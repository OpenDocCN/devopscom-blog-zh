# 4 容器编排安全问题

> 原文:[https://devo PS . com/4-container-orchestration-security-concerns/](https://devops.com/4-container-orchestration-security-concerns/)

根据 DevOps Institute 的 [2021 年技能提升调查，51%的受访者表示，集装箱编排是一项必备的技术技能。所以，毫不奇怪，](https://info.devopsinstitute.com/2021-upskilling-report-download?utm_campaign=Upskilling%202021&utm_source=partner-devops-dot-com-register-co)[容器技术](https://devops.com/?s=containers)变得越来越流行。然而，容器环境远比其他基础设施更加复杂和动态，这使得容器安全性至关重要。安全性确保容器按预期运行，保护基础设施、软件供应链、运行时以及两者之间的一切。

如果您希望在您的软件基础设施中使用基于容器的应用程序，您可能需要记住这项技术带来的安全风险。为了提供帮助，我请了 [DevOps Institute 大使](https://www.devopsinstitute.com/about-us/ambassadors/) 来权衡他们对容器编排的安全顾虑。 下面是他们分享的:

[**安舒尔·拉利特**](https://www.linkedin.com/in/a-lalit/) **，康斯伯格数字技术与改造负责人**

“容器编排最常见的安全问题之一是它共享主机操作系统和主机内核。因此，它提出了一个问题，即主机操作系统或内核漏洞可能会在容器中被利用和利用。这方面的一个例子是最近对 Linux 的 mmap syscall 的攻击。虽然您可以通过使用定义的最佳实践来避免这种情况，但值得注意的是，主机操作系统中的任何危害都可能危害该主机中的容器。这种情况与虚拟机(VM)形成对比，在虚拟机中，来宾操作系统、主机操作系统和内核不共享公共二进制文件，因此本质上是分离的。

同样，一个流氓进程、一个正在运行的易受攻击的容器映像以及 orchestrator 中的一个安全漏洞都可以为攻击者提供一个进入大门的途径。例如，如果容器被设计为以 root 权限运行，那么在编排主机上运行漏洞扫描器的攻击者将很快找到易受攻击的容器。

用户应遵循最佳实践，通过提供 Linux 安全模块来保护在容器编排平台中运行的容器，例如，Linux 安全模块可以实施策略来防止流氓容器或用户从主机系统提取敏感数据。"

[**supra tip baner JEE**](https://www.linkedin.com/in/supratipb/)**，解决方案架构师，全球服务负责人**

“容器为云安全带来了新的挑战。虽然这些结构是动态的和自动化的，但是新的威胁载体也存在，原因与容器有用的原因完全相同。DevOps 和 SecOps 团队必须意识到并准备好应对这些可能的问题:

*   与虚拟机不同，容器共享它们执行的主机操作系统。如果没有正确维护设置，主机和容器可能会面临安全问题。
*   流程编排自动化增加了复杂性，可能导致过度配置访问的错误配置，从而扩大攻击面。
*   使用保存在公共或私人储存库中的图片来构建容器；他们经常依赖于其他图像，其中任何一个图像中的一个缺陷都可能传播到数千个集装箱。
*   因为集装箱是短暂的，基于 IP 地址的安全措施是无效的，并使取证调查复杂化。"

[**帕维恩·阿罗拉**](https://www.linkedin.com/in/parveen-arora-india/) **，VVnT 续集联合创始人兼导演**

“如果你的项目涉及到容器的使用，你将把你的应用程序分解成不同的组成服务，这虽然有益，但不一定像你使用虚拟机那样安全。由于像 Docker 这样的容器平台从主机上借用代码库，它们从根本上来说不如 VM 安全。因此，容器环境中的漏洞可能允许恶意软件突破容器并感染主机环境。”

[**Vishnu Vasudevan**](https://www.linkedin.com/in/vishnuvasudevan811/)**产品工程与开发负责人 Opsera**

“安全挑战，尤其是在容器编排方面，是人们认为他们可以构建自己的映像，或者他们可以从云中下载映像并开始使用。这不像大家想的那么简单。每分钟都有漏洞被引入。您需要一个安全工具和流程，能够自动扫描每个被拆分的容器。此外，团队需要围绕蓝绿部署或容器变更的回滚创建共享的实践，这些变更将被推入生产环境。

仅在疫情，我们就看到了显著的云移动，但平均每天都会出现 50 个漏洞。事实上，有人只是在等待漏洞，在那里他们可以执行恶意脚本。有些人，尤其是从事挖掘的人，试图了解云帐户中的漏洞，并渗透到这些漏洞中，以运行巨大的容器来实现他们自己的云挖掘目的。平均来说，检测到任何漏洞或入侵系统都需要 280 天。

使用容器的公司必须将威胁漏洞管理作为其协调策略的一部分。你认为这个团队有足够的经验一天扫描 50 个漏洞并修复它们吗？可能不会，因为发布速度正在将现代应用计划的发布周期缩短到一到两周。这产生了大量的技术债务，这些债务会迅速积累，这就是为什么在迁移到云时必须考虑威胁漏洞策略和流程。”
