# SLI、SLA 和 SLO 有什么区别？

> 原文:[https://devo PS . com/what-the-difference-sli-SLA-and-SLO/](https://devops.com/whats-the-difference-between-sli-sla-and-slo/)

如果你刚刚接触站点可靠性工程(SRE)或平台工程，你可能会遇到一堆新术语，比如 SLI、SLA 和 SLO。这些基准在 SRE 人的日常生活中经常被提及，但对外人来说可能有些陌生。那么，这些缩写有什么区别呢？

嗯，[SRE 方法](https://devops.com/site-reliability-engineering-sre-comes-of-age-in-2022/)的一个标志是评估性能并为应用程序指标设定目标。这些目标可以被认为是可靠性阈值。例如，如果一个站点的延迟达到某个点，它可能会破坏软件提供商与其消费者之间的协议。

sre 创建并监控各种指标的服务水平性能基准，如正常运行时间、延迟、错误计数、[平均恢复时间](https://devops.com/why-mttr-is-a-vital-metric-for-devops-teams/)和吞吐量等。但并非所有这些绩效目标都是公共协议的一部分。公司通常采取更主动的方法来设定更多的内部目标，以远离破坏外部协议和用户信任的行为。设置这些基线来保留功能性的、可靠的系统已经成为一种最佳实践。

如上所述，目标的类型从坚定的合作伙伴承诺到更多的内部目标。下面，我们将尝试用简单的术语来定义服务水平协议(SLA)、服务水平目标(SLO)和服务水平指标(SLI)之间的区别。虽然这些概念有许多相似之处，但理解它们在实践中的应用是很重要的。请记住，每个公司可能对这些术语有细微的理解，并可能以不同的方式构思和应用它们。

## 服务水平指示器(SLI)

首先，服务水平指标(SLIs)指的是软件服务产生的实际指标。这是对服务行为的直接度量。这些是真正的数字，*表明了*的整体性能，比如错误率和延时。

SLI 的一个例子是在一个月的时间内所有请求中成功请求的数量。假设一名工程师使用了一个应用程序监控工具，该工具从生产日志中获取数据。该数据显示，在一百万次请求中，有十次请求失败。因此，可用性的 SLI 就是 99.999%的正常运行时间。

## 服务水平协议

第二，大多数技术人员都熟悉服务水平协议(SLA)。即使你不是，在你作为数字用户的历史中，你可能已经同意了许多。SLA 就像是软件供应商和软件用户或客户之间的协议。这些有约束力的承诺通常会注明必须遵守的可用性预期。SLA 还可能包括对事件和错误的响应。这取决于合同，但如果违反了 SLA，可能会产生某种惩罚，如退款或服务订购积分。

如今，普通企业依赖于许多基于云的 SaaS、PaaS 和 IaaS。仅仅启动像网上支付这样简单的事情就可能需要访问多个远程服务器。但是，如果所有这些服务在它们想离线的时候就离线，我们的数字操作将会嘎然而止。因此，SLA 对于企业软件合同协议来说是必要的，以确保双方满足特定的标准。

一旦工程师跟踪了 SLIs 并对典型行为有了概念，他们就可以设置有意义的 SLA。如果我们以上面的例子为例，SRE 可以考虑保证一个比被主动监控的阈值更低的阈值。这将确保应用程序的性能在当前的速度下不会违反任何法律或合同义务。在这个场景中，也许 SLA 提到了可用性目标:每一百万个请求中失败的请求不超过 100 个。这基本上相当于 99.99%的正常运行时间。

## 服务水平目标(SLO)

最后，服务级别目标(SLO)类似于 SLA，但明确指的是性能或可靠性目标。SLA 可能指特定的 SLO。或者跟踪 SLO 只是为了内部目的。正如 Google 描述的那样，“SLA 中的可用性 SLO 通常比内部可用性 SLO 更宽松。”

您不希望最终用户成为第一个抱怨移动 web 应用程序延迟增加 400%以上的人。因此，SRE 团队通常会密切关注性能，以确保他们从未接近违反 SLA。有些人通过设置和监控比 SLA 阈值更高的内部基线来做到这一点。

以上面的例子为例，如果一个 SLA 保证了 99.99%的服务正常运行时间，那么企业可能会设定一个 99.995%的内部目标。换句话说，对于每一百万个请求，不应该超过 50 个请求失败。如果软件系统没有达到这些标准，这表明公司必须重新评估设计，寻找瓶颈。或者，也许工程团队的目标是改善下一季度的平均停机时间——内部*目标*可以设定为比当前感知性能更高的标准。

正如谷歌云博客所描述的，“每项服务都应该有一个可用性 SLO——没有它，你的团队和利益相关者就无法做出原则性的判断。”他们补充说，这是一个很好的提醒，提醒你在可靠性承诺方面要适度，因为消费者将会期待这种水平的性能。另一方面，设定更雄心勃勃的内部绩效目标的好处是提供比书面协议更好的结果，增加软件服务的竞争力。

## 真实世界的性能基准

请记住，以上数字仅用于演示目的。真实世界人物的一个有趣资源是 [API。Expert](https://www.api.expert/) ，一个查询流行 API 并发布每周性能统计的服务。由于 API 是许多基于 UI 的平台(以及我们的[数字经济](https://hbr.org/2021/04/apis-arent-just-for-tech-companies))的核心，这些基准可以作为行业平均正常运行时间和延迟的良好指标。

比如写的时候，API。专家的[企业 API](https://www.api.expert/collection/enterprise-it)集合将微软 Office365 和 Pivotal Tracker 排在首位，两者的通过率均为 100.00%，延迟分别为 220 毫秒和 248 毫秒。另一方面，Docusign 为 99.93%，延迟为 877 毫秒，Box 为 99.99%，延迟为 414 毫秒。

## SLIs，SLAs 和 SLOs——哦，我的天！

尽管对于没有经验的人来说这听起来不错，但是 99.99%的 SLA 仍然相当于每年 52 分 36 秒的停机时间。这相当于将近一个小时的停机时间，客户在这段时间里绞尽脑汁，或者更糟糕的是，寻找其他选择。对于创伤医疗情况，失去连接可能是生死攸关的问题。

尽管创建 SLA 和 SLO 对于衡量系统健康状况很重要，但现实情况是跟踪和执行它们可能很有挑战性。“这些协议——通常由不在技术领域的人撰写——通常做出团队难以衡量的承诺，”根据亚特兰大知识中心的说法。

总之，sli 展示了软件系统的真实行为。这些指标为 SLA 的创建提供了信息，必须满足 SLA 才能确保 B2B 协议。这些 SLA 通常引用必须满足的特定服务级别目标(SLO ),这通常为 sli 提供了更多的喘息空间。最后，在数字经济中，随着数字期望值的提高，监控内部 SLO 并随着时间的推移改进基线是有意义的。
