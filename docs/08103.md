# 是的，您确实需要 SCA

> 原文：<https://devops.com/yes-you-do-need-sca/>

我们经常去餐馆，吃一些我们不熟悉的异国食物，这些食物的成分我们只是模糊地知道。厨师和他们的团队(或者经理和他们的团队)是我们的保证，那里的东西不是致命的。大多数时候，这样做很好；但是，很少，我们最终付出了代价。从超市带回家的食物也是如此，尽管对于较小的疾病来说更难追踪，因为他们不像餐馆顾客那样聚集在一起。虽然致命的结果确实极其罕见，但美国疾病控制和预防中心估计，美国每年约有 3000 人死于食源性疾病，另有数百万人患病。美国有令人难以置信的安全食品，然而，我们仍然有这些疾病。即使在安全的环境中，简单的事情，如将原料放在外面太久，或食用来自受污染的生产设施的食品，也会引起问题。

我们的代码是相似的——它是如何被创建和消费的——并且越来越多地属于这一类。我们依赖大量的资源来运行我们的网络系统，而且，很多时候，没有人告诉我们我们的资源有多安全，或者是否发生了我们注意不到的变化。

这就是为什么我们绝对必须有[软件组成分析](https://devops.com/?s=software+composition+analysis) (SCA)。如果没有其他人会站出来验证我们引入的所有代码的安全性，那么我们至少需要知道那里有什么。我们需要理解底层软件的内部关系，包括已知的和未知的。有些库无处不在，以至于你正在使用它们，甚至可能不知道。但那不是好的管理；知道自己拥有什么，并照顾好它，越来越成为它的核心部分。了解在您首选的(或自动的)包含出现问题的情况下有哪些替代方案可用对于连续性来说是必不可少的，这使得 SCA 成为连续性规划的关键部分，同时也是良好的业务实践。

好消息是，如果您正在使用 DevSecOps 空间中的任何主要代码扫描工具，您要么已经有 SCA 可用，要么可以从同一供应商那里购买它作为附件。点击他们的网站，找到答案。

您的系统每天都在为用户的需求服务。保护这些用户免受供应商问题、开源变更、安全漏洞和嵌套依赖等不可预见情况的影响。虽然绝大多数开源软件都是安全可靠的，但是我们使用了太多的开源软件——我们使用的工具链接了太多的开源软件——几乎可以肯定在你的代码中隐藏着一个炸弹。想象一下上面的食源性疾病的例子:你会在一家说“是的，我们 90%的食物是安全的，但我们不确定剩下的 10% …”的餐馆吃饭吗？

你在建造令人惊叹的东西，让世界继续前进。不要停下来，但也不要放松警惕。了解这些应用程序中运行的是什么，其漏洞(安全性和停机时间)是什么，以及您可以做些什么来减轻这些弱点——这样您就可以继续做出令人震惊的东西，而不是花时间修复已经存在的东西。
