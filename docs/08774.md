# 30 年的 Linux 教会了软件行业什么

> 原文:[https://devo PS . com/what-30-years-of-Linux-teached-the-software-industry/](https://devops.com/what-30-years-of-linux-taught-the-software-industry/)

在过去的 30 年里，Linux 已经成为计算机历史上最大的合作开发项目。反思是什么使这成为可能，以及它的开源哲学是如何最终在行业中实施的，可以为软件供应商从这个惊人的成功故事中提供宝贵的经验。

网络可能还没有完全成熟，但它已经创造了自己的神话。

1991 年 8 月 25 日:来自芬兰的 21 岁大学生 Linus Torvalds 在新闻组的一个小组上写了一篇文章:“大家好，正在使用 minix 的人——我正在克隆公司做一个(免费的)操作系统(只是一个爱好，不会像 gnu 那样庞大和专业，价格是 386 英镑(486 英镑)[……]”。几周后，这个最终被称为 Linux 的项目首次发布。

这是一部很少有人能预见的史诗的起点。

一晃 30 年过去了，Linux 内核不仅运行在全球大多数网络服务器和智能手机上，而且还支持几乎所有更新的云基础设施。如果没有像 Linux 这样的开源程序，云计算就不会出现。

推动 Linux 走向成功的主要因素之一是安全性。今天，世界上最大的软件公司正在将开源安全性提升到新的水平，但是 Linux 项目是最先强调这一点的项目之一。

## Linux 如何成为现代 IT 世界的支柱

### 简史

开源比 Linux 项目早很多年，可以说和软件本身一样古老。然而，正是后者的成功推动了上世纪 90 年代的这场运动。当 Torvalds 在 1991 年第一次提交贡献时，Linux 内核是 GNU 项目到一个完全自由的软件操作系统的“缺失的一环”,它可以不受限制地分发甚至出售。在接下来的几年里，随着该项目开始包含专有的许可组件并越来越受欢迎，澄清“自由软件”的含义变得很有必要。

这导致了我们今天使用的“开源”一词的产生，部分归功于 Eric Raymond 的开创性论文[The Cathedral and The Bazaar](http://users.ece.utexas.edu/~perry/education/382v-s08/papers/raymond.pdf)，这是一篇“对黑客社区和自由软件原则的反思性分析”选择开放源代码是为了使软件合格，根据许可证的条款，公众可以使用或修改其原始设计的源代码。然后，人们可以下载、修改并发布他们的源代码版本(fork)回社区。

由于 Apache HTTP Server、MySQL 和 PHP 等软件在互联网上运行第一个动态网站的流行，开源项目在 90 年代后期开始受到关注。

### 准确的情报

今天，不仅是 Linux 驱动了数字时代的大部分，而且开源已经成为我们构建和发布软件的主导模式。虽然大多数人没有意识到，但我们每天依赖的许多技术都运行在自由和开源软件(FOSS)上。电话、汽车、飞机甚至许多尖端的人工智能程序都使用开源软件。根据 Linux 基金会的数据，全球前 100 万台服务器中有 96.3%运行在 Linux 上，所有云基础设施中有 95%运行在 Linux 上。其他基础设施也依赖于开源:70%的全球移动用户使用在 ONAP(开放网络自动化平台)构建的网络上运行的设备。

Linux 在专业 IT 领域的采用率非常高，已经成为事实上的标准，尤其是随着云时代的到来。事实上，83.1%的开发者说 Linux 是他们更喜欢使用的平台。这一成功在很大程度上要归功于自其创建以来为其源代码做出贡献的社区:来自 1500 多家公司的 15000 多名开发人员。可以说，Linux 后来成为自由软件运动中最成功的故事，证明了开源可以创造出与公司销售的任何软件一样强大的软件。

Linux 基金会是一个成立于 2000 年的非盈利技术联盟，旨在支持 Linux 和 OS 软件项目的合作开发，它本身就是一个巨大的成功。它现在旗下有 100 多个项目，分布在人工智能、自动驾驶汽车、网络和安全等技术领域。几年来也出现了几个子集基金会，包括 Cloud Foundry 基金会、有影响力的 Cloud Native Computing 基金会和最近宣布的开源安全基金会。该基金会估计，其社区的集体贡献创造的总共享价值高达 541 亿美元。

如果没有企业界对开源的支持，所有这些成就是不可能的，这可能是它最大的胜利。

### 企业采用

公司开始意识到，与要求他们的开发者从头开始一遍又一遍地构建互联网业务的基础部分相比，许多开源项目更容易实施，成本也更低。

20 年前，大多数企业运行在微软、甲骨文和 IBM 的专有软件上，在大型软件项目上合作的想法对他们来说可能听起来很可笑。如今，这些公司，以及相对较新的公司，如谷歌、脸书和亚马逊，不仅雇佣了数千名全职贡献者从事 Linux 等开源项目，他们还定期选择开源一些最先进的项目；从谷歌大脑的机器学习平台 TensorFlow 和容器编排平台 Kubernetes 到脸书的 React。

毫无疑问，开源软件创造了一波新的商机。随着越来越多的公司对开源项目感兴趣，他们意识到自己不一定有管理这些项目的内部专业知识，于是转向初创公司和更大的公司寻求帮助。

就连以反对 Linux 概念近十年而闻名的微软，在首席执行官塞特亚·纳德拉的领导下，也在 2010 年代做出了拥抱开源的战略转变。这家 IT 巨头终于在 2016 年加入了 Linux 基金会，并在两年后收购了开源项目的最大主机 GitHub。从那以后，它成为了开源项目的最大赞助商之一。

因此，开源软件的风险增加了，开源软件是推动几乎所有公司向云转变的引擎。在这种情况下，安全性成为一个最重要的话题，保护开源生态系统的承诺也在快速增长。

## 设定安全和信任的标准

### 开源安全问题

随着开放源码软件的采用热潮，这些软件包的可持续性、稳定性和安全性现在是每一个使用它们的公司主要关心的问题。

关于现代供应链中结构和安全复杂性的普查二报告 揭示了两个令人担忧的趋势，这两个趋势可能使自由/开源软件更容易受到安全漏洞的攻击。首先，该报告称，在个人开发者的账户下发布流行的软件包是很常见的，这引发了安全性和可靠性的问题。其次，经常可以看到过时版本的开源程序在使用，这意味着它们包含的安全补丁更少。

[2021 OSSRA 报告](https://www.synopsys.com/content/dam/synopsys/sig-assets/reports/rep-ossra-2021.pdf) 对此表示赞同:“过去一年审计的 98%的代码库包含至少一个开源组件，开源占全部代码的 75%。”该报告还指出，85%被审计的代码库包含“过期四年以上”的组件。

这凸显了“未受管理的”开源软件带来的日益增加的安全风险:“84%的经审计的代码库包含具有已知安全漏洞的开源组件，高于前一年的 75%。同样，60%的代码库包含高风险漏洞，而 12 个月前这一比例为 49%。”不仅安全状况受到影响，而且由于许可证可能会相互冲突甚至不存在，开源内容的无监督集成也会产生合规性问题。

因为大公司现在是开源生态系统的一个重要组成部分，他们的赞助对许多人来说是一个受欢迎的资金来源，他们的工作到目前为止都是免费的，但这可能还不够。开源社区以其对独立性、归属感和自给自足的承诺而闻名，期望贡献者自愿解决安全问题不太可能成功。

在这一点上，30 年来构建 Linux 和协调成千上万个人贡献者的工作的经验是值得效仿的。

## Linux 基础

在 Linux 内核开发中，安全性是非常重要的。因为它是数字世界中许多公共和私有软件“砖块”的底层，这意味着任何错误都会给企业带来数百万美元的损失，如果不是生命的话。从一开始，它就采用了一种分散的开发方法，大量的贡献者不断地协作。因此，随着社区发展工作的增长和扩大，它巩固了一个强大的同行审查流程。

撰写本文时的最后一个稳定版本是 5.14，发布于 2021 年 8 月 29 日，距离项目 30 岁生日只有几天。该版本中最重要的功能是与安全相关的:一个旨在帮助缓解处理器级漏洞，如 Spectre 和 Meltdown，另一个涉及系统内存保护，这是一个主要的攻击面。每个 Linux 内核版本每周都有近 100 个新的补丁，由来自英特尔、AMD、IBM、Oracle 和三星等公司的个人和专业人士提供。

随着如此广泛的采用和悠久的历史，Linux 项目已经达到了其他自由/开源软件项目很少见到的成熟水平。审查过程和发布模式为众多下游供应商树立了信心。虽然这个世界并不完美，而且可以说他们很难跟上如此高的变化率，但他们至少可以从强大的安全实施机制中受益，并且他们可以根据他们的“风险偏好”调整他们的安全状况:供应商能够进行计算，确定他们可以容忍用户接触多旧的内核。

## 拓展开源安全的边界

### 心脏出血和操作系统安全性的脆弱性

2014 年 4 月，影响 OpenSSL 密码库的重大安全事件；披露为“心脏出血”引入这个 bug 的开发人员承认，尽管他和少数其他工程师一起从事这个项目:

“我要对这个错误负责，因为我写了代码，由于疏忽错过了必要的验证。不幸的是，这个错误也通过了审查过程，因此进入了发布的版本。”OpenSSL 是一个开源项目，广泛用于实现传输层安全(TLS)协议。换句话说，它是用来保护大部分网络的基础部分。

开源在很长一段时间内被认为是基本安全的，因为检查一行代码的人越多，发现任何弱点的机会就越大。此外，这种模式防止了“通过模糊来实现安全性”，即大部分保护来自于不知道安全软件如何工作的人，如果机密信息被泄露或被外部发现，这可能导致整个体系崩溃。

这一事件对于大部分最大的网络公司来说是一个重要的转折点:他们意识到支撑他们核心运作的许多开源技术不能再被认为是安全的了。任何人为错误都可能产生巨大的影响；因此，必须做出特别的努力来提高这个特定空间的安全性。

### 开源的新时代

随着我们进入一个开源无处不在的时代，开源在代码库、工具、网络和基础设施中无处不在，甚至在软件以外的领域中也是如此，安全意识开始占据主导地位。但是它需要更多的工作。

首先，挑战的很大一部分是行业对问题范围的理解。

谷歌 [刚刚宣布](https://security.googleblog.com/2021/10/introducing-secure-open-source-pilot.html) 将承诺“1 亿美元支持管理开源安全优先级并帮助修复漏洞的第三方基金会。”

由 Linux 基金会运行的安全开源(SOS)试点项目将奖励开发者，因为他们增强了我们都依赖的关键开源项目的安全性。

通过这样做，谷歌在扩大像公司和政府这样的大玩家的财政赞助方面起了带头作用——这些大玩家越来越多地直接或间接地赞助开源软件。然而，他们也建议组织“理解他们对自由/开源软件生态系统的未来的影响，并遵循一些指导原则。”

这些原则会是什么样的呢？

## 安全使用和贡献开源的路线图

Linux 基金会提出了一个 [特定的信任和安全倡议](https://linuxfoundation.org/wp-content/uploads/improving_trust_security_in_oss_projects.pdf) ，它描述了开源团队应该使用的八个最佳实践(具有三个成熟度等级)来保护他们生产的软件，以及由更大的受众来“提高集体安全标准”他们在这里:

*   明确角色和职责，并确保每个人都知道他们在整个组织中的安全职责。

*   为每个人建立安全策略；换句话说，组织的所有成员都有一颗清晰的北极星。

*   “了解你的贡献者”被定义为一套实践，以基于风险做出决定，决定谁该信任谁，并对抗攻击性网络战技术，如上游代码中毒。

*   锁定软件供应链:这已经成为一个首选目标，因为对手清楚地知道，与针对单个系统相比，他们可以用更少的努力产生更大、更有效的影响。

*   提供技术安全指导，将潜在的解决方案缩小到在安全性方面更合适的解决方案。

*   部署安全行动手册以定义如何执行特定的安全流程，特别是事件响应和漏洞管理流程，如创建角色和职责或发布安全策略。这可能感觉很正式、过时和老派，但拥有预定义的剧本意味着团队可以专注于软件的发布，而不是学习如何实现安全性，尤其是在最不方便和最有压力的时候。

*   强烈推荐使用自动化测试来开发安全测试技术，因为它的伸缩性更好，对团队来说摩擦更少，成本更低，并且与现代的连续交付管道保持一致。

然而，该指南的作者意识到该行业仍然面临一些重大挑战，因此需要加以解决。他们提到:

*   缺乏开源安全测试工具
*   事实上，开源软件包的发行已经被打破了
*   漏洞披露的 CVE 格式也被打破的事实
*   缺乏安全构建证书标准，该标准允许任何消费者透明地验证产品或组件是否符合公布的规范

“验证的类型可以而且应该包括使用自动安全工具，如 SAST、DAST 和 SCA，以及验证安全流程，如回购中是否存在安全回复邮件以及安全回复电子邮件是否有效。”

像这样的计划可能会对开源软件和互联网的安全质量产生重大而持久的影响。

30 年前诞生的 Linux 项目，如今出现在现代软件栈的所有层中。它被所有支持现代网络的大型服务器集群所使用，任何走向数字化的企业都会在某个时候使用它。这种无与伦比的寿命和成功证明了开放源码模型与企业级服务的需求是兼容的，并且在经济上是可行的。既然开放源码在软件行业风靡一时，就如何确保这个生态系统的可持续性达成共识并制定行动计划就变得刻不容缓。对于依赖 it 的企业来说，当务之急是采用强有力的应用程序安全准则，就像 Linux 基金会提倡的那些准则，它们已经证明了自己的价值。

关于开源本质的最后一点:由于企业现在更多地被开源组件的普遍使用所束缚，他们不应该落入“公共悲剧”的陷阱。这意味着等待，直到其他人采取行动；例如，改善全球软件安全状况。这可能是我们高度协作的行业面临的最大挑战之一。
