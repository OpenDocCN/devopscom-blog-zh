# 成功定位 ML 开发人员和团队

> 原文:[https://devo PS . com/positioning-ml-devs-and-teams-for-success/](https://devops.com/positioning-ml-devs-and-teams-for-success/)

智能应用程序(本质上)是复杂的。传统软件基本上由一个东西(代码)组成，而智能软件包括代码、模型和数据。正如 [之前讨论过的](https://devops.com/what-is-mlops-dataops-and-why-do-they-matter/) ，三个截然不同的领域——devo PS、 [MLOps](https://info.verta.ai/applying-mlops-at-scale) 和 data ops——已经发展到管理这些相互关联的学科中的每一个。

![](../Images/1d8d42d4c40710da97ea6b48aa47fdcd.png)

快速有效地通过 ML 生命周期需要团队在上述每个领域的合作。组织应该努力使这种合作尽可能地顺畅，以使他们的 ML 努力取得成功。以下是我们认为组织应该关注的四个方面:

*   基础设施
*   团队结构
*   工具
*   项目所有权和 KPI

### 为每个支柱构建专用基础设施

基础设施软件在代码、模型和数据这三大支柱上并不是一刀切的。例如，DevOps 团队在软件开发、支持应用程序或产品服务中一直扮演着基础设施的角色。想想由 DevOps 支持的常规软件工程团队，他们生产不同种类的应用服务。应用程序基础设施是软件开发的一部分，以至于我们大多数人都认为这是理所当然的。

就像常规的软件服务需要 DevOps 一样，数据和模型服务也需要相应的基础设施。数据基础设施通常涉及管理工具(如气流装置、Spark 集群和功能存储)，构建数据管道，并确保数据服务对数据消费者保持顺畅和可操作。

另一方面，模型服务主要关心以一种其他人可以有效使用的方式创建、运行和组织模型。由于 AI/ML 生命周期极其复杂，需要考虑许多因素(例如，硬件、并行化库、模型打包格式等。)来帮助数据科学家和工程师高效地完成工作。强大的 ML 基础设施以及专门的 ML 基础设施团队在帮助组织扩展其 AI 方面有很大的帮助。

### 优化团队结构

人工智能和人工智能的最佳实践在不断发展，多年来组织已经尝试了许多团队结构。我们注意到近年来出现了两种流行的智能应用程序开发团队结构:

*   **豆荚**

Pod 结构涉及较小的团队——通常是数据科学家、数据工程师和应用软件工程师——紧密合作来交付智能功能或产品。由于他们紧密结合的本质，这些团队定位于极快的迭代，这通常会带来更好的产品和特性。然而，pods 紧密结合的本质也带来了不利的一面，因为不同目标的 pods 之间的知识共享可能是困难的。

![](../Images/0114565e4bd1c40ad6da9f5a14d585c9.png)T2】

*   **卓越中心**

在卓越中心(COE)结构中，所有数据科学家或数据工程师都被“集中”起来，然后根据项目需求和环境分配到不同的产品或功能。与 pod 相反，COE 结构倾向于非常容易地共享知识。然而，这种安排会导致更慢的迭代周期，因为团队成员并不致力于特定的产品或特性。

![](../Images/54f147bf17e1e9376c20eb0246de827c.png)T2】

我们认为，根据一个组织的实际情况，特遣队所属装备和特遣队所属装备安排都可能有效。选择最佳的团队结构取决于组织的其他部分是如何构建的，以及如何建立报告关系。

### 选择合适的工具

要为您的组织选择最好的工具，请选择那些促进协作并维护代码、模型和数据团队安全的工具。

合适的工具有助于团队快速高效地工作。团队应该能够为自己发现相关的信息，而不必把其他人从他们正在做的事情中拉出来。例如，数据科学团队应该能够发现哪些数据访问或了解存在哪些功能。类似地，应用程序团队应该能够发现存在什么模型，什么版本是最新的，或者如何以及何时使用模型。您越能使您的代码、模型和数据自助化，团队就能越快地构建出可以投入生产的产品。

此外，合适的工具使调试更快更容易。如果你已经接触了 ML 使能的特性或产品有一段时间了，你可能已经被未声明的依赖和由于上游变化导致的应用中断所困扰。在智能应用中，找到问题的根源可能极其困难，因为根源可能存在于三大支柱中的任何一个或三者的任何组合中。强大的工具提供日志和信息，有助于更容易地发现问题所在。

最后，治理应该在工具选择中发挥关键作用，因为安全应该是 ML 团队的主要关注点。例如，考虑一个基于用户的应用使用预测用户人口统计的模型。虽然这些预测在某些应用中可能被认为是可接受的，但如果模型被用于提出财务建议或批准信贷申请，使用某些人口统计数据可能会违反各种消费者保护法。由于使用该模型的软件工程师可能不知道与其使用相关的细微差别，因此对模型和数据使用进行安全检查变得至关重要。

### 建立所有权并确定适当的 KPI

无论我们是在公司工作还是在学术实验室工作，我们的目标都是一样的:制造最好的产品。

但是“最好的产品”可能意味着许多不同的东西，这取决于用例。例如，有时团队可能愿意牺牲一点准确性，如果一个模型以可预测的方式失败(例如，在医疗保健应用中。)相比之下，电子商务推荐系统可以优化整体模型准确性，即使模型不太稳定。因此，选择合适的[KPI](https://devops.com/?s=KPIs)进行模型优化需要非常小心。T3】

类似地，选择对产品和业务有意义的适当指标通常需要对许多因素有一个高层次的看法——用户期望、收入、用户投诉等。根据我们的经验，产品负责人通常是做这些决定的最佳人选。支持智能应用的每个支柱的 KPI 应基于产品所有者确定的关键指标。虽然这些 KPI 可能会有所不同，但我们已经看到以下 KPI 表现良好:

*   代码——发布版本的数量、错误修复的数量、已发布的特性的数量、解决问题的平均时间
*   模型—交付的模型迭代次数、生产中的模型质量、模型性能 SLA
*   数据—数据刷新的频率(例如，每天还是每月？)、数据质量问题的数量、访问数据的速度

这里的关键点是基于产品 KPI 来推动所有这些，并关注敏捷原则，即您移动的速度有多快以及正在进行多少部署。

了解您的团队希望优化哪些指标，以便您可以将资源投入到正确的地方。
