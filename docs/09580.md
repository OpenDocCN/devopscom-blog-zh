# JFrog 将 DevOps 产品组合与微软团队集成在一起

> 原文:[https://devo PS . com/jfrog-integrated-devo PS-portfolio-with-Microsoft-teams/](https://devops.com/jfrog-integrates-devops-portfolio-with-microsoft-teams/)

JFrog 本周宣布，它已经与微软团队整合了 DevOps 产品组合，以加强合作。

JFrog 的高级产品营销经理迪普·达塔表示，JFrog Artifactory repository 和 JFrog Xray 软件组合分析(SCA)工具已经通过一个适用于微软团队的 JFrog 应用程序工具进行了集成。

达塔说，我们的目标是让 DevOps 团队更容易与一个组织的其他成员协作，这个组织已经在微软团队的协作上实现了标准化。

达塔指出，总的来说，随着越来越多的组织意识到他们的业务是多么依赖软件，从应用程序所有者到网络安全专业人员的个人现在都在寻求与 DevOps 团队合作。他补充说，事实上，在许多情况下，随着企业越来越依赖软件，业务流程工作流现在需要与 DevOps 工作流保持一致。

达塔说，虽然许多 DevOps 团队已经使用了一系列协作工具，但这些团队中的许多人还需要访问微软团队，以便与这些团队以外的个人进行互动。

新出现的挑战是 DevOps 团队现在发现他们自己使用的沟通渠道的数量。每一个新增加的沟通渠道不一定会取代现有的平台；根据所需的消息传递和[协作](https://devops.com/?s=collaboration)的类型，开发运维团队使用微软团队以及 Slack 和传统电子邮件的情况并不少见。在某些情况下，不管是好是坏，DevOps 团队发现他们需要在任何应用程序开发项目的过程中处理多个协作平台。

总的来说，在[数字业务转型](https://digitalcxo.com)的时代，开发运维团队和业务其余部分之间的关系也在不断发展。应用程序交付和更新的速度现在对业务目标有着[的实质性影响](https://devops.com/?s=value+stream+management)。组织发现他们需要将从市场营销和客户支持到财务规划的一切都与软件开发结合起来。一个软件开发项目的延迟会产生连锁反应，影响到业务的方方面面。越早通过协作平台共享信息，其余业务就越容易适应和响应。

自然，并不是每个 DevOps 团队都对所需的透明度级别感到兴奋。许多软件开发团队以前会在内部处理的问题现在暴露在董事会的议程中。收入目标比可用的开发人员专业知识对应用程序交付和更新期限的影响更大。管理所有这些期望的关键是更频繁地沟通。然而，所有这些通信也会占用构建应用程序的时间。将开发团队与干扰隔离开来是很重要的——比如与多个业务线和多个沟通渠道共享的持续不断的消息。

当然，唯一比这些信息更糟糕的是未能及时传达关键信息。面临的挑战是找到一种方法来过滤掉所有的噪音，并在正确的时间将关键信息传递给正确的人。
