# 云原生成熟度模型 2.0

> 原文:[https://devops.com/cloud-native-maturity-model-2-0/](https://devops.com/cloud-native-maturity-model-2-0/)

KubeCon 2021 在北美推出了 [云原生成熟度模型](https://github.com/cncf/cartografos/blob/main/reference/prologue.md) ，该模型由 [Cartografos 工作组](https://github.com/cncf/cartografos) 推出，旨在帮助采用者和最终用户在 CNCF 环境和更广泛的云原生生态系统中导航。作为发布会的一部分，该集团还推出了图书 [巴什上将的岛屿冒险。](https://www.cncf.io/wp-content/uploads/2021/10/Admiral-Bash.pdf)

今天，我们很高兴地宣布云原生成熟度模型的更新。该小组与 CNCF 社区的新成员合作，包括 Annalisa Gennaro、Marcello Testi 和 Jonathan Gershater，以及现有成员 Danielle Cook、SImon Forster、Robert Glenn 和 John Forman，他们花了六个月的时间审查该模型，并与 TAGs 进行交流，以确保该模型能够反映市场变化的速度。

成熟度模型包括 5 个级别，每个级别涵盖人员、流程、策略和技术。这些按关键主题分类。以下是一些更新的概述。

**新部分:业务成果**

作为成熟度模型的一部分，我们认为重要的是不仅要概述对技术的期望，还要概述业务的期望。今天我们推出新的 [业务成果](https://github.com/cncf/cartografos/blob/main/reference/business_outcomes.md) 部分。根据该模型，我们考虑了业务领导(首席执行官、首席财务官、董事会成员等)对原生云的期望。

这包括确保业务目标与原生云如何帮助您实现这些目标保持一致。一些例子可能包括:

*   扩展到 100 万用户:在任何给定时间根据用户提供灵活、可扩展的基础架构，并配备在出现问题时的快速故障转移功能。
*   提供卓越的客户体验:确保应用程序可靠，不会让用户失望
*   更快地将功能推向市场:支持微服务方法来构建应用。较小的团队更敏捷，因为每个团队都有一个专注的功能。API 最大限度地减少了构建和部署所需的跨团队交流。

虽然云原生并不是一个线性的发展过程，并且会经历挫折，但是云原生成熟度的第一步是了解如何衡量成功(KPI)以及如何向利益相关方展示这些成功。

一些定量和定性的示例 KPI 可以包括:

*   通过优化成本，应用基础设施支出减少 25%
*   开发成本降低 10%
*   通过尽可能自动化，团队对应用基础设施的关注减少了 15%
*   通过自动识别集装箱中的 CVE 提高了应用的安全性
*   提高合规性，因为您可以限制和跟踪对应用程序的访问；证明符合 SOC 2
*   加快开发生命周期，因为您实施的 CI/CD 管道每季度交付 10%以上的功能
*   迁移计划–这将因您的组织而异，但您应该有一个适当的迁移计划。无论是先迁移一个应用程序，还是先迁移几个应用程序，您都应该做好准备。
*   通过增加净推介值(NPS)来衡量客户体验的改善
*   消除信息孤岛:各部门不再孤立；独特的集成生态系统到位。
*   业务和 IT 目标的一致性:每个人都参与并意识到这一点，以便更好地分配资源，高效地实现这些目标。
*   增加内部交流:交叉授粉提供了共享知识的新视角。

您可以在 GitHub 上查看完整的 [业务成果。](https://github.com/cncf/cartografos/blob/main/reference/business_outcomes.md)

**流程更新**

流程部分现在包含了关于 GitOps 的信息以及建立 CI/CD 卓越中心的要求。我们还调整了第三级，以包括标准如何出现的信息。

**技术更新**

我们致力于通过技术进一步提升安全性。我们还致力于进一步将基础设施集成为代码。

**接下来的步骤**

Cartografos 工作组致力于保持成熟度模型的更新和相关性，以便各种规模的组织在经历各个阶段时都可以使用它。作为其中的一部分，我们接下来的步骤包括:

*   重新评估模型结构和布局
*   继续改进已发布的工件
*   网站整合到 cncf.io(目标)
*   与标签合作并将材料整合到模型中
*   审查工作组章程
