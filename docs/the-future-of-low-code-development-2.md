# 低代码开发的未来

> 原文：<https://devops.com/the-future-of-low-code-development-2/>

对自动化的恐惧(自动化焦虑)从 16世纪就有了。这是对技术和机器持消极态度的主要原因(也称为技术恐惧症)。随着更复杂的[低代码和无代码开发解决方案的出现，即使是软件开发人员也面临着与他们的工作安全相关的焦虑。](https://devops.com/?s=low-code+no-code)

虽然低代码解决方案有能力通过自动化冗余的业务流程来 [快速改善客户体验](https://www.bonitasoft.com/news/improve-customer-experiences-business-process-automation) ，但是它们能完全取代开发人员吗？下面的指南将试图通过分析当前的趋势和即将到来的进步来窥视低代码开发的未来。

## 了解低代码开发最近的流行

低代码描述了工具、平台和其他技术，它们允许您用尽可能少的代码构建应用程序。许多这样的平台让你通过一个拖放界面来构建整个网络或电话应用程序。

然而，大多数中小型企业使用这些低代码工具来构建简单的后台服务，以帮助优化员工工作流。最近我们看到了 [快速软件开发](https://www.bonitasoft.com/priorites-for-cios-2022) 以及敏捷、DevOps 等部署方法论的流行。

许多企业已经开始整合跨职能协作模式来优化这些渠道。这催生了被称为融合团队的多学科开发团队。这些团队通常由软件开发人员和非技术业务分析师组成，他们可能没有编码经验。

IT 领导者发现，弥合融合团队成员之间差距的最佳方式是通过低代码开发平台。这些平台中的许多已经开始推出 [新功能](https://www.bonitasoft.com/news/bonita-2021.2-for-low-code-development-and-fusion-teams) 来促进融合团队之间的顺利协作。

低代码平台总是被用来创建紧急业务问题的基本解决方案。然而，最近，他们已经取得了进展，以解决新的问题领域。

## 低代码简史

Microsoft Access 是商业无代码/低代码工具的最早例子之一。它允许您使用表单和其他图形工具来构建数据库。该工具最初旨在帮助用户在不使用 SQL(或 Transact-SQL)的情况下构建小型个人数据库。然而，员工们开始在同事之间共享这些 Access 创建的数据库。

最终，大公司将开始围绕这些数据库构建完整的工作流。这可能成为这些公司的负担，原因有很多。使用完全专门化的关系数据库管理系统，如 Microsoft SQL 或 MySQL，是更安全的选择。

组织仍然必须使用经验丰富的软件开发人员和数据库管理员的服务。这些专业人员可以确保数据库安全并符合 HIPAA 等 [标准以及欧盟的《通用数据保护条例》(GDPR)。](https://www.atlantic.net/hipaa-compliant-hosting/mysql-and-ms-sql-and-hipaa-compliance/)

因此，尽管 Microsoft Access 允许非技术用户在没有任何 SQL 知识的情况下创建数据库，程序员仍然需要将这些点连接起来。像 Access 这样的低代码工具允许商业用户在 [引入开发者](https://dev.to/study_web_dev/with-no-code-builders-why-should-clients-hire-a-developer-53ip) 扩展核心思想之前开发概念和证明市场。

然后，开发人员将重塑数据(为了优化)并围绕这些 Access 数据库构建应用程序。

## 低代码开发的现状

如今，更多复杂的低代码开发平台(LCDPs)可用于构建桌面和移动应用。我们已经看到由云原生 SaaS 应用支持的低代码工具，允许用户轻松构建可扩展的架构。这些 LCDPs 可以让您构建定制的移动、桌面和 web 应用程序，这些应用程序可以满足接受支付等高级用例，也可以满足限制访问某些内容等简单用例。

软件开发人员的重点是编写代码；这就是他们的日常生活。因此，他们可能没有专业人员的在职经验或他们编写应用程序的用例。谁比医生和其他医务人员更有条件编写 [健壮的医疗应用](https://www.jotform.com/blog/medical-apps/) ？谁能比其他建筑工人更能充分同情建筑工人的困境？

这就是为什么“[公民开发者](https://devops.com/?s=citizen+developers)”运动变得如此突出。事实上，Gartner 进行的研究显示， [超过 41%的非 IT 专业人士](https://www.gartner.com/en/newsroom/press-releases/2021-02-15-gartner-forecasts-worldwide-low-code-development-technologies-market-to-grow-23-percent-in-2021) 定制或构建自己的软件解决方案来解决业务问题。尽管有这些统计数据，企业仍然需要开发人员的投入。问题域刚刚发展起来。

## 我们将何去何从？

尽管 LCDP 和 NCDP 工具变得越来越复杂，但它们仍有其局限性。在未来，企业将能够以几种方式解决这些限制。

### 人工智能和机器学习

机器学习和人工智能将在低代码/无代码技术方向产生巨大影响。像微软的 AI builder 这样的平台已经提供了一种低代码方法来构建机器学习驱动的自动化软件。然后，您可以将这些模板构建的 ML 模型集成到您的业务应用程序或软件栈中。

我们已经看到它们是如何在医疗和兽医通信软件 [中被用于自动调度](https://www.getweave.com/industry/veterinary/) 和通信的。因此，LCDPs 可用于使机器学习更容易被非编码人员使用。

然而，机器学习也可以使低代码平台更加有效。例如，ML 驱动的数据挖掘可以用来发现最新的更新和商业问题的最佳解决方案。它将允许 ncdp/lcdp 向公民开发者提供更相关和定制的建议。

在未来，通过文本、UML 用例图或语音简单地提供一个规范，就有可能构建一个自动化过程或应用程序。这允许更灵活的项目组合选项。

改善用户体验是当今 IT 领导者 最重要的 [要务之一。机器学习结合低代码将帮助企业快速应对这些挑战。](https://www.bonitasoft.com/priorites-for-cios-2022)

### 雇佣专业开发人员来克服低代码的局限性

最终，业务所有者和非 IT 人员会遇到低代码解决方案根本无法解决的问题——至少不是现成的。在这种情况下，最好雇用一名开发人员。

对于更复杂的用例，企业仍然需要专业应用程序开发人员的服务。即使机器学习和人工智能为未来的低代码解决方案提供动力，情况也将如此。

事实是，虽然业务用户可能对业务对象之间的关系有更深的理解，但只有训练有素、经验丰富的软件开发人员才能在代码中有效地实现这些概念。

因此，无代码不会导致专业开发人员的灭绝。相反，它将促进公民开发者和专业开发者之间更好的合作。这对于中等复杂的项目尤其重要。

### 必须解决安全问题

该行业最近的趋势以及对网络安全和网络安全的高度重视是软件开发人员不会过时的另一个原因。例如，许多开发人员发现他们需要更加关注 [API 安全](https://www.redhat.com/en/topics/security/api-security) ，这要归功于诸如 [Log4j](https://community.bonitasoft.com/blog/log4j-library-issue-does-not-affect-bonita) 等 API 中被发现的漏洞。

因此，他们必须[在开发管道的早期集成安全性](https://devops.com/?s=DevSecOps),以防止任何可能的数据泄露和破坏。大多数低代码工具使用 API 来构建应用程序。不幸的是，公民开发人员不能像专业应用程序开发人员那样保护他们的应用程序。即使 LCDPs 定期接收安全更新，也不足以保护业务应用程序免受未来的网络攻击。

同样，这也是市民和专业开发人员之间的合作如此重要的原因。只有后者才能真正确保商业软件得到适当的保护。

### 更好的协作是低代码的未来

未来的无代码和低代码工具必须能够方便地促进专业开发人员和非开发人员之间的协作。这些平台应该为开发人员提供相对容易和安全的方法来访问代码，尽可能自由地修改和扩展代码。

在大多数情况下，软件开发人员已经有了他们更喜欢在工作流程中使用的框架和工具。从 IDEs、CI/CD 自动化到测试软件；低代码平台应该能够插入或连接这些已建立的环境和工具。

同样，这个属性非常重要，尤其是随着敏捷和 DevOps 等开发和交付方法的广泛流行。这些技术和模型需要跨职能团队之间的顺畅协作。LCDP 越灵活越好。我们已经看到现代 LCDPs 不必要的编程约束如何阻碍跨职能设置的进展。这些缺点必须得到解决。如果 lcdp/ncdp 做不到这一点，它们在市场上的存在很可能是短暂的。

## 结论

我们不会看到低代码平台在不远的将来完全贬低专业软件开发人员。然而，它们将改变开发者构建软件的方式。对程序员的业务和工作要求也将发展，因为低代码有可能改善开发人员和非开发人员之间的协作。

在允许有经验的开发人员扩展和构建这些想法之前，公民开发人员可以使用低代码工具来为他们的最终产品建模。因此，这些低代码开发平台对于普通开发人员来说必须易于访问和使用，但是对于专业应用程序开发人员来说限制更少，更加灵活。他们必须允许专业开发人员更自由地编码。这将是未来几年低代码平台开发人员关注的焦点。
