# 应用程序现代化报告显示需要特定于 Kubernetes 的迁移工具

> 原文:[https://devo PS . com/application-modernization-report-shows-need-for-kubernetes-specific-migration-tooling/](https://devops.com/application-modernization-report-shows-need-for-kubernetes-specific-migration-tooling/)

Konveyor 是一个充满热情的人组成的社区，他们通过构建工具、识别模式，并就如何打破铁板一块、采用容器和拥抱 Kubernetes 提供建议，来帮助其他人实现应用程序现代化并将其迁移到混合云。为了做到这一点，Konveyor 社区目前正在致力于五个项目，以帮助用户将他们的应用程序重新托管、重新平台化和重构到 Kubernetes。

虽然与用户的对话和各种来源的数据让我们对迁移过程有了一些了解，但我们希望通过最近的详细数据来告知我们的优先事项。为此，Konveyor 社区贡献者 Red Hat， [发起了一项针对美国、英国和讲英语的亚太地区(APAC)的 600 名 IT 决策者、后端开发人员和软件架构师的调查](https://www.konveyor.io/modernization-report/) ，以更好地了解应用程序现代化的前景。

以下是一些与 Konveyor 和应用程序现代化相关的发现:

**应用程序现代化正在进行**

很明显，推出迁移工具迫在眉睫。在我们调查的那些正在迁移或计划迁移的客户中，我们发现他们计划在未来一年内实现 54%的自定义应用程序的现代化，并在未来六个月内实现超过四分之一的现代化。只有 20%的人预计现代化需要两年以上的时间。很明显这正在发生。

也就是说，这不是一帆风顺的，也不是一蹴而就的。虽然一些地区在应用程序现代化的道路上还没有取得足够大的进展，没有遇到严重的障碍，但其他地区(尤其是美国)报告称，选择一种简单的方法具有挑战性。组织有许多选项可供选择，每个选项都呈现不同的复杂程度。

**将现有应用升级到 Kubernetes 的激励因素**

调查告诉我们，对可靠性、可扩展性和安全性的需求是推动现代化的主要因素。降低成本也在列表中，但正如我们在其他关于技术采用或变革的调查中经常看到的那样，这不是驱动因素。

Konveyor 的工具套件可用于帮助实现组织的迁移计划。

根据这项调查，Konveyor 工具套件与公司打算使用的迁移策略密切相关:重新托管、重新平台化和重构 。

**使用叉车和起重机将应用重新托管到云**

重新托管(占总数的 20%)包括将应用程序复制到云中(无论是托管的还是本地的),只做最少的更改，即“提升和转移”这对于任何有虚拟化经验的人来说都很熟悉，当时裸机服务器通常被移至虚拟机(VM)以获得更高的效率，但对软件的更改很少或没有更改，并且操作更改相对较少。

虽然再托管不能提供与平台重组或重构同样深度的好处，但这是朝着正确方向迈出的第一步。主机更换是最常见的方法

[叉车](https://forklift.konveyor.io/) 和 **起重机** 都是这里相关的康威项目。叉车专注于将虚拟机迁移到 Kubernetes，并提供在最短停机时间内将多个虚拟机迁移到 KubeVirt 的能力。

Crane 是一种满足不同用例的再集成工具。Crane 帮助组织在 Kubernetes 集群之间迁移应用程序。很多时候，开发人员和运营团队希望在新旧版本的 Kubernetes 之间进行迁移，撤出集群，或者迁移到不同的底层基础设施。

**使用 Move2Kube 重新构建应用程序**

重新平台化(占总数的 18%)是使用云迁移来升级操作系统、数据库和其他所需组件，以便在不改变核心应用程序代码或架构的情况下支持云应用程序。把它想象成“举起、修补和移动”

这实际上是重新托管和重构之间的一个中间步骤。其实往往是敲门砖；调查发现，90%计划重构应用程序的人计划首先重新搭建平台。

**Move2Kube** 是一个 Konveyor 项目，它允许客户将他们的应用程序移植到 Kubernetes 编排的平台上。正在进行平台移植的一个领域是将容器编排平台整合到 Kubernetes。

由于这种整合，Move2Kube 项目开始专注于加速从 Swarm 和 Cloud Foundry 等平台到 Kubernetes 的平台转换过程。这个项目将现有的工件转换成 Kubernetes 工件，以加快能够在 Kubernetes 上运行应用程序的过程。

**用攻略重构应用**

重构(17%)通常被认为是最具挑战性的应用程序现代化策略，但却是最具长期影响的策略。重构包括将架构重新设计为云原生架构，例如，将工作负载容器化，将整体架构转变为微服务，或将功能转移到无服务器架构。

虽然不是一个完整的重构工具，但它提供了一系列相互关联的工具，允许用户评估、分析并最终将他们的应用程序迁移到 Kubernetes 编排的平台上。

正如你所看到的，组织正在采取深思熟虑的方法来进行应用程序迁移，没有单一的正确途径。这些数据支持 Konveyor 社区提供工具箱的方法，而不仅仅是一个单一的工具，它可以支持应用程序的过渡。阅读应用现代化报告的完整状态 [这里](https://www.konveyor.io/modernization-report/) 。
