# 保持德沃普斯信念的红帽文化

> 原文:[https://devops.com/redhat-culture-keeping-devops-faith/](https://devops.com/redhat-culture-keeping-devops-faith/)

Red Hat 因其在 Linux 和软件设计方面的成就而广受认可。Red Hat DevOps 文化本身最初是作为一个项目设计的，该项目旨在培训开发人员的操作技能，并增加操作人员对这些开发人员的信任。红帽公司的 IT 平台运营经理安德森·席尔瓦告诉 DevOps.com，内部团队是如何在这个 Linux 开发图标内部发起文化变革的。

回到队伍里去！

Silva 说，IT 部门的平台运营团队管理并执行了这些环境中的大量 IT 托管应用程序部署。在建立 Red Hat 内部 DevOps 王朝的过程中，在管理平台运营团队之前，Silva 在一个名为生产控制的部门领导了一小群运营人员。在那里，Silva 和他的团队建立了控制机制，以确保任何新的部署都不会也不可能破坏生产环境。

“有时，这意味着我们不得不暂停整个部署，因为开发人员忘记添加一个链接到我们的发布文档，将它们与请求部署许可的其他人放在一起，”Silva 说。Silva 认为生产控制本身是一个瓶颈，而不是一个部署解决方案，他认为这是对 Red Hat 的一个挑战，将变得越来越重要，越来越难以应对。在 DevOps 流行之前的那些日子里，一位 Red Hat 开发经理提出了一项计划，将瓶颈转化为可以加快部署过程的解决方案。根据 Silva 的说法，这第一个计划是红帽称为上尉计划的根源，该计划支持愿意获得[红帽认证工程师](https://www.redhat.com/en/services/certification/rhce)认证的开发人员，以便他们可以享受生产和预生产访问，并根据他们的喜好频繁地管理他们自己的部署。Red Hat 最终添加了一个“语料库”组件作为附加的类似程序。这些“语料库”中的开发人员获得了 QA 权限，并不需要获得 RHCE 认证。

引人入胜的结果

“这两个项目在过去几年中增加了巨大的价值。Silva 说:“通过在开发人员和运营人员之间建立信任，我们解决了许多过去被视为‘他人问题’的技术问题。”Silva 还拥有 RHCE 和 RHCA 认证，并积极维护 Fedora 软件包。Red Hat 继续培养一种灵活的 DevOps 文化，并培养源于工程师和开发人员之间信任的灵活性。

这种灵活性正在带来回报。Silva 说，以前，Red Hat 需要一周的时间来开发和部署大约 20 台服务器的软件，而现在，Linux 巨人一天可以完成几次这样的发布。“如果在不久的将来，我们能够在必要时每小时做几次，我不会感到惊讶，”席尔瓦说。

通过保持运营和开发之间的信任，Red Hat 可以限制迁移到混合云可能带来的风险。这种基于 DevOps 关系的文化也将有利于 Red Hat 新的基于映像的部署，这些部署依赖于这种文化成员中常见的工具，如[Red Hat Enterprise Linux open stack 平台](https://www.redhat.com/en/technologies/linux-platforms/openstack-platform)、 [OpenShift](https://www.openshift.com/) 、Kubernetes(一种数据中心工具)和 [Docker](https://www.docker.com) 。

“作为一家开源公司，我们尽可能使用开源解决方案，并且我们编写了自己的工具来帮助我们实现自动化部署，”Silva 说。随着 Red Hat 的平台运营团队即将淘汰其中的一些工具，以支持其他工具，它还将向其他内部 it 部门推广基于映像的部署，这也利用了 [Jenkins](https://www.cloudbees.com/) 、 [AWS](https://aws.amazon.com) 、 [Puppet](https://puppetlabs.com/) 、 [Ansible](https://www.ansible.com) 和 [Git](https://git-scm.com) 。

几个经理在部门内播种的 DevOps 文化已经结出了果实，形成了自上而下的领导范例，奖励和鼓励创新的开发人员。Red Hat 召集了包括系统管理员和开发人员在内的临时团队来解决困扰开发的问题。席尔瓦说，成功的关键是共同应对这些挑战，而不是责怪某个人。

免费建议

Silva 建议，那些想要努力培养 DevOps 文化的企业领导人，在处理 DevOps 方法的众多选择中，无论什么看起来新的和闪亮的，都要先关注他们的员工。“相信你的人，让他们做好自己的工作。当错误发生时，不要责怪别人；寻找解决方案。然后看看有什么新的和闪亮的，”席尔瓦说。
