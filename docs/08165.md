# 一些技术债务可以自行解决

> 原文:[https://devops.com/some-technical-debt-is-self-resolving/](https://devops.com/some-technical-debt-is-self-resolving/)

但是不要指望它。

DevOps.com 的另一位撰稿人 Bill Doerrfeld 最近写道，“[企业面临着越来越多的技术债务](https://devops.com/enterprises-face-growing-technical-debt/)”，这是一篇关于技术债务趋势的伟大文章，它让我想起了过去几年该行业一直在发生的一些事情。

正如我在之前提到的[，我们系统的潜在变化，加上保持这些变化之前就存在的系统存活的需要，导致了相当高的债务比率，而且债务即将到期，可能比你想象的要早。](https://devops.com/our-api-mess-is-coming/)

我们已经在 CI/CD 工具中内置了跟踪技术债务消耗时间的增强方法，但是这些工具存在固有的弱点。跟踪花费在问题 X 上的时间并将问题 X 归类为债务的能力需要一个可靠的分类过程，并依赖工作人员密切跟踪他们的时间。无论对错，该行业目前(也许是永久地)已经不再以小增量跟踪时间，取而代之的是，大多数组织跟踪结果。但是除非你知道你的主线开发工具集之外的系统保持运行所花费的*时间百分比*，否则你无法确定技术债务到底有多糟糕。

虽然有一些亮点，但我(当然还有其他人)一直在警告技术债务，因为 DevOps 团队都被允许选择他们自己的工具链(很高兴这基本上已经结束了)，我们需要意识到反作用力也将有助于缓解技术债务问题。

首先是简化事物的趋势。让我们面对现实吧，在开发运营方面，Python 正以疯狂的速度流行起来。这有一个*非常*充分的理由。即使是优秀的 IT 人员也不可能遵循由脚本大师为实现某个目标而编写的 Shell 脚本。你不能理解的东西将需要更长的时间来修复——因为你必须把它拆开，找出各个部分，解析大量复杂的模式匹配，等等。Python 简化了这一切。如果您编写一个 Python 脚本来做与 bash 脚本相同的事情，那么很大一部分人会看一眼就说，“我知道这是做什么的。”这种趋势在整个 IT 领域都存在(bug dashboards 是另一个更高层次的好例子)，随着更容易理解的工具越来越多地成为组合的一部分，旧的和晦涩难懂的工具将不再是债务。

第二，虽然我对 API 使用的增长和 CI/CD 工具集的复杂性提出了警告，但事实是这两件事都减少了我们内部维护的代码量。自动化的步骤，无论是通过调用其他人的 API，还是调用您的 CI/CD 供应商来处理构建或部署步骤，都是我们需要担心的较少的步骤(当然，只要它们能够工作，并且有值得信任的合作伙伴)。

但是我们仍然需要意识到技术债务，因为我们在过去提到的所有事情——我们的环境越来越复杂、人员流动、团队特定的工具和语言——都在推高债务。

请咨询您的 CI/CD 供应商，看看他们的债务跟踪产品是什么样的。其他不需要跟踪花费在单个活动上的实际时间的方法包括记录 bug 的数量。找到你的弱点的一个可靠的方法是寻找没人想使用的软件，或者有人警告新人不要接触的软件。那是你债务负担最重的地方。到处都是修补匠；“如果它没坏，就不要修理它……”只适用于“……因为这里没有人确切知道它是如何工作的”也适用。

让您的团队列出他们所在领域(无论是 silos 还是 DevOps 团队)需要更新和/或替换的工具，然后对它们进行排序。然后把这些数据带给团队领导，让他们决定你有限的时间和金钱应该用在哪里。提供你在上面收集的相对债务的数据，这样就可以比较团队排名和债务成本。

然后，留出时间来减轻债务。它不一定是巨大的，只是继续前进。你为各种各样的事情留出时间，即使一周几个工时的债务也会成倍增加。开始会很慢，但是一旦你每周花在修复“产品 X”上的时间减少了，其中的一些时间可以添加到技术债务解决上，增加可以做的事情。寻找低垂的果实；一个占用大量时间的小应用程序或脚本，或者一个维护起来极其昂贵的系统，而市场已经超出了它在你的部门中的实际用途，等等。

然后享受你的劳动成果，把那些惊人的技能和额外的时间用于为你的用户开发更多的新应用。
