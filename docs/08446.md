# 开发运营和持续变革

> 原文:[https://devops.com/devops-continuous-change/](https://devops.com/devops-continuous-change/)

一位同事在等待咖啡机完成循环时的一句话开启了我的思路。"我们应该在一年内发布多个小版本还是只发布几个大版本？"

在大型组织中，由于许多因素，单个成功发布的周转时间相当长；但是，如果我们在使用#DEVOPS 方法时所谈论的概念上的改变不仅会减少这些长周期，而且会交付更快更高质量的软件。

我已经在我之前的文章“DEVOPS:让我们的组织保持一致”中讨论了传统方法，以及这两者(开发和运营)的结合如何在各种组织中被证明是有益的，以及有多少人仍在犹豫是否走上这条道路。

**持续变革**包括持续集成和交付，对于实现这一目标至关重要。为了达到定期向涉众交付快速和一致的价值的目标，我们必须有持续的变化发生。

我确信我们已经从大型机时代进入了一个更加 T2 化的分布式系统时代。但不幸的是，我们还没有摆脱这种文化，即我们将组件堆积在一个准分布式应用程序上，而不是创建更小的独立但相互依赖的系统。

所以，回到同一个问题；我们是应该进行一些恐龙大小的费力的释放，还是进行多次更短更快的奔跑？在我看来，我们应该朝着更精简的模式努力。你说起来容易做起来难！同意。如果必须的话，我会这样做:

在具有一套分布式应用程序的环境中，从分析组成主要版本的组件开始，列出计划的更改(修复和新功能)，评估功能和技术影响，确定依赖性、上游/下游连接、体系结构缺陷，按功能列出已确定的组件子集或组件组，可以作为独立的版本。

我忍住了说“独立”的冲动，因为对于当前复杂的应用程序干预集合，这将是一个不正确的说法。它更像是可以独立升级的组件的子集。现在最大的任务是审查和决定这个子集或组是否会在没有其他组件加入的情况下为整个应用程序套件增加功能价值。

一旦确定，此版本的成功工作预期将在这些组件的广泛测试(单元/回归/性能)中，并且当我们在此拆分和部署过程中获得信心时，我们可以开始优先考虑和安排关键必须具有的功能或错误修复，这些功能或错误修复也应该与落后的其他组件向后兼容。我们甚至可以想出这样的场景，其中这些组件不仅仅是频繁更新，还会有多个独立的轨道。当然，不言而喻，测试和签署过程是严格遵循，虽然在一个较短的窗口。

生产部署一直是运营团队心痛的来源，我们经常看到相互指责和遏制变化的冲突失败。为了保持稳定性并最大限度地提高可用性，实施了更严格的控制并缩短了部署时间。在促进频繁的和较小的组件级变更时，OPS 也将控制正在发生的事情，并且如果变更没有按预期工作，较小的组件级版本可以在绿区窗口内快速回滚。虽然我将这种倒退归类为过程失败，而不是部署失败，并且应该通过正确的缓解计划和经验教训回顾进行非常认真的审查，但是在每一次成功或失败的 sprint 之后的持续学习可以确保胜任和自信的发布。

影响多次运行生产版本的另一个主要因素是长时间的**绿区**窗口，通过转向更快、更精简的部署周期，这些请求可以大大减少，并最终可以达到在线部署变更而无需长时间停机的阶段。

同样，在没有计划部署的情况下频繁进行维护活动，实际上也存在环境不可用的风险。分析和转移到这些活动在线进行而没有绿区的状态将减少维护停机时间，并确保无缝可用性，增加利益相关者的价值。

由于明显的原因，我没有提到部署过程的**持续集成**或**自动化**，如果没有集成过程，我们就无法实现**持续变更**的目标，在集成过程中，代码的增量变更被快速无缝地构建到一个包中，甚至可以集成健全性或基本测试来确保构建的质量。自动化部署也是如此，开发快速部署所需的&集成工具。我们已经看到这样的例子，自动化一个活动或重新审视一个过程已经节省了大量的时间。

虽然对**发布流程**提出变更或提倡快速部署周期很容易，但我们必须确保适当的检查和平衡到位，遵守合规性并创建审计跟踪。

现在来看本文最重要的衡量标准，**成本节约！**

恐龙发布版本的支持者会争辩说，需要一个发布版本的公共活动，如构建、测试、发布管理、操作可用性或减少的 GZ，会降低成本，而多次运行会增加成本。

但这绝对是另一种方式，我们不再需要喂养恐龙，这将是一个值得成本的快速交付，具有优化的资源，一个更精简的流程，开发和运营部门共同努力，以确保更快地向业务提供高质量的增值交付品..这种文化变革和合作努力也创造了新的发展前景，并提供了一个完美的平台。成本效益分析最终会支持持续交付。

**机会成本**是另一个没有考虑的变量；在这个竞争激烈的时代，客户可以快速获得创新或新功能，缩短上市周期，从而获得急需的优势。我记得在一些对话中，由于各种原因，一些必须要做的事情被推掉了，使业务受挫。

DEVOPS 从未打算取代传统的软件开发方法；相反，它是在一个协作模型中对现有资源的有效利用，以快速交付高质量的软件产品。我们的方法需要一个重大的文化转变，为了达到这个目标，我们需要接受灰色的阴影，很多时候超过 50 倍(抱歉，无法抗拒)，当它不是真正的黑色或白色。
