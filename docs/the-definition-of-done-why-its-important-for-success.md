# 完成的定义:为什么它对成功很重要

> 原文：<https://devops.com/the-definition-of-done-why-its-important-for-success/>

在软件组织之间建立清晰的沟通和期望对于任何项目的成功都是至关重要的。如果每个人都在同一页上，公司就不太可能经历意想不到的延迟或其他不愉快的惊喜。为了确保所有团队成员同步，并消除不必要的歧义，工程团队应该创建一个完成的“[定义来指导项目。](https://devops.com/guide-for-software-development-software-security/)

### 「完成的定义」是什么？

在[敏捷软件开发](https://www.agilealliance.org/agile101/)中，完成的定义是团队之间的工作协议，它定义了一套标准来标识“完成”的事情因为敏捷是软件开发的迭代方法，有时不清楚任务何时完成。产品经理经常报告说，当“开发完成”而不是“质量保证完成”时，就可以交付了换句话说，代码已经写好了，但是还没有创建或执行任何文档或测试。对于软件组织的其他部分，没有 QA 或文档的代码并没有“完成”直观地说，完成的定义是为了消除对“完成”在整个组织中的含义的混淆。

### 如何为您的软件组织创建完成的定义

对于一些软件组织来说，完成的定义是在项目开始时建立的一套静态的指导方针和基准。对其他组织来说，包括 Allstacks，工程师们有自由修改他们所做的定义。这种方法使工程团队能够做出能够产生最佳结果的决策。

在工程组织中建立完成的定义时，考虑以下提示:

●以整个软件组织能够普遍理解的方式交流标准。确保完成的定义同时满足战术和技术要求。
●创建一份活的调整文档，可以在不可预见的情况下进行修改，如失去团队成员或重大市场变化。
●考虑到文档的健壮性将随着团队的成熟度而变化，特定的团队可能需要特殊需求的附录。例如，与两个人的商店相比，更大的企业 IT 团队经常需要更详细的文档。
●在所有团队中建立一种理解，直到所有标准都满足，用户故事、任务或项目才算“完成”

完成的定义承认前端、后端和 QA 之间的共同责任。除了这些角色之外，其他利益相关者可能会对定义“完成”做出贡献，比如 Scrum masters、董事会成员或业务分析师，这取决于组织及其结构。此外，产品所有者还可以定义“验收标准”,目标是最终用户体验，工程团队也必须将此作为他们工作的一部分。

### 完成的定义如何帮助团队取得成功

done 的定义有助于软件组织保持项目的一致性，并使每个人都在同一页上。错位会带来严重的后果。例如，它可能会在产品真正准备好之前就与客户展开过早的对话，在内部利益相关者之间制造摩擦和/或给其他上市计划带来风险。当“完成”的定义包含了项目到任务级别的定义步骤时，敏捷团队可以在整个组织中创建更多的凝聚力，并消除不可预见的延迟。

完成的定义概括了软件工程项目中的每一项任务。记录每项任务的完成情况可以作为项目是否在既定期限内完成的指标。通过记录完成每项任务所需的时间，工程团队可以衡量项目的速度，这有助于保护到期日期或发出需要改变时间表的警告信号。

### Done 的定义有助于支持软件团队 OKRs

目标和关键结果(OKRs)是一个目标设定框架，帮助高层领导关注对组织重要的事情。通过定义软件工程 okr，高级开发人员领导可以让他们的团队专注于将工程目标和活动与业务成果相结合，同时也给他们实验的空间。

创建工程 okr 包括接受组织的目标，制定可操作的、通用的工程目标，同时确定推动进展的指标和里程碑。为工程团队翻译组织的目标有助于他们准确理解他们的工作如何影响公司。

OKR 框架的主要好处之一是给工程团队一个共享的语言和成功的愿景。但是，没有一个既定的 done 定义，OKRs 会有很多无意的差距(团队也会有很多误解)。从“完成”的定义开始，以确保每个团队都知道他们的部分是什么，以及何时可以正式移交给另一个团队。团队将在整个组织中顺利地传递接力棒，在市场发布的竞争中将会有更少的障碍。

### 完成的定义可以帮助组织变得更聪明

在一个只有少数软件员工的小型创业公司中，建立对时间表和目标的共同理解是很容易的。随着软件组织的成长，在饮水机旁的随意聊天不足以让组织保持一致。所有团队成员都必须理解并参与目标和目的，这些目标必须是书面的，并且每个人都可以访问。建立 done 定义的好处包括为软件交付创建一个共享的理解和统一的语言，确保新员工获得部落知识和过程期望，以及检测和去除团队之间的虚假协议以创建真正的共识。通过在一个组织的历史早期创建一个“完成”的定义，公司领导可以确保在更多人加入时保持一致。
