# 嵌入式 SREs 的利弊

> 原文:[https://devops.com/the-pros-and-cons-of-embedded-sres/](https://devops.com/the-pros-and-cons-of-embedded-sres/)

嵌入还是不嵌入:这是一个问题。

至少，这是公司在决定如何实施 [现场可靠性工程](https://rootly.com/blog/what-is-an-sre) 时必须回答的问题之一。他们可以将 sre 嵌入到现有的团队中，也可以建立一个新的、独立的团队。

这两种方法各有利弊。当然，对你的公司或团队来说，正确的策略取决于你的需求和优先事项。

## 什么是嵌入式 SRE？

嵌入式 SRE 作为非 SRE 团队的一部分工作。例如，他们可能加入开发或 IT 运营团队——尽管他们也可能是质量保证团队、 [安全团队](https://rootly.com/blog/when-you-do-devsecops-don-t-forget-the-sres) 或组织内任何其他可以从他们的专业知识中受益的单位的一部分。

嵌入式 SRE 与创建专用的通用 SRE 团队相反。在后一种方法中，一个组织雇佣与非 SRE 团队合作的人才，但这些人才并不直接融入他们。

还有其他几种[](https://cloud.google.com/blog/products/devops-sre/how-sre-teams-are-organized-and-how-to-get-started)组织模式介于这两个极端之间。但是典型地，两种主要的结构类型是嵌入式的或者专门的团队。

## 嵌入式 SREs 的优势

将 SREs 嵌入到其他团队中有几个好处。

可以说，最重要的优势是嵌入式 SREs 确保了现场可靠性工程师和其他利益相关者之间最高水平的合作。当他们直接与其他工程师一起持续工作时，他们的观点会影响其他工程师做出的每一个决定。独立的团队通常仍会与其他团队合作，但不一定像嵌入式 sre 那样投入。

嵌入式 sre 对于站点可靠性工程概念仍然是新的，并且利益相关者仍然需要了解 sre 做什么以及它们可以带来的价值的组织也是有益的。当你提议建立一个完全独立的团队时，要获得认同可能有点困难；事实上，开发人员或 IT 工程师甚至可能将这样的团队视为对自己工作的威胁(尽管事实上，当然， [他们补充其他团队](https://rootly.com/blog/de-siloing-incident-management-how-to-make-reliability-engineering-everyone-s-job) 而不是与之竞争)。但是当你在更大的团队中加入 sre 时，可能更容易让每个人都接受这个概念，也更容易让其他类型的工程师理解 sre 如何让他们的工作变得更容易。

最后，嵌入式 sre 非常适合小型组织，不需要整个团队。如果你的员工中只有几十名工程师，而不是几百名，雇佣几名 sre 加入你现有的团队可能比投资一个全新的团队更有意义。

## 嵌入式 SRE 的挑战

嵌入式模型也有它的缺点。

一个主要的风险是，SRE 在一个更大的团队中的影响力可能不足以对该团队的运营产生重大影响。例如，一个 SRE 与十几个开发人员或 IT 工程师一起工作，可能很难将站点可靠性工程技术和工具集成到现有团队的工作流中——特别是如果团队已经对如何操作有了强烈的意见。

类似地，在使用嵌入式模型的团队中工作的单个 SRE 可能最终会被拉得太瘦而无法很好地完成他或她的工作。如果其他团队成员希望他们“拥有”本应是集体、团队责任的职能，那就更是如此了——比如应对突发事件或使用混沌工程 测试可靠性的 [。后面的过程可能由 sre 领导，但它们不应该是 sre 一个人的责任，当其他团队成员认为 sre 是他们可以简单地将所有可靠性相关任务“转储”给他们的人时，这个功能可能会开始崩溃。](https://rootly.com/blog/creating-chaos-to-achieve-reliability)

最后，嵌入式模型可能很难扩展，这也是它在大型组织中不能很好工作的原因之一。如果您有几十个不同的工程团队，将 SRE 嵌入每个团队需要大量的管理工作。它还剥夺了您组织内各种 sre 之间密切合作的能力。在这种情况下，更有意义的做法可能是创建一个专门的团队，并根据需要让这些 sre 与其他团队进行交互，而不是试图通过将现场可靠性工程师直接嵌入其他团队来将他们分散到整个企业中。

## 结论

因此，一般来说，嵌入式 sre 对于那些拥有相对较小的工程组织的公司来说最有意义，在这些公司中，功能的概念还没有被完全接受。对于拥有众多工程团队的大型企业来说，只为他们创建一个单独的团队可能会更容易管理 SREs。

也就是说，你的里程数可能与标准有所不同，可能值得尝试一下，以确定哪种模型(嵌入式或非嵌入式)对你的公司最有意义。请记住，您还可以使用混合方法，将 SREs 嵌入到一些团队中，同时保持一个专门的团队，领导整个组织的站点可靠性计划。

毕竟，站点可靠性工程就是要创造性地思考，并提出创新的解决方案来应对挑战，甚至是解决如何在您的组织内构建 SREs 的挑战。
