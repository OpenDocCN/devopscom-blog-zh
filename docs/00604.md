# 赛博靶场、云计算自动化和开发运维

> 原文:[https://devo PS . com/cyber-ranges-cloud-automation-devo PS-Alex-hen thorn-iwane/](https://devops.com/cyber-ranges-cloud-automation-devops-alex-henthorn-iwane/)

网络安全测试代表了从传统方式转换到开发方式的一些挑战。虽然网络安全实验室和网络靶场通常是发现自己成为网络攻击的高价值目标的组织的关键投资，但太多时候网络安全测试流程是孤立的和高度手动的。是时候让赛博靶场走向 DevOps 了。

网络靶场本质上是实验室基础设施，用于各种与安全相关的活动，包括单点产品测试、网络拓扑级测试以及红队/蓝队练习等培训。虽然网络靶场通常由政府和军事组织使用，但越来越多的商业组织(如金融机构和公用事业公司)正在建设这些基础设施，以确保它们尽可能地准备好预测、响应和挫败攻击。特别是当不断增长的移动性和多孔性的 IT 边界不可避免地会受到损害时，快速有效的响应能力至关重要。这意味着除了产品测试，开发响应技能和系统是一个关键的功能。

所有这些当然都是好的非常重要的东西。那么问题出在哪里？嗯，像许多 IT 实践一样，赛博靶场通常生活在自己的筒仓里。如今，网络范围成为更广泛的持续集成或持续交付测试周期的一部分，以解决应用和基础设施问题，这种情况非常罕见。事实上，电脑靶场远远不是一个连续过程的一部分，而是以高度人工的方式运作，这非常普遍，这意味着变化发生得缓慢而痛苦，相对于它们所代表的重大投资而言，生产率很低。

**如今网络靶场是如何运作的**

今天的网络范围市场由两大类解决方案解决:主要由中小型系统集成商执行的定制部署，以及由大型系统集成商提供的昂贵的“盒中网络范围”解决方案，这些解决方案通常将一系列功能打包在由操作员 GUI 驱动的单片管理应用程序下。后者的优势很明显，它使用户更容易操作，但这些“机箱内”系统是为一组离散、孤立的操作而构建的，它们通常依赖供应商路线图驱动的更新来扩展功能或更新对新基础架构或工具的支持。讽刺的是，这使得他们不那么敏捷。至于定制部署，它们本质上是非常手动的，仅设置测试场景就可能需要一周或更长时间，这与非云化 IT 基础架构相关的时间框架完全一致。Gartner 将 DevOps 描述为“敏捷方法和系统思维的结合”。在这两个方面，今天的赛博靶场都离 DevOps 很远。

**云自动化是开发友好型网络范围的关键**

cyber-ranges 进入 DevOps 滑流的路径主要是云自动化的问题。赛博靶场基本上是私人基础设施，因为你必须利用你操作的实际网络和数据中心设备来真正了解攻击将如何影响他们，症状会是什么样子。此外，网络安全问题确实需要在现实的网络拓扑中解决，这意味着我们正在谈论阶段/交付类型的测试环境。我们需要的是转变这些复杂的基础架构，并能够以私有云的方式快速建立、拆除和修改这些基础架构环境。转变为私有云的 cyber-range 可以支持 UI 和 API 驱动的程序化自助服务。云自动化不仅可以带来巨大的短期投资回报优势，如生产力和资源利用率的大幅提高，而且还开辟了将网络范围与更广泛的开发运维实践相结合的途径。

在 DevOps 中，就像一般的生活一样，没有人或组织(包括安全团队)是一座孤岛。通过转向私有云自动化模式，cyber-ranges 可以从一个(非常重要的)筒仓演变为与 DevOps 计划保持一致。
