# 没有足够的银子弹

> 原文：<https://devops.com/there-are-few-enough-silver-bullets/>

今天早上我浏览了本周的博客，它聚焦在一个狭窄的话题上。我举了一些例子，说明为什么太多的好东西是不好的，关于方法的绝对主义是如何伤害大多数组织的，以及如何克服这个问题以继续改进它对组织的作用。

当我发现有太多的领域受到绝对主义的困扰时，我就快完成了，所以我会提供一些一般症状和步骤来摆脱它们。

我说的“绝对主义”是什么意思？嗯，人们过去会说，“我们是 X 店”，这是事实，在一些领域仍然如此真心实意。如果你在上世纪 90 年代或 2000 年代是一家思科商店，那就是你所购买的全部。不允许竞争者进入；也没有考虑竞争对手。这适用于方法和产品类别。事实上，今天它比产品更适用于各种方法。

不管它来自哪里，它对组织都是有害的。虽然对于供应商来说，它简化了采购并提供了著名的“单一瓶颈”，但它也导致了成本的低效率(从您的角度来看，他们是垄断者，一些大型供应商因试图利用这一事实而臭名昭著)，并导致人们以最低效的方式做事，因为这是所选择的方法/解决方案所要求的。

有时，我们设法跳舞并避免“一种真实的方式”——例如，向云迁移并继续向云迁移。虽然云供应商在定价或优先级方面没有给客户提供太多的余地(我们[客户都是牛，而不是小猫](https://devops.com/cattle-not-kittens-revisited/))，但多重云仍然让我们有机会使用最便宜或最合适的云供应商，而没有太多的戏剧性。

很多时候，我们并不能有效地避免银弹的想法。以敏捷为例。如果这十年有一个真正的光环，我希望它是[容器](https://containerjournal.com)，但它实际上是敏捷的。大多数 IT 部门都极力维护敏捷是唯一正确的方法。早期阶段的新产品开发非常适合敏捷，我们需要它——即使只是为了那部分。后期的产品维护也非常适合敏捷，我们应该很高兴拥有它，在这种情况下，变化并不是系统复杂性的特征丰碑。介于两者之间？大型更新可能跨越数据中心，甚至跨越大洲，需要地理上分散的工作人员和数百(或更多)小时的开发才能完成一个子功能？不太适合敏捷，虽然我们中的许多人已经说了很多年，但我们希望事情会有所改变。正如我以前说过的，尽管瀑布被诽谤，但它做了几十年的工作，并且很好地处理了大型复杂项目。也许我们需要一个更符合时代的替代者；它融合了我们从敏捷中学到的东西。实际上，我很乐意。但是我们*不需要的是“我们是一个敏捷商店”它没有谈论产品时有用(至少如果你只选择一种产品，你会得到单一来源和简化采购。有了方法论，你甚至都不明白)。*

注意:敏捷是一个*的例子*，不是帖子的重点。在你的组织中避免“一个真实的方法”。找出它并检查它。有时候，拥有单一专业技能的单一解决方案是完美的答案——至少目前是这样。太多时候，“我们是 X 店”是某个人的偏好，被写进了公司政策。

要走过一条真正的路，你必须在那里向管理层展示它的弱点。当允许多种方法或产品时，会有一些技能重叠，但是如果管理层开始对此喋喋不休，只需指出组织当前支持的开发和/或 CI/CD 环境的数量。为了显示开销，跟踪花费的时间。多年来，员工短缺一直是该公司的通病。确保您知道在产品或流程的管理费用上花费了多少时间，并尝试将这些时间映射到在这段时间内*可以实现的目标上。它并不完美，但可以说明问题。别停下来。做你的工作，但每次都要指出，一个真正的方法会让工作变得更难或更昂贵。*

你正在*钉*它。不要让敏捷之类的东西阻止你为公司生产出优秀的应用程序。我所在的团队中，花费在过程开销上的时间拖累了交付质量的能力。反抗吧。坚持不懈地努力，使用最好的工具/流程来完成工作。
