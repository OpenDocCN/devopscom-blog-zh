# DevOps 的状态很好

> 原文:[https://devops.com/the-state-of-devops-is-just-fine/](https://devops.com/the-state-of-devops-is-just-fine/)

越来越多的人对债务状况感到绝望；[的采用](https://devops.com/?s=adoption)进展缓慢，或者大多数追随者没有意识到该方法的全部好处。我们能够而且必须做得更多、更好，等等。

这太愚蠢了。或者，我的第一反应:“扯淡！”

是的，我说一个主意很蠢。不，我没有说*你*傻。别给我理由。这个世界充满了愚蠢的想法，这是其中之一。原因如下。

DevOps 已经经历了前所未有的采用的十年——是的，前所未有。一旦 DevOps 流行起来，企业采用 PC 的速度并没有 devo PS 那样快。虚拟化席卷了整个商业世界，坦率地说，它的采用率远不及 DevOps。敏捷润滑了轮子，让人们思考更好的方法；我们*都*知道我们做事方式的弱点，DevOps 给了我们一个解决这些弱点的选择。我们拿走了。我们带着它跑了。

那么，当收养几乎无处不在的时候，谈论它是如何停滞不前的呢？傻。

那些担心的人进一步集中在 DevOps 的一个方面——没有足够的文化变革。让我们面对它——自动化更容易，投资回报更明显，即使投资回报模糊，从开发人员和运营人员的时间表中消除无用功也是巨大的。一些组织永远不会超越这一点；其他人最终会的，但并不着急。组织的变化并不比自动化更能定义开发运维——开发运维是简化软件创建和交付的过程——自动化和文化变化都有所贡献。

DevOps 的“全部好处”是…具体是什么？这是命题的另一个愚蠢的部分，即减缓 DevOps 的采用是负面的。自动化改进了软件的创建和交付，竖井的消除改进了软件的创建和交付 DevOps 的全部好处是改进。这一直是既定的目标——让它更好、更流畅、更可重复。无论一个组织在哪里进行实现，几乎可以保证他们的创建和交付是“更好、更流畅、更可重复的”因此，利益正在累积。

每个组织都有不同的需求。DevOps 是一种方法论。它从来不是所有 It 问题的完美解决方案——尽管仍有专家试图告诉你它是。自动化永远是积极的；敏捷和 devo PS——嗯，这不太正确。你知道你的组织和你的应用，所以“正确的工作方法”绝对是你的电话，而不是一些供应商或专家的。因此，“采用放缓”部分代表组织/项目/应用选择不追求 DevOps 暗示的一些更激进的改变(是的，激进。自动化只是脚本的逻辑延伸；相比之下，重新设计团队组织和报告关系绝对是激进的)。

这是一个停下来的好地方。您的应用、您的组织、您的环境。你知道什么是最好的。所以继续做下去。如果组织还不存在，那就去推动它，成为需要的改变。但是要注意成本/收益。有时候，被要求的改变并不值得。所以，在你逼得太紧之前，确保他们是对的，如果他们不是，确保你对为什么这样做感到舒服，并能为这个决定辩护。

继续摇摆。无论您处于开发运维的哪个阶段，这些应用都将交付给用户。这说明了你和你的团队。感谢你的用户，他们从来没有想到要说这句话。
