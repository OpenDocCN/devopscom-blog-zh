# 盘点 DevOps:何去何从？

> 原文：<https://devops.com/taking-stock-devops/>

你有没有注意到每一种技术趋势是如何在主流采用的道路上经历一系列阶段的？首先是怀疑和否认，然后是默认，最后是承认它确实工作得很好。DevOps 运动也不例外。

首先是关于 DevOps 实际上是什么以及它在企业环境中的有用性的问题(无论企业现在实际上意味着什么)，然后是一些成功的故事，最后是有形的和可量化的好处的发布。

谢天谢地，我们已经从“它是一只鸟还是一架飞机？”舞台。当然，这并不能阻止大肆宣传，但至少益处的公布——如最近在全球研究中说明的那些，“组装 DevOps 拼图”——允许我们穿过所有“DevOps 解决全球变暖”类型的废话。

因此，随着价值评审团最终进入，我们应该问的关于 DevOps 的下一个问题是，从这里到哪里？有趣的是，DevOps 的旧维基百科条目提供了一些指导，指出 DevOps 是:

… *强调软件开发人员*[](https://en.wikipedia.org/wiki/Software_developer)***和其他* [*信息技术*](https://en.wikipedia.org/wiki/Information_technology)**【IT】(IT)专业人员在自动化软件交付和基础设施变更过程中的协作和沟通的文化、运动或实践。****

**这很好，因为它强调了跨所有 it 职能部门(不仅仅是开发和运营部门)协作的重要性。然而，这仍然不太适合我。尽管在沟通开发和运营方面已经做了很多文化和自动化方面的工作，但还需要做更多的工作来理解将其他 IT 职能与开发运维结合起来实际上如何有利于职能本身和业务。同样重要的是，调整其他业务职能，如人力资源和财务，可以带来更多的成功。**

## **DevOps 之所以有效，是因为它对业务有益**

**业务领导人谈论的术语包括更快的交付周期、客户转化和价值实现时间，这(将文化讨论放在一边)确实是 DevOps 的本质。因为采用 DevOps 与赚钱直接相关，所以它很有可能获得企业的关注和投资。**

**也就是说，信息安全和其他专业人员最好把他们的马车拴在 DevOps 上。这一运动的美妙之处在于，团队协作并在开发生命周期的更早阶段提供输入，然后过程自动化以确保更快的发布时间和质量。通过更早地涉及安全性，企业可以切实加强安全性，用自动化方法平衡所有这些“妨碍您的审核”和清单，确保合规性配置导致更快的部署。**

**一个有趣但重要的好处是，这种方法如何帮助 it 安全性与业务成果保持一致。安全性可以报告业务目标环境中的功能价值，例如自动化合规性如何等同于更快的发布，从而增加收入，而不是通过一系列技术探测措施来证明其价值，例如渗透测试分数、修补的漏洞或检测到的安全事件。**

**当然，DevOps 不仅仅涉及开发、运营和安全，它还将涉及提供有价值的输入以帮助提高速度和质量的其他功能。你可以把它想象成一个古老的“石头汤”故事，但没有诡计——每个人，无论是软件设计师、产品经理还是企业架构师，都在软件生命周期的早期提供建议和专业知识，因此弹性、可支持性和规模等基本元素从一开始就被融入其中。**

## **DevOps 不应该在其中开始和结束**

**DevOps 方法建议我们不应该仅仅局限于 IT 职能部门——从那里开始，是的，但是业务的其他方面也应该参与进来。这听起来像是常识，但实际上更重要，因为商业和数字之间的界限已经变得非常模糊。认识不到这一点，我们可能会以良好意愿的筒仓而告终:文化上正确，但缺乏牙齿。**

**这解释了为什么即使是备受指责的人力资源部门也要扮演 DevOps 的角色。长期以来，许多人力资源部门基本上都是后台办公，通过定期的员工调查和绩效评估来涉足组织文化的浑水。但这还不够。现在，人力资源领导必须走上软件工厂，与其他 DevOps 从业者携手合作，找出所有抑制业务增长和人才发展的组织和心理障碍。**

**金融也必须与债务挂钩。这包括超越传统的 IT 成本中心思想，开发新的流动资金模型，以刺激敏捷团队的软件创新。这意味着业务变革代理必须与 DevOps 从业者联手，寻找更好的方法来资本化或消耗软件开发成本，或者以最佳方式将新应用、API 和信息资产货币化。同样，这在软件生命周期中发生得越早越好。**

**从 IT 的角度来看，我们在相对较短的时间内通过 DevOps 取得了长足的进步。但这只是旅程的开始。大多数企业甚至没有考虑过使用 DevOps 原则作为变革代理来改造整个组织。那些最有可能获得先发优势的公司——在越来越受应用软件影响的经济中，这是一个很好的位置。**
