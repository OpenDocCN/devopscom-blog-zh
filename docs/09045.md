# 持续安全:CI/CD 的下一次发展

> 原文:[https://devo PS . com/continuous-security-the-next-evolution-of-ci-CD/](https://devops.com/continuous-security-the-next-evolution-of-ci-cd/)

在一波高调的网络攻击和白宫发布关于改善国家网络安全的行政命令之后，如何建立一个成功的网络安全计划从未像现在这样受到如此热烈的讨论。从网络安全管理软件产品到 Colonial Pipeline 攻击，这些事件表明，单靠防御措施无法降低网络安全风险，任何网络安全计划都必须从十个漏洞中的九个开始——代码——开始。

近年来采用 CI/CD 作为 Agile 和 DevOps 的扩展，使组织能够接受数字化转型，并确保代码始终能够满足当前的业务需求。不幸的是，这进一步孤立了安全性，因为流程跟不上步伐。虽然有许多单点解决方案可以解决不断扩大的应用程序安全领域(从传统的 pentesting 和漏洞扫描器到监控 DevOps 工具链(如容器)的各个组件),但太多的工具会导致 DevSecOps 成为“一团乱麻”,使组织淹没在数据中，却没有可操作的信息。

## 超越安全开发运维

随着组织不断发展他们的交付流程，安全已经在安全开发运维(SecDevOps)计划中找到了自己的角色，以确保在软件开发生命周期的每一步执行安全流程(SLDC)。通过持续管理现有管道中的安全实践、政策和[债务](https://devops.com/?s=technical+debt)，该方法可确保所有利益相关方——从最高管理层到开发人员和 AppSec 经理——在开发的每一步都拥有所需的信息，以分担交付安全软件的责任。

然而，这些计划虽然是一个很好的起点，但在 pdf、Excel 文档和电子邮件中留下了安全要求和检查，未能像开发人员和运营人员所要求的那样，将安全集成到 SDLC 中。如果没有 SecDevOps 编排，信息通常会滞后于管道，从而产生新的瓶颈和不确定性。SecDevOps 编排让安全和开发团队确信他们遵循了正确的策略，并消除了对 SDLC 造成拖累的手动工作。随着应用程序安全流程的自动化，企业获得了对其计划的一致性和洞察力，从而可以专注于交付代码，同时知道代码符合其组织安全标准。

## 增加持续安全性

借助 CI/CD，组织能够快速满足其快速发展的最新业务需求，但如果没有持续的安全方法，安全性又会被孤立起来。组织必须能够根据内部变化(不断变化的数据库、不同的软件版本等)调整其安全要求。)和外部因素(合规性要求、不断变化的威胁形势等)。我们需要在 CI/CD 的基础上更进一步，还需要持续集成安全性。

推而广之，持续安全性(CI/CD/CS)是指持续交付满足最新业务安全标准的软件的理念，考虑了整个 SDLC 中的内部和外部变化。

为了有效地采用 CI/CD/CS，组织必须关注三个关键原则:

*   **没有完美的代码，因此也就没有完美的安全**—良好的应用程序安全可根据整体业务战略管理网络风险。当你有能力在企业的风险承受范围内接受风险时，你就知道什么时候停止，什么时候继续，因为其他控制措施已经到位。不要让完美成为你的敌人。
*   **一口大小的应用程序安全性**—开发运维不是快速移动和破坏东西，而是高效移动和在正确的时间修复东西。在正确的时间将正确的应用程序安全信息交付给负责执行它的相关人员。这确保了更好的协作，因此利益相关者知道在哪里、何时以及与谁一起引导他们的注意力。
*   **微反馈循环—** 利用流程编排，在政策合规性造成瓶颈之前获得实时反馈。这使您能够尽可能快地对内部和外部的变化做出反应，为未来的决策创建一个信息的迭代循环。

通过持续管理现有 CI/CD 管道中的安全实践、政策和债务，这种方法允许在任何 SDLC 中部署可扩展的应用安全计划，因此它可以持续发展，始终与当今的业务保持同步。

不同的组织需要考虑不同的风险，这意味着安全性必须与业务战略和优先级保持一致，以便动态响应。借助 SDLC 中的端到端集成，持续安全性支持 CI/CD 来提高工作效率和加快上市时间，同时降低可能影响特定业务的风险。软件本质上是短暂的，组织需要能够不断平衡安全性、技术和业务优先级，以确保他们保持对最重要的事情的关注:为客户和股东提供价值。
