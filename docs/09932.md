# 使用 DevSecOps 实现联邦网络安全标准化

> 原文：<https://devops.com/standardizing-federal-cybersecurity-with-devsecops/>

DevSecOps 在联邦政府里有一段时间。随着 [拜登总统关于改善国家网络安全的行政命令](https://www.whitehouse.gov/briefing-room/presidential-actions/2021/05/12/executive-order-on-improving-the-nations-cybersecurity/) 和联邦机构发布基于持久安全框架 的 [DevSecOps 最佳实践，政府组织正在支撑和标准化其 DevSecOps 流程，以正面应对当今的网络安全挑战。](https://devops.com/federal-agencies-share-devsecops-guidelines/)

将安全性集成到开发过程中对于联邦机构来说尤其重要，因为他们已经投入了大量的时间、精力和金钱来创建和运行软件开发工厂。这些机构正在积极尝试平衡安全和创新——许多机构取得了成功。美国空军的 [凯塞尔运行](https://kesselrun.af.mil/) 计划是紧密运行和创新软件工厂的一个例子。

问题是:随着标准化网络安全的呼声越来越高，他们如何继续这一成功？

### 安全和创新之间的微妙平衡

很难在安全性和创新性之间找到平衡，也很难在一个标准化的流程中实现这两者。这当然不是不可能的，凯塞尔运行证明了这一点。但是它确实需要高水平的严格性，以确保应用程序被快速开发，同时花费必要的时间来减轻和最小化安全风险和漏洞。

像凯塞尔·朗这样的项目工作得很好，部分原因是他们从一开始就将安全性融入了开发过程。DevSecOps 的核心原则允许团队在整个开发周期中集成和标准化安全性，而不会失去动力。

但是，这些标准化工作面临着一些挑战，阻碍了 DevSecOps 的实践。

首先，许多政府机构仍然在使用遗留系统和工具集，而不是许多使用 DevSecOps 的商业公司可以获得的现代技术。这些技术可以包括从协作平台到消息应用程序，再到将所有工作联系在一起的底层平台。

第二，尽管政府致力于创新，但旧的思维方式在公共部门的许多领域仍然存在。一些重要的人仍然认为，“电子邮件这么多年来一直运行良好，为什么要改变呢？”那些人没有立即看到他们的工具集和实践现代化的价值。

但是，过时的技术和流程不会帮助政府机构平衡安全和创新——它们也不会与新的网络安全政策和命令一起工作。例如，拜登总统的 EO 充满了确保软件开发实践和软件供应链可见性的暗示。要做到这两点，需要开发人员、安全经理和运营经理保持一致，并在开发周期的各个阶段保持沟通。

### 在整个 DevSecOps 过程中保持态势感知

然而，即使在最成熟的 DevSecOps 环境中，这也不总是可能的。因为虽然 DevSecOps 是由协作驱动的，但这并不意味着开发人员、安全经理和运营经理总是知道谁在做什么，或者特定项目的状态如何。在旧的流程、技术和孤立的团队仍然存在的政府环境中，这一点尤其如此。

但是维护安全性和创新性之间的平衡高度依赖于在整个开发生命周期中能够不受约束地了解情况。例如，开发人员需要知道安全操作团队是否正在修复开发人员在测试中发现的漏洞。同样，运营经理需要知道补丁的状态，因为这可能会影响开发过程中的其他重要因素，包括部署时间。

高级仪表板有助于所有团队成员快速确定开发中项目的状态，并有助于个人了解项目在管道中的位置。他们可以提供一个快速、一目了然的视图，领导可以用它来规划他们的项目，发现积压或潜在的部署障碍，等等。

但是即使是仪表板也不一定足够，因为涉众需要能够快速和安全地相互交流状态更新。

### 收集信息，促进协作

问题是，并非所有这些人都必须使用相同的工具。有些人可能使用一种信息服务。其他人可能正在举行即兴视频会议。其他人可能会向项目管理委员会发布更新。软件开发是一个活的、敏捷的过程，沟通通过不同的渠道无时无刻不在发生。

这就是为什么对于机构来说，将所有这些不同的线索整合在一起是至关重要的，特别是如果他们要正确地管理安全性并保持对 EO 的遵从。关于软件漏洞的一次沟通失误可能会成为明天的突然袭击。必须有一种方法让所有 DevSecOps 涉众相互交流，不管他们选择什么工具或者使用什么渠道。

这就是现代工具集和平台选择发挥作用的地方。将不同的线程整合在一起以促进更好的通信和协作，需要能够连接各种服务、自动化任务管理和警报以及打破任何挥之不去的通信孤岛的技术。它需要一个连接所有关键利益相关者的基础架构，以便他们能够协作并确保他们正在工作的应用程序的安全性。

### 迎接网络安全时刻

旧的技术和实践无法满足政府机构目前面临的需求。它们不适合今天的软件开发过程，也无助于寻求在今天的网络威胁中保持领先。

为了迎接这一时刻，政府机构必须采用新的沟通和协作方式。他们必须使用可用的工具来保持创新，即使是在根据网络安全法规进行标准化的同时。他们必须为他们的 DevSecOps 团队提供自由而安全地工作的能力，并为他们的成功做好准备。
