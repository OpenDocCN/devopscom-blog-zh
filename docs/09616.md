# 什么是 DevSecOps？

> 原文:[https://devops.com/what-is-devsecops/](https://devops.com/what-is-devsecops/)

虽然传统的开发实践长期以来一直将安全性和合规性分开，但作为一系列最佳实践，DevSecOps 将安全性集成到了 DevOps 软件开发生命周期的每个阶段。DevSecOps 在软件开发生命周期的早期阶段引入并自动化了安全性，而不是在最后附加上安全性。这种方法节省了资金，节省了繁琐的手动任务的时间，帮助组织满足法规遵从性要求，并显著降低了在应用程序最终构建后被发现的严重安全漏洞的风险。例如，DevSecOps 提倡在敏捷开发过程中使用主动渗透测试、安全审计和其他安全工具。

DevSecOps 的目标是在开发人员和安全专业人员之间创建一个协作环境，使组织能够更快、更容易地构建安全代码。通过从过程的最开始就强调安全性，它成为应用程序开发过程中的优先事项，而不是事后的想法。随着时间的推移，开发人员越来越熟悉软件中的常见弱点，这些弱点导致今天部署的应用程序比大多数人真正想承认的更不安全。最终目标是让开发人员、安全专家和运营专业人员在应用程序开发的每个阶段都能更轻松地协作，从而创建安全的应用程序。

## 最小化应用程序漏洞

将安全性集成到开发流程中可能看起来单调乏味且成本高昂，但对于组织来说，采用 DevSecOps 来最大限度地减少网络罪犯可能利用的应用程序漏洞的数量变得至关重要。在整个开发过程中应用时，扫描和报告可以在几分钟内完成，而不是在过程结束时暂停部署来检查整个应用程序。

将安全性留到应用程序开发过程的最后，或者更糟的是，留到应用程序部署之后，只会增加总拥有成本。与在应用程序开发过程中解决问题相比，在应用程序部署后修复漏洞的成本要高得多。

实施 DevSecOps 要求组织建立通过持续集成/持续交付(CI/CD)平台管理的工作流。最重要的是，开发人员必须了解安全性在使组织能够在应用程序开发生命周期中尽早识别漏洞方面所扮演的角色。

例如，在开发过程中插入安全审计和渗透测试有助于确保应用程序的安全性。当项目发生变化时，可以在开发流程中触发扫描、架构审查和渗透测试，以节省时间和资源。

## DevSecOps 采用

然而，从工具集成到开发人员和安全团队之间缺乏信任，有许多技术和文化挑战会阻碍 DevSecOps 的采用。安全专业人员的任务是识别和防止应用程序中的漏洞。验收测试标准、用户设计和威胁模型应由安全专业人员创建。然后，开发团队需要定义一个代码审查系统，以确保一致性。

尽管有这些顾虑，但最重要的是开始行动。理想情况下，开发和安全团队应该合作创建一个安全的应用程序开发和软件开发环境。没有必要创建单独的团队或应用不同的策略。

DevSecOps 最大的好处是它消除了开发和操作之间的孤岛。因此，软件工程师从开发过程的开始就集成了网络安全流程。这包括确保每个组件、配置项目和安装过程都得到安全修补和记录。这种将安全转移到左边的概念允许安全团队在早期识别和补救安全威胁。

然而，从长远来看，在一系列备受瞩目的网络安全违规事件发生后，采用 DevSecOps 最佳实践将确保安全仍然是重中之重。简而言之，现在人们更加关注 DevSecOps，目标是保护软件供应链。
