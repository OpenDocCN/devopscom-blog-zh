# JFrog 旨在将 DevOps 推向边缘

> 原文:[https://devops.com/jfrog-aims-to-push-devops-to-the-edge/](https://devops.com/jfrog-aims-to-push-devops-to-the-edge/)

在其 [swampUP](https://swampup.jfrog.com/) 活动上，JFrog 推出了一个 [JFrog Connect](https://jfrog.com/press/jfrog-unveils-devsecops-for-iot-bringing-trusted-software-to-the-edge/) 平台，用于更新、管理、监控和保护远程 Linux 和物联网(IoT)设备。

JFrog Connect 平台基于一个轻量级代理 JFrog，它是在去年收购 Upswift 时获得的，现在已经与 JFrog Artifactory 存储库集成在一起。DevOps 团队现在可以使用[低代码](https://devops.com/?s=low-code)工具来构建流逻辑，该逻辑除了触发回滚以确保设备在必要时返回到其先前已知的软件状态之外，还与 Git 存储库集成。

DevOps 团队还将能够从世界任何地方连接到各种基于 Linux 的物联网设备，然后通过名称、描述或标签实时管理和监控它们。

JFrog 首席技术官 Yoav Landman 表示，在不久的将来，DevOps 团队将发现自己正在构建和部署应用程序，这些应用程序不仅会部署在网络边缘的终端设备上，而且会不断更新。

他说，历史上，这些端点主要由运营技术(OT)团队管理，但很快将由 DevOps 团队使用 Upswift 创建的轻量级代理来自动管理这些平台，直至单个容器。

随着组织寻求在更接近数据创建和消费点的地方处理和分析数据，部署在边缘计算平台上的应用程序工作负载数量正在急剧增加。事实上，这一转变是数字业务转型计划的核心，这些计划依赖于边缘计算平台和设备运行，例如近实时处理数据的物联网应用。

今天的挑战是，在许多情况下，组织缺乏在[边缘计算](https://devops.com/?s=edge+computing)平台上一致部署软件的能力，这导致了在边缘计算平台上手动部署软件的自制工具的大杂烩。兰德曼说，显然需要提高开发运维标准，以满足边缘计算需求。

现在说有多少软件将被部署在边缘还为时过早，但也许有一天边缘的软件会比云中的多。与此同时，边缘计算和云计算之间的界限继续模糊。越来越多的 DevOps 团队很快会发现自己要从一个中央云中管理数千个分布式端点。

与此同时，DevOps 团队应该期望发现他们与通常具有显著不同文化的 OT 团队更紧密地合作。这些 OT 团队中的许多人现在才开始接受将端点连接到互联网的含义。在这些平台上持续更新软件的需求通常不是他们首先想到的问题。如今，运行在边缘计算平台上的许多软件很少更新。然而，在未来，软件可能会以许多 OT 团队还没有准备好的速度更新，更不用说管理了。
