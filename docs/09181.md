随着对各种事物的供应链攻击的增加，从日志软件如 [Log4j](https://devops.com/?s=Log4j) 到重要 JavaScript 包的接管，再到网络实用工具如网络安全管理软件产品的入侵，越来越多的组织认识到采取零信任心态的必要性。零信任可以提高安全性、降低风险，并让组织对其 IT 基础架构和应用程序的完整性更有信心。为了正确地建立一个零信任架构，您需要理解使系统零信任实际上需要什么。

## 零信任的定义和指导原则

由于对零信任的兴趣增加，许多误解已经形成，需要纠正。

*   零信任不是一种产品或工具。 事实上，没有一种工具可以应用于一个环境来配置全栈零信任实现。相反，零信任更多的是关于创建和提炼一个架构，并应用一致的心态。
*   零信任是一个永无止境的旅程。 它不是目的地。你永远不会结束；因为软件应用程序和 IT 环境在不断发展，所以从未完成过零信任的设置。随着您的环境的发展，应用零信任的一部分是定期回去重新评估您正在做的一切，以确保它符合零信任的原则，并涵盖您所有的软件和技术资产。
*   零信任并不意味着您的环境不会被黑客攻击或破坏。 相反，在零信任状态下，你会不断假设你的环境已经被攻破。零信任强调构建防御层，减少任何黑客攻击或违规的潜在影响，保护您的资产和数据。

那么，什么是零信任呢？零信任是一种安全框架和方法，它假设组织的 IT 资产(软件、网络、数据)始终处于风险之中，必须通过持续的身份验证以及对 IT 环境中运行的任何身份(人或机器)应用最小特权原则来保护这些资产。零信任的核心是缺乏对任何身份的内在信任，作为构建更具弹性的应用程序和分段风险的管理方法。

零信任有几个关键的指导原则。

1.  **使用一种强大的身份认证形式**——而不仅仅是用户名和密码，随时验证所有身份。具体来说，使用强形式的身份验证，例如多因素身份验证。这一原则不仅适用于人类用户，也适用于访问您环境的所有设备、机器人、服务、API 和合作伙伴系统。当它们在您的环境中处于活动状态时，必须在每次试图访问时对它们进行验证。
2.  总是假设你的环境已经被破坏了。 从这一点出发，设计您的系统以减少爆炸半径并保护您的资产，就好像攻击者持续存在并在您的环境中移动以执行恶意行为—数据泄露、系统接管或其他形式的滥用。
3.  始终对在您的环境中运行的所有人和所有事物实施最低特权。 最小权限是指只限制访问每个任务或交互所需的系统、应用、数据和服务。要正确应用最小特权，需要了解应用程序和服务的预期行为，它们应该做什么和不应该做什么，还需要创建一个上下文概念。对上下文应用最低特权还取决于对任何设备、服务、应用程序、API 或寻求访问的身份的安全状态的评估。您的零信任策略执行引擎应该始终询问入站和正在进行的连接，以了解身份来自何处、它试图连接到什么以及该身份的当前安全状态。这些问题的答案应该告知和形成最低特权权限，并提供对提供访问的风险的额外洞察。 T6】
4.  **对您的环境进行分段，以限制水平遍历或穿透攻击。** 这与最少特权密切相关。底线:不要授予任何身份全权访问您的网络，应用程序数据或基础设施。仅允许身份访问他们执行任务真正需要的部分，并且仅在他们执行任务时。

## 使用零信任保护软件和软件供应链

零信任的概念语言让它听起来像是一种身份架构和网络安全方法，漂浮在开发者之上。为了构建和部署安全的应用程序，开发人员和 DevOps 团队还必须采取零信任的心态。事实上，零信任和应用程序安全之间有明显的协同作用——特别是在开源软件(OSS)方面。OSS 开发人员可能不会花足够的时间和精力来保护他们的代码，因为这不是他们喜欢写软件的地方。由于这个原因，如果你在你的应用程序和环境中包含了 OSS 代码(和嵌套的 OSS 库),那么你就更有理由应用零信任框架。

考虑这一点的一个有用方法是将零信任视为从内到外、从外到内以及链接内部和外部、应用程序和 API、代码和集成/部署管道以及合作伙伴环境之间的一切都适用。如果执行得当，零信任可以让你的开发者更快也更安全。此外，从机密性、完整性和可用性(CIA)的角度来看待零信任也很有用。您应该设计您的零信任体系结构，以映射到您的业务需求的优先级层次结构来交付这些元素。有几种不同的方法可以在软件供应链中实现零信任原则。

### 假设您的代码已经被破坏并且不安全

进行软件资产盘点，查看并分类您环境中运行的所有代码。这使您更容易识别未经授权的更改，并构建更具弹性的应用程序。建立机制来检测未经授权的更改或异常的软件行为。您可能希望将这些指导原则应用到亲密的合作伙伴及其软件和供应链中。请记住，攻击可能来自你，通过你，或针对你。

### 小心管理身份

对用户权利进行分析和分类，以设定最低权限的基准。对您环境中的所有活动身份进行清点，以识别已知身份(并快速识别未知身份)。建立分析身份行为的机制，以发现异常，并为上下文和行为权限以及访问策略决策创建基线。在此清单中包括合作伙伴身份:您最大的风险可能来自攻击者用来访问您的环境的不安全的合作伙伴。清单还应包括服务和管理帐户。倾向于设置他们的特权，然后忘记它们，通常授予广泛的访问权限，以免减慢他们的速度，并且还可能为攻击者提供不受监控的后门。

### 清点库存并锁定集成

盘点你的外部和内部供应链。这包括代码中的所有依赖项、调用的服务、公开或访问的 API 以及用于开发、托管、集成和构建应用程序的工具。例如，您可能希望通过分段和最低权限来分析您的 CI/CD 管道和部署环境，以根据管道阶段来限制访问。采取措施分割软件环境并防止横向移动，如应用程序沙箱(但也要确保严密保护您的沙箱)。

### 代码评估

开发人员需要将威胁模型应用到他们的代码中，并采取这样的心态:考虑攻击者可能访问什么，如果他们危及应用程序或周围的供应链，他们可能会做什么。显然，对于所有代码，尤其是 OSS 代码，开发人员需要不断寻找漏洞，并使用静态代码分析和动态应用程序安全测试等工具扫描代码。零信任意味着持续验证你自己的代码，而不仅仅是使用代码的实体。

## 零信任外卖

随着我们继续走向更加复杂和集成的应用环境的未来，以及核心应用功能更多地使用 OSS 代码，零信任的未来将永远变成零信任。它将成为应用程序开发中的一种生活方式。也就是说，我们应该牢记四个关键的指导原则。

**了解您的当前状态**—了解您的环境和安全状况，以及安全漏洞的直接和间接影响。

**假设违规**—能够识别异常行为，并持续考虑如果您的应用被恶意使用，攻击者会做什么？

**评估**—持续评估环境的状态和变化。问这样的问题:谁拥有什么，何时何地？在安全性方面，您的集成合作伙伴和供应商处于什么位置？

**持续改进**—定期评估和比较您的零信任工作与期望的安全状态，并根据机密性、完整性和可用性目标进行改进。不断防范风险。

这四个原则对应着五个关键要点，任何开发人员、应用安全团队、开发安全团队或平台团队现在和将来都可以应用。

*   用零信任的眼光审视你的环境
*   连续重新认证所有身份
*   建立基线应用和身份行为
*   专注于检测和防范攻击
*   弹性设计应用

遵循这些指导原则和基本框架将确保您的零信任之旅创建强大的防御，帮助从外到内、从内到外保护您的开发供应链，并与整合的合作伙伴合作。
