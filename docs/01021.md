# 不要让我编写播客:一个关于开发者体验的新的 Heavybit 系列

> 原文：<https://devops.com/dont-make-code-new-heavybit-series-developer-experience/>

欢迎来到“不要让我写代码”,这是一个新的播客，讨论了开发者体验设计以及你在开发面向开发者的产品时所面临的独特挑战。《不要让我写代码》由 [Opsee](https://opsee.com/) 的联合创始人兼首席采购官[史蒂夫·博克](https://twitter.com/sboak)和 [Convox](https://convox.com/) 的联合创始人兼首席执行官[大卫·多勒](https://heavybit-blog.squarespace.com/config#)主持。

这个播客是由 heavy bit(T1)带给你的，这是一个帮助以开发者为中心的公司将他们的产品推向市场的项目。

在第一集中，Steve 和 David 讨论了术语“DX”(开发人员体验)的起源，他们分享了关于如何打造出色的开发人员体验的想法，并讨论了他们在设计面向开发人员的产品时面临的一些挑战。

大卫:大家好，我是[大卫·多乐](https://twitter.com/ddollar)。我是[康沃克斯、](https://convox.com/)一家重量级公司[的首席执行官和联合创始人。我在过去的三四年里一直从事开发经验方面的工作。](http://Heavybit.com)

我在 Heroku 成立了 DX 团队，开发了包括 Foreman 在内的一系列开源工具，现在正在开发一个开源平台服务。

**史蒂夫:**我是[史蒂夫·博克](https://twitter.com/sboak?lang=en),[Opsee](https://opsee.com/)的联合创始人兼产品负责人。我们是一家监控公司，在过去的六年里，我一直在几家监控公司工作，帮助开发人员改善他们产品的体验，并发现他们代码中的错误，提高他们产品的质量。

虽然我直到几个月前才第一次听到 DX 这个词，但这是我几年来的一个热情所在，很高兴终于给它起了个名字。我对这第一集很兴奋，因为我们可以稍微解开这个术语，弄清楚它的意思。

大卫，我认为你的背景中最有趣的一部分是，你是 Heroku 创始团队的一员，这是创建 DX 的一部分，我有兴趣听到更多关于这方面的信息。

大卫:是的，我想这可能是 2012 年末，也可能是 2013 年初，就在 Heroku 我们开始扩张的时候，从一个 Ruby 主机变成了一个任何语言的主机。这意味着我们也开始看到更多，你知道我们的标准用户在那之前有点像，我是十个构建 rails 的用户，非常，有一种快乐的方式。

当我们扩展语言的时候，我们有了 Windows 用户和 Linux 用户，每个人都聚在一起尝试构建任何东西。这很快就失控了。所以，真的，我们组建了 DX 团队，把 Heroku 负责各种语言的人召集在一起。有人负责 Ruby，有人负责 Python 和 Node。将我们所有人聚集到一个团队中，并真正决定尝试在所有这些语言之间创造一致的体验，有点像 Heroku 的方式…

史蒂夫:它从一开始就被称为开发者体验团队吗？

大卫:是的。

所以我第一次听到这个术语是来自亚当·威金斯。他好像过来对我说，“嘿，我想我们应该建立这个 DX 团队。你怎么看？”是的，所以我的意思是它准确地描述了我们正在努力做的事情，那就是为人们在 Heroku 上建造东西提供良好的体验。

史蒂夫:那是什么时候？那是哪一年？

大卫:我想那是 2012 年年底，但现在这个时间点有点模糊。不过，是的，是有一段时间了。

史蒂夫:这对 Heroku 来说似乎仍然是一件重要的事情。我只是在研究它的术语，看到有一个 Heroku.com/dx,，这是他们如何营销自己的一个重要部分。

很明显这对 Heroku 来说很重要。就像我说的，现在用一个名字来谈论这个问题很有趣，因为我们很多人都在做这件事，而且已经做了一段时间了，但是，给它一个名字会让它更有分量，这也标志着它对我们所有人有多重要。嗯嗯。

大卫:这是兴趣……你会想到 DX，那到底是什么意思？你现在说的是某种类型的用户，对吗？当你谈论为这种用户设计一种体验时，这种用户喜欢打开引擎盖，打破常规，以自己的方式做事情，并且知道发生了什么，也许你甚至可以考虑超级用户。

是的，你要确保他们可以做他们想做的任何事情，但选项不是压倒性的，也不是完全的，只是完全的混乱。

**Steve:** 是的，这就像是一种合作关系，我们作为工具的制造者，如果开发人员不喜欢他们正在使用的产品，这种合作关系可能会在他们最终试图构建的产品中表现出来，那么他们就用这些工具为他们的客户构建工具。因此，正是这种合作关系，让我们一起努力写出好的代码，创造出优秀的产品。

我认为这是它不同于整体用户体验设计的部分原因，也是为什么它本身作为一个话题很有趣。

大卫:我绝对同意。

> 我想，在我发现的开发经验中，工作最有回报的事情是当人们做了你从未期望他们做的事情。

你把所有的部分都放好，你想，“哦，也许有人会用这些部分来建造 X，”结果是有人来了，建造了一些你从未想过的东西。我认为看到这些真的很有意义。

**Steve:** 有意思的是，我认为很多产品开发人员可能会分享这种感觉，他们会发现人们使用产品的意想不到的方式，以及他们创造的黑客技术，我听说，即使是消费产品，人们也会对用户以他们从未预料到的方式使用产品感到惊讶，但对于开发人员来说，这真的是另一个层面，因为他们可能会试图破坏你的 API。

他们更擅长于探索代码的深度，然后用它创造出你从未想到过的工具，这对我们来说既令人惊讶又美妙。

大卫:这在很大程度上还是关于审美，对吗？比如，你是一个好的用户体验，一个好的开发者体验。

说到底这个东西好用吗？用起来舒服吗？你能弄清楚发生了什么吗？在一天结束的时候用这个东西会让我开心吗？所以，是的，当有人试图进来并在你的产品上实际构建一些东西时，它需要一套不同的东西。

史蒂夫:我记得在谷歌地图应用编程接口的早期，有多少实验和应用程序是用它编写的。

我认为，部分原因是因为它们让开发者变得非常容易和容易理解。我听到该团队中有人谈到，我们都知道并喜爱的产品并不是它的第一次迭代，他们实际上有一个更难使用的初始版本。你知道更多关于那的历史吗？

大卫:我真的不知道。

史蒂夫:是的，但那是我第一次接触使用另一家公司的 API，是一次非常有益的经历。他们有所有这些样板代码，你可以很容易地放下来，在地图上绘制图钉，并看到这一点。

> 尤其是对作为设计师的我来说，获得令人满意的成果是非常值得的。屏幕上的实际地图，上面有我布置好的图钉。我认为这是一个非常好的方法来衡量你的开发者体验设计有多有效，有多优雅，开发者自己能多快做出东西。

大卫:我用来衡量我正在建造的东西的一个晴雨表是你寻找人们围绕你的工具写的博客帖子。

他们是不是在写一篇博文，说这东西太疯狂了，我都不知道怎么用它？就像我的博文教你如何使用它，因为我必须弄清楚这一切。还是他们在谈论这让他们的生活变得多么轻松？看这有多棒，你也应该试试。

史蒂夫:是的，这让我想到了今天的 JavaScript 世界，以及有多少框架正在推出，以及关于这个或那个的争论。从某些方面来说，这对整个开发社区来说确实是件好事，因为现在有这么多的选择，有这么多好的框架可供选择。实际上，太多了。

我认为我们已经开始在 JavaScript 世界中看到一些了。只是关于应该如何做事的不同观点。这几乎引发了对开发者体验设计价值的反驳，即当我们打开太多的可能性而不创造一套工具时，我想你在你的长篇大论中描述过它

大卫:我喜欢制作锋利的工具。我的意思是这有点像 Unix 哲学，对吗？比如说，你想做一些能很好地做一件事的小工具。你通常希望有一些好的例子或默认，比如，这是如何结合使用这些工具来制作有趣的东西。但是让这些工具变得有些独立和可组合，你就会开始得到意想不到的有趣的东西。

史蒂夫:

> 是的，每个部分都做得很好，但是当它们组合在一起时，它们可以形成更好的东西，整体大于部分的总和。

大卫:对。就像，一个很好的例子，这不是真正的开发者体验，但我有 TTT，它有点像所有这些小组件的无限变化，你可以把它们放在一起，但它们也有一些默认的管道，你说，就像，“好吧，我真的只想让我的电子邮件去谷歌驱动器或类似的东西，就像这些漂亮的默认设置。”

我认为，这真的很重要，当你有所有这些选择的时候，也有通过它们的指导。

史蒂夫:是的。我们现在大量使用亚马逊 API，它们本身就是一个生态系统。对于亚马逊在 AWS 中提供的每个产品，都有一个独特的 API，其中许多 API 似乎有不同的世界观，因此，从某种意义上说，他们为开发人员创建的工具数量很多，他们显然凭借他们提供的数量赢得了巨大的市场份额，但我仍然觉得他们是一个反例。

> 他们似乎有我们很多人使用的 API，但没有很多开发人员喜欢使用的 API 和工具。

**大卫:**尽管如此，几乎每个人都还在使用他们的狮身人面像，对吗？

史蒂夫:是的。

大卫:所以这是有原因的。

史蒂夫:你知道，脸书的 API 或推特的 API，我不能说它们有多好或多坏，但是很多人都在使用它们。还有一点，产品作为一个整体对世界有如此大的价值和影响，以至于你被迫进入一个你需要使用它的地方。

这并不能免除你的责任，让你的工具对你的开发者和你的合作伙伴来说都很棒，但是，是的，当你在市场上占有很大份额的时候，看到能做多少事情是令人兴奋的。

David:AWS 数据库是另一个很好的例子。有这么多的选择，这么多的方式来做事情，你可以，喜欢，这是伟大的。你真的可以做任何事情。好像天空才是极限。但是，就像，找到幸福的道路往往是，它只是在开始的时候有很多尝试和错误。

你最终只需要围绕这些东西积累一些专业知识，最终你可以，好的，现在我可以理解如何使用这些微小的永久物来设计东西，但是没错，没有真正伟大的，就像我只有一些代码，我想把它放到互联网上。

Steve: 是的，Convox 的部分业务就是帮助开发人员完成这项工作，对吗？事实上，在 AWS 中创建一个基础设施有很多可能性，给予一点指导可能是帮助开发人员完成这一过程的好方法。

大卫:是的，基本上就是想创造一条快乐的道路。我认为有很多选择不是你必须要做的。

Steve: 是的，我们在 Opsee 也大量使用 AWS。这是一款专为亚马逊虚拟主机设计的监控产品。当我们创建该产品时，我们假设我们会看到我们的客户在如何配置其环境方面有相当高的一致性，但这几乎立即被否决了。

> 没有两个环境是完全相同的，所以现在我们要应对各种各样的客户环境，这是一个例子，说明人们可以利用这些工具做多少事情，但是他们选择做这些事情的方式有多么不同。

不过，这确实引出了另一个有趣的问题，即我们如何衡量开发人员体验的质量，以及我们如何知道我们做得很好。我们已经谈了一点关于价值实现的时间以及开发人员用我们的产品制作工具的速度，但我认为谈论这如何成为我们对这些工具定价的一部分也很有趣。比如，我们试图看看我们能为客户节省多少开发时间，然后根据这些时间给产品定价。你尝试过做类似的事情吗？

**David:** 我们试着把它想成一个同行开发者工具，所以，你知道，很多人试图对资源之类的东西收费，我们更像是对使用这个东西的每个开发者收费。所以我们实际上，实际上为开发者创造价值，我认为这是我们努力做的核心，所以我们从这个角度出发。

**Steve:** 是的，监控有点不同，我们主要是想衡量我们的开发人员在使用这个工具时不需要写多少代码，以及他们在竞争激烈的世界中需要做什么，但是

> 我认为所有的重量级公司，当然也包括所有的开发工具公司都有一个共同点，那就是我们正试图用这种方式来衡量我们的价值:我们能为开发人员节省多少代码，我们能为他们节省多少时间？

这在很多方面都不同于普通的用户体验设计和消费者用户体验设计。我们必须看到这一点，我们为用户创造的价值，以评估我们自己的价值。所以作为开发者体验产品的设计师，我认为谈论灵感和其他做得好的公司是很有趣的。

我第一次真正将开发者体验作为衡量标准是在看 Stripe，以及…你之前说过博客帖子和社区内的认可是多么重要。我记得听到许多开发人员谈论 Stripe 文档的质量，以至于它成为了优秀开发人员产品的同义词。

比如你有条纹级文档吗？所以他们总是一个晴雨表，你知道，我们做得好不好，我们用谁来衡量自己？

我觉得他们不仅在创建伟大的文档和伟大的 API 方面，而且在伟大的产品方面都做得很好。我是说甚至他们的网页设计。他们把设计的各个方面都提升到了一个我认为在我们的世界里不常见的水平。

大卫:是的，当然。我的意思是，我们使用 Stripe，我只记得当我第一次点击他们的网站时，我只是想，“哇，这真的很好。”当你使用它的时候，你只是对你自己有这样的想法，所以是的，它使用起来非常愉快。

史蒂夫:我不知道，你知道在 Heroku 的早期，我不知道你是否见过很多这样的东西，但可以肯定的是，在五、六年前，如果我们设计了一个非常漂亮的用户界面，我们的开发者客户几乎会有一点犹豫，我们没有街头信誉，“不，你不可能是好的，你不可能是真实的，因为在我们的世界里没有人会这样做，不会制造这样的产品。”

我认为这在过去几年里发生了很大的变化，但我们几乎是在打一场艰苦的战斗，试图创造一种优雅而美丽的用户体验，因为人们并不期望这样。

大卫:我当然认为这种情绪已经改变了。我认为有一个好的设计美学，这种感觉，实际上在大多数开发者圈子里被认为是一件好事，而不是像这样，如果你有一种风格感或其他东西，也许你不知道你在说什么。

是的，看起来很重要。当人们寻找工具或产品时，他们真的在寻找它，这也是我们倾向于看到庆祝的事情。

史蒂夫:这就是我们今天的全部时间。
