# 哈希公司预览领事服务网的更新

> 原文:[https://devo PS . com/hashi corp-previews-updates-to-consul-service-mesh/](https://devops.com/hashicorp-previews-updates-to-consul-service-mesh/)

在一次在线[hashi conf 全球](https://hashiconf.com/global/)会议期间，HashiCorp 今天发布了一个针对虚拟机和 Kubernetes 集群的开源咨询服务网格更新的[测试版，这使得多个管理员可以共享多租户服务网格的同一个实例。](https://www.hashicorp.com/blog/announcing-hashicorp-consul-1-11)

Consul 的 1.11 版本还将为 Kubernetes 添加对命令行界面(CLI)的支持，这将为在这些集群上安装 Consul 提供一种替代掌舵图的方法。

HashiCorp 的产品营销总监 Raymond Austin 表示，显然需要更高层次的抽象来管理应用程序编程接口(API ),以便在基于虚拟机的传统 it 环境和基于 Kubernetes 集群的云原生环境中轻松部署。

当然，在许多情况下，这些 Kubernetes 集群部署在虚拟机之上。然而，现在也有更多的 Kubernetes 被部署在[裸机](https://devops.com/?s=bare+metal)平台上，比如网络边缘。

随着 IT 环境变得越来越复杂，服务网格使得发现、访问和保护 API 变得更加简单，这些 API 现在正在企业中迅速扩散。奥斯汀补充说，这种转变需要一种更动态的方法，不仅管理 API，还管理这些 API 所依赖的网络服务。

![](../Images/9b923ec36466fa3ab160fa2998e42e15.png)

HashiCorp 打赌，许多 IT 组织将更喜欢依赖由单一供应商支持的轻量级开源平台，而不是依赖在财团赞助下开发的更复杂的开源服务网格产品。

仍然不太清楚 It 组织中谁负责部署服务网格。大多数情况下，是开发团队做出最初的决定。然而，随着 IT 运营团队越来越意识到服务网格代表了一种需要跨多个应用程序横向提供的能力，一些组织决定在单个服务网格上实现标准化以减少运营开销可能只是时间问题，Austin 指出。

在更大的 DevOps 工作流中，服务网格还将在更高的抽象级别上简化网络和安全服务的管理。

就企业对服务网格的采用而言，现在还处于早期阶段，但是已经有了大量的服务网格选项。事实上，可能永远不会有一个单一的主导服务网格平台。仍然需要批准互操作性标准，以使在不同服务网格上实现标准化的组织能够将策略应用于跨混合云计算环境的微服务。

当然，也不是每个 IT 组织都需要服务网格。许多 IT 团队仍然可以使用传统的代理服务器或 API 网关，直到部署的基于微服务的应用程序数量开始达到临界质量。然而，无论如何，要管理的 API 越多，it 团队部署服务网格就越不可避免。
