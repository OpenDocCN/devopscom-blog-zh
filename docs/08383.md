# 企业数字化转型的三大基石

> 原文：<https://devops.com/3-building-blocks-for-enterprise-digital-transformation/>

在过去的十年里，DevOps 已经从一个新的想法变成了每个会议室里的流行语。随着社会越来越依赖数字化，并推动他们对速度和便利的偏好，各种类型的企业都必须改变他们的业务文化，以帮助他们的团队以更高的速度和更大的自主权一起工作。

虽然疫情无疑提高了 DevOps 在工作场所的重要性，但它的广泛使用要早于前几年。早在 2018 年的 HGS 数字研究发现，只有 3%的软件开发企业从未听说过 DevOps。[2021 年:亚特兰蒂斯生态系统的未来报告](https://www.adaptavist.com/state-of-atlassian-report-2021)显示，54%的组织已经实施了 DevOps 战略，比 2020 年的 48%有所上升。还有 27%的受访者希望在未来三年内采用 DevOps 策略。

事实上，DevOps 现在已经成为主流，职位列表中“DevOps 工程师”的数量上升，以及企业迅速吹捧他们的“经验丰富的 DevOps 团队”，可能是这一点的进一步证明。然而，这些帖子和情绪中的一些也可以被认为是危险信号，因为它们表明组织正在将 DevOps 实践应用于孤立的团队。或者更糟的是，他们只是简单地给传统的工程角色贴上 DevOps 的标签，而不是实施适用于整个组织并使其能够以更快的速度发布可靠产品的完整思维转变。

考虑到这一点，对于希望避免常见问题并有效扩展其开发运维战略的企业来说，这里有三个关键的构建模块。这将有助于他们在开发和运营的各个方面改变工作和思考方式。

## 改变文化和赋予员工权力

DevOps 要取得成功，每个人都需要参与进来，而接受度在很大程度上取决于他们所处的商业文化。例如， [Gartner 预测](https://www.gartner.com/en/conferences/apac/infrastructure-operations-cloud-india/gartner-insights/swg-the-secret-to-devops-success#:~:text=The%20Secret%20to%20DevOps%20Success&text=DevOps%20has%20picked%20up%20pace,around%20organizational%20learning%20and%20change.%20%22)到 2022 年，75%的开发运维计划将无法达到预期。其原因与组织学习和变革有关。

换句话说，如果员工不理解 DevOps 更深层次的思考以及对他们个人和更广泛的群体的好处，他们自然会抵制变革。毕竟，对于许多部门来说，这是一种全新的思维方式，尤其是运营团队，他们传统上通过最小化变化和慢慢来降低风险。DevOps 鼓励他们通过[持续集成](https://devops.com/?s=continuous+integration)在故障快速且通常是故障前向的管道中运行来加速小的变更。

因此，教育员工对成功至关重要。特别是，组织应该让员工清楚地了解以不同方式工作的感受。共享成功的定义、共享词汇和共享要做的工作的可见性是关键。有了这一点，再加上分享可操作的见解，团队可以将这些见解应用到他们的工作中，然后观察结果，无论他们处于流程的哪个阶段，都可以实现 DevOps 的真正价值。

事实是，人是任何 DevOps 转型的最重要组成部分，这使得他们对事业的承诺对于成功来说是不可协商的。尽管软件开发的本质是技术密集型的，但像沟通、协作、灵活性和决策这样的软技能必须像安全、编码和基础设施知识一样优先考虑。所有企业都必须采取 DevOps 的思维方式，并在未来将这些特质融入他们的文化中。

## 将四个基本流程应用于 DevOps 的基础

也就是说，除非企业摆脱传统 IT 的“信任但验证”方法，否则不可能实现成功的 DevOps 转型。相反，它们必须包含一套基本的过程，使每个人都能够更加高效和有效地工作，以实现更快、更高质量的软件交付，从而满足最终用户的需求。这些基本流程包括协作、自动化、持续集成和持续交付(CI/CD)以及监控。

虽然协作一直是使全球分布的 IT 团队能够协调其目标并实现其目标的最重要的战斗口号之一，但自动化也非常重要。许多在 DevOps 环境中根深蒂固的人会告诉你，自动化使 DevOps 变得非常有价值。它帮助团队实现高水平的效率，并允许员工更多地关注复杂的问题，而不是重复性的任务。当公司的自动化战略与其更广泛的业务目标相一致时，它可以帮助实现大规模价值。

但是由于软件开发的本质，问题可能会出现。这就是持续集成和交付发挥作用的地方；它迫使开发人员将他们的源代码更新与其他人的更新进行合并，这比他们原本选择的频率要高得多。运营团队可以确保他们不会通过手动测试向用户发布版本来制造瓶颈。这种方法意味着更小的变更集；帮助更有效地识别、定位和解决缺陷。这增加了工作的流程，提高了发布的质量，并有助于避免开发周期结束时的棘手问题。

但是，如果企业不花时间来衡量什么在起作用，并评估哪些行动需要改进，那么拥有一个具有真正协作文化的快速、高度自动化的 CI/CD 流程是没有意义的。这就是为什么有效的 DevOps 计划需要建立在“实施、衡量和改进”的基础上。毕竟，DevOps 是关于通过增加工作的可见性和不断的反馈来最大化价值和效率。测量实时指标对于确保业务转型始终与其目标保持一致变得至关重要。如果企业坚持这些基本原则，他们就能在完全接受的情况下顺利实现成功的开发运维战略。

## 采用正确的转型工具

随着团队的成长和开发运维流程的扩展，可预测的性能是必不可少的。因此，企业的工具必须全天候可用，并且应该高性能地支持他们的协作管道。

但是使用 DevOps 工具并不自动意味着组织正在应用 DevOps 原则。工具的价值受限于企业在上述文化和过程中整体使用它们的承诺。正是这种结合使他们能够转变开发和交付产品的方式。综上所述，重要的是要理解采用适当的工具链——以及各种各样的选择和组合——是多么重要，这就是为什么很容易理解为什么决策者可能会不知所措。

毕竟，要记住的东西很多。他们需要选择能够加强协作、将上下文切换保持在最低限度、有助于自动化、鼓励可见性并允许有效监控的工具。但也有风险，他们会选择他们觉得最舒服的解决方案，这可能与转型的直觉相反。这种方法会导致工具狂潮，在集成不是优先考虑的地方带来太多的技术，并使企业没有任何东西来完成他们需要的工作。

考虑到这一点，企业需要花时间咨询哪些工具适合他们，同时时刻牢记 DevOps 的基本原则。为了说明这一点有多么有价值， [DORA 和谷歌云的 2019 年 DevOps 报告](https://services.google.com/fh/files/misc/state-of-devops-2019.pdf)发现，表现最好的工程师利用易于使用的工具的可能性是其他人的 1.5 倍。同时，未能做到这一点可能是 DevOps Institute 的另一项[研究发现超过 50%的受访者的 DevOps 转型之旅仍然艰巨的主要原因。](https://www.bmc.com/documents/analyst-research/upskilling-2020-enterprise-devops-skills-report.html)

## 向 DevOps 迈出下一步

DevOps 的成功转型依赖于对基本原则和实践的深入理解、大规模认同和全组织采纳。当这种敏捷思维应用于组成组织的人员、流程和工具时，企业可以推动团队的重大变革，同时提高产品的可靠性、稳定性和安全性。

记住 DevOps 的设计目标是让谁受益最大——客户，这一点非常重要。这种心态在今天的环境中尤其有价值，因为企业和消费者都比以往拥有更多的选择。随着速度和便利性成为大多数人购买决策的首要考虑因素，DevOps 可以在帮助组织追求卓越方面发挥巨大作用，并为客户提供他们甚至不知道自己需要的东西。使用上述构建模块，各种规模的组织都可以显著改善他们的客户体验，并更有效地快速将正确的产品送到他们手中。
