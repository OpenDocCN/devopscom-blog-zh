# 所有的测试都去哪里了？

> 原文:[https://devops.com/where-has-all-the-testing-gone/](https://devops.com/where-has-all-the-testing-gone/)

总的来说，测试一直是 IT 的一个问题，特别是 AppDev，现在它是 DevOps 的问题。DevOps 可以做一些事情来[提高您的应用程序实际发生测试的机会](https://www.cprime.com/resources/blog/test-automation-and-devops-what-you-need-to-know/)。有趣的是，有些人认为我们不再需要测试；理由是，如果某个东西需要修复，新的版本可以很快推出，以至于它是不相关的。将特性标志与推出新版本的能力结合起来，嗯，这可能是有一定道理的——至少对于有限的测试子集来说是如此。

如果一个组织为用户接受和新特性测试实现了[特性标志](https://devops.com/?s=feature+flags)，然后为 DevOps 开发团队标准化了测试驱动开发，那就剩下系统和安全测试了。理论上，系统测试可以被推出新版本的能力所取代。尽管系统问题可能很复杂，很难解决，但这比将测试驱动开发扩展到系统层面更具风险。对于 sprint，指派一个人负责整个集成系统，并让团队的其他成员在确保所有工作按预期进行的过程中支持他们。

测试——尤其是系统测试——会减缓发布。我明白了。一旦 DevOps 系统和过程就位，我们大多数人迭代的速度足够快，以至于你可以承受一点减速。这使得安全性测试成为最后一个独立的部分。但是如果使用 DevSecOps 工具，在开发过程中会发生一些安全验证。这仍然留下一些安全问题需要测试，但确实消除了一大块。

这些最后一点的关键是将它们整合到流程/工具链中。DevOps 直接穿过，一遍又一遍。任何进入“直通运行”的位都将被执行。如果今天的迭代陷入了“我们超时了，我们会用更少的测试来完成最后期限”的心态；这不是问题，这只是一次迭代。只是不要让这成为某人不能按时完成迭代工作的默认答案。开发可能会遇到意想不到的阻碍，从而延迟交付，承认这一点并记录下来以备将来参考是非常重要的。通过减少测试时间来掩盖这一点会导致产品不稳定，并且无助于将来解决这个问题。

所以开始从 DevOpsTest 的角度来思考。确保你交付的产品代表了你在生产和维护上投入的大量时间。你在摇摆它——像集成测试这样的东西可以确保用户看到你在摇摆它，而不是一个错误页面。
