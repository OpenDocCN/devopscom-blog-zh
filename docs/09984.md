# 数据狗深入通用服务监控

> 原文:[https://devo PS . com/data dog-dives-into-universal-service-monitoring/](https://devops.com/datadog-dives-into-universal-service-monitoring/)

Datadog，Inc .今天使通用服务监控服务普遍可用，该服务利用 Linux 操作系统中的扩展的 Berkeley 包过滤(eBPF)微内核来自动检测构成应用环境的所有服务，而无需改变用于构造它们的代码。

Datadog 产品副总裁 Yrieix Garnier 表示，随着应用环境的不断发展，IT 团队现在不仅试图跟踪单个微服务，还试图跟踪整个业务流程，如订单到现金。然而，在这些服务可以被装备之前，DevOps 团队需要一种简单的方法来发现它们，他指出。

Garnier 补充说，在 Linux 操作系统中，eBPF 使得采用沙盒环境来识别和映射组成应用环境的服务成为可能，包括内部开发的和第三方服务，而不管使用什么编程语言来构建它们。

开发运维团队还可以通过实时请求率、错误和持续时间(RED)指标以及相关的基础设施指标和应用程序日志，了解每项服务和部署的运行状况。然后，他们可以扩展通过 Datadog 代理软件提供的监控功能，以采用可以与[可观察性](https://devops.com/?s=observability)数据相关联的分布式跟踪。

Datadog 通用服务监控服务还与服务目录集成，Datadog 提供服务目录来确定哪个开发团队创建了特定的服务。

Garnier 指出，目标是缩小可见性差距，因为在数字化转型时代，越来越多的 IT 组织围绕特定的应用服务而不是组成服务的组件来调整管理工作。

如今，DevOps 团队同时承担着深入应用环境的任务，以便在服务中断之前发现问题，并跟踪跨多个应用组件的服务。总的来说，我们期望开发运维团队能够确保应用服务不仅可用，而且能够持续交付。最终，IT 组织的任务是消除过去 IT 管理中的大量猜测。

不太清楚的是，这些转变最终将在多大程度上推动 IT 团队的重组，这些团队仍然主要围绕他们支持的平台和交付的服务进行组织。然而，随着企业领导越来越意识到整个组织对应用程序服务的依赖程度，以更全面的方式管理它们的需求变得显而易见。

与此同时，DevOps 团队正在评估他们现有的工具和平台是否能够帮助他们实现这一目标。Datadog 正在为采用它代表 DevOps 团队管理的服务提供理由，而不是每个 DevOps 团队都必须构建和维护自己的可观测性平台。最近，[data dog 扩展了其同名云交付监控和可观察性平台](https://devops.com/datadog-extends-reach-of-integrated-devops-platform/)的范围，以解决持续测试、应用安全和成本管理问题。

将由每个组织决定如何最好地实现他们的可观测性目标。可以肯定的一点是，他们目前对应用程序环境的可见性还远远不够。
