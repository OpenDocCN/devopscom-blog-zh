# 将移动开发平台转变为移动开发平台

> 原文：<https://devops.com/transform-mobile-devops-into-mobile-devsecops/>

移动开发者大多专注于为他们的应用添加新的特性和功能。安全特性——如 RASP 保护、混淆、加密、越狱/root 预防和 MiTM 预防——通常会在开发周期的最后解决。不幸的是，大多数应用程序都有严重的漏洞。一项内部应用测试表明，大多数应用可以在 15 分钟内被破解，最近的一项学术研究发现， [67%的恶意应用安装来自谷歌 Play 商店](https://arxiv.org/pdf/2010.10088.pdf) 。

2021 年 8 月发布的一项针对 1 万多名智能手机消费者 的 Appdome [全球调查显示，这种将功能置于安全之上的方法是一个巨大的错误。近三分之二(63%)的消费者表示，在使用](https://www.appdome.com/consumer-security-survey/)[移动应用](https://devops.com/?s=mobile+apps)时，他们对安全性的重视程度不亚于功能。近四分之三(73%)的人表示，如果手机应用程序无法抵御攻击，他们就会停止使用。

这项调查揭示了消费者情绪的巨大转变。近年来，消费者第一次站在了安全团队一边！他们期望得到强有力的保护，免受威胁和欺诈。但是满足这些期望对于开发团队来说是极具挑战性的，因为开发和安全经常是不一致的。

原因如下:从 GitHub 到 Jenkins 再到浪子，DevOps 中几乎所有的工作都是自动化的，而几乎所有的安全工作都是手动的。当然，安全软件开发工具包(SDK)供应商将自己定位为备选方案，但是购买者要小心。实现安全 SDK 绝不仅仅是几行代码。这仍然是一个脆弱的手动过程，不能提供有保证的结果。

目前可用的移动应用安全解决方案不适合 DevOps 这种自动化交付模式。自动化安全性并使其适合 DevOps 工作流的唯一方法是停止试图在开发周期结束时将其挤进去。因此，安全性在今天的移动应用程序开发中没有一席之地。只要安全性是手动的，并且不能无缝地插入到现有的 DevOps 工作流中，它充其量将仍然是一种事后想法。

### **加密示例:为什么安全性如此困难**

作为一个例子，让我们来看看如何整合移动加密，这是保护数据免遭窃取、窃取、拦截、欺骗和欺诈的绝对必要条件。对于应用中的每种数据类型，开发人员必须为加密模型的每个组件选择正确的组合，包括加密算法/协议、密码套件、密钥强度、密钥派生技术、密钥保护和安全措施。它必须满足高度可变的数据格式特征集的安全性和性能要求。即使是很小的错误或不匹配也会降低应用程序的性能或导致应用程序不可用。另一方面，应用程序可能运行良好，但加密可能太弱，无法提供足够的保护。

这是一项复杂、耗时的工作，开发人员不想重复做。这就是为什么他们倾向于等到应用程序构建完成后才添加加密。不幸的是，这意味着所有的网络信息都以不加密的方式存储在应用程序中，如果一个坏演员得到了该应用程序的 UAT 版本，那么“王国的钥匙”就在明处，随时可以被窃取。

对开发人员来说，快速发布新版本的要求很高。安全性通常被视为阻碍创新和延迟发布的负面力量，尤其是当它被留到软件开发生命周期(SDLC)的最后。术语 DevSecOps 用于描述以安全为中心的连续交付 SDLC。但是这应该不仅仅是渗透测试。识别问题是不够的。安全性需要被构建到开发周期中，而不减慢发布时间或产生不可控的成本。

将安全性整合到 DevOps 中并实现 DevSecOps 的唯一方法是将安全性与开发目前所享受的自动化程度相结合。通过使用将安全性直接构建到应用中的自动化、无代码平台，安全性可以成为 DevSecOps 工作流的自然组成部分。此外，安全性可以与证明保护的文档一致地应用。这是使开发人员能够快速将其安全的移动应用交付到最终用户手中、将所有利益相关方与一组共同的目标和成果联系起来，并消除开发人员和安全团队之间常见的摩擦来源的唯一方法。
