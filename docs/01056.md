艾伦·希梅尔:大家好，我是《DevOps.com》的主编艾伦·希梅尔，欢迎来到另一个 DevOps 聊天室。今天，我们 DevOps 聊天的嘉宾是 IBM 的 Dan Berg，我们将讨论 DevOps 和开放技术。丹，欢迎来到 DevOps 聊天。

丹·伯格:谢谢你，艾伦。

*Alan Shimel:*Dan，在我们进入整个开放技术、开放企业主题之前，你能给我们的听众简单介绍一下你是谁、你做什么以及你今天为什么在这里吗？

丹伯格:当然。正如你所说，我叫丹·伯格。我确实为 IBM 工作。我是一名杰出的工程师，在我们的云平台组织中工作，这基本上意味着我负责与一组非常有活力的团队合作，这些团队正在我们的云基础中构建服务。因此，IBM 云平台，特别是 IBM Bluemix，像我们的容器服务、登录和监控服务，几个基于 DevOps 的服务。因此，我们称之为主动部署的服务有助于在云架构中部署微服务。所以我在这方面做了很多工作，帮助团队确定我们将使用的策略和架构。我还积极参与了 IBM cloud 中的 DevOps 组织，重点关注云原生应用程序和混合云应用程序。

*Alan Shimel:* 太棒了，Dan，你知道，我们谈到 DevOps，当然当我们谈到云时，有人会说如果没有开源，我们就没有 DevOps 或云。你认为那个声明怎么样？

丹·伯格:我认为这是一个非常深刻的说法。开源是云战略的重要组成部分。

我认为几乎所有的客户都将云视为一个方向。我与我们的许多客户交谈过，我不能说他们中没有任何人没有云战略，或者至少没有追求云战略。

经常出现的一个领域是能够灵活选择的概念。我希望能够选择我的云提供商，我可能需要多个云提供商。

真正实现这一点的唯一方法是拥有一个开放的社区和开放的标准，允许您开发工作负载，开发应用程序，以便您可以在运行时将它们从一个提供商转移到下一个提供商，就像您希望它们跨多个提供商一样。

正如我所指出的，除非你专注于一个推动开放标准的强大的开放社区，否则你不会得到它。

*Alan Shimel:* 是的，你知道 Dan，我已经老了，我记得当我第一次接触 IT 和技术时，曾经有一段时间对开源企业有巨大的阻力。

我们都读过这些书，听过理查德·斯托尔曼的歌，但这本书现在我已经记不起来了。关于开源的伟大著作是什么？关于集市和——哦，随便啦。

天哪，我会找到的，但更重要的是，你知道我们——是什么阻碍了？

没有支持，没有审查等等，企业怎么能依赖开源呢？

事实上，像 IBM 和微软这样的公司已经为开源进入企业铺平了道路。你对此有什么想法？

Dan Berg: 嗯，看看开源是如何发展的，开放开发是如何真正改变开发者学习的方式，以及他们如何采用技术，这很有趣。

首先，您将开放社区视为了解新技术并加入这些技术不断发展的社区的一种方式，您是对的。

企业通常认为或者在过去认为开源是人们兼职做的廉价软件。他们不属于任何公司。我们怎么能相信这个？

it 企业准备好了吗？我们最好不要碰它。对，我们最好开发自己的。我认为这种心态已经发生了巨大的变化，原因有很多。

第一，社区变得非常大。我的意思是看看像开放堆栈社区或云铸造社区，Docker 社区。

这些社区已经建立了一组基本的用户和贡献者，并且在过去的几年里有了很大的发展，他们正在开发的东西是以企业的心态开发的。我的意思是它不仅仅是用来踢轮胎或玩耍的。

我们正在开发开放的企业级解决方案，之所以这样做是因为有了围绕这些计划的社区，你最终会从各种利益相关者那里获得更多的投入，最终，你会得到更好的解决方案，满足更多利益相关者的需求。

我知道在 IBM 内部，我的意思是我们已经开放开发了很多年。我记得很久以前，15 年前，在野外工作。

虽然最初可能会很难，但当你想到哦，我必须与一群人一起工作并达成一致时，这在某种程度上是正确的。

但你会发现，随着越来越多的人开始在全球范围内就一套技术，一套开放的技术，一起工作和协作，它实际上加快了步伐，你可以更快地创新，因为你只是有一个更大的资源池和开发人员在那个代码基础上工作。

这就是我们所看到的，在企业中采用并使用它变得更加容易，因为您已经有了像 IBM、微软这样的企业，以及其他为这些开放社区做出贡献的企业，并确保将企业需求添加到这些代码库中，以便企业可以使用它们。

因此，它正在转变。我认为越来越多的企业正在采用开源功能，因为这些社区带来了创新和企业级的质量。

艾伦·希梅尔:当然。顺便说一下，丹，这本书是《大教堂和集市》。

丹·伯格:是的。

艾伦·希梅尔:作者埃里克·雷蒙德，我相信你也读过。你知道吗，在这一点上，我很惭愧地告诉你。它已经有 15，20 年的历史了，但是，不管怎样，它还是不错的。

丹，让我带你转一点，让我们快速转一下。这个月我们有 IBM 互连展，这是 IBM 在移动领域的大云，我也想说是 DevOps 活动。

根据您刚才告诉我们的内容，开放企业和开放源代码是最重要的，也是理所当然的。

关于今年的互连大会上的开源、开放企业，您能为我们重点介绍一下什么吗？

丹·伯格:是的，所以会有很多令人兴奋的事情。我的意思是，我们确实推动了开源和开放技术的概念。

显然，我们将围绕开放堆栈、node 和 JavaScript 以及 Docker 讨论一些核心计划。

我们在这些社区中举办了许多激动人心的会议和活动。

open container initiative 是我们参与并讨论的另一个大项目，我们正在帮助该社区打造一个方向，推动容器的发展。

这就是基于 Docker 的容器和其他容器，真正推动了围绕容器运动的更强大的社区。甚至在 DevOps 周围，我们将展示的非常非常棒的东西是我们用 IBM garage 方法完成的一些工作。

因此，这是我们去年年底宣布的一套开放的流程和实践，然后在今年进一步推进，我们正在做的是将实践与一套工具相关联。

这些工具显然是我们提供的一些工具，我们提供的服务，但它们中的许多也是开放工具，正在开发并在开放社区中提供。

但我们在这一领域推动的令人兴奋的事情之一是我们称之为开放工具链的东西，开放工具链是一组功能，允许组织在工具链中定义他们的最佳实践和开发运维功能。

可以由其他方实例化的一组集成的实践和工具。

因此，我们对这项倡议感到非常兴奋。虽然现在还为时尚早，但我们确实看到了通过将这些开放技术整合到一个统一的工具链中来推动云原生部署和交付机制所带来的潜力和灵活性。

所以我们非常兴奋。这些社区正在发生很多事情，我们真的很高兴能在两周半或三周内在互联上分享这些信息。

*Alan Shimel:* 当然，你知道，Dan，我想当他们在旧金山宣布开放车库方法时，我可能会和你在一起——我忘了是什么会议，但无论如何，这是一个 DevOps 企业峰会。

*丹伯格:*没错。

艾伦·希梅尔:正如你提到的。

*丹伯格:*没错。

*Alan Shimel:* 你知道，对我来说这有点不同，尤其是对 IBM 来说，因为它不是一个开源项目。对，它不是一种特殊的工具。

这实际上更像是一种开放的方法，其中捆绑了一些工具，你可以谈论供应链。

这是一种软件供应链，你正在开发的软件或应用程序，并不是所有的都是开源的。

但是有一些开源组件可以让我们更容易地持续实时开发应用程序，持续交付和集成，以及我们在 DevOps 中看到的所有这些优点。

对我来说，这就是车库方法的力量，对吧，就是这个软件供应链，如果你愿意的话，用开源组件，构成了它的一大部分，围绕着 IBM 的经验和技术将它结合在一起。我觉得很厉害。

丹·伯格:我完全同意，正如我指出的那样，现在还为时尚早，但它显示出了希望。我的意思是，我们都知道有时将您的交付工具链与各种开源工具以及第三方供应商特定的工具拼凑在一起是多么困难。

这可能非常复杂，而开放工具链真正的好处是它消除了一些复杂性，并真正帮助您专注于您的关键计划，即尽快交付高质量的代码。

所以我真的很兴奋。它确实建立在我们的一些核心技术之上，但我真的很兴奋它如何引入这些关键的开放技术和开放社区来推动更好的整体开发运维体验。

艾伦·希梅尔:是的，我同意。你知道，丹，不幸的是我们的时间不多了，但这是——你知道，软件供应链的开源，开放企业的整个主题。

企业如何将开源组件整合到他们的供应链中，这让我很感兴趣，我想你会在 DevOps.com 上看到更多，听起来也会在本月晚些时候的互连上看到很多。
