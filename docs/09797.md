# 利用经验数据来避免开发人员精疲力竭

> 原文:[https://devo PS . com/leverage-experimental-data-to-avoid-devo PS-burnout/](https://devops.com/leverage-empirical-data-to-avoid-devops-burnout/)

任何行业都有可能出现倦怠。但它似乎在科技领域尤为猖獗，这个领域充满了膨胀的预期和不断加速的趋势。TeamBlind 的一项研究发现，近 60%的科技工作者表示，他们目前在工作场所感到筋疲力尽。

限制倦怠和支持员工心理健康是必不可少的，这不仅是为了员工，也是为了企业。在这个被评论者称为大辞职的时期，如果员工感觉不到被尊重，他们可能会去其他地方寻找新的机会。在持续的[云技能短缺](https://devops.com/4-tips-to-plug-the-cloud-it-skills-gap/)中，确保工程师的需求得到满足是一个至关重要的目标，DevOps 领域当然也不能幸免于这一人才缺口。

我最近会见了 AHEAD 的高级解决方案架构师 Eric Mader，以了解与开发运维相关的倦怠的根本原因。根据 Mader 的说法，倦怠可能由许多因素引起，包括工作生活不平衡和工作场所中相互冲突的自我。但最重要的是，倦怠通常是由管理层设定不切实际的目标引起的，这些目标并不是基于经验的团队绩效数据。

## 哪些因素导致 DevOps 倦怠？

那么，在 DevOps 中什么会导致倦怠呢？首先，工作场所[的文化](https://devops.com/majority-of-open-source-devs-open-to-change-jobs-in-2022/)会极大地导致倦怠。马德解释说，在大多数远程工作模式中，很难确定正确的工作生活平衡。例如，可能有一个不成文的期望，即在下班后或周末立即回复垃圾邮件或迅速回复电子邮件。

“永远在线”的态度会导致不健康的工作习惯。虽然这不一定是 DevOps 独有的，但由于执行工作所需的高连接性，这有时是不可避免的。马德尔指出，另一个导致倦怠的潜在因素是团队动力。“每个团队似乎都有自己独特的方式，”他说。

尽管敏捷和 DevOps 涉及非常技术性的工作，但有时自我会妨碍健康的团队动态。例如，在许多技术团队中，通常会有一位“大师”，如高级开发人员或技术负责人，每个人都会去寻求建议，Mader 说。这个人可能最终完成了大部分工作(包括技术和人际关系)，并且通常感觉最疲惫。或者，其他想成为领导者的人可能会扮演不必要的指挥和控制角色。

与同事的性格差异可能会导致倦怠，但其他因素如处理[辛劳](https://devops.com/shedding-light-on-toil-ways-engineers-can-reduce-toil/)或[管理遗留技术债务](https://devops.com/enterprises-face-growing-technical-debt/)也会同样令人厌倦。

## 快速移动并打破东西

根据 Mader 的说法，也许导致倦怠的最大因素是高管们坚持认为技术人员一直超过他们以前的产出——特别是当过去的团队绩效数据不支持这些目标时。设定可实现的目标是很好的，但是当可实现的目标超出范围时，它会让人精疲力尽。

马德尔说，提高速度的部分压力来自于“越快越好”的持续错觉。价值通常归因于一次冲刺中完成的项目数量，而不是每一点的质量。例如，一个团队可能在 sprint 中交付 200 个工作故事点，但是其中只有 30 个点具有重要的价值。或者，一个领导者可能看到一个团队在一次冲刺中完成 60 分的工作，然后在下一次冲刺中鼓励 200 分。

这种高期望受到基本能力问题的困扰。马德解释说，增加的压力会很快意识到自己是一种负担，会导致精疲力竭。他鼓励技术领导不要强迫做更多的工作，而是“基于你所拥有的经验数据，专注于最有价值的工作。”

## 避免精疲力尽的五个技巧

Mader 分享了一些关于如何在 DevOps 团队中避免倦怠的技巧。这里有一些简单而有效的方法来保持更快乐的工程师和更高效、更健康的团队关系。

**扼杀永远在线的态度**。在精神健康意识方面，组织需要言行一致。因此，确保员工知道他们的生活界限得到了尊重。通过尊重假期、陪产假或个人假期来鼓励这一点。不要在周末发邮件，鼓励在团队聊天和工作邮件账户上打开“请勿打扰”提醒。

**区分自动化的优先顺序**。任何自动化都是好的自动化，这是一种误称。事实上，软件开发自动化的[过度规定](https://accelerationeconomy.com/automation/the-benefits-and-pitfalls-of-software-development-automation/)会导致一些负面后果。Mader 说:“我见过许多 DevOps 团队将如此多的时间集中在自动化低影响业务流程上。相反，优先考虑影响更多人的自动化项目和那些会被持续使用的自动化项目。

**试试“五拳**”确保每个人的声音都被听到有助于避免有毒的工作环境。在团队会议中快速评估反应的一个策略是使用 Mader 所说的五个拳头:如果一个想法完全错了，就给一个，如果一个想法完全有意义，就给五个。这种简单的做法可以确保每个人都能对新项目做出贡献。

把房间里最安静的人拉到一边。马德指出，只有吱吱响的轮子才能得到油。记住这一点，会后把最安静的人拉到一边，收集他们的意见——你可能会感到惊讶。资历较浅的人可能不太容易说出自己的想法，他们通常有一种不那么制度化的心态，可以提出更多创新的想法。

**用经验数据告知未来的目标**。最后，Mader 鼓励 DevOps 团队“使用经验证据来支持你未来的发展方向。”具体来说，在 DevOps 领域，经验数据采用的是像[平均恢复时间](https://devops.com/5-mean-time-reliability-metrics-to-follow/)或[SLA、SLIs 和 SLOs](https://devops.com/whats-the-difference-between-sli-sla-and-slo/) 这样的指标形式。

请记住，即使一个团队在一次冲刺中完成了一定数量的目标，也并不意味着他们下一次能够产生完全相同的结果——他们可能已经被推到了极限，产生了过去的结果。因此，管理人员应该致力于逐步推进指标，而不是大踏步前进，以免挫伤士气。

## 对 DevOps 采取固执己见的态度

如今，开发运维团队被宠坏了——市场上有[无数的工具](https://devops.com/11-open-source-devops-tools-we-love-for-2021/)和源源不断的最佳实践和方法。即使在一个公司内， [DevOps 工具集](https://devops.com/how-does-it-management-govern-everythingops/)也可能因团队不同而有很大差异。为了帮助理解这一切，Mader 鼓励 DevOps 领导通过设置防护栏和采用更具主见的方法来简化转型之旅。
