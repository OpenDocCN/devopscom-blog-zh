# 15 个开发部门最佳实践

> 原文：<https://devops.com/15-devsecops-best-practices/>

DevOps 的核心是速度、灵活性和协作。但是当涉及到安全性时，DevOps 团队经常面临独特的挑战。从保护应用程序开发过程到保护生产环境，DevOps 和 DevSecOps 团队需要了解各种潜在的安全风险。

为了帮助您保持领先，我们编制了 15 个 DevOps 安全最佳实践和挑战的列表。

### 1.保护您的应用开发流程

保护 DevOps 管道的第一步是确保您的应用程序开发过程是安全的。这意味着确保只有经过授权的开发人员才能访问您的代码库，并且所有的代码更改在合并到主分支之前都经过了合格的评审人员的评审和批准。这也有助于让您信任的开发人员正确地完成工作，并自始至终遵守网络安全最佳实践。像[【rightpeoplegroup.com】](https://rightpeoplegroup.com/it-security-experts/)这样的咨询场所让寻找这类专业人士变得简单多了。

### 2.保护您的生产环境

生产环境是客户最终部署和使用应用程序的地方。因此，确保这种环境尽可能安全非常重要。

一种方法是将您的生产环境划分为单独的层，每层都有自己的访问级别和 [安全控制](https://webinars.devops.com/integrate-security-controls-wherever-you-build-applications) 。这样，即使一层受到威胁，其他层仍会受到保护。

### 3.实现最小特权原则

一般来说，在授予对 DevOps 资源的访问权时，最好遵循最小特权原则。这意味着只给用户执行工作所需的权限，仅此而已。遵循这一点非常重要的原因是，您的员工构成了您最大的网络安全威胁[](https://www.cnbc.com/2018/06/21/the-biggest-cybersecurity-risk-to-us-businesses-is-employee-negligence-study-says.html)。这并不总是出于邪恶的原因，而往往只是因为他们没有知识或理解来随时保持您的业务的数字安全。

### 4.使用基于角色的访问控制(RBAC)

基于角色的访问控制(RBAC)是一种访问控制类型，可用于根据用户的角色限制对 DevOps 资源的访问。例如，您可以创建一个可以访问您的代码库的“开发人员”角色和一个可以访问您的测试环境的“测试人员”角色。通过使用 RBAC，您可以帮助限制内部威胁可能造成的损害。

### 5.加密敏感数据

任何可能用于识别或伤害个人的数据都应该加密，无论是静态数据还是传输中的数据。这包括数据，如社会安全号码，信用卡号码和健康信息。

加密数据的一种方法是使用非常好的隐私(PGP)加密。PGP 结合使用公钥和对称密钥加密来保护您的数据。

### 6.使用双因素身份验证

双因素身份验证( [2FA](https://devops.com/setting-up-google-authenticator-on-multiple-servers/) )是一个额外的安全层，可用于保护对 DevOps 资源的访问。使用 2FA，用户需要提供两个证据来验证他们的身份。第一块是他们知道的东西，比如密码，第二块是他们有的东西，比如手机。

实施 2FA 有助于防止对资源和系统的未经授权的访问，即使用户的密码被泄露也是如此。

### 7.使用机密管理工具

秘密是任何应该保密的敏感信息，如密码或 API 密钥。机密管理是安全存储和管理机密的过程。

有许多可用的秘密管理工具，如 Hashicorp 的 Vault 和 AWS Secrets Manager。这些工具可以帮助您集中管理机密，并提供访问控制和审计功能。

### 8.对员工进行安全意识培训

提高 DevOps 安全性的最佳方法之一是对您的员工进行安全意识培训。这可以帮助他们理解安全的重要性，并识别和降低风险。

有许多不同的安全意识培训计划，如 SANS 安全意识计划。或者，您可以创建自己的程序来满足您组织的特定需求。

### 9.使用网络应用防火墙(WAF)

web 应用程序防火墙(WAF)是一种可用于保护 web 应用程序免受攻击的防火墙。WAFs 通过检查传入流量和阻止包含恶意负载的请求来工作。

有许多不同的 waf 可用，包括开源的和商业的。WAFs 的一些例子包括 Apache 的 mod_security、NGINX Plus 和 F5 的 BIG-IP ASM。

### 10.执行定期安全审计

定期安全审计是 DevOps 安全的[](https://www.linkedin.com/pulse/importance-regular-security-audits-mike-lorenzen-sphr-mcp-vcp?trk=pulse-article_more-articles_related-content-card)重要组成部分。它们可以帮助您识别系统中的弱点，并确保您的安全控制是有效的。

有许多不同类型的安全审计，例如渗透测试和代码审查。根据您的需求选择正确的审计类型非常重要。如果不确定，可以咨询安全专家。

### 11.使用入侵检测和预防系统(IDPS)

入侵检测和防御系统(IDPS)旨在检测和阻止恶意活动。IDPSes 可用于保护物理和虚拟资源。

有许多不同的 IDPSes 可用，既有开源的，也有商业的。IDPSes 的一些例子包括 Snort、Suricata 和 Bro。它们通常被部署为安全信息和事件管理(SIEM)系统的一部分。

### 12.实施灾难恢复计划

灾难恢复计划(DRP)是概述在发生灾难、违规或其他安全事件时应采取的步骤的文档。DRP 应包含关键人员的详细联系信息和恢复系统的程序等信息。

DRP 有助于将灾难的影响降至最低，并确保您的组织能够及时恢复。

### 13.使用日志和监控工具

日志和监控工具可用于收集系统活动的相关数据。这些数据可用于检测和调查安全事件。

有许多不同的日志和监控工具可用，既有开源的也有商业的。日志和监控工具的一些例子包括 Splunk、ELK Stack 和 Nagios。

### 14.进行定期渗透测试

渗透测试是一种模拟对系统攻击的安全测试。pentesting 的目标是识别可能被攻击者利用的漏洞。

渗透测试可以在内部或外部进行。外部渗透测试通常由第三方安全公司执行。内部渗透测试可以由您自己的员工进行，或者使用 Metasploit 之类的工具进行。

### 15.使用访问控制列表

访问控制列表(ACL)是一种安全措施，可用于限制对 DevOps 资源的访问。ACL 通过定义一组规则来决定谁可以访问什么。

ACL 可用于实施最低权限策略，并有助于防止对敏感数据的未授权访问。

### 结论

当涉及到 DevOps 和 DevSecOps 时，存在大量的威胁，同样，也存在大量可用于改进 DevSecOps 的最佳实践。通过实施这些最佳实践，您可以帮助保护您的系统免受攻击。
