# DevOps 聊天:死胡同计算机语言与斯科特·威尔森

> 原文:[https://devops.com/devops-chat-scott-wilson-automic/](https://devops.com/devops-chat-scott-wilson-automic/)

Automic 的斯科特·威尔森总是 DevOps 知识的重要来源。我和他谈过几次，他从来没有失望过。在这次 DevOps 聊天中，Scott 向我们讲述了没有出路的计算机语言。Scott 本人也是一名开发人员，他经常惊叹于当今可用语言的爆炸式增长，并想知道有多少语言能够在编写它们的应用程序的使用寿命结束后继续存在。

这是一个伟大的观点。我们不久前做了一项调查，调查显示，一般组织同时支持 10 种以上的语言。说说巴别塔吧。斯科特给了我们一些很好的背景知识。

**艾伦·希梅尔:**大家好——艾伦·希梅尔，DevOps.com，又来参加 DevOps 聊天了。本期 DevOps Chat 的嘉宾是 Automic 的斯科特·威尔森。斯科特，欢迎来到 DevOps 聊天。

威尔逊:好的，太好了。谢谢你艾伦。我目前是 Automic Software 的产品营销总监，基本上，我特别专注于 DevOps 和应用程序发布自动化。

我有很好的技术背景。我从高中开始写代码。我是一名很长时间的软件开发人员，你知道，在许多发展中，不同的垂直市场编写代码，我现在最终演变成营销，所以。

**Shimel:** 但是不管怎样，Scott，我们今天要谈一点，你知道，我们可以称之为死胡同语言——我的意思是，你知道，现在是开发人员的大好时机。在你想发展的方面，你有很多选择。

但是，就像其他事情一样，你知道，在你可以选择的开发语言中，会有赢家和输家。我知道这是一个—

威尔逊:绝对是。

**Shimel:**–是的，这是你说过和写过的话题。所以我想，你知道，在这个前提下，你为什么不向我们的观众解释一下你在说什么？

威尔逊:好的。你知道，正如我在介绍中提到的，Alan，我对代码并不陌生，我承认我仍然为各种各样的事情写代码。在我的经历中，特别是在与客户交谈，甚至为几家应用发布自动化产品供应商工作的经历中，有一件事让我印象深刻，那就是，一般来说，在 DevOps 中，人们对语言有着极大的兴趣和实验。例如，许多 ARA 工具为您提供了构建自己的插件的方法，这样您就可以扩展这些产品，并将其扩展到构建一个良好的、内聚的工具链——如果您愿意，可以称之为 DevOps 管道。

挑战在于，当我开始关注这些产品时，DevOps 的发展势头很大，我开始意识到人们选择的语言在到达时就已经死了。真正激发我对此感兴趣的不仅仅是我在这里的观察，Alan，我还与一家大公司的高管交谈过，他告诉我，他们不得不重写大量的 DevOps 代码，也可以说是部署代码，因为五年前编写管道的人使用了一种语言，他们再也不能雇人来编写这种语言。所以他们不得不重写一遍，事实上，我在会见美国各地的高管时，也听到过几次这个话题。

这让我真的一头扎进去，试图弄清楚，这是一个多大的问题？有——我不会说出是哪家公司，但是有一家供应商确实提供了 PERL 工具包，例如，连接到他们的 API 并与之交互。问题是，PERL 是一种垂死的语言。所以，如果——现在，这个供应商也没有专门要求你使用 PERL，但是我发现这是一个奇怪的选择，他们甚至会支持 PERL，因为它越来越不受欢迎了。此外，去年夏天，也就是 2016 年夏天，我参加了一个厨师会议，他们宣布了他们新的 Habitat write 平台，它都是用 Rust 编写的。又是一种奇怪、晦涩的语言。

所以，艾伦，我所做的是，我四处寻找答案，为什么有些公司选择了一种语言，但五年后，却无法支持这种语言和他们内置的工具？事实证明，有几个流行指数，艾伦，你可以四处看看。有一个叫做编程语言的流行度，PYPL，它被列在 GitHub 上，它列出了所有的流行度——按流行度和份额排列的语言。还有 TIOBE 指数，有好几个指数。他们中的一些人编制索引，我想——是什么？有一个是由所谓的编码 Dojo 收集的，还有几个是去查看工作列表，看看工作列表中最流行的语言要求是什么。

如果你想更好地了解这个，实际上，你可以去 ZDNet.com。他们写了一篇很棒的文章，基本上采用了所有不同的索引，将它们汇总，放在一起。在他们的发现中有趣的是，你看不到 PERL，你看不到 Rust，你看不到很多这样的语言，我称之为博士语言，意思是那些将要获得计算机科学博士学位的人最终创造了这些语言，他们每个人都有自己的狂热追随者，Alan。这在一方面很酷，但在另一方面，这意味着如果您不小心，您可能会为了 DevOps 实践而使用某种编程语言，最终选择一种实际上不可持续的语言。这是一种趋势，你最终—

当然，斯科特，就 Rust 和我们遇到的其他一些更晦涩的语言而言，我明白你的意思，我不反对，但是你知道，当你看 PERL 时，你知道，你和我都不是 20 多岁的年轻人了，斯科特，对吗？我的意思是，PERL 已经存在很长时间了，在它的日子里，你知道，回到网络泡沫时期，你知道，PERL 相当于 PHP，你知道吗？最近我看到，我认为 PHP 已经过时了——PHP 5 或其他现在已经过时的东西。

但是，你知道，PERL——如果你在 1998 年问我赌哪种语言，我可能会赌 PERL。你知道，我不认为它会变得模糊或过时——尽管，你看，98 年，你知道，我们谈论的是 18，19 年，当然没有什么是永恒的。

但是外面的人怎么样——你知道，太难了。我们没有水晶球，对吧？我们确实有这些指数，但是我们怎么知道，比如，下一个热门的是什么，下一个无用的是什么？

**Wilson:** 这是一个很好的问题，Alan，请告诉我你所指出的故事——因为没有水晶球，这很难——我认为现在的整个 it 行业，包括编程语言，真的让我想起了 20 世纪之交，当时对汽车应该是什么样子还没有明确的定义。有各种各样的版本——爱迪生有他的电池驱动的汽车，有几个蒸汽机，有几个内燃机，有很多很多失败者，很多投资破产。

它花了，你知道，基本上几十年的时间，但是到了 20 世纪 20 年代，每个人都理解和知道汽车看起来像什么，它看起来像 T 型车，操作起来像 T 型车，对吗？

所以现在真的有点困难。因此，我喜欢做的，或者说我在我们这里谈论语言的背景下建议的，是真正看看它的趋势和它的寿命。如果你看那个，你知道，Java，C++是流行的。可能是这样的，C 已经有了，甚至 JavaScript 也已经有了几十年的持续发展势头。

你是对的——PERL 在当时是伟大的，但是根据流行指数，它是自 2001 年以来最受欢迎的，并且持续下降。PHP 一直坚守着自己的阵地，在综合指数中，它仍然被列为比较受欢迎的语言之一。

所以，你知道，我会——坦率地说，如果我在练习一门语言来帮助扩展我的 DevOps 实践，我肯定会看看过去几年——更确切地说是几十年——哪些东西是可持续的，并真正看看趋势。这就是为什么——听起来不是太挑剔，但当我看着 Chef 时，例如，将 Habitat 放在 Rust 上，对未来来说可能不是一个好主意。如果你看看在过去的 20 年里有哪些语言得到了持续发展，那么选择 Java 会更好，因为你有一个庞大的 Java 开发人员基础，然后你就可以对这个项目做出更大的贡献。所以我认为你可以这样做。

然后，你知道，另一件事，艾伦，你的观点——这样说很不性感，但有一种，我称之为准编程接口，它跨越了整个时间的考验，并保持一致。据我观察，它还超越了每个 it 专家，作为命令行界面或 CLI。我发现，在看这个的时候，非常有趣的是，比方说，使用编程语言扩展一些新的时尚 DevOps 工具的挑战之一是，它需要一个程序员，并且你必须理解该语言的语法。这通常会储存专业知识，而这些专业知识是有能力的，或者很容易就能掌握的。坦率地说，与命令行界面相比，它们是为易于理解和掌握的用例而设计的，它们超越了任何人，无论您是软件开发人员还是数据库管理员，每个人都能很快理解命令行界面的工作方式。

因此，如果我今天试图构建我的 DevOps 实践，我实际上会更热衷于使用命令行接口，已发布的，来建立我的许多扩展和自动化机制，只是因为它们超越了时间，并且在任何语言中具有最长的生命周期，尽管我不会说命令行接口是一种语言，它是一种接口，对吗？但好的一点是，它们大多是自文档化的，编程语言，你必须了解这种语言才能理解它在做什么，它们通常不是自文档化的。

**Shimel:** 明白了。Scott，我不想让你为难，但是如果可以的话，给我们的听众，你认为在可预见的将来，你最喜欢使用的三种语言。
