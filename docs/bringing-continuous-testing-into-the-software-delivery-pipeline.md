# 将连续测试引入软件交付管道

> 原文：<https://devops.com/bringing-continuous-testing-into-the-software-delivery-pipeline/>

对于任何依赖于平稳用户体验的高度竞争行业来说，其开发团队加速发布周期、降低运营成本并最终降低风险是至关重要的。持续测试是公司实现这些目标的一种方式。

作为软件测试的一个子集，连续测试(CT)是有价值的，因为它揭示并解决了开发过程中不同阶段的不同问题，这是其他方法所没有的。大多数其他测试方法需要大量的员工；CT 没有。CT 利用冲刺自动化，包括跨技术前端和后端的端到端测试，以削减成本和促进快速交付，而不会给内部团队增加更多压力。此外，作为一个更大的软件交付管道的一部分，CT 被定位在更接近开始的位置，以便尽早地、经常地和全面地执行测试，从而实现内聚的发布目标。

CT 引入了质量保证，在开发的每个阶段不断重复测试和可操作的反馈，以评估业务风险覆盖范围，并确保最佳的用户体验和客户满意度。顾名思义，CT 不仅在开发的早期阶段，而且在整个软件开发生命周期(SDLC)中持续地帮助开发人员和产品经理检测和修复关键错误，确保代码中的错误不会被无意地发布出去。

随着软件公司加快发布频率，持续测试为数字工具开发和质量保证过程开创了先例。

## 持续测试的好处

持续测试在 CI/CD 管道中扮演着不可或缺的角色，是 DevOps 最佳实践的一部分，它提供的信息对开发团队来说是无价的。CT 的好处也是深远的，对整个组织都有积极的影响，包括:

### 更好的全公司反馈

持续测试有助于向所有参与的团队和涉众提供反馈。例如，当在代码中检测到一个 bug 时，理想情况下，唯一被通知的团队应该是负责有 bug 的那部分代码的团队。如果没有 CT 自动检测 bug 并通知那些负责修复错误的人，对 bug 进行分类可能是整个团队的工作，或者更糟，是一个专门团队的主要职能。这种方法效率很低，浪费了代码没有错误的工程师的时间。CT 提供的反馈减少了这些无益的场景，并允许利益相关者做出即时和明智的决策。

### 更高质量的代码和用户体验

一个有效使用 CT 的公司应该期待更高质量的代码和更稳定的用户体验。[稳定性](https://www.bugsnag.com/research/app-stability-index-report)衡量应用健康和用户体验；它会受到崩溃等会话结束事件的负面影响。有了 CT，企业将让他们的开发团队更好地了解产品质量和性能，同时通过更快地修复损坏的应用程序，显著提高整体业务指标。

### CT 打破了孤立的团队

CT 不仅增强了质量保证，它同样通过分解孤立的团队来改进数字工具的开发。当团队变得孤立时，他们会不经意地将过程和知识留给自己，抵制跨职能的协作，并且孤立地朝着他们自己的目标工作。可以预见，不同的群体无法分享见解或想法，导致停滞和创新减少。然而，通过利用 CT，孤立的团队可以在开发、测试和操作中变得更加协作，从而提高整体业务效率。

### CT 支持客观评估

最后，CT 使组织能够停下来，客观地评估它的工作是否与其特定的业务目标相一致。如前所述，CT 使用自动化来提供关于业务风险的及时反馈，减少了数字应用程序或工具在发布时失败的机会。同样，开发团队和经理可以更容易地满足他们的业务目标，并优化一个版本的价值。此外，客观评估有助于关键决策，因为 CT 为决策者提供了可操作的数据来支持他们的选择。

今天，消费者已经开始期望他们的软件应用程序中没有错误的体验，这些应用程序中最稳定的应用程序推动了转化、参与和保留。将连续测试模型应用到软件开发管道中，最终将会使应用程序比以前交付得更快、质量更高，并增强最终用户体验。随着公司将敏捷开发与持续测试相结合，他们将优化开发生命周期的每个方面以及他们更大的业务目标。
