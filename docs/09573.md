# DevOps 不是一个角色，而是一种文化

> 原文：<https://devops.com/devops-is-not-a-role-its-a-culture/>

如今，全栈开发人员越来越少了。由于现代软件开发越来越复杂，程序员通常专注于特定的语言、框架或软件堆栈级别，如前端或后端。

这些相同的骨折似乎也发生在 DevOps 内部。如今，大公司不再有单一的 DevOps——通常有许多风格的快速部署和跨组织开发团队使用的各种 devo PS 工具链。DevOps 现在也要求对每个云服务提供商平台的细微差别进行专门培训，要求专家在单独的[云平台](https://nordicapis.com/5-cloud-computing-service-providers-for-saas-in-2021/)上接受培训，如 AWS、Azure 或 GCP。随着平台工程和[现场可靠性工程的成熟](https://devops.com/site-reliability-engineering-sre-comes-of-age-in-2022/)，这一角色正在扩展并呈现出新的形式。

因此，公司不应该将 DevOps 视为一个特定的角色或一套工具，而应该将其视为一种 T2 文化。我最近会见了 BOS Framework 的首席执行官和创始人 Sashank Purighalla，以收集对 DevOps 不断演变的角色和文化的见解。根据普里加拉的说法，“DevOps 是文化、实践、流程和工具的结合。”下面，我们将确定引入 DevOps 带来的挑战，并考虑完成工作所需的技术、人类专家和治理的正确组合。

## 实施 DevOps 的挑战

实施 DevOps 的一个挑战是简单地定义职位本身。“DevOps 并不是一份真正的工作，”普里加拉解释道。相反，它包含许多方面，如 [CI/CD](https://containerjournal.com/features/6-cncf-projects-for-ci-cd/) 、供应、维护和[合规性](https://containerjournal.com/features/6-cloud-native-tools-for-security-and-compliance/)，所有这些现在都采用了过多的不同工具。CloudOps 和 DevOps 之间也有重叠，这种模糊性使得为 DevOps 工程师编写工作描述变得很棘手。

各种角色组成了负责现代云原生运营的团队。就像程序员有特殊的专长一样，DevOps 也有专门的领域。这种细分可能基于云供应商套件或正在使用的工具，例如对 Jenkins、Docker 或 Kubernetes 的熟悉程度。特定的领域，如 DevSecOps 或 BizDevOps，可能也需要专门关注。Purighalla 分享了一些现代 CloudOps 团队中常见的角色示例:

*   云架构师
*   应用架构师
*   安全专家
*   合规专家
*   AWS 开发工程师
*   DevOps 工程师
*   AWS 的 SRE
*   蔚蓝的 SRE

更复杂的是，DevOps 可以超越基于云的系统，应用于内部或边缘网络维护。普里加拉说，考虑到许多可能性，为 DevOps 职位起草职位描述需要“对你希望如何在你的组织中建立 DevOps 职能的认识和愿景”。他说，不幸的是，大多数人不清楚从哪里开始。设定这一路线图将因行业而异，需要对公司及其技术资产有深入的了解。

## DevOps 不应该制造另一堵墙

DevOps 是一种更快速地交付高质量代码的机制——这一业务成果自其诞生以来一直是指导原则。为了实现这一目标，DevOps 试图将开发和运营放在“同一个房间”以减少摩擦。然而，在一些组织中，DevOps 作为一个全新的角色出现在组织结构图中。“在某些情况下，它创造了一堵墙，”普里加拉说。"当开发人员将代码交给基础设施团队时，他们不能想得太多."

如果这堵墙还在，它可能会进一步分裂团队动机。当然，建立一个健康的 DevOps 文化会因公司而异，需要明确最终目标。但是创建公共激励可以包括转移一些安全和供应责任，以便开发人员有更多的视角并可以参与开发运维。

接受 DevOps 文化的另一个原因是这个领域人才稀缺。这是各种规模的企业都面临的问题。普里加拉说，初创公司可能不够大，不足以聘请开发人员或网站可靠性工程师，但开始看到将可靠性和合规性作为核心业务功能的必要性。企业通常拥有多个云和混合环境，很难找到人才来支持。中型市场的公司经常输给创业公司的吸引力和大公司的大笔资金。

## 外包 DevOps:它曾经是一个可行的选择吗？

那么，考虑到寻找工程师是如此具有挑战性，外包 DevOps 是一个可行的选择吗？据普里加拉称，“如果你的公司有预建的云解决方案，并且能够为你提供类似的功能，而不需要全新的定制工作，那么外包是一个不错的选择。”云支持解决方案可用于降低准入门槛，但绝不能替代整个开发运维实践。

原因是软件编程是一个创造性的工程过程。“我从未见过一个工程师每次都做同样的事情。”人们继续以“构建和修复”的心态进行实验。这意味着，一般来说，雇佣外部工程师是很棘手的，因为他们并不熟悉手头的系统。

此外，随着后端基础设施的不断发展，采用标准方式与多家云提供商合作的平台必须灵活多变。因此，DevOps 必须被视为更加持续的东西，需要持续的维护。这不是外包一次就能解决的事情。

“这变得像会计——如果你不这样做，你将无法通过审计，”普里加拉说。“我们需要这样看待外包。您需要引入一个与专家相结合的云支持解决方案。不能非此即彼。”

## DevOps:不仅仅是一个角色或工具集

许多人认为 DevOps 只是一套工具。相反，一些专家认为 DevOps 更多的是不同角色、工具和过程的组合。“DevOps 一开始是一个模型——它不是一个工具，”普里加拉重申。

本质上，它更多的是一种文化，而不是一个具体的职位名称。从这个角度来看，组织可以开始消除孤岛，这是 DevOps 首先应该消除的。凭借正确的技术和协作精神，公司可以创造一种文化，满足 DevOps 旨在实现的效率目标。
