# DevOps 神话被揭穿

> 原文:[https://devops.com/devops-myths-debunked/](https://devops.com/devops-myths-debunked/)

领导者不必是 DevOps 专家，但他们需要区分 DevOps 神话和现实，以领导他们的数字化转型项目。

正如我在最近的一篇文章中指出的，[工程实践可以克服 DevOps 的挑战，](https://devops.com/engineering-practices-can-overcome-devops-challenges/)领导者需要为组织设定一个鼓舞人心的 DevOps 方向的愿景，并积极地激励和赞助团队活动以实现目标。为了可信，来自领导者的沟通需要区分神话和现实。领导者很容易感到困惑，因为围绕 DevOps 有许多流行的神话。重要的是要记住，在开发运维中，开发运维的定义和工程最佳实践还没有完全确定下来。

正如战略合作伙伴高级总监 Doug Miller 所说, [Perforce Software](https://www.perforce.com/) 表示,“在一个尚未确定 DevOps 公认‘定义’的行业工作，每天都是一个挑战。不断变化的环境使得技术供应商和消费者很难确保我们都在以同样的方式谈论同样的事情。”

以下是领导者需要了解的一些最重要的误区。

## 揭穿 DevOps 神话

神话:“DevOps 是一场文化运动。”

事实是，实施 DevOps 需要一种谨慎而平衡的工程方法，要考虑到广泛的人员、流程和技术方面。在我的文章中，[DevOps 的工程方法的时候到了，](https://devops.com/the-time-has-come-for-an-engineering-approach-to-devops/)我解释了虽然文化是一个关键的成功因素，但 devo PS 不仅仅是文化。此外，DevOps 不仅仅是一项运动。在软件工程历史的大背景下,“DevOps”这个词是新的，也许有一天它会被另一个词取代，但是构成 DevOps 的底层工程原则和实践可以适用于几乎所有的工程项目。

**谬论:** **“开发运维需要持续部署。”**

事实是，DevOps 支持将发布工件连续交付到准备安全部署的阶段环境，但是频繁部署到生产环境并不总是一个目标。有些有效的客户使用案例不希望频繁部署到生产环境中。例如，客户可能不希望经常收到软件平台产品的新版本，因为每个新版本都可能因为超出产品本身范围的原因而具有破坏性和风险。在不部署到生产环境的情况下，将完整版本交付到试运行环境的工程优势仍然相当可观。每个版本都是在生产前环境中生产和验证的，并且随时准备按需部署，不会干扰不想使用新版本的客户。

**误区:****“devo PS 适用于任何软件产品、服务或应用。”**

事实是，DevOps 工程在许多情况下会产生很高的投资回报率，但有些情况下并不保证对 DevOps 所需的人员、流程和技术变革进行投资。例如，很少更改的应用程序不保证 DevOps 更改的成本。高度孤岛化的组织会发现实施 DevOps 太令人沮丧且适得其反。将 DevOps 应用于一些商业现货(COTS)产品可能不会产生足够的 ROI，因为企业对企业的壁垒可能表现为难以逾越的组织孤岛。工具链中几乎没有适合开发运维的工具的应用可能会发现向开发运维的过渡过于昂贵。

神话:“DevOps 需要敏捷。”

事实是 DevOps 和敏捷可以是互补的，也可以是竞争的。敏捷和开发人员都有处理小的增量变更的概念。然而，采用敏捷的组织专注于管道开发端的优化，可能会与强调端到端流程优化工程的 DevOps 相冲突。

**误区:“DevOps 的作用域与 CI/CD 相同。”**

事实是，一个设计良好的 DevOps 实施包含持续集成/持续交付(CI/CD)，但也扩展到集成和交付之外的其他实践领域，如规划、运营、治理、灾难恢复和其他必须与 CI/CD 一起工作以实现业务目标的实践。

**误区:“DevOps 不适用于平台产品，如嵌入在制造产品中的软件代码。”**

事实是，DevOps 确实适用于嵌入式代码产品，以及 web 服务、业务应用程序和其他类型的产品。它甚至适用于 DevOps 本身的软件和服务开发！在平台产品的情况下，部署通常是有计划的。它们不是连续的，而是，候选版本的连续交付被交付到一个阶段化环境，以确保在平台构建时为平台实现的变更被充分开发和验证，而不是等待一个大爆炸式的发布。

误区:**“开发运维是对 IT 运营的消除。”**

Jeremy Bosch 在他的文章[中揭穿了 DevOps 的 9 大神话](https://dzone.com/articles/top-9-debunked-devops-myths)，“根据 NoOps 的概念，IT 部门将变得如此自动化，以至于不需要任何内部团队来管理软件。然而，无论服务变得多么自动化，总会有运营的需要。虽然运营可能会有变化，但这些变化在 DevOps 中仍然具有重要意义。”

关于 DevOps 有许多持续的神话，比如本文中的例子，它们会迷惑和误导不知情的领导者。学会区分 DevOps 神话和现实的领导者能够更好地传达现实目标，以领导他们的数字化转型项目。
