# 将安全性融入开发运维转型的 12 种方式

> 原文：<https://devops.com/12-ways-to-bake-security-into-a-devops-transformation/>

安全性已经成为任何开发运维转型不可或缺的一部分。根据 [Upskilling 2021:企业 DevOps 技能报告](https://info.devopsinstitute.com/2021-upskilling-report-download?utm_campaign=Upskilling%25202021&utm_source=Devops-dot-com-12-ways-to-bake-security-practices-into-DevOps%2520transformation) ， [DevSecOps](https://devops.com/?s=DevSecOps) 在自动化工具类别中取得了 56%的必备百分比投票。安全性不仅保护企业及其客户，还使公司能够实施内部和外部策略。

在考虑如何将安全实践转化为 DevOps 转型时，我邀请了几位演讲者和即将到来的[](https://devopsinstitute.com/dso-2021/?utm_campaign=SKILup%2520Day-DevSecOps%2520August%25202021&utm_source=DevOps-dot-com&utm_term=12-ways-to-bake-security-practices-into-a-DevOps-transformation)滑雪日的赞助商以及几位 DevOps 协会大使来分享他们的想法。以下是他们分享的:

**赞助商，** [**肯德尔米勒**](https://www.linkedin.com/in/kendallamiller/) **，总裁，费尔温兹**

“DevOps 转型是关于构建优秀的工具，以便开发人员可以在生产过程中一直拥有他们的服务。安全的增加是一回事；它正在构建优秀的工具，这样开发人员也可以拥有他们的服务的安全姿态，一直到产品化。这种转变是为了让开发人员能够理解应用程序级别的安全问题。服务所有权仍然是服务所有权，即运营和安全部分的所有权。”

**赞助商，**[**Guy Eisenkot**](https://www.linkedin.com/in/guy-eisen-3012a597/)**，产品副总裁，Bridgecrew by Prisma Cloud**

“将安全性放在首位的强大 DevOps 转型的关键是选择和利用有助于您的特定堆栈实现自动化的工具和流程。对于 AppSec 和云安全来说，自动化是游戏的名字。但是自动化的强大取决于它能融入的流程和它提供的反馈。

如果没有可操作性，没有在正确的时间或正确的地点给出，安全反馈将是无用的。当经历 DevOps 转换时，在实现工具和流程时记住这一点很重要。同样重要的是，出现的反馈是可操作的和有用的，以便工程师可以实际学习和实现安全的编码实践。"

**发起人，**[**Rob cud dy**](https://www.linkedin.com/in/rcuddy/)**，全球应用安全布道者，HCL 软件 DevOps**

“最好从设计开始，确保你的史诗、故事、希尔声明和用例都包含安全元素。这里的一个好主意是制作“误用用例”或对抗性用例，然后对照它们进行测试。  例如——误用案例可能是这样的，“作为一个不良参与者，我可以使用一个格式错误的 URL 来获得对 web 服务器的未授权访问”，或者“作为一个不良参与者，我可以在事务中扮演一个端点，并且在扮演的同时发送和接收数据，而其他方对此一无所知。'

拥有优秀的 SCA——软件组成分析也是至关重要的。您无法保护您不知道自己拥有的东西，这包括开源和第三方工具。在很多情况下，简单的代码共享可能会无意中导致添加许多新的依赖项，这些依赖项可能会增加新的漏洞。

在整个管道中对安全方面进行测试和验证是很重要的，尤其是在管道中还进行其他检查的任何点。结合这一点的一个很好的地方是在 QA 和任何正在进行功能测试的地方。  在这样的环境中利用交互式安全测试(IAST ),可以在测试应用程序功能的同时进行监控并发现漏洞。  这大大减少了误报的出现，并提供了关于应用程序安全状况的实时信息。"

**发起人，** [**亚瑟·富恩特斯**](https://www.linkedin.com/in/yasser-fuentes-cissp-cism-38395b71/) **，云工作负载安全技术产品经理，Bitdefender**

“简而言之，安全性不应仅在 CI、开发和测试阶段考虑，也不应仅限于漏洞评估，因为 DevOps 团队目前用于检查 CI/CD 管道的大多数漏洞扫描器和评估工具都依赖于已知的漏洞、签名和已知的攻击技术。作为一项建议，我们应该强调整合工具或解决方案的重要性，这些工具或解决方案允许开发人员在运行时级别识别潜在的威胁和安全缺陷，并在典型的反恶意软件警报之外提供对可疑和潜在异常的流程相关行为的可见性。”

**，** [**乔尼·克利珀特**](https://www.linkedin.com/in/joniklippert/) **，联合创始人兼 CEO，****stack hawk**

“要将安全实践融入开发运维转型，团队需要重新想象安全在软件开发生命周期中的作用。对于今天的许多组织来说，安全测试阻止应用程序进入生产环境，或者试图在软件发布后赶上进度。但不一定非要这样。

安全性可以(也应该)贯穿于软件开发的生产前阶段。现代安全工具在 IDE 或 CI/CD 管道中自动进行安全测试，如果工程师引入了漏洞或易受攻击的依赖项，就会向他们发出警报。

使安全性成为在生产前运行的另一种测试意味着团队可以更快地发现漏洞，即时进行补救，最重要的是，回到功能开发。将安全性融入 DevOps 管道是所有团队的目标！"

[**斯蒂芬·沃尔特斯**](https://www.linkedin.com/in/1stephenwalters/) **，销售工程师，光大桥**

“与任何其他实践成为 DevOps 一部分的方式完全相同，遵循 CALMS 原则。实施包容和接受的文化；将自动化集成到安全测试、监控、构建和存储库管理中；确保交付和支持中的安全流程和反馈以精益的方式执行，增加价值而不是浪费；通过使用记分卡将安全性作为业务价值和业务成功标准进行衡量；分享关注点、经验教训、新实践和技术。”

[**Dheeraj Nayal**](https://www.linkedin.com/in/dheerajnayal/)**，** g **全球社区大使&亚太、中东和非洲区域负责人，DevOps Institute**

“积极主动的方法是构建 DevSecOps 要素的最佳方法。这可以通过遵循一些最佳实践来实现。

1.  团队应该保持一致和同步
    实现 DevSecOps 文化的第一步也是最重要的一步是培训你的团队，安全是所有三个系统的团队的共同责任。一旦开发和操作单元承担起保护代码和基础设施的分布式责任，DevSecOps 就成为开发周期的一个常规部分。
2.  **保持代码简单** 复杂的代码会携带更多的安全漏洞。简单的代码协作起来更舒服。所有的开发人员都应该能够看到彼此的代码，并且知道发生了什么。
3.  **测试** SAST(静态应用程序安全测试)是一个重要的安全工具，可以在开发过程的早期定期检查安全问题，使 it 部门能够更有效地解决问题。
4.  **部署** 为了加快产品交付并增加开发过程的灵活性，自动化部署非常有用。人们可以检查整个 IT 基础设施的属性，并在采用基础设施即代码工具的系统中实现安全配置。
5.  **运营**  运营团队必须与安全从业人员勾结。他们负责控制基础设施和网络安排。运营和安全团队联合起来建立手动和自动安全测试，以保证符合网络配置。"

[**彼得·马迪森**](https://www.linkedin.com/in/peter-maddison/) **，Xodiac 公司创始人**

“两个主要的地方:在平台工程内，收集您创建可见性、反馈和学习所需的数据。接下来，重要的是让安全和法规遵从性团队参与进来，以确定您可以在哪里集成到现有标准中，或者可能需要在哪里进行修改。”

[**马克·彼得斯**](https://www.linkedin.com/in/tinycyber/) **，技术负责人，诺维塔**

“安全实践可以通过标准开发运维方法、人员、流程或技术融入其中。在实践中与人一起烘焙意味着让你的团队中能够倡导安全的人参与进来。这可以通过让您的安全专家花时间与团队在一起、指派安全人员负责团队，甚至通过在那些定期与安全人员互动的团队中选择冠军来实现。毕竟，第一个敏捷原则是“人高于过程”

这一组成部分不允许人们忽视这一进程。构建正确的过程意味着从一开始就使用测试驱动开发。每次交付都应该进行测试，这些测试应该包括安全性测试。一些常见的安全测试，如代码质量，现在发生得如此频繁，以至于许多人不再认为它是一种独立的安全实践；仅仅是发展的一个常规部分。

最后，人们可以通过技术烘焙过程。许多安全工具允许通过仪表板上传递的指标、日志和跟踪信息走捷径。这些工具评估并突出安全挑战发生的位置。工具允许通过基础设施进行集成，并保持对何时发生什么功能的了解。一个古老的军事战略使用了约翰·伯伊德的 OODA 循环:观察、定位、决定和行动。就像军队中的情报系统一样，通过仪表板更好地了解您的系统可以让您观察网络中的功能，针对可能的问题，决定解决方案并在这些问题达到临界质量之前采取行动。"

[**supra tip baner JEE**](https://www.linkedin.com/in/supratipb/)**，解决方案架构师，全球服务负责人**

“有几种方法可以将安全性融入开发运维中:

1.  **安全是一种心态**:开发过程中的安全可能会减缓开发过程，这是一个思考过程。但是我们必须教育团队，在过程的早期修复安全缺陷所花费的时间和精力要比以后做的少。
2.  **平台/环境:**部署平台/环境的适当配置/卫生极其重要。基础设施即代码(IaC)、配置即代码、合规性/策略即代码、机密管理等。可以帮助这个过程。容器和云平台的使用增加可以保持 it 整洁，因为底层基础设施由提供商管理。
3.  **自动化:**在 CI/CD 管道中包含大量的自动化测试。还增加了静态代码分析、第三方测试、AVA 测试、代码漏洞检查、容器安全、入侵检测与防御、APM 等。自动化过程。进行变更管理、审计等。有规律，尽可能自我操作。
4.  **安全架构:**确保在软件架构/设计阶段启用安全性。架构风险评估必须是一个强制性的过程，以便在开发开始之前发现任何设计漏洞。
5.  好奇的文化:创造一种强调安全重要性的文化，这样每个人都有兴趣并随时准备发现他们系统中的安全是否出了问题。"

[**Amiran Alavidze**](https://www.linkedin.com/in/aalavidze/)**，安全与风险管理总监，Tasktop**

“DevSecOps 是关于文化和授权，并使软件交付团队自治和高效。培训和认知是这种支持的核心组成部分。这还包括专注于开发人员友好和自动化的工具，以便关键的安全控制成为如何完成工作的不可分割的一部分，并允许开发团队承担所有责任，而没有对安全领域的专家级理解的认知开销。”

[**沙拉思·多达**](https://www.linkedin.com/in/sharathdodda/) **，IT 开发经理，TD**

“通过在开发运维的早期尽可能自动化地实施安全性。所有可能的和相关的扫描都应该在管道中提前做好，以便清除工件中的所有发现。”
