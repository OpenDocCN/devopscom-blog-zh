# 连续测试与测试自动化:3 个关键区别

> 原文:[https://devops.com/continuous-testing-vs-test-automation/](https://devops.com/continuous-testing-vs-test-automation/)

测试人员已经与测试自动化搏斗了多年，然而大多数团队对他们当前的测试自动化水平或者维护它所需的开销并不满意。此外，过去几年中，应用程序的设计、开发和使用方式发生了巨大变化，增加了测试的复杂性和软件故障的业务影响。

鉴于现代应用程序交付日益增长的复杂性和速度，软件测试专业人员如何帮助企业控制风险？进入连续测试。

## 什么是持续测试？

持续测试是[执行自动化测试](https://www.tricentis.com/what-is-continuous-testing/)的过程，作为软件交付管道的一部分，以尽快获得与候选软件发布相关的业务风险的反馈。

测试自动化旨在产生一组与用户故事或应用程序需求相关的通过/失败数据点。另一方面，持续测试关注业务风险，并提供软件是否可以发布的洞察力。为了实现这种转变，我们需要停止问“我们完成测试了吗？”取而代之的是，集中精力于“发布候选具有可接受的商业风险水平吗？”

## 为什么我们需要持续测试？

今天，整个行业的变化对测试提出了更高的要求，同时使得测试自动化更难实现(至少使用传统的工具和方法):

*   应用架构越来越分散和复杂，包括云、API、微服务等。，并在单个业务交易中创建不同协议和技术的几乎无穷无尽的组合。
*   由于敏捷、DevOps 和持续交付，许多应用程序现在每两周发布一次，甚至一天发布数千次。结果，可用于测试设计、维护，尤其是执行的时间急剧减少。
*   既然软件是业务的主要接口，那么应用程序故障就是业务故障——如果影响用户体验，即使是看似很小的故障也会产生严重的后果。因此，与应用程序相关的风险甚至已经成为非技术型企业领导的主要担忧。

## 持续测试和测试自动化有什么不同？

持续测试和测试自动化之间的主要区别可以分为三大类:风险、广度和时间。

### 风险

如今，企业不仅向最终用户公开了许多内部应用程序，还开发了大量扩展和补充这些应用程序的附加软件。例如，航空公司已经不仅仅是公开他们曾经的内部订票系统。他们现在让客户计划和预订完整的假期，包括酒店、租车和活动。向用户展示越来越多的创新功能现在是一个竞争优势，但它也增加了潜在故障点的数量、种类和复杂性。

大规模的“软件故障”具有如此严重的业务影响，以至于与应用程序相关的风险现在是企业公开财务申报的重要组成部分。鉴于[显著的软件故障导致股价平均下跌 4.06%](https://www.tricentis.com/resource-assets/software-fail-watch-2016/)(这相当于平均损失 25.5 亿美元的市值)，业务领导者注意到这一点并期待 it 领导者采取行动就不足为奇了。

如果您的测试用例没有考虑到业务风险，您的测试结果将不会提供评估风险所需的洞察力。大多数测试被设计来提供关于用户故事是否正确地实现了需求的低级细节——而不是一个发布候选是否风险太大而不能发布的高级评估。你会根据测试结果立即停止发布吗？如果没有，您的测试就没有与业务风险恰当地结合。

需要澄清的是:我们并不是说低粒度测试没有价值；我们声明需要更多的措施来阻止高风险的候选人进入野外。

### 宽度

即使一家企业设法避开了成为头条新闻的大规模软件故障，如今即使看似很小的故障也可能造成麻烦。如果用户体验的任何一部分未能满足他的期望，你不仅面临失去竞争对手的客户的风险，如果用户决定将问题提交到社交媒体，你还面临品牌受损的风险。

仅仅知道一个单元测试失败了或者一个 UI 测试通过了，并不能告诉你整体用户体验是否受到了最近应用程序变化的影响。为了保护最终用户体验，您需要足够广泛的测试来检测应用程序更改何时会无意中影响用户已经开始依赖的功能。

### 时间

既然组织交付软件的速度已经成为一个竞争优势，绝大多数组织都转向敏捷和开发运维来加速他们的交付过程。

当自动化测试出现时，它专注于测试根据瀑布开发过程构建和更新的内部系统。系统都在组织的控制之下，在测试阶段准备开始时，一切都已完成并准备好进行测试。既然敏捷过程正在成为规范，测试必须与开发并行开始；否则，用户故事不太可能在极度压缩的迭代时间框架内(通常是两周)被测试并被认为“完成了”。

如果您的组织已经采用了 DevOps 并且正在执行连续交付，软件可能每小时发布一次——甚至更频繁。在这种情况下，过程中每个阶段的反馈都不能只是“快”；它必须几乎是瞬间的。如果质量不是您的应用程序最关心的问题(例如，如果在产品中发现缺陷时进行回滚的影响很小)，那么在每个版本上运行一些快速的单元测试和冒烟测试就足够了。然而，如果企业想要最小化错误软件到达最终用户的风险，您需要一些方法来实现必要的风险覆盖水平和测试广度——快速。

对于测试，有几个重要的影响:

*   测试必须成为开发过程的组成部分(而不是开发完成后附加的“卫生”任务)
*   相关功能一实现，测试就必须准备好运行
*   组织必须有一种方法来确定在交付管道的不同阶段执行的正确测试(签入时的冒烟测试、集成后的 API/消息层测试、系统级的端到端测试等等)
*   每组测试必须足够快地执行，以便不会在软件交付管道的相关阶段产生瓶颈
*   需要一种稳定测试环境的方法来防止频繁的更改导致大量的误报

## 持续测试>测试自动化

如果你只从这篇文章中得到一个想法，我们希望是这个:

测试自动化≠连续测试

持续测试>测试自动化

当他们的组织采用现代的架构和交付方法时，即使是使用传统的测试自动化工具取得了相当程度的成功的团队也会遇到严重的障碍:

*   他们不能足够快或足够频繁地创建和执行真实的测试
*   不断变化的应用程序会导致大量的误报，并且需要永无止境的测试维护
*   他们不能提供即时的洞察力来判断候选版本是否风险太大而无法通过交付管道

重要的是要认识到，没有任何工具或技术可以立即“给予”您持续的测试。像敏捷和 DevOps 一样，持续测试需要人员、过程和技术的变化。然而，当你的技术不能胜任这项任务时，试图在人员和流程上发起相关的改变，从一开始就是一场艰苦的战斗，而且最终会失败。

如果您的组织正在开始或扩展您的持续测试自动化工作，我们鼓励您查看由 [Forrester](https://www.tricentis.com/resource-assets/continuous-testing-tool-comparison/) 和 [Gartner](https://www.tricentis.com/resource-assets/software-testing-tool-comparison/) 所做的两篇新研究。这两份报告都提供了对持续测试和测试自动化趋势的洞察，以及顶级持续测试工具之间的比较。
