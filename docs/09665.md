# 构建 DevOps 演进平台，第一部分

> 原文：<https://devops.com/building-a-platform-for-devops-evolution-part-one/>

温斯顿·丘吉尔曾经说过，“改善就是改变；完美就是要经常改变。”虽然几乎每个人都承认变革是成长和成功的必要条件，但我们都知道这要困难得多，尤其是当涉及到人类行为的时候。在 DevOps 环境中工作的专业人员对此再清楚不过了。

DevOps 原则和行为的采用应被视为一个渐进的过程。许多组织一开始就搞砸了，或者试图实施变革，结果却失败了。有时这是由于领导层、市场条件或技术的变化，但随后组织会重新振作起来，建立势头，重新加速其发展。

### DevOps 可以改变

有了共同的愿景、共同的目标和努力的工作，DevOps 可以改变以跟上用户的需求。DevOps 专业人员将永远面临两个强大的障碍:文化和流程债务和技术债务。

当领导者拒绝授权、不信任他们的团队以及缺乏价值流工作方法时，就会产生文化和流程债务。当整体系统紧密耦合，测试、环境和版本控制的自动化被延迟，并且无法访问云技术时，技术债务就会产生。

尽管这些障碍既具有挑战性又很复杂，但许多组织仍然会正面应对。进化依赖于每天通过使用改进形式进行改进的习惯，如 [PDCA](https://www.plutora.com/blog/plan-do-check-act-understanding-and-applying-pdca?utm_source=ebook_release_manager) (计划、执行、检查、行动)，这涉及长期战略和目标以及当前和下一个目标状态之间的持续实验。

一个不断发展的 DevOps 方法在一个由成百上千甚至上万人组成的大型组织中维护移动的部分，这些人不可能在一夜之间改变。许多人缺乏接受和实施变革的意愿和认知能力。

这种环境要求发布经理下放权力，从集中指挥者转变为推动者，理解不同的价值流具有不同的技能和需求。马克·彼得斯(Mark Peters)是一名网络安全专家，也是 DevOps 的领导者，拥有多年的军事行动经验，他在 DevOps 的工作方式中概述了他的主要发布管理特征，如自动化流程、分布式责任和加速交付。

### PDCA 改进卡塔

DevOps 建议我们在日常工作中采用一种改进型式，作为一种改进习惯，包括四个重复的步骤:

1.设定长期愿景
2。了解当前状态
3。确定下一个目标州
4。各州之间的 PDCA

实验是 PDCA 循环的核心。最实用的实验是经验性的、数据驱动的和特定日期的。使用价值流管理平台来提供数据以支持实验，从而引发关于真实且可衡量的成功的数据驱动的讨论。

在本系列的下一部分中，我们将开始探索 DevOps 组织可以进行的不同实验，以驱动变化并改进他们的发布管理过程。
