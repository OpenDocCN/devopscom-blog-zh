# 时间和复杂性；或者说，为什么有些事情很难做？

> 原文：<https://devops.com/time-and-complexity-things-hard/>

为什么有些事情很难做？如果我们想让复杂的任务变得简单，这是 DevOps 的目标之一，我们需要有一些理论来解释为什么它们首先是困难的。

数学家兼游戏设计师理查德·加菲尔德提出了两个让任务变得更加困难的因素:[时间限制和复杂性](https://www.youtube.com/watch?v=av5Hf7uOu-o&t=7m11s)。我们解决问题的能力取决于我们在这个问题上花费的时间和问题的复杂程度。

## 时间和难度

DevOps 的讨论通常侧重于时间这一相关维度。商业价值是用时间来衡量的:上市时间、恢复时间和变更的前置时间都是 DevOps 的重要指标。在更短的时间内做同样多的工作更好。但是对时间的强调也会产生误导。为一项任务分配更少的时间并不等同于真正的更快。

那么是什么让事情花费更多或更少的时间呢？一项任务的部分难度与完成这项任务的人有关。配置新服务器对管理员来说可能很简单，但对应用程序开发人员来说却很困难，对图形设计人员来说则不可能。雇佣优秀的人才，并给他们分配合适的任务，可以降低任务的难度，减少完成任务所需的时间。

## 复杂性和难度

但是不管是谁在做，要做的任务都是一样的。规范是客观存在的，不属于任何从事这项工作的人。任务难度的客观部分是它的复杂性。

时间和复杂性之间的相似性在理查德·加菲尔德提出的思想实验中显而易见。想象你站在两扇门前。一个背后有龙，一个有宝。每扇门上都印着一道数学题，揭示门里是藏着龙还是宝藏。如果你有很短的时间来决定，在被迫猜测哪扇门是安全的之前，你只能解决一个相当简单的问题。同样，如果你有很多时间，但必须解决一个非常复杂的问题，你将再次沦为猜测。时间不足或过于复杂都会迫使一个人在已知答案存在的情况下进行猜测。

## 自动化和复杂性

DevOps 通常旨在通过 [CAMS](http://www.slideshare.net/matthewskelton/what-does-devops-culture-feel-like) 定义的“自动化”部分来降低复杂性。现在我们已经更广泛地研究了这个问题，很明显自动化是“降低复杂性的事物”的一个子集在自动化之前，或者如果您的组织的自动化工作产生了令人失望的结果，一个好的检查将是询问“这将减少部署代码的复杂性吗？”

在不了解自动化内容/方式的情况下进行自动化将会降低复杂性，但不会降低复杂性。一旦你找到了推最新版本的人，并且忘记了提及你需要改变的三个小的配置设置，你将能够非常快速地构建新的版本。配置偏差或自动化的局部重新引入了复杂性。没有可见性和监控，这些问题就像无人照管的水果上的霉菌一样在自动化系统上滋生。

这并不是说自动化不好。当在具有测试和可见性的[成熟环境](/blog/228391/editor/www.scriptrock.com/blog/configuration-automation-maturity-model)中完成时，自动化是降低复杂性的最佳工具。但是如果没有独立的验证和监控，您会发现自动化也可能是复杂性的最大来源之一。
