# 为测试自动化制定一个计划

> 原文:[https://devops.com/make-a-plan-for-test-automation/](https://devops.com/make-a-plan-for-test-automation/)

我们能够做的测试数量是巨大的。我们中的大多数人并不关注跨领域和全面的测试，但是让我们先这样做一下。

我们有代码级别的功能测试，这已经是相当好的自动化了，如果你的商店正在使用这种自动化的话。然后我们有集成测试，它是 API 测试、代码集成测试、可用性和回归测试的超集集合。集成测试是自动化的，有时做得很好，但是这个巨大的空间还在增长。接下来是安全性测试，从严格的安全性角度来看，它涵盖了其他测试感兴趣的大部分代码。然后是实际的可用性测试，这对于某些领域来说比其他领域更重要——例如，嵌入式应用比 web 应用需要更少的可用性测试——并且在这个时间点上具有不同的自动化水平。

真的，仔细想想，这个环境的复杂性是巨大的。我们不仅需要从一系列不同的角度测试一系列不同的功能，还必须在一系列平台上进行测试。在 Windows 上测试不同于在 Linux 上测试；在云中测试可以使用不同于内部测试的工具；客户可以在任何给定的平台上，这取决于应用程序和目标市场…在这方面，我们真的有一头猛兽。

而测试人员一直落后，或者说没有把东西覆盖好。这不是对测试人员的指责，这是手边的复杂性和事实的结果，在许多情况下，一行代码可能以一堆不同的方式出错。

进入自动化领域，它在持续集成(CI)领域发展迅速。自动化测试工具的范围从宽泛到非常集中，都有助于促进测试。开发中 DevOps 的增加意味着更多的测试，而测试已经落后了。这一点在过去已经讨论得相当透彻，但最终感觉我们已经找到了解决问题的可行方案。如果测试人员成为审查自动化测试结果的测试开发人员，并且通过 CI，那些自动化测试越来越多地进入开发过程，我们将会看到测试人员获得更好的覆盖率和质量，看到持续的增长。

但是这确实意味着你应该继续关注[市场](https://www.gartner.com/en/documents/3975493/magic-quadrant-for-software-test-automation),因为它在继续增长，并且选择那些提供对你的测试团队最有用的功能的工具。让它们运转起来。更多的测试意味着更多的设置测试的工作。即使是 UI 的无代码测试解决方案也需要有人来记录完成测试所必需的动作，这需要花费一些时间。但是回报是不需要人的长期测试，直到自动化发出危险信号。

我完全赞成。测试和安全都是自动化的最大受益者，因为它们在历史上是最落后的，而抄近路让组织付出了沉重的代价。所以，让我们开始吧；查看一些新的测试工具(有很多 listware 文章给你一个开始的地方)，让我们把伟大的软件做得更好！
