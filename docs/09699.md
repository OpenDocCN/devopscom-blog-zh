# DevOps 云安全中缺失的一环

> 原文：<https://devops.com/the-missing-link-in-devops-cloud-security/>

我们都看到了最新的[威瑞森数据泄露事件报告](https://www.verizon.com/business/solutions/secure-your-business/business-security-tips/?cmp=knc:ggl:ac:wls:dpr:8888855284&utm_term=verizon%20dbir&utm_medium=cpc&utm_source=google&utm_campaign=GGL_BND_Security_Phrase&utm_content=DBIR2022&ds_cid=71700000082349844&gclid=CMX51oLIwfkCFZ1GHQkdLokKUw&gclsrc=ds)中的数据，该报告显示一半的安全漏洞源于凭据滥用。很明显，凭据泄露是网络安全中的一种流行病，可以通过多因素身份验证和最低权限策略在很大程度上避免。根据威瑞森的数据，在过去的五年里，被滥用的证书数量上升了近 30%。工程师和开发人员也不能幸免于这种瘟疫；它们可能对攻击者更有吸引力，因为管理员权限允许的访问权限远远超出了普通用户的权限。事实上，他们经常被授予始终在线的访问权限，而实际上，他们只需要短暂、间歇的访问权限就可以完成工作。据 Gartner 称，特权访问会带来巨大的风险。即使有了 [PAM 工具](https://devops.com/3-reasons-why-ephemeral-access-is-best-for-devops/)，拥有长期特权的用户带来的剩余风险仍然很高。他们建议身份和访问管理(IAM)领导者实施准时制(JIT)策略，并最终实现零特权。

当这些凭证用于维护生产资源或对生产资源进行更改时，给予工程师适当的权限，使其仅具有执行特定任务所需的特权，这一点尤为重要。

### 对开发者来说正是时候

几十年来，JIT 库存管理已经成为许多行业的标准。特别是在制造业，工厂不能让供应品闲置在仓库里，装配线闲置，或者在等待零件时生产停顿。在信息安全中，JIT 管理只在用户完成任务所需的时间内授予用户特权。这有助于最大限度地减少攻击面，并避免长期未撤销的权限带来的风险。

在当今的敏捷环境中，开发工程师需要快速移动和修复事物。例如，云基础设施环境通常由开发人员和开发团队通过持续集成/持续交付(CI/CD)管道来建立和维护。但是开发人员有时也需要直接、高度特权地访问生产环境来完成更具体的任务。

虽然允许开发人员执行日常任务的静态权限通常是通过网络的 IAM 系统提供的，但组织通常缺乏为云环境动态授予和撤销权限的能力。这是因为传统的本地特权访问管理(PAM)和身份治理和管理(IGA)解决方案不支持云原生基础架构，也不支持越来越多的云计算基础架构。

开发人员总是需要快速、高权限地访问敏感的云环境，以进行调试或手动部署服务等活动。但是，这些广泛的权利不一定是永久性的，如果在工作完成后不取消，就会带来巨大的风险。

这使得组织求助于高风险、高特权的实践。不活跃的账户是许多网络安全问题的根源。事实上，在去年切断天然气供应的勒索软件攻击中，网络犯罪分子就是这样攻破 Colonial Pipeline 网络的。特权过高的账户是另一个令人头疼的问题，这是现在著名的网络安全管理软件产品袭击事件的一个重要因素。

### JIT 访问

这就是 JIT 访问发挥作用的地方，它可以对云中的特权访问实施高度细粒度的权限管理。Gartner 已经成为 JIT 的倡导者，因为它实现了许多安全最佳实践。他们最近建议安全负责人实施“一个快速简便地请求和授予额外权限的流程，同时尽量减少对个人工作流程的干扰。”

实现 JIT 权限管理的最大挑战之一是供应。安全人员负担过重，尤其是在采用多种云环境不断增加其工作复杂性的情况下。当有如此多的任务需要监督时，为每项任务提供单独的临时权限变得很繁重。

面向云的 JIT 权限管理必须满足开发人员对速度、自给自足和透明性的需求。一种方法是使用自助服务门户，接受提升权限的请求，管理审批流程，授予授权权限，并在权限到期时自动撤销。

### 开发人员安全性的 JIT 框架

该框架提供了安全性和生产力优势。

首先，它通过强制实施临时的细粒度最低权限策略，最大限度地减少了云攻击面。这可以防止已经泄露用户凭据的攻击者使用过多权限来访问敏感资源和泄漏数据。

其次，它使工程团队能够快速提交请求、通知审批者并获得临时访问权，从而节省了他们的时间和精力。这种自动化的工作流程消除了延迟，使 DevOps 工作人员能够更快、更安全地完成工作。

最后，在特权提升会话期间监视用户活动使得为所有 JIT 访问请求和授权生成报告以实施安全策略成为可能。它还为安全事件中的取证分析创建了审计跟踪。

在为云权限管理实现 JIT 框架时，请考虑以下最佳实践:

寻找通过自助服务门户为开发人员提供无缝体验的工具。通过自动提交、处理、批准和记录访问请求，节省了每个人的时间。

定义 JIT 批准的每个步骤的所有者，包括谁管理资格、谁批准访问范围以及谁审查请求。

对于低风险访问请求或涉及非生产环境的访问请求，使用无需人工干预的策略。系统应该能够记录请求，评估权限提升的原因和时间范围，并快速响应。

### 用 JIT 减少攻击面

让 DevOps 员工长时间不受限制地访问敏感资源(尤其是在生产环境中)违反了基本的安全卫生最佳实践。JIT 权限管理使组织能够保持工程活动在轨道上和时间上，而不会招致过多的风险。通过减少与 DevOps 身份相关联的攻击面，即使用户的凭证受到损害，黑客也不会拥有通往众所周知的王国的钥匙。这将限制攻击的爆炸半径。

公司面临的挑战是将传统的权限管理功能扩展到云环境。幸运的是，IAM 和 PAM 功能的新的云原生优势使得弥补云安全中的这一缺失环节成为可能。
