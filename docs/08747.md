# 零信任网络访问平台消除了 DevOps 瓶颈

> 原文：<https://devops.com/zero-trust-network-access-platforms-slash-devops-bottlenecks/>

在之前的一篇文章中，我指出了为什么需要一个新的[远程零信任平台来支持 DevOps 团队](https://securityboulevard.com/2021/07/a-new-remote-zero-trust-platform-is-needed/)。

传统的远程访问配置不太适合开发运维，因为不断变化的用户群包括员工、承包商和其他第三方，他们通常使用自己的设备；必须提供对大量工具和系统的访问，以及基础设施本身的混合性质(内部、混合和多云)。

## DevOps 痛点

与 DevOps 一起使用时，传统方法有三个痛点。

1.在传统的远程访问配置中，DevOps 用户体验很差。每个 DevOps 服务都必须作为单独的安全会话来访问，这是令人困惑和乏味的。
2。系统管理需要 VPN 中的 IP 白名单规则、Bastion 中的静态 SSH 密钥、由不同团队控制的防火墙规则和特定于应用程序的身份验证。协调和执行如此多的常见操作接触点—包括新团队成员的加入、角色的改变或添加新服务—会导致错误，从而导致生产力问题并引入不可预见的安全漏洞。
3。安全性是复杂且容易出错的。开发运维用户必须被授予对他们高效工作所需的个别服务的受限访问权限。管理个人和组的受限访问非常耗时。安全策略通常要求监控用户访问的详细审计日志。

## 零信任远程访问平台如何与 DevOps 协同工作

在我的白皮书[Zero-Trust-Remote Access-For-DevOps 中，](https://www.banyansecurity.io/wp-content/uploads/White-Paper-Zero-Trust-Remote-Access-For-DevOps-21210810.pdf)我解释了零信任远程访问平台如何解决改进 devo PS 用户体验、一致且可管理的管理以及简化且不易出错的安全性的需求。

### **改进的 DevOps 用户体验**

通过取代用户为他们需要使用的每个 DevOps 服务设置单独会话的繁琐需要，客户端 DevOps 服务目录使所有 DevOps 服务都可以通过单击来访问。这种服务目录方法对于 DevOps 用户来说更加方便，并提供了一种 DevOps 治理形式。它通过鼓励 DevOps 团队选择目录中可用的选项来实现这一点，而不是创建导致昂贵的 DevOps 工具蔓延的新选项。

### **简化的开发运维管理**

通过将所有 DevOps 用户和服务的访问整合到一个统一的服务管理器和连接器下，简化了远程 DevOps 用户的管理。当 DevOps 用户和服务的管理要求整合到一个通用的零信任远程访问平台下时，通过不同的团队(每个团队都有自己的流程)管理每个用户和服务的需求得到了统一和简化。

此外，这种方法使得跨用户和服务范围实施适应性更强的策略变得更加容易。

借助零信任远程访问平台，新 DevOps 用户和组的入职在单个会话中完成，而不是针对每个入职要求的单独会话。

### 简化且不易出错的安全性

将对 DevOps 服务的访问整合到一个具有统一服务和服务连接器的通用零信任远程访问平台下，可实现简化且不易出错的安全控制、改进的安全监控和策略管理。所有服务的访问审计日志都会得到一致的报告。调整策略时，可以支持个人或组监控和更改访问规则。可以一致地实施基于信任的访问控制。精细策略确保访问基于符合适当的设备信任分数阈值。这些生产服务策略不是在授予访问权限之前进行一次检查，而是持续评估访问服务的用户和设备的安全性。

## 向面向 DevOps 的零信任远程访问平台转型

所有类型的转换，尤其是开发运维的转换，通过遵循我的书[工程开发运维](http://mybook.to/engineeringdevops)中描述的
七步转换过程变得更加确定。这个过程作为实现零信任远程访问平台的框架。

前两个步骤，即愿景和团队调整，涉及让组织的领导者、关键影响者和实施者认识并建立可操作的目标，以实施零信任远程访问平台。

第三步是评估，需要清点新平台需要支持的开发运维用户、开发运维服务和政策的当前状态。

第四步，路线图，需要确定实施计划。通常，从一个应用程序开始是有意义的。从单个应用程序或管道所需的一组 DevOps 用户和 DevOps 服务开始，随着解决方案得到验证和接受，这些应用程序或管道可以作为推动其他应用程序采用的模型。

第五步，realize，是为初始模型应用程序或管道安装和设置零信任远程访问平台的步骤。 [Banyan Security](https://www.banyansecurity.io/test-drive/) 解释了示例工具供应商如何帮助简化这一步骤。

在零信任远程访问工具中可以寻找的功能包括:

在几分钟内完成部署，无需集成任何现有的 DevOps 服务或基础架构。
无需设置 VPN、打开入站防火墙端口或管理 DNS。
·在不降低安全性的情况下使用现有的 DevOps 用户设备。
“开发运维服务目录”为用户提供了对开发运维服务的可见性和一键式访问。
支持捆绑服务、收藏夹和自动运行功能。

在第六步“实施”中，试运行和使用零信任远程访问平台的程序已经过验证，收益已经过衡量，行动手册也已经过强化。

在第七步“扩展”中，零信任远程访问平台部署扩展到整个企业中的其他开发运维应用管道、开发运维用户和开发运维服务。

## **这意味着什么**

鉴于服务、团队成员和环境的动态性质，传统的[远程访问](https://devops.com/?s=remote+access)配置不太适合开发运维管理员或用户。零信任远程访问平台可满足以下需求:改善开发运维管理和用户体验，简化开发运维服务的访问管理，以及更简单、更少出错的安全实践和监控。
