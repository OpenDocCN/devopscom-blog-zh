# 完整的管道:用于网络自动化和流程编排的 CI/CD

在过去的几年里， [CI/CD 和网络自动化](https://www.itential.com/solutions/network-infrastructure-as-code/) 的交集已经大大增加。网络团队似乎已经到达了他们的自动化和编排之旅的关键点，在这一点上，实施一种策略来管理网络变更和自动化资产的测试、版本控制和部署变得至关重要。

虽然 [CI/CD](https://devops.com/?s=CI/CD) 越来越受欢迎，但至少从表面上看，网络团队希望应用该技术的管道实现类型存在许多差异:不同的工具、不同的场景和不同的问题被解决。尽管表面上存在差异，但在大多数管道实现中，有一些关键主题是一致的。例如:

网络团队要么希望开发管道来更好地管理网络的特定变化，要么希望使用管道来更好地管理自动化和编排资产(如工作流或脚本)的测试和部署。最初，这些团队可能会转向网络基础设施即代码(IaC ),但有相当数量(且越来越多)的团队正在投资自动化资产管道。

投入到这些管道中的信任和资源只是越来越多的指标之一，这些指标表明自动化和流程编排解决方案正在走向成熟，并从最初的实施转向大规模的常规业务技术。

### 管道和网络自动化如何互补

对最近管道开发的仔细研究，更清楚地说明了 CI/CD 工具和网络自动化工具在网络变更管理和生命周期方面各自的优势——git lab、GitHub、Jenkins 和 CircleCI 等等。大多数网络团队更愿意坚持“正确工作的正确工具”的理念，其中任何给定任务的最佳工具可以很容易地集成到整体解决方案中。

在这种情况下，“正确的工具”哲学着眼于管道和网络自动化/协调工具，以确定它们做得最好。然后，制定一个策略，最大限度地发挥两种技术的优势。使用这种方法，团队可以避免试图用一种或另一种方法解决所有问题的陷阱，从而导致次优的过程。

当前一些好的经验法则基于两个简单的要点:

*   **管道**通过管理与代码版本、测试协调、代码合并、代码验证和批准相关的活动，专注于创建一致性和可重复性。管道的主要好处是自动化变更过程的能力。
*   **网络自动化**专注于生成网络有效负载并将该有效负载应用于网络元素。这可能还包括通常必须以特定顺序执行的工程和业务逻辑步骤的自动化。网络自动化在变革过程中的最佳应用需要关注业务逻辑、操作顺序和网络工程规则。

通过将这两种技术集成到一个公共管道中，网络团队可以获得这两种工具集的优点。
