# 实现可持续开发运维转型的八个步骤

> 原文：<https://devops.com/eight-steps-sustainable-devops-transformation/>

[DevOps](http://automic.com/devops) 不是标准。它更像是一套松散的概念和工具所描述的哲学，许多都植根于敏捷方法和持续交付。这通常会导致对 DevOps 潜在范围的混淆，并使确定 DevOps 计划的确切起点和终点变得困难。此外，DevOps 需要重大的文化和组织变革，这可能很难实现和维持。

现在我们知道了什么是 DevOps，在这里，我们提出了实现可持续发展的 DevOps 转型的八个步骤，这些步骤基于 Kotter change framework (1)的修改版本:

1.  **培养紧迫感**

“更快、更便宜地交付”。来自企业的两个最常见的要求—无论是交付数字化转型、寻找更灵活、更具成本效益的内部 it 替代方案，还是其他要求。

IT 组织必须充分意识到变革的紧迫性，以便保持与业务的相关性。

在开发/运营环境中，这种紧迫性是通过发布速度、新特性的周转时间和其他指标来衡量的。基准测试在这方面很有帮助:例如，使用 DevOps 工具将交付时间结果与其他同等公司进行比较。任何不足都可以通过价值流分析和 TIMWOOD 浪费分析提出来。

2.  **组成指导联盟**

为了成功的开发运维转型，确定一个既支持变革又能够交流进展的团队。包括来自运营和开发部门的高级 IT 管理人员，以及开发运维流程中的主要利益相关方。

3.  **创造愿景**

持续交付的 DevOps 愿景应该与业务愿景保持一致。它应该足够宽泛以涵盖 DevOps 的所有方面，但也应该足够狭窄以指导后续变更过程中的决策。

Gartner 使用术语“边界对象”来表示可以“在不同的社区中有不同的解释，但拥有足够的共识，使其成为交流和改进知识的有用机制”的事物。因此，边界对象被认为是在主要概念上对齐的一种可能方式；这些对象对于本地需求是灵活的，但是足够健壮，可以跨站点维护一个公共的身份。开发运维/持续部署路线图也是必要的，它将重要的主题及其优先级封装在一个大致的时间表中。避免详细的项目计划，基于 DevOps 应该遵循敏捷的方法，并且都是关于持续改进的。

整个组织应该采用 DevOps，还是应该将 DevOps 限制在那些对敏捷性的需求最终胜过强大的流程合规性需求、风险缓解和可用性需求的领域？

对此没有明确的答案，但请记住:

*   DevOps 和文化有很大关系。从长远来看，很难维持一个建立在两种不同文化基础上的单一组织。
*   IT 系统通常是相互依赖的；必须在一系列系统中实施简单的更改。

4.  **传达愿景**

一次又一次地传达愿景、战略和路线图。此外，开始实践持续交付的文化价值观，例如避免责备、将个人置于过程之前以及促进透明度。

5.  **授权他人按照愿景行动**

愿景的许多方面——如降低流程复杂性和修订协作理解——应该已经扩散到公司文化中，即使“全面”开发运维尚未到位。

指导联盟需要消除障碍(例如过多的流程步骤或批准以及僵化的系统访问权限)。这些潜在地限制了试验新工具或方法的能力。事实上，任何严重破坏愿景的系统或结构都应该被取代。同样，应该鼓励冒险和非传统的想法、活动和行动。

6.  **创造短期胜利**

对于成功的 DevOps 部署和转换，交付可见的性能改进非常重要，最好是使用已建立的 DevOps 团队的几个试点项目。

选择试点团队时，请确保:

*   可以信任该团队来管理持续交付
*   该应用程序在架构和技术方面非常合适
*   开发运维的优势对企业来说是显而易见的

7.  **巩固改进成果并扩大发展规模**

利用从成功试点中获得的信誉来改变不符合持续交付愿景的系统、结构和政策。

让不同的团队实现他们自己的 DevOps“风格”,最适合他们的特定环境。

这些团队可以利用通过试点开发的经验、流程和工具，但只要他们的实现符合整体的持续交付愿景，就不必这样做。

避免治理框架，它违背了敏捷的概念。

8.  **使新方法制度化**

阐明新行为和持续交付的企业成功之间的联系。要做到这一点，您需要适当的指标，并与顶级 KPI 相关联。在这个阶段，整个业务系统必须适应新的文化，这样变革就成为“一切照旧”的一部分。

这种“制度化”还需要度量标准、激励机制和适当的管理风格的支持。
