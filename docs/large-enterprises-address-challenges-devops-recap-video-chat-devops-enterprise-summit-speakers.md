简单介绍之后，我总结了 DevOps 社区目前面临的五大问题，这也是今年 DevOps 企业峰会的重点关注领域。在去年的峰会上，我要求每一位演讲者在结束演讲时，用一张幻灯片来说明以下两个标题中的一个:“这是我不知道如何做的”或“这是我正在寻求帮助的”我们想了解整个社区面临的最大挑战是什么，然后我们可以将其用作研究路线图。

每位演讲者都分享了他们的组织如何应对这些挑战，并分享了他们在 DevOps 企业峰会上的演讲内容。

1.  **如何解决文化和领导力问题，尤其是围绕过渡和转型的问题**

科妮莉亚·戴维斯建议从小处着手，也许可以派遣一些精选成员从事结对编程，然后让他们回来把技能带回公司内部。“这是学习新编程的最佳方式之一。这更像是一个基层过程，”戴维斯解释道。

Heather Mickman 对此表示赞同，并补充道:“我们开始了自己的内部开发日，在此期间，我们可以迅速建立势头并展示成果。为了加速采用，我们创建了一个内部 DevOps dojo 来帮助指导团队使用工具和流程。”

芝华士·南比亚尔建议:“找到与你有共同信念的领导者，他与工作关系密切，并且了解这些类型的技术趋势。第二点是，你不可能在一夜之间完成这种转变。我们正进入转型的第四年。找到你的关键合作伙伴，给他们成功所需的一切，然后分享他们的成功。”

当我问“我如何让我老板的老板关心 DevOps？”拉尔夫·娄艺潇列举了对抗怀疑和抵制变革的“企业抗体”的方法:“你最需要的是信任、信任和信任，”他说。“为了建立这种信任，我们通过协作工具和将人们配对来提供透明度；我们创建并嵌入布道者，帮助下一波团队成功并传播知识，在出现问题时提供帮助；我们需要成功的数据和证据。

该小组讨论了基层努力与获得领导认同之间的平衡，每个人都讨论了团队需要创造小成功以赢得怀疑者的价值。娄艺潇提出了营火的比喻:“如果你点燃了营火，并且过快地添加了过多的燃料，它会使火熄灭。重要的是一开始要把它遮挡住，直到火有足够的热量。”

2.  **如何处理自动化测试问题，特别是遗留环境**

谈话转到了我们都有过的软件产品或服务的经历，在这些经历中，我们给出了开发、测试、操作等等。一个不同的采购伙伴，认为这样我们会得到更好的结果。Heather Mickman 分享了从该模型到当前模型的转变，“我们过去在 Target 进行软件开发的方式是将每个功能交给不同的团队和卓越中心。移交给了 10 到 15 个不同的团队。这些每季度一次的发布没有端到端的责任。因此，当我们开始组建为 Target 创建 API 的团队时，我们决定要端到端的问责制，所以我们打破了这些孤岛，让每个人都关心最终产品。”这只是从这些复杂的组织中涌现出来的模式的一个例子。

3.  **如何解决 IT 安全性和合规性问题–让您的安全团队参与您的开发运维计划**

所有这些演示者都来自大型上市公司，他们必须遵守许多监管措施。我想更多地了解他们是如何处理安全性和法规遵从性的。

Neil Manvar 建议 DevOps 从业者完全透明。他说，“向安全团队展示其价值，这样他们才能参与进来，而且你不只是想让他们不好过。给他们访问变更日志的权限，非常好地格式化它或者链接到提交。提供更多关于正在发生的事情的信息确实有助于让每个人都参与进来。提出的任何问题，都会围绕它创建一些自动化，并将其集成到管道中。”
