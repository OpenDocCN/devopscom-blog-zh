# CloudBees 收购 ReleaseIQ 以推进 DevOps 流程编排

> 原文:[https://devo PS . com/cloud bees-acquisites-release IQ-to-advance-devo PS-orchestration/](https://devops.com/cloudbees-acquires-releaseiq-to-advance-devops-orchestration/)

CloudBees 今天宣布收购了 ReleaseIQ，这是一家通过软件即服务(SaaS)平台自动化软件交付的编排平台提供商。

CloudBees 首席战略官 Sacha laboury 表示，将 ReleaseIQ 添加到 CloudBees 平台将提供对端到端软件交付的更多可见性和控制。

CloudBees 已经在其[连续交付](https://devops.com/?s=continuous+delivery)平台中提供了一个流程编排引擎。然而，ReleaseIQ 使用一个更易访问的 SaaS 平台提供了跨多个持续集成/持续交付(CI/CD)平台的类似功能。

[ReleaseIQ 平台](https://vimeo.com/754401251)旨在使 DevOps 团队能够使用无代码工具创建和维护管道。这使得工作流更加透明，因为仪表盘还能提供优化建议。该功能使得软件部署的故障排除变得更加简单。

![](../Images/9e6ba7c1bf6476afbd8b5999665fd6bf.png)

Labourey 表示，虽然 ReleaseIQ 可以与多个平台集成，包括 CloudBees CI、Jenkins、CircleCI、GitLab、Bamboo、ArgoCD 等，但主要焦点仍将是开源的 Jenkins 平台，该平台仍拥有 DevOps 市场的最大份额。他指出，CloudBees 的总体目标仍然是采用自动化，并尽可能多地消除 DevOps 工作流中的摩擦，作为提高开发人员生产力的持续努力的一部分。然而，不管任何组织内的 DevOps 成熟度如何，CloudBees 都致力于满足客户的需求，而不是要求他们淘汰和替换现有投资，Labourey 补充道。

在组织内部和组织之间，开发运维成熟度的总体水平可能会有很大差异。然而，随着越来越多的组织意识到他们在多大程度上依赖软件来驱动[数字业务转型](https://digitalcxo.com)流程，进一步自动化应用程序开发和交付的需求变得更加明显。虽然 CI 在今天被广泛使用，但持续实施 CD 的组织数量相当有限。部署软件的底层基础设施通常具有每个 CD 平台需要支持的一组独特的应用编程接口(API)。ReleaseIQ 等平台提供了更高层次的抽象，通过这些抽象，管理交付软件的管道变得更加简单。

一般来说，大多数组织都试图在集中管理 DevOps 工作流和支持开发工具采用他们喜欢的任何工具之间取得某种程度的平衡。对于应用程序开发团队来说，独立采用最终需要与 DevOps 工具链集成的任意数量的工具并不罕见。然而，这些工具中的每一个都包含需要与集中式编排引擎共享的元数据，该引擎基于从每个工具收集的元数据跨多个管道自动化一系列任务。

当然，仍有许多手动开发运维流程应该实现自动化。CloudBees 将 ReleaseIQ 定位为编排引擎，用于统一管道管理，并最终。在日益复杂的应用开发环境中实现价值流管理。可能需要一段时间才能完全实现这一雄心，但随着对 ReleaseIQ 的收购，一个新的基础正在奠定。
