# 展望开发者体验的未来

> 原文:[https://devo PS . com/展望未来的开发者体验/](https://devops.com/looking-to-the-future-of-developer-experience/)

开发者体验是最重要的！[开发者体验](https://devops.com/understanding-the-importance-of-developer-experience/) (DX)意味着在开发者之旅中采用用户体验的方法。DX(也缩写为 DevEx)通常应用于文档、沙盒环境、SDK 人体工程学等领域。通过改善开发者对某个工具的体验，软件提供商可以增加粘性，留住更多用户。

如今，拥有流畅的开发人员体验是一种竞争优势——尤其是因为开发人员通常有购买新工具的决定权。在持续的技能差距中，保持内部软件生态系统的高质量对于留住快乐的工程师和避免疲惫的 T2 也很重要。所有这些因素意味着开发人员体验正在成为高管团队的目标。那么，DX 走向何方？开发人员的工作流程在不久的将来会有什么变化？

我最近会见了 Nylas 的开发人员关系总监 Ash Arnwine，思考关于开发人员体验的近期前景。根据 Arnwine 的说法，DX 仍然是一个非常新的概念。然而，它已经成为定位现代软件的一个不可或缺的方面，特别是对于面向公众的 SaaS 来说，并且无疑将在未来的一年中推动购买决策。下面，我们将分享一些关于 DX 将如何在四大类别中发展的预测:入职、构建、部署和维护。

## 开发者体验的状态

首先，在整个技术世界中，开发人员的体验状况如何？嗯，Arnwine 很快指出，这个概念仍然相对新生。“开发人员体验是我们不久前才谈论的东西。”然而，随着[开发者对软件生命周期行使更大的控制权](https://containerjournal.com/features/can-great-dx-enable-full-cycle-development/),这种想法在人们心中越来越流行。

事实上， [2022 年 StackOverflow 调查](https://survey.stackoverflow.co/2022)发现，66%的专业开发人员至少对他们的组织购买新技术有一定影响，高于 2020 年的 56%。此外，开发人员的学习方式正在发生变化。新一代程序员主要是自学，依靠编码训练营或其他在线资源。令人印象深刻的是，70%的开发人员从在线资源中学到了如何编码。

Arnwine 说，我们最终得到的是越来越多的开发人员，他们拥有比过去更广泛的技能。由于没有人亲自操作，技术本身必须是可用和直观的——软件提供商有责任设计无摩擦的自助服务工作流。Arnwine 说，随着开发商在采购方面的影响力越来越大，DX 成为了一种竞争优势和高管的谈资。

### 入职

开发人员体验是一个广泛的话题，但它通常与开始使用一个工具所需的时间有关。这通常被称为“第一次你好世界的时间”但是尽管这是重要的一步，度量可能并不总是具有很大的意义。Arnwine 说，另一方面,“价值实现时间”则更进了一步，是衡量成功的更好标准。

例如，加入 web API 可能涉及注册过程、创建帐户、获取 API 密钥和阅读文档——所有这些都在第一个调用发出之前。根据 Arnwine 的说法，在这个阶段有一些技术可以帮助开发人员在整个入职过程中更快地移动，从而更快地实现价值。

以下是开发人员入职变得更加简化的一些方式:

*   注册时自动生成密钥。
*   在开发人员仪表板中提供一个终端来测试示例代码。
*   将开发人员需要定制的代码交给他们特定的堆栈。
*   使用规范作为文档的唯一来源。

### 建筑物

SaaS 公司在建设过程中可以做些什么来帮助提高 DX？一种方法是在 ide 中启用自动完成。当开发人员使用新工具时，他们通常会收到 SDK *和*参考文档。但是，让程序员在浏览器之间来回查找方法名并没有增加价值。相反，Arnwine 认为开发者工具应该“以一种现代开发者所期望的方式融入到 IDE 中”

类固醇自动完成的一个例子是[图](https://fig.io/)，这是支持 500 多种流行 CLI 工具自动完成的下一代命令行。[更进一步](https://devops.com/github-previews-free-ci-cd-migration-tool-due-in-2023/)是 [GitHub Copilot](https://github.com/features/copilot) ，这是一个人工智能助手，可以接受自然语言提示并生成非常复杂的代码建议。该行业似乎正在走向更多由人工智能驱动的低代码软件开发——这一趋势可能会显著改善开发人员在构建过程中的体验。

然而，增加软件开发自动化的一个潜在缺点是确保人工智能建议正确的代码片段。因此，Arnwine 说，他预计可能会出现故障排除和调试问题。此外，SaaS 供应商将不得不审查生成胶水代码的系统，以确保它的准确性和最新版本。他对 SaaS 供应商的建议是，在他们所在的地方与开发者会面，但尽最大努力保持他们可以控制的高质量体验。

### 部署

在很长一段软件开发历史中，IT 管理员是唯一能够部署软件的人。老派的方法需要获得一个主机提供商，安装 Ubuntu 和大量其他跑腿工作。然而现在，软件部署变得更加拖拉。DevOps 在最近几年变得更加大众化，我们很可能会在未来几年看到更好的可用性出现。

增强开发人员部署体验的一种方法是通过 GitHub Actions 或类似的基于事件的系统来定制更多的自动化例程。即插即用的预制动作可以减少使用这种系统的障碍。Arnwine 说，这种连接是实现“文档即代码”方法所必需的。

### 维持

另一个需要认真升级开发人员体验的领域是依赖性管理。软件依赖经历一个生命周期。它们被版本化、更新、淘汰，有时甚至被彻底替换。因此，跟踪第三方依赖关系的不断发展的表面区域可能是一件令人头痛的事情。更糟糕的是，[软件供应链威胁](https://devops.com/the-age-of-software-supply-chain-disruption/)使得大多数应用程序使用的许多开源依赖项的完整性变得复杂。

尽管组织必须维护依赖关系的对等性，但是很少有人承担持续检查依赖关系和监控安全威胁的任务。因此，依赖关系管理和安全监控需要更多的自动化。例如，Arnwine 使用 Snyk，它每周发送一份关于其依赖项状态的报告。

## 最后的想法

假设你有强大的功能来支持它，投资开发人员体验可以帮助开发人员工具在拥挤的市场中脱颖而出。但是当然，提高 DX 意味着许多不同的事情，取决于你的环境。这可能是一个[微妙的用户界面调整](https://nordicapis.com/8-tiny-api-docs-tweaks-that-deliver-big-on-dx/)，添加相关的代码样本或使你的文档更加直观。

另一方面，[负面的开发者体验](https://nordicapis.com/what-does-a-bad-developer-experience-look-like/)可能会拒绝潜在用户，并在整个应用支持生命周期中造成挫折。这可能会给程序员带来不好的名声。

回顾一下，在不久的将来，我们可能会见证开发人员体验在以下方面的发展:

*   简化注册流程的增强方法。
*   自动完成和人工智能驱动的代码建议，了解开发人员的堆栈，并在他们工作的地方满足他们。
*   更直观、自动化的部署模型。
*   依赖关系维护的自动化程度提高。

虽然开发者体验越来越受到关注，[它不应该是 SaaS](https://youtu.be/ezOh-Vd09ZQ) 的唯一衡量标准——它给开发者消费者和最终用户带来的持续价值同样重要(如果不是更重要的话)。
