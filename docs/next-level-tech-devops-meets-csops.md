# 下一级技术:DevOps 满足 CSOps

> 原文：<https://devops.com/next-level-tech-devops-meets-csops/>

运营是一个组织最重要的方面。在技术领域，DevOps 占据了主导地位，因为它必须跟上用代码构建产品、应用和基础设施的速度。DevOps 包含的不仅仅是一个开发团队；它是一种文化，一套实践和工具，允许组织快速、大规模地交付应用程序，然后有效地运行。这一理念在过去的十年中非常有用，但缺乏与组织其他部分的联系。开发团队是相互联系的，但是在他们的竖井中，他们的工作缺乏与最终用户的集成，削弱了为他们的客户提供最佳解决方案的机会。更直接的是，开发人员、DevOps 和 ITops 在自主环境中工作时看不到影响的广度，导致优先级和补救措施不佳。

进入客服。组织力量的另一个支柱，客户服务不仅是一个必需品，而且已经成为企业的一个战略职能。当数字服务没有按预期工作或没有满足客户期望时，他们是企业的第一反应者。客服人员训练有素，资金充足，拥有最新的技术，因此他们可以成为客户最有效的第一接触点。客户服务团队可以从他们组织中的开发人员那里学到一些东西，并在指导哲学和实践的同一种“运营”模型中工作:换句话说，就是 CSOps 或客户服务运营。

## 为什么选择 CSOps？

客户服务是商业运作中一个非常强大的元素，因为它与用户体验紧密相关。当消费者经历了糟糕的客户服务时，他们不会羞于宣布，反之亦然。终端用户也大声吹嘘他们得到了强有力的客户服务的时刻；这对于创造忠诚度、回头客和客户保持率大有帮助。贝恩&公司 T3[的一项研究指出，如果客户遇到与服务相关的问题，而不是基于价格或产品的问题，他们从竞争对手那里购买的可能性会增加四倍。像许多电子商务平台一样，以客户服务为主导是一种新的运营标准，对于满足客户的期望至关重要。在 B2B 环境中，客户服务也不例外。客户或用户被吸引到能够以无摩擦和高效的方式快速修复他们的问题的组织。](https://www.bain.com/insights/topics/loyalty)

客户服务现在处于企业计划的最前沿，不会消失。在最近来自《华尔街日报》 的一篇文章中，分析师和高管们对首席信息官的预算进行了权衡，并预测了在日益增长的经济衰退担忧中的优先事项。随着一些领域的裁员，客户服务不在其中。总体观点是，“企业比以往任何时候都更看重技术，这将增强客户体验，并在日益紧张的市场中赋予他们优势。”

在当今的组织中，如果您没有 CSOps，就无法与 DevOps 竞争。它们在与底线的关系中是共生的——如果产品或应用没有达到客户期望的结果，来自客户服务团队的沟通必须与开发人员联系，并传达错误或缺失的内容。然后，开发人员可以提供适当的解决方案。客户服务运营为 DevOps 渠道提供支持，以创造满足客户需求的产品和服务。换句话说，如果没有 CSOps，您就无法在开发运维方面取得成功。

## 创建 DevOps 和 CSOps 关系

事实是，大多数 CSOps 和像 SREs 或 ITOps 这样的后台团队没有良好的关系，也没有有效的流程来相互沟通。这种脱节不仅会给客户带来问题，而且内部沟通的缺乏会造成团队之间的断层，无法高效地产生结果。随着 SaaS 模式、灵活工作和应用程序的兴起点燃了技术组织在过去十年中的发展轨迹，审视组织内的最佳实践并了解团队如何更好地合作非常重要。

当开发团队继续迭代和产生代码时，客户服务团队不断地处理客户的反馈、关注和投诉。客户服务代理通常是第一个了解故障情况和故障位置的人——如果组织没有将一线员工作为补救和[事件响应](https://devops.com/?s=incident+response)的工具，他们就错过了可以帮助他们找到解决方案的宝贵信息。

除了处理有关产品和服务问题的投诉和报告之外，客户服务团队还能够汇总和关联报告，并从报告用户那里收集更多信息。当出现问题时，动员所需的快速反应通常会很困难。如果客户服务团队与开发人员或工程团队之间缺乏沟通，客户对产品或服务的投诉可能需要很长时间才能解决。创建从客户服务团队到开发人员团队的快速连接允许客户服务代理了解问题和解决方案，在实施解决方案时实时向客户提供反馈。正是客户服务和开发团队的这种“运营”文化从始至终支持着组织。

DevOps 和 CSOps 的共生关系可以使任何组织受益，但采用需要可见性和双向沟通。与客户服务团队保持密切联系的开发团队受益于更好的、真实的用户信息，这些信息可以被合并到新的特性和改进中。当客户服务团队有一条清晰的路径来上报客户报告的问题时，通过消除孤岛和打破 CSOps 与 DevOps 之间的壁垒，解决问题的时间会缩短。

现在是每个组织整合客户服务团队作为实现客户满意度的一部分，而不仅仅是报告的时候了。与工程和后勤团队的整合创造了服务至上的运营文化。
