# 软件架构卓越的创新

> 原文：<https://devops.com/innovation-with-software-architectural-excellence/>

[软件架构](https://devops.com/devops-chats-software-architecture-for-cloud-native-net-core-and-open-source/)在让企业高效交付价值方面发挥着不可替代的作用。这一点在数字原生者如何挑战现有者并迫使他们快速数字化和转变他们的运营和创新方式中显而易见。因此，实现卓越的架构是一个高效实施、高级规划和执行领导参与的持续过程。此外，认识到软件架构策略和计划不应该局限于 it 是很重要的。支持 IT 和业务领导之间的协作是必要的，因为软件架构在交付价值、使流程健壮以及应对价值创造和交付的下一波变革中起着关键作用。

## 好的软件架构的目标

一个好的软件架构促进了正在进行的业务流程的稳健运行，并确保业务能力在面对威胁性挑战时具有竞争力。除此之外，软件架构也应该成为驱动下一波业务变化的基本方面——无论是流程、模型，还是创新的产品和特性。这些变化源于不断增长的数字化、转型和基于生态系统的整合的竞争力量，使变化成为各种业务方面的首要常数。因此，传统上在有限的软件或产品级别上查看软件架构不再像过去那样有效。随着数字化和创新使业务和生态系统在交互方式上数字化地交织在一起，在业务和生态系统层面上看待软件架构的实践得到了更多的重视。

以下基本方面描述了如何利用卓越的架构进行创新:

### 安全性和隐私

客户越来越担心如何处理和保护他们的数据，如果一个组织没有关于如何保护和处理客户数据的明确政策，他们甚至不愿意做生意。另一方面，监管机构也在不断提高合规要求，以保护客户。这一格局在多个国家正以不同的速度发生变化，为快速适应者带来挑战的同时也创造了机遇。所有这些都意味着企业需要从道德上管理客户数据，同时防范不断变化的威胁形势。为了满足这种需求，有必要修改现有的软件架构，并让高级管理人员参与这方面的关键决策。此外，架构规划周期应该缩短到几个月——而不是几年。这是快速构建和实施创新解决方案以应对不断变化的监管环境和客户期望的关键。

### 满足用户需求

自 2008 年以来，智能手机一直在改变客户的体验和期望。数字颠覆者不断挑战现有者，威胁他们的生存。在这种范式转变中，企业应该更深入地了解他们的客户，快速调整他们的业务模式，并推出新产品来满足客户的需求。它需要架构的灵活性，能够在更短的周期内灵活地创新业务模型，并根据客户的反馈更快地改进它们。这也意味着对软件架构规划和策略开发保持长期的观点，以实现无缝创新并消除流程和性能瓶颈。从这个角度来看，软件架构的改进和决策应该与业务目标相一致，并增强业务能力，而不仅仅是 IT 已经交付的实现或技术解决方案的数量。

### 授权给开发者

如今，越来越多的企业需要开发者帮助他们实现数字化和转型。因此，授权给开发人员是必不可少的，而软件架构是帮助他们推动创新和建立关键业务能力的基本方法。开发人员授权通常以开发人员的速度来衡量——它定义了开发人员的敏捷程度以及他们的才能被利用的程度。为了提高开发速度，为他们提供一个定义良好的软件架构来关注质量和一致的开发。此外，软件架构必须有定制的余地，以便开发人员可以充分利用它的潜力。

虽然以上描述了总体方法，但以下部分详细描述了通过卓越架构实现创新的基本方面。

### 可量测性

软件架构应支持按需无缝扩展应用程序，以适应用户增长高峰，并缩减规模以节省运营成本。因此，这种方法需要在云中部署应用程序，同时确保不仅底层基础设施，而且整个系统中的多个组件都可以无摩擦、无干预地扩展。高性能应该伴随着这一点，以确保客户满意度和提供卓越的体验。这意味着您的系统应该在任何时候、任何规模下都能理想地运行。

### 轻便

系统本身或其组件能否部署在任何云平台上，包括公共云、私有云或混合云？就时间、精力和成本而言，搬家要花多少钱？在一个拥有[超大规模](https://devops.com/three-cloud-trends-to-watch-in-2022/)云提供商和尖端云服务的世界中，公司的 IT 生态系统不应局限于一个平台或供应商，而应利用当前可用的最佳服务和技术，这些服务和技术也是面向未来的，可以提高性能和优化总体成本。

### 服从

业务部门应制定合规框架，使人员、流程和战略符合监管要求，以确保合规并避免不合规风险。此外，该框架应该是正在进行的工作，以适应监管机构即将到来的全面变化。

### 可维护性

类似于软件组件的源代码，软件架构应该易于维护——这意味着在修改和改进软件架构时，它不应该引入瓶颈。

可维护性 问题可能有多种形式:软件架构对于当前的技术和商业前景是否过时了？当其他组件和依赖项更新时，它是否使用不兼容的遗留版本？确保软件架构的可维护性有助于促进创新，并以有竞争力的速度前进。如果不确定从哪里开始改进或重新定义软件架构，就从可维护性开始。

### 费用

it 部门经常受到审查，以证明其支出的合理性，衡量 IT 部门对业务成果的有意义贡献也很复杂。因此，IT 不断面临预算削减的风险和优化成本的挑战。典型地，软件架构被允许有机地增长，导致系统和工作的重复以及跨系统的不一致的数据。为了克服这一点，在持续的基础上控制软件架构，这反过来允许更可预测的成本控制。

在这个瞬息万变的数字世界中，企业应该保持灵活的架构以促进创新，并使用敏捷方法不断改进软件架构，从而在业务能力方面获得竞争优势。这种做法是非常必要的，尤其是当数字颠覆者越来越有可能把一切都夺走——把落后者甩在后面的时候。
