# 管理企业架构中的微服务

> 原文：<https://devops.com/governing-microservices-in-an-enterprise-architecture/>

软件开发策略不断发展，以满足不断变化的业务需求。最近有很多关于数字转型的讨论，定制软件开发正在推动这种转型。随着公司努力开发更好、更快、更灵活的应用程序，敏捷软件开发和 DevOps 已经成为规范。微服务提供了企业应用开发演进的下一步。

DevOps 将开发人员(Dev)和运营人员(Ops)分成团队，共同开发新的企业解决方案。类似地，微服务扩展了专业化的思想，创建专门的应用程序构建块来构建新的应用程序。微服务支持更短的开发时间和更高的灵活性。它们也为企业架构师提出了一系列新的挑战，他们必须监督这些微服务并管理[开发过程](https://devops.com/new-aws-services-optimize-application-development-process/)。

### 一系列积木

如果您对这个术语不熟悉，微服务是一系列组装在一起用于开发新应用程序的构建模块。正如敏捷软件开发是一种哲学和开发方法一样，微服务是企业软件开发的一种架构和组织方法。

微服务的价值在于构建模块方法。对于老式的整体式服务，您必须设计和维护一个应用程序及其各种组件，以作为一个单元。更改或删除一个元素，整个应用程序都需要重新编写和测试，这既费钱又费时。

借助微服务，单个服务作为独立的功能单元运行，可以根据需要进行混合和匹配，以创建不同的定制应用。使用微服务更快、更灵活、更可靠，简化了大型复杂应用程序的交付。

微服务也是围绕业务功能构建的，因此业务应用程序组件可以与 DevOps 敏捷企业开发协同创建。不同的团队一起工作来创建更小的组件，出现单点故障的可能性就更小了。

无代码/低代码开发解决方案将微服务的力量放在了业务用户的手中。产品开发、金融、物流和其他部门的领域专家可以使用拖放界面来开发微服务，然后可以将这些微服务集成到其他应用程序中。简而言之，微服务为主题专家提供了创建可添加到企业架构中的定制应用的工具。

### 微服务的利与弊

微服务为企业和企业架构师提供了几个好处。新应用程序的部署速度更快，您可以更灵活地适应不断变化的企业和业务需求。它们还可以快速扩展，并使向现有应用程序添加新功能变得简单。并且微服务元素是可重复使用的。

当然，也有挑战，包括治理。随着架构的发展，需要管理的组件越来越多。除了不同部门构建微服务，DevOps 团队还负责监督应用开发和解决日常问题。拥有成熟的工程文化有助于自动化应用开发和跟踪微服务，提供必要的指导以确保应用开发支持更大的业务目标。

微服务还会产生架构开销。必须记录应用程序，以确保持续的业务价值。然而，试图记录所有的事情会造成延迟，所以文档永远不会与开发阶段保持同步。挑战在于提供足够有用的文档，而不延迟正在进行的软件变更。

这需要新的企业应用思维。DevOps 需要密切关注自治团队，以控制开发并确保他们支持更大的公司目标。使用无代码/低代码工具也可能导致影子 IT 得不到网络架构师或开发人员的认可。

### 管理发展

推动微服务采用的市场力量之一是对“由外向内”应用开发的需求。所有的微服务都应该在客户的请求下开发，不管客户是在组织内部还是外部。客户需求不断变化，微服务可以更快、更有效地适应这些变化。

微服务开发在领域驱动的架构中工作得最好，该架构根据组织的现实挑战对应用程序进行建模。领域驱动的架构根据业务需求以及如何满足这些需求来评估企业基础设施。大多数组织已经有了领域驱动的设计策略，将架构映射到业务能力。

[有界上下文](https://www.martinfowler.com/bliki/BoundedContext.html)是一种策略，是领域驱动设计的一部分。负责微服务的自治团队是围绕库存管理、产品发现、订单管理和在线交易(即有界上下文)等责任领域形成的。领域专家驻留在团队中，因此企业架构师的责任是指导开发与战略目标保持一致，平衡当前需求和未来业务目标。

当将微服务作为企业的一部分来管理时，应用软件架构的 C4 模型——上下文、容器、组件和代码——是有意义的。上下文定义了客户或最终用户与应用程序之间的关系。容器代表微服务本身，以及如何将其打包以用作应用程序构建块。

在任何企业架构中管理微服务的秘密是将业务能力映射到数字基础设施，然后促进企业架构背后的愿景以及实现该愿景的指导方针。关注业务转型和 IT 战略的全组织目标。教育涉众何时让企业架构师参与进来。此外，尝试预测 DevOps 团队视角之外的主题，并关注大局——什么是长期业务战略，微服务如何为该战略做出贡献？
