# 营业额、文件和缺失环节

> 原文：<https://devops.com/turnover-documentation-and-missing-links/>

当员工流动率激增时，IT 部门，尤其是 DevOps 团队需要做好准备。

在我写这篇文章的时候，股票市场正在动荡，DevOps [补偿](https://www.payscale.com/)正在反弹。这对人员流动有影响，因为 DevOps 中的许多人(以及几乎所有高级职位)的薪酬都包括股票，即可能大幅下跌或目前下跌风险很高的股票。这导致了两种类型的流动——一种是为了避免薪酬进一步损失而离开公司，另一种是为了加入股价下跌的公司。结合那些为了更大底薪的角色而离开的人，有为了赚更多钱而离开的，也有为了避免更多损失而离开的。在疫情时期，IT 预算膨胀，但也在紧缩，所以尽管我们有工人，而不是职位短缺，但双方都会有压力。

那么，为什么这个怪胎一直在谈论就业、薪酬和空缺职位呢？

简单:因为 DevOps 在文档方面并不比它的任何其他版本更好。在管理严格的商店中，文档是可用的并且被维护。在大多数商店，它充其量是参差不齐的。在某些情况下，它是完全不存在的。这一点在 IT 界一直都是正确的，如果说有什么不同的话，那就是更加正确，因为 DevOps 实施的早期阶段会导致大量的变化。

人们离开一个没有一流文档的环境会是一个问题。用什么工具？它们位于哪里？这些 SaaS 工具的登录名是什么？等等。等。等。如果人员流动意味着你在相对较短的时间内失去整个 DevOps 团队，这可能会很快变成一场噩梦。那个噩梦对留下的人和进来的人来说都是真实的。总得有人去解决这个问题。IT 管理层必须让业务的其余部分知道，保留问题正在导致积压。不是负责解决这个问题的每个人都必须为那些…你明白的。

花一到四个小时，记录下你所拥有的。它不必很全面，只需列出哪些工具，它们在哪里，组织中谁负责任何 SaaS 供应商的凭据，以及在什么情况下使用这些工具。如果每个团队都记录了他们自己的东西，即使是在共享的工具链环境中，比较文档也会发现一个团队遗漏了另一个团队没有遗漏的小细节。

让我们面对现实吧，不管是好是坏，我们已经在很大程度上进入了一个忠诚建立在补偿基础上的世界。就像雇佣兵一样，大多数 IT 员工总是在寻找更好的工作机会。因此，组织在承认这一事实的同时，需要采取措施将[人员流动](https://devops.com/?s=Great+Resignation)的负面影响降至最低。改变事物的新想法和建议都是好的，你不想干涉这方面的变化。但是，任何阻碍本组织交付能力的事情都应该事先解决。这意味着剩余的文档和新员工可以快速上手，并自信地快速掌握所有权。

继续摇摆。无论你是否在寻找新的工作，保留你负责运行的应用程序，并记录下来。你为组织的成功做出了贡献；前进时不要松懈。不要讨厌那些正在前进的人。并非我们所有人都将忠诚建立在薪酬之上，但我们确实都允许薪酬影响我们的决定(除非你是免费工作，而我错过了这一点？)继续摇摆，不管你的状态如何。不要犹豫，告诉面试官你帮助你的上一家公司取得了成功。
