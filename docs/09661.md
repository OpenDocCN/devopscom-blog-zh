# 移动应用的四个安全编码最佳实践

> 原文：<https://devops.com/four-secure-coding-best-practices-for-mobile-apps/>

移动应用已经成为许多公司的主要创新点，熟练的移动开发人员需求量很大。该企业要求开发人员利用最新的智能手机特性和功能快速创新，以推动高下载率，让用户惊叹，吸引客户并发展业务。面临着持续压力的开发团队可能会将安全性推到次要位置，将注意力集中在业务要求的必需特性和交付日期上。然而，许多开发团队已经发现如何通过确保开发人员理解关键的安全需求和编码最佳实践来更快地创新内置的安全性。我们与数百个开发团队合作，保护数千款移动应用，发现了四个常见的安全漏洞领域,很容易解决。

### 通过 HTTPS 使用 SSL

移动应用程序开发人员可能会本能地使用超文本传输协议(HTTP)进行网络通信。然而，HTTP 将私人用户信息公开给威胁者拦截。开发人员可以通过使用 HTTPS 来解决这个常见的错误，它通过行业标准的 SSL 加密服务器发送和接收的数据。Android 开发人员可以使用可用的 NetworkSecurityConfig 文件为应用程序中的所有网络连接设置预定的配置，或者手动加入 HTTPS。iOS 开发人员可以使用应用传输安全(ATS)作为默认功能，强制 iOS 应用中的安全通信，并防止移动应用和服务器之间的任何不安全连接完成。

### 验证证书的内容

证书通过在执行连接时强制执行额外的验证，为 HTTPS 连接增加了额外的安全层。证书包括签署它的证书颁发机构(CA)和应用程序已知或接受的主机名列表。在执行与服务器的连接时验证这些组件的应用程序大大降低了窃取凭据和敏感数据的中间人(MITM)攻击的风险。

为了验证证书是否由有效的 CA 颁发，Android 开发人员可以参考运行移动操作系统的设备上包含的预配置 CA 列表。Android 开发人员可以实现诸如 HostnameVerifier 之类的本地类，以在他们的应用程序中实现正确的主机名验证。

iOS 开发人员可以利用提供内置验证的 ATS，或者使用 NSURLSession 类中的方法来手动编写这些代码。

### 避免对移动应用的资源进行硬编码

攻击者经常在移动应用的源代码中使用硬编码信息来利用用户。例如，攻击者可以使用存储在应用程序文件中的凭据来访问用户帐户。攻击者还可以找到硬编码的 API 密钥或 URL 来收集私人数据或完全接管应用程序。移动应用程序开发人员可以通过不将密钥、密码和 URL 硬编码到源代码中来防止这些区域受到损害。在后端加密传输和解密数据也增加了一层额外的安全性。

### 使用最新的加密技术保护移动用户

SHA-2、RC4 和 DES 等过时的加密算法使得攻击者能够轻松破解看似安全的移动应用。使用旧算法还可能使移动应用不符合行业法规，使组织面临罚款或法律风险。开发人员可以通过使用最新的加密算法来避免 Android 和 iOS 移动应用程序的这个问题，选择那些适合特定应用程序场景的算法。在将二进制文件发布到商店之前对其进行签名时，请使用长度至少为 2048 位(最好是 4096 位)的密钥，在 Android 上，为加密实现生成随机值时，请使用 iOS 上的 SecureRandom 或 SecRandomCopyBytes。Android 开发人员可以利用 Keystore 类，iOS 开发人员可以利用 Keychain 服务来存储高度敏感的数据。dev 应避免不安全的操作模式、不正确生成的密钥和初始化向量(IVs ),以保证加密的信息不会被恶意参与者解密。
