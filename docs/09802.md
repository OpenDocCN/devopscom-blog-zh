# 开发人员在软件安全方面取得成功需要什么

> 原文:[https://devo PS . com/what-developers-need-for-software-security-success/](https://devops.com/what-developers-need-for-software-security-success/)

鉴于当今[不断演变的威胁形势](https://techstrong.tv/videos/interviews/prioritizing-application-security-matias-madou-secure-code-warrior)，各个领域的组织和企业现在都迫切需要生产安全的软件。犯罪团伙、职业攻击者和敌对的民族国家正在采用先进的战术，旨在利用程序、应用程序、网络甚至原始代码中的任何漏洞。攻击者不断寻找新的方法来绕过甚至是最先进的保护和防御。例如，许多人已经将他们的重点从提供恶意软件转移到损害凭证，或者对[供应链](https://www.cisa.gov/sites/default/files/publications/defending_against_software_supply_chain_attacks_508_1.pdf)发起有针对性的攻击。虽然这些高级别的入侵正在以更高的频率发生，但更基本的攻击也是如此，如勒索软件和 [SQL](https://devops.com/redgate-boosts-team-productivity-in-its-sql-coding-and-formatting-tool-sql-prompt/) 注入，这些攻击多年来一直是网络安全防御的祸害。越来越明显的是，虽然网络安全平台和防御是防御现代攻击的关键组件，但真正需要的是可以部署的无漏洞的安全代码。这就需要有安全意识的开发人员具备经过验证的安全技能。

大多数开发人员表示，他们愿意捍卫安全性，并致力于更高标准的代码质量和安全输出，但如果没有大量的支持，以及对雇主和组织经常用来评判他们的传统指标进行改造，他们就无法做到这一点。

### 为什么大多数开发人员不优先考虑安全性

多年来，编码最佳实践一直在不断发展，以响应业务需求和市场趋势。在过去，大多数应用程序是使用所谓的瀑布开发模型创建的，软件工程师在进入下一个开发阶段之前，努力使他们的代码准备好满足一系列正在进行的里程碑或目标。瀑布式开发倾向于支持这样的程序开发，即在开发过程中已经满足了所有先前的里程碑，并且在为生产环境做好准备时没有 bug 或操作缺陷。但是进展缓慢，从开始一个项目到到达终点有时需要 18 个月甚至更长时间。

敏捷方法倾向于取代瀑布方法，更加强调速度。接下来是 DevOps，它将开发和运营结合在一起，以确保程序在完成最终开发调整后立即投入生产，从而实现更高的速度。

随着业务环境的发展，将速度置于安全之上——以及除功能之外的几乎所有东西——是必要的。在一个基于云的世界中，每个人都一直在线，每几秒钟就可能发生数百万次移动交易，因此尽快部署软件并将其纳入持续集成和持续交付(CI/CD)渠道对于企业成功至关重要。

这并不是说组织不关心安全性。只是在大多数行业都存在的竞争激烈的商业环境中，速度被认为更重要。能够匹配这一速度的开发人员蓬勃发展，成为评判他们工作表现的主要手段。

现在高级攻击急剧增加，部署易受攻击的代码正成为一种负担。这种偏好再一次发生了转变，安全性越来越成为软件开发的首要焦点，速度紧随其后。事后逃避安全不仅危险，还会减缓软件部署的过程。这导致了像 DevSecOps 这样的程序的出现，这些程序试图将速度和安全性结合在一起，以帮助生成安全的代码。但是，如果没有组织的大量帮助和支持，受过纯速度训练的开发人员不可能成为安全专家。

### 开发者需要什么

好消息是，大多数开发人员希望看到向安全编码的转变，以及作为开发过程一部分的安全性的重新优先化。今年早些时候，Evans Data 对活跃在世界各地的 1200 多名专业开发人员进行了一项全面的调查，绝大多数人表示他们支持创建安全代码的概念。大多数人还期望它成为他们组织的优先事项。然而，只有 8%的受访者表示编写安全代码很容易完成。这给大多数组织的开发团队留下了很大的改进空间，在什么是需要的和为了达到那个目标需要什么之间。

简单地强制安全代码无法完成工作。开发团队需要培训、支持和改变软件工程师在他们的组织中的价值和判断方式。

最需要的是为他们提供更多更好的培训。它应该是定制的，以便经验较少的开发人员可以通过学习如何识别代码中经常出现的各种常见漏洞开始他们的培训，通过大量的实践学习和示例。与此同时，展示其安全技能的更高级的开发人员可以承担诸如高级威胁建模概念之类的任务。

团队合作也应该被强调，这样开发者社区可以互相帮助提高他们的技能。了解安全性的有技能且有意愿的开发人员应该被任命为安全性拥护者。作为冠军，他们的责任是帮助其他开发者提高他们的技能。虽然安全冠军几乎总是一个非正式的头衔，但他们应该得到这样一个重要职位应得的尊重、奖励和补偿。

除了资助和支持培训项目和安全冠军，包括给开发人员足够的时间远离编码，以便正确地参与这些项目，组织还需要改变评估开发人员的方式。奖励开发人员的主要标准需要从单纯的速度上转移。相反，评估可以奖励那些能够创建没有漏洞或利用的安全代码的开发人员。是的，速度也是一个需要评估的因素，但是最重要的是，代码需要安全并且没有漏洞。

对于大多数企业来说，产生不安全或易受攻击的代码不再是可接受的风险。事后逃避安全措施变得越来越无效。幸运的是，对抗这种令人不安的趋势的最佳武器是让开发人员社区生成攻击者无法利用的安全代码。大多数开发人员都愿意迎接这一挑战。他们只是需要适当的培训和支持来帮助实现这一目标。
