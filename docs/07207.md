**软件项目的交付方式出了问题。LinearB 的联合创始人兼首席执行官 Ori Keren 表示:“虽然 Git 和 CI/CD 工具采用了开发优先的方法，但项目管理和工程效率工具采用了自上而下的思维模式，实际上让开发人员的生活更加艰难。“开发团队需要工具，这些工具存在于他们的工作流程中，减少手工工作，并帮助他们专注于构建和改进。将软件交付智能视为 Git 关联和重建技术，它使项目管理更新变得多余，并内置了团队指标和运营洞察力。”**

****LinearB 解决开发团队目前面临的挑战:****

**项目管理工具对于计划来说是好的，但是一旦开发团队开始构建，就不会增加价值。如果没有手动的状态更新，他们就无法理解进度，并且当涉及到分支、拉请求和发布中发生的开发人员特定的工作时，他们就会失败。**

**效率&价值流工具错过了打开工程“黑匣子”的目标，因为这些指标不可行。由于“老大哥”文化的影响和使用执行仪表板来微观管理开发团队活动的诱惑，他们也不能在团队中获得采用。**

**LinearB 自动构建并可视化开放项目问题的详细进度时间表，无需人工输入，并同步项目系统的更新，消除了代价高昂的开发中断。该平台自动化了团队和项目绩效数据的收集和显示，这些数据通常需要数据工程师花费数小时进行手动编译。**

**开发人员可以深入了解他们在 Slack 和 Git 中的工作，开发团队领导可以从第一个项目板上获得清晰的可见性，每个问题都有完整的 Git 活动时间表，工程副总裁可以获得基于团队的可操作指标，从而推动日常改进。**

****结果是一种新的数据驱动的方法，以团队方式构建和交付软件:****

**更好的决策 : LinearB 标记哪些团队和项目需要帮助，以便开发领导知道他们的时间和注意力每天可以在哪里做得最好。**

**更快的交付:团队实时收到项目风险、延迟和依赖性的警告，以便他们能够更快地纠正和交付更多功能。**

**提高可见性:开发主管现在有了团队绩效、效率、质量和重点领域的衡量标准，明确了在自动化方面要做什么、何时招聘以及在哪里投资。**

**更好的沟通:指标是商业领袖的语言。LinearB 将最重要的基于团队的度量可视化，如周期时间，以帮助将工程翻译给非技术高管。**

**有意义的会议 : LinearB 为站立会议、回顾会议、冲刺计划会议和发布计划会议提供数据驱动的仪表盘，确保对话以事实为基础，时间得到有效利用。**

**异步开发: LinearB 让每个人都能实时了解项目，无需手动更新、中断或状态会议，让混合远程团队能够异步工作。**

**“LinearB 将我们的 Git 活动和项目问题之间的点连接起来，为我们的团队提供了更大的日常工作背景，并帮助我们更有效地交付，”Grin 的工程副总裁布伦特·巴特利特说。“我使用团队绩效报告来把握我们远程团队的脉搏。它们是自动生成的，省去了我过去花在从所有不同来源手动收集数据上的时间。”**
