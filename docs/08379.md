# 分阶段推出软件交付的注意事项

> 原文:[https://devo PS . com/dos-and-donts-of-phased-rollouts-software-delivery/](https://devops.com/dos-and-donts-of-phased-rollouts-software-delivery/)

当涉及到应用程序稳定性和用户体验时，“快速移动和打破东西”的方法是一个灾难的处方。为了保护品牌声誉，并领先于不断变化的需求和客户期望，领先的硅谷公司的软件工程团队采用了分阶段部署，这是一种软件开发方法，涉及一次向较小的用户群而不是一次向所有用户逐步实施新功能。

分阶段推出有助于公司在进行变更和软件更新的同时，为用户维持最高质量的[客户体验](https://hbr.org/2007/02/understanding-customer-experience) (CX)。当重要的特性和更新没有立即投入生产时，组织的软件工程团队不必立即处理所有潜在的实现问题。通过分阶段推出，工程团队可以在流程的早期发现问题，以便他们可以快速修复问题并恢复试验的推出，从而防止问题影响未来数百万用户。

虽然各地的公司都试图模仿这些硅谷技术领导者的应用开发技术，但许多公司无法成功实现大规模的分阶段推广。他们通常没有意识到的是，在实施分阶段部署方法之前，无论是针对 5%、10%还是 15%的用户群，都必须采取一些步骤来确保成功。下面，我将带您了解一个成功的分阶段推广计划的注意事项。

## Do:发布稳定性管理

为了在当今市场上取得成功，公司必须推动持续稳定的应用和网络体验。因此，在整个开发和测试过程中持续监控版本的稳定性以尽早发现问题并决定何时向用户推出功能是至关重要的。通过实时测量和监控应用程序的稳定性，发布经理为组织注入了责任感、可预测性和完全的安心。控制发布稳定性管理有助于工程师交付稳定的应用，推动世界级的数字体验。

最重要的是，[软件开发](https://devops.com/?s=software+development)团队必须不断改进移动和网络应用，增加新的特性和功能，让用户能够更好地完成他们想做的事情。这些改进为公司提供了在当今竞争激烈的商业环境中生存所需的竞争优势。此外，监控发布稳定性为工程团队提供了对影响应用程序稳定性和 CX 的每个问题的实时可见性。这种级别的可见性对于快速诊断和理解错误的根本原因至关重要，以便工程师可以在用户受到负面影响之前推出修复措施。

## Do:测试和早期访问

对于分阶段部署，工程团队必须运行 A/B 测试和质量保证(QA)测试。A/B 测试(也称为分割测试或桶测试)是一种比较网页或应用程序的两个版本以确定哪一个性能更好的方法。这包括尝试不同的截图、视频甚至不同的应用程序图标等功能，以衡量用户最喜欢哪种格式。

QA 测试通过检测可能不是技术性的可用性问题来确保组织交付最好的产品或服务。它测试任何影响最终用户体验的问题，从糟糕的导航到缓慢的加载时间或不清晰的网页内容。

对于分阶段推出，您还必须为精选用户提供对新功能的早期访问，以便在发布前收集反馈并消除关键问题。许多软件产品通过在某处包含一个小复选框来收集用户反馈，要求用户通过发送他们如何使用应用程序的见解来帮助改进软件。此外，社交媒体已经成为收集坦率用户反馈的不可否认的途径。我们都见过这种情况——当一个平台在其他用户之前为某些用户改变其界面时，用户会在社交媒体网站上发布他们是否对新的更新感到满意。从历史上看，当进行重大更改时，会有强烈的反对意见，然后公司必须决定是保留、更改还是取消对平台的新更改。

如上所述，通过应用程序稳定性监控工具，您可以看到新功能或实验是否会带来新的稳定性问题，并做出决定是撤回还是推广到更大的用户子集。通过进行测试并只允许特定用户群的早期访问，软件工程团队可以在发布速度和应用程序稳定性之间取得平衡，从而尽可能创造最佳体验。

## Do:在投入生产之前考虑一个阶段

当您发布新特性时，您会希望在它们通过 QA 测试后，在类似生产的环境中对它们进行测试。这就是所谓的分期，它已经成为硅谷公司的一个流行概念。登台是用于软件测试的生产环境的近乎精确的复制品。试运行是指在应用程序部署之前，在类似生产的环境下测试代码、构建和更新，以确保质量。

一个类似生产的环境类似于当应用程序为用户发布到生产中时，但它还没有公开。它提供了更新或新功能在生产环境中如何运行和外观的准确视图，以避免只有在生产环境中才会出现的小故障。阶段化可以包括手动测试，或者在某些情况下，工程团队使用自动化软件测试，这种测试实现起来成本更高，可能只有大型企业才能实现。

在这个阶段，软件工程师应该对新功能进行额外的测试和实验，以确保这些功能在发布到生产或更大的用户群时不会损害应用程序的稳定性。你最不希望的就是一个新特性破坏用户体验。

## 不要:尝试在每一个 Bug 或错误出现时就修复它

由于工程团队通常没有足够的带宽来修复每一个可能出现的编码错误，他们需要优先处理并修复直接影响客户的错误，以保护收入。尽管组织应该能够看到每一个错误或缺陷，但并不是所有的错误都值得修复。这导致了困扰开发者效率的信噪比。

这就是应用稳定性管理发挥重要作用的地方。然而，目前市场上的许多应用程序稳定性监控解决方案可以帮助识别错误，但让开发团队来确定哪些需要优先处理。工程团队必须快速确定哪些错误是最迫切需要修复的，因此他们需要可靠的工具，如应用程序稳定性和错误监控平台，为他们提供实时做出这些重要决策的可见性。

## 逐步交付的分阶段推广

分阶段部署允许软件工程师不断优化数字体验，而不会影响整个用户群和品牌。一旦掌握了，分阶段的推广就可以作为软件工程和开发的尖端渐进交付方法的一部分。

渐进交付方法包括一天多次将代码(有时是未完成的代码)交付到生产环境中，这超过了许多公司的风险承受能力。为了成功做到这一点，工程师必须能够控制谁可以在何时发布版本，以及哪些用户可以在何时访问某些功能和更新。分阶段的推出提供了一种方法来保护大多数客户免受在快速发布速度下发生的故障的影响。为了更快地发布新的软件版本，工程师必须在 QA 和测试上花费更少的时间，但这并不意味着他们应该被完全忽略。然后，通过首先向一小部分用户交付新版本，您可以更有效地管理风险，并相应地进行迭代。

重要的是要指出，如果工程团队正在做微小的改变，没有用户会看到或关心，你不需要使用分阶段的展示方法。也就是说，如果这是一个重大的修饰性或操作性的变化，比如对 UI 的更新，那么分阶段的推出是至关重要的。分阶段部署策略是大规模更新软件或应用的关键，这是在当今竞争激烈的市场中成功推出的必要条件。
