# CNCF 调查预测 Wasm 增长势头

> 原文:[https://devo PS . com/cncf-survey-predicts-growing-wasm-momentum/](https://devops.com/cncf-survey-predicts-growing-wasm-momentum/)

云原生计算基金会(CNCF)对 93 名开发人员进行的调查发现，超过四分之一(28%)的人正在他们的云原生开发项目中使用 WebAssembly (Wasm ),另有 36%的人计划在未来 6 到 12 个月内这样做。

本周公布的调查实际上是一项民意调查，发现 42%的受访者正在开发或计划开发使用 Wasm 的服务器端应用程序，而 48%的受访者正在将服务器端工作负载与客户端应用程序结合或计划结合。

Wasm 最初是由万维网联盟(W3C)在五年前开发的，目的是为执行 JavaScript 代码的浏览器创建一种通用格式，现在 Wasm 开始被用来快速构建可以部署在任何服务器平台上的轻量级应用程序。在最初引入 Java 编程语言大约 25 年后，能够编写一个应用程序并将其部署到任何地方的承诺终于实现了。

调查发现，70%的受访者认为 Wasm 运行时是一种颠覆性的新兴技术。调查发现，与 Wasm 相关的主要编程语言是 Rust T1(59%)、Go (47%)和 JavaScript (46%)。

近三分之二(63%)的受访者表示，他们正在使用 Wasm 移植新应用并迁移现有工作负载，相比之下，37%的受访者表示，他们只是在创建新应用。将工作负载迁移到 Wasm 的主要原因是应用程序可移植性(78%)、更好的安全性(77%)和更高的运行时性能(70%)。

安全性排名很高，因为构建应用程序的现有方法依赖于软件组件的集合，而这些组件之间往往缺乏明确的界限。因此，恶意软件感染应用程序的所有组件变得相对容易。Wasm 代码在沙箱环境中运行，沙箱环境以某种方式隔离执行环境，从而消除恶意软件在应用环境中横向移动的能力。

调查发现，Wasm 应用的目标平台包括无服务器计算框架(63%)、边缘平台(54%)和网络(52%)。

采用的最大障碍是工具(46%)和缺乏标准(45%)。超过一半(56%)的受访者表示 Wasm 开发者工具缺少功能，难以使用。整整 81%的受访者表示，他们希望看到供应商提供功能全面且稳定的 Wasm 工具和项目。

Fermyon 的首席执行官 Matt Butcher 表示，虽然调查的受访者很可能代表更高级的开发人员，但很明显 Wasm 的势头正在开始形成。他指出，就像之前的 JavaScript 一样，Wasm 即将从客户端跳到服务器端应用环境。

不太清楚的是 Wasm 是否代表了云原生计算的下一个时代，或者仅仅是另一种类型的软件构件，将被用来与其他构件(如[容器](https://containerjournal.com))一起构建应用程序。

Wasm 不一定会很快取代现有的构建应用程序的方法，但它确实为开发运行速度更快的轻量级应用程序提供了一种替代无服务器计算框架的方法。现有的无服务器计算框架通常使用启动速度比平台上本地运行的 Wasm 代码慢得多的容器，这解释了为什么开发人员希望采用这种格式来提高整体应用程序性能。

无论如何，更多的 Wasm 代码将很快进入 DevOps 管道。问题是确定如何将它与通过这些管道的所有其他代码合并在一起。
