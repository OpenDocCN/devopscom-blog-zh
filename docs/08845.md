# Cortex 扩展微服务目录以改善协作

> 原文:[https://devo PS . com/cortex-expands-micro services-catalog-to-improve-collaboration/](https://devops.com/cortex-expands-microservices-catalog-to-improve-collaboration/)

Cortex 是一家跟踪微服务所有权的目录提供商，它今天宣布[增加了一项服务创建功能](https://www.businesswire.com/news/home/20211118005546/en/Cortex-Secures-2.5M-in-Funding-from-Sequoia----New-Reliability-as-Code-Platform-Provides-Comprehensive-Microservices-Visibility-and-Control-for-Engineering-and-SRE-Teams)，使开发人员能够使用模板在五分钟内搭建新服务。

此外，该公司还增加了一个 Cortex Teams 产品，以改善在相互依赖的微服务上工作的开发人员团队之间的协作。

Cortex 刚刚筹集了 250 万美元的额外资金，它声称自去年 5 月首次推出以来，其客户群增加了 10 倍。这些客户包括 Rappi、Grammarly 和 8×8。

Cortex 首席技术官加内什·达塔表示，在软件开发变得越来越复杂的时候，该公司的微服务目录使开发人员、开发人员工程师和安全专业人员的团队更容易合作。虽然基于微服务的应用程序比单一的应用程序更容易更新，通常也更有弹性，但基于不同开发团队创建的相互依赖的微服务来构建应用程序，在构建和维护方面可能更具挑战性。

在某些情况下，组织甚至可以发现他们组织内的不同团队已经构建了冗余的微服务。部分原因是许多 IT 团队仍然依赖电子表格来跟踪哪些微服务正在构建，哪些已经部署在生产环境中。

相比之下，Cortex 会自动将驻留在开发人员工具中的数据导入到 surface 中，组织中谁负责维护哪些微服务。Cortex 仪表板还使 DevOps 团队能够看到哪些开发人员在发生意外事件时随叫随到，并监控和跟踪微服务正在经历的延迟率以及哪些漏洞尚未修复。

该平台的核心是 Cortex Query Language (CQL ),这是一种特定于领域的语言，允许 DevOps 团队定义粒度规则，这些规则可应用于例如待命轮换、安全漏洞、包版本、服务级别目标(SLO)和微服务的整体健康状况。工程师可以使用 YAML 文件形式的代码来设置跨团队和服务类型的可靠性标准。

还可以根据相关的[指标](https://devops.com/?s=metrics)将等级分配给特定的微服务。这种方法允许 DevOps 团队使用记分卡来应用游戏化技术，以鼓励开发人员不断改进他们的微服务。这种能力让其他开发者清楚地知道哪些微服务是最可靠的。

基于微服务的应用程序主宰企业市场可能还需要一段时间。在接下来的十年中，单片应用程序将在生产环境中运行。然而，这些单块应用中的许多实际上正在被转换成更大的微服务，在最终被重新设计成一组逻辑连接的微服务之前，其他应用可以通过应用编程接口(API)调用该微服务。

然而，有一点是清楚的，随着微服务数量的增加，精确确定高度分布式应用程序开发团队已经可以使用哪些功能变得更加困难。
