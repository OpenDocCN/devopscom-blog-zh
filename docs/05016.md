# 2020 年预测:DevOps 赋权年

> 原文:[https://devo PS . com/predictions-2020-the year-of-devo PS-empowerment/](https://devops.com/predictions-2020-the-year-of-devops-empowerment/)

从十年中期开始，似乎每年都有这样或那样的出版物宣称这是“DevOps 年”请允许我证明这真的会在 2020 年发生。

原因？IT 领导者最终赋予他们的工程师全方位服务所有权。

在这个世界上，速度、质量和精确度将继续决定持续成功和无情失败之间的差异——特别是在为所有利益相关方提供一流的端到端体验方面——全方位服务所有权是一个现实和必要的问题。更不用说常识了。

开发和交付能够跟上步伐并满足业务需求的产品和服务依赖于在不牺牲质量的情况下实现更快的发布周期。但是以不断增长的速度运行可靠的系统带来了巨大的挑战，软件团队可以通过改变他们的服务所有权策略来应对这些挑战。好消息是 DevOps 实践已经获得了相当大的吸引力，因为它们帮助团队以更高的速度交付，同时降低了风险。与引入的任何变更一样，这种支持需要时间和实践来满足您的需求。

## **编码，运输，拥有**

全面服务所有权赋予工程师对他们编写的代码和他们在生产过程中开发的服务更大的责任。不应该低估拥有和操作自己的代码并对所有可交付成果负责的重要性，因为毕竟是工程师应该——也有权利——负责。他们是设计架构和系统、编写软件和操作手册的人。他们肩负着整体服务质量的重任。当问题出现时，他们是随叫随到的人(正如墨菲定律所暗示的，总是在半夜)。

如果他们是编写代码的人，那么他们就是发布和拥有它的人。工程师将代码“扔过围墙”投入运营，或者依靠站点可靠性工程(SRE)团队来保证野外服务性能的日子已经不再适用，更不可行了。

考虑到预计到 2025 年[devo PS 市场将增长 18%，达到近 130 亿美元](https://www.grandviewresearch.com/press-release/global-development-to-operations-devops-market) ，这一点对于未来尤为重要。随着 DevOps 工具和解决方案在 it 运营中变得越来越根深蒂固，管理这种增长以及它将带来的不可避免的混乱的最合理、最积极的方法是让 IT 拥有全面的服务所有权。

## **全业务所有权背后的力量**

发生这种情况有三个原因:

**问责**

这是一个简单的事实；消费者隐含地期望他们所使用的服务能够无缝地工作，并且具有最小的延迟。考虑到当一个页面加载缓慢或返回 404 错误时，消费者会简单地转移到另一个网站，增加企业竞争优势的回报是巨大的。

全面服务所有权促进了更高质量的工作，因为它让工程师清楚地了解他们创建的代码和服务是如何执行的，以及如何影响客户的日常活动。

**可靠性**

当火箭坏了的时候，要求建造火箭的科学家去修理它，这不是火箭科学。同样，当发生严重事故时，企业可以通过立即引入编写代码或创建服务的工程师来减少停机时间和对客户的影响。

全面服务所有权提供了另一个重要优势:IT 团队的整体机构知识。单个工程师从代码审查、随叫随到的移交、每日站立、随意的信息共享和其他建立知识和专业技能冗余的练习中受益匪浅。

**持续改进**

工程界的一个常见误解是“随叫随到”是“永远在线”的同义词这本身是适得其反的，因为它会直接导致警觉疲劳。在一个理想的世界中，团队将在他们的开发工作中建立“待命”的时间，这是一种减轻警报疲劳的实践，也是全服务所有权所鼓励的。

这也完全符合工程师文化的一个组成部分:致力于不断完善和改进他们的代码、产品、服务和警报。最有效的流程不仅仅是解决问题，它还包括通过自动化预防问题所需的工作来提前解决问题，让工程师有时间专注于最重要的工作。

## **全方位服务所有权的五大优势**

1.  代码的质量将会提高。
2.  工程师会知道他们什么时候下班，警报会由另一个具有同等专业知识的开发人员负责。
3.  通过对代码库的深入理解，团队的知识和技能将会增加。
4.  服务将变得更加可靠。
5.  运营将变得更加敏捷。

当工程师能够编码、运输和拥有信息技术时，企业就能更好地提升客户体验，减少中断带来的影响，从而更专注于创新。通过定义和细化角色和职责，全方位服务所有权还减少了与事件相关的混乱，消除了孤岛和不必要的层级，并最终促进了授权和问责文化。
