# 长期云原生安全性的实用方法

> 原文：<https://devops.com/practical-approaches-to-long-term-cloud-native-security/>

关于如何保护现代云原生工作负载的建议并不缺乏。到目前为止，大多数从事云原生部署的开发人员和 IT 工程师都听说过关于 DevSecOps、左移安全性、多层防御和动态基线的所有咒语(仅举几个最近推动 IT 安全对话的关键概念为例)。

一件事是谈论安全最佳实践，另一件事是设计一个不仅可行而且易于实施的云战略。更具挑战性的是规划一个策略来促进长期的安全最佳实践。

请继续阅读关于应对这些挑战和设计长期安全策略的技巧。

## **安全是一场持久战**

很容易认为安全是你必须实时做的事情。毕竟，威胁和攻击通常是突然发生的，快速做出反应是防止严重损害的关键。

如果您的安全策略以发现和补救出现的威胁为中心，那么您最终会陷入本质上的中断/修复模式。你总是被动反应，而不是主动出击。

更有效的安全策略是首先最大限度地减少您面临的威胁。这种战略需要长期规划。

当然，您将始终需要准备好检测一些您没有预料到的威胁，并在发生违规时快速做出反应。没有完美的安全计划，意外有时仍会发生。但是，通过尽可能关注长期安全解决方案，阻止大多数威胁成为现实，您最终会获得更安全、更可靠的安全态势。

## **构建长期安全战略**

实施长期保护您的安全策略实际需要什么？这一切归结为四个要素:数据、基础设施、流程和文化。

**数据**

对于许多组织来说，存储在云中的数据是带来最大风险的工作负载。这就是为什么似乎永远不会有关于重大安全漏洞的头条新闻，这些安全漏洞涉及窃取存储在云中的敏感数据。

因此，减轻云中数据的威胁是长期安全的关键要求。这方面的一些最佳做法包括:

*   **评估您的数据** :数据安全始于了解您的数据中有什么，以及这些数据有多敏感。数据编目可以帮助这一过程，但亚马逊 Macie 等工具也可以，它有助于通过基于云的数据识别个人身份信息。
*   **访问控制** :访问控制框架是抵御基于云的数据泄露的另一条重要防线。请记住，仅仅启用 IAM 是不够的；您还必须审计 IAM 策略，以避免可能导致违规的错误配置(这是 Capital One 等公司吸取的 [教训)。](https://krebsonsecurity.com/2019/08/what-we-can-learn-from-the-capital-one-hack/)

**基础设施**

如今，我们倾向于将基础设施——即托管工作负载的基于云的和/或本地数据中心——视为解决方案堆栈中相对通用和可互换的部分。虽然一般来说，没有一个云或数据中心天生就比另一个更安全，但您设计基础架构的方式对您长期保护现代工作负载的能力起着关键作用。

基础设施方面的长期安全最佳实践包括:

*   不可变的基础设施 :采用基础设施平台，如 ECS 或 EKS，以不可变的方式轻松部署和更新应用程序。不可变的构建块可能不适用于所有用例，但是在可能的情况下，它们减少了您为了部署或更新应用程序而必须应对的变量和移交的数量(并且变量和移交越多，产生安全漏洞的疏忽的机会就越大)。
*   **访问控制** :虽然 IAM 框架可以帮助防止某些类型的未授权访问，但是您也可以使用访问控制列表来提高基础设施的安全性。最佳实践是采用白名单方法，默认情况下阻止所有流量，只允许显式信任的主机。
*   **避免膨胀** :云原生基础设施可以轻松启动虚拟机、数据库或其他您并不真正需要的基础设施组件，并让它们保持运行。这样做可以增加你的攻击面。因此，识别和关闭未使用的基础设施对于确保长期安全至关重要。这也有助于削减云成本，但这是另一个话题。
*   **奇偶校验** :为了让开发人员、ITOps 和安全团队能够有效地沟通和协调，他们都应该在相同类型的基础设施环境中工作。例如，如果您使用本地数据中心进行开发，然后将生产工作负载部署到云中，协作将变得更加困难。通过尽可能在您的基础架构中创建奇偶校验来避免这种情况。

**流程**

流程是为云原生工作负载创建长期安全计划的第二个关键要素。显然，您使用的过程将部分反映您特定的工作负载和工具。但是无论您的情况如何，您的流程都应该考虑以下安全目标:

*   一致性 :你的 IT 组织可能由不同的团队组成，每个团队都有不同的职责和议程。尽管如此，在开发、测试、试运行和部署应用程序时，所有团队都应该尽可能遵循相同的程序。这样做可以减少可能破坏安全性的变量或疏忽的机会。您可以通过在整个组织中创建和实施 IT 治理过程来帮助实现这种一致性。
*   **左移** :流程是 IT 运营中左移安全最有效的部分。左移安全性意味着从应用交付管道的开始就将安全性作为优先事项。然而，重要的是(这是一些组织忽略的一点)，正确地进行左移安全性也意味着在管道的后面阶段加强安全性操作。不要只是将安全转移到左边，然后就到此为止；目标是在交付链的所有阶段将安全性作为首要考虑因素。

**文化**

文化是很难形式化的东西；事实上，如果你试图向员工灌输文化价值观，你就有可能损害建立有机文化的整体意义。相反，您希望鼓励您的团队成员自然而然地接受促进安全文化的价值观。有助于实现这一目标的策略包括:

*   **漏洞奖金** :通过为您组织的工作负载提供漏洞奖金，激励您的团队寻找并解决安全问题。为此，“bug”应该广义地定义，不仅包括代码中的安全 bug，还包括您的基础设施、工具或应用程序中可能导致安全漏洞的任何类型的配置问题。这样，您的 bug 奖金不仅包括开发人员，还包括 IT 组织的所有成员。
*   **鼓励共享所有权** :每个工程师都拥有特定应用程序、基础设施或流程的文化会限制透明度和解决安全风险的机会。相反，努力促进 IT 资产的共享所有权，这样每个团队成员都知道他或她的工作可能会被其他人审查，这有助于鼓励最佳实践并减少安全问题。本质上，这些是 DevOps 和 DevSecOps 优先考虑的价值的一部分。

## 结论

通过设计促进安全性的基础设施、流程和文化实践，您可以让您的组织长期优化安全性。提前规划安全性是避免漏洞被发现后立即响应的中断/修复循环的唯一方法，这种循环会让您总是踩水，而不是推针。你不可能防止所有的安全问题，但是你可以通过在你的基础设施、流程和文化中增加安全性来大大减少出现的安全问题。
