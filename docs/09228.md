# 低代码 SaaS 解决方案的核心需要安全性

> 原文:[https://devo PS . com/low-code-SaaS-solutions-need-security-at-their-core/](https://devops.com/low-code-saas-solutions-need-security-at-their-core/)

更快的集成。加速软件交付。提升用户体验。这些只是组织通过将低代码平台、应用和解决方案整合到软件开发和业务流程管理中而获得的三大好处。

低代码平台并不新鲜，但在过去两年里，随着公司面临满足利益相关者对更多数字化转型需求的挑战，对该技术的需求激增。借助低代码工具，公司发现他们可以通过更快的业务应用交付来扩展数字化转型工作。此外，使用低代码工具，创新所需的时间大大减少了。

低代码解决方案不是过时的趋势。 [在最近关于全球云收入](https://www.gartner.com/en/newsroom/press-releases/2021-11-10-gartner-says-cloud-will-be-the-centerpiece-of-new-digital-experiences) 的报告中，Gartner 预测，到 2025 年，低代码和无代码的使用将“增加近两倍”，并补充说“组织开发的 70%的新应用程序将使用低代码或无代码技术，而 2020 年这一比例不到 25%。”

与更传统的内部开发相比，低代码解决方案使组织能够以更快的速度和更灵活的方式生产工作解决方案和集成。集成曾经是一个劳动密集型的 IT 过程，需要在集成的两端进行定制开发。随着越来越多的组织采用“购买而不是构建”的软件开发方法，他们开始转向自助服务和其他具有“开箱即用”功能的解决方案来提高速度和效率。此外，供应商开发的低代码工具组件也提供了更好的用户体验。

### 对基于平台的低代码解决方案的需求日益增长

随着组织扩展其产品和服务，对低代码、基于平台的解决方案的需求将会增加。关注低代码的 SaaS 和 PaaS 解决方案对于寻求加速功能的组织来说可能是有效的。这些工具通常包括预构建的模型和模板，企业可以快速实施并轻松复制，以实现灵活性和多功能性。

必须具备与其他应用程序无缝集成的能力。同类最佳的低代码平台使用图形用户界面(GUI)以及供应商提供的环境中的 JSON 和 API 等行业标准接口，实现了多个应用程序之间的智能集成。直观的可视化用户界面和拖放功能允许企业定制解决方案，以满足他们的需求和偏好。

但是除了易用性和快速的功能性，组织还必须寻找内置安全性的低代码工具。

### 安全性:低代码工具的优先选择

从使用低代码平台进行应用程序开发，到使用无代码 SaaS 解决方案进行业务流程自动化，安全必须是使用低代码工具的组织的首要任务。

无论何时组织将外部应用程序或任何第三方 API 集成引入环境，该组织都完全依赖于供应商的安全实施。当组合各种解决方案来创建统一的产品或服务时，尤其如此。如果供应商不重视安全性，解决方案的其他部分可能会因为另一个领域的弱点而面临风险。

正如最近的 Apache Log4j 漏洞所显示的，问题可以有意或无意地进入。这一常用程序中的漏洞被发现具有巨大的网络安全攻击潜力。 [正如 CNET 警告的](https://www.cnet.com/tech/services-and-software/log4j-software-bug-cisa-issues-emergency-directive-to-federal-agencies/) ，“广泛使用的软件中的漏洞可能被网络攻击者用来接管计算机服务器，潜在地将从消费电子产品到政府和企业系统的一切置于网络攻击的风险之中。”

### 补救低代码工具安全风险的方法

组织不能对他们带入环境的每一个工具、解决方案或平台的安全性做出假设。他们必须在软件开发过程中或任何时候引入第三方应用程序或工具时，完成自己的安全尽职调查。

解决安全性问题的一个关键步骤是整合 DevSecOps，并将安全性转移到软件开发流程中。Log4j 漏洞将对 [DevSecOps 的需求置于聚光灯下](https://devops.com/how-log4j-becomes-a-serious-devops-problem/) 。在 DevOps 过程中采取左移方法并更早、更频繁地实现安全性有助于捕捉更多潜在的漏洞。在开发周期的早期引入 DevSecOps 是至关重要的，否则组织会有遗漏 bug 或漏洞的风险。在开发周期中，安全性不是防病毒场景；必须对所有代码进行漏洞扫描。

组织在探索将选项和解决方案引入其环境时，也需要进行尽职调查。寻找以安全性为核心的解决方案。在集成任何低代码解决方案之前，检查供应商的安全性方法。

在考虑供应商或解决方案时，要注意的一些安全措施包括:

*   [确保他们使用最新的 OAuth/2 模型](https://www.lumicademy.com/for-developers/) 用于客户端到服务器安全以及服务器到服务器安全或等效标准。

*   所有通信应至少利用 TLS 1.2(传输层安全性)并符合证书和加密密钥的行业标准。
*   所有哈希都应使用 HMAC 当前接受的标准，他们应执行密码安全规则，例如仅存储密码哈希，如果直接与客户帐户交互，则要求 2FA。

低代码平台和工具能够加速数字化转型，而无需依赖手工编码的内部解决方案。节省时间和成本的潜力是巨大的，但对于组织来说，确保在选择解决方案时不牺牲安全性也很重要。
