# 2021 年最佳——不工作时招聘

> 原文：<https://devops.com/hiring-when-youre-not-faang/>

随着 2021 年的临近，我们 DevOps.com 想要突出今年最受欢迎的文章。以下是我们 2021 年最佳系列的第二十期。

我们大多数人都不是 FAANG，我们也不应该表现得像这样。我将使用我们过去用来指代他们的术语:“网络怪兽”，这不是 100%准确，但在本文中，它不仅仅包括五个 FAANG 成员(脸书、亚马逊、苹果、网飞和谷歌)。在某种程度上，我们需要一个新的术语。网络怪兽没有明确包括苹果或花旗，但 FAANG 明确排除了花旗和易贝。但是这个博客是关于其他事情的，所以请接受我的道歉。假设“网络怪兽”意味着“大的、受欢迎的高科技公司，薪水高，简历好看。”

我们可以从几个角度来讨论“非网络怪物”，但我主要关注的是 [DevOps 招聘](https://devops.com/?s=DevOps+hiring)，并提及一些不相关的销售/营销技巧。招聘是一个容易的目标，因为网络怪兽提供的简历制作、薪酬和福利在业内极具吸引力，所以他们可以(至少现在)拒绝 99%的申请人，但仍然有很多候选人可供选择。大多数组织甚至在其中一个领域都无法竞争，尽管少数组织在薪资和/或福利方面可以。

我们 IT 界长期以来有一个坏习惯，就是要求我们的申请人比一般员工更合格。有时这很可笑；比如要求候选人在某项技术上拥有比该技术本身实际存在的时间更长的经验。当你要求不可能的事情时会发生什么？诚实、合格的申请人会自己选择，只有那些没有意识到这项技术存在时间不长的人和骗子才会申请。这只是糟糕的招聘政策，即使你是网络怪物。但在较小的组织中，每次招聘都会对组织产生较大的影响，这可能是毁灭性的。

在 DevOps 似乎更糟。事实上，这里列出的所有招聘项目在 DevOps 中似乎更糟，因为 DevOps 实际上是一个技能组合的噩梦。没有两个人对工作的定义是一样的，因为“工作”涵盖了一个应用程序从摇篮到坟墓的所有内容。应用程序开发的任何部分都是“DevOps”无法触及的。这是第一个问题。清楚地定义角色的职责，不要要求所有可能的事情。作为一个规则，一个可回答的专家不会成为 DevSecOps 专家，你也不应该期望他们是。我认为，至少，我们需要按照我的传统重点划分招聘……*开发*运营和开发*运营*。如果你在 GitLab 构建结果的时候编写代码和使用 Checkmarx，并且你配置它来进行测试…你不是一个开发*运营*工程师，即使那里有一些运营。你的优势需要在开发和应用配置上。你的工作描述需要关注这一点。

会有需要这一切的角色吗？是的，特别是在 SRE 或混沌工程领域；我看得出来。但肯定不是入门级的角色，甚至接近。接触整个 DevOps 工具链所需的技术范围的知识是多年积累的，并带着健康的好奇心。当然，你雇佣的人应该和现在做这份工作的人有相似的技能。这是对“扔进厨房水槽”综合症的一个很好的理智检查。

接下来是带回家的测试和面试中的共享编码。我们知道你在获得最好的人才方面有既得利益，但是如果你和主要是开发运营的人一起做编码练习，你就是在浪费每个人的时间。大多数开发*操作*工作都是用 PowerScript 和/或 YAML 文件 bash 完成的。其他偶尔使用的语言在核心工作面试中并不比其他偶尔使用的语言有更多的位置。“关键在于你如何思考”是一句常见的套话。我认为，申请人需要几个小时(据我所知，甚至几天)才能完成的任务，并不是评估未来员工思维方式的有效方式。而且，至少在一个案例中，我很确定面试解决了一个实际问题，而这个问题是员工们没有或者不能解决的。问应聘者他们会如何解决问题 x。问他们一些复杂的问题。他们不需要达到代码级别来告诉你他们是如何思考的。

最后，我向糟糕的营销点头致意:除非你真的在做某个网络怪物的交易量，否则脸书是供应商 X 的客户这个事实是完全不相关的。事实上，像政府一样，大多数供应商可以将一个或多个网络怪物标记为他们的客户，只是因为网络怪物太大了，他们在许多不同的团队中尝试了许多不同的工具。因此，将他们作为客户应该是无关紧要的，但如果不认为这是无关紧要的，应该是意料之中的。

你今天的团队很棒。雇佣更多像他们一样的人。你知道什么有效，你需要什么。回到定制招聘流程，提供更多你需要的东西，停止过度要求。有更多优秀的人在寻找角色；不要问无关的经历，把他们吓跑了。坚持核心。他们每天要做什么？要求，然后在面试的时候寻找其他有用的东西。让更多的好观众坐在座位上！
