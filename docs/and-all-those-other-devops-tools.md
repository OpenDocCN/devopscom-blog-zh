# …以及所有其他 DevOps 工具

> 原文：<https://devops.com/and-all-those-other-devops-tools/>

当我们开始为我们的新项目检查 DevOps [工具链](https://devops.com/?s=toolchain)时，我们发现了大量通常我们(很可能还有你)不会太注意的细节，因为它们并不重要——直到它们变得重要。然后我们都关注。这就是为什么在成为问题之前，我们将它们添加到选项*的工具链列表中。*

是时候停下来了，四处看看，确保你有一份制作秘制酱料的工具清单。简单的是 IDE。在过去的一小段时间里——比如说十年——我们让这在很大程度上成为一个团队或个人的决定，但是做事情所需的步骤和所使用的附加组件/库使得这越来越不属于个人的选择，而更属于至少一个团队，通常是一个组织的决定。我们是在项目层面上做的，所以功能上等同于团队。我们的 Java/Spring 应用程序是在 Eclipse 中完成的，因为你们大多数人都在为那个架构做同样的事情。但是我们新的 Godot 项目将在内置的 Godot 编辑器中(用于 Godot 脚本)和 VS 代码中用于 C#。同样，因为这在几种语言中都很流行，包括 C#。

事情是这样的——如果你使用 VS 代码，你应该保存一份详细的最新插件列表。我不喜欢他们通过命令面板安装整个扩展的方法，我想确保每个人手头都有需要的工具。在企业环境中，ROI 只是用来设置它，并在一个包中向开发人员分发他们需要的东西。但这只有在大多数开发者都在同一个平台上的情况下才是真正的可能。

正在使用什么 Git 桌面客户端？虽然它们都以相同的格式支持 Git 命令行，但是互相帮助使用 UI 来进行不同的操作，这种做法需要标准化。我们的团队变化不大，所以我并不特别担心这个问题。我们正式在用 [GitHub](https://github.com/) Desktop，但是只要我不是做技术支持的(因为你有一个你用不好的工具)，我不在乎别人是否坚持。你可能比我们更关心这一点——在这方面，我们更像一家初创公司，而不是企业。因为它们共用一个命令行，所以我们可以不关心。如果你，作为一个企业，正在提供现成的代码环境，那就是另一回事了。

几年前，我们不得不担心容器，但现在我们可以在本地使用 Docker，在生产中使用 Kubernetes。虽然 Godot 应用程序没有服务器组件(因此我们保留了 Java/Spring 应用程序)，但构建环境将是容器，因为我们希望能够在任何需要的地方进行构建。

继续摇吧。我只是想我应该把我们正在讨论的事情列出来——当然还有更多；这一部分集中在开发人员工具方面。我最终会更新最终的完整堆栈，但现在，当我查看其他一些区域时，这一点被搁置。*向保安人员挥手示意:
