# 如何从 DevOps 获得更大的价值

> 原文：<https://devops.com/how-to-gain-greater-value-from-devops/>

DevOps 不仅仅是部署和管理。充分利用 DevOps 团队及其活动实际上可以增强运营并提高客户满意度。

## 积极主动，而不是被动反应

假设您部署了一个新版本，遇到一些问题，然后做出反应。如果你能通过变得更加积极主动至少领先一两步呢？

检查您的指标并寻找趋势。随着时间的推移，您可以发现错误率或延迟的上升，并在问题出现时识别问题。理想情况下，当它们只是小问题时，你会发现它们，而不是当你已经需要按下紧急按钮或回滚时。向前看，看看拐角处有什么会让你处于一个更好的位置。

## 利用反馈改进运营

虽然投入时间和资源来自动化任务(如部署)的确是一件好事，但更好的做法是专注于另一项高价值的 DevOps 活动——将来自运营的见解引入您的开发管道。

将反馈和想法整合到您的流程中会让您处于一个更好的位置，尤其是因为已经有工具可以帮助您非常轻松地完成这些部署和其他任务。例如，一旦你把你的应用打包到一个容器中，你只需要按几个按钮来设置自动伸缩、负载平衡和路由。所有这些都实现了自动化，您可以专注于真正重要的东西—您的应用程序。

## 跟踪客户行为

这里有两个宝石——都易于设置和使用，出自同一位 [GitHub](https://github.com/) 作者——您可以使用它们来跟踪客户的行为，监控使用情况，并获得更好的见解以改进运营。

**Ahoy**—使用它来跟踪用户对您的网页的访问，以确定您想要添加到代码中的特定命名事件。您可以从服务器端和控制器、JavaScript 或通过简单的 API 来实现这一点。它使用两个可以轻松查询的表，加上一些智能的内置数据可视化功能。
**Blazer**—只需在 routes 文件中添加一个挂载点，就可以访问 web 应用程序，从而获得数据库的可见性。它提供了一个简单的界面来执行 SQL 查询并显示结果。

## 经常检查指标

一旦你设置好这些工具，不要止步于此。检查[指标](https://devops.com/?s=metrics)——如果可能的话每周检查一次——看看趋势是什么。提问。运营最酷的一点——不像你试图解决一个问题的工程和编写软件——是它并不总是被很好地定义。你可能一开始甚至不知道该问什么问题。然而，钻研度量标准，看看从使用和事件中出现了什么样的模式，不仅会使您的客户受益，还会反馈到您的开发周期中。这让你能够回头重新评估之前所做的决定，并了解你应该把注意力和精力放在哪里。

一定要加强你对 DevOps 实际能带你去的地方的关注。不仅仅是写自动化和部署脚本。分析和软件的协同是 DevOps 的未来。
