# Dynatrace 增加了安全门以促进 DevSecOps 的采用

> 原文:[https://devo PS . com/dyna trace-adds-security-gates-to-advance-devsecops-adoption/](https://devops.com/dynatrace-adds-security-gates-to-advance-devsecops-adoption/)

Dynatrace 今天在其可观察性平台中增加了[安全门功能，以便在应用交付管道中自动采用 DevSecOps 最佳实践。](https://www.businesswire.com/news/home/20211110005489/en/Dynatrace-Introduces-Security-Gates-Bringing-Automation-and-Intelligence-to-DevSecOps)

Dynatrace 负责产品管理的高级副总裁 Steve Tack 表示，安全门的功能与 Dynatrace 之前添加到该平台的质量门非常相似，因为现在每个版本都会被自动评估，以确保只部署安全的代码。

Tack 指出，Dynatrace 平台现在使用戴维斯人工智能(AI)引擎来实时扫描应用程序工作负载中的漏洞，然后根据它们带来的风险对它们进行优先排序。他补充说，目标是使组织更容易将安全审查纳入应用程序开发过程，而不一定要求开发人员成为安全专家。

相反，核心 DevOps 平台应该能够在应用程序部署到生产环境之前发现可以补救的问题，Tack 说。

Dynatrace 早期增加了自动识别存在最大安全风险的软件库和开源包的功能。安全门功能现在扩展了这项工作的范围，将使用定制代码构建的应用程序包括在内。

一般来说，大多数组织都在寻找采用最佳实践的方法。面临的挑战是在不过度降低应用程序开发速度的情况下实现这一目标。事实上，最近由 Dynatrace 委托进行的一项[调查](https://devops.com/dynatrace-survey-sees-continuous-delivery-challenges-ahead/)发现，平均而言，组织期望在未来两年内将软件发布频率提高 58%。然而，近四分之一(22%)的受访者承认，他们经常在满足应用程序部署需求的压力下不得不牺牲代码质量，这通常包括安全问题。

目前还不清楚安全审查将在多大程度上自动包含在质量保证过程中，但 Tack 表示，目标应该是尽可能限制应用程序部署过程中的人工参与。Tack 指出，如今许多开发团队遇到的问题是，应用程序被网络安全团队拒绝，网络安全团队在应用程序部署前对其进行审查。他补充说，结果是，开发者竞相加快应用程序的开发速度，结果却看到它们被拒绝，并在最后一刻被退回给他们。

在一系列引人注目的软件供应链违规事件发生后，大多数组织对 DevSecOps 的关注度大幅提高。通常不太清楚谁负责应用安全，但随着 DevOps 平台的不断发展，可能会有一天安全问题会在每个 DevOps 工作流中得到例行解决。

与此同时，需要更加关注弥合开发人员和安全团队之间的鸿沟。鉴于安全专业人员的长期短缺，毫无疑问，DevOps 团队需要为应用程序安全承担更多责任。挑战和机遇是找到一种尽可能减少摩擦的方式来实现目标。
