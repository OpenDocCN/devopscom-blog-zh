# 如何实现聚合工具链

> 原文：<https://devops.com/how-to-achieve-a-converged-toolchain/>

最近几年，开发和交付软件的工具激增。在单个管道中，公司将拥有独立的工具来构思、计划、编码、验证、配置、发布和监控新软件，并且每个管道可以拥有用于每个单独阶段(即构思)的不同工具。这需要大量的工具，并且提供应用程序开发和交付过程的可见性是困难的，会带来许多挑战。

消除这些挑战的关键是开发一个融合的工具链。

## 什么是工具链？

在软件交付的背景下，工具链是管理软件开发的计划、执行、质量和交付的工具的集合，从想法产生一直到生产。开发的每个阶段都有许多工具，对于一个组织来说，使用多个测试工具、多个用于 backlog/用户故事计划的工具或者多个缺陷跟踪工具并不罕见。

默认情况下，每个工具都在一个竖井中运行——并不真正与其他工具交互，并且对其他组不可见。如果没有一个融合的工具链，公司在应用程序开发和交付过程中只能看到支离破碎的视图。这意味着开发团队不知道计划团队在做什么，运营团队看不到开发团队发生了什么，等等。当然，开发团队中的某个人可以走到规划团队面前问:“发生了什么事？”但是这停止了势头，减缓了整个进程。

## **融合工具链的优势**

有了聚合工具链，分散孤立的工具可以在一个互联的生态系统中工作。帮助台票证可以将错误与从事该功能的特定开发人员相匹配，并且可以将计划与应用程序、环境和/或生产变更请求相匹配。工具数据的融合实现了组之间的可见性——允许每个组使用他们自己的工具，但同时使用从工具链的其余部分增加的数据——这种可见性实现了组之间的协作。连接的工具链还使每个团队能够继续使用他们自己的工具进行他们的工作，并且团队可以评估软件交付的整个过程，并监控移交、浪费和低效率。这种整体可见性带来了前所未有的效率。

没有聚合的工具链，团队很可能会遇到时间和计划方面的问题。例如，当测试团队准备处理项目中他们的部分时，他们的第一个问题通常是，“它在什么环境中？”以及“它包括哪些功能？”不幸的是，这些可能是最难回答的问题。但是一个聚合的工具链允许测试团队很容易地看到未验证特性的确切位置，并沿着管道快速地移动它。

## **实现收敛**

通过整合工具链，您可以整合和互连工具链中的工具。为此，您需要完成三个步骤。首先，您需要将工具中的数据关联并集成到一个公共数据模型中。这将有助于创建一种标准的方式来看待特性、测试、环境、发布过程和交付阶段。所有这些工具都需要输入到它们的应用程序交付部分，以便您可以将所有数据关联在一起。

一旦你有了这些，第二步就是在工具之间同步数据，并将所有数据放入你的公共数据模型。这将为您提供您的团队所需的可见性，以便您可以将特定的功能与他们所处的环境联系起来，或者知道哪些功能仍然需要构建。

第三步是创建工作流程。这不仅允许您引入数据，还允许您对数据采取行动，并根据事件对事情做出反应。例如，当运营团队将一个新特性转移到生产中时，他们可以将其标记为生产就绪，所有工具和所有团队都将这样标记。这确保了每个团队都在同一个页面上，没有任何工作会过时。

此工作流程也有助于解决向组织服务台或内部应用程序技术支持部门报告的问题。如果没有聚合的工具链，服务台工作人员必须创建一个工单，打电话给开发人员团队，并试图找到可以解决问题的人。但是，使用聚合工具链，帮助台工作人员可以自动为该系统的开发人员创建警报。然后，开发人员能够很快发现错误，开始进行热修复，并向最终用户发送一个通知，告知预计的完成时间。所有这些都在聚合工具链的工作流中完成，任何人都不必亲自去找另一个人来寻找答案。

大多数组织将自己限制在各自为政的状态中，这给跨团队的沟通和工作流程带来了问题。团队没有致力于向客户交付价值的项目，而是在移交上花费了大量的时间。虽然了解项目所处的位置很重要，但是团队应该专注于向客户交付价值，而不是收集状态更新。

这种应用程序开发和交付流程支离破碎的日子再也无法忍受了。每一个组织的工具链都应该专注于消除执行障碍，并自动化开发和持续交付过程。为了取得这样的成功，团队需要能够在他们选择的工具中完全透明地工作，同时还能够看到他们的工作如何适应工具链的其余部分的更广泛的背景。借助精心设计、集成和自动化的 DevOps 工具链，开发和运营团队成员能够根据共同的目标和指标开展合作，以确保向客户按时交付高质量的应用。
