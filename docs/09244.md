# 利用全栈可观察性克服 IT 复杂性

> 原文:[https://devo PS . com/comprising-it-complexity-with-full-stack-observability/](https://devops.com/overcoming-it-complexity-with-full-stack-observability/)

踏上数字化转型之旅的组织往往会开发复杂的基础设施。他们更新其传统应用，同时添加多云、虚拟和[云原生](https://containerjournal.com/?s=cloud-native)功能。最终，IT 专业人员发现自己管理着多样化、复杂和分布式的网络，包括云、系统、应用程序和数据库基础架构。

为了控制由此产生的复杂性，组织倾向于积累监控和管理工具。目标是简化系统监督——但相反，由于团队使用各种各样的工具来管理他们的网络或基础设施，孤岛往往会发展。

这种零敲碎打的方法恶化了操作盲点，并延迟了问题的解决。这也增加了安全风险。很快，不堪重负的 IT 专业人员无法跟上应用程序现代化或基础架构动态的步伐，因为他们被复杂性淹没了。

尽管这种情况很常见，但并非不可避免。通过实施能够克服复杂性和孤岛的集成式、经济高效的全堆栈、端到端监控服务，IT 团队可以简化数字化转型之旅。

组织需要全栈[可观察性](https://devops.com/?s=observability)。

## 可观察性与传统监控有何不同

可观察性超越了传统的监控。传统监控有助于 IT 组织了解其基础设施和应用程序的实际状态。它捕获和处理大量基础设施和应用遥测数据和通知，并显示哪些组件正常运行，哪些组件关闭，哪些组件发生了变化。T3T5】

通常，传统监控侧重于特定的网络、云或基础架构。它跟踪应用程序和基础架构元素，以便 It 专业人员能够发现异常并在问题出现时进行调查。

监控依赖于以度量为导向的仪表板，该仪表板用于根据手动或基本统计相关阈值评估遥测数据。监控工具是非常宝贵的，但是它们不提供跨域关联、服务交付洞察力、操作依赖性或可预测性。现代系统有复杂的多云环境和大量的遥测数据:这被认为是一个不足。

[可观性](https://www.solarwinds.com/resources/it-glossary/observability) 确实比较多。它通过检查输出来测量系统的内部状态，并从整体上查看应用程序和系统——从最终用户体验到服务器端指标和日志。

但是可观察性也包括作为关键要素的监控。要获得可观察性，首先必须通过监控收集信息。可观察性使用通过监控产生的洞察力和度量来理解是什么导致了手边的问题。

监控汇总并显示数据，以确定系统是否按预期运行。将对这些信息的分析与预期结果和目标进行比较。这使 IT 专业人员能够了解其基础架构和应用程序的状态。

这样就避免了孤岛，因为复杂的环境是从整体上来看的。

## 行动中的可观察性

一旦到位，可观察性允许 IT 组织在复杂、多样和分布式的混合和云环境中不断提高性能、可用性和数字体验。

通过可观察性，组织可以快速找到并解决异常情况。然而，全栈可观察性不仅仅是监控和加快问题解决:它通过跨域数据关联、机器学习(ML)和人工智能为 It 运营(AIOps)提供洞察力、自动化分析和可操作的智能。它适用于大量实时和历史指标、日志和跟踪数据。

可观察性超越了与传统监控相关的孤岛和零敲碎打的方法。当可观察性不受限制(包括 ML 和 AIOps)时，它会使用大量收集的数据，并提供见解、自动化分析和可操作的情报，以帮助 it 人员加快问题的解决。它还使 ItOps、DevOps 和安全组织能够实现一致、优化和可预测的业务服务交付，并不断改善数字体验和 IT 工作效率。

结果是客户和员工从运行更好的系统中受益。该技术通过云连接内部部署或软件即服务(SaaS)部署灵活性，为各种规模和行业的组织提供全面、集成且经济高效的功能。

在开始数字化转型之旅时，组织不需要进一步的复杂性，尤其是在他们更新传统应用程序并向其堆栈中添加过多的现代服务和功能时。降低复杂性的关键是可观察性。

可观察性简化了转换过程。It 部门降低了运营噪音，有利于 ITOps、DevOps 和安全团队。他们可以在问题和异常检测方面变得更加主动，以实现最佳的 IT 性能、合规性和弹性。借助全栈可观察性，任何组织(无论其规模或行业如何)都可以降低 IT 复杂性，同时为数字化转型做好准备。
