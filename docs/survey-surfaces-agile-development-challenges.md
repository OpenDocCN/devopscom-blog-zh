# 调查揭示了敏捷开发的挑战

> 原文：<https://devops.com/survey-surfaces-agile-development-challenges/>

本周公布的一项对 3220 名软件开发专业人员的调查发现，在接受敏捷实践构建软件的 80%受访者中，只有不到一半的人跟踪按时交付(47%)和实现的业务目标(44%)。

这项调查由 Regina Corso Consulting 代表 Digital.ai 进行，digital . ai 是一个[价值流管理](https://devops.com/?s=value+stream+management) (VSM)平台的提供商。调查还发现，总共有 19%的受访者不确定使用什么指标来跟踪软件开发进度。

与此同时，40%的受访者指出他们对组织中的敏捷不满意；与公司文化的冲突被列为交付失败的主要原因(41%)。一半的受访者使用敏捷、瀑布和/或迭代过程的组合，但 27%的人指出这种框架组合对他们来说是一种挣扎。

调查还指出，参与调查的产品和应用程序所有者中，超过三分之一的人(36%)已经采用了敏捷。

Digital.ai 的敏捷和 DevOps 总经理德里克·霍尔特(Derek Holt)表示，这一比例表明，敏捷原则现在已经超越了构建软件的工程团队。他补充说，随着对敏捷原则的欣赏不断增长，这些人工作的组织也倾向于开始跟踪价值流指标。Holt 指出，在组织成熟到足以证明采用 VSM 平台的合理性之前，这种过渡通常始于一些实用的指标。

然而，随着时间的推移，VSM 平台不仅可以帮助组织识别瓶颈，还可以识别潜在的质量问题和业务风险，Holt 补充道。

总体而言，受访者确定的前三大敏捷开发优先事项是加快上市时间(52%)、交付可预测性(44%)和降低风险(31%)。然而，从业务角度来看，前三个优先事项是公司目标(54%)和最终客户满意度(43%)，其次是交付时间(35%)。

然而，超过一半(51%)的受访者也注意到 DevOps 团队没有定期整合业务反馈。超过一半(58%)的受访者表示，对 DevOps 团队而言，最有价值的改进是从业务计划到开发、测试和部署到最终用户的端到端可见性和可追溯性，而 55%的受访者表示，测量周期时间、等待时间和瓶颈的能力将是一项重要的改进。

关于使用的敏捷工具，Scrum 以 87%领先，其次是看板 56%和 Scrum ban 27%。根据调查受访者的说法，超过一半的受访者采用了扩展敏捷框架(SAFe)框架(53%)，而 28%的受访者采用了 Scrum 的[【电子邮件保护】](/cdn-cgi/l/email-protection) /Scrum，其次是精益管理，比例为 8%。

敏捷的最大好处是增加了协作(69%)，其次是更好的业务一致性(54%)和更好的工作环境，以及应用程序开发生命周期中增加的可见性(各占 39%)。

当然，关于敏捷框架优点的争论[仍然很激烈](https://devops.com/agile-sucks-devops-mars-richixbw/)，没有哪两个组织的采用程度是一样的。然而，随着在[数字化商业转型](https://digitalcxo.com)时代更快交付高质量软件的压力持续增加，[调整敏捷和 DevOps 最佳实践](https://devops.com/how-to-combine-devops-and-agile/)将变得更加重要。
