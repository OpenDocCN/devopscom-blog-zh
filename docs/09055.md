# 流处理如何超越实时

> 原文:[https://devo PS . com/how-stream-processing-goes-beyond-real-time/](https://devops.com/how-stream-processing-goes-beyond-real-time/)

实时有多快？根据罗伯特·米勒 1968 年关于人机交互的开创性研究，我们认为 100 毫秒或更短的反应时间是瞬间的。这比人类的平均反应时间要快得多，[在线测试](https://humanbenchmark.com/tests/reactiontime)的平均反应时间为 273 毫秒。

如今，面向客户的应用程序需要尽可能接近实时地做出响应，以赢得我们的关注并满足标准要求。例如，100 毫秒是信用卡处理者满足[SLA](https://devops.com/site-reliability-engineering-sre-comes-of-age-in-2022/)的黄金法则。特别是在我们新的远程优先经济中，媒体、医疗保健、零售和制造等其他领域需要充分利用实时数据处理来增强数字业务并保持相关性。然而，提高实时性需要一些技术上的细微差别，以目前开发人员的技能差距来看，这可能很难实现。

根据 Hazelcast 首席产品官 Manish Devgan 的说法，大多数数据管理技术的问题是数据没有在边缘进行流处理。相反，数据被存储并扔进数据湖，在那里积累，无法实现。更复杂的是，分散的数据管理意味着组织可能要监管许多不同的数据湖，增加了未使用数据的可能性和延迟。

我最近会见了 Devgan，了解内存流处理如何超越实时数据。根据 Devgan 的说法，秘密酱正在处理传入的流数据，同时将它绑定到持久存储以创建上下文。他说，通过这样做，公司可以获得更多创新的实时体验。

## 实时的用例

实时为业务运营和最终用户客户互动提供了许多机会。在金融领域尤其如此。例如，Devgan 解释了一家银行如何能够在客户在自动取款机上进行交互时使用实时数据。该系统能够获取用户帐户的历史记录，并提供低信用额度，使他们的贷款总额增加了 400%。德弗根说，抓住用户意图的那一刻是一个千载难逢的机会。

Capital One 是金融机构利用先进技术改善产品体验的又一个例子。“就技术而言，他们是领先的银行之一，”德弗根说。Capital One 的[威胁检测系统](https://www.capitalone.com/tech/machine-learning/)实时调用机器学习模型来检测和防止欺诈。只有通过在交互时对输入数据采取行动，才能发出类似这样的预防性响应信号。

其他场景也开始增加实时分析。“这种瞬间先发制人的决策对许多其他用例都很重要，”Devgan 说。实体零售可能会融入销售点系统的客户忠诚度计划，以增加店内支出。或者，汽车制造商可能希望使用实时处理来尽早检测机器人手臂的问题。

## 实时与流处理

当一个事件发生时，它通常只有很少的信息。但是，当您查找客户资料并将其与更大的背景联系起来时，您可以指导进一步的行动。“将运动中的数据与静止中的数据相关联是神奇的地方，”Devgan 说。这种区别揭示了实时数据和流处理之间的差异。

实时标签往往被抛来抛去，而且对不同的群体有不同的量化。对于某些应用程序，实时时间可能是 5 分钟，但对于其他应用程序，实时时间可能是 10 毫秒。实时可以简单地理解为系统非常快速地访问静态数据，提供可感知的“实时”结果。

另一方面，流处理包含静态数据和动态数据。它计算传入的*流内*数据，并将其与静态数据相关联。“你不能让数据着陆——你要在它进来的时候处理它，”Devgan 说。

多年来，企业一直在收集和存储大量数据。希捷和 IDC 的一项调查显示，收集的数据中有 43%是未利用的，其他研究估计这种“[黑暗数据](https://priceonomics.com/companies-collect-a-lot-of-data-but-how-much-do/)的数量超过 50%。此外，没有有效地根据数据采取行动只会使它成为[不断增长的技术债务山](https://devops.com/enterprises-face-growing-technical-debt/)上的又一堆。

无论是在内部还是在云中，保存过多的数据都会产生巨大的存储开销。但是，这种模式正在发生一些变化，德弗根说。组织不想抓住一切不放。人们的态度开始转向“垃圾进，垃圾出”

## 实时处理背后的技术

那么，流处理需要什么样的技术？在边缘处理数据需要对设计思维进行一些根本性的改变。首先，因为数据必须在流中计算，所以更加强调内存中的处理。Devgan 将此解释为“推动计算而不是推动数据。”为了实现这一点，Devgan 提到了像 [Jet](https://github.com/hazelcast/hazelcast-jet) 这样的开源项目，它们提供了强大的分布式批处理和流处理。

SQL 有着处理静态数据的传奇历史。Devgan 说，现在他注意到 SQL 在流处理上下文中的[重现。尽管业界仍在定义 SQL 方言如何适应流媒体世界，但他看好 SQL 引导这一领域的前景。“流 SQL 将成为流的通用语言，”他坚持说。Devgan 补充说，我们还处于决定如何在有状态和无服务器世界(T3)之间架起桥梁的早期时代。](https://www.confluent.io/blog/databases-meet-stream-processing-the-future-of-sql/)

但是不管计算位置在哪里，它都将归结为净客户体验结果。“人们希望构建他们的业务应用程序，不太担心系统的运营管理，”Devgan 说。无论是建立电子商务的实时报价还是实施欺诈检测，都将归结于数字。在医疗保健中，流处理还可以优化物理资源，并通过实时数据增强患者体验。在所有这些场景中，在边缘快速访问数据和处理将决定他们的成功。
