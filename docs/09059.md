# 用财务纪律提升敏捷

> 原文:[https://devo PS . com/lifting-agile-with-financial-discipline/](https://devops.com/elevating-agile-with-financial-discipline/)

一条企业格言正在改变，从“大吃小”转变为“快吃慢”但是改变不会在一夜之间发生。

虽然[敏捷和精益方法](https://devops.com/combining-the-best-of-agile-waterfall-and-devops-techniques/)继续打破现状，取得收益，并越来越多地超越团队口袋，进入组合和企业范围的采用水平，瀑布式交付方法仍然存在。

瀑布公司倒闭的传言似乎有些夸张。Scrum Alliance (2017)对 Scrum 实践者进行的一项[调查显示，78%的受访者表示他们使用敏捷-Scrum 实践与瀑布实践并行。混合瀑布/敏捷环境是复杂的。对于很多大公司来说，这是常态。这是敏捷和传统工作方式共存的挑战变得更加突出的时候。](https://www.scrumalliance.org/ScrumRedesignDEVSite/media/ScrumAllianceMedia/Files%20and%20PDFs/State%20of%20Scrum/2017-SoSR-Final-Version-(Pages).pdf)

重要的是，正如 IT 财务管理(ITFM)的领导者所知，在财务纪律和监督方面，变得“更加敏捷”并不能让你过关。敏捷组织仍然对他们的投资回报率负责，说你采用了精益方法并不会自动让你变得精益。或者负责财政。或者治理有方。或者有利可图。

## ITFM 领导人面临压力

技术组织是 在 中，从以项目为中心发展到以产品为导向的解决方案交付，并从团队级别扩展敏捷到组合/企业。在这些不断发展的混合工作环境中，面对更大更活跃的敏捷社区，ITFM 的经理们正面临着传统方法和敏捷方法之间越来越多的过程不协调，对他们的报告、计划、预算和监督造成了严重破坏。

在混合的传统/敏捷财务组合中，了解项目、组合、价值流和 okr 的真实成本有助于公司更好地利用其财务资源来推动盈利结果。

随着人们对信息技术的价值从首席信息官扩展到其他高管的兴趣，将敏捷工作与传统信息技术项目相结合的综合报告和决策，ITFM 的角色本身变得更加重要。

## 技术业务管理:准确计量 IT 成本

对于许多 ITFM 领导人来说，对技术业务管理(TBM)规则的关注提高了他们的能力，通过对 IT 资源的实时报告(和计费)进行有针对性的分析，他们能够更好地了解 IT 成本并影响治理和行为。TBM 提供了一个决策框架，帮助领导者了解向客户交付产品和服务所涉及的真实成本。这种方法效果很好，特别是在以项目为中心、传统的瀑布式项目是标准的行业。事实证明，它对于支持敏捷也非常有效。

随着敏捷获得动力，财务是 it 与 P & L 报表、预算美元和财政季度相碰撞的地方。看似不可调和的敏捷发布系列和程序增量、故事点和周期时间，很难看到一条整合的前进道路。幸运的是，对于已经接受它的团队来说，TBM 提供了一个经过充分测试的、成熟的框架，可以轻松扩展以支持企业敏捷转型。

当团队寻求获得端到端企业敏捷性的回报时，敏捷转型和扩展计划就会出现。

随着企业敏捷性的扩展，ITFM 的领导者可能会遇到对敏捷转型至关重要并直接影响 IT 财务的流程改进计划。其中的核心是精益投资组合管理(LPM)和价值流管理(T2)(VSM)。由于他们对 ITFM 的直接影响和他们交织在一起的敏捷特性，这两个特别值得早期关注。此外，虽然企业敏捷性还有更多的维度，但这是一个很好的起点。
