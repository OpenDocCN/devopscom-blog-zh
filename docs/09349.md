# 为什么过度许可的 CI/CD 管道是不必要的祸害

> 原文:[https://devo PS . com/why-over-permissive-ci-CD-pipelines-are-an-needly-evil/](https://devops.com/why-over-permissive-ci-cd-pipelines-are-an-unnecessary-evil/)

在软件开发中，平衡创新速度和安全性是一个古老的问题。这在现代 DevOps 中没有什么不同。 开发团队想要一条无摩擦的路径来部署新代码，但是经常与链接到 服务的访问权限相冲突。CI/CD 管道让开发人员更容易以敏捷的方式快速引入新功能和更新。当应用程序更改通过 CI/CD 管道时，自动化可以让更改非常快速地发生，而不会在客户端造成停机或延迟。

### CI/CD 服务器权限

当 涉及到 CI/CD 服务器的权限时，DevOps 往往不要 要 拒绝访问。毕竟，广泛的访问权限是允许创新速度的一部分，并且开发团队经常——并且通常可以理解——对放置最小特权或其他访问规则持谨慎态度。

但是有一些安全风险需要考虑。任何被授予管理员权限来部署新代码的开发者都容易受到 漏洞、弱点和错误配置的影响。[云蔓延](https://devops.com/devops-chat-cloud-sprawl-devops-park-cloud-w-jay-chapel/)导致无法了解基础设施是否正确设置并与测井工具连接。开发团队如何知道事情是否得到了适当的修补？

这种平衡极其敏感。开发人员需要变化的速度和自由创新的能力，但安全团队无法应对不断增长的风险。有些事情 不得不 改变，这可以允许[安全性向左转移](https://containerjournal.com/features/gitops-security-compliance-and-resilience-shift-left/) ，并且 更早地参与 CI/CD 管道，而不是碍事。

### 在不影响创新的情况下降低风险

以下是一些加快 创新速度同时保持健康安全态势的建议:

*   **地图威胁与安全连接** **。**进行威胁建模练习，将威胁映射到管道。请记住，到 CI/CD 管道的每个连接都是潜在的危险点。定期修补和扫描连接到管道的所有服务，并在设备不符合安全策略要求时阻止设备。
*   **学会爱 日志 。** 当生产中出现安全问题时，开发人员、IT 工程师和安全工程师需要快速高效地做出反应来解决问题。日志记录对于实现快速、协调的响应至关重要。等待敏感数据的手动共享会降低反应速度。但是，当每个利益相关者都可以从日志中获得他们需要的数据时，对信息的访问就不再是安全事件补救过程中最薄弱的环节。
*   **加强访问控制，实施强管理权限** **。明确谁有权访问哪些数据。清晰和强制的职责分离应该确保在连续的软件交付管道中，没有人或帐户可以访问不必要的内容。不适当的认证用户或不良行为者不能建立对整个管道的控制并扰乱代码或流程。此外，它们也不容易更深入地渗透到其他企业系统和数据中。**
*   **借力旗帜经营风险** **。开发人员通常使用功能标志(或功能切换)向应用程序添加新功能，同时降低这些功能带来的性能或稳定性问题。尽管大多数开发人员可能不认为特性标志是一种安全工具，但是它们也可以用于这个目的。** 新的 特性是安全漏洞的主要载体，尤其是如果开发者还没有彻底地测试它们的话。

许多安全团队没有意识到，仅仅因为他们的网络是私有的，他们仍然与第三方共享管道，包括任何人 从维护 到部署您的移动应用 的团队。 通过采用这些工具，开发人员可以 左移， 在过程早期支撑防御，而不会给来自 DevOps CI/CD 管道的创新步伐增加障碍。
