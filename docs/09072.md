# 面向客户体验的开发运维

奥汉隆:好吧。听起来不错，我想您的业务在过去一年左右的时间里一直在蓬勃发展，因为在过去一年半左右的时间里，客户真的成为了许多组织的首要目标，因为他们不得不加倍努力进行数字化转型，才能在疫情上生存下来。有些人做得很好，有些人做得不好。

那么，就客户体验而言，你们现在在市场上看到了什么？

哈德逊:是的，不，这是一个很好的问题，因为在过去的一年里，疫情发生了很大的变化。我的意思是，听着，顾客总是要求很高，对吧？这是我们作为顾客的天性。但是现在，随着他们对技术的掌握，客户现在有了很高的期望。并且转换成本低。所以，如果客户没有从体验中得到他们想要的东西，你可能会失去他们。

因此，从我们的角度来看，拥有良好的客户体验并保持领先地位的重要性达到了前所未有的高度。

欧汉伦:所以，很明显，你知道，通过 DevOps.com，我们关注的是采用 DevOps 方法、心态和文化转变以及所有这些好东西的好处。我们已经看到 DevOps 的整个前提在某种程度上转移到了组织的其他部分，因为他们认识到了将两种不同的思维模式和组织领域融合在一起的好处，从而简化了运营，并从长远来看实际上构建了更好的产品。

我认为，当我们谈论采用 DevOps 原则来实现客户体验的组织时，客户服务并没有被排除在对话之外。

哈德森:对，没错。很有意思。因此，大约四五年前，在 Cyara，我们开始看到 DevOps 进入联络中心/客户体验领域，但这在当时还比较罕见。开发运维那时还没有必要与客户体验交叉，但是在过去的三年左右，我们真的看到开发运维在客户体验领域变得越来越普遍。我认为这是一件好事，因为正如我之前提到的，你需要保持领先。你需要能够为客户体验快速创新，我认为这些公司已经开始看到这一点。

所以，我们开始看到 DevOps 几乎被普遍采用。要么是有人已经开始了，要么是正在进行中，要么是稍微成熟了一点，但就客户体验而言，这是这些天我们几乎与每个人交谈的话题。

奥汉隆:太好了。这真是太好了，因为 DevOps 在客户体验领域肯定有很多好处，尤其是能够更有效、更快速地为客户服务。

但这不是一对一的调整。你知道，看起来，特别是因为，当我们谈论 DevOps 时，我们认识到每个 DevOps 实施和每个 DevOps 体验都是不同的。你知道，每个人都是一片雪花。

那么，这与客户体验有什么关系？如果组织希望将 DevOps 添加到他们的客户体验中，这实际上会给他们带来什么样的麻烦？

哈德森:是的，绝对的。所以，当我刚开始在 Cyara 工作时，我领导了一个非常小的战略咨询小组，我们与客户一起研究 Cyara 如何帮助他们完成任何重大转型。由于行业的转变，它最终变成了许多 DevOps 转换。因此，我们看到了客户在旅程中的多个不同点，以及他们遇到的困难。我认为有许多不同种类的陷阱，或者从中吸取的教训。

你知道，首先要做的事情之一——我们可以尽可能详细地讨论这些感兴趣的事情，但首先最重要的事情之一是领导层面的文化转变。我认为，如此频繁——我以此打头，因为如果你没有做好这一部分，它真的会阻碍你。但是，开发运维经常被视为技术或流程的改变，因此，它每天都在发生。但实际上，这是一种思想转变，文化转变，必须自上而下，必须融入文化，必须由领导来推动。

奥汉隆:嗯，是的。我同意，同意，这适用于整个组织，无论他们在哪里实施开发运维，是在 it 部门，还是在后台办公室或营销部门。因为我们看到，如今组织中如此多不同学科的融合，是推进和加速他们开展业务的方式。

但就你的观点而言，如果我们没有，或者如果一个组织没有来自高层的认同，无论他们在组织中的什么地方实施 DevOps，都注定会失败。

哈德森:没错，没错。你知道，这不仅仅是一个开始，也不仅仅是某个 C 级别的人在开始时说“啊啊”。

奥汉隆:对。

哈德森:这就像，通过这个过程，嵌入到你如何激励、激励、如何奖励人们，你如何——你知道，在你的常规节奏中。到处都是。这需要被视为一种文化转变。

奥汉隆:是的，是的。我完全同意。现在，你知道，除了最初的那种，随着组织开始采用这种思维模式和这些原则，我们可能会称之为 DevOps 第二阶段或 DevOps 2.0，第二天 devo PS-你们看到组织需要非常注意的一些事情是什么，以确保他们成功？

Hudson:是的，我们在运营层面遇到的另一件事是，如你所知，流程在 DevOps 中的重要性，对吗？

奥汉隆:对。

哈德森:你不会想扰乱流程的。要做到这一点，你必须考虑自动化，对吗？这只是—这是显而易见的。是柱子吧？对我来说，当我想到自动化时，有许多不同的口袋可以自动化。但是对我来说，我主要考虑部署自动化和测试自动化。

我们进行了一次采访——或者说我采访了一家咨询公司，一家拥有 DevOps 专业知识的四大咨询公司之一和一位客户。他们都同意自动化测试，虽然它并不总是完全简单或直接，但是，他们几乎认为这是唾手可得的，对吗？让测试自动化，这样你就可以即插即用，不要中断流程。

从部署的角度来看，有许多不同的环境，对吧，让一些东西进入生产。因此，从这个角度来看，你越能自动化，你就越能继续推动事情的发展，快速学习，快速失败，如果你愿意的话。

因此，找到这些自动化领域，并制定一个策略来指导您如何尽可能多地实现自动化而不中断流程，这是我们真正向人们推荐的，以实现转型的总体好处。

奥汉隆:从人的角度来看呢？众所周知，DevOps 不仅与流程和技术有关，也与文化有关。那么，有没有什么方法可以让组织——也许他们在采用 DevOps 时，在文化和人员方面没有考虑特定的事情，特别是为了客户体验？

哈德森:是啊，当然。所以，我会用几种不同的方式来回答。我要讲的第一个基本上是关于——这又是一个文化问题。DevOps 的支柱之一是创造一个鼓励学习的环境和一个安全的失败场所。所以，如果你的团队不得不失败，你希望你的团队得到鼓励，从失败中吸取教训，下次做得更好，并感到安全。这是从人的角度来看的。

我想提的另一件事就是衡量关键绩效指标，对吗？没有测量就无法知道自己是否成功，对吧？

奥汉隆:嗯。

Hudson:然后公布这些 KPI，这样它们就透明了。透明度真的很重要，所以每个人都朝着同一个目标努力，每个人都觉得自己是实现这个目标的一部分。所以，让我们拿出业绩指标，让我们明确我们要实现的目标，让人们在这个过程中感到安全。你知道，这就是你想要围绕它建立的文化。

欧汉伦:那么，当我们谈论客户服务应用程序和软件与内部流程或内部 DevOps 流程或由这些 DevOps 流程推进的内部应用程序相比时，当我们看这些 KPI 是什么或服务水平目标是什么时，有什么区别？当一个组织在开发客户体验型软件时，他们是否需要更加注意某些事情，而不是更加关注内部或者其他客户，而不是面向客户、最终用户和客户？

哈德森:是的，我认为这是一个很好的问题。我们在行业内尝试的是创造这种思维转变，让您时刻考虑最终客户，而不仅仅是在生产环境中，您在旅程的每一步都要考虑最终客户的体验。

我会举一个例子来说明我的意思。传统上，联系中心的空间，他们将如何监控测试或生产中的事情是一种服务器负载或容量，对不对？如果达到了最大容量，就会触发警报，这是个问题，对吗？你不希望它达到最大容量。

很好，但是如果服务器达到最大容量，您可能不知道客户体验如何，对吗？

奥汉隆:嗯。

Hudson:因此，在 Cyara，我们试图做的是说——当然，服务器可能会达到最大容量，这是一个问题，但客户目前在生产中会遇到什么情况？但是我们也想在开发、质量保证、UAT 和性能测试方面进行测试，看看客户的体验如何。因此，如果出现问题，你知道这可能是一个非常严重的问题，可能是电话被切断。如果在你的创收电话线路或聊天窗口中，客户打不通电话——你想在投入生产之前知道这一点。

奥汉隆:嗯。

Hudson:你不一定想知道——或者知道某项技术是否达到了极限可能没什么帮助。最有价值的是知道客户在那一刻会体验到什么，甚至在投入生产之前，如果你能做到的话。

奥汉隆:有意思。你知道，在不同的垂直市场中讨论 DevOps 真的很有趣，因为每个人都有不同的东西，他们真的真的非常专注于这些东西，但最终，这真的是要确保最终用户体验尽可能好，应用程序如宣传的那样工作，当然应用程序中也有一定的安全级别。

因此，我确实认为安全性也是等式中的一个重要部分，特别是对于客户体验和大量的敏感数据，正如你所说的，许多联络中心都在定期处理这些数据。那么，在 DevSecOps 上重点强调 Sec？

哈德森:哦，当然。这实际上是来自两端。一个是客户数据，对吗？你必须保护它。这里有一些敏感性，需要保护，我们需要在这些方面进行投资，如果可能的话，您需要能够在生产中或在此之前进行测试，对吗？

但是它的另一面也是围绕着法规要求，对吗？如果某些要求没有得到满足，客户——我们的客户、大公司——将会被罚款，对吗？因此，从纯业务的角度来看，您有动力确保客户体验到该需求。但是从他们自身保护的角度来看，你也有动力。

因此，DevSecOps，我们真的鼓励人们在讨论的早期就考虑安全性。这通常是最后出现的事情之一——“哦，等等，我们讨论过安全性吗，我们有没有引入那个团队？”尽早把它们折叠起来，完全透明，进行对话，并把它们编织到流程中，因为这很重要。

奥汉隆:太好了，太好了。艾米，这是一次很棒的谈话。我真的真的很喜欢它，并且非常希望将来能与您讨论与开发运维及客户体验相关的其他话题，因为我真的认为这不会是一次彻底的讨论。就像 DevOps 一样。客户是一个不断发展的对话。

但是现在，非常感谢您抽出时间与我交谈。我真的很感激。

哈德森:谢谢你邀请我。很荣幸来到这里。
