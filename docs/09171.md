# 有时候，你应该修理没坏的东西

> 原文：<https://devops.com/sometimes-you-should-fix-what-isnt-broken/>

DevOps 已经和我们在一起足够长的时间了，现在在我所关注的所有地方都有许多遗留的 DevOps 东西在运行。忽略实际上不起作用的项目；那些构成严重安全威胁的部分，因为它们包含已被修复的漏洞——有许多“它只是工作”，管道到位，甚至有点“我们一直是这样做的”

考虑到 DevOps 仍然经常被营销人员推为“不可避免的新热点”，我觉得这很有趣；DevOps 现在是跨企业的事实上的标准，这是另一个话题。我想谈谈安静运行的东西，让你不再需要或渴望去思考它，同时专注于那些需要你去思考的问题。

但是这种类型的东西正是最初创造开发运维需求的东西。DevOps 就像一个破坏性的球，迫使你查看围绕组织/文化/基础架构中的项目建立的不必要的步骤，通常是围绕这些非常“这只是我们如何做的”或“这没有坏，让我们看看另一件事，”的态度。

在[根深蒂固的技术](https://ingrainedtech.com/)，我们有脚本运行我们的备份到云。它们是全自动过程的一部分；我们设计它们是为了确保我们的系统在高度适应的情况下备份什么。这个系统每天晚上都像冠军一样运行，除非我们告诉它不要这样。

自从我们第一次使用他们的 API 获取数据并编写脚本以来，我们的云存储提供商已经取得了很大的进步。事实上，像我们这样的客户是供应商保持与那些 API 向后兼容的原因。这意味着是时候让我们重新审视、使用最新的[API](https://devops.com/?s=APIs)并看看我们想要利用众多新特性中的哪一个。“足够好”也不过如此。系统没有崩溃——我们从中获得了可行的备份(几个月前我们还不需要，所以我们*知道*我们得到了好的备份)，但是我们没有从中获得最大价值；我们不得不做一些跳跃性的工作，让数据集按照我们想要的方式恢复。

与我在大公司看到的类似“足够好”的安排相比，我们简单的备份程序算不了什么。“足够好”的安排产生了越来越大的减速带，有时类似于使早期 DevOps 实现的巨大收益不可避免的确切环境。

以批判的眼光审视您的基础设施和工具链。寻找唾手可得的水果，甩掉一些堆积如山的包袱。安排一些流程更新日，让一两个团队成员专门用批判的眼光审视环境中的一切，看看他们有什么建议。

继续摇摆。不管有没有减速带，你都在创造维持整个公司运转的应用和基础设施。记住这一点，当你解决反复出现的问题时，脸上要保持微笑。
