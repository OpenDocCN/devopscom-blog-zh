许多人说站点可靠性工程(SRE)正在“将运营放在开发运维中”在本期 DevOps 聊天节目中，我们将采访彭博 SRE 团队的 Stig Sorensen。Stig 告诉我们 SRE 是如何帮助彭博跟上快速发展的新闻和信息世界的步伐的。

与现实生活中的从业者就如何在现实世界的公司中部署 SRE 进行了精彩的讨论。像往常一样，下面是流媒体音频，然后是我们的谈话记录。

# 副本

艾伦·希梅尔:大家好。我是艾伦·希梅尔，您正在收听的是另一个 DevOps 聊天。我们今天有一个非常好的 DevOps 聊天，关于一个非常热门的话题，这是 SRE，网站可靠性工程。我的嘉宾是斯蒂格·索伦森，他是彭博 SRE 团队的负责人。斯蒂格，欢迎来到 DevOps 聊天。

**斯蒂格索伦森:**非常感谢。我很高兴能上。T3

Shimel: 那我们就不拘泥于形式了。斯蒂格，把你的头衔和背景给我们的人，这样我们就可以开始了。T3

好的，我的名字是斯蒂格·索伦森。我是彭博生产可见性小组的经理，负责构建我们的 CMDB 工具，并构建遥测分析平台。我也是将 SRE 作为一个角色和一个新的运动在彭博推广的主要倡导者之一。

**Shimel:** 当然，彭博不是一家小公司，所以推出这一新角色，这种看待运营等的新方式，当然不是一件——你知道，这不是我们可以掉以轻心或认为容易的事情。

索伦森:嗯，的确如此。我们的 5000 名工程师为我们的一切提供支持，从我们的商业网站到市场数据反馈，再到我们的主要产品，彭博专业终端，全世界成千上万的重要影响者都依赖它。而且，由于各种状态和各种不同的技术堆栈，这实际上是一个相当大的挑战。

**希梅尔:**优秀。斯蒂格，为什么是 SRE，为什么是现在的彭博？

**Sorensen:** 所以 SRE，就像我们所做的，对彭博来说并不新鲜。一直以来，工程师都拥有生产，拥有生产的可用性，拥有生产过程。

我们现在尝试的是在应用工程师和 sre 之间划分更多的责任，这样我们就可以拥有这方面的专家，但在当今复杂的世界中，让每个人都知道从如何在街上交易仪器到如何运行和管理高可用性系统的所有事情是不公平的。对吗？所以它更适合基础设施，让人们成为专家。

Stig，你提出了一个有趣的观点，你说，我们一直在做这种事情。这是真的。就任务和责任而言，许多我们今天称之为 SRE 的东西并不一定是全新的。它-它可能有更多的包装和一些混合在一起。

什么时候，比如说，彭博开始把这个角色，这个，你知道，需要称为“SRE”了？

**Sorensen:** 大约两年前，我们开始小规模经营，我想说，在去年，我们在全公司范围内真正做到了这一点。一直以来都是在一些地方做的，但是我们真的，像，这个难题的一部分是内部品牌化，也就是给人贴上标签——不是给人贴上标签，而是确定谁在那里，为这些人创造合适的学习和分享想法的环境。对吗？

我认为，正如你所说，这并不一定是一件新鲜事。对我们来说，新的是在全公司范围内以更标准、更持久的方式来做这件事。

是的。斯蒂格，你知道，5000 名工程师。你知道,“SRE 队”有多少人？

**Sorensen:** 所以他们——我们现在实际上没有一个集中的 SRE 团队。我们在某种程度上推出了它，让 SRE 团队与应用团队保持一致。原因有很多——为了让 SRE 团队取得成功，应用程序也必须改变。把责任分得太开会让事情变得更难。

所以我们把他们放在平行的、肩并肩的、有着不同职责的两个平等的团队中，这就是我们启动团队的方式。大多数 SRE 团队——很多——相当多的 SRE 团队是从应用程序团队和许多核心基础架构团队等中产生的。

因此，总的来说，我们有大约 75 人，他们属于 SRE 团队。正如我所说的，我们公司有更多的人在做这件事，但他们是在做兼职工作。

**Shimel:** 明白了。所以斯蒂格，你知道，SRE——这个 SRE 的概念——就像你说的，虽然他们——我们一直在做这种事情，把他们叫做 SRE 并给他们贴上这样的标签的想法是相对较新的。

有没有什么特别的训练或技能，你要求人们去确保他们，你知道，为了成为“SRE”而精通的？

**Sorensen:** 我认为——我认为某种程度上——答案是我们正在招聘，而且我们正在努力招聘更多的人。他们似乎来自两个不同的阵营。要么是具有运营或系统管理背景的人学会编程并以这种方式前进，要么是我们的应用工程师——对系统和可用性有更积极的看法——因为我们认为软件工程师只是在做一些事情——构建不同类型的软件。

我开玩笑地试着将软件工程师和应用程序开发人员进行分类，我认为应用程序开发人员天生乐观，他们认为所有东西都会一直工作，而软件工程师则相反，他们认为所有东西都会一直坏掉。这种关系——我认为更多的是一种心态，而不一定是技能。你总能学到的技能。然后我认为有些——只要你有正确的心态，正确的能力去做，那——嗯，那是我们的关键。对吗？

我认为有——我认为现在 SRE 的工作比以前在 SRE 工作的人多得多。所以我们不能出去尝试雇佣现成的 SRE。我们需要弄清楚如何让聪明的人具备合适的资质，并对他们进行培训。

人们提到培训，我们试图通过各种会议，各种内部会议，外部会议，参加会议来了解行业内发生的事情，在会议上发言，这样我们也可以展示我们正在做的事情。

我认为这是-我认为这真的是一种大量的在职培训，与应用团队合作，与其他 SRE 团队合作，你会意识到你并不孤单。总有——总有人在做和你非常相似的事情。你不需要每次都发明轮子。

是的，斯蒂格，我们不应该低估你刚才所说的话的重要性。对于像 SRE 这样的新领域，我们在 DevOps 早期就看到了这一点，因为没有任何成文的最佳实践，对吗？谷歌团队出了一本书，现在也有其他东西出来了，但真的没有最佳实践。新兴的实践。因此，点对点的分享和分享学习的想法对于继续发展非常重要，你知道，如果你愿意的话，是最先进的。

除了我的 DevOps dot com 职责之外，我还是 DevOps Institute 的联合创始人之一，这正是我们在 DevOps 中看到的事实模式。这是我们在 SRE 看到的同样的事实模式，坦率地说，这是我们正在努力的原因之一，并希望很快推出一个 SRE 课程，你知道，将所有这些新兴的实践汇集在一起，并围绕它拥有一个共同的知识体系。

**Sorensen:** 我认为这很好，我们正在尝试标准化这些东西，但是——因为很多东西也变得有点——你在推出时必须灵活，因为不是每个应用程序或每个系统都可以推出——可以——可以让 SRE 团队以相同的方式推出，因为系统的成熟度不同，问题也不同。对吗？

就像，就像，你是侧重于监控和生产环境，还是侧重于更经典的开发运维以及如何将东西投入生产？就像，你真的要从整体的角度来看待它。SRE 团队的目标是提高生产的可用性。做这件事有多种方法。你可以选择一个并使它变得更好，你可能会——你可能会改进它。对吗？

这种认为 it 必须竭尽全力才能有所作为的想法可能也不是正确的想法。

希梅尔:是啊。绝对的。那么，斯蒂格，让我——让我稍微转一下话题，让我们谈谈——彭博和 SRE 有什么关系？他们为什么支持你？

**Sorensen:** 嗯，他们——我认为他们真的只是关注我们提供的服务。对吗？例如，我们正在向全球 30 万最具影响力的人提供实时金融信息，这不仅仅是可用性的问题，还有性能和延迟的问题——数据在全球流动。对吗？我们——我们每天处理 1000 亿条市场数据和 150 万条新闻。对吗？所以，为了保持所有这些管道的运转，这实际上是一项相当大的工作。对吗？我认为只是——只是确保我们是可用的，不会因为不在那里而影响金融市场。

希梅尔:是啊。我的意思是，那——你知道，那带回家，你知道，我们不是——对不起，彭博不卖没人关心的小配件。停机时间和性能是结合在一起的，或者说，与金融市场性能和报告密不可分。您无法承受–您无法承受停机时间。你不能承受这种混乱。

但是 Stig，我们就这个问题谈了一会儿，我想提出的话题是，你知道，在彭博，你们现在一直在做 SRE 的事情，你认为它-你正在取得进展，这很好，但是我们真的没有做过任何形式的 ROI 分析，或者，你知道，类似的工作。正确吗？

索伦森:是的，不，的确如此。这可能是最大的挑战之一，我们得到了支持，每个人都认为这是正确的做法。你有一种直觉，我们必须这样做，以使它变得更好，或者保持我们一直拥有的服务。

但我认为这是一种——对我们来说，这是——对我们来说，就像是测量——测量可用性实际上并不是非黑即白的。这不像是，你在一个网站上经历了多少次失败。对吗？因为如果你是某个市场参与者，而实时市场数据延迟了几分钟——一毫秒或数百毫秒，这可能会对你产生很大影响。对吗？这是完全失败还是部分失败？以及介于两者之间的所有东西。

这是我们正在努力尝试定义的更好的衡量标准，这样我们就可以有效地跟踪我们的进展，看看我们是否成功了。这是——如果你凭直觉，它只会持续这么长时间，但在某些时候，你需要真正问自己，你需要确保回报在那里

绝对的。斯蒂格，我是说，让我们说清楚。这根本不是彭博的问题。我–你知道，我们还没有真正看到它的全面发展。你知道，SRE 的正式 ROI 类型的分析，但正如你所说，它-你知道，它显然是有意义的，或者，你知道，认为它对组织有好处是合乎逻辑的，我认为这就是为什么我们看到这么多组织开始采用它。

不过，Stig，我想谈的另一件事是，与 DevOps 体验类似，最初每个人都说，噢，DevOps 对独角兽来说太棒了。这对谷歌很好，对脸书很好，对全世界的推特用户也很好，但这对马真的不起作用。对吗？只是正规的大型企业。

SRE:嗯，是的，这是谷歌的事情，对谷歌来说很好，但它对彭博有用吗？对吗？或者类似的事情。你知道，你就是活生生的例子，如果你愿意的话。但是每条路，斯蒂格，都有–这是–你知道，我称之为 DevOps 恰恰。它从来都不是线性的，一步一步向前，它只是，你知道，一帆风顺。这里总有失败，这里有教训，退一步，进两步，一二，一二三。

给我们，你知道，我肯定你能给我们很多成功。给我们一两个你们真正从中吸取教训的失败案例。吸取的教训。

**Sorensen:** 酷，是的，我认为这是转型失败的教训，我认为，一些被称为 SRE 的团队被分配了太多的运营工作，他们不一定有足够的带宽来摆脱困境。这就创造了一种新的模式，就像，SREs 是操作，而它们不是。sre 是开发者。但是开发东西来摆脱运营工作，但如果你没有时间去开发，你永远不会摆脱它。

因此，我认为这可能是我们学到的最大的一课——在建立文化和获得某种——内部正确的光学方面。同样，我的意思是，这也是应用程序团队内部的 SREs，没有专门的团队也是一种反模式，我们看到在这种情况下，更容易窃取 SRE 的时间来做应用程序工作。再说一次，我们实际上并没有前进。所以，这可能是最大的失败。

我认为我们还需要确保我们有一套好的基本工具和基本的东西，可以在团队之间共享，这样每个人都需要从头开始构建一切。我们可能——我们可能开始于类似的筒仓中，在那里建立了类似的东西，但它得到了改进。它提高了——它提高了生产质量，但它不一定会让我们在推出 SRE 方面走得更快，所以它——我们不得不后退一步，对更多的工具进行标准化，并确保我们对其中一些工具拥有中央所有权，而不是由个别团队拥有。

**希梅尔:**优秀。斯蒂格，我们的时间快结束了。我只是想问你，所以，你知道好的进展，一个好的开始。如果你有一个水晶球，在未来的三到五年内，你认为 SRE 在彭博和更大市场的未来在哪里？

**Sorensen:** 我认为——我认为在这里是内部的，对，就像我们——我们正试图大幅增长，这样我们就能够——在内部将人们从一个应用程序转换成一个 SRE，并且还能雇用这些人。

我希望我们和像我们这样的大公司能够进行转型，变得更加标准化，这样我们就可以有一个像 SRE 这样的中央集权的组织，并且有更多的——我们可以——我们可以扩展——扩展和支持更多的团队，而现在是一对一，不一定是一对一的人，而是一对一的团队。

我希望整个行业也开始考虑不在大型云提供商上运行的人，他们实际上像我们一样从头开始拥有一切，我希望更多像我们一样的公司进入这一领域

将会有更多的游泳池和更多的东西来支持我们。对吗？

Shimel: 当然可以。当然可以。嗯，彭博 SRE 团队的 Stig Sorensen，我想感谢你成为我们今天 DevOps 聊天的嘉宾，你知道，这真的是——整个 SRE 事件肯定是我们密切关注的一个领域。也许几个月后我们可以让你回来，了解最新情况。
