# 合作进行中:使开发运维成为现实

> 原文：<https://devops.com/collaboration-progress-making-devops-reality/>

贵公司 DevOps 清单上的一切都已考虑在内。它有远见和才华。它发现了问题并制定了解决方案。它的文化允许这种计划所需要的灵活性。现在是时候将计划付诸行动，让收益开始。

事实是，事情可能没那么简单。除非每个人都明白如何实现，否则伟大的计划没有任何意义。无论一家公司的 DevOps 战略由什么组成，协作都是实现它的关键。

协作是 DevOps 的核心。它甚至在短语本身中被传达。无论你的公司规模如何，这种团队合作和透明的新流程依赖于开发和运营之间的优势结合，以实现否则不可能实现的目标。

**用全循环方法打破筒仓**

一个部门负责编写代码，另一个部门负责实现代码，当每个小组专注于自己的工作时，自然会形成孤岛。这种结构通常会导致软件部署时间过长，甚至两个部门之间很少或根本没有交流。

DevOps 是针对这种孤立情况的解毒剂，在这种情况下，效率低下和缺乏团队合作是常态。然而，要让 DevOps 起飞，开发和运营都必须充分参与，并理解协作需要双方的平等合作。这种协作不仅要在特定阶段进行，还要贯穿项目的整个生命周期。当每个团队都通过提供反馈、帮助确定未来的问题并不断从日常部署中学习来投入工作时，公司就有很大的机会从这些新的协作努力中受益。没有每个人的完全认同，这个过程将永远不会开始。

**协作的演变**

正如 DevOps 是关于超越软件开发的流水线过程一样，协作的概念本身也在发展。DevOps 风格的团队合作不仅仅是打破不同团队之间的隔阂。在敏捷基础设施中，团队成员可以根据当天的需求戴上不同的帽子并交换职责。不同于人们在没有与更大的团队进行任何互动的情况下从事单一的专门任务的场景，您将拥有可以在团队中扮演多个角色的贡献者，并且理解这如何提高整个公司的生产力，而不仅仅是单个部门。团队成员有可能某一天作为开发人员工作，然后另一天从事部署工作。通过 DevOps，组织可以超越一维的部门和一维的贡献。

**现实世界开发团队合作**

对于 IT 行业的许多人来说，DevOps 已经是一个可行的现实。打破孤岛和变得更加敏捷是那些知道增加协作如何帮助他们的业务的公司的目标。最近出版的 Rackspace 电子书[*《DevOps 心态:来自技术领袖的真实世界洞察*](http://www.rackspace.com/devops/the-devops-mindset/?cm_mmc=or-_-dr-_-devops-_-or) 》提供了 devo PS 从业者的一些第一手经验。它包含了对 It 专家的采访，这些专家讨论了是什么导致他们采用 DevOps 方法，并为那些对新的软件部署敏捷方法感兴趣的人提供了建议。

在讨论敏捷团队合作的重要性时，2U 的 James Kenigsberg 评论说，“要实现真正的 DevOps 协作，你需要你的员工真正像一个人一样思考和行动，而不仅仅是名义上的合并。”作为在线大学课程的提供商，2U 面临着快速、可靠部署的巨大需求，并经历了一次转型以增强公司的团队合作。“我们必须翻转模式，”Kenigsberg 解释说，“从个人英雄主义和孤立责任的想法到真正的跨职能合作。”

基金管理公司 HedgeServ 的吉姆·金博尔重申了这种跨部门方法。“可重复性和可扩展性是我们的业务，”他解释道。“DevOps 为我们提供的功能通过鼓励组织内的协作来提出更好的解决方案，从而直接实现了这一点。”

**在开发人员和运营人员之间建立合作伙伴关系**

组建一个技术娴熟、才华横溢的团队，让你的行业羡慕不已，这是每个公司都希望拥有的理想。如果你能够组建一个相当于 X-Men 的团队，仅仅聚集人才并给他们分配孤立的角色是不够的。没有频繁的团队合作，不仅公司目标难以实现，解决问题也几乎不可能。

当开发和运营部门认为自己是在相互合作时，加强合作的基础就会到位，公司的目标，如更快的上市速度、更高的生产率和全方位的敏捷绩效，都将触手可及。有了正确的心态，DevOps 可以成为公司功能的进化步骤，并达到新的效率水平，否则将无法实现。
