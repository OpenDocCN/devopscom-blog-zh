# DevOps Chat:与 IBM 的 Matt Tarnawsky 讨论应用程序现代化和服务虚拟化

> 原文：<https://devops.com/devops-chat-app-modernization-and-service-virtualization-with-ibms-matt-tarnawsky/>

虽然每个人都想专注于开发一个新的应用程序，但事实是已经有很多很多的应用程序了。它们需要维护、更新和现代化。如何将这些传统应用升级到新的基础设施、框架、语言等？App 现代化是一件事。

在本次 DevOps 聊天中，我们与 IBM 的 Matt Tarnawsky 讨论了应用程序现代化和服务虚拟化、测试等问题。

对你有好处。这是一个伟大的故事。你知道，这很公平——在 it 领域，当你看到某人——当我们审视我们的职业生涯时，谁知道五年后我们会从事什么工作。它甚至可能不存在，或者我们今天才知道。所以这是理所当然的。

Matt，正如我之前提到的，我想谈一谈应用程序现代化。在我看来，无论如何，这是一个加载——这个短语是一个加载短语。我们听到很多这样的说法，但是当你和人们交谈时，有些人会说，“我正在更新我现有的应用程序。因此，我正在利用我的传统应用程序，并针对新的基础架构、新的交付方式、新的导入等等对它们进行现代化改造。”因此，真正要做的是使您现有的传统应用程序现代化。

但是，对其他人来说，它更多的是关于开发、部署和管理应用程序的现代化方式。如此现代——应用程序现代化。当然，无论我们如何谈论它，测试在其中扮演着至关重要的角色。所以我们为什么不先做一点水平设置。你知道，Matt，当你听到应用程序现代化时，这对你意味着什么？

**Tarnawsky:** 所以对我来说，它包含了你提到的很多事情 _____ _____ _____ __ 人们谈论的是，我如何 _____ _____ __ 我的应用程序到新的市场，新的界面。比如手机。可以是展示应用程序 _____ _____ ____。这通常需要 _ _ _ _ _ _ _ _ _ _ _ _ _ _ _ _ _ _ _ _ _ _ 他们的界面，将他们从我们的系统中带出来，他们过去可能已经有了这些界面。

但是通常，这也意味着他们想要寻找更快的交付方法，新的开发风格。这些事情通常是密切相关的。你经常不能得到那些快速反馈的更快的开发方法和其他东西，而不看看，我们如何改变我们的整个开发生命周期；尝试陷入瀑布式方法并不能真正让你获得与开始考虑我们如何自动化一切并转移到 DevOps 时相同的加速。

**Shimel:** 明白了。明白了。现在，你在那里中断了一点——我希望正在听的人能够理解这一切。但是让我们——这样我们可以填补。但是，让我们稍微谈谈测试的作用和应用程序现代化的框架。显然，测试很重要，但是如何重要，在什么环境下重要呢？

**Tarnawsky:** 所以我实际上看到——你知道，如果我们采用旧的、单一的应用程序，并希望将它们迁移到微服务上，这是一个很好的测试机会。你把东西放入可以独立测试的小单元中。比如，假设您正在迁移到基于 _____ 的服务，您有机会用 _____ Swiger 或 _____ 或开放 API 来描述它。你有一份合同，然后你可以对照它进行测试。这意味着现在事情有了参数，需要测试什么，如何测试，它可以被导入到许多不同的测试工具中，以帮助他们开始测试，但它是更小的功能。因此，这是一个很好的机会来打破东西，真正看看如何在一个较小的水平上工作。

但与此同时，它也为我们之前没有预料到的更复杂的交互带来了新的机会。假设你拿走了那块巨石。您想要向世界的其他地方或者您组织中的其他团队公开微服务，突然之间，您过去所依赖的集成可能与您将来要使用的不一样了。你需要开始考虑我们如何测试不仅仅是那些更小级别的微服务，还要考虑我们如何随着时间的推移测试这些集成。

我的一个客户在测试方面有一个很好的类比。他们说当你测试一辆车的时候，你不仅仅是测试这辆车。你测试它的一小部分。所以你可以从测试火花塞开始。然后你可能会说，我们将整合一些部分，看看它们是如何一起工作的。我们测试发动机。在未来的某个时候，我们会将发动机放入汽车中进行测试——比如试驾、试车，并确保整个端到端系统也能正常工作。

**希梅尔:**优秀。非常好。实际上，这是一个很好的类比。很好的类比。Matt，我们听到的另一个短语是服务虚拟化，我们中的许多人长期以来都在听这个短语。我认为，当我们谈论服务虚拟化时，人们会更好地理解我们的意思。不过还是那句话，先打个基础。当您谈到服务虚拟化时，特别是在应用程序现代化和测试方面，这对您意味着什么？

Tarnawsky: 因此，服务虚拟化是我们采用某种接口的地方，我们可能已经有了这种接口，它可能是某种无声的休息。它可能类似于 Mqueue。它可能是像 Kicks 这样的传统主机接口。我们提供了某种模拟来处理与界面的交互。可以耦合到应用程序的其余部分。它开始给我们提供——对不起。

它开始为我们提供环顾四周的能力——对此我很抱歉。我有点搞不清楚状况。

Shimel: 没问题。

Tarnawsky: 基本上，它给了我们去除自我限制的能力。因此，如果我们的环境中有云环境中可能还没有的遗留项目，那么实际上，我们可以说，看，大型机连接、SAP 连接，无论是我们当前本地环境中的什么，我们都可以虚拟化，我们可以在云中放置一些东西。事实上，我们的开发人员可以在云中拥有一个完整的环境，不再与那些遗留系统相关联。

很公平。那么这一切是怎么进行的呢，马特？对吗？现在我们有了我们的小零件，我们已经定义了它的含义。在你的世界里，如果你愿意，从组织部署、雇佣、做这种事情的角度来看，你会在哪里看到橡胶遇到道路？有道理吗？

塔尔纳沃斯基:所以我们看到了各种各样的事情。我们确实看到人们在关注这一点，我们希望进行迁移。因此，我们将记录各种环境片段，并让它们可以重放。只是创建虚拟服务的非常非常基本的风格，但它可能是有效的。事实上，他们也可以为他们的 API 测试做同样的事情，只需说，“我们正确地迁移了吗？它是否仍具有上周我们在内部部署时的功能，就像我们现在在云上部署时一样？”

这是一种非常有效的看待事物的方式。但同样，我们看到客户提出了各种有趣的故事，无论是寻求第三方虚拟化的方式，我们都看到客户关注 Apple Pay 或 Experian 等东西，他们每次都必须支付第三方服务的费用。或者实际上，他们可能无法控制它，他们希望模拟如果第三方服务响应或抛出错误代码或其他情况会发生什么。

我们看到客户真的做了一些——我最喜欢的一个故事来自一个客户，他正在测试他们的信用卡发行系统。现在，你可以想象当你在考虑发行一张信用卡时，在某个时候你会去打印一张信用卡。现在，想象一下这类事情的自动化测试。你真的想让信用卡从机器里蹦出来吗？不，你希望能够有效地测试，但不要有这些物理依赖。

所以他们所做的是，他们观察通过信用卡机，发卡机的流量。他们说，这里我们有一个点，我们可以虚拟化我们环境中的一些东西，并消除我们目前拥有的这种依赖性。我们不想印刷真正的信用卡。这是对时间和资源的浪费，事实上，它降低了测试自动化的速度。因此，如果我们能够将其虚拟化，我们可以更快地向开发团队反馈。这正是他们所做的。

**Shimel:**Matt，我们最近在 DevOps.com 举办了一场网络研讨会，你可能已经知道了，通过应用重构实现测试现代化。这是我们正在谈论的那种东西吗？
