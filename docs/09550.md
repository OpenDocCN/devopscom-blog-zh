# 为什么您的 DevSecOps 计划会失败

> 原文:[https://devops.com/why-your-devsecops-initiative-will-fail/](https://devops.com/why-your-devsecops-initiative-will-fail/)

当你在安全行业呆得够久了，你就会习惯于行业炒作机器把一个很酷的创新变成，嗯，哼。这种炒作周期始于每年的 RSA 大会，像我这样的人在展会上寻找新的热点。也许这是 25 年来第二次，我没有亲自参加展会，但似乎 DevSecOps 是每个供应商都喜欢的趋势。叹气。

我不确定是我越来越大的年龄还是这个行业让我变得愤世嫉俗，但我们在这里。

需要说明的是，从客户的角度来看，DevSecOps 的概念有很大的价值，正如即将举行的[devo PS Connect:DevSecOps @ RSAC 2022](https://www.techstrongevents.com/devopsconnect-devsecops-rsac-2022/begin)虚拟活动所证明的那样。从理论上来说，让这些(有时是)不共戴天的敌人一起朝着一个共同的目标努力是很棒的。但是，唉，在实践中(类似于 DevOps 出现时发生的情况)，说起来容易(或者在厂商展台上展示)做起来难。

因此，我得出了一个清醒的结论:您的 DevSecOps 计划将会失败。

大多数组织会谈论一个好游戏一段时间，他们甚至会尝试一些东西，买一些工具包，送人们去培训等。但最终，尽管付出了所有这些努力，你还是猜对了——他们还是会失败。

这是不是每个机构的定局？嗯，没有。但这不是我的第一次牛仔竞技表演，我很确定这一次会像其他的一样以失败告终。

但假设你希望这次有所不同。真的，你知道。你愿意采取一些行动给自己一个成功的机会。如果是这样的话，这里有几个小贴士可以增加你的机会，不管多小。

## 文化

首先，DevSecOps 不是一个东西。它不是一套工具(不管供应商告诉你什么)；你不是在 VAR 线卡上买的。DevSecOps(像 DevOps 一样)是一种文化转变。你的团队需要合作。他们需要合作。他们需要共享激励。他们需要体验在沙盒里玩得不好的后果。

文化转变是艰难的。人们不喜欢改变。他们反抗。他们破坏。他们搞破坏。这听起来很邪恶，但事实如此。所以除非你能改变你的文化，否则你没有机会。

## 左移位

另一个安全营销术语是左移。这意味着您希望尽早将安全测试直接集成到开发过程中。您应该在开发人员的 IDE 中运行安全检查，以确保他们在提交代码之前没有违反安全策略(例如，在源代码中硬编码 API 键)。

您还必须将安全测试集成到 CI/CD 管道中。因此，您需要与您的首选管道一起工作的工具。

## 安全冠军

安全测试会给你的开发者带来焦虑。他们会有假阳性。他们将不得不解决他们可能不理解的问题。这会导致他们错过交货日期。当然，他们应该在第一次就构建好，但是在他们看来，他们会认为是安全人员造成了延迟。

开发人员会很恼火，你必须控制这种恼火——或者让高级管理层平息任何反抗，但这不是一个长期的解决方案。

另一方面，您可以建立一个安全冠军计划，在每个团队中指定(并培训)开发人员来“代表”开发团队中的安全。他们帮助开发人员了解工具和安全编码。出问题时，他们会分散愤怒。

强大的[安全冠军](https://devops.com/?s=security+champions)计划是 DevSecOps 成功的主要标志之一。

## 打破常规

最后，如果在部署过程中发现关键问题，安全团队必须有能力打破构建。我们与许多组织讨论这个问题，他们都点头表示同意。但是他们在骗我，也在骗他们自己。当遇到这种情况时，开发团队中的某个人打电话给高级管理层中的某个人，突然之间，关键问题变成了警报，而不是失败的部署。

无法停止构建将会杀死您的 DevSecOps 程序。这就像质量控制问题迫使某人停止装配线。如果您不停止生产线，可能会产生不合格产品。在安全环境中，这意味着一些关键的漏洞将在以后花费更大的代价来修复。

所以，你有它。如果您能够发展您的文化，将安全测试集成到您的工具链中，授权安全卫士并授权您的团队停止部署，您就有机会成功完成 DevSecOps 计划。

不幸的是，你很可能做不到这些事情，你很可能会失败。这样的话，我想把瀑布边的一些可爱的房产卖给你。
