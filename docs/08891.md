# API 蔓延对数字经济的潜在威胁

> 原文:[https://devo PS . com/API-sprawl-a-emerging-threat-to-digital-economy/](https://devops.com/api-sprawl-a-looming-threat-to-digital-economy/)

新的估计表明，正在使用的公共和私有 API 的总数接近 2 亿个。API 对全球数字经济越来越重要。他们是许多数字平台的支柱，并推动着 T2 的可组合企业模式。但是这种普遍存在带来了蔓延问题。

F5 最近发布了一项研究，调查了 API 经济的总体状况。由 F5 的 Rajesh Narayanan 和 Mike Wiley 撰写的论文[持续的 API 蔓延:API 驱动的经济中的挑战和机遇](https://www.f5.com/pdf/reports/f5-office-of-the-cto-report-continuous-api-sprawl.pdf)阐明了 API 蔓延的状态及其背后的条件。

根据 Narayanan 和 Wiley 的说法，“如果数据是新的石油，那么 API 将成为新的塑料。”这种迫在眉睫的现实将需要一种持续的 API 管理方法，以避免进一步污染数字生态系统。

下面，我将回顾该报告的主要观点，看看是什么导致了 API 的蔓延，并考虑 IT 领导者应该如何应对。

## API 增长状况

API 已经发展成为企业和服务连接和共享价值的标准机制。而且，随着越来越多的公司开始依赖它们，API 经济正在成为大企业——根据 [2020 年云元素 API 集成状态报告](https://nordicapis.com/trends-unveiled-summarizing-the-2021-cloud-elements-state-of-api-integration-report/)，今天 83%的组织认为 API 集成是其业务战略的重要部分。

API 风格越来越多。存在数以千计的公共产品化 API，但更多的私有和合作伙伴 API 正在使用中。从技术上来说，API 有许多不同的形式——它们可能是基于网络的、基于浏览器的或嵌入到设备中的。该报告还确定了单用途 API 和聚合多个数据提供者的 API。

API 用例无处不在，从酒店预订到天气、股票报价机、交通、物联网、DevOps 工作流和许多其他领域。“API 驱动的应用已经渗透到我们生活的方方面面，”该报告称。F5 表示，然而，在评估这一市场时，典型的市场估计通常相当保守。根据 F5 的大胆计算，到 2030 年，我们的活跃 API 将接近 17 亿个。

从启动用例到企业应用程序，API 都具有很高的价值。然而，纳拉亚南和威利说，不利的一面是，这种增长正在导致蔓延。

API 蔓延是一个术语，用于描述正在创建的数量呈指数级增长的 API，以及部署 API 的分布式基础设施位置的物理分布。

## 推动 API 蔓延的因素

纯粹的增长。有消息称，到 2030 年，开发者的数量将增长到 4500 万。SlashData 还估计 30%的开发者已经在使用 API。随着市场上的 API 数量达到数百万，管理增长带来了巨大的挑战，尤其是在没有适当的治理和最佳实践的情况下。

**缺乏标准**。“缺乏一个通用的共享模型导致了 API 的蔓延，”该报告称。API 设计标准确实存在，但是指导方针通常为服务之间的细微差别留有余地。围绕特定行业也出现了标准，如金融数据交换(FDX)。虽然对单个行业有帮助，但这不会同时推动多个行业的发展。缺乏标准化会导致不同的版本化方法和集成挑战。

**新的开发方法**。集成需求迫使许多新的 API 联合不同的业务应用。[微服务架构趋势](https://devops.com/webinars/get-loose-microservices-loosely-coupled-architectures/)也加剧了蔓延，因为“API 既可以通过微服务向北连接到接口，也可以在微服务之间水平连接。”

**持续软件开发**。持续的开发，加上将业务单元与定制需求联系起来的需要，可能会产生同一个 API 的多个版本。这很快会导致维护困难、文档过时和客户端崩溃。该报告还引用了不断增长的数据创建和“一切即服务”趋势作为 API 蔓延的先兆。

**各种计算进化**。报告称，新的计算趋势也在促进蔓延。业务部门可能在不同的[内部](https://devops.com/the-role-of-apis-in-mainframe-modernization/)或云环境中运营。当这种混合情况发生时，API 可能会分散在许多位置，变得难以跟踪。可能会创建特定的连接来迎合边缘计算和物联网设备等新环境，从而进一步增加蔓延。

## API 蔓延的问题

依赖 API 的数字经济也依赖于这些服务的潜在稳定性和可用性。然而，“API 有一个保质期，如果被开发者忽视，就会变得不受支持，”该论文承认。对于分散在分布式云中的 API，很难保持服务的可靠性。除了可靠性问题，API 蔓延还有许多其他可能的后果。

首先，大规模的管理和运营变得困难。并不是所有的 API 都是基于规范的，比如 [OpenAPI](https://www.openapis.org/) 。这意味着 API 文档可能不那么精简和易于访问，从而损害了可发现性和入门。“用传统的或遗留的网络方法，连接这些 API 的简单方法可能是不可能的，”报告补充道。

由于开发人员是 API 的主要消费者，蔓延可能会对他们的集成体验产生负面影响。维护一个不断膨胀的 API 依赖库可能很麻烦。此外，API 很容易随着时间的推移而演变和升级，当端点在没有预先通知的情况下改变时，会导致客户端崩溃。即使是 API 功能的微小变化也会产生巨大的影响，因为所有可观察到的 API 行为都会被依赖，无论是否有文档记录。(这就是所谓的[海勒姆定律](https://www.hyrumslaw.com/))。

但是 API 蔓延的安全后果可能是最麻烦的。2020 年，多达 91%的企业经历过 API 安全事件。Salt Labs 发现，2021 年年中，恶意 API 流量也增长了惊人的 300%。API 使用的增加无疑会导致更频繁的攻击。

“不受管理的 API 蔓延是一个迟早会发生的安全漏洞，”该报告称。API 通常采用 API 密钥进行身份验证和授权，但是这种方法容易被误用。凭据经常暴露或配置错误。坏人可以利用 API 窃取大量数据和计算能力。因此，蔓延正在破坏对 API 连接和响应的信任。

## 缓解蔓延

随着 API 经济的增长，预计在未来十年将达到十亿大关，解决安全、稳定的集群间通信将变得更加重要。F5 表示，现有解决方案(网关、入口控制器、服务网格以及正向和反向代理)不足以应对这一问题。

“因此，我们认为该解决方案应该包含一个中介(代理)设备，专注于解决群集间的连接、安全性和集成挑战。”作者称这个解决方案为 API Gateway 2.0，他们将其描述为双向应用程序级代理。实现这种中间件的技术已经存在，也许它可以帮助公司解决蔓延问题。

除了 Narayanan 和 Wiley 的建议之外，API 所有者还可以采取很多措施来避免增加蔓延。以下是一些例子:

*   将 API 视为一种产品
*   改善开发人员体验
*   使用规格驱动的开发
*   确保最新的文档和代码库
*   使用一致的端点命名
*   为版本控制和弃用设置明确的指导原则
*   通过 OAuth 和 OpenID Connect 超越 API 密钥
