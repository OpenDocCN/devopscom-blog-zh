# 锁定您的工具链

> 原文：<https://devops.com/lock-down-your-toolchain/>

我们在敏捷和 DevOps 方面取得了惊人的成就，将 IT 响应能力提高到了大多数人都不敢相信的水平，我们的业务对手甚至在十年前还只是梦想。想想看，我们可以签入单个源文件，然后启动一系列事件，这些事件涉及六个或十几个软件工具，它们共同引导整个应用程序完成从开发到部署的整个过程。通过所有这些，只有部署可能需要人工交互，这取决于公司策略。在许多地方，甚至部署都不需要一个人。只是发展。

我们没有因为行动缓慢而走到这一步，可悲的是，我们也没有因为以安全第一的方式行事而走到这一步。由于 DevOps 诞生时的安全有毒心态，我们不得不回头尝试在最初被忽视的地方增加安全性。但是我们在很大程度上控制了这个过程。我们拥有处理权限、监控连接、扫描源代码、查找支持软件中的漏洞，甚至在构建之前扫描整个容器定义的工具。

知道你们大多数人还没有什么吗？真正安全的工具链。这是一项糟糕的政策。一个有经验的攻击者将会追踪其中一个聚集位置。像[容器管理](https://containerjournal.com/?s=container+management)服务器这样的地方，让他们可以访问大量资源。或者工具链，这使他们能够访问组织开发的每一个软件。如果攻击者想在不被发现的情况下进行代码注入，那么负责构建和部署的工具链就是这样。

然而，为了易于使用和跟上迭代，你们中的许多人将工具链的凭证存储在容易访问的地方，以便无论哪个工具正在运行，都可以访问工具链的每个部分的凭证。我希望我可以说“没有人再把数据保存在平面文件中了”，但是，唉，我不能。

它不够安全的原因之一是构建工具链介于开发安全和运营安全之间。这意味着我们来自开发背景的团队并不关注构建链的安全性，我们来自运营背景的团队/团队成员也不关注开发人员最初选择和部署的工具链。

这就引出了我的建议。让安全专业人员负责工具链，以验证它在每一步都被锁定。工具链有很多不安全的地方，不仅仅是糟糕的存储凭证。工具链触及了它所有的皇冠上的宝石。所以确保它不是敞开的。如果您的组织没有专门的[安全人员](https://devops.com/?s=DevSecOps)，请顾问来审查工具链并提出建议。有一些很棒的安全团队会乐意帮忙。

继续摇摆。您的团队正在以前所未有的速度构建和部署——即使 DevOps 在您的组织中很“慢”,它仍然可能比它的前辈更快——来做*任何事情*。所以，继续创造奇迹，锁定你的城堡的内部保持。
