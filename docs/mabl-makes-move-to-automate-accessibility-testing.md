# mabl 开始自动化可访问性测试

> 原文：<https://devops.com/mabl-makes-move-to-automate-accessibility-testing/>

mabl 今天发布了测试版[的自动化可访问性测试功能](https://www.prnewswire.com/news-releases/mabl-extends-low-code-test-automation-platform-to-include-accessibility-testing-301533708.html)，它将其添加到其软件即服务(SaaS)平台中，用于测试应用程序。

mabl 联合创始人 Dan Belcher 表示，鉴于各种法规遵从性要求中包含的可访问性要求越来越高，因此更有必要在应用程序开发生命周期中尽早解决这些问题。否则，组织可能会发现他们要为未能遵守[网站内容可访问性指南(WCAG)2.0 版和 2.1 版中概述的](https://devops.com/?s=accessibility)可访问性要求负责。

贝尔彻指出，与此同时，不解决可访问性问题的应用程序提供商也很可能会发现自己在公共社交媒体论坛上受到指责。对品牌声誉的损害可能相当大，他补充道。

mabl 创建的 SaaS 平台使得使用被称为 axe-core 的低代码引擎创建可重用的测试成为可能。这些测试可以被适当地插入到 DevOps 工作流中。Belcher 说，这种方法允许专门的测试团队以不需要在 DevOps 平台上进行的方式创建测试。

不管法规要求如何，Belcher 表示，在持续快速发展的数字世界中，显然需要将残疾终端用户包括在内。不幸的是，作为一个整体，软件业才刚刚开始把易访问性作为头等大事来考虑，他补充道。

Belcher 说，目标应该是通过易访问性回归测试实现快速反馈，例如，解决对比检查。然而，他指出，随着应用程序的不断发展，这些测试不太可能消除由可访问性主题专家进行定期审核的需要。贝尔彻指出，这些审查将解决从用户界面到整体性能影响的所有问题。然而，他指出，也许有一天可以通过自动化解决 WCAG 指南中确定的 50%的无障碍问题。Belcher 补充说，在开发过程中扫描代码将防止积压和可访问性债务的累积，在应用程序部署后，解决这些债务的成本将会高得多。

Belcher 说，总的来说，自动化不可能完全消除对专门测试团队的需求。他补充说，mabl 等测试平台旨在使有限的测试资源能够跟上应用程序开发的加速速度。

对于所有的意图和目的，可访问性是任何全面的软件质量评审的子集。开发团队越早认识到这一事实，他们就越不可能在将来为没有早点认识到这一点而道歉。
