艾伦·希梅尔:大家好。艾伦·希梅尔——DevOps.com 又来参加 DevOps 聊天了。这一集的嘉宾是 LeftShift IT 的约翰·哈里斯。约翰，欢迎你。

约翰·哈里斯:谢谢你，艾伦。很高兴来到这里。

艾伦·希梅尔:谢谢。约翰，你在《T2》T3 中的角色是什么

*John Harris:* 我的角色是 LeftShift 在 DevOps 的持续交付主管。

艾伦·希梅尔:好的。约翰，你知道吗？我们的观众中有些人可能不熟悉或者可能以前没有听说过 LeftShift IT。为什么不快点给我们一个左移的背景？

约翰·哈里斯:是的。我们主要是持续交付领域的服务咨询公司——devo PS 持续测试领域。我们与 IBM 和许多其他不同的供应商进行了大量的合作。总部设在伦敦和英国，但我们在美国也有很多业务办事处。

艾伦·希梅尔:当然。John，在我们进入今天关于持续测试的话题之前，我希望你花点时间谈谈 DevOps 咨询这个新兴领域。好像要爆炸了。上个月，我在伦敦参加了 DevOps 企业峰会，提供 DevOps 咨询服务的公司从大到小，多种多样，让我大吃一惊。LeftShift 进入 DevOps 咨询领域多久了？

约翰·哈里斯:我想你可以说，我们断断续续地在这个地区生活了十年。我想大多数人会说 DevOps 只是在比利时最初的 DevOps 创造时代才出现的，但我想很多人会说这个学科本身在这个名字之前就已经存在了，所以我想 DevOps 的相关流程你可以说我们已经参与了很长时间。*T3】*

艾伦·希梅尔:当然。John，你知道，如果没有 DevOps consulting 持续交付咨询服务的受众，我不认为我们会看到 DevOps consulting 的爆炸式增长。这就是你在《左移》中看到的吗？我想生意不错。

约翰·哈里斯:是的。这是一个有趣的领域，我想 DevOps 顾问可能会说，你不只是想在你的公司里创建一个新的 DevOps 实践。你不只是想引入这些部分，但这是一个困难的循环，因为你必须支持这些流程，你必须以某种方式将这些知识融入到组织中，所以我认为我们的模式是坚定地帮助人们开始并美化他们自己的实践和他们自己的良好流程，然后引导他们去做，所以是的，绝对是这样。

*艾伦·希梅尔:*明白了。John，让我们稍微转到这里，讨论一下持续测试。所以，我要告诉你们，在 DevOps.com，测试、持续测试、自动化测试的整个概念是 DevOps 中争论最激烈的话题之一，在某些方面也是有争议的话题。想知道你是否看到类似的？

约翰·哈里斯:是的。我也这么认为我的意思是，我认为在迄今为止的开发中，测试传统上被视为最终必须完成的必要部分， 但我认为，现在连续交付正在成为一种真正的纪律，人们被迫越来越快地走在市场的前面，像这样的事情——再也不能把测试视为开发的穷亲戚了，我认为测试真的变得——人们意识到我可以以我想要的速度把东西投入生产，但实际上它是以它需要的质量投入生产——产品仍然在市场上可行。 所以我觉得绝对可以。早就应该进行测试了，但是它确实已经开始了。*T3】*

*艾伦·希梅尔:*明白了。约翰，你能在没有自动化的情况下持续工作吗？

*约翰哈里斯:*我个人不这么认为。我认为 DevOps 真的——对我来说最热门的词之一——devo PS 和持续交付真的是组织。我认为你必须有流程支持，但我认为当你在生产中通过我们开发生命周期的所有这些阶段更快地交付任何东西时，我认为你不能在没有自动化流程的情况下按我们需要的规模进行扩展。

我的意思是，当你谈到它形成的联盟，如网飞和 Etsy 以及所有这些人，现在三、四年前，你不可能提供这种规模，而不加入自动化工具以及围绕它的流程和文化，以使它能够达到你需要的效果。

艾伦·希梅尔:当然。约翰，我认为争议之一——不是争议，而是围绕这个问题的一个大讨论点是，这对传统问答角色和问答领域的人们意味着什么？

约翰·哈里斯:是的。我认为团队传统上是分裂的。我的意思是，显然我们有这些孤岛和开发运维，试图解决开发、测试和运维等方面的孤岛。我再次认为，测试是被排除在外的事情之一，但当然在我们工作的客户中，我们看到许多 Q/A 团队以某种方式被重新培训，被带到组织的开发方面，他们的实践被改变为更像开发软件工程的工作方式。

我们肯定会看到一些在这个领域传统的地理分裂正在被打破，也许是测试，相当多的测试外包工作。其中一些会上岸。测试人员变得越来越技术化，但是，是的，再培训和问答角色现在需要的不同技能，使之更加技术自动化。这肯定是我们的许多客户现在正在努力解决的问题，我认为他们的决定不仅仅是技术性的，而是涉及到公司的文化方面以及公司如何组织自己。所以，是的，这是一个有争议的领域。我同意。

艾伦·希梅尔:是的。的确是这样，这确实是其中之一——对于问答领域的人来说，这是一个钱包问题，对吗？他们认为生计受到威胁，尽管我不这么认为。我认为这实际上允许他们飞得更高，做更大更好的事情，但肯定有这个因素。

John，我想谈一下服务虚拟化，实际上我们将在 8 月 23 日举办一场网络研讨会，我们将在会上看到一家真正的独角兽公司——Mertz Shipping Lines，我认为它在传统航运领域已有 80 年的历史，而不是你认为的谷歌、脸书那种公司。但他们将谈论服务虚拟化，向左转。有什么想法吗？

约翰·哈里斯:是的。服务虚拟化是我们经常使用的一个领域，尤其是围绕 IBM 工具集。是的，默茨做得很好。我们工作的几乎所有组织都在尝试进行某种并行开发，我的意思是我们看到的专利之一是在他们的一些遗产之上构建 y 和 Rex API 层，所以业务服务之类的东西，他们希望在空间层中与 ____ 和 Rex 以及用户进行一些并行开发。因此，SV 真的能够为这些团队提供一个非常坚实的平台，因为一切都基于他们的服务接口定义，它真的可以在那里进行一些适当的并行开发，我们看到这些开发团队留下了一些巨大的转变，实际上有能力在他们甚至开始相互集成之前识别一大堆缺陷。

我想我们有一个团队在使用服务虚拟化的早期测试阶段发现了 100 多个缺陷，甚至在他们与其他服务集成之前。他们甚至能够在进入下一个集成环境之前就解决这些问题，因此这是一个真正伟大的成功故事—服务虚拟化。

艾伦·希梅尔:是的，绝对是。我想我去年采访了互连公司的 Mertz 人员，这是一个有趣的案例研究。我想我们应该提到——有一本*虚拟服务虚拟化的书，人们可以下载，但我面前没有网址。我会尽量把它写进我们的笔记里。我认为这是 IBM 为对此感兴趣的人提供的。但是，John，让我们回到——你认为——你在 LeftShift 中的角色是什么，你认为什么是下一个前沿领域——不是战斗，而是有意义的地方，因为在不久的将来缺乏更好的词，所以有需要赢得的战斗？*

*John Harris:* 我认为最大的变化是，我们现在开始看到这种转变，但这仍处于早期阶段，习惯是，对于独角兽和初创公司来说，这可能不会是一个大问题，但对于企业来说，用户界面测试肯定仍然非常普遍，我们谈到了测试金字塔，我们的大多数测试都是单元测试之一，然后我们将有一个良好的集成测试基础，如果你愿意，还有一小部分 UI 测试。

我认为目前我们的许多组织都有一个颠倒的概念，因为它很容易进行 UI 测试并将其应用到业务中，但现在当我们走得更快时，我认为传统的 UI 测试重点将不得不消失，但当我们谈论 API 经济等一切事物时，许多系统现在都没有 UI。推出了许多都是 API 驱动的智能设备。

我认为所有东西都必须经过集成测试。我认为 API 及其背后的逻辑才是核心业务功能的真正所在，所以我认为这才是重要性所在，它将推动组织并向他们展示集成测试和 API 测试并不是正确的方法，显然，自动化是他们交付管道的一部分，这有时在 UI 端更难做到。

艾伦·希梅尔:是的。因为我思考过如何直接处理这个问题。难道 API 测试——至少测试本身不应该来自编写 API 的同一批人吗？换句话说，负担在哪里？提供 API 的人是否也应该为它提供测试，或者这有点像把狼放进鸡笼，对吗？你在用别人的 API。你的责任是确保这实际上与你的系统一起工作。

约翰·哈里斯:是的。我认为这是一个非常有趣的观点，围绕它有不同的哲学。我自己，我们在虚拟化和测试方面做了很多工作，所以我们会说，如果你正在开发接口，那么你也负责开发基本的虚拟服务，以便与你的消费者集成，但我非常支持消费者开发 API 测试或你消费的 API，因为如果你想一想我是消费者，我可能不需要 API 中的每个字段。比方说，如果我正在开发松散耦合的微服务，我可能只使用几个。

所以我真的很赞成每个使用 API 的团队开发他们自己的测试，然后将这些测试贡献给 API 团队，这样他们就可以在调用、每次构建或每次部署时运行。因此，如果我更改了功能，而那些测试失败了，我知道是哪个团队给了我这些测试，我可以去提醒他们，“嗯，看，我已经做了更改，我只是要中断您的实现或您对该 API 的调用，但我可能没有中断任何其他人的实现。”

我的意思是，我认为让测试每个功能的负担，特别是当我们去更松散耦合的服务时，对开发 API 的团队来说是一种不合理的负担。

*艾伦·希梅尔:*明白了。你知道吗，约翰？信不信由你，正如我告诉你的，我认为在我们上线之前，时间过得真的很快，我们就要结束了。如果有人想了解更多关于 LeftShift 的信息，他们可以去哪里？

John Harris: 我认为是 LeftShiftIT.com，我们刚刚更新了网站，所以他们可以在那里找到我们的服务和产品以及我们提供的一切。

艾伦·希梅尔:好的。最后，约翰，我想以一个问题来结束我的演讲，我想我会问每一位来宾，这是为了我们的观众，如果他们要读书，他们应该——如果他们——希望如果他们真的读书，每个人都应该读书。你推荐我们的读者读的下一本书是什么？

*John Harris:* 我是 Jez Humble 的超级粉丝，我不打算推荐*连续交付*，因为我相信听过播客的每个人都读过，但《T4》是一本关于精益企业的书-*高绩效组织如何大规模创新*，它是 Jez Humble，Joanne Molesky 和 Barry O'Reilly 写的一本非常棒的书，适合任何希望进入这类领域的企业。

艾伦·希梅尔:当然。实际上我把它放在办公室的书架上。干得好，约翰。无论如何，我想这将会耗尽我们在本期 DevOps Chat 的时间。John Harris，LeftShift IT，非常感谢您成为我们的客人，我们期待听到更多关于 LeftShift 和持续测试服务虚拟化的信息，并期待继续取得成功。

约翰·哈里斯:太好了。谢谢你邀请我，艾伦。

艾伦·希梅尔:谢谢。我是艾伦·希梅尔，在 DevOps.com 和德沃普斯聊天。下次再见，祝你愉快。*T3】*
