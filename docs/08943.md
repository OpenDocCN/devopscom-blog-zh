# Log4j:都是供应链的问题，宝贝！

> 原文:[https://devo PS . com/log4j-its-all-about-the-supply-chain-baby/](https://devops.com/log4j-its-all-about-the-supply-chain-baby/)

2021 年，*t**he*devo PS 和 DevSecOps 的安全故事一直是[供应链](https://devops.com/?s=supply+chain)。因此，我们目前正经历着所有供应链问题之母，Log4j Log4Shell RCE 漏洞将在今年结束。

我不会浪费你的时间重复什么是 Log4j，为什么它如此危险，并警告你测试所有你自己的和第三方的应用程序，以确保你不受这种影响。不过，我要说的是，我不认为 Log4j 是一个 bug——我认为它是设计上的一个缺陷。更糟糕的是，正如我们所看到的，设计中有*多个*缺陷。所有讨厌 Java 的人都回到盒子里。Java 并不比我们经常用来构建应用程序的任何其他语言更好或更差。

那么，供应链安全到底有什么特别之处呢？为什么 DevSecOps 团队、开发人员和我们的安全事件管理团队必须将此解释为一个警告，并从现在起保持在他们游戏的顶端？很简单。在萨苏普斯的时代；在这个世界上，我们通过拼凑和缝合众多第三方组件(一些[开源](https://devops.com/log4j-is-there-such-a-thing-as-too-much-open-source/)，一些没有)来制造应用程序，我们越来越依赖于代码，即使是在我们自己的应用程序中，也不是我们编写的。我们可能也没有测试它。我们比以往更加依赖“社区”和未知的第三方供应商。这些应用程序和组件一直是一个信念的飞跃，今年其中一些鸡已经回家休息了。

白宫关于改善国家网络安全的行政命令要求为应用程序制定软件材料清单(SBOM );正如他们在拉斯维加斯的餐桌上所说，这是一个良好的开端，但我们需要更多。如果我们使用的应用程序(包括 SaaS 应用程序)真的有 BOM，你们中有多少人真的会去阅读它并根据信息采取行动？除了在 BOM 中包含所有这些组件，供应商还有什么义务来确保他们替换和更新易受攻击的代码？

随着开源的出现，这个问题变得越来越尖锐。我的合作伙伴兼同事 Mitchell Ashley 最近询问过多的开源是否是一件坏事。我不知道我会不会去*那个*远；就我而言，越多越好。但是我们确实需要更好的监控和管理开源代码。在正式上市前，可能还需要更多的测试？是的，凯迪拉克开源项目可能有资金和时间来做这件事，但以 Log4j 为例:这是两个家伙在空闲时间维护它，并试图为社区做正确的事情。他们怎么可能跟得上？

我曾经以为答案就在回购中。我在 Sonatype 的朋友有一个组件防火墙，它不会让你在没有警告的情况下下载旧的或易受攻击的代码版本。如果你是第一次下载，这没什么。但是如果它已经被嵌入了——然后呢？我们可以扫描应用程序中的旧组件吗？我在 JFrog 的朋友可以用 x 射线做类似的事情，甚至更多。Sonatype 的 Nexus 平台还允许当前和潜在用户扫描旧应用程序的漏洞— [包括扫描 Log4j，绝对免费](https://www.sonatype.com/products/vulnerability-scanner)。但这就够了吗？

所有这些问题都涉及到开源组件。第三方 SaaS 应用呢？通常，我们甚至不知道那些 SaaS 应用程序中有哪些组件或代码。再说一次，这是我们在使用它们时的一次信念飞跃。

这些供应链安全问题是一个我们刚刚开始理解的问题。虽然 2021 年是这种东西受到追捧的一年，Log4j 可能会被认为是第一次最大的一次，但就我而言，我不敢想象还会发生什么。

我知道解决这一问题和任何未来供应链问题的唯一方法是开发人员、DevSecOps、DevOps、SREs 和我们的安全事件响应团队通力合作。就像大多数安全问题一样，供应链安全是我们的全部责任。  我希望 Log4j 没有把你咬得太重，并且你在你的曝光率之上。但是，让我们开始思考下一个问题，以及如何更好地为它做准备。
