# 你在 DevOps 上犯了什么错误(以及如何改正)

DevOps 现在是遥遥领先的顶级软件部署策略—[](https://www.ahead.com/resources/devops-stats-growth-value/)77%的组织表示他们使用该方法来推出新软件。这是一个好消息，因为 DevOps 方法是业务成熟度的重要标志，它的好处(稍后将详细介绍)是如此明显，以至于大多数组织不需要确信 DevOps 就是未来。

坏消息是大多数组织在开发操作上是错误的，这可能是为什么只有 [10%](https://www.ahead.com/resources/devops-stats-growth-value/) 报告他们的组织在实现快速软件开发和部署方面“非常成功”。我的公司在 2019 年开始了 DevOps 之旅，我们开始构建下一代 ERP 软件。我们致力于这一战略，但在此过程中也犯了错误。

我们的经验告诉我，DevOps 的典型方法(我们最初采用的方法，现在许多组织都采用的方法)是一种改进，但思维模式的转变可以真正释放 DevOps 战略的潜力，使公司能够实现利益最大化。下面让我们仔细看看公司可能出错的地方——以及当他们做对时会得到什么。

## 当组织把开发工作做错了会发生什么

DevOps 的核心概念很简单:在开发阶段就应该考虑操作上的问题。随着 SaaS 解决方案越来越受欢迎，像停机这样的问题应该是每个人的问题的想法越来越流行。许多组织将此解释为应该有一个 DevOps 团队作为开发和运营团队之间的桥梁。

实际上，完全合并团队更有意义。如果 DevOps 团队位于中心，开发和运营方面的孤岛仍然存在。如果有三个团队而不是两个团队，沟通会更加困难，用户需求仍然会被忽略。一个合并的操作通过创造一种“你建立它，你运行它”的思维方式来解决这个问题，这消除了脱节。

我们发现，合并后的 DevOps 团队从最早的阶段就将运营问题作为工程流程的一部分来处理。这种方法简化了识别和解决操作问题的过程，使软件更有弹性，更好地满足用户的期望。

## 当公司正确开发时会发生什么

简而言之，让这种心态转变以获得正确的 DevOps 可以让您的组织构建更好的产品，加速更新，在团队之间建立更好的一致性，并更有效地执行任务。这改善了每个人的体验，包括组织、客户和最终用户。

其中一个主要好处是更长的正常运行时间。如果你从一开始就整合了操作元素，你就降低了失败的风险。当这些组被分开时，运营团队通常会发现很难查明问题，而且系统越大，找到问题的时间就越长，这可能会导致客户不满意和无法接受的长时间停机。

通过合并工程和运营，你消除了一些工程师将运营视为他人问题的倾向。合并团队的 DevOps 策略使运营成为每个人的问题。如果出现问题，每个人都会在深夜接到求助电话，这让我们认识到首先要主动预防问题的必要性。结果是更高的稳定性和可靠性，以及更满意的客户。

## 最大化 DevOps 优势

我们的经验告诉我们，合并工程和运营是最大化 DevOps 优势的最佳方式。这是一个没有立即发生的思维转变；一旦它发生了，我们意识到它的力量有多大。这种转变的一个重要组成部分是通过雇佣运营专家和必要时磨练现有员工技能来提高团队效率。

另一个重要的因素是让软件工程师参与到新的方法中来，并明确这种改变是优先考虑的。如果目前的方法运行得非常完美，那就更容易随波逐流了，但是这阻止了您的 DevOps 策略发挥其全部潜力。如果你优先考虑改变并承认其中的挑战，你就能克服它们并获得全部收益。
