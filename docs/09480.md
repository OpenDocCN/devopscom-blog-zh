# 让我重申一下——不要急于重复

> 原文:[https://devops.com/let-me-reiterate-dont-rush-to-iterate/](https://devops.com/let-me-reiterate-dont-rush-to-iterate/)

在 DevOps 世界中，我们花了大量的时间来讨论工具和角色——而且总是有新的工具和角色的修改要讨论。

你知道作为一个行业我们不讨论什么吗？处于 DevOps 的日复一日的[折磨](https://devops.com/?s=toil)。哦，你可以去像 Reddit 这样的论坛，和其他从业者进行讨论，但是权威人士不会讨论实际工作是什么样的。我暗自怀疑这是因为我们大多数专家一般不再开发代码，更不用说有一个工作的 DevOps 环境了。我当然不像以前那样多写代码了，而且我比我在权威领域的许多同行开发得都多。声明一下，我们*确实*在我的公司有一个工作 DevOps 环境(两个，但是一个几乎从来没有启动过)，但是我会很友好，不会因为在这里谈论它而转移话题。

我们已经不着边际地谈了这件事，但我想对你们大家直说。您的绝大多数组织不需要不停地迭代。而那种不能过多迭代的心态，根本就是假的。虽然过去保护产品免受流氓开发者攻击的门卫和德国牧羊犬反应过度了，但应该有一个令人高兴的媒介。每次在生产中引入变更，都会有一定的风险。如今，我们有大量的工具来限制这种风险——开启/关闭新功能的特征标志，在进入生产之前自动运行测试，等等。—但仍有可衡量的风险。而“我们可以回滚”只适用于小的、未分段的项目。一旦你进入了嵌套依赖(微服务是一个很好的例子)，那么版本控制就开始使回滚成为一个问题。了解正在改变的事物之间的相互联系是一件大事。因此，这并不像耸耸肩那样简单:“只需重新构建和部署，无需更改，”人们会让你相信。

在很大程度上，这就是为什么*这位*专家没有出来谈论迭代速度——每六个月或每次 Bob 签入代码时进行迭代的决定在很大程度上是组织/应用程序的决定。组织处理完整迭代的速度有多快，应用程序多长时间需要一次？答案很少是供应商为了向你展示他们有多棒而抛出的疯狂数字。有大量的组织/应用程序组合，它们的变化率足够慢，以至于它们可以诚实地安排完整的迭代，并计划围绕底层组合件中的漏洞补丁进行更新。大多数只需要迭代几个月。很少(我强调*很少*)人需要每天迭代，更不用说一天超过一百次了。它们确实是特殊的雪花，有着超级扭曲的环境。

让我有点吃惊的是，敏捷和 DevOps 的兴起部分是因为我们超负荷工作而释放了 It 时间。然后一旦完成，我们决定提高变化率，这样它就…超载了。一个发布需要开发*运营*时间，*开发*运营时间和安全时间。构建、测试和部署需要资源。不要过度。

按需构建。在对客户有利时部署。继续摇吧。我们在这里使用您的应用程序，或者您的应用程序在幕后工作，保持业务运行。不管怎样，你比任何人都清楚你的应用和组织需要什么。不要跟着“一天 100 亿部署！”人群；推动有效的方法。而*你*就是那个部门的 SME。
