# 更快更好的测试？

> 原文：<https://devops.com/faster-and-better-testing/>

敏捷和 DevOps 的整个概念是更快地迭代开发，并以更及时的方式交付结果。随着我们对这两种方法的了解越来越多，我们制定了一些流程和策略来提高所创建内容的质量。早期的质量思想，如“我们可以再做一次迭代”，仍然存在，但是很大程度上没有被使用。回滚在大多数组织中都是一件事，尽管环境更加复杂，但是在迭代通常代表的较小开发变更中，回滚更容易管理。

一旦我们超越了整体，“任何减慢这个过程的东西都是不好的！”贯穿开发运维与敏捷开发早期实现的心态，我们开始看到令人惊奇的事情。但是在敏捷和 DevOps 创造的快速迭代世界中，包括安全性和功能测试这样的事情是不可行的。在瀑布的缓慢迭代世界中，它们在大多数组织中也是不可行的。在绝大多数组织中，人员配备和时间限制使它们远未达到最佳状态。但是将它们放入 DevOps 过程中，挑战了供应商找到更快做事的方法，挑战了组织以稍微不同的方式看待质量和安全性。

现在，我们有工具可以以 DevOps 的速度进行测试，无论是功能测试、集成测试还是安全测试。支持着眼于安全性的敏捷开发的工具。现在，越来越多的开发人员认识到他们在应用程序安全中的作用，并且正在创建更安全的代码。

关注 AI/ML 在测试中的应用——所有形式的测试——以及它在未来一两年的发展。完全自动化也是如此。我毫不掩饰我将测试领域的 AI/ML 视为高级自动化，因为它支持什么，所以也许“看看有什么 AI/ML 支持的自动化可用”更准确。创建、运行、处理结果，甚至修改代码来解决问题的能力如今都已经存在，而且越来越好。更快、更准确、更自动化。

就像入侵检测变成入侵防御时一样，我不会只是打开它然后走开，而是考虑一下提升开发人员技能的能力；"以下是自动系统发现的问题列表，以及它建议的修复措施列表."他们实际上是在展示如何做得更好。

我是一名开发人员，所以我不想冒犯你。如果你得到的信息告诉你，“这是一个问题；下面是发生这种情况的原因，下面是解决这种情况的方法。仔细看看是什么，为什么和如何解决它，并将其融入到你的工作中。

继续摇摆。我们将制作更稳定、更安全的代码，但你让系统在没有反馈的情况下继续运行，并修正了错误。这使得尽早发现错误变得更加容易——在它们破坏生产和进行消防演习之前。
