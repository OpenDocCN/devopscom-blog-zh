# DevSecOps 洪水:选择正确的工具

> 原文：<https://devops.com/devsecops-deluge-choosing-the-right-tools/>

在过去几年中， [DevSecOps](https://devops.com/?s=DevSecOps) 已经成为许多前瞻性企业的首选安全流程。

这些组织已经认识到，在产品和应用程序开发的后期阶段修复漏洞对任何人都没有好处，除了网络罪犯。因此，他们彻底改革了传统流程，将开发、运营和安全团队联合起来，组建了 [DevSecOps](https://www.uleska.com/blog/ultimate-guide-devsecops) ，这些团队现在协同工作，合作将安全性嵌入到应用和软件开发的整个流程中。

这样做的好处是显而易见的；尽早发现并修复漏洞可以阻止网络犯罪，同时让客户高枕无忧、节省成本并获得更好的产品。然而，当过渡到 DevSecOps 时，许多组织面临的挑战之一是使过程无缝，确保它不会减慢软件开发生命周期，使安全检查更加繁琐或在团队中引起挫折。

安全团队需要确保他们的 DevSecOps 计划为组织的文化带来价值，同时加快修复时间，获取正确的指标，并根据风险对问题进行分类和优先排序，以便不会出现可能造成重大损害的情况。

此外，他们还需要选择正确的 [DevSecOps toolkit](https://www.uleska.com/blog/devsecops-tool-examples-that-will-alleviate-your-workload) 来执行应用程序安全性测试，同时确保这些工具可以轻松集成到软件开发中，并可以跨多个项目使用。当确定测试工具时，DevSecOps 团队经常被可用的巨大范围和数量所淹没，这使得选择使用哪些工具和学习如何使用它们成为一个雷区，即使对于那些受过适当培训和知道如何使用它们的人来说也是如此。

一个 DevSecOps 自动化程序需要大量的技术工具和文化因素来使其工作。安全团队需要静态分析工具来检查代码，需要第三方库分析来检查依赖性，需要单独分析来检查基础架构即代码(IaC)配置，需要扫描仪来检查容器中的问题，需要工具来测试运行的系统，需要云安全检查和基础架构测试来检查补丁和端口。他们还需要将这些工具与每个团队使用的正确技术相匹配，并跟上不断的变化。

考虑到所有这些复杂性，DevSecOps 团队如何克服这些挑战并使用正确的工具集构建一个有效的 DevSecOps 程序呢？

以下是一些帮助他们一路走来的提示。

### 保持安全流程的灵活性

您的团队将使用不同的技术栈、不同的语言、框架等等。如果你把你的过程和一些工具绑得太紧，当事情发生变化的时候，就很难加入新的检查。请记住，我们的目标是一个具有适当可见性的一致的、可重复的安全流程——技术工具是其中的一部分，但它们不是整个流程。12 个月后，您将需要哪些现在没有的安全检查？36 个月呢？这些是必须不断提出的问题。

### 自动化是你的朋友

如果开发管道是平稳地自动运行的，那么任何手动的安全步骤都不适合这个过程。将安全工具一起自动化—从协调它们的运行到聚合它们的响应和管理问题—将节省大量时间，并为您提供与您的版本一致的结果。

### 关注你的投资回报率

高价商业工具未被充分利用是很常见的；您有重复测试的许可，并且您有频繁的代码丢失，但是过程或集成意味着测试没有按照需要经常应用，或者结果需要时间来收集和处理。探索将商业工具轻松引入现有流程的方法。

### 天下没有免费的午餐

开源社区提供了很好的安全工具，但是请记住:团队使用它们和管理输出的时间是有成本的。从开发人员学习如何运行它们，到实际运行它们、提取结果或管理误报所需的时间，这些都不是免费的。一个每次发布都要花费两个小时的“免费”工具可能不值得。

### DevSecOps 是关于协作的

将安全检查编织到开发过程、CI/CD 管道、票务系统和 sprint 会议中有助于安全成为开发过程的一部分，但并不是每个开发人员都是——或者都想成为——安全专家。想一想发展和安全的目标如何在中间相遇。开发人员如何不用太多的努力或思考就能包含安全性？安全性如何帮助提供工具，并帮助甄别开发人员管道中的问题？该流程如何符合审计员的工具和正确的配置文件，而不仅仅是为了避免报告问题而将其降为零？无论使用何种工具，安全性如何增加价值，帮助开发人员更快地解决问题？
