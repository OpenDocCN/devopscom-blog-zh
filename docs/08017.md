# 无代码/低代码工具的安全风险

> 原文：<https://devops.com/security-risks-with-no-code-low-code-tools/>

***随着无代码和低代码工具的流行，安全问题也随之增加***

对新应用的需求正在快速增长。许多个人和企业单位不会容忍延误。因此，[公民开发者](https://devops.com/?s=citizen+developers)正在介入，其中一些人可能会受到公司的制裁，但本质上是作为影子 IT 运营。

这两个团队的共同点是，他们越来越多地受益于当今市场上无代码/低代码开发工具和平台的广泛可用性。这种解决方案抽象了许多编码任务，为开发人员提供了一个 visual studio 或调色板，通过拖放软件组件和较小的应用程序来创建应用程序。

无代码/低代码工具将繁琐的杂务从专业开发人员身上卸下，使他们能够更高效地工作。对于那些没有这方面专业知识的人来说，这些工具使熟悉业务的人能够创建他们自己的应用程序并自动化流程。

开发不仅更快，而且结构化程度更低。因此，这些技术掌握在没有接受过安全培训的人手中会带来风险。

公民开发者可以通过多种方式让企业面临网络风险。许多公司的一个常见问题是，公民开发人员(像他们的专业同行一样)不想重新发明轮子。如果他们可以找到一个应用程序的现有或开源组件，他们可能会在开发应用程序时免费使用它。例如，他们不需要为不同的平台硬编码一个前端移动用户界面，而只需要将现有的界面用于他们的新应用。类似地，如果他们希望他们的应用程序执行实时分析，他们可以使用开源事件流引擎。如果他们想创建一个交互式聊天机器人，他们可能会使用开源的认知应用程序。问题是，他们选择这些无代码/低代码应用程序时，很少审查他们正在使用的代码组件。

另一个常见问题是，公民开发人员可能会忽略组成复合应用程序的组件中的内置安全特性。例如，他们可能选择不对数据库进行密码保护，以简化其使用。或者他们可能决定放弃最佳实践，通过公共网络不加密地发送数据。

有许多这样的选择需要注意安全性。然而，大多数公民开发者甚至不知道要注意潜在的安全弱点。

## **在应用创建之前、期间和之后进行安全分层**

显然，在潜在的漏洞被攻击者利用之前，必须对公民开发者应用进行跟踪、审查和修复。

为此，安全性需要在应用程序的整个生命周期中集成。仅仅在开发应用程序后测试漏洞并让它永远不受检查地运行是不够的。此外，安全性必须是实时的，而不是被动的。它必须在应用程序运行和修改时不断检查应用程序的元素。

使用无代码/低代码技术拼凑组件会引入额外的问题。虽然单个组件可能是可信的，但是几个组件一起使用可能会暴露数据或提供可被利用的弱点。此外，如果应用程序中的某个软件元素出现新的漏洞，那么今天安全的复合应用程序可能不再安全。

许多组织发现，他们需要将开发运维与安全性结合起来。因此，业界越来越关注 DevSecOps。DevSecOps 越来越有必要发现由松散耦合的组件组成的现代应用程序中的弱点和漏洞。

应用程序的不断刷新、更新和替换决定了安全性必须是应用程序整个生命周期中不可或缺的一部分。这也是将安全性引入应用程序生命周期的另一个原因，就像 DevOps 为开发和部署所做的那样。

即使使用低代码/无代码方法，构建的应用程序也很复杂，这意味着安全性需要更加复杂，才能理解当今松耦合应用程序中隐藏的依赖关系。任何安全解决方案都不能是被动的。简单地扫描已知的漏洞是不够的。一个解决方案必须理解 citizen developer 应用程序的组件如何随着时间的推移而变化，以及如何相互交互。

底线是网络犯罪防护必须植入代码级别，并且必须制定安全的编码实践，尤其是当公民开发者正在推动应用程序开发时。
