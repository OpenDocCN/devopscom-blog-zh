# 纽奥普斯？AIOps？NoOps？只是不要叫我吃晚饭迟到

> 原文:[https://devo PS . com/new ops-aiops-noops-just-don-t-call-me-late-for-dinner/](https://devops.com/newops-aiops-noops-just-dont-call-me-late-for-dinner/)

再一次，我们看到 NoOps 这个词的使用有所上升。我总觉得 NoOps 是一个误导，事实上，是一个空洞的短语。我不在乎你认为你有什么魔力；你永远不会消除对操作你的软件、应用和基础设施的需求。

我第一次听到这个术语是在 DevOps 成熟曲线的早期。当时，人们的想法似乎是 Dev 将吸收 Ops。在这和云之间，传统的运营部门真的没有什么可部署的。有了云和自动化，管理应用程序将变得容易。至少，这是不切实际的想法——但我从不相信。T3】

还出现了其他术语，如 NewOps 和 AIOps，尽管它们完全变成了另外的东西。但实际上，我认为系统可靠性工程师(SRE)角色的崛起将 NoOps 运动推入了坟墓(它可能属于那里)。SRE 成了 Ops 人群中闪亮的新身份，而 SRE 继续成为 DevOps.com 最受欢迎的话题之一。

NoOps 的最新版本似乎有点不同。从我所看到的，它不是真正的 NoOps 相反，这更像是一场“不关你的事”的运动。It 正在将负责运营的人从开发和使用应用的公司转移到负责运营的服务提供商(或 SaaS 提供商)身上。这更像是一场“我的 Ops”运动，而不是“你的 Ops”运动，但肯定还是有一个 Ops。

在我成长的街区，我们曾经有一句谚语，“我不在乎你叫我什么，只是不要叫我吃饭迟到。”最初，对我来说，NoOps 很像那个样子。我们改变了某个事物的名称，但它并没有消失或从根本上改变事物本身。

现在，我们仍然有 Ops，只是谁来做的问题。对我来说那不是 NoOps。这是“有人在为我做近视手术。”

我们在安全领域看到了类似的模式。外包安全的想法从来没有被称为“NoSec”，因为坦白地说，没有安全的想法是非常可怕的。相反，这仅仅意味着依靠云供应商或 MSSP 来为你处理安全问题；是外包你的安保。从中得到的一个重要教训是，你可以外包你的安全行动，但你不能逃避你的安全责任。也就是说，当你的客户信息被泄露时，说“有人在为我们做这件事！”并没有免除你的责任。

同样的事情也适用于这种新的“外包服务”在运营部门，外包商可能比你做得更好。他们可能会向您提供 SLA 和 SLO，这些 SLA 和 SLO 会为您提供合理的性能预期和补救措施(如果这些 SLA 和 SLO 未得到满足)。但这仍然是你公司的名字和名誉。

那么我们看到的是真正的 NoOps 吗？不，这是外包业务。但就像我们在老街区说的，我不管你叫它什么，只要别叫它吃饭晚了就行。
