# Cribl 自动化了可观测性工作

> 原文:[https://devops.com/cribl-automates-observability-efforts/](https://devops.com/cribl-automates-observability-efforts/)

[Cribl](https://cribl.io/) 今天宣布，除了全面提供名为 LogStream Cloud 的软件即服务(SaaS)之外，它还更新了 Cribl LogStream observability tool。

Cribl 市场战略和竞争情报高级总监 Nick Heudecker 表示，Cribl LogStream 提供了一种方法，可以将从多个平台和应用程序收集的日志数据以任何可观察性或监控工具都可以访问的格式进行聚合。

如今，可观察性平台面临的挑战是，他们将数据复制和粘贴到平台中的任务留给了 it 组织。LogStream 通过日志路由器自动执行该过程，该路由器从现有安装的代理软件中收集和优化数据流，然后将数据整形并发送到分析平台。

![](../Images/f31e03daf6a7f908df95290544e30979.png)

版本 3.0 增加了 LogStream 包，这是一个用于加速日志路由器部署的框架。或者，DevOps 团队可以使用 LogStream 云来构建和共享 LogStream 配置模型。

Heudecker 表示，目标是通过减少设置和管理数据管道所需的时间和精力来降低实现可观测性的成本。Heudecker 指出，DevOps 团队无需定制即可从数百个来源收集数据并将其发送到任意数量的目的地。

虽然可观察性是 DevOps 的核心原则，但实现它却是一个重大挑战。除了部署和管理代理软件以管理 IT 环境之外，还需要大量的时间和数据工程工作来建立用于将数据传输到平台进行分析的管道。Cribl 正在利用一个软件层将数据抽象成一种可以被多种可观测性和监控平台使用的格式，来实现这些数据工程任务的自动化。

现在还不清楚 It 组织在多大程度上认为[可观察性](https://devops.com/?s=observability)是监控的进化，或者仅仅是监控的下一个逻辑进化。无论如何，IT 团队需要比以往使用遗留监控工具所能获得的更多的上下文。企业中采用的每个平台通常都有自己的监控工具，由负责管理它的 IT 团队使用。然而，无论何时出现问题，IT 团队都会花费数小时来关联来自多个工具的数据，以确定问题的根本原因。

可观察性平台承诺通过对从多个平台收集的数据进行分析来减少时间和精力。现在的挑战是找到一种有效收集所有数据的方法。

It 组织可能需要一段时间才能开始放弃他们现有的所有监控工具，转而支持可观察性平台。然而，随着 IT 环境变得越来越复杂(部分原因是基于微服务的云原生计算平台的兴起)，对这些平台的需求变得更加迫切。问题是使 DevOps 团队能够实现这一目标，而不必等待超负荷工作的数据工程师找到时间来建立管道。
