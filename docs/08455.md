# 性能测试是一项团队运动

> 原文:[https://devops.com/performance-testing-team-sport/](https://devops.com/performance-testing-team-sport/)

性能测试是任何开发人员工作过程中的重要组成部分。你的网站或应用程序的性能可以由你组织中的很多人来评判，包括技术人员和非技术人员。

当涉及到一个功能网站/应用程序的开发时，开发人员、操作人员和性能工程师站在第一线，但是总会有一些远离工程的人提出意见。

“我们能让它更快吗？”

*“用户界面会不会让这个有点太慢了？”*

c 级高管、董事会成员和营销人员经常会成为业余的 web 性能测试专家和评论家，但这没关系！

Web 性能被广泛认为是用户体验的一个关键因素，例如，当涉及到电子商务买家的旅程时。你的应用程序最好是快速的，否则你将会为自己的转换和金钱付出代价。

我们不应该在开发和测试完成后进行这些对话，我们应该明确地从一开始就让所有的利益相关者参与到这个过程中来。听起来熟悉吗？(咳咳，DevOps，咳咳)

## 集合队伍

这里有一些简单的方法可以让每个人都参与到你的性能测试周期中来:

**测试初期**

在收集数据的业务中，获得基线是关键。像测试“最终”产品一样积极地测试你的最小可行产品(MVP)。如果 MVP 不能在早期顶住一些压力，它可能会在前进中遇到很大的麻烦。

**经常测试—并实现自动化**

这里的想法是创建一个性能趋势来显示你是否在朝着正确的方向前进。数据点越多越好。

生成许多数据点的最佳方式是在每次代码提交后以编程方式运行测试，要么使用持续集成软件，要么在源代码控制系统中使用提交后挂钩或类似的东西，通过 API 调用来启动负载测试。

这样，您就可以用正确的信息武装自己，知道新代码何时会影响应用程序的性能，并且知道何时应该立即介入并调查变化。

**给予人们访问权限**

我们生活在软件即服务(SaaS)透明化的时代，所以让你的同事访问你的测试仪表板数据总是一个好主意。虽然组织中的一些人可能不完全理解技术方面，但他们不难理解增加加载时间可能是一种不好的趋势。

有各种各样的仪表板 SaaS 公司可以帮你做到这一点——假设你不想每天把数字敲进电子表格或表格。

在 Load Impact，我们收到了许多工程师的反馈，他们正在寻找一种简单的方法来与团队成员共享测试信息，由此，我们创建了“团队成员”功能。

测试人员简单地邀请团队成员到他们的组织，新的被邀请者可以看到过去的结果，甚至创建要测试的场景。这样，每个人都在循环中，每个人都可以在他们的专业知识范围内参与进来。

**玩得开心并分享**

持续测试既是一种方法，也是一种思维模式。它完全符合 DevOps 的理想，其中很大一部分是与您的团队成员的包容、协作文化。

所以请记住，与你的同事友好相处——无论是技术型还是非技术型的——你将为每个人的成长创造一个富有成效和创造性的环境。
