Mike Vizard: 你认为在过去的几个月里，随着向数字化转型的整体转变，整个测试的重点有所增加，人们正试图开发比以往更多的应用程序吗？我认为这年头第一次就把事情做对是很重要的。我认为在最终用户身上做实验的容忍度要低得多，尤其是在数字商业环境中，但是你看到的测试是否正在发生某种根本性的转变？

Guljeet Nagpaul: 是的，Mike，你提到的两件事都是对的。在过去的几年里，数字转型显然一直在加速，因为我们已经看到远程工作改变了我们周围的世界。许多流程甚至在医疗保健和金融领域也受到繁文缛节的困扰，所有这些流程几乎都在加速实现自动化，组织不得不加快实现转型目标。就像你说的那样，这自然会导致必须更快地将应用程序交付给客户，以便更快地运营他们的业务。为了更快地发布他们的软件，自动化测试的需求变得更加重要。

Mike Vizard: 自动化在哪里发生？当我们将事情向左转移时，我们是否为开发人员自动化测试，或者自动化更多地在 CI/CD DevOps 平台中，我们试图在那里自动化它，因为那里可能是应用程序本身聚集的地方；否则，每个开发人员都在测试某个东西的一小部分，但可能不是全部，然后呢？

Guljeet Nagpaul: 完全正确。我们生活在一个什么都有应用的世界。从消费者的角度来看确实如此，但有趣的是，现在在我们的企业应用程序方面更是如此。如果你还记得，过去我们有几个 ERP，然后我们有一些自主开发的系统，这就是组织的 IT 前景。但是现在，随着云应用程序、打包应用程序和垂直应用程序的出现，企业可以使用各种应用程序来实现更高的自动化。这意味着高度集成的企业应用生态系统。这意味着单元测试或者仅仅是自动化方面的左移是不够的。

你真的需要端到端验证，每个应用程序至少有五个上游和五个下游集成，这意味着真正的自动化，真正的客户体验在你的端到端工作流方面必须全面自动化。您谈到的 DevOps CI/CD 管道在能够验证端到端工作流方面变得更加重要。这是我们真正看到自动化需求的地方，而不是我们从左移的角度所认为的。这是绝对必要的，因为你想建立一个更强大的产品。你不想被动地构建一个强大的产品，你想把质量融入到产品中，但在下游，你想确保你的端到端工作流程按照集成的端到端系统的设计工作。

Mike Vizard: 我们离实现这个目标还有多远，因为我认为在这个过程中的某个地方，每个人都可能认为测试是别人的工作，但是就自动化而言，测试的成熟度如何？

*Guljeet Nagpaul:* 是的，就像在我们的 IT 中有一些不同的功能，我们有点看到一个完整的圆圈回来了，对吗？许多测试开始被认为是商品化的，我们看到许多测试从成本函数的角度有所改变。但是，组织很快意识到，没有持续的测试，就没有持续的交付，这真的是它的症结所在，如果你不能有一个可靠的，可靠的，持续的测试自动化来弥合你的开发和运营之间的差距，就没有开发运维，因为这是你的开发和运营之间的关键桥梁，对不对？有了这个概念，测试成熟度水平就变得困难了。

老实说，许多自动化方法，我在惠普工作了十多年，许多传统的自动化工具和方法都更关注孤岛，因为我们的应用生态系统当年还没有集成到一起，对吧？它曾经有过集成，但没有达到您今天看到的水平，因此现在需要一种可以跨越这些技术堆栈的自动化方法，这是一种更新的需求，现在也更为关键。就现有方法而言，测试成熟度一直很艰难，但我们现在开始看到客户真正需要跨技术堆栈的方法，这就是 ACCELQ 的优势所在。

Mike Vizard: 你是否认为微服务的兴起迫使自动化测试问题出现，因为我们已经达到了无法真正手动测试所有这些东西的复杂程度？

Guljeet Nagpaul: 我会说是的。有趣的是，Mike，就在过去的五年里，我们已经将 API 测试视为关键的概念验证的一部分，当客户来尝试 ACCELQ 这样的平台时，就要看它是否适合他们的需求。仅仅在过去的五年里，我们已经看到至少有三倍的人对确保 API 可以脱离 UI 进行验证的兴趣。但是，就像我说的，这实际上更多的是关于如何在技术堆栈之间进行验证。举个例子，我们的一个医疗保健客户，世界五大公司之一，他们有一个使用案例，一个病人走进一家药店放入他们的处方，但现在有了 COVID 和一切，自动化程度有所提高。

现在，他们可以通过连接到医疗服务提供商的移动应用程序直接输入处方。一旦处方被键入，至少有 20 个下游流程被启动，包括主机系统、微服务、可能位于云上的数据库，但随后又回到仍生活在老派桌面世界中的药房系统。真正的需求，我们所看到的是跨技术方面的，因为这就是用户体验。

Mike Vizard: 测试可以变得多聪明？这些天我们听到了很多关于人工智能的事情。整个事情会变得自动化吗，或者测试人员的角色是什么？

Guljeet Nagpaul: 嗯，这是个很好的问题。不幸的是，如今人工智能被说得很松散，尤其是在谈论人工智能如何成为平台的时候。我们一直将它视为我们工具箱中的一个工具。自从我们的平台成立以来，我们一直在考虑如何利用人工智能来提高效率，而不是说人工智能就是平台。我们总是更现实地看待它。如果你暂时忘记人工智能，这就是我们人类拥抱科技的方式。在各个方面，技术已经使我们能够增加我们的吞吐量，不管我们选择哪个过程领域，这就是我们如何看待人工智能。如果它被准确地利用，它将增加我们测试工程师的吞吐量。这将提高我们自动化工程师的生产能力。在测试自动化生命周期的各种过程领域，这就是我们如何利用人工智能的。

Mike Vizard: 你是否认为我们正在接近这样一个点，即正在构建的应用程序的数量超过了现有的测试人员，并且工作的速度正在达到成为瓶颈的点？

*Guljeet Nagpaul:* 是的，我与我们的 Forrester 分析师就同样的问题进行了讨论，其中一个有趣的讨论点是，作为 DevOps 领域的测试人员，我们都必须后退一步，了解在以传统方式进行测试之前需要测试什么。是的，就能够为我们的业务提供服务而言，如此多的应用程序如此快速地聚集在一起，这可能会令人不知所措，但为我们的业务和这些组织提供的许多应用程序都是云应用程序，或者打包应用程序、垂直应用程序。他们所需要的只是配置和与其他应用程序的集成，然后他们就可以按照企业期望的结果开始工作了。它几乎需要某种程度的重构，一种新的思维方式，当你在这个新的事实世界中思考，你应该如何测试，你应该测试什么来获得最大的回报，所以很容易被淹没。但是，有了自动化和需要测试的正确方法，您可以更加有效地工作。

Mike Vizard: 现在谁来写测试？在我看来，在创建测试上花费了大量的时间和精力，并且，当我们自动化这个过程时，我们能达到机器可以创建测试的程度吗？我的意思是，与运行测试相比，我需要在多大程度上创建测试？

Guljeet Nagpaul: 是的，这是一个有趣的观点。不幸的是，我们看待它的方式是测试和自动化特别发展的方式，我谈到的传统方法，它们存在于光谱的两端，Mike，光谱的一端非常面向记录和回放。但是，在过去的 20 年里，我们已经看到自动化并不新鲜；它已经存在了至少几十年。回到我们发行《风行者》的时候，我不知道这是否让我想起了什么，我是最初的风行者团队的一员，我们一度拥有大约 70%的市场份额，录制和回放是自动化的事情。但是，在过去的十年中，有太多的工具出现在录制和回放的同一条线上，但我们已经看到，它们的可扩展性不足以应对不断变化的技术堆栈。它们只是不够真实，无法跨技术堆栈工作，以及在设计挑战等方面的许多其他事情。

我们生活的另一端是开源，你似乎指出了测试是如何作为一种成本功能被推出的，开源在测试自动化领域掀起了一股浪潮，但这变成了一种专业功能。对你的问题来说，这一点很重要，因为自动化已经成为一种专业功能，因为我们已经看到了大约 60%到 65%的市场份额——Gartner 对接管多种测试和自动化方法的开源软件进行了调查。但是，当你看到这一点时，基本上它变成了一个专门的功能，我的测试团队是一个测试人员的混合体，可能有一些自动化工程师，但是有很多业务主题的专业知识。如果你真的想要一个强大的测试方法，你需要让主题专家成为理解业务规则的人。

这需要作为自动化的一部分进行验证，这是两个极端必须满足的地方，非技术、非专业的测试人员也应该能够成为自动化的一部分。只有我们的平台变得更智能，这才有可能实现，对吗？也许依赖于该套件来适应，也许在某些情况下桥接机器，对你来说，能够编写更智能的测试用例，基本上使你的自动化更精简和更有效，以便它可以在创作和设计更改方面以更少的资源做更多的事情，并且随着这么多版本的出现而改变，所以绝对是这样。我的意思是，测试必须脱离专门的功能角色。

迈克·维扎德:好吧。当然，这些天我们听到很多关于安全性的关注。你认为安全性最终会成为质量保证测试过程的一部分吗，因为我不再清楚这是两个不同的功能。他们可能会以某种方式结合在一起。

我当然希望如此。最强大的软件和组织在过去的一年中，我们看到了我们从未想象过的停机，对吗？AWS 退出了几次，脸书，Meta，退出了几次，你说得出的。安全必须成为不可或缺的一部分，我们越来越多地看到这一点。作为功能回归测试的一部分，客户希望进行那些安全性回归测试。到目前为止，安全性存在于两个世界中。一个是静态代码分析，“让我们做静态代码分析”，这很好，这有点像向左移动到你的单元测试。是的，当你写代码的时候，你必须找到你的漏洞，但是，一旦代码开始运行，什么样的漏洞会出现就不一样了。

静态代码分析是安全性的所在，正如你所说，这就是为什么它在不同的世界里有点不同。然后，你有动态安全防火墙，这是一种更美化的防火墙。但是，在这中间，利用安全验证的功能测试是目前行业中的一个空白，我们当然希望填补这个空白。我们已经帮助许多财富 100 强客户，包括某些金融服务客户，利用功能自动化方法填补了这一空白，因为他们已经有了一些围绕这两个方面的解决方案，但功能是它需要进入的地方，对吗？如你所说，它必须成为测试和功能自动化领域的主流。

迈克·维扎德:好吧。除了在你的网站上闲逛之外，我想，关于如何到达这个持续测试的天堂，你对人们最好的建议是什么，因为，归根结底，时间就是金钱，所以每当一个应用程序有一个脱节的体验时，就有人在某个地方损失金钱，对吗？

古尔吉特·纳格保罗:共。自动化第一。自动化第一是关键。许多传统的思维模式仍然停留在，“让我们写我的手动测试，让我们把它们转换成自动化。”心态必须改变。组织方法必须首先转变为自动化。对于自动化优先，你只能先做自动化，如果它不是专业功能的话。不要陷入技术债务中。你不想让一百万行代码测试一百万行代码，对吗？找到一个更强大的解决方案，首先实现自动化，这将使你走上正确的道路。
