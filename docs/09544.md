# 网上研讨会:为什么持续交付软件对你的商业成功至关重要

> 原文:[https://devo PS . com/webinar-持续交付软件对您的业务成功至关重要/](https://devops.com/webinar-continuous-delivery-of-software-is-paramount-to-your-business-success/)

在应用经济中，你必须交付软件，就好像你的业务依赖于它一样…因为它确实如此！这是一个大胆的说法，但却是事实。想想一个人今天可能进行的几乎每一次互动——无论是工作、商务还是娱乐。大多数都有一个数字维度，依赖于数字技术和平台。

为了生存和发展，每个公司都需要成为技术公司，每个企业都需要成为数字企业。成功的转型需要持续交付，这是一种新的业务需求，使您能够快速开发和交付应用程序，推动卓越的用户体验并吸引您的客户和员工。

但是传统的“软件工厂”或者将想法转化为客户体验的过程被交付管道中的许多瓶颈所抑制。这意味着更快、更频繁地交付创新、高质量的应用程序可能是一个混乱而复杂的过程，特别是如果您的应用程序交付系统和过程被设计为一年只推出一两个版本。

加入主持人艾伦·希梅尔，DevOps.com；专家 Georg von Sperling(CA Technologies 的开发人员和技术宣传员)和 Julie Craig(Enterprise Management Associates(EMA)的研究总监)在 2015 年 7 月 21 日讨论了如何利用旨在将您的软件交付生命周期转变为数字创新的连续交付渠道的解决方案来克服当今的应用程序经济挑战。

# 问&答

**1。根据你的经验告诉我:连续交付与传统开发。持续交付模式对每个企业都是完美的？**

作为一个目标，是的，绝对是。因为持续交付实践增强了已经在做的事情，并且简单地促进了良好行为而不是浪费的产生。无论你从事什么行业，消除瓶颈以提高质量、频率和速度都是有意义的。也就是说，如果你将持续交付定义为持续交付到生产，那么持续交付不一定对每个企业都是完美的。但是什么样的开发组织会避免在用户接受度测试中获得早期和放大的反馈呢(UAT)？为了实现这一点，您仍然需要连续交付中的每一个概念。

ANBU:如果你认为实现连续交付是实现模块化和去耦合的一种手段，以达到更高的员工生产率，每个企业都会被激励去做。作为一个概念，它对每个企业都是完美的，但是这个过程的目标和它们的重要性顺序会因行业而异。只要做好这些调整，你就会受益匪浅。

**2。我们面临的一个挑战是持续交付对我们的环境战略和用于维护它们的机制的影响。例如，依赖于其他应用程序/组件的应用程序将需要进行测试的环境基础结构中的那些其他应用程序/组件。您如何看待 Compnay 环境战略的变化以支持连续交付？什么时候有必要改变应用程序的架构来减少或消除这些依赖性？或者有没有不需要大型 QA 环境就能准确测试变更的测试方法？**

全球之声:阻力最小的道路似乎是从飞机制造业学来的。当有一个新的机翼设计需要进行质量验证时，相关的系统和数据集是如此复杂，如果他们在不同的天气条件(环境)下将不同乘客或行李负载(数据)的飞机抛向空中，并连接到完整的飞机机身(相关性)，他们就不可能发布这些设计。很可能，这将导致我们在将数字资产从我们的 IDE 推到第一个环境时所经历的同样的负面影响——通常是灾难性的。用实际行为虚拟化依赖服务——就像在风洞中一样——是一个好方法。这也允许发现可消费的、可共享的格式中的依赖性，如果您发现需要重新访问架构，这实际上有助于重新设计的过程。

ANBU:我建议根据你的情况采用基于模型的方法。在你陈述的例子中，如果你将“基础架构作为代码”建模，并试图在那个层次上实现模块化，那么开发人员和测试人员在他们的计划/设计/开发过程中调用它，以定义他们的“可用性和用途”，这个问题就可以解决。为了对其建模，需要对所有功能组件(包括下文)进行评估。有一些工具可以帮助你实现这样的建模。

**3。除了更快的&质量发布，你如何衡量 DevOps 的成功？有没有可以定期收集的指标？**

GvS:在持续交付的环境中，有许多可以并且应该被捕获的度量标准。在 DevOps 的上下文中还有更多。我们可能会花一整节课来讨论这个问题。然而，让我把重点放在持续交付度量上。除了关注面向业务的结果之外，就像在 DevOps 中一样，还可以捕获一些运营指标。例如:

–生产准备时间。测量从工作开始到团队定义的“完成”的周期时间。

–缺陷解决时间。真正成功的团队能够在发现缺陷时尽快修复缺陷。缺陷发现越左移，解决的越快。

–生产缺陷泄漏。度量两者:新发现的缺陷和以前发现的缺陷，并且仍然可以生产。

–缺陷。发布漏洞百出的应用程序已经够糟糕了。持续这样做更可怕。在循环的每个阶段进行测量，如果 CD 运行良好，请注意左移。

–构建时间中断。构建失败的时间越长，对质量的承诺就不存在了。

–部署期间的生产停机时间。如果你实现了零停机，你可以在这里停止测量。

–回归测试持续时间。具有手工实践的团队将在几个月或几周内进行度量。CD 团队可能会以分钟或小时为单位进行测量。还包括回归测试的准备时间。

–性能测试结果。如果您在性能测试中发现大量的功能性错误，那么很可能有一个瓶颈没有在周期的早期得到解决。

触摸。在每个阶段进行部署所需的票证、电子邮件或其他通信的数量。如果不是“零接触”，就有改进的潜力。

这不是一个完整的列表，但肯定是一个开始

ANBU: GVS 的回应是完整的。

**4。如何将遗留代码/系统引入 CD 模型？还是 CD 只针对新代码/项目/系统？**

GvS:CD 提倡的实践并不特定于项目的期限或技术的时代。可以说它们都是好的实践，非常有意义。现在，如果您是一个正在采用 CD 实践的企业，那么推荐的做法当然是从对业务影响最大的项目开始。一个好的热图分析确实有很长的路要走。如果事实证明这是您的遗留代码/系统，那就这样吧。并行化工作流，提高质量，自动化和协调交付管道。

ANBU:遗留代码/系统通常是整体的/紧密耦合的。这就是问题所在。不幸的是，它们也是企业应用程序的一部分，所以我们不能远离它们。我们已经看到这样的例子，这些系统作为一个“组件/工作流”插入到更大的 CD 流中，形成了一个混合模型。是的，它并不完美，但会帮助你实现 CD。此外，这些遗留系统的维护/再工程是修改代码以适应您所实施的更大的 CD 策略的机会。如果它们是真正的遗产，就不会有大的“修补”工作，所以它们保持原样。

**5。你如何调和产品管理和 CD，即，一个故事的定义来封装一个可部署的或可交付的原子元素？通常情况下，故事是一个更大的史诗的一部分，需要作为一个整体来交付，而不是单独交付。所以问题是部署原子单元/故事会产生交付部分功能的恐惧。企业如何处理这种情况并克服这种情况**

如果你原谅我，但这似乎是对 CD 本身的误解。部署和交付是两件不同的事情，如果不是三件的话。我相信您可能已经抓住了生产中原子变化的“实时”部署。给定建模管道本身的正确策略，所有 CD 元素仍然有效，原子元素可能与许多其他原子元素一起“活”起来。但是，与过去不同的是，随着质量的显著提高，对部署失败的担心降到了最低，并且利用相同的、已经过测试的流程来设置原子单元“活动”,这一点已经在较低的环境中得到验证。这是一次旅行。

ANBU:CD 的好处之一是只需按一下按钮，就可以实现(所有或部分原子单位)的生产部署就绪。它不会强迫你(但你可以选择)将每一个原子单元部署到产品中。如果你所说的“整体上更大的史诗或者没有”是重要的，那么你对“选择”原子单元的定义应该与之相关。配置管理和发布管理应该有适当的“关口”,以确保它们能够正确推出。

**6。我关于度量的问题是因为在遗留组织中，需要花费更多的时间&精力来达到 CD，所以通过度量的度量是关键的。**

GvS:当然，度量和测量是携手并进的。以 CALMS(文化、自动化、精益、度量和共享)为指导，通过度量和共享来影响组织文化，这两者都是从自动化中衍生出来的，这是一种涅槃。除了 IT 组织的意愿之外，还必须有管理层对战略的认同。是的，基层是好的，但是如果公司支持战略，很多事情都可以完成。在这一过程中定义积极的业务成果指标有助于巩固这一地位。

ANBU:同意 GVS 的观点。

**7。连续交货的结果是什么质量？对于生产中的缺陷泄漏的业务影响，权衡是什么？**

GvS:如果实现了 CD 租户，质量度量应该显示出显著的改进。已经有一个关于度量的问题了。这不是关于持续部署糟糕的代码，这只会让事情变得更糟，而是关于持续交付，质量当然是一个决定性因素。

ANBU:你需要在裁谈会前后确定“质量”基线。绝对没有办法，这些措施应该消极的趋势，如果你正确地执行 CD。任何模型中的缺陷泄漏都会有同样的影响，只是在 CD 中——解决(MTTR)这些问题的平均时间应该更快。

**8。对于采用 CD 模型的企业来说，难道不是所有的规程/功能都必须改变以适应原子交付概念吗？**

GvS:一旦涅磐被实现，很可能所有的学科和功能都会发生变化。这是一个必须开始的旅程。这是企业的当务之急。但是，在 DevOps 的背景下也讨论了很多，CD 是一个转变的旅程-不是你一天就能完成的。

ANBU:是的。这是一个生命周期过程，因此对每个人都有影响。但是你可以从一个单位/组织开始，展开来进化它。如果你有奢侈，开始一个新的项目/单位，以 CD 为基础。

**9。你如何看待 ITIL 和 DevOps 的互动，以最大限度地发挥持续交付的优势？**

全球之声:两者之间有很大的协同作用。实际上，在变更管理流程中进行“标准变更”、用不会过时的实际数据丰富 CMDB、拥有最新的服务模型并专注于问题管理会容易多少呢？

ANBU:一个定义良好的 devops 策略可以帮助你更有效地实施 ITIL。配置管理和发布管理是 ITIL 可以为 CD 策略提供最大利益的两个特定领域。

10。Jez Humble 和 David Farley 建议不要使用功能分支，因为这将鼓励后台渠道开发，并且只使用主分支来做 CD，但该领域的其他公司表示功能分支是理想的，因为 git-flow 鼓励这样做，在您的分支策略中有没有更好的策略来做 CD。是否需要更好地定义“持续交付”的含义？我担心有人会说 CD 意味着代码未经测试(这是我理解的一个反定义),经过充分测试的代码是 CD 的一部分。

GvS:我相信 Jez 的观点，我也同意，这就是为什么必须有一个从主干触发的部署管道，以确保所有必要的验证，包括功能测试。部署管道是采用 CD 的组织使用的一种模式。关键是验证应该根据构建主线来执行。我还提倡许多有助于验证工作的好方法，比如自动化功能测试的生成，以及让正确的数据集可用于相关的验证。

ANBU:无可奉告。

**11。持续交付是否意味着持续部署到生产？对于马来说，部署的频率不是太频繁，但是他们仍然需要有持续的交付来获得测试自动化、敏捷方法等的好处。**

GvS:我认为不是。持续交付并不等同于生产的持续部署。如果我可以这么大胆的话，我想说连续交付是一种实践，它允许在任何时候以高质量和低风险的方式部署到生产中，并且没有停机时间和接触。一位业务领导最近对我说，过去，“生产部署是一项 IT 决策，而现在是一项按下按钮的业务决策”。但是，CD 无疑在这种范式转变中发挥了作用，因为整个交付过程，包括测试，都是可见的和可预测的。

ANBU:持续生产部署不是 CD 的最终目标。它让您可以将多个工作流作为原子单元并行管理，通过高度自动化来提高质量，并提供学习/持续改进流程的能力。通过这种方式，您可以比平时更快地准备好部署您的更改。你应该致力于在整个生命周期中实现无摩擦的自动化，而不仅仅是在 QA 活动中。
