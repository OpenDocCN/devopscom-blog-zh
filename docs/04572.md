# Elasticsearch 勒索软件攻击凸显对更高安全性的需求

> 原文:[https://devo PS . com/elastic search-勒索软件-攻击-高亮-需要-更好-安全/](https://devops.com/elasticsearch-ransomware-attacks-highlight-need-better-security/)

最近，有报道称大量 Elasticsearch 服务器成为潜在勒索软件攻击的受害者。勒索软件是一种公司不希望在其系统或网络上出现的恶意软件。它劫持系统，最常见的方式是加密或窃取数据，并使所有者面临勒索企图。根据 Herjavec Group 的一份报告，到 2016 年底，勒索软件造成的损失预计将达到 10 亿美元。

过去几周观察到的新一轮勒索攻击的目标是不安全的 MongoDB 数据库。安全研究人员 Victor Gevers 和 Niall Merrigan 将这些攻击称为“洗劫”，Merrigan 估计仅在前两周就有超过 40，000 个数据库受到影响。

现在研究表明，Elasticsearch 服务器被配置为不安全的，因此可以通过公共互联网访问，正在遭受类似的赎金攻击。Victor Gevers 在推特上说，在头三天内，2515 个 Elasticsearch 服务器被清除和赎回，34298 个有漏洞的 Elasticsearch 实例仍然开放。在接下来的几天里，受影响的服务器数量已经上升到 5000 多台。Shodan 的创始人约翰·马瑟利(John Matherly)在推特上表示，绝大多数易受攻击的 Elasticsearch 服务器都开放在亚马逊网络服务(AWS)上。

如果 Elasticsearch 服务器遭到黑客攻击，用户会发现数据索引不见了，并显示一条消息:“如果您想恢复数据库，请向此钱包发送 0.2 BTC:1 das gy 4k t1 a4 LCT pmh 5 vm5 pqx 32 ezmot 4 r！发送比特币后，将您的服务器 IP 发送到此电子邮件……”

联邦调查局强调，受害者应该拒绝支付比特币赎金，因此用户可能会也可能不会拿回他们的数据，这取决于他们在遭受攻击时的安全流程。目前，还不清楚谁是这些袭击的幕后黑手。

具有讽刺意味的是，使这些攻击成为可能的并不是 Elasticsearch 本身不安全，因为它不是。勒索攻击是可能的，因为这些实例的配置方式使它们容易受到攻击。就像开着前门一样。

ZDNet [的科技记者 Steven Vaughan-Nichols 给出了一个极好的总结](https://www.zdnet.com/article/elasticsearch-ransomware-attacks-now-number-in-the-thousands/)，他解释说，如果由没有任何安全技能的业余爱好者使用，Elasticsearch 很容易被破解。在 AWS 云上部署实例的人以为 AWS 在保护他们，但事实并非如此。虽然 AWS 告诉用户如何保护他们的 AWS Elasticsearch 实例，但用户仍然需要自己完成这项工作。

他写道:“最糟糕的事情是什么？就像 MongoDB 攻击一样，如果它的程序员用基本的、众所周知的安全措施保护它的实例，这一切都不会发生。”

Elasticsearch 经常用于日志管理，通常作为 Elastic Stack 或 ELK 的一部分，代表其主要的开源成分 Elasticsearch、Lucene 和 Kibana。由于它是免费的开源软件，ELK 是许多人的首选。这是一款强大的软件。这些开源项目非常活跃，每个月都有成千上万的代码贡献，以及大约 250 万行代码的不断增长的组合代码库。

然而，用户需要专业知识来高效、安全地部署和运行它，这意味着组织中需要有技能和时间来维护 ELK 集群的人。如果这些人走了，公司需要有后备。如果他们不愿意或不能投资这些资源，他们很可能会陷入困境，就像最近的赎金袭击事件所表明的那样。

ELK 是自由软件，但请记住自由软件基金会和 GNU 项目的创始人理查德·斯托尔曼对自由的看法:“‘自由软件’是自由的问题，而不是价格的问题。为了理解这个概念，你应该把“免费”理解为“言论自由”，而不是“免费啤酒”。"

无论一家公司是为日志管理服务付费还是运营 ELK，一种方法并不一定比另一种更好。对于一些公司来说，运行基于 ELK 的内部构建的日志管理解决方案，甚至是从头开始构建的解决方案，是非常有意义的。对于其他人来说，交付的解决方案可能是最好的。

无论如何，公司和团队需要仔细评估开源是否有商业意义，以及它们是否能够现实地适当支持部署而不会带来风险。如果没有正确权衡或完成这些因素，弹性搜索勒索软件攻击的数量将继续增长，并成为黑客有利可图的努力。
