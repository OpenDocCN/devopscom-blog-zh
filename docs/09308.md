# SRE Vs .德沃普斯:错误的问题？

> 原文:[https://devops.com/sre-vs-devops-the-wrong-question/](https://devops.com/sre-vs-devops-the-wrong-question/)

关于德沃普斯和 SRE 之间竞争的老问题建立了一个错误的二分法。DevOps 是一种方法，而 SRE 是一个运营团队。尽管这两者经常相互对立，但是开发人员也体现了实现修复的技能和能力。

传统上，应用程序的更改是被动的。为了在开发生命周期中获得更高的效率， [SRE](https://devops.com/?s=SRE) 的过程将问题解决剧本形式化。这在过去几年里发生了很大的变化，DevOps 获得了更多的关注和主流采用。

那么，DevOps 和 SRE 之间的现代关系是什么，如何才能最好地将它们结合在一起使用？先来了解一下 DevOps。

## 定义 DevOps

根据 [厨师的拥抱 DevOps 指南](https://d1l5pp53ux74mz.cloudfront.net/docs/default-source/ebook/embracedevops.pdf?sfvrsn=ae0a7c00_7) 的说法，DevOps 是关于转变公司的运营方式；这种转变的一部分是理解公司应该专注于管理人而不是产品。DevOps 在过去十年中不断发展，最初诞生于软件世界。云技术的采用、技术堆栈架构的变化、SaaS 平台和安全法规都对开发运维文化产生了影响

现代 DevOps 的一些基础是:

**一切如代码**:IT 架构的当前状态已经采用了“一切如代码”的方法。基础设施、配置、合规性和安全性都可以转化为代码。

**自动化一切**:将一切整理成文，使整个组织的流程和政策标准化。这些过程被简化为可重复的功能代码，这使得自动化开发和部署过程更加容易，同时大大减少了人为错误。

**验证**:除了用于维护基础设施和合规性的其他工具之外，还有许多用于持续集成、交付和部署的工具。没有验证，就没有办法知道一切是否都按需要运行，也很难确保在高度敏捷的环境中无缝交付。

最重要的是，采用 DevOps 方法的团队承担管理应用程序开发生命周期(开发、部署和维护)的端到端责任。他们将开发人员和运营人员之间的职责差距缩小到一个团队中。

## SRE 关注领域

成功的站点可靠性工程(SRE)团队关注性能指标的数量和转移管理和减轻风险的责任的过程。从根本上说，在创建高效应用程序方面，他们与开发人员有着相同的价值观。然而，SRE 团队专注于根据发生的事件引入效率，有时需要开发团队的干预来转移产品开发周期中留下的风险。

故障排除是 SRE 团队的主要职责之一，因为他们必须管理事故。因此，SRE 团队拥有丰富的产品设计知识，这对于产品可靠性、推广和缩短恢复时间至关重要。可扩展的自动化也是 SRE 团队面临的一大挑战。通常，任务是手动完成的，这增加了人为错误的可能性。一个解决方案是与开发团队一起实现策略即代码的方法。

应该注意的是，当谈到 SRE 和 DevOps 之间的责任和工作范围时，界限是模糊的——特别是随着 API 的激增和评估微服务时。所使用的工具类型是一个优势，因为 SRE 团队通过监控、检测和事故报告系统来衡量效率。

## 打破筒仓

这些基本价值对于寻求实现高效 CI/CD 渠道的企业至关重要。在反应周期中，SRE 团队介入并负责运营的某些部分，但 DevOps 模式有所不同，因为它集成了整个团队的部署、监控和补救责任。

采用 DevOps 实践的组织应该接受 SREs 的专业知识，并欢迎他们的实践加入开发团队。sre 可以利用自动化来解决出现的问题，还可以对软件架构进行更改，以避免将来出现此类问题。与反应式方法相比，开发团队被激励保持事物持续运行。

扩展 DevOps 实践需要分担责任。值得注意的是，开发人员必须在电话响起时接听，这可能会给负责大型企业级应用程序的工程团队带来混乱，这些应用程序的上市时间很快，更新周期也很频繁。

## 现代视角

效率是 SRE 团队和 DevOps 方法的核心价值观。当组织能够在开发运维周期内完全整合运营时，这些孤岛就可以被打破，并整合左移思维。传统的反应式补救周期不再有效。主动性对于当今闪电般的应用需求和持续更新至关重要。

当使用现代工具和策略即代码方法处理应用程序问题时，可以立即解决应用程序问题，使开发人员和 SREs 团队能够协同工作。DevOps 和 SRE 团队之间的现代关系应该是共存的关系。两者都是企业内部成功的可伸缩性所需要的。许多人曾经认为 SRE 团队和开发人员不能和谐共存，但是揭穿这个神话有助于组织优化他们的协作努力。
