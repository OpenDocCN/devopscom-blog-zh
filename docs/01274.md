# 拥抱 DevOps 文化，第 2 部分

> 原文:[https://devops.com/embracing-devops-culture-part-2/](https://devops.com/embracing-devops-culture-part-2/)

在本系列的第 1 部分中，我们讨论了越来越多的组织希望接受“DevOps 文化”的事实然而，与 ITIL 不同的是，在组织中实施 DevOps 并不是一蹴而就的。DevOps 更多的是一种哲学而不是程序，选择正确的领导团队和选择正确的工具一样重要。

在第 2 部分中，我们将探讨成功接受 DevOps 文化的组织的其他重要元素。

## 没有适当的协作，DevOps 计划将会失败

确定您的组织是否真正接受了 DevOps 文化的一个很好的测试是，当产品环境中发生重大事件时，您的 IT 部门的响应如何。例如，一家电子商务公司的网站速度非常慢，一家医院的电子病历(EMR)系统无响应，以及一个故障的航空预订系统。在这些混乱的情况下，每一分钟都很重要，所有开发和运营人员都需要在甲板上保持您的组织运转。

当事故发生时，组织的开发和运营团队合作解决问题并尽可能快速高效地恢复故障服务至关重要。

## 事件响应:重大事件团队

一家拥有全球业务的国际公司可能在美国和英国拥有支持 50 多个关键业务应用的网络运营中心(NOC ),在中国和美国拥有开发中心，在美国拥有服务台。当事故发生时，公司的服务台会收到大量警报，从各种 IT 监控工具和客户电话到在线用户开罚单和利益相关者的询问。当发生这种情况时，服务台唯一关心的是解决问题，而不是追踪世界另一端可能对问题负责的部门负责人。

DevOps 应该使组织能够在几分钟内组建其关键事件团队。运营团队应该确切地知道针对给定事件应该联系哪个(些)团队，以及这些团队中的谁被分配到应用程序支持。他们应该根据时区和星期几知道谁在打电话，并且有容易获得的正确电话号码。

接下来，团队应该审查事件经理提供的问题的技术细节。普遍使用的 DevOps 工具集使每个人都能使用同一种语言，快速确定根本原因，制定补救计划以获得紧急 cab 的批准，部署修复程序，进行测试并确认服务已恢复运行。

## IT 沟通:维系开发人员和运营人员的粘合剂

为了取得成功，组织需要能够在重大 IT 事故期间尽快做出响应，以限制对业务的负面影响。当开发人员和运营人员共同创建一个中央沟通和协作中心，以确保与关键利益相关方的适当沟通时，这一目标就能实现。

一个组织有效减轻 IT 问题影响的能力在很大程度上依赖于其访问和交流关键信息的能力，以及确保合适的人员能够分析这些信息并启动适当的操作以保持业务平稳运行的能力。配备信息技术通信解决方案的通信枢纽将有助于:

*   联系所有不同团队中合适的待命人员:基础架构、服务器、系统管理、中间件、网络、DBA、QA、支持团队、服务台和应用程序开发人员。
*   因为电子邮件不会唤醒人们，他们将能够利用 IT 通信解决方案通过多种渠道联系人们，直到他们做出回应(语音、短信、电子邮件、推送通知应用程序、寻呼等)。)，或者自动升级到待命列表中的下一个解决程序。
*   提供正确的信息，以便 IT 解决人员开始调查问题，找出根本原因，并在不浪费时间的情况下制定解决方案。
*   让人们使用相同的电信和协作工具进行协作，而不管他们在哪个时区，也不管他们在哪里。
*   如果问题不是由公司造成的，而是由外部软件引起的，请联系第三方供应商。
*   如果业务影响变得如此之大，以至于影响到公司的盈利能力或声誉，例如导致数据泄露的网络攻击，通知其他部门。可能需要通知首席执行官、法律部和市场部，以预测后果。
*   通知最终用户或客户限制帮助台的来电数量。

快速协作的好处，而不仅仅是团队之间的交接，甚至不需要演示。消除中断的途径、过时的交付方法和多余的平台将大大提高本组织在关键时刻的沟通能力。提高效率并消除浪费的时间将自动对平均知晓时间(MTTK)产生巨大影响，进而对平均解决时间(MTTR)产生巨大影响，从而最大限度地减少对业务的灾难性影响，并让公司高管晚上睡得着觉。

## DevOps 文化可以发生

拥抱 DevOps 文化不是一夜之间发生的巨大变化。根据现有文化以及 IT 组织的历史和成熟度，这是一个或多或少有些困难的旅程。更重要的是，这将取决于高层管理人员对引入新文化的承诺程度。领导者需要踏上这一旅程，牢记这不仅需要新的通用工具和新的通用流程，还需要让人们相信这种转型的价值。为了实现这一目标，管理层应该将 IT 沟通视为一个关键要素，这将是将开发和运营结合在一起的粘合剂。因为总会有持续改进的空间，通往 DevOps 的道路没有尽头。
