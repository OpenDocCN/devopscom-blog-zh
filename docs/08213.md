## 工程应用需要考虑的因素

围绕 DevOps 产品和服务的极端全球宣传似乎表明 DevOps 对每个人都有好处，并且同样适用于每个组织中的每个基于软件的产品或服务。以下是我对这一假设的体验:这不是真的！

另一方面，我听人们说 DevOps 不适用于它绝对适用的应用程序！在您的 DevOps 之旅更进一步之前，或者决定您的应用程序不符合 DevOps 的条件之前，您应该仔细考虑您的应用程序在多大程度上被设计为 DevOps 实施的模型。

## DevOps 测定的最佳实践

以下是您可以用来决定应用程序是否适合开发运维的重要因素:
应用程序将受益于更快的交付周期。
·负责该应用的领导乐于合作，并将成为变革的发起者。
与应用团队相关的团队成员(产品负责人、开发人员、质量保证人员、运营人员、基础设施人员、安全人员、项目管理人员)乐于合作和变革。
应用目前正在使用或计划使用面向服务的模块化架构或微服务。
至少有十个人与应用程序相关，包括产品负责人、开发人员、质量保证人员、运营人员、基础设施人员、安全人员和项目管理人员。拥有较小团队的较小项目可能不会显示出足够的影响力或证明实质性开发运维投资的合理性。
应用程序预计会在一年多的时间内发生变化。更少的时间不太可能产生开发运维所需的投资回报。
该应用程序表现出良好的业务影响和可见性，但不会给业务带来极大的风险。
·应用程序经历了业务变化的频繁需求。
实施 DevOps 工具链并不需要更换应用程序使用的所有工具。更换所有工具的成本会对开发运维转型的投资回报和业务案例产生负面影响。
构建、测试或部署应用程序版本的工作量很大，可以通过自动化大大减少。

应用程序没有必要完全符合所有因素，但是越符合，您的应用程序越有可能从使用 DevOps 中受益，因此，值得花费成本来实施设计良好的 DevOps 实践。

DevOps 可用于所有类型的软件应用程序和服务。然而，并不是所有的应用和服务都应该设计为使用 DevOps。决定是否为某个应用设计开发运维的因素与[业务因素](https://devops.com/survey-finds-biggest-devops-obstacles-are-organizational/)的关系更大，而不是不同类型应用之间的技术差异。应用程序必须以特定的方式进行设计，开发运维才能有效运行。
