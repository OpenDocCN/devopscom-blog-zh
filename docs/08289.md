# 如何发现 SRE 的收养障碍

> 原文:[https://devops.com/how-to-spot-sre-adoption-barriers/](https://devops.com/how-to-spot-sre-adoption-barriers/)

众所周知，[站点可靠性工程](https://devops.com/?s=SRE) (SRE)的采用正在增长。运营模式似乎是整个企业数字化转型的关键。根据[2021 年技能提升:企业开发技能报告](https://info.devopsinstitute.com/2021-upskilling-report-download?utm_campaign=Upskilling%202021&utm_source=DevOps.com&utm_content=2021-upskilling-report-download-devopsdotcom)，22%的受访者(比去年增加 7%)认为 SRE 是他们在 2021 年继续自动化所需的方法。

## 常见的采纳障碍

然而，对于一些公司来说，采用 SRE 原则和实践并不总是容易的。在即将举行的以 SRE 为主题的 SKILup Day 会议之前([注册 SKILup Day: SRE 在此](https://devopsinstitute.com/sre-2021/?utm_campaign=SKILup%20Day-SRE%20May%202021&utm_source=SRE-2021-devopsdotcom))，我邀请了几位 DevOps Institute 大使和 SRE 主题专家就 SRE 采用的障碍提供见解。以下是他们分享的内容:

### *[马克·霍恩贝克](https://www.linkedin.com/in/marchornbeek/?originalSubdomain=mx)，工程开发咨询公司首席执行官兼首席顾问，《工程开发》一书的作者*

“采用 SRE 的主要障碍是管理层愿景和团队围绕向 SRE 实践转变的目标的一致性。SRE 会影响整个组织的实践、角色和投资优先级。没有高管的支持，SRE 的转型不会走得太远。”

### *[斯蒂芬·沃尔特斯](https://www.linkedin.com/in/1stephenwalters/)，xMatters 公司的解决方案架构师*

“文化，尤其是文化变化，是采用 SRE 的最大障碍。关于人类的一切，那些让我们成为今天的生物的东西，都与进化有关，这是变化，而我们并不特别擅长于此。或者更确切地说，变化越大，我们对它的抵触情绪就越强。为了“转变”，你必须改变，这在 It 哲学中一直是一个很大的矛盾，但是这种转变正在发生，因为我们不擅长改变。早期的敏捷也是如此。希望转换到敏捷(小的、迭代的变更)的组织将在转换项目下进行，这通常是瀑布(大爆炸)。大变革，大文化阻力。关键是以更小、更敏捷的块来实现转换。它们更容易被接受和测试，如果它们不起作用或不可接受，也更容易回滚或适应。敏捷开发的所有好处都适用于敏捷转型。

第二个最大的障碍是“买入”，不幸的是，简单的答案是向最终用户展示“他们从中获得了什么”这通常会超越企业级的“这对我们有什么好处？”但是，随着时间的推移，随着任何文化转型，这是应该反复改变的事情之一，也是发展更全面和更具同理心的文化的重点。"

### *[希瓦加米·古干](https://www.linkedin.com/in/shivagami-gugan-3980181/)，CX 科技独角兽公司的首席技术官*

“任何组织，无论其目前处于何种成熟水平，都可以采用 SRE 并改善其业务成果。然而，应该正确理解 SRE 的关键基本原则及其适用性。围绕转型有很多宣传，第一个障碍是理解 SRE 在您的组织环境中意味着什么。我发现大多数时候，人们试图模仿高度成熟的公司，如谷歌或 Spotify 或其他一些云公司的做法，这通常会导致失败。SRE 是 DevOps 最纯粹的实现形式。SRE 是关于消除产品生命周期中的筒仓，以更安全、更快、更便宜和更好的方式实现业务成果。如果清楚地理解了这一点，第一个障碍就消除了。

第二个障碍是煮沸海洋的复杂愿景——实现 100%敏捷、100% DevOps 或 100% SRE。这是行不通的，尤其是当您拥有传统的记录系统、内部服务和传统基础设施，以及一大堆历经数年(有时是数十年)构建的包袱时，这是大多数公司的常见情况。所以在选择目标区域时要非常明确。

第三个障碍是当您认为 sre 专门负责构建操作可靠性，但与生命周期的开发或部署阶段没有任何关系。可靠性是每个人的问题——技术产品经理、开发人员、测试人员、支持工程师；不仅仅是 SREs。SRE 角色的现实是通过支持拥有这些服务的其他团队来确保服务的可用性和可靠性。sre 是推动者，他们是合作者，他们的目标是确保服务总体上是有弹性的、可靠的，并且以持续的方式交付增值。当一个组织认识到实施 SRE 是一种影响组织所有部分的基础文化变革时，那么就更容易消除主要障碍。"

### *[马来西亚国家石油公司软件工程负责人&陈丽莎](https://www.linkedin.com/in/lisa-chan-b448ab1b/)devo PS*

“并非所有组织都拥有由熟练开发人员组成的运营团队，这是采用 SRE 工作模式的关键成功因素。就我而言，我所在的 IT 组织有 1，800 名员工，自 40 年前一家石油和天然气公司成立以来，他们一直为该公司提供支持。我们服务于综合能源业务的垄断市场，其投资组合下有 100 多家公司。因此，遗留下了 2000 多个应用程序。IT 组织仍然保持着常见的“开发就是开发，操作就是操作”的反模式。即使我们正在向一个更具协作性的工作模式过渡，期望我们的 300 名运营人员将自己重新技能化为开发人员也是不现实的，因为他们中的许多人拥有传统的运营技能；例如网络、数据库管理员、服务器管理员、数据中心管理、呼叫中心代理等。”

### *[BMK·拉克什米纳拉亚南](https://www.linkedin.com/in/bmknz/)，新西兰银行价值流架构师*

“在企业内部建立 SRE 实践和技能总是具有挑战性，因为一些企业仍受旧思维方式的困扰。思维、相关技术、可用工具和有限的技能限制了大多数企业对 SRE 的采用。

将系统管理员改名为 SREs 既无济于事，也不会让 SRE 成为专家。投资昂贵的工具也于事无补。任何采用和改造都是一个渐进的过程。一个典型的大中型企业有数百个系统、应用程序和服务，要支持各种操作系统，要管理数据库平台，还要监控和管理数百个应用程序数据库。对于这种复杂性，还要加上其他服务和组件的所有依赖关系。因此，对于大多数企业来说，重新定义这些服务的 SLO、SLA 和 sli，自动化日常维护任务，了解 toil，在这些系统和应用程序中建立弹性和容错特性是一项艰巨的任务。"

## 确定其他常见的采纳障碍

企业面临的许多其他常见障碍是:

*   在一个集中的平台上提供最新的系统和应用程序信息
*   遗留系统的可观察性
*   了解依赖关系和服务级别(承诺理论)
*   为正确的工作提供正确的工具
*   技能——旧的方法不会打开新的大门。
*   时间——人们提升技能、训练和实验的时间和空间。
