# 重新思考您在云中的工具链

> 原文:[https://devops.com/rethinking-your-tool-chain-in-the-cloud/](https://devops.com/rethinking-your-tool-chain-in-the-cloud/)

每个 IT 部门都有一个“工具链”，即除操作系统和应用程序之外的所有软件，它们填补了空白并保持一切正常运行。这包括用于监控应用程序、数据保护、软件开发、版本管理、供应和取消供应资源等的解决方案。随着时间的推移，这些工具逐渐建立起来，随着公司的发展，这些工具通常从自制的解决方案演变成商业软件。

云代表了另一个转折点:一种管理系统和应用程序的新方式。当一家公司考虑将应用程序迁移到云中时，IT 和[应用程序开发](https://devops.com/?s=application+development)团队应该抓住机会重新评估他们的工具链。他们现有的解决方案是否仍然适合这个新的基于云的世界，或者是否有更好的替代方案？以下是做出决定的流程，以及在评估某项变革是否符合组织的最佳利益时需要考虑的一些重要问题。

## 第一步:盘点当前的工具链

首先，IT 团队应该对他们当前的堆栈进行全面盘点，以确保他们不会忽略任何东西。该审核应包括以下方面:

*   供应工具(如 Ansible 或 Terraform)。
*   身份和访问管理(IAM)解决方案，允许 IT 部门将对服务器、资源和软件的访问权限仅限于相关用户。
*   开发工具包括 ide、持续集成/持续交付(CI/CD)、源代码控制等等。
*   数据保护、高可用性、灾难恢复和备份软件。
*   监控和审计软件(如 Datadog)。

## 第二步:评估每个工具满足其需求的程度

接下来，考虑每个产品现在满足其需求的情况，以及随着工作负载转移到[云](https://www.skytap.com/)这些需求将如何变化。第一个也是最重要的问题:供应商是否仍然支持您使用的工具和/或版本？像这种向云迁移的变化很少见，所以一个或多个产品不再受支持的可能性比人们想象的要高。其次，考虑软件的许可条款，以及它们是否允许在云中使用。例如，一些工具被许可给特定的物理服务器，而一些供应商要求他们的硬件归持有许可证的同一实体所有。这两种场景在云中都不太适用。第三，应用程序在迁移后会以同样的方式进行管理吗？云通常会消除约束；它提供几乎无限的计算和存储、动态工作负载、新的 API 和全球多个地区。IT 团队可能希望通过迁移到基于云的工具(例如，取代内部应用程序日志报告解决方案)来利用这些功能，或者他们可能希望尽可能保持管理的一致性，以最大限度地减少员工所需的再培训量。

不要忘记考虑非技术性的论点。团队是否对当前的解决方案感到满意并富有成效，或者他们是否感到被迫使用它？它是否满足组织的业务需求？这种变化会在短期内创造更多的工作吗？这些理由不足以让工具独立存在，但是它们应该成为讨论的一部分。

## 第三步:创建备选方案图

接下来，看看可能需要替换的工具的潜在替代品。这可能是一个很好的整合机会，所以先看看组织中的其他团队已经在目标云中使用了什么。在这里，IT 部门必须做出的最常见的决定是，是否切换到基于云的工具(要么是云供应商提供的工具，要么是独立的 SaaS 产品)。这些通常提供随用随付许可，并且通常更容易扩大或缩小规模、转移工作负载以及扩展到其他地区。传统供应商的许可差别很大；一些现在提供基于消费的选项来与云竞争，而一些则坚持永久许可。基于特定公司的需求，这些因素可能会使基于云的工具对特定公司或多或少具有吸引力。最后，考虑做出改变是否会允许 IT 团队扩展他们的技能。对于喜欢学习新事物的工程师来说，添加新产品实际上可以增加满意度和保留率，因为这给了他们学习新事物的机会。

## 第四步:与现有的工具供应商交流

即使不同的选择似乎是更好的选择，与当前的供应商联系并给他们一个保留您业务的机会总是值得的。他们可能会提供更灵活的条款来留住你这个顾客。相反也是可能的；一些供应商不会在他们的许可上让步，因为他们认为云是一种威胁。即使与供应商的对话没有改变他们愿意提供的东西，通常也会揭示出他们如何接近客户的有用见解。

## 第五步:为每个工具创建一个计划

根据在最后四个步骤中收集的信息，是时候决定使用每个工具做什么，并为每个过渡制定计划了。必要时，记下需要尽快更换的产品、需要进行更多研究的产品，并汇编在此过程中获得的任何其他有用信息，以便更大的 IT 团队可以访问这些信息。还要考虑团队是否需要培训，内部文档或行动手册是否需要更新，以及新工具如何插入现有的授权/身份验证解决方案。每个变更还需要某种类型的迁移计划，详细说明组织将如何以及何时从旧产品迁移到新产品，需要重写哪些脚本，以及如何处理旧产品的日志文件等历史数据。

基于云的工具不会适合所有人，但它们的灵活性、潜在的成本节约和可扩展性不应该被低估。虽然没有人渴望向云迁移项目添加更多工作，但花时间完成这一过程可以发现更适合企业正在转向的新云模式的有用的新选项。
