# 这是一次考验

> 原文：<https://devops.com/this-has-been-a-test/>

在应用程序开发的几乎整个历史中——当然是从应用程序开发随着个人计算机被广泛用于商业用途而进入更大的市场以来——测试在大多数行业中一直处于次要地位。这并不是因为我们不关心代码质量，而是因为它占用了开发酷的新工具——甚至产品——的资源，并减缓了发布周期。从历史上看，这两个问题都是巨大的。手动测试人员从几个人迅速增长到一个大部门，等待这个过程完成可能要花很长时间。

所以，我们还没有做很多。哦，当高管或领导层要求更彻底的测试时，IT 管理层会承诺这样做，然后——为了符合预算和时间限制——会告诉开发人员测试他们自己的代码。有几个地方，你不能让负责创建最终产品的人来审查最终产品——因为如果他们知道自己做错了，他们会改正的。写作就是这样一个领域——这就是为什么好的编辑是物有所值的，即使在互联网时代，任何人都可以(很多人都这样做)在没有编辑帮助的情况下把他们喜欢的东西放到数字页面上。另一个是测试应用程序代码。我一边写代码一边测试，就像每个花时间做专业开发人员的人一样，我犯了几个惊人的错误。如果我知道我在系统中留下的特别的弱点在那里，我不会说它已经完成了。

当 DevOps 出现时，我期望事情会有所改变——直到我明白 DevOps(和敏捷，步调一致)是为了交付更快的*而不是更好的*。任何减慢这个过程的事情——测试、安全性，甚至是不可避免的遵从性——都是不被允许的。当[测试驱动开发](https://www.agilealliance.org/glossary/tdd/) (TDD)出现时，我立刻对它产生了爱恨交加的感情。我们在我参与的一个项目中使用了它，坦率地说，它将质量问题转移到了测试上——代码可以被强制通过所有的测试，但是测试本身有多彻底和准确——特别是随着项目的增长和变化——成了问题。如果做得好，TDD 将会解决我们的许多测试问题。见鬼，即使做错了，它也能解决简单的单元测试所能覆盖的大多数问题。

但是几十年后，我认为我们终于到了可以要求和要求全面测试可接受和不断增长的覆盖率的时候了。你需要考虑一下。测试自动化已经走了很长的路，但是你们中有太多的人还没有跟上它。如果你努力证明投资测试套件是正确的，我理解。我曾经是一名经理，不得不在提高数据库性能和测试之间做出选择。但是现代的迭代包括安全测试。因此，如果开箱即用，您可以获得一个工具来检查有效输入，并提醒无效数据可能会破坏应用程序的地方，这可能会有所帮助。而这只是最简单的情况。我目前在一个相邻的空间工作，所以我不想太疯狂地在这里推销你。我只会说，“测试已经不是几年前的样子了。看看吧。”应用程序和安全性测试。随着使用人工智能/人工智能的主张转变为应用实际算法，你的人工智能或数据科学人员可以进行比较，以帮助选择高质量的产品，这些变化将继续出现。

我们可以整天争论 DevOps 是否让应用程序变得更加稳定和安全，但我们不需要这么做。最后，[连续测试](https://devops.com/?s=continuous+testing)真实。查看这样做的供应商，比较他们，看看将他们融入您的环境有多容易。更稳定，更安全，而不是你头脑中的障碍。不喜欢什么？你已经开始摇摆了；自动化测试会让你看起来更好。
