# 为什么可观察性和监控对 SREs 很重要

> 原文：<https://devops.com/why-observability-and-monitoring-matter-to-sres/>

现场可靠性工程师，或 SREs，做很多事情。它们帮助开发人员在应用程序中建立可靠性。他们管理 SLA 和 SLO。他们在事件管理和[事件响应](https://devops.com/?s=incident+response)中发挥着主导作用。

对于所有这些任务，SREs 在很大程度上依赖于可观察性和监控。尽管 IT 组织的其他部门通常也帮助管理可观察性和监控，但如果没有可观察性和监控工具及数据，sre 几乎不可能在当今的云原生世界中有效地完成工作。

以下是可观察性和监控的含义，为什么它们对 sre 特别重要，以及 sre 如何系统地衡量他们的监控和可观察性实践的成功。

## 定义可观察性与监控

监控是从应用程序、基础架构或其他 IT 资源收集数据，以跟踪这些资源的状态和性能。自 20 世纪 90 年代末第一批现代监控工具(如 [、Nagios](https://www.nagios.org/about/history/) )出现以来，监控一直是 IT 实践的核心。

与监控相比，可观察性更强。可观察性是从多个系统收集离散类型的数据，并关联这些数据集，以实现对 IT 资源的可操作可见性。有一点需要注意:可观测性在控制理论领域有一个更专业的定义[](https://www.ece.rutgers.edu/~gajic/psfiles/chap5traCO.pdf)，它与使用外部输出来推断系统的内部状态有关；然而，当人们在 IT 和 SRE 的背景下谈论可观测性时，他们通常指的是为了理解复杂系统而收集和关联大量的数据。

关于 [可观察性和](https://dzone.com/articles/observability-vs-monitoring) 可监控性之间差异的争论正在进行，但共识是它们归结为:

*   **什么与为什么**:监控有助于 SREs 确定是否存在问题。通过允许他们理解*为什么*它存在，可观察性走得更远。
*   **上下文**:可观察性提供了关于系统问题的上下文。在很大程度上，监控提供了关于问题存在的表面信息。
*   **可操作性**:可观察性使 sre 能够有效地应对中断、性能下降或其他问题。监控只是让他们知道出了问题，可能需要干预。

在大多数方面，监测是更大的可观测性过程中的一个步骤。但是可观察性涉及到额外的实践，例如各种类型数据的相关性，以达到某种程度的上下文和可操作性，这是单独的监控所不能提供的。

## 为什么 SREs 应该理解可观测性

同样，可观察性已经成为各种角色的工程师的核心实践。开发人员可以在部署之前使用可观察性工具来帮助测量和优化应用程序性能。IT 工程师可以利用可观察性来了解生产环境中存在的问题。可观察性工具可以帮助 QA 工程师确定应用程序测试失败的原因。

但是没有一个群体比 SREs 从可观察性中获益更多。考虑到最大限度地提高系统的可靠性和性能是 SREs 的核心任务，不仅能够通过监控发现问题，还能够通过可观察性了解问题，这对于现代 SRE 团队来说至关重要。

对于负责管理复杂系统(如基于微服务的系统)可靠性的 sre 来说，更是如此。在复杂的应用中，确定可靠性问题的根本原因——例如哪个特定的微服务包含错误代码，或者哪种类型的请求触发了性能错误——可能非常困难。通过最大化可靠性问题的上下文，以及允许团队跨复杂的分布式系统跟踪请求，可观测性将 SREs 置于快速发现和修复可靠性问题的最有利位置。

此外，除了检测已经部署的系统中的问题，SREs 还可以利用可观察性工具来帮助在应用程序投入生产之前为其构建可靠性。通过观察开发/测试环境中的应用程序，SREs 可能能够识别可靠性风险，然后利用这种洞察力找到使应用程序本质上更可靠的方法。例如，可观察性可能揭示出源于应用程序的架构设计或管理应用程序的编排工具的可靠性弱点。

## 如何衡量可观察性和监控

从可观察性和监控中获得最大收益需要系统地测量可观察性和监控计划。只有通过收集关于可观察性和监控结果的指标，sre 才能确保他们提高可靠性，而不仅仅是收集和分析数据，没有特定的目的。

虽然对可观察性和监控度量的全面讨论超出了本文的范围，但是考虑以下类型的度量来评估这些实践:

*   **事故频率**:影响生产系统的事故(尤其是严重事故)发生的频率如何？理想情况下，SREs 将会看到与可观察性和监控工具使用增加相关的事件频率降低。
*   回滚频率:一个应用程序版本回滚到先前版本的频率越低，SRE 团队的可观察性和监控实践可能就越有效。
*   **平均修复时间(MTTR)** : MTTR 是解决可靠性问题所需总时间的度量。它反映了您的监控能力(这决定了您发现问题的速度)和观察能力(这是解决问题的关键)。
*   **分析的数据总量**:通过可观察性和监控工具跟踪您收集和分析的数据总量，有助于您评估实践的总体范围和可伸缩性。
*   **分析的总系统**:除了跟踪总数据量之外，考虑测量您作为可观察性和监控工作流的一部分管理的离散应用程序、托管环境、CI/CD 工具和其他系统的数量。

SREs 的最终目标超越了可观测性和监控。然而，可观察性和监控是实现 sre 的更广泛目标的重要手段:最大化正常运行时间和性能。虽然监控本身曾经足以支持 SREs 的目标，但很难想象今天的 SRE 团队不广泛利用可观察性，从而将监控提升到一个新的水平。
