# 开源与专有数据操作

> 原文：<https://devops.com/open-source-vs-proprietary-dataops/>

核心数据操作概念正在进入数据工程团队，并从那里进入更广泛的企业。数据工程师正在改变他们创建数据产品的方式，其中大部分工作围绕着创建数据管道。

DataOps 管道提供了传统数据集成和 ETL 过程所没有或不能提供的可观察性。它们允许您持续集成和测试新的数据源，并在流或批处理环境中提供比传统的孤立方法更高质量和可靠性的数据。他们还可以通过为训练、测试和部署准备数据来支持机器学习工作。

DataOps 管道缩短了数据消费者的请求与数据生产者满足请求的速度和可靠性之间的周期时间，同时在整个生命周期中观察服务质量。团队在构建管道时面临的挑战是如何构建管道。在我看来，有两种方法，一种是使用开源软件自己动手，另一种是通过购买商业产品。

开源当然有它的拥护者，也有一些明显的成功，但是采用开源路线也有利弊。

## 构建或购买您的管道

开源的一个优势在于它的灵活性和可用性。开源许可证，不包括 SSPL，给了用户难以置信的自由，他们可以用软件做什么。如果您有这方面的技能，您可以构建一个 DataOps 管道，它可以接收任何数据，丰富数据并将其路由到正确的位置。然而，这种灵活性也是一个缺点。虽然你可以做任何你想做的事，但你也必须做这件事。像[卡夫卡](https://kafka.apache.org/)、[脉冲星](https://pulsar.apache.org/)、[火花](https://spark.apache.org/)、[气流](https://airflow.apache.org/)和 [Flink](https://flink.apache.org/) 这样的开源项目对他们正在处理的数据一无所知。这取决于开发商。

这听起来可能不是问题，但今天的数据工程师正在处理数百甚至数千种不同格式的数十种数据类型。如果添加运营数据，您还会看到从防火墙、容器、SNMP 陷阱和 HTTP 来源涌入的数据。这就是你要面对的。您还需要从对象存储、多个活动中心和其他消息来源获取数据。没有一个开源项目本身支持现代数据操作管道中所需的数据种类和数据量。您添加的每个新数据源都意味着从原始组件开始，而不是从更高层次的抽象开始，为您的工作增加时间和无差别的努力。

您还需要一些基本特性，比如每个源的背压、对一系列协议和数据存储的支持、角色和权限管理等等。对于超负荷工作的数据工程团队来说，在管道的整个生命周期中，这是一个很大的工作量。

专利产品也没有“简单按钮”。许多商业产品仅支持少数源和目的地，并且可能具有与数据接收速率或每个连接器许可费用相关的过高许可成本。此外，从运营的角度来看，它们的扩展也很有挑战性。在演示过程中，看到一个很酷的拖放界面可能看起来很棒，但当您看到成千上万个类似的数据处理管道时，它很快就会变得杂乱无章。

商业产品的优势集中在企业需要的特性上，比如治理、管理和安全性。它们也可能比开源软件更容易与其他组件集成。最后，尽管开源核心公司也提供支持，但在生产环境中运行时，当您需要支持时，有人打电话也是必不可少的。

## 从更高的角度看问题

在评估 DataOps 管道选项时，请记住两点。首先，无论您选择哪种方式，都要确保您选择的技术提供可观察性和监控功能。数据运营是为了更快地对变化做出反应，如果你不知道发生了什么，你就无法做出反应。构建可观察的数据运营管道可确保您向企业中的数据消费者提供有价值的数据产品。

接下来，越过技术。DataOps 与其说是一项技术实践，不如说是一项人员培训。成功的 DataOps 实施的一个关键组成部分是跨团队调整指标和激励措施。如果运行 DataOps 管道的员工与他们的数据产品所传达和影响的业务成果脱节，他们就没有动力将管道作为任务关键型基础设施组件来运行。关键是让你的所有团队都有共同的衡量标准和激励机制。

## 数据操作确定

在决定 DataOps 管道实现之前，您应该做一些事情。

*   确定谁是您的客户，以及他们对数据交付、丰富、用户体验和可观察性要求的期望。你是只为数据科学家服务，还是也为基础设施和运营、商业智能和营销服务？了解你的客户可以让你做出更好的决定。
*   在真实环境中测试软件。每个人都不喜欢在购买/构建之前被告知要进行概念验证，因为概念验证非常耗时。记住——这比选择错误的工具，然后看起来很愚蠢要省时。

工程团队有时会选择流行的技术，而不是为工作选择合适的工具。以务实的态度看待性能、可管理性和工程工作，并让广泛的利益相关方参与进来，将为您带来更高的 DataOps 成功机会。
