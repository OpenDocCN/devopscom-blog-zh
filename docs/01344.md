# 在 DevOps 世界中测试会发生什么？

> 原文:[https://devops.com/happens-test-devops-world/](https://devops.com/happens-test-devops-world/)

我总是对 DevOps 周围的都市传说感到惊讶。我最喜欢的一点是，如果你实现了一个 DevOps 方法，你可以停止做测试。人们引用了一句著名的话，网飞只在生产中进行测试来证明这一点。这也许更像是电话游戏，故事的大部分在翻译中丢失了。的确，网飞创造了一个名为“混沌猴子”的工具，它在生产中产生故障，以此来测试其软件在大规模生产中的容错能力。然而，这不是网飞唯一的*测试；这是一种*附加*类型的大规模测试，允许公司以新的和不同的方式进行测试。网飞仍然在测试软件开发。*

好了，现在我们已经过了 DevOps 是关于消除测试的概念，让我们看看在 DevOps 世界中测试到底发生了什么。事实上，我们的目标是让测试随时随地进行。让我们看看两个方向的测试可以转移:左或右。

## 左移位

DevOps 方法鼓励整个开发过程中活动的自动化。这可以通过自动化开发和测试之间的链接来实现。通常使用诸如 Jenkins 之类的自动化构建工具来实现，每当开发人员完成新代码的构建时，就会触发一组自动化测试。当然，这意味着开发和测试过程都必须自动化。一旦你达到这个目标，那么测试就可以一直进行下去。每当开发人员想要签入新代码时，他们可以触发一小组“冒烟测试”如果这些成功了，他们可以签入代码，当构建发生时，他们可以触发一组更大的回归测试。结果是测试向开发的左移。

左移测试的好处包括:

*   将测试人员移入 scrum 团队，与开发人员一起开发故事。
*   与开发并行的设计测试。
*   让开发人员和测试人员灵活地互相帮助来减少瓶颈。
*   bug 的早期检测，缩短创建 bug 和修复 bug 之间的关键时间。
*   增加了代码依赖问题的透明度，因为它们是在夜间(或更频繁地)回归测试中提出的。
*   虚拟消除开发完成后出现的测试瓶颈。

## 右移

关于网飞的都市传说可能是错误的，但这并不意味着将测试转向生产没有好处。生产中的自动化测试对于在生产之外不可能执行的测试是有效的，包括规模测试和弹性测试。生产也是一个引入 A/B 测试的好地方，它为一些客户提供不同的接口，并测量业务结果的差异。

生产测试的好处包括:

*   更好地了解真实的生产环境和工作负载。
*   客户对新功能和用户体验的早期反馈。
*   更好的弹性。
*   为客户提供更好的整体用户体验。

## 无换档

等等，如果我们左移和右移，test 中还剩下什么测试要执行吗？仍然有一系列的测试需要由测试团队来执行。有了全自动化，这些测试可以在不同的时间执行，不需要成为发布的瓶颈。这些测试包括:

*   性能试验
*   端到端用例测试
*   安全测试(不要想着在生产中做这个！)
*   手动测试——通常用于最难自动化的用户界面或移动测试

DevOps 方法并没有促进测试的结束，而是事实上扩展了测试的范围，包含了 DevOps 周期中的所有步骤。你知道他们是怎么说的:早测试，多测试！下次当你们围坐在篝火旁讨论城市传说时，让我们把网飞排除在外。
