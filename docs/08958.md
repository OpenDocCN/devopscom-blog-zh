# 2021 年最佳——从敏捷到 DevOps 再到 DevSecOps:下一次变革

> 原文：<https://devops.com/from-agile-to-devops-to-devsecops-the-next-evolution/>

随着 2021 年的临近，我们 DevOps.com 想要突出今年最受欢迎的文章。以下是我们 2021 年最佳系列的第九部。

随着工作人员转移到远程位置，网络变得更加分散，因为边缘越来越远。与此同时，这些网络上的设备和应用数量也在持续增长。一些估计表明，到 2030 年，地球上每个人将拥有 [15 台联网设备](https://www.statista.com/statistics/802690/worldwide-connected-devices-by-access-technology/)。

远程工作人员及其相关应用程序需求的快速增长给应用程序开发人员和负责应用程序部署和维护的 IT 专业人员带来了巨大的挑战。在发布速度与应用程序的稳定性和可靠性之间找到正确的平衡——这从来都不是一件容易的事情——对于任何组织的成功都是至关重要的。

敏捷和 DevOps 的发展是为了帮助组织[达到这种平衡](https://devops.com/how-to-combine-devops-and-agile/)，在最短的时间内提供最好的客户体验。但是敏捷和 DevOps 实现了这个承诺吗？

## 敏捷宣言的承诺

虽然敏捷作为一种方法论已经被人们所知并遵循了近二十年，但是组织应用敏捷的方式却和组织本身一样多种多样。

敏捷宣言在 2001 年第一次被写出来时，列出了四个基本价值观:

*   个人和交互优先于过程和工具。
*   工作软件优先于全面的文档。
*   客户协作优先于合同谈判。
*   对变化的反应优先于盲目地跟随计划。

敏捷流程适用于一系列开发活动，从独立应用程序的开发到云原生业务应用程序的开发。敏捷宣言还包含了更多的原则，进一步解释了这些价值观在实践中的意义。

这些敏捷原则可以浓缩成几个重点领域:

### 优先考虑客户而不是产品

敏捷开发需要[持续的客户参与](https://www.researchgate.net/publication/221440354_Agile_customer_engagement_A_longitudinal_qualitative_case_study)贯穿整个开发过程，最终目标是基于特定客户反馈的频繁产品交付。虽然这可能会给敏捷团队带来挑战，尤其是当客户在开发周期的后期要求变更时，但是敏捷坚持认为提升客户体验的最佳方式是让客户成为开发过程的一部分。

### 相信敏捷团队能够自我组织和自我管理

敏捷专注于与客户以及彼此之间保持持续沟通的小型团队。这些团队应该包括开发人员和那些了解通过开发来满足更大的业务需求的人员。要让这些团队有效工作，信任是关键。敏捷认为，自组织的团队，被给予了以他们认为合适的方式追求项目的余地，会导致最短的开发周期。

### 保持简单

敏捷团队需要简单。这与敏捷对过程过度使用的蔑视相关。敏捷寻求最小化所需的工作总量，从而最小化总体开发时间。

### 敏捷导致敏捷

正如敏捷团队不断改进他们的产品和客户体验一样，他们也在不断改进自己。团队和他们所采用的过程的不断发展(记住，团队先于过程)提高了团队敏捷性本身。

虽然敏捷的价值观和原则看起来很简单，但是价值观和原则的有效应用完全是另一回事。事实上，有时候，敏捷开发的实现已经违背了宣言的第一个价值，因为它关注于应该使用哪些工具和过程。但是，在一个越来越多的用户要求更多产品和服务的环境中，实施敏捷方法不仅重要，而且势在必行。

## 使用 DevOps 构建敏捷

该术语本身的确切含义和 DevOps 的范围仍然是一个争论的来源。在高层次上，DevOps 是开发周期与后期开发活动(如部署和维护)的结合。

根据开源贡献者 [Cloud Defense](https://www.clouddefense.ai/blog/what-is-devops) 的 Barbara Ericson 的说法，“DevOps 打破了之前存在于 IT 运营和软件开发之间的传统界限，以实现向最终用户持续集成和持续交付(CI/CD)高质量的软件功能和应用，而不是将 IT 运营和软件开发相互隔离。”

正如敏捷一样，开发运维的有效实施需要在组织内培养开发运维文化。然而，敏捷和 DevOps 也不是互斥的。也就是说，DevOps 不仅仅是敏捷的替代或发展。尽管敏捷和开发运维在结构和方法上有所不同，但它们可以[协同工作](https://www.simplilearn.com/agile-vs-devops-article)以实现快速部署和增强客户体验的最终目标。

虽然敏捷和开发运维有着共同的目标，但是他们并不总是在如何实现这些目标上达成一致。DevOps 在许多方面与敏捷不同，但是，在最好的情况下，DevOps 应用敏捷方法和精益制造原则来加速软件部署。

敏捷和 DevOps 之间特别紧张的一个领域是，后者严重依赖工具；特别是，当涉及到测试和部署过程的自动化时。但是 DevOps 可以简单地通过应用敏捷原则本身来克服敏捷开发人员对工具使用的抵制。

实际上，DevOps 的支持者必须让敏捷团队相信，教条地坚持敏捷的基本原则实际上与敏捷根本就不一致。具有讽刺意味的是，坚持认为过程总是不好的敏捷开发人员实际上违反了敏捷原则，因为他们拒绝承认变更带来的好处，这是另一个敏捷原则。

挑战在于让敏捷开发团队[信任 DevOps](https://techbeacon.com/devops/how-agile-teams-can-get-devops-right-6-essential-steps) 的自动化工作，同时鼓励 DevOps 团队考虑部署的业务目标，而不是为了部署速度而追求部署速度。

随着敏捷团队和 DevOps 团队之间的不断交流(另一个敏捷原则)，开发可以在一定程度上适应 DevOps 任务和过程。这意味着测试和部署自动化可以快速进行，从而缩短整体上市时间。

最终，当敏捷和开发运维回忆起他们的共同目标时，他们可以最有效地合作:基于反馈的更小的变更增量导致更快的交付周期。

## 持续进化:DevSecOps

随着网络的扩展和企业不断集成基于云的服务(在许多情况下，构建多云环境)，安全问题变得越来越复杂。敏捷缺乏的一个领域是将安全性集成到开发过程中。不幸的是，在敏捷环境中，应用程序安全性通常是在开发之后解决的问题[，而不是作为过程的一部分。进入 DevOps 的下一个迭代:DevSecOps。](https://devops.com/devsecops-vs-agile-development-putting-security-at-the-heart-of-program-development/)

DevSecOps 围绕三个基本原则:确保数据安全，同时尽量减少用户访问数据的不便，使用开发工具在开发过程中尽早识别风险，并使用安全套接字层(SSL)和虚拟专用网络(VPN)等技术确保数据加密。

虽然 DevSecOps 和 Agile 可能也不同意工具的优先级，但是安全性必须成为任何开发项目不可或缺的一部分。这就是为什么 Agile、DevOps 和 DevSecOps 必须通力合作，以确保快速部署和安全部署。

## 下一次进化

敏捷和 DevOps 都是软件开发过程的重要演进，但是 DevSecOps 是下一个演进步骤。尽管有一些完全不同的特征，敏捷、开发运维以及开发安全运维可以而且应该协同工作来实现他们的共同目标:尽可能好的客户体验和短的部署周期。
