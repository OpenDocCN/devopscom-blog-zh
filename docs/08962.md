# 2021 年最佳——如何结合开发运维与敏捷

> 原文：<https://devops.com/how-to-combine-devops-and-agile/>

随着 2021 年的临近，我们 DevOps.com 想要突出今年最受欢迎的文章。以下是我们 2021 年最佳系列的第十三部。

近年来，应用程序开发和部署已经成为业务运营中越来越重要的一部分。正因为如此，各种实体已经寻求优化他们的产品开发过程。

这使得专为这一目的设计的 DevOps 越来越受欢迎。简而言之，软件开发过程中的 DevOps 应用程序减少了将软件推向市场所需的步骤。这些更快的发布和简化的流程意味着迅速的用户反馈。

DevOps 的实施还关注软件的可扩展性、部署情况以及未来版本发布后的监控和维护。然而，传统的 DevOps 也有不利之处。该系统不包括敏捷所提供的那种持续测试和改进。

这导致敏捷实践压倒性地聚焦于你可能定义的软件交付的开发方面。不过，对运营方面的关注较少。

因此，在任何产品的 SDLC 中，这两种实践都是非常关键的。

将软件开发的敏捷方法和 DevOps 方法分开会导致产品的构建，但是它的部署、工作自动化以及基础设施管理会失败，因为在敏捷团队看来这是“别人的问题”。此外，“可操作性”消失在背景中。

解决方案是将敏捷冲刺与 DevOps 提供的整合团队合作相结合。通过这样做，您可以逐步优化产品的开发生命周期和维护。它有助于纠正不平衡，但是对持续开发阶段发生的实践几乎没有影响。

## DevOps 和敏捷集成实现了什么？

开发运维与敏捷的集成有助于以下方面:

*   简化发布流程，改进产品供应。
*   允许更好的协作。
*   每个版本的价值更高，风险更小。
*   更少的错误和更快的修复。
*   能见度提高。
*   更高的用户满意度，因为产品质量更高。

## **集成 DevOps 和敏捷需要注意什么**

为了简化在敏捷产品开发中实施 DevOps 的过程，我们收集了在此过程中可能发生的最常见的陷阱，以及如何避免它们，使您的过程顺利进行。

**改善团队合作流程**

将 DevOps 框架和敏捷方法结合使用使得团队成员对所有开发方面有更广泛的理解变得至关重要。你将 DevOps 的商业价值和敏捷的实用性结合在一起。

产品负责人、[Scrum Master](https://www.scrum.org/resources/what-is-a-scrum-master)和 PM 等团队成员，以及运营、基础设施和系统管理员角色不仅需要考虑软件开发过程，还需要考虑交付和维护。您的团队应该具备发布、服务和变更管理、环境供应、自动化和工具以及应用程序部署方面的知识。构建并运行以产品为中心的团队解决了敏捷开发到运营的交接问题。

**定义生命周期**

使用敏捷框架成功尝试开发运维实施计划意味着定义产品生命周期。这提高了一致性，通过最大限度地减少浪费降低了成本，并加快了上市时间。团队现在在整个生命周期中继承了更多的运营问题。因此，如果您在开发过程的一开始就开始实施一些 DevOps 原则，那将是非常理想的。

**sprint 采用 DevOps】**

敏捷工作流假设软件开发过程分为 sprint，因此在处理 sprint 的同时集成 DevOps 管理具有战略意义。

当你开始在短跑中运用 DevOps 方法时，请遵循这些指导。

*   邀请运营/基础设施/支持人员参加规划会议。
*   讨论产品的功能和可操作性特点。
*   将他们纳入即将到来的冲刺阶段。
*   让 DevOps 团队参与 sprint backlog 计划和每日会议，以及 sprint 评审、scrum 和计划协调。

开发团队和运营团队的参与和协作也让运营团队与功能发布时间表保持同步。然后，运营团队可以帮助开发团队更准确地规划发布时间表，并帮助开发团队更快地交付产品。

**包括每个阶段的质量保证**

使 QA 成为整个开发生命周期的一部分是集成开发运维与敏捷的另一个重要因素。将两者结合起来时，测试占据了重要的位置。除了敏捷中应用的功能测试，DevOps 方法还需要软件的性能和负载测试。因此，持续的测试和持续的开发同样重要。

**实施 DevOps 下的服务积压**

当同时实施 DevOps 和 Agile 时，您需要重建您的服务积压流程。在 DevOps 框架下，应该包括:

*   软件的可扩展性。
*   部署能力。
*   服务监控。
*   伐木。
*   设置报警。
*   软件测试。
*   安全性和合规性方面。
*   运营绩效。

**利用正确的工具**

使用正确的工具是在软件开发中成功采用敏捷和 DevOps 的关键。请记住，适用于您的软件开发流程配置管理工具，使用 DevOps 所需的基础架构即代码(IaaC)概念来创建和复制基础架构。这有助于开发人员在不同类型的平台上部署应用程序，而无需任何返工。

**自动化**

工作流自动化是敏捷与 DevOps 结合的另一部分。尝试自动化所有代码扫描过程，避免任何潜在的漏洞。在存储库中构建工件，或者将发布自动化。将这些元素完全自动化，这样您的端到端标杆就是一个可能需要人工检查问题的地方。

**文档**

在敏捷方法中，团队不记录他们的会议记录或其他交流。相反，他们更喜欢笔和纸等低保真方法。另一方面，DevOps 需要完整的设计文档和规格来理解一个软件版本。

**测量和分析**

在将 DevOps 构建到敏捷项目管理中以跟踪其进展之后，您需要关注建立度量标准来衡量其有效性。这使得多个版本能够更快地成功投入生产。根据 Scrum 联盟组织的建议，其中一些可能是:

*   遵守发布日期的百分比。
*   发布数量的百分比增长。
*   发布到生产所需的时间。
*   可归因于平台/支持要求的缺陷。
*   符合 NFRs 的百分比。

虽然您可以在开发运维实施期间定义其他指标来衡量更高的业务价值。

为什么 DevOps 和敏捷都很重要现在应该很明显了。尽管这些实践有助于精简和简化产品创建和部署过程，但是将敏捷和开发运维结合起来需要组织的转变。但是努力以正确的方式结合敏捷和 DevOps 管理，你会看到它如何改进你的开发过程，并在可靠的、可伸缩的和可维护的应用程序中交付结果。
