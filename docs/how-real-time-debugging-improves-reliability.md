# 实时调试如何提高可靠性

> 原文：<https://devops.com/how-real-time-debugging-improves-reliability/>

当设计和构建软件时，服务可靠性总是开发团队最关注的领域。每个构建软件的团队通常直接或间接地与他们的客户签订[服务水平协议](https://devops.com/?s=SLAs)。本质上，这些是团队用来测量和确保软件系统可靠性的一致同意的度量或性能标准。

组织可能会以不同的方式来衡量可靠性，查看服务可用性、平均无故障时间或平均修复时间等指标。不管组织选择如何度量可靠性，您通常都可以根据应用程序的可靠性来评估组织的健康状况。因为我们知道，无论我们准备得多么充分，可靠性问题都会发生，所以拥有快速有效地解决这些问题的工具对于保持平稳运营至关重要。

## 您组织的健康状况

今天，大多数软件开发组织都有关键的性能指标([KPI](https://kpi.org/KPI-Basics))，他们通过测量这些指标来了解他们做得有多好，并且能够快速、轻松地测量他们的应用程序开发过程的健康状况。如果团队发现他们没有达到 KPI，这通常是一个迹象，表明一些基本的东西需要改变，或者新的和改进的过程需要到位。用数据驱动的方法不断衡量他们做得好的地方和他们做得差的地方的团队通常是领先的团队。

简而言之，一个团队衡量和达到 KPI 和客户服务水平协议的能力直接影响到团队的整体成功和健康。如果一个组织不能在客户需要时生产出可靠可用的应用程序，那么这个公司就不太可能长期拥有客户。当可靠性问题发生时，开发人员能够立即找到代码中问题的根源，以确保他们能够尽可能快速有效地解决这些客户问题，这一点至关重要。其中一种方法是让开发人员能够在出现问题的环境中直接进行实时调试，而不需要创建成本高昂的再现环境，也不需要想出如何在本地再现问题，因为这些问题往往依赖于数据。

## 可靠性正在左移

从历史上看，软件可靠性已经落在运营或生产支持团队的肩上，他们在软件被部署到生产之后(以及在开发人员将代码扔过隐喻墙之后)管理软件的部署和运行时。近年来情况发生了变化，现在许多开发团队负责他们的应用程序，从开发一直到在生产中运行和支持这些应用程序。正因为如此，开发人员比以往任何时候都更有动力确保他们的软件具有内置的可靠性，或者至少有能力在出现问题时快速调试可靠性问题。

实时调试器就是这样一类工具，开发团队选择将其打包到应用程序中，以便更快地解决那些难以解决的客户问题。实时调试器是允许开发团队收集代码级调试信息的解决方案，例如局部变量的快照、堆栈跟踪、跟踪信息和分析数据。每当出现问题时，所有这些数据(传统上依赖于开发人员在他们的代码中拥有正确的日志，如果没有，则部署新的调试版本)立即变得可用。这种快速识别问题的方法可以极大地提高应用程序的可靠性，因为它允许开发人员在正常情况下所需的一小部分时间内解决问题。

## 关注可靠性的团队行动更快

许多开发团队已经转向了交付软件的敏捷方法；目标是快速响应业务需求，并尽快为客户提供新的有价值的功能。尽管这是一个有益的步骤，但计划外的事件可能会发生，需要尽可能提前计划。在 Rookout 首席执行官沙哈尔·福格尔最近发表的一篇《福布斯》文章中，他讨论了企业应该如何将这些软件错误视为小故障。任何时候团队处理软件错误或其他计划外的问题，都会消耗业务收入，并可能让客户感到沮丧。关注平均修复时间(并为开发人员提供减少平均修复时间所需的工具)等指标，可以让团队更好地预测计划外问题一旦发生需要多长时间来修复。

在 Nicole Forsgren、Jez Humble 和 Gene Kim 所著的《加速》( Accelerate )( T1)一书中，作者讨论了这样一个概念:当问题出现时，通过向开发人员提供工具来解决问题，团队创造了一个环境，在这个环境中，开发人员接受对质量、稳定性和可靠性等全球结果的责任。对技术的投资也是对人的投资。因此，难怪通过投资有助于提高可靠性的技术，团队将能够更快地前进。这意味着更快乐的客户、更快乐的员工和繁荣的业务。

通过关注服务的可靠性，开发团队不仅能够更快地行动，而且还直接对他们组织的整体健康和成功做出贡献。我们看到可靠性的责任从操作团队转移到开发团队，这增加了新发现的确保开发团队将可靠性构建到他们的产品中的重要性。实时调试工具是一种潜在的解决方案，可以确保开发人员能够快速解决计划外的客户问题，这些问题会直接影响应用程序的可靠性。
