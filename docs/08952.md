# 2021 年最佳——不应该雇佣 DevOps 工程师的七个理由

> 原文:[https://devo PS . com/seven-reasons-you-should-hire-a-devo PS-engineer/](https://devops.com/seven-reasons-you-shouldnt-hire-a-devops-engineer/)

随着 2021 年的临近，我们 DevOps.com 想要突出今年最受欢迎的文章。以下是我们 2021 年最佳系列的第五部。

DevOps 是一种文化，不是一个头衔。因此，公司应该遵循 DevOps 原则，而不是雇佣一个专家来修复整个组织的交付模式。

我最近与 Replix 的 DevOps 主管 Uri Zaidenwerg 进行了交谈，以了解 DevOps 的现状。随着更强大的 DevOps 工具的出现和实践在组织间的扩散，开发人员现在能够更好地拥有他们的端到端应用，包括部署、[安全性](https://containerjournal.com/features/leveling-up-container-security-with-security-as-code/)甚至云成本。

从开发周期中提取 DevOps 可能会违背开发和运营去孤岛化的目的。下面，我们将概述一些原因，为什么你应该专注于建立一个整体文化，而不是填补 DevOps 工程师的角色。

### 1.DevOps 是一种文化，而不是一个角色

第一个原因也是最包罗万象的——devo PS 不是一个头衔；这是一种文化。DevOps 作为 IT 部门采用的一种思维模式发挥了最佳作用。Zaidenwerg 说，所有的“开发者都应该做运营”，而不是雇佣一个专门的角色。

这意味着让开发人员更好地了解 DevOps KPIs，如部署频率、部署时间、变更失败率、检测时间和可用性，我们将在下面详细解释。

### 2.开发人员最了解他们的应用程序代码

开发人员比任何人都了解他们的代码。在不熟悉的环境中诊断错误是很棘手的，只有程序员才真正理解自己代码的来龙去脉。如果用 DevOps 技能武装起来，他们可以比外部眼睛更有效地解决问题。

Zaidenwerg 是一名 DevOps 系统管理员，他描述了“必须监控一个不是自己编写或构建的应用程序的可用性是什么感觉。”很难；他解释说，同时诊断你不熟悉的多个应用程序也是相当具有挑战性的。

如果您不能快速将错误与应用程序流程联系起来，可能会损害可用性和 SLA。相反，鼓励开发人员诊断问题可以减少补救时间，因为他们有更多的背景，Zaidenwerg 指出。

### 3.开发商看到周围的环境

除了了解单个应用程序如何工作，开发人员还对周围环境有独特的视角——他们认识到构成微服务系统的依赖关系和互连的复杂网络。

要了解发生了什么，你真的需要知道如何编码。“DevOps 工程师需要成为软件开发人员，”Zaidenwerg 说。将 DevOps 任务交给没有编码知识的管理员会严重限制他们的能力。他们将无法钻研代码来解决实际问题或自信地提出拉取请求。

### 4.分离角色使运营处于孤岛状态

将 DevOps 的职责移交给一个集中的团队违背了一个基本原则:分离部门。“DevOps 的发明就是为了缩小这个差距，”Zaidenwerg 说。因此，单独发挥作用似乎违背了这一方法的基本原则之一。

诺华公司的首席开发人员 Tom Marsh 在 Quora 上说:“DevOps 实践的要点是，开发团队有权控制他们自己的操作，而不是你有知道如何编写操作脚本的专家。”。

### 5.一个 DevOps 工程师不能对抗洪水

就目前情况来看，没有足够的 DevOps 工程师。根据 Quora 上的回答，DevOps 工程师与开发人员的比例往往在 1:10 到 1:12 之间。Zaidenwerg 说他见过更糟的情况——从 5 点 50 分到 6 点 120 分。相反，他说，“零到零是最佳数字。”

一个 DevOps 工程师对抗潮水般的请求和应用程序部署管道是不可持续的。"你能想象已经编写的特性和代码的数量吗？"Zaidenwerg 问道。“他们需要扑灭的火灾数量巨大。”当那个单身的 DevOps 工程师去度假时会发生什么？为了避免延迟，他鼓励开发人员从头到尾拥有他们的 CI/CD 过程。

### 6.当 DevOps 工程师自动化他们的工作时会发生什么？

DevOps 和 [SRE 实践](https://devops.com/how-the-sre-role-is-evolving/)的另一个信条是引入自动化——最终目标是让他们的工作自动化。但是，如果这真的发生了，DevOps 工程师会怎么样呢？他们会坐享其成，在随后的几年里享受前期工作的红利吗？让程序员将[自动化](https://devops.com/ai-automation-wont-replace-jobs-it-will-replace-processes/)引入他们的工作流程会更有意义。

### 7.K8s +其他 DevOps 工具增强了开发人员的能力

功能强大的 DevOps 工具如 Kubernetes、ELK stack 和云基础设施解决方案，现在让程序员自己承担更多责任变得更加现实。Zaidenwerg 说，将监控、负载平衡或[存储](https://devops.com/survey-the-state-of-database-devops/)的“即服务”解决方案民主化，可以让每个人“分担开发运维工作的负担”。

许多组织已经开始将 Kubernetes 视为真相的来源。Zaidenwerg 全心全意地鼓励这种做法。“一切都应该在 K8s 上管理，从 K8s 开始，”他说。"试着把 K8s 想象成你的最低层."他补充说，围绕 K8s 标准化微服务有助于保持对多云平台的不可知论。

## 怎么去？跟踪 DevOps KPIs

因此，如果我们不把 DevOps 任务委托给 DevOps 工程师，我们如何鼓励开发人员接受 DevOps 原则呢？对 Zaidenwerg 来说，答案归结于我们如何定义成功。我们需要一种使用 DevOps [关键绩效指标](https://phoenixnap.com/blog/devops-metrics-kpis)(KPI)来衡量开发人员工作的文化，而不是使用传统的开发产出指标，如生产率、scrum 或完成的任务数量来衡量成功。

有许多 DevOps KPIs 需要考虑。有些可能是:

*   **部署频率**:你多久提交一次？
*   **部署时间**:从提交到实际环境需要多长时间？
*   自动化:构建过程的自动化程度如何？
*   **可用性**:该计划是否在全球范围内可用并具有高性能？延迟和正常运行时间是多少？
*   **监控**:您是否主动监控并解决了实际环境中的问题？
*   **优化**:应用是否划算？能更有效率吗？是否有闲置资源在运行？
*   **策略作为代码**:安全策略作为代码是自动化的吗？“安全性需要用代码来编写……如果手动完成，你就完了，”Zaidenberg 直截了当地说。
*   DRP :应用程序有弹性吗？什么是灾难恢复计划(DRP)？

在上述关键绩效指标中，优化最为突出，因为[不断上升的云成本](https://devops.com/how-to-respond-to-rising-cloud-costs/)是目前一直存在的障碍。Zaidenwerg 鼓励开发人员跟踪云环境费用。他说，当你有一个每日支出图表时，你可以利用这种可见性来“创建支持系统，以自动减少闲置资源”。其他人将此描述为 [BizDevSecOps](https://devops.com/seeing-the-bigger-picture-with-bizdevsecops/) 。

## 鼓励 DevOps 文化

我们厚脸皮的标题并不是要贬低那些头衔中带有 DevOps 的人，相反，雇佣一个有这种罕见经历的人可以极大地帮助组织引入 DevOps 原则。他们可以作为帮助培养这种文化的典范。

除了跟踪 KPI，Zaidenwerg 还推荐了一种自上而下的方法来传播 DevOps 原则。该方法必须由 CTO、架构师或技术负责人批准和领导，具体取决于公司的规模。DevOps 文化对于 SaaS 或 IaaS 公司尤其重要，它们的架构必须不仅仅是防错的——“它必须每次都成功，”Zaidenwerg 说。

传播现代云最佳实践对于构建 DevOps 文化也至关重要。当您从硬件过渡到云时，您经常会遇到类似的技术(负载平衡器、防火墙、可用性区域等。)，但每个功能都有不同的名称和实现过程，云服务提供商(CSP)之间的细微差别使其变得复杂。Zaidenwerg 说，所有这些都需要“将内部最佳实践转化为云最佳实践”。

另一个重要的问题是，DevOps 并没有真正在大学计算机科学课程中讲授。展望未来，Zaidenwerg(和我自己)希望在教育系统中看到更多的重视。尽早灌输 DevOps 思维模式可以让开发人员为更真实的环境做好准备，包括部署、可用性和代码安全性。

## 这是谁的房子？

Zaidenwerg 将软件应用程序比作某人的房子。房主知道每一个细节——餐具在哪里，如何操作淋浴喷头，哪天收垃圾，等等。邀请一个陌生人进新房子，他们会不知所措。

同样，邀请外部运营部门来管理国外基础设施也会让他们不知所措。“如果你让别人处理基础设施，那么你有两个政党，没有人完全意识到，”Zaidenwerg 说。

总结一下——devo PS 是一种文化，而不是一个角色。将开发运维视为一种角色，而不是一种实践，可能会违背目的，并导致孤立的、行动缓慢的团队。相反，跟踪 KPI 和用 DevOps 工具授权开发人员可以帮助他们保持秩序。

当然，这是一个不断发展的实践，每个组织对于实现 DevOps 意味着什么都有自己的理解。现在，你还认为它配得上这个头衔吗？请在下面的评论中告诉我们！
