# 调查发现最大的发展障碍是组织

> 原文：<https://devops.com/survey-finds-biggest-devops-obstacles-are-organizational/>

测试自动化平台提供商 mabl 本周公布了一项针对 600 名 IT 专业人员的调查,发现采用 DevOps 实践的最大障碍与组织问题而不是任何技术更密切相关。

该调查确定非技术问题是 DevOps 的主要抑制因素(82%)。只有 11%的受访者表示他们的组织已经完全实施了 DevOps 实践，而另外 24%的受访者表示他们正在努力实现这一目标。

采用的两大障碍是缓慢的过程和适应速度(29%)，其次是预算和资金(21%)。只有 18%的人认为技术限制是一个问题。

近一半的受访者(46%)表示他们已经能够实施持续集成，而超过三分之一的受访者(36%)表示他们也已经能够实现[持续交付](https://devops.com/category/blogs/continuous-delivery/)。超过四分之一(27%)的受访者表示他们也实现了持续部署。然而，只有三分之一的受访者表示他们每周至少部署一次新代码。几乎一半的人仍然每月部署不到一次。每天或每天多次部署新代码的团队数量(15%)大致相当于每季度部署代码的团队数量(16%)。

使用最广泛的 DevOps 工具是 GitHub (34%)和 Jenkins (33%)，其次是 GitLab (21%)、Azure Pipelines (20%)和 Bitbucket (19%)。

在测试方面，最广泛的自动化测试是用户界面(UI)/功能和单元测试，分别占 42%，其次是回归测试，分别占 41%和 35%。使用最广泛的功能测试工具是 Selenium (41%)，其次是 Cypress(14%)。测试应用程序编程接口(API)最广泛使用的工具是 Postman，其次是 SoapUI (21%)和自制工具(17%)。

mabl 的联合创始人 Dan Belcher 表示，虽然很明显大多数组织在完全采用 DevOps 最佳实践之前还有很长的路要走，但现在更多的是程度问题，而不是是否。随着组织意识到他们今天对软件的依赖程度，对质量和确保及时交付的关注将会增加。贝尔彻指出，如今软件交付的风险从未如此之高。

然而，调查清楚地表明了组织对变革的抵制程度。几十年来，一些组织一直以同样的方式构建软件。唯一比说服 IT 和业务领导存在根本的变革需求更困难的事情是将变革一路推进到单个开发人员和 IT 运营团队。

然而，竞争压力将不可避免地迫使发行。如果竞争对手正在交付更好的应用程序体验，在数字业务流程的环境中创造业务优势，组织将不得不适应。否则，随着时间的推移，他们的命运将是逐渐消失或被收购。将 DevOps 与存在的威胁等同起来可能是一个挑战，但除非建立这种联系，否则不采用 DevOps 最佳实践的组织将滑向无关紧要只是时间问题。
