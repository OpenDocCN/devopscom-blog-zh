# 了解一点人工智能

> 原文：<https://devops.com/learn-a-bit-about-ai/>

偶尔，一种趋势变得足够广泛，以至于我觉得有必要提供职业建议。今年会很简单:对 AI/ML 有足够的了解，让它变得危险。我们已经过了没有起码的基本理解是一个好主意的阶段。环顾数据中心，从开发的最早阶段到生产网络性能评估，AI 都在使用。在你环境的某个地方，你正在使用 AI 和 ML。更重要的是，在未来你可能采访的地方，他们正在使用人工智能和人工智能。

我并不是建议你成为一名人工智能/人工智能专家——尽管你当然可以；我有一个亲戚是数据科学家，这就是他所做的一切——设计和调整算法。他几乎每天都在学习新的东西，所以如果你喜欢(像我一样)，这是一个选择。但是对于我们这些做 DevOps 或者 SRE 或者甚至常规旧操作的人，我们只需要理解一点点。

了解像 [Jupyter](https://jupyter.org/) 这样的工具是什么，你所在领域最常用的 ML 算法是什么。然后，了解 AI 和 ML 如何帮助你正在做的事情。正如更多地了解你选择的语言的编译器/解释器有助于你解决更多奇怪的问题一样，人工智能也将面临同样的问题。想想那些了解路由本质的人是我们想要解决路由问题的人。同样的场景。

展望未来，AIOps、NetOps、DevSecOps……都打算增加卸载到相关 AI 上的工作量。在很多情况下，这很好。我们有足够的事情要做；如果你能翻阅这些日志，并给我相关的摘要信息，去吧。但事情并没有就此结束——一些人工智能引擎正在做出决定并实施改变。在网络和应用程序安全方面，相当多的供应商会告诉你，没有时间等待人类来阻止攻击，他们鼓励他们的客户打开自动响应。这是 AI/ML 众多应用中的第一个。我们将会看到越来越多的人工智能/人工智能工具被用于完全自动化。我们都做着困难的工作，有着复杂的——有时令人困惑的——大量的输入。但是我们使用模式识别和经验来做出决定，做对我们组织最有利的事情。我们的软件已经达到了这样一个程度，有时是 T1，它可以为我们完成更简单的任务，这样我们就可以专注于更复杂的任务。当 AI/ML 失败时——它会失败，因为模式识别和历史趋势并不总是正常使用的指示——了解幕后发生的事情可以帮助您确定它是否以错误的方式做了正确的事情，或者只是做了错误的事情。它可以帮助你快速恢复。我不能代表每个组织，但我从未见过一个组织哀叹道:“真希望停机再持续几个小时！”

继续摇摆。在遥远的未来的某个地方，人工智能也许能像你一样做好你的工作。但那不是今天。即使它能处理一点，它肯定不会像你一样动摇你的全部责任。谢谢你，所有不知道你让他们活着的用户。
