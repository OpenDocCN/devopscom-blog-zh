# GitLab 研究显示开发团队中的安全问题

> 原文：<https://devops.com/gitlab-research-shows-security-concerns-in-devops-teams/>

违规威胁是许多 DevOps 团队的首要考虑，尽管安全性继续左移，但它似乎是许多决策者选择 DevOps 平台或其他技术的驱动力。

这些是 GitLabs [调查](https://about.gitlab.com/developer-survey/) 的结果之一，该调查还发现获得正确的安全性是 DevOps 团队的头号挑战。

## 把事情做好

受访者表示，他们选择 DevOps 的主要原因是更好的代码质量、开发人员生产力和运营效率，37%的受访者提到了这一点，紧随其后的是更好的安全性/更安全的应用程序。

调查还显示，53%的开发人员表示他们对其组织的安全负全责，比 2021 年增加了 14%。

Salt Security 的副总裁 Michelle McLean 说，对于组织来说，仅仅依靠开发团队来保证安全是有风险的。

“安全和开发团队需要[协作](https://devops.com/why-devops-teams-need-security-engineers/)并一起工作——在应用生命周期的每个阶段——而不仅仅是开发阶段,”她解释道。

例如，安全团队必须将运行时洞察带回开发中，以便它们可以应用于教育和培训。

“选择一个内置安全功能或者可以轻松与安全平台集成以促进安全和开发运维团队协作的开发运维平台至关重要，”她说。否则，组织将面临推出不安全软件或在软件供应链中引入其他风险的风险

Synopsys 网络安全研究中心的首席安全策略师蒂姆·麦基(Tim Mackey)补充说，云原生社区中一个过时的咒语认为，安全不是一个高优先级的问题，因为适当的 DevOps 工作流意味着增量更改较小且易于部署。

他说 Log4j 和 Log4Shell 的经历对这一理念提出了质疑，因为人们认识到安全问题可能并且确实会发生在一个版本或功能的带外，并且持续监控外部威胁是正常运行的安全流程的一部分。

“通过对新威胁实施监控，无论其来源或性质如何，云原生开发快速发布测试补丁的能力本身就成为了一项安全功能，”他解释道。

## 了解管道

McLean 指出，虽然对 DevSecOps 的安全性有了越来越多的了解，但这还不够。

“例如，在原料药的情况下，尤其如此。因为 API 不仅仅是简单的代码，你必须看到它们被用来发现逻辑缺陷，”她说。“您需要能够在运行时监控它们，以便发现异常并找到 API 可能暴露关键数据的区域。尽管开发人员编写 API，但他们的测试往往达不到安全性要求。”

为了准确地了解 API 发生了什么，开发人员需要练习 API——即使是在生产前。

McLean 表示，这是一个安全与开发协作和集成至关重要的领域——API 支撑着现代经济的重要组成部分，安全与开发团队之间的脱节可能会产生直接的经济影响。

SaaS 安全和运营分析公司 Netenrich 的首席威胁猎人 John Bambenek 补充说，在新平台发布和意识到这些平台的安全功能需要实施之间总是存在延迟。

“有了云原生和无服务器技术，这种延迟是我职业生涯中记忆中最短的，”他说。“我非常喜欢这样一个世界，在这里，每个人至少都明白安全的必要性，而不是花时间去接受安全的重要性。”

他说，如果你能够在开发过程的早期以无缝和高效的方式管理和实现安全性，那么解决问题要比在代码发布后解决问题容易得多，也便宜得多——而且这还不会增加违规或责任成本。

“你可以在 dev 或 prod 中修复它，但你迟早要修复它，”他说。

McLean 指出，许多组织的 DevOps 实践仍在不断成熟，更不用说他们的 DevSecOps 计划了。

从她的角度来看，在编写一行代码之前，教育开发人员从安全角度对他们的期望是关键。

“为了帮助提高合规性，我们已经开始看到组织采取措施将安全工具引入他们的管道流程，并将测试添加到构建流程中，”她说。

然后，他们会添加一个步骤来触发对不符合的构建的警告，但不会阻止构建。然后他们会添加一个步骤，要求开发人员签字确认他们已经看到了不符合的构建的安全警告，但仍然不会中断构建。

“在极少数拥有高度成熟的 DevSecOps 流程的组织中，他们会因为不符合而导致构建失败，”她补充道。

McLean 说，组织需要了解什么是现实的，考虑到他们的成熟度和风险承受能力，并实施匹配的控制。

他们还需要重新评估这些控制措施，以寻找提高对安全策略和其他法规遵从性的机会。

总的来说，在软件开发生命周期(在开发、测试和部署阶段)中通过合规自动化继续教育进一步帮助强化最佳实践，同时确保问题在成为风险或技术债务之前被发现。

Mackey 说，由于 DevOps 平台涉及到为您的业务提供动力的软件，因此在选择任何 DevOps 平台时，平台本身的安全性及其支持的能力都应该是必备的。

“实际上，任何关于新软件的决定都应该基于它如何提高企业当前的安全能力，”他说。
