# AWS 推出一系列自动化工具来简化开发工作

> 原文:[https://devo PS . com/AWS-un furls-bevy-of-automation-tools-to-streamline-devo PS/](https://devops.com/aws-unfurls-bevy-of-automation-tools-to-streamline-devops/)

在 [AWS re:Invent](https://reinvent.awsevents.com/) 大会上，亚马逊网络服务( [AWS](https://devops.com/?s=AWS) )向其产品组合添加了一系列工具，旨在加快应用开发的步伐，同时简化 DevOps 流程。

本周发布的 AWS 开发工具包括 [AWS Amplify Studio](https://press.aboutamazon.com/news-releases/news-release-details/aws-announces-aws-amplify-studio) ，这是一个可视化开发环境，允许开发人员用最少的代码创建 web 应用程序用户界面(ui)。

AWS 开发工具产品负责人 Ken Exner 表示，作为现有 AWS Amplify Studio 工具的扩展，这一最新功能允许开发人员使用组件库在更高的抽象层次上定制应用程序 ui，同时使他们能够下降到更低的编码层次，以便在需要时进一步定制他们的应用程序。UI 设计完成后，AWS Amplify Studio 自动为开发者生成关联的 JavaScript 或 [TypeScript](https://devops.com/is-typescript-the-new-javascript/) 代码。

他说，总的来说，AWS 致力于通过提供自动化推理工具来提高开发人员的生产力，这些工具可以自动化流程，而不会将开发人员锁定在一个抽象层中，最终可能会形成一堵墙，阻止他们满足定制需求。

AWS 本周推出的其他开发者工具包括对亚马逊 CodeGuru Reviewer 的[更新，该更新添加了一个秘密检测器，该检测器使用机器学习算法来识别可以使用 AWS Secrets Manager 保护的硬编码秘密；经过改进的 Amazon Inspector 漏洞管理工具，提供自动评估扫描、自动资源发现、对基于容器的工作负载的支持、改进的风险评分、与 Amazon EventBridge 和 AWS Security Hub 的集成以及独立 Amazon Inspector 扫描代理的移除。](https://aws.amazon.com/blogs/aws/codeguru-reviewer-secrets-detector-identify-hardcoded-secrets/)

AWS 还透露，它已经[通过添加自动同步功能，使私有和公共注册保持一致，消除了从亚马逊弹性容器注册(ECR)库](https://aws.amazon.com/blogs/aws/announcing-pull-through-cache-repositories-for-amazon-elastic-container-registry/)手动获取图像的需要。

在 [IT 运营工具](https://press.aboutamazon.com/news-releases/news-release-details/aws-announces-three-new-database-capabilities)方面，AWS 在其亚马逊关系数据库服务组合中添加了一项托管数据库服务，允许 IT 团队使用 RDS 定制选项定制底层数据库和操作系统，这是亚马逊 DynamoDB 的一个新表类，旨在降低不经常访问的数据的存储成本，以及一项名为 Amazon DevOps Guru 的 RDS 服务，它扩展了现有的机器学习能力，以自动检测和诊断性能瓶颈和运营数据库问题。

AWS 负责关系数据库的副总裁杰夫·卡特表示，随着数据库越来越多地由 DevOps 团队而不是传统的数据库管理员(DBA)管理，云服务提供商正在通过尽可能多地自动化手动任务来减少摩擦。他补充说，在 IT 团队部署更广泛的专门构建的数据库，为不同类别的应用程序进行优化的时候，这一点至关重要。卡特说，总体目标是更多地依靠自动化来减少 IT 环境中的摩擦。

总的来说，有两类 AWS 客户:第一类是开发团队，他们越来越依赖 AWS 提供的托管服务来自动化各种各样的开发运维流程。第二种倾向是企业 IT 组织，他们更喜欢自己管理云服务。

DevOps.com 母公司的分公司 Techstrong Research 的负责人 Mitch Ashley 指出，在许多情况下，企业 IT 团队正在管理跨越多个云的 DevOps 工作流。因此，他们倾向于选择可以应用于多个云和内部 IT 环境的工具和平台。这些组织越依赖云服务提供商提供的托管服务，就越难以在多个 it 环境中一致地管理开发运维工作流。

现在说 It 组织未来将在多大程度上依赖托管服务还为时过早。然而，当越来越多的组织使用比以往更多的 it 平台时，很明显，随着 IT 环境中引入每个新平台，DevOps 管理挑战也会增加。
