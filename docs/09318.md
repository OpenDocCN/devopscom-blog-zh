# 这不仅仅是一个测试:DevOps 和自动化的需求

> 原文:[https://devo PS . com/this-is-not-just-a-test-devo PS-and-the-need-to-automate/](https://devops.com/this-is-not-just-a-test-devops-and-the-need-to-automate/)

创造软件与过程和产品一样重要。近年来，由于更有效的开发和质量控制，我们看到创新的速度达到了新的高度。DevOps 通过在统一的、自我改进的团队中结合软件和操作，进一步加速了 开发 。随着自动化处理机械任务， 开发人员 现在可以处理更困难的问题，从而带来更高的软件质量和更快的开发速度。 市场对速度的需求至关重要；一两个月可以决定一个产品发布的成败。话又说回来，如果一个产品在匆忙推向市场的过程中没有经过彻底的测试，缺陷可能会从 溜到 ，一个组织的 收入 和声誉 可能会受到严重打击。 在美国，企业运营系统的软件故障总共造成了几乎[1.6 万亿美元 2019](https://www.it-cisq.org/pdf/CPSQ-2020-report.pdf) 。 问题 是 它可以成为一个瓶颈并在发布周期 中创建重大 。 事实上， A [最近对 4300 名开发者](https://www.globenewswire.com/news-release/2021/05/04/2222642/0/en/GitLab-s-Fifth-Annual-Global-DevSecOps-Survey-Reveals-2020-Was-Catalyst-for-DevOps-Tool-Adoption.html) 的调查显示it是发布延迟的首要原因。

过去，瀑布方法很常见 。 测试 通常 发生在代码编写之后； 由与原开发人员没有互动的测试人员进行。结果，在开发过程中可能需要几分钟才能纠正的错误需要几天甚至几周才能修复。更糟糕的是，完成的软件的复杂性和它的许多依赖性意味着修复一个本地化的错误可能会引入额外的问题，这些问题需要在另一轮测试后修复，从而进一步推迟的发布。

但是即使在现代的 DevOps 环境中，测试人员和开发人员仍然说着完全不同的语言。总的来说，测试人员不编码，软件开发人员也不是专业的测试人员。两个团队 通常通过多个日志、事件查看器和其他操作系统工具进行对话。它会产生很大的摩擦并减慢这个过程。

### 左移

DevOps 代表了对瀑布方法的一大改进，但即使如此，它也远不是一成不变的过程。它也在发展，现在，[左移测试](https://devops.com/when-shift-right-is-shift-left/)已经成为焦点。

在这种方法中， 的指导思想是考早而常， 初起发展 循环不断每隔因此，当开发人员对相关代码记忆犹新时，就能及早发现错误，从而更容易修复。最终，企业 能够 更快更频繁地发布更高质量的 软件。

### 环境测试

谈到开发运维流程，并不总是清楚在哪里以及如何实现流程自动化 。下面的 着眼于 如何将 整合到基于连续交付(CD)管道中四种不同环境的 DevOps 中。

se环境包括本地开发(通常是程序员的个人设备)、持续集成(CI)或开发服务器、用于用户界面(UI)测试的测试服务器以及生产环境本身。每个阶段都有要求，必须满足这些要求代码才能前进。

然而，无论何时考虑任何新的软件特性或功能， 开发人员和 QA 团队 都应该以回答这个问题开始:“如何测试？”

### 移动通过各阶段

一旦代码被创建，它 必须能够 被无错误地构建，通过 ing 所有单元测试和本地验证。

这个阶段特别重要，因为开发环境与生产的需求相呼应；如果它在这里有效，它也可能对客户有效。

在这个阶段，运行早期回归和开发人员验证测试确保构建就绪，同时提供关于环境可靠性的反馈。将单独的测试用例作为多个计划的一部分是很有用的。这使得开发人员能够测试软件的次要部分，而无需经历所有产品特性的回归测试。这些测试应该作为同一流程的一部分自动进行和触发。

开发人员应该有能力对他们自己的代码进行基本测试。毕竟，缺陷越早发现，就越容易根除。 正如测试员不是编码专家一样， 开发人员 专家测试员 也不是。为了左移并有效地集成开发生命周期所有阶段的测试，它必须是自动化的。这将允许开发人员更早、更频繁地测试大量代码 。

一旦通过这个阶段，软件就可以进入测试环境。

这里是进行各种产品测试的地方，包括针对[功能](https://devops.com/applause-report-surfaces-functional-testing-issues/)、、负载、和性能的测试。通过这一阶段需要完成所有自动化功能 UI 测试、视觉外观验证、服务器日志检查和性能评估。

这是大多数回归测试进行的地方，自动化对于预定测试的触发至关重要。

接下来，产品进入生产阶段。

测试仍然在这个阶段进行，以查明任何剩余的错误。然后，针对生产环境运行自动化测试，以密切反映与真实最终用户的交互。在这一点上，不太可能有大的问题，但是，如果有，在早期阶段运行测试很可能是罪魁祸首。

在整个过程中，理想情况下，测试人员将使用可视化的、无代码的工具来创建这些自动化测试。通过这种方式，测试人员能够 更快速、更直观地实现测试自动化 ，而无需利用编程资源 。此外，自动化过程中所采取的行动的集成文档将与测试结果的可视化文档交互连接。结果是测试人员和开发人员获得了一种共享的语言，既可以构建测试自动化，也可以解决它所识别的问题。

### 这不仅仅是一次测试

有效的、可扩展的测试对于交付可靠的、高质量的软件至关重要，这些软件是今天的企业 日常运营 所依赖的。 这就是为什么 DevOps 和开发团队需要在软件开发的各个阶段集成测试。向左变速 向上变速质量更好的 软件 ，这提供了显著的 市场优势。毕竟这不仅仅是关于软件质量的；它可以显示哪些公司为满足市场需求做了最好的准备。
