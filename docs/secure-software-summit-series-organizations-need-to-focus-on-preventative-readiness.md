## 安全软件峰会

2022 年初， ShiftLeft 主办了[Secure Software Summit](https://go.shiftleft.io/secure-software-summit-2022-replay)为软件开发界的行业专家和从业者提供一个 平台，共同探讨安全编码和开发实践的最新方法和突破——更早更好地保护代码已经成为

此次活动的一些主要收获是:

### SBOMs 在你的未来

在不久的将来，组织将需要更好地说明其应用程序的所有软件和组件，最有可能的方式是通过软件材料清单(SBOM)。美国政府已经规定了准系统业务模型，我们很快就会看到私营企业也开始规定它们，作为采购流程和审计的一部分。这将增强透明度，并以一种迄今为止并不常见的方式自动发现所有依赖项和组件。软件组合分析(SCA)将使这个过程变得更容易，并将成为构建过程和应用程序开发生命周期的标准部分。

### 保护开源至关重要

面对 Log4Shell 和其他开源软件(OSS)供应链攻击，应用安全团队必须学会如何更智能地更新和保护最关键的 OSS 组件和基础设施。普通的 AppSec 团队需要筛选大量的漏洞和建议的安全修复措施——远远超过他们可能修复的数量。在过去的四年中，我们每年都看到创纪录数量的新漏洞披露。根据漏洞是否会对组织自身的应用程序和基础设施造成重大影响来确定适当的优先级，这在构成现代应用程序的大量 OSS 依赖关系中是必不可少的。

### 数字威胁影响着现实世界

对于医疗保健等消耗大量软件的非技术垂直行业的许多组织来说，应用程序安全性正处于危机关头——软件漏洞确实会危及生命和我们的经济。研究人员已经将死亡率的增加与医院在更高压力和更大容量下运行联系起来。当勒索软件或其他攻击影响医疗保健机构时，其效果就像一场大规模的疫情淹没了急诊室:医生无法使用系统或设备，护理是定量供应的，病人被拒之门外，一切都变得更加耗时。最终结果是更多的死亡。保护这些系统免受攻击成了生死攸关的事情——毫不夸张地说。

### 文化变革必须放在首位

组织将部署新的安全架构，如零信任，但只有当 AppSec 和开发团队改变安全文化时，这些尝试才会成功。组织必须认识到，新常态是一种不断更新信任的状态。这将具有挑战性；信任的不断更新需要全新的基础设施和新的心态，这对人类来说可能很难。这意味着在更多的地方实现多因素身份验证，并取消对系统拥有全面访问权限的管理帐户等便利条件；它需要实现最小特权实践。软件开发需要将这种不断更新的信任融入到工作流和工具中，使之成为新的常态。

### 从被动转向主动

AppSec 和开发团队必须改变在违规或事故期间倾向于调动大部分资源的被动方法，而强调更主动的方法，如更好的软件安全分析和安全混乱工程。在未知成为问题之前了解未知是关键。网飞是混沌工程的先驱，他认识到频繁而持续地对系统施加压力，观察它们在不利条件下的表现，往往会产生令人惊讶的洞见，从而提高安全性(以及更广泛的弹性)。如上所述，创建更好的方法来区分漏洞的优先级并专注于实际可攻击的漏洞，可以将任务从无望的烧毁转变为可管理的有针对性的主动战术练习。
