# DevOps 和混合云:快车道上的生活？

> 原文：<https://devops.com/devops-and-hybrid-cloud-life-in-the-fast-lane/>

如果你想走得更快就尖叫吧！当谈到混合云和开发运维的双重部署时，大多数组织都会承认他们渴望更快一点。但是，当他们在没有考虑所有必要因素的情况下加速进行这一重大转变时，开始时如此热情的事情可能会变成沮丧(以及另一种尖叫的开始)。

好处显而易见。由 DevOps 支持的混合云是一种高效的变革工具。一个强大的 DevOps 平台可能是开发长期混合云战略的成败因素。总之，他们是变革的力量，如果组织还没有实施开发运维与混合云，他们应该问自己为什么不这样做。

的确，对于启用云的组织来说，迁移到混合云是一个不同于云原生组织的旅程。对于启用云的企业来说，这一过程更加艰难，尤其是对于大型企业。更高的发布频率要求业务线、开发和 IT 运营之间比传统上更紧密的协调和协作，这反过来又推动了对增强协作、自动化和信息透明度的需求。

尽管这种转变可能会更加艰难，但它必须被视为不可避免。组织可以再等一年，也许更久一点，但是技术和业务的转变即将发生，所以为什么不开始呢？在两篇文章的第一篇中，我将介绍需要做出哪些改变，以便组织可以开始在混合云和 DevOps 快车道上生活。

## 一种文化现象

正如这些页面已经详细介绍的那样，如果组织认为他们需要担心的只是工具和招聘，那么由 DevOps 支持的混合云迁移注定会失败。不是的。这两种转变的背后是一场彻底的文化变革。DevOps 关乎敏捷性、信任和自主性，混合云也是如此。

然而，最重要的是，DevOps 和[混合云](https://devops.com/?s=hybrid+cloud)致力于增强组织的开发人员体验(DevX ),这不是简单的票务工具或采用 GitOps 方法可以实现的。DevX 的意思是自主的，自由的运动。它意味着授权、用户体验和自助服务。这些变化只有在商业文化成功改变的情况下才会发生，而不是一种工具。这是一个重要的起点，许多组织仍然难以理解。

## 人的问题

人是另一个老生常谈的话题，但仍然是一个重大挑战。理论上，成功实现 DevOps 所需的文化转变可以从一个人开始。但要取得成功，实际上需要长期的改变，因为根深蒂固的信仰被瓦解，部落忠诚被打破。

例如，许多人认为实现 DevOps 需要雇佣新人。这种信念包含两个谬误:第一，组织将能够雇佣他们需要的团队；第二，DevOps 只是一个职位头衔。正如我们所知，一个好的 DevOps 团队不是雇佣来的，它是通过开发和运营的融合发展起来的。dev 开发和 ops 运行的老筒仓是不太开明时代的遗迹。

如今，it 需要学习如何构建自动化(通常是开源的)，同时管理内部、私有和公共云基础架构。DevOps 是一种寻求帮助他人的哲学，随之而来的是对合作、鼓励、说服和其他软技能的需求。

## 安全和隐私并不简单

迁移到混合云意味着在安全性和隐私方面需要更加小心。一个主要的问题是，安全团队不再能够高高在上。随着控制权从服务器机房转移到云，从高度孤立的运营工程师转移到整个团队，安全策略也需要改变。有能力的开发人员*将*做出正确的决策，但是他们需要正确的自下而上的工具，将他们的决策从理论变为现实，并保护那些没有意识到的人。

这些必须成为一种催化剂，促使组织坐起来，从长计议云、容器和微服务时代的安全到底意味着什么。如果同时迁移到 DevOps 和混合云是催化剂，那就这样吧。正如 DevOps 正在彻底改变团队的工作方式一样，DevSecOps 也将彻底改变他们看待安全性的方式。

安全性在基础设施的各个阶段都有所体现:当云组织被创建和结构化时；当集成保险库以确保密钥密码安全时，甚至当启动触发专用安全软件的管道时。这不是一个详尽的列表，但它说明了组织需要如何开始有条不紊地、一步一步地审视其混合云中的安全性。

## 吉托普斯呢？

Gartner 最近发现，普通公司使用 28 种基础设施工具。这太多了，一个人无法处理和理解。即使是专业的 DevOps 工程师也不可能是所有云提供商、OpenStack 和VMware 的专家。其他研究表明，开发人员通常只使用 5%的 Git 存储库，或者平均使用 5 个存储库。

许多公司犯的错误是认为突然转向使用 API 和 GitOps 方法就足以应对向混合云的转变。它不是，而且实际上可能对 DevX 不利。为什么？差距太大了——这就像在马拉松比赛的前一天进行每周一次的长跑。你可能会开始，但你会严重损害你实现目标的能力。即使是 DevOps 工程师也很难维护单一的事实来源和维护良好的可重用存储库。

解决办法是做长期规划。相信一两个 DevOps 工程师可以成为混合云方法的救星是不现实的。有太多的工具、最佳实践和选择 Ansible、Terraform、Kubernetes 和 Concourse CI/CD，仅举几个例子——由少数几个开发人员管理，不管他们有多有才华。

这就是为什么掌握混合云的真正方法在于能够让整个技术团队朝着同一个方向前进，协作、提升技能并利用他们的优势来支持组成整体的各个部分。这听起来可能像是白日梦，但确实有公司在说空话，在做实事。
