# Splunk 向 CNCF 捐赠 eBPF 遥测数据采集器

> 原文：<https://devops.com/splunk-donates-ebpf-telemetry-data-collector-to-cncf/>

Splunk Inc】在 KubeCon + CloudNativeCon 北美会议期间宣布，[向云本地计算基金会(CNCF)运营的 OpenTelemetry 项目捐赠了一个数据收集器](https://www.businesswire.com/news/home/20211011005068/en/Splunk-Advances-Future-of-Observability-with-OpenTelemetry-Donations)。

Splunk 的数据收集器作为 Linux 操作系统内核中的沙盒级程序运行，并利用扩展的 Berkeley 数据包过滤器(eBPF)技术来简化网络遥测数据的收集。

OpenTelemetry 计划本身涵盖了一系列用于仪器应用的开源工具、应用编程接口(API)和软件开发工具包(SDK)。此前，Splunk 为 OpenTelemetry 项目贡献了 SignalFx 智能代理和智能网关以及超过 64，000 条代码。

Splunk 的产品管理总监 Morgan McLean 表示，虽然 open telemetry 还处于正式测试阶段，但该项目的一些元素比其他元素更成熟。他说，例如，用于捕获度量和跟踪的工具已经在使用，而另一套用于捕获日志数据的工具将在明年的某个时候准备好。

Splunk 提供的数据收集器是首批在内核级别运行以收集数据的数据收集器之一，McLean 补充道。这种方法使得默认捕获某些类型的数据成为可能，而不是要求开发人员总是使用代理来检测他们的应用程序，以实现可观察性平台。它还将有助于网络运营和开发运维流程的最终融合。

McLean 说，展望未来，DevOps 团队应该期望采用在 eBPF 和应用程序级别运行的开源数据收集器的组合。最终的目标是通过使收集数据变得更加简单，在默认情况下检测每个应用程序。如今，大多数 DevOps 团队混合使用专有和开源代理软件来收集数据，这些数据必须进行部署，然后与他们构建的所有应用程序集成。考虑到所需的成本和工作量，实际上被检测的应用程序的百分比相当低，这并不奇怪。

然而，随着代理软件变得越来越容易访问，在未来的几年中，应用程序的百分比应该会大大增加。对于依赖这些数据来优化应用程序性能的开发运维团队来说，这种能力将有助于提高 IT 环境中的整体可观察性。

可观察性，以某种形式，一直是任何 DevOps 最佳实践的核心原则。最初，DevOps 团队将持续监控作为主动管理应用环境的最有效方式。然而，发现问题的根本原因仍然需要几天甚至几周的时间。

监控侧重于预定义的指标，以确定特定平台或应用程序何时在预期范围内运行。例如，被跟踪的度量通常集中在资源利用上。可观察性平台将指标、日志和跟踪(一种专门的日志记录形式)结合起来，以一种更简单的方式对应用程序进行检测，从而在不依赖于专注于特定流程或功能的一组有限的预定义指标的情况下解决问题。

然后，这些可观察性平台使关联事件成为可能，以便更容易识别指示问题根本原因的异常行为。有了这些见解，it 团队更快地解决问题就变得简单多了。

尚不清楚 OpenTelemetry 工具何时会得到更广泛的应用，但随着更多收集数据的工具变得可用，对 DevOps 的影响将是深远的。
