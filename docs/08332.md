# 云安全需要共同承担责任

> 原文：<https://devops.com/cloud-security-requires-shared-responsibility/>

## 公共云的增长和对云安全的需求

自 2006 年亚马逊首次推出 EC2(首个公共云服务)以来，云已经成为企业信息技术战略的主要内容。基础设施即服务(IaaS)和更新的云服务继续强劲增长。输入对云安全的需求。

虽然公共云产品为用户提供了不可否认的好处，如可扩展性、上市速度和成本节约，但也带来了新的困境:当平台不再受用户保护，并且在某种程度上由第三方供应商控制时，谁来负责安全性？

![](img/81c81b6abe1eaf7bf888563ce09d1de9.png)

为了实现如此惊人的增长，云安全是该行业必须解决的一个问题。自 2014 年以来，安全服务一直是云提供商的重点关注领域。这包括保护基础架构本身和租户边界，以及保护数据本身的高级服务。没有安全性，任何理性的企业都不会在生产中使用公共云。

## AWS 共享责任模型方法

作为公共云的先驱，亚马逊网络服务(AWS)是第一个引入术语“共享责任模型”及其组件的提供商，开始考虑将云安全提高到足以保护企业生产工作负载的水平。

根据亚马逊的说法:

> ‍*[*安全性*](https://wa.aws.amazon.com/wat.pillar.security.en.html) *和合规性是 AWS 和客户的共同责任。这种共享模式有助于减轻客户的运营负担，因为 AWS 运营、管理和控制从主机操作系统和虚拟化层到服务运营设施的物理*[](https://wa.aws.amazon.com/wat.pillar.security.en.html)**的**组件* *。客户负责并管理客户操作系统(包括更新和* [*安全*](https://wa.aws.amazon.com/wat.pillar.security.en.html) *补丁)、其他相关应用软件以及 AWS 提供的配置* [*安全*](https://wa.aws.amazon.com/wat.pillar.security.en.html) *群组防火墙***

**简而言之，AWS 保护云的物理基础设施、计算和网络系统免受外部物理访问、数字访问和内部威胁。客户的责任是保护他们在云中运行的一切。**

## **在云部署中需要保护什么？**

**安全在公共云环境中的重要性导致了云安全联盟(CSA)的形成。CSA 对共同责任模式有自己的解释。**

**从 CSA 的角度来看，以下是需要保护的云系统类别:**

****信息和数据–**数据是任何数字企业皇冠上的宝石。从付费客户的地址到敏感的财务记录和员工信息，保护数据对于任何在线业务的成功都至关重要。此外，许多法规遵从性管理法规，尤其是最近对 GDPR 和 CCPA 的关注，给保护个人身份信息带来了额外的压力。**

**‍ **应用程序逻辑、API 和代码–**数据由应用程序逻辑和应用程序编程接口(API)解释和处理。实现应用程序的代码可以代表企业的核心竞争优势，但是逻辑本身和对 API 的保护对于保护业务及其数据极其重要。**

**‍ **用户、身份验证和访问控制—**用户既是数据的所有者，也是在线业务的收入来源，他们也需要保护。从帐户接管到身份盗窃，恶意行为者可能会伤害用户。发生这种攻击的服务的所有者最终负责提供安全性。**

**‍ **平台和资源配置–**这是云提供商自己保护的唯一主要云组件。保护通常扩展到物理场所、计算资源(包括冷却和物理维护要求)、网络资源、虚拟化系统和虚拟机管理程序、资源管理系统、存储系统等。**

## **随着平台的发展，分担的责任也会改变吗？**

**较新的云服务不仅仅提供基础设施即服务，还可能影响我们对共同责任的看法。**

**例如，一些更有趣的平台产品包括无服务器计算服务，这是一种云开发模型，允许开发人员构建和管理应用程序，而不必担心服务器管理。这些产品包括 AWS Lambda、人工智能即服务(如谷歌云人工智能)和 Medium 等托管博客。**

**如果应用程序使用这些高级服务之一，它将改变共享责任模型的应用方式，因为提供者不仅要对主机负责，还要对平台本身的安全性负责。**

**虽然这有助于我们理清安全责任，但我们仍然必须考虑信息和数据的安全(想想托管的 WordPress 博客)；知识产权和代码的安全性(如无服务器的情况)；以及 API 和应用程序逻辑的安全性(托管的 Kubernetes 基础设施、Amazon Elastic Kubernetes 服务或 Azure Kubernetes 服务可能就是这种情况)。根据共同责任模型，所有这些数据、应用程序逻辑和代码元素仍然应该由云用户或企业来保护。**

**对于较新的平台，还有几个额外的重要安全注意事项。小心云锁定。虽然运行来宾 Linux 操作系统可以非常容易地将工作负载迁移回内部或不同的云，但在托管平台上运行逻辑会使迁移变得更加困难:逻辑变得依赖于提供商 API。其次，数据流和合规性变得更加困难，因为新平台的最佳实践还没有建立起来。**
