# 迭代指标:测量 DevOps ROI

> 原文:[https://devo PS . com/iterative-indicators-measuring-devo PS-ROI/](https://devops.com/iterative-indicators-measuring-devops-roi/)

众所周知，将 ROI 度量应用于 It 投资仍然是一门不完善的科学，开发运维方法的持续采用也是如此。

抛开供应商驱动的投资回报率计算器的模糊数学不谈，即使是最受尊敬的行业观察人士也一直在努力创建评估技术支出的具体参数。这在很大程度上与持续投资的需求超过了短期指标的价值这一事实有关。

与此同时，包括移动性、虚拟化和云在内的范式对当今商业世界的影响难以估量。所谓的应用经济的范围才真正开始被认识到。

在 DevOps 中， [ROI 测量的挑战](https://devops.com/2016/06/23/state-devops-report-reaping-results/)唤起了与这些其他领头羊运动相似的特征。从长远来看，利用 DevOps 变得更加敏捷并熟练地响应不断变化的客户期望的影响最终将在组织的生存能力方面实现，就像在任何直接的成本节约或其他因素方面一样。

也就是说，由 devo PS Research and Assessment(DORA)的研究人员汇编的最近发布的 2016 年 devo PS State 报告，在创造有意义的投资回报率指标方面比任何其他类似的[先前的努力](https://devops.com/2015/03/09/the-roi-of-enterprise-devops/)走得更远。

首先， [2016 年报告](https://devops.com/2016/06/22/puppet-2016-state-devops-report-addresses-pressing-issues-devops-today/)建立在去年的结论基础上，即所谓的“高绩效组织”——那些已经深入参与开发运维实践的组织——“在吞吐量方面明显优于表现较差的同行，”并且这一差距还在继续扩大。

此外， [DORA](https://devops-research.com/) 认为高绩效者部署应用程序代码的频率是低绩效者的 200 倍，交付周期是低绩效者的 2555 倍。也许更重要的是，特别是当它涉及到提高应用程序性能的能力时，报告估计领先的 DevOps 实践者声称恢复时间快 24 倍，更改失败率低 3 倍。

由大约 4600 名技术从业者完成的 DevOps 调查还发现，高绩效团队在所谓的“计划外工作和返工”上花费的时间要少得多，这通常是解决现有问题所需的时间，这使他们可以将近 50%的时间花在新工作上。后一种努力通常旨在添加或改进特性功能，或者简单地说，创新。

随着软件速度的提高和 MTTR 的减少[代表了当今环境中最关键的差异化，在这种环境中，最终用户可能只需等待几秒钟就能加载应用程序，然后就会完全放弃你的品牌，这些结论显然具有令人印象深刻的分量。](https://devops.com/2014/04/28/devops-storytime-mttr-vs-goodhearts-law/)

最后，根据 IDC 的研究分析师建立的框架[,该框架表明，对于财富 1000 强企业，每小时的应用程序宕机成本可能在 12.5 亿美元到 25 亿美元之间，关键应用程序故障的平均成本为每小时 50 万美元到 100 万美元，DevOps 报告的状态允许得出这样的结论:高绩效组织每年平均节省超过 9100 万美元。](https://www.idc.com/getdoc.jsp?containerId=253155)

研究人员指出，这种优势在很大程度上是以较低的应用程序更改失败率的形式发现的，或者是那些导致后续性能问题或中断的软件代码更新。虽然高绩效的 DevOps 实践者经历了 7.5%的平均变更失败率，但是中等绩效的组织看到这个数字急剧上升，达到了惊人的 38%。

如果不可否认的格言“时间就是金钱”在今天的应用程序经济中同样适用(如果不是比以往任何时候都更适用)，那么这些 ROI 计算似乎可以推断出实质性的意义。
