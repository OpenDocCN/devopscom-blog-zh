# 持续播客:我们忘记如何编码了吗？

> 原文:[https://devops.com/continuous-podcast-forgotten-code/](https://devops.com/continuous-podcast-forgotten-code/)

欢迎来到 To Be Continuous，这是一个关于持续交付和软件开发的节目，由 [Heavybit](https://twitter.com/heavybit) 成员、 [CircleCI](https://circleci.com/) 的创始人保罗·比格和[launch darky](http://blog.heavybit.com/blog/to-be-continuous-16#)的首席执行官伊迪丝·哈博主持。

在这一集中，Paul 和 Edith 讨论了广泛报道的“左键盘”事件的后果，虽然他们同意这些时刻推动了整个开源社区的发展，但他们想知道:我们忘记了如何编码吗？

伊迪丝:嘿，保罗。所以，有人发布了一个叫做 left-pad 的模块，互联网基本上爆炸了。现在已经过去几周了，所以有很多不同的话题要讨论，这对开源意味着什么，是否意味着人们已经忘记了如何编程。

保罗:这么黑地发射。或者，不好意思，左边的键盘。很明显我在看我自己的 t 恤。

伊迪丝:你在看你的黑色 t 恤？

**Paul:**left-pad 事件非常有趣，我们在此之前做的事情之一就是查看所有关于[黑客新闻的讨论](https://news.ycombinator.com/item?id=11348798)和评论。

对于这对开源意味着什么，对人们是否可以编程意味着什么，对节点生态系统意味着什么，有着非常不同的立场。所以作为开始，你为什么不给我们回顾一下整个事件？

伊迪丝:是的。简而言之，一个程序员有一个模块叫做 Kik，K-I-K，这也是一个非常流行的消息应用程序的名字。我是说，即使我听说过，我也不是青少年了。

保罗:犹太人律师联系了那个家伙。说，“请取下你的模块，换一个。”程序员说，“不。我基本上不相信版权法。”于是，犹太人联系了 NPM。NPM 人拆除了名为 Kik 的模块。那个家伙，我会说，有点赌气，说，“好吧，你拆了我的一个模块，我要拆了我所有的模块。”基本上，我要带着我的弹珠回家。

所以，他拆掉了所有的模块。其中一个被证明是一个非常受欢迎的模块，叫做 left-pad，它基本上完成了名字所暗示的功能。它排序左垫。在 NPM，如果没有这些，我读到的引语是，“世界各地的建筑很快就开始失败。”

**Paul:** 据我所知，left-pad 被另一个模块使用，它被另一个模块使用，被某个核心模块使用，或者像这个非常受欢迎的模块—

**伊迪丝:**是的，基本上就是一只只海龟放在另一只海龟的上面，在海龟的底部有一个家伙—

保罗带着他的弹珠回家了。

伊迪丝:是的，或者基本上是拉最下面的叠树。

保罗:明白了。所以所有的建造都开始失败。人们的部署开始失败。人们很生气。

伊迪丝:人们很生气，因为他们甚至没有意识到自己有这种依赖性。所以 NPM 基本上重新出版了它。他们说，即使你是最初的出版商—

保罗:所以网上有很多抗议，然后他们重新发布了它。

伊迪丝:只是简单地说，“你没有权利取消发布这个。”

保罗:明白了。据我所知，他们有一个取消发布的按钮。所以他做了 NPM 内置的事情，移除了一个模块。

伊迪丝:是啊，从那以后，他们收紧了很多规定。他们基本上是说，这应该是很少使用的东西，而不是你可以随意做的事情。

**Paul:** 关于 NPM 生态系统和整个节点生态系统，有趣的一点是他们已经有了很多这样的东西。与安全相关或与戏剧相关的事情。

人们不断地出来说，“哦，不，JS 是一个沥青坑，或者贫民窟，”或者人们用来描述糟糕的生态系统的任何东西，普通的程序员不知道如何编码，等等，等等。

在我看来，你知道，这些事件中的每一个都将节点生态系统向前推进了一点。就这一点而言，我的意思是 Node 相对较新，它不像 Ruby 或 Python 那样有 15 年的黄金时间，也不像它变大之前有 20 年，它是一种真正的自下而上的东西。

我的意思是，如果有人建立了它，有一些类固醇，然后别人接管，然后 NPM 接管，然后。从生态系统的角度来看，这是一种奇怪的语言。它不具备 Java、C 或 C++所具备的优势，无法走向成熟。所以每一次这样的事件都促使它更接近成熟。

伊迪丝:是的，我把它比作电刚出现的时候。然后你就解决了所有可能发生的问题。比如说，当一根电线断了或者一个变压器烧了。

就像，你解决了，好吧，我们的失败点是什么？让我们不要再让这成为失败的一点。

保罗:嗯。所以，NPM 在此基础上改变了他们的政策，对吗？

伊迪丝:是的，所以现在你不能让个人贡献者取消发布，即使他们最初已经发布了。现在你得联系他们，而且严格多了。

保罗:明白了。人们的反应是，尽管人们……人们对他们的反应很恼火。

伊迪丝:在我看来，这完全合理。

保罗:哦，我认为这看起来非常合理。这就是流行趋势。总有一些人对你做的每件事都感到恼火。

伊迪丝:你知道，互联网上有数十亿人。我的意思是，肯定有某种标准偏差，否则这将是一个非常无聊的互联网。

保罗:没错，没错，这就是为什么黑客新闻上总是有仇恨者。无论你做什么？如果你用一种方式做，会有一群讨厌的人，如果你用另一种方式做，会有另一群讨厌的人。

因此，对于 left-pad，有些人对 NPM 追溯性地决定他们对自己的名称空间拥有所有权感到恼火，在此之前，他们在某种程度上是将该名称空间赠予社区中的人，即第一批获得该名称空间的人。

伊迪丝:你是说 Kik 最初的事？

**Paul:** 所以有那个，但是也有左拍子。他们剥夺了左翼联盟的自主权。

伊迪丝:我读过一篇很好的文章，基本上是说开源的一部分是，如果它真的是开源的，你会维护它的权利吗？还是社区项目？

保罗:嗯。决定这种所有权的传统方式是分叉。你可以去，你可以叉，当你叉，你不一定有权相同的名字，你没有权利相同的商标。当 Debian 分叉 Firefox 时，他们称之为 Iceweasel。

当詹金斯被分叉，或者哈德森被分叉，就变成了詹金斯。GCC 分叉了。他们都有了新名字。但是这里的叉子是什么，它甚至不是叉子。这是同样的东西，同样的软件的再版，命名空间不受原作者控制。

所以通常你会拥有 URL，你知道，如果你有一个软件库，你会拥有 URL。在某种意义上，他们确实拥有 NPM 的网址，或者 NPM 空间的名字，而 NPM，这个名字空间的所有者，否决了它。

伊迪丝:我认为他们做了维持良好秩序所必须做的事情。

保罗:当然可以。我完全同意，我认为这是一个好决定。

伊迪丝:我的意思是，从逻辑上来说，如果你有一个名为 left-pad 的模块，你决定，就像恶作剧一样，把它重写为 right-pad，会怎么样？就像，当然，那是…那是你的权利吗？有点，但这是一个真正的混蛋做的事情，你可能想阻止。

**Paul:** 这是围绕集中式包管理器的整个生态系统的危险之一，这是一种可信任的包管理器，其中不涉及实际的信任。所以，我的意思是，发布的内容没有任何把关人。

伊迪丝:是的，没错。

保罗:我的意思是，这很好，它让一个生态系统增长得更快，但它也有安全问题，因为人们通常不会将他们的包绑定到一个确切的版本。

伊迪丝:你基本上是把它固定在一个浮点值上。

保罗:你把它固定在一个浮点值上。并且该浮动值可以改变。不保证次要编号意味着什么，主要编号意味着什么，补丁编号意味着什么，甚至补丁意味着什么。在语义版本控制方面有一些类似的通用社区指南，但是没有强制执行。

伊迪丝:是的，我的意思是这基本上可以追溯到整个大教堂和集市的事情。

保罗:对，对。很巴扎。而且是故意的。这是允许一个社区成长的事情，开源社区是否需要，比如，封闭舱口以防止这种行为？或者防止这种风险？

伊迪丝:是啊，而且真的很有趣。我认为这可以追溯到我们几集前[与 Nadia Eghbal 和 Sean Byrnes 谈论的事情](http://www.heavybit.com/library/podcasts/to-be-continuous/ep-17-open-source-economics/)当你从一家公司购买时，你有他们在某处发布的标准的某种保证。你知道，这是我们对开源的保证，你通常不会有同样的保证。

保罗:即使你这么做了，也往往是—

伊迪丝:没有强制机制。

保罗:对。所以集市上没有强制机制。

**伊迪丝:**嗯，没有，嗯……比如，对于一家公司来说，如果一家公司说，“好的，我们将一年发行两次，质量将达到这个水平”，而他们打破了这个标准，你可以用你的脚或你的钱走路。如果一个开源项目停止了或者做了一些你不喜欢的事情，那么很少有强制执行的结果。

理论上，你可以走路。你知道，你可以叉东西，但当你最终在一个有巨大网络效应的生态系统中，像 NPM 那样，你不能带着它走。因为 pends 都是可传递的。你知道，左边的垫子是一堆最下面的乌龟。

伊迪丝:它几乎就像一只刚孵化的海龟。

**Paul:** 所以，要分叉 left-pad，你就得分叉整个 NPM 生态系统。

伊迪丝:是啊，而且还没扯平。不过，叉子不是问题所在。问题是人们依赖于它以特定的方式存在于特定的地方。并不是说它没有分叉，只是底部的整个结构都消失了。

**Paul:** 我想我要说的是，如果你对 NPM 社区中的某一组开发人员或某一组实践感到恼火，因为他们是如此的相互关联，你就没有传统的开源分叉选项。

伊迪丝:绝对是。绝对的。

保罗:你可以给它重新命名。你可以选择一个不同的库，但是因为它的方式——我不确定他们是否还这样做，因为我读过一些关于它的东西，但是在 NPM，它曾经是你的库使用的软件版本是库得到的版本。你知道我的意思吗，好吧。所以假设有一些软件，我们称之为 left-pad。

伊迪丝:我们就叫它右垫吧。

**保罗:**右键。因此，库 a 正在使用右 pad 版本 1。库 b 正在使用右 pad 版本 2。在 Java 生态系统和许多许多其他生态系统中，只会使用一个右 pad 版本。

我相信在 NPM，两种版本的右键盘都会被使用。所以库 A 会用版本一，库 B 会用版本二。现在这肯定是过去的方式，我不确定他们是否真的改变了这一点。

伊迪丝:是的，我不知道，但是听起来很乱，很快。

保罗:所以两种选择都很糟糕。没有正确的方法去做。

所以，真的，这取决于什么。所以，NPM 的做法，每个库都有自己的依赖版本，是很棒的。因为它让这个库在幕后做它真正想做的事情。

然而，如果你开始有一个对象被发送出去，或者两个模块被发送出去，比如说它接收到一个，我认为，一个 HTTP 包或者类似的东西，就是一个很好的例子。比方说，你从一个模块中取出一个 HTTP 数据包，然后把它交给另一个模块，他们对该对象的形状有不同的理解，或者你可以在该对象上运行什么功能，这就很麻烦了。

但是，另一方面，如果您覆盖了其他库正在使用的 right-pad 版本，您最终会遇到各种各样的问题。这是 Maven 生态系统中的一个主要问题。在 CircleCI，我们使用 Clojure，Clojure 是建立在 Maven 生态系统之上的，我们使用的库可能会有错误。

我们使用的一个库有一个依赖项，它在另一个东西的包文件中。这就是我们使用的数据包版本。所以我们必须选择。因此，我们必须选择有效的库。而且，通常，没有一个库对两者都适用，或者没有一种子依赖对两者都适用。所以不管你用哪种方式，这些都是接近分叉的。

伊迪丝:是的，我是说，我们在 STK 上遇到了同样的问题。我的意思是，因为我们对其他库的依赖有时会发生冲突。

**保罗:**对，灾难。

另一方面，我认为更重要的一点是。所以有一种有趣的反弹，我认为，每个人都在 NPM 的事情上投射了他们想要的东西。

人们说，哦，人们不再知道如何编程了。你知道，你为什么要用他们所谓的左键盘？你知道，你为什么不自己磨小麦，从头开始做面包，然后在午餐时吃呢？

保罗:我一点也不赞同这种世界观。

伊迪丝:嗯，你应该从头开始做所有的事情？或者—

保罗:那个，“哦，我们已经忘记如何编程了，因为我们需要左键盘。”

伊迪丝:是的。事实上，我们有一些模块可以让编程变得越来越容易，这是一件好事。

保罗:我认为和那个有点不同。我认为这更像是一种坏脾气的世界观。这是一个“滚出我的草坪”—

伊迪丝:等等，你就是那个坏脾气的人？

保罗:不，是那些抱怨的人，“我们忘记编程了吗？”我认为这不是真正的苏格兰人的观点。真正的程序员知道如何编写自己的左键盘，真正的程序员用静态类型的语言编写，真正的程序员用 C 语言编写，真正的程序员，我在这里要回到过去，真正的程序员用汇编语言编写，真正的程序员手工编写他们的。拧动针。

伊迪丝:他们自己收集垃圾。

保罗:哦，还有垃圾收集，这是什么？垃圾收集是给业余程序员用的吧？我的意思是，你可以很容易地写，我们忘记了如何手动分配内存吗？

伊迪丝:或者，你也可以说，没有人知道如何制造自己的晶体管了。我的意思是，整个计算机革命就是你不再需要手工组装电脑。

保罗:对。生态系统的每一层都允许更多的人参与其中。所以我认为我们现在看到的。这种情况已经发生了 5 到 10 年，但是现在变得非常，非常重要的是那些只知道如何做前台的人。

那些只知道 Javascript 和 HTML/CSS 的人，他们去了新兵训练营，出来后，他们可能有一些模糊的真正理解。

他们自称程序员，而我们，真正的程序员，知道如何写快速排序，知道如何写左补，我们不欢迎他们加入我们的社区。坦率地说，我的意思是，我过去也有过这种感觉，后来我改变了想法，但这是一个非常非常普遍的观点。
