# API 分析帮助应用所有者的 3 种方式

> 原文：<https://devops.com/3-ways-api-analytics-can-help-application-owners/>

软件开发充满了风险，可能会使项目搁浅，就像一艘船搁浅在多岩石的海滩上一样。无论是被误解的需求，快速发展的市场，还是过时的错误和进度延误，都可能破坏资源充足和管理良好的软件开发项目。

即使软件开发团队最终交付了他们的产品并从客户那里收取了费用，也不能保证产品会被使用。一项 [2022 年的研究](https://www.ciodive.com/news/saas-spend-control-enterprise-flexera/608257/) 发现，在那些为 SaaS 软件付费的人中，29%的软件要么没有得到充分利用，要么被浪费了。这项研究还指出，只有 33%的 IT 专业人员跟踪 SaaS 软件的使用情况。这里的要点很明显:作为一个行业，我们的大部分劳动力都是*而不是*用于生产目的。

这不是软件的新趋势；一个完整的工具生态系统的存在是为了帮助软件开发公司收集用户参与度量来驱动开发决策。事实上，在全球排名前 1000 万的网站中，近三分之二的[](https://w3techs.com/technologies/overview/traffic_analysis)拥有分析解决方案。然而，相比之下， [API](https://devops.com/?s=API) 应用程序在采用这种实践方面远远落后。很难获得 [API analytics](https://devops.com/connections-matter-gaining-visibility-into-apis/) 的市场渗透率数据，但 API 管理供应商 Apigee 的一份报告显示，即使在其客户(即那些已经为包含 API analytics 的服务付费的客户)中， [也只有 26%](https://cloud.google.com/blog/products/api-management/api-monitoring-and-analytics-for-2021) 使用了分析功能。这让人们相信大多数功能都没有被使用。

没有将真实世界的分析使用数据集成到决策过程中的 API 应用程序所有者正在浪费时间和金钱。尽管无法直接获得用户行为，API 分析在帮助 API 应用程序所有者做出关于应用程序开发和维护的更好决策方面非常有价值。

API 应用程序所有者应该注意以下三种方式来做出更好的产品开发决策。

### 在应用程序最常用的领域中优先考虑功能开发

我在自行车店工作时学到的最好的一课是“卖什么就储存什么”换句话说，在你的货架上摆放顾客不买的产品是没有意义的。同样的道理也适用于在 API 产品目录中添加客户不使用的端点。

API 分析允许 API 产品所有者测量特性的使用并评估它们的有效性。产品所有者可以使用这些有价值的信息来优先扩展流行功能或开发新功能，并有一个清晰的标准来衡量使用情况。此外，这些数据使开发团队能够“最大化未完成的工作量”，方法是在完全实现之前发布初步的特性集并评估使用度量。同样，如果现实世界的反馈不令人鼓舞，它允许你转向产品中的另一个机会。

### 弃用或废弃不用的应用程序功能

未使用的代码和特性对 API 产品组织来说是一种负担，可能会带来错误、性能问题和安全风险。作为一名软件开发人员，我曾经见过这样的情况，这些老的特性是由早已离开组织的开发人员编写的。这些未使用的部分通常隐藏着严重的安全缺陷和漏洞(例如多年未修补的软件依赖)以及公司中没有人完全理解的可疑授权逻辑。

未使用的、无关的特性也会导致产品臃肿，分散最终用户对 API 核心价值的注意力。提倡删除“工作产品特性”可能很难，但 API 分析数据可以帮助证明某些特性只不过是一种昂贵的干扰。删除应用程序中旧的、不用的部分也让开发人员可以专注于用户喜欢的部分。

### 优先考虑性能改进

性能通常是实现时间/成本和产品需求之间的微妙平衡。

软件开发中有一条广为人知的格言:过早的优化是万恶之源 。然而，在缺乏真实性能数据的情况下，确定什么代表不成熟的优化，什么代表完全必要和谨慎的优化，往往会归结为工程师之间的主观争论，或者更糟糕的是，导致上市时间缓慢的规避风险的软件开发文化。

对性能影响进行客观衡量，例如客户端受到次优性能影响的实例数量，可以帮助应用程序所有者决定如何分配预算来提高性能，而不是开发新功能或修复错误。此外，拥有这个反馈循环可以使开发团队能够采取敏捷的“快速进行，经常发布和评估”的开发方法。不要因为担心看不见的问题会导致糟糕的性能和不满意的用户而预先优化性能，而是可以随着时间的推移跟踪和检查性能。

### 选择 API 分析解决方案

运营单一应用程序的 API 产品所有者在决定为他们的产品使用哪种分析工具时有多种选择。对于单独的 API 开发语言和框架，存在许多专门的解决方案。然而，安全团队有一个独特的问题:他们需要跨组织内所有应用程序和所有技术平台的所有数据。在这种情况下，我推荐一个具有强大 API 分析能力的 API 保护供应商，它不仅可以防御威胁，还能为您提供确定工作优先级所需的可见性。

API 分析应该成为推动创新的工具。不要把它闲置在你的工具箱里。
