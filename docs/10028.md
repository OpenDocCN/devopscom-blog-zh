# 利用 DevSecOps 优先考虑产品安全性

> 原文:[https://devo PS . com/prioritizing-product-security-with-devsecops/](https://devops.com/prioritizing-product-security-with-devsecops/)

构建具有强大安全性的软件不再是组织事后的想法。事实证明，在全球各行各业不断遭受攻击的情况下，可靠的网络安全态势至关重要。特定于产品的安全性是大多数公司的头等大事，成功的产品安全性实践需要与工程团队的精神相结合。考虑一个三重策略，这个策略在行业中被一次又一次地证明是成功的。

它包括首先[创造一种安全文化](https://devops.com/motorola-solutions-driving-a-thriving-security-culture/)，让所有利益相关者理解优先事项、策略和商业价值。其次，团队必须将安全性与软件开发生命周期(SDLC)相结合。向左转变是不够的，组织必须将其思维从向左转变为到处转变。最后，始终监控产品系统是不容置疑的。在一个 24/7/365 的世界里，必须有团队 24/7/365 监视安全。工程交付的一部分变成了向这些团队提供数据，这样他们就可以一直关注产品。

### 成功的实践始于文化

在建立强大文化的过程中，目标是让整个工程团队成为使命的一部分。这包括组建一个产品安全团队来支持工程需求，以及提供持续教育和资源来应对任何可能发生的情况。

虽然每个组织在创建产品安全团队时都有自己独特的需求、考虑因素和方法，但强烈推荐以下五个角色:

**1。技术产品经理(TPM):** 负责推动规划中的安全优先级
**2。安全架构师:**是架构评审委员会的成员，确保架构和设计包含安全流程
**3。AppSec 工程师:**负责实现安全开发生命周期
**4。敌对工程师:**敲打攻击面，找出弱点
**5。安全冠军:**来自每个 Scrum 团队的工程师，有兴趣用安全基础知识支持他们的团队；它们有助于扩展整个产品安全工作

创建和执行引人入胜的安全教育计划对于赢得员工的信任也至关重要。考虑诸如深入的安全训练营、每月漏洞报告、夺旗(CTF)竞赛和安全咨询计划等计划。

### 将安全性集成到 SDLC 中

从高层次来看，所有 SDLCs 都包括规划、开发和发布，每个阶段都应该有自己的一套安全控制措施。

**规划:**产品安全 TPM 负责与产品管理团队合作，确定安全优先级——来源于风险降低、执行优先级、客户请求、安全调查结果和监管要求。确定什么是最重要的是关键。

**开发**:许多重大的产品安全提升都发生在开发阶段，包括实现威胁建模、安全基线和自动扫描。

**威胁建模:**识别系统中潜在威胁的过程。一旦确定，威胁建模团队就可以创建缓解措施，并将结果集成到渗透测试中，以验证威胁没有进入实施阶段。在编码开始之前，在架构阶段测试模型是最有效的，这样可以避免返工和权宜之计。例如，一家公司可能有一个策略，禁止从互联网访问端口 22。威胁建模自动化可以检测设计中的违规行为，并通知所属团队要求他们关闭端口。支持这种用例的工具包括 [IriusRisk](https://www.iriusrisk.com/) 和 ThreatModeler，以及微软威胁建模工具，这是一个免费选项，但不包括自动化。

**安全基准:**团队应创建完全修补的安全映像，满足安全策略并预装强制安全代理。对操作系统映像和容器都这样做可以让开发人员“免费”获得一定的安全级别，并避免持续的安全审查。基线应该自动更新并流入 CI/CD 系统。

**自动化扫描:**为了跟上敏捷流程，扫描应该是自动化的，并且具有较低的误报率。事实上，少报道一些问题比用很多噪音扼杀可信度要好。持续调优将增加保真度规则集，有用的工具包括静态应用安全测试(SAST)、软件组合分析(SCA)和动态应用安全测试(DAST)。自动化对于保护使用 CI/CD 开发过程的应用程序是至关重要的，并且在代码推送时提供安全性是非常宝贵的——开发人员可以立即获得反馈，而不是几周后他们可能忘记重要上下文时的罚单。将这些工具集成到构建管道中是一个已经解决的问题。

例如，下面的 YAML 演示了对嵌入机密(例如，密码和密钥)的自动化源扫描。

sast:secrets-detection:
扩展:。先科:秘密侦查
阶段:部署:开发

**发布:**当团队到达发布阶段时，对抗性测试成为一个重要的组成部分。渗透测试可以是内部的、外包的或者两者都有。内部的 red 团队对技术有更好的理解，因此比外部的安全合作伙伴有更广阔的视野。例如，他们跨产品边界进行测试，测试流程和人员，测试实际生产系统。对于一个成熟的产品，考虑一个 bug 赏金计划。这会邀请白帽黑客来寻找弱点以换取现金。但“成熟产品”是必不可少的；否则，支出的数量将变得难以承受。

### 全天候监控

如前所述，全天候监控是关键，尤其是对于任何基于云的技术。专门的安全运营中心(SOC)可以提供帮助，检查针对公司、员工和客户的威胁。SOC 的职责比产品安全团队的职责要广泛得多，他们让产品安全人员晚上睡个好觉。安全团队应配置系统、负载平衡器、公共云平台和网络，以便将数据直接输入安全信息和事件管理(SIEM)系统。然后，SOC 可以响应定制的自动警报，这些警报可以识别流向它们的任何数据异常。

随着产品以及整体技术变得越来越先进，组织保护产品的方式也必须如此。完美的安全方法因业务而异，但授权给 IT、工程和安全团队是为员工和客户识别、优化和集成安全性的有益的第一步。
