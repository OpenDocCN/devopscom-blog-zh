# 您的应用程序是最薄弱的安全环节

> 原文:[https://devo PS . com/your-applications-is-weak-security-link/](https://devops.com/your-applications-are-the-weakest-security-link/)

亚马逊、微软和谷歌的安全预算比你多得多。他们在世界各地有数百名员工，确保他们的基础架构是安全的。他们的团队不断审查他们的代码，寻找缺陷、漏洞和潜在的漏洞。他们监控黑客论坛，运行有吸引力的漏洞奖励程序，以保持主动。

仅仅几年前，内核错误还很普遍。现在，微软和 Linux 的团队几乎已经消除了内核漏洞。例如，根据 CVE 数据库，2017 年 Linux 内核有 169 个代码执行漏洞，2018 年只有 3 个，2019 年只有 5 个。

说到云平台，云提供商确实要对云的安全性负责，而客户仍然要对云中的安全性负责——这也是危险所在。

解决方案架构分布在公共云和传统内部环境中。它们包含客户自己的软件以及客户无法控制的开源和第三方产品。微服务的激增显著增加了网络接口的数量。不断增长的发布速度使得没有时间去识别和解决潜在的安全问题和配置疏忽。

随着操作系统和关键基础设施软件的改进，应用软件中的漏洞数量也在显著增加。改变现有应用程序或注入额外代码的能力使得攻击者能够将合法软件转化为恶意软件。这种漏洞被列在“代码执行”类别中，这恰好是 CVE 数据库中最大的类别。

修补已知漏洞平均需要 50 多天，而攻击者只需要几分钟就可以利用它们。除此之外，还有大量未知/未发布的零日漏洞，经常被商业攻击者使用，他们为这些漏洞付费并将其隐藏起来。

在众所周知的 Equifax 数据泄露事件中，恶意软件利用 Apache Struts 漏洞渗透到合法的 web 服务器中，该漏洞允许任意代码插入。自从这次攻击发生在 2017 年，CVE 数据库显示，Apache Struts 至少有五个类似类型的漏洞。

基本上，客户应用程序现在是，将来也永远是最弱、最易被利用的软件。我们如何保护他们？

软件运行时代码执行漏洞被利用。有些攻击根本不会留下任何文件系统痕迹——无文件恶意软件。该漏洞可能在执行会话的几天甚至几个月内出现。识别入侵的唯一可靠方法是在整个执行会话中运行代码的同一存储器中验证软件完整性。对代码的任何更改或未签名代码的添加都会损害其完整性，因此会被识别。

传统的、基于文件的签名机制在这里不起作用，因为可执行文件在加载到内存中时会发生变化；因此，它必须是一种足够健壮的特殊算法，以保护自身和应用软件。

就像人类 DNA 识别一个人一样，加密签名可以唯一地识别软件。这种机制可以进一步用于为应用程序提供适当的授权，例如通信和信息访问。这将确保只有具有正确“DNA”的应用程序才能相互通信并访问加密信息，这是最终的零信任范例。
