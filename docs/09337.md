# OpenSSF 增加了开源包分析工具原型

> 原文：<https://devops.com/openssf-adds-open-source-package-analysis-tool-prototype/>

开源安全基金会(OpenSSF)已经提供了一个包分析工具的[原型，该工具已经识别了 200 多个上传到 PyPI 和 npm 软件组件的恶意包。](https://openssf.org/blog/2022/04/28/introducing-package-analysis-scanning-open-source-packages-for-malicious-behavior/)

该项目的 OpenSSF 维护者 Caleb Brown 表示，目标是理解开源存储库中可用的包的行为和功能。它还跟踪软件包在一段时间内的行为，以便更容易识别以前安全的软件何时开始以可疑的方式运行。

Brown 说，大多数检测到的恶意软件包都采用了依赖性混淆和域名仿冒技术。它们通常包含一个在安装过程中运行的简单脚本，并向外部命令和控制系统提供有关主机的一些详细信息。除了机器名或用户名之外，这些包中的大部分都没有泄漏有意义的数据，但是它们没有试图掩饰自己的行为。布朗指出，问题是这些方案中的任何一个都可能造成更大的损害。

该项目的未来目标包括随着时间的推移检测包装行为的差异；自动化包装分析结果的处理；在处理过程中存储包装本身，以便进行长期分析，并提高管道的可靠性。因此，OpenSSF 正在寻找应用程序开发人员和网络安全研究人员为该项目做出贡献。Brown 说，预计许多应用程序安全工具提供商将把包分析工具集成到他们的产品中，而不是要求每个供应商创建他们自己版本的相同工具。

OpenSSF 已经启动了一系列计划来改善开源安全的整体状况。最近，它增加了一个 Alpha-Omega 项目，利用微软和谷歌提供的 500 万美元初始投资，自动化开源软件的安全测试过程。核心挑战是许多组织依赖于由少数志愿者维护者和贡献者创建和维护的[开源](https://devops.com/?s=open+source)软件项目。创建这些项目的个人并不总是有很多网络安全专业知识。事实上，他们中的许多人会争辩说保护开源软件的责任在于那些使用相当于自由软件的组织。例如，贡献者和维护者没有责任放弃一切并立即创建一个补丁来解决零日漏洞。

目前还不清楚开源软件有多容易被网络犯罪分子破坏，他们会将恶意软件插入依赖该代码的下游应用程序中。然而，在一系列高调的违规事件之后，可以肯定的是，越来越多的网络犯罪分子正试图危害开源软件项目。虽然一些组织可能会选择限制对未经适当审查的开源代码的依赖，但没有办法完全避免使用开源代码。面临的挑战是找到一种更有效的方法来确保所有相关代码的完整性。
