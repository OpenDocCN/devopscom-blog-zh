# GitHub 爆炸半径突破重大安全隐患

> 原文:[https://devo PS . com/blast-radius-of-github-breach-major-security-concern/](https://devops.com/blast-radius-of-github-breach-major-security-concern/)

在 GitHub 披露的安全漏洞之后，软件供应链可能受到损害的程度可能包括数千家组织。

GitHub 透露，未授权方[破坏了由 Heroku 和 Travis CI 维护的 OAuth 用户令牌](https://github.blog/2022-04-15-security-alert-stolen-oauth-user-tokens/)，Heroku 是 Salesforce 的一个分支，提供平台即服务(PaaS)环境，Travis CI 是持续集成/持续交付(CI/CD)平台的提供商。GitHub 已经断开了使用这些令牌访问存储库的第三方应用程序，但有多少代码可能已经被泄露，或者更糟的是，被修改，可能几周内都不会知道。

“这可能有很多方向，”Techstrong Research 的首席研究员米奇·阿什利(Mitch Ashley)说，tech strong Research 是 Techstrong Group 的一个分支，拥有 DevOps.com。“这是对认证机制的攻击。”

事实上，除非每个使用 Heroku 或 Travis CI 平台的组织都公开他们的软件供应链是否遭到破坏，否则可能永远也不会知道破坏的整个范围。AppOmni 的首席技术官 Brian Soby 指出，GitHub 还没有提供违规程度的任何时间表，因此没有人知道应该开展多广泛的调查，或者需要多长时间才能发现可能已经注入应用程序的恶意代码。AppOmni 是一个提供平台配置可见性的平台提供商。

“我们还不知道这是什么时候发生的，”他说。"这可能会有很大的爆炸半径."

与此同时，GitHub 的披露只会加强对软件供应链的安全审查，在一系列高调的违规事件发生后，这种审查已经开始认真进行，这些违规事件始于涉及网络安全管理软件产品提供的 IT 服务管理平台的事件。

在最新的 GitHub 漏洞案例中，安全专业人士推测，开发人员将令牌硬编码到访问 GitHub 的应用程序中，以使其更容易访问存储库。“开发者很懒，”Panorays 的首席技术官黛米·本·阿里(Demi Ben-Ari)说，Panorays 是一家评估第三方安全风险的平台提供商。“他们喜欢走捷径。”

Ben-Ari 指出，作为更好地保护其知识产权的任何努力的一部分，一些组织在受到攻击后可能会跨多个供应商的存储库分发源代码。

组织也可能决定依赖不能通过云访问的代码库。然而，考虑到跨软件供应链集成不同服务的需要，无法保证在内部 IT 环境中运行存储库的组织不会受到类似攻击的影响。

另一方面，随着越来越多的组织意识到危害 GitHub 存储库的威胁媒介，实现最佳 DevSecOps 实践的意识水平应该会提高。当然，挑战仍然是弥合技术和文化鸿沟，这些鸿沟经常导致开发人员在构建和部署应用程序时不遵循最佳安全实践。然而，随着事件数量的不断增加，越来越多的组织开始意识到他们用来吸引客户和供应商的代码是多么容易被篡改或直接窃取。
