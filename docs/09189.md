# 安全软件峰会:看 SBOM

> 原文:[https://devops.com/secure-software-summit-behold-the-sbom/](https://devops.com/secure-software-summit-behold-the-sbom/)

随着供应链安全越来越受到关注， [SBOM](https://devops.com/?s=SBOM) 现在被视为支撑供应链安全的关键要素。SBOM 代表软件材料清单。在最基本的层面上，SBOM 是一个成分列表。想一想超市里的每种食品是如何列出成分的，例如，对坚果过敏的人可以不吃坚果。SBOM 为软件应用程序做同样的事情。

如果你想和美国政府做生意，SBOMs 不再是可有可无的。行政命令 14028 要求所有美国联邦软件采购都要有 SBOM。为什么现在推动 SBOMs？软件公司通常不会列出他们生产的软件中包含的所有组件。了解应用程序中包含哪些组件对于主动漏洞管理至关重要。也就是说，政府对 SBOM 的要求只要求包含最少量的信息——供应商名称、组件名称和版本、依赖关系、版本或修改的时间戳，以及 SBOM 本身的作者信息。

## 在 SBOM 中寻找什么

这种最低要求的数据对降低网络安全风险和漏洞管理的益处很小。SBOMs 实际上可以包括许多其他类型的信息。对于贵组织生产、购买或消费的软件应用程序，您应该在 SBOMs 中包含哪些信息？这是一份清单。

**真实性**—真实性询问并验证某人确实创建了一个列出的组件。理想情况下，这需要某种数字签名。在嵌套组件的情况下，真实性尤其重要，在嵌套组件中，应用程序的提供者包括多个其他组件，这在用于通用功能的 JavaScript 和 Java 库中很常见。

**完整性**—这解决了组件是否被篡改或未经授权修改的问题。SBOM 应该包含某种程度的完整性。同样重要的是证明 SBOM 本身的完整性。正如校验和可以告诉您正在加载的软件是否与该软件的来源相符一样，完整性检查应该允许您验证 SBOM 上的组件(以及 SBOM 本身)是否正是您所期望的。

**服务**—由应用程序调用的服务的清单是至关重要的。现代软件不是生活在真空中。它调用互联网来执行各种任务，并与其他系统进行通信。那些扩展到应用程序之外的服务是清单的一部分，因为它们可能是漏洞的指示或更大问题的指示。

**漏洞修复状态**—开发团队一直在平衡修复漏洞的需求和修复会破坏应用程序的风险。例如，一些开源项目有不回推安全补丁的历史。项目的最新版本可能会破坏应用程序。组织被迫在破坏应用程序和冒险引入可能被利用的高严重性漏洞之间做出选择。选择保留旧版本并且不破坏软件的负责任的组织将会反向移植他们自己的修复。SBOM 应该描述这些变化和任何其他漏洞补救活动。

**组件安全建议清单**—SBOM 应描述针对所有组件的未决安全建议。这很可能是一个动态列表，您需要使用一种公认的漏洞公告格式(CSAF 和 CycloneDX，举两个例子)来自动更新。事实上，这是 SBOMs 的优势之一——它们是为自动化和以编程方式操作而设计的。

**完整性**—由于技术或知识产权方面的原因，并不总是能够发运完整的 SBOMs。SBOM 应该指出它是否完整，是否包含依赖项和服务的全面清单。

**血统**——这个信息描述了 SBOM 的 DNA 和每个特定组件的原始构成和变更记录。SBOM 应该描述原始组件，确切地说做了什么更改，谁提交了更改以及为什么，不同版本之间有什么差异，以及每个变化和版本修复了什么安全问题。

**出处**—这与组件或服务的起源有关。它可以描述下载该组件的存储库、原产国、软件制造商或开源项目的名称，或者指示该组件或服务的原产地的其他信息。

**许可证类型**—许可证信息与合规性相关，可确保您的组织不以违反许可条款的方式消费或生产软件。与安全性和漏洞相比，这些信息更适用于风险管理、合规性实施和审计。

## 定制 SBOMs 以满足您的需求

SBOM 是一种灵活且适应性强的方法，可以轻松定制以适应特定的使用情形。无论是作为软件提供商还是消费者，您都可以设计 SBOMs 来满足您的用例的特定需求。随着需求的发展，您可能还需要修改 SBOM 字段和数据设计。幸运的是，使用软件安全工具，如软件组合分析(SCA)和交互式应用安全测试(IAST)解决方案，自动生成 SBOMs 正变得越来越容易。SBOMs 也可以在持续集成/持续部署(CI/CD)过程中的构建生命周期的其他时间点自动创建或读取。正如你不会买一盒没有列出成分的麦片，将来你也不会想使用没有详细 SBOM 的软件。你对所运行软件的不了解会对你造成伤害，而 SBOMs 将成为一个重要的工具，为软件供应链提供更大的透明度和可观察性。
