# 过渡到 DevSecOps 时会发生什么

> 原文:[https://devo PS . com/what-to-expect-when-transitioning-to-devsecops/](https://devops.com/what-to-expect-when-transitioning-to-devsecops/)

您如何确保 DevOps 管道的安全？DevSecOps 是否保护您免受严重的破坏，或者它只是一种减轻利益相关者对 DevOps 安全性的担忧的方法？

根据 IBM 的研究 [数据泄露的成本](https://www.ibm.com/security/data-breach) ，数据泄露的平均成本为 392 万美元。在该报告中，医疗保健被认为是最昂贵的行业，违规罚款平均为 645 万美元。

虽然 DevOps 兑现了提高交付率的承诺，但传统的 CI/CD 框架并不强调安全性。为了提高速度，错误配置和已知的漏洞可能会被允许通过开发管道进入生产环境。

安全和测试通常是低优先级的活动，这意味着这些漏洞可以追溯修复。这需要更多的时间、精力和金钱。随着开源工具的猖獗、未经验证的集成，您向攻击者开放了软件，这可能会危及您的应用程序、系统和用户数据的安全。

DevSecOps 是 DevOps 的升级版，允许您持续进行安全性和合规性测试。这些检查被集成到了 [CI/CD](https://devops.com/?s=CI/CD) 管道中，以便更快地交付更安全的代码。

## 从 DevOps 到 DecSecOps:期待什么

DevOps 的自动化性质带来了几个好处，使团队能够在筒仓破裂和交付率飙升时认识到 DevOps 的真正威力。

由于 DevOps 实践跨越整个交付渠道，它提供了必要的机制来交付审计、合规性和安全性要求以及速度和效率。

从 DevOps 升级到专门的 DevSecOps 实践有许多好处，在开始这一旅程之前了解这些好处非常重要。

### 1.没有针对开发人员的限制性访问控制策略

必须严格控制和监控对基础设施资产的访问，以实施必要的安全措施。这意味着批准门、安全配置参数和其他安全措施。

通过将 DevSecOps 内置到管道中，可以自动应用这些访问控制。这使得团队能够按需访问基础设施资源，并减少他们对团队外部因素的依赖。

例如，许多组织已经开始部署内部开发运维团队，实现对部署流程、工具、存储库、工作流等共享池的自动化访问。

这个团队通常独立于 IT 团队，但以同样的严格性实施安全机制。它确保根据指导原则自动跟踪、监控和控制对开发、QA 或生产所需的所有基础架构实例的访问。此外，不一致的流程或错误配置也得到控制。

通常被称为影子 IT 团队，这个团队的想法不是绕开 IT 部门，而是创建一个额外的团队来匹配 DevOps 团队的敏捷性。

### 2.无缝的安全性和合规性测试

传统上，安全性和合规性测试在接近发布日期的发布周期结束时同时进行。这通常意味着一个大的问题电子表格被发送到开发团队来立即修复，因为保持一致的发布速度是 DevOps 成功的关键。

这要求开发团队至少有一个专门的资源来处理这些问题。当度量被跟踪时，毫不奇怪，这种方法对下一个发布周期的速度产生了负面影响。

DevSecOps 批量应用所有安全和符合性检查。因此，安全性和法规遵从性的测试被转移到左边，分散到整个管道中。

这使得安全反馈循环更短、更频繁，从而使安全性和合规性测试与 DevOps 理念保持一致。由于每次都提交更安全的代码，QA 团队报告的安全性和遵从性错误的可能性被最小化。

第二种更个性化的方法是使用集成开发环境(IDE)本身对开发人员进行安全性和合规性检查。随着定期和自动化的安全性和遵从性被融入到开发过程中，这个过程被大大加速，开发团队变得自力更生。

### 3.克服与开源软件相关的风险

来自 Sonatype 的 [2019 年软件供应链状况报告](https://www.sonatype.com/press-releases/2019-state-of-the-software-supply-chain-report-reveals-best-practices-from-36000-open-source-software-development-teams)显示，平均而言，现代应用程序中 85%的代码库使用开源组件。虽然开源组件促进了创新，但它们也带来了相当大的安全风险。随着越来越多的开源工具被添加到软件供应链中，它增加了网络安全方面的风险。

同一份报告显示，有证据支持需要更好的开源软件安全检查和流程。管理开源组件的公司比那些没有可验证的开源管理流程的公司发布的易受攻击的组件少 55%。

虽然这些开源组件经常发布补丁，但 DevSecOps 团队介入并监管开源组件也很重要。DevSecOps 团队可以这样做的一些方法有:

*   创建一个正式的组件管理计划，确保当员工离开时，实践不会随之消失。拥有一个记录在案的计划可以让你从更大的角度来看待你的管道，并找到那些可能在没有正式流程的情况下被忽略的脆弱部分。如果这个计划被很好地记录，对它的渐进改进也是可能的。
*   使用扫描工具，如 Gemnasium 和 Snyk，帮助定期检查漏洞。
*   如果在组件中检测到或报告了漏洞，请考虑更新或更换组件。
*   能够快速批准新的开源软件。或者，让开发人员准备好使用安全性和遵从性清单来测试新组件。

### 4.降低总体成本，加快交付速度

devo PS 中集成的安全性和合规性框架可防止违规，并在违规发生时保证更快的恢复。由于开发人员不必留出大量时间来处理安全性和合规性违规，这缩短了发布周期，同时确保了更安全的代码。

您识别和修复的漏洞越少，您在克服安全灾难方面节省的资金就越多。

### 5.自动化合规报告

DevSecOps 自动执行您的安全性和合规性测试，并记录和记录所有测试。DevOps 自动化平台为构建、测试、集成和部署阶段收集大量信息。这些数据可用于创建端到端的审计跟踪，无需任何手动干预，也无需花费额外的时间来浏览各种工具以准备合规性报告。

DevSecOps 的一大优势是从第一天起就具有完全的可追溯性和自动化的合规性报告。DevSecOps 还关注测试的自动化，允许您在持续的基础上交付安全的开发框架。

## DevSecOps 是 DevOps 的未来

DevOps 提供的快速释放速度并非没有缺点。特别是，在管道中不加检查地使用开源工具会使您的应用程序和系统暴露于许多已知的漏洞，增加了违规的可能性。再加上普遍缺乏对安全性的关注，这就导致了代价高昂的安全事故。

为了应对这种情况，DevSecOps 通过将安全性和合规性测试向左转移，彻底改变了管道。开发人员在每次提交之前都会检查代码。通过快速反馈循环，他们可以修复任何报告的漏洞以及应用程序中的其他错误。

使用 DevSecOps，合规性和安全性检查不会一次完成；相反，开发人员在每次发布时都会消除漏洞和安全缺陷。
