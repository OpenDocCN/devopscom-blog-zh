# 将秘密管理应用于开发运维

> 原文:[https://devops.com/applying-secrets-management-to-devops/](https://devops.com/applying-secrets-management-to-devops/)

DevOps 改变了企业的运营方式，允许快速高效的产品开发。为了让公司跟上市场竞争，许多在结构上没有准备好实施 DevOps 的公司仍然会组建一个团队来满足需求，这反过来又会在其安全态势中产生漏洞和缺口。

今天，只有 46%的 IT 专业人员在开发的初始阶段面临安全风险。将安全性作为事后的想法，DevOps 团队可能会忽略甚至最基本的安全实践。此外，在一个严重依赖代码的环境中，我们一次又一次地看到粗心的开发人员通过 GitHub 等网站上的 API 或密钥泄露机密信息。作为代码的圣地，它是网络犯罪分子获取开发人员存储在其平台上但在生产前忘记删除的凭据的主要场所。

虽然 GitHub 漏洞继续将焦点放在 DevOps 安全上，但大多数组织仍然没有看到全局。现实情况是，安全卫生仍然至关重要，开发人员可能会因为忽视基本的安全实践(如监控访问和嵌入的机密，如密码、密钥和 API)而将他们的组织置于风险之中。

## DevOps 团队应该从哪里开始？

解决方案很简单:将非常有价值的秘密与开发人员分开。幸运的是，精简秘密的关键不一定要费时费力。事实上，机密管理的核心是特权访问管理(PAM)的扩展，许多企业已经有了这种管理。

作为保护组织免受不良行为者侵害的重要方法，降低开发运维相关数据风险的最快方法之一是补救与 PAM 相关的薄弱安全措施。

以下是 DevOps 团队应该应用于其交付渠道的一些最重要的安全协议:

## 清点特权帐户和访问权限

拥有数千台服务器和网络设备的大型企业通常缺乏准确的资产清单。由于不充分的取消配置流程，大多数组织不了解其系统中的凭据，从而造成了安全漏洞。DevOps 环境对自动化和脚本的使用使得管理特权帐户和访问变得更加复杂。

为了确保 DevOps 实践在初始配置阶段之后仍然安全，安全专业人员必须了解自动化存储在何处，以及自动化中存储了哪些嵌入式凭据。通过了解整个嵌入式特权凭据流程，坚定地跟踪每个 DevOps 团队成员的访问权限并利用自动化，企业可以在用户角色发生变化时即时更新用户的访问权限，以确保他们拥有正确的访问权限来完成工作，仅此而已。

## 机密管理集成

DevOps 团队在管理特权凭证时面临的一个重大挑战是找到一个可以无缝集成到现有开发工具集中的解决方案。

许多 DevOps 团队在开发过程的不同阶段依赖于几种不同的工具，并且根本不能使用不集成和处理所有这些工具的秘密管理解决方案和过程。例如，如果一个企业投资了 AWS，并且有一个特定于 Azure 的秘密管理解决方案，那么繁琐的步骤可能会被添加到流程中。在一个关注快速开发的环境中，开发人员很可能会找到方法来解决这些会给企业带来风险的不便。

为了确保实施有效的机密管理策略，企业需要了解它必须与 DevOps 工作流中的每个工具配合使用。通过将安全系统与开发人员熟悉和使用的工具相集成，企业可以确保没有捷径会将敏感信息置于风险之中。

## 了解 DevOps 管道

传统上，企业认为获取秘密的唯一方式是登录服务器，但在 DevOps 中并非如此。在 DevOps 中，当拥有特权的用户输入一段代码时，它会通过一个管道，而许多企业并不知道管道中发生了什么。这个盲点可能会给组织带来严重的网络风险。例如，如果三个开发人员正在处理一个项目，他们签入需要特权访问的代码，例如复制信用卡信息，开发人员可能会无意中在该项目中插入危险的代码，而安全团队对此一无所知。

对于专注于个性化操作的环境，应用秘密管理并再三考虑谁有权在这个管道环境中进行构建是至关重要的。由于不了解管道如何工作，也不知道谁有权访问特定数据，企业增加了内部违规或数据泄漏的风险。

事实上，管理 DevOps 安全与实际的 DevOps 流程没有任何关系。通过集成基本的特权管理协议，企业可以确保安全性处于开发过程的最前端，而不是后端。
