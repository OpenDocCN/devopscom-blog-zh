# 用于可观测性的开放式遥测技术的未来

> 原文:[https://devo PS . com/the-future-of-open telemetry-for-observability/](https://devops.com/the-future-of-opentelemetry-for-observability/)

可观测性的前景正在发生变化。当今的数字化转型之旅凸显了对可观察性的需求，因为组织希望捕获、分析和关联应用程序堆栈每个级别的所有数据，包括日志、指标、跟踪和事件。随着[数据](https://devops.com/how-developers-can-handle-the-finserv-data-deluge/)的迅猛增长，收集这些信号对于商业洞察和系统优化都是必不可少的。

在过去的几年中，开源数据收集让软件和工程团队可以自由地决定他们希望数据收集标准化的技术，以及他们希望将数据发送给哪些供应商。虽然开源数据收集带来了很多自由和优势，但也产生了一些新的问题。缺乏一个单一的、全面的框架来跟踪所有关键信号的可观测性给开发人员带来了很多摩擦。随着我们迈入新的一年，企业正在借此机会思考如何更好地利用数据和应用智能来帮助客户做出决策，并筛选所有噪音。输入 OpenTelemetry。

## 什么是 OpenTelemetry？

OpenTelemetry 是一个工具、API 和 SDK 的集合，用于检测、生成、收集和导出所有机器数据；它诞生于为软件工具开发行业标准的需要。通过利用单一的云原生框架来完成可观测性，OpenTelemetry 正在成为收集机器数据的黄金标准。

OpenTelemetry 由三个主要组件组成:规范、收集器和仪器。该规范使用 API、SDK 和语义约定以及 OpenTelemetry 协议来确定所有实现的跨语言要求。收集器不再需要多个数据收集代理。它是一种供应商中立的方法，用于接收、处理、转换和导出数据。仪器库允许用户使用自动仪器和来自多种框架和语言的支持将其应用与 OpenTelemetry 集成。它们一起为公司提供了一个健壮的框架来破坏他们现有的可观察性实践，并进一步放大它们。

## 一切都是为了顾客

大多数现代企业都有复杂的分布式微服务，由数十个、数百个或数千个组件组成，这些组件协同工作以提供客户体验。如果组织不能轻松地在这些组件或不同信号之间建立关联，将会降低企业有效监控、诊断和解决问题的能力。T3】

多年来，组织依靠多个基础设施供应商收集、丰富数据并将其发送给供应商，从而形成了[可观察性](https://devops.com/?s=observability)。尽管如此，许多人仍然没有最大化可观测性最显著的能力:关联复杂的系统和信号。T3】

由于最终用户希望深入了解他们的数据，他们也希望供应商能够提供最佳体验，在可观察性的帮助下识别和解决他们系统中的问题。通过真正的可观察性，组织可以让客户有时间专注于重要的事情，即管理客户的数字体验，并赋予他们满足客户需求的能力。

## 类似的收养之路

就像 Kubernetes 对于软件容器的日益普及至关重要一样，业界已经见证了围绕 OpenTelemetry 实现可观测性的相同采用方法。 [Kubernetes 赢得了成为容器编排标准的战争](https://containerjournal.com/editorial-calendar/could-nomad-reignite-orchestrator-wars/)。现在，OpenTelemetry 将带来可观测性所需的相同标准，全面了解堆栈每一层的健康状况和性能。T3】

OpenTelemetry 已被软件开发社区广泛采用，但它仍处于旅程的开始。开发人员社区使 Kubernetes 取得了成功，随着客户要求更好的集成和更少的锁定，我们将看到 OpenTelemetry 取得同样的成功。对于涉及应用和软件的供应商来说，从应用性能管理等更传统的方法到软件开发管道和管理工具，再到云原生服务，支持 OpenTelemetry 将是未来的赌注。

凭借广泛的可移植性、更强的开发者控制力以及强大的供应商和云提供商支持，OpenTelemetry 通过帮助改善整个应用基础设施(从软件开发流程到安全等领域)的可实现结果，为客户打开了一个新的世界。

企业如何在平衡不断增长的客户需求的同时，继续推进可观察性？很简单:如果组织现在不考虑 OpenTelemetry，他们就会因缺乏适当的可见性而失去竞争优势，无法实现当今时代客户期望的服务水平目标。
