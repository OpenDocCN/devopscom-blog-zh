# DevOps 原则、实施和文化

> 原文:[https://devo PS . com/devo PS-principles-implementations-and-culture/](https://devops.com/devops-principles-implementations-and-culture/)

当组织开始使用 DevOps 进行应用程序开发时，许多组织仅将 DevOps 作为其基础架构的一部分，并在了解成功所需的文化和流程变化之前选择 DevOps 的子集。相反，组织必须首先确定哪种“运营”方法最能促进开发运维心态，并将他们的工作流融入他们的文化。通过这样做，这些组织可以更好地将他们的开发和运营团队与共同的目标和协议结合起来。

DevOps 有许多变种和亚属。但是，重要的是组织要了解可用的“运营”选项，并选择最适合他们的选项。

## **原理与实现差异**

一般 DevOps 及其亚属可以分为两类。第一个也是“最纯粹的”是这些术语专注于将两个或更多行业与运营统一起来的原则。

第二种类型是关注实现类型的。

## **原理类型**

**DevOps:改善技术流程和文化**

DevOps 是运营的总括术语，它将敏捷开发概念与 IT 运营相结合，并在云环境中快速跟踪应用程序的开发和部署。虽然 DevOps 有不同的子类，但每个子类都是 DevOps 方法的不同应用，都是为了使开发更加敏捷、稳定和高效。因为 DevOps 大大缩短了开发生命周期，所以这种简化的应用程序开发方法非常流行。

**DevSecOps:添加安全检查**

DevSecOps 在工作流中加入了额外的安全措施。DevOps 通常在周期结束时进行安全检查，利用 [DevSecOps](https://devops.com/?s=DevSecOps) 的 IT 专业人员在 DevOps 旅程中尽早集成安全性。使用 DevSecOps，开发人员、安全专家和运营人员之间的协作对于保持敏捷性和交付速度至关重要，但额外的安全层会降低流程速度。DevSecOps 利用快速测试和健壮的管道来支持更好的代码完整性。如果组织需要在开发过程中提升安全性，这种方法可能是有益的。

**NoOps:减少内部运营团队**

NoOps 是一个崇高的愿景，即 DevOps 是如此自动化，以至于不需要内部专业知识，这意味着在流程中不需要人工干预。NoOps 不同于 DevOps，因为它不太关注敏捷和流程管理。这个过程假设开发人员不需要知道如何使用自动化工具，并使用持续集成技术来允许开发人员只关注应用程序开发。

尽管 NoOps 加快了部署过程，但如果单独使用，NoOps 缺乏开放的交流和团队管理。如果一个组织没有崇高的技术目标，或者如果它的开发团队很小但是很强大，并且能够处理开发和操作，那么 NoOps 的成功实现是可以实现的。

**GitOps:使用 Git 实现自动化**

在过去的两年中，GitOps 在开发行业中获得了越来越多的关注，因为它使用 Git 来自动化连续交付管道。GitOps 是一种健壮且可管理的操作形式，因为它使用 Git 作为操作的唯一真实来源。此外，由于开发人员熟悉 Git 和 pull 请求，因此没有复杂的工具需要学习，对现有工作流的更改通常是不必要的。

GitOps 方法还可以夸耀它得到了市场上一些最具创新性的云服务的支持，从而自动化了构建更新、测试错误、审查代码和将更新推向生产的过程。因为 Git 是 GitOps 的唯一来源，所以有效地加强 Git 存储库以避免不必要的提交或拉请求是至关重要的。

**CloudOps:实现云原生特性**

CloudOps 试图更多地利用现代服务提供商提供的云原生功能。由于 CloudOps 专注于分布式开发和部署，因此不存在单点故障，从而使整个云环境更加可靠。此外，在这种类型的操作中，工作流的某些部分可能会变成无状态的，这使得成本效益降低，因为组织只需为他们实际使用的资源付费。虽然 CloudOps 可以节省时间和成本，但组织必须获得正确的配置才能使该方法受益，这可能意味着配置 CI/CD 管道会更加复杂。

**CIOps:要求 CI 操作员进行配置**

持续集成运营(CIOps)是 DevOps 的最新版本，要求 CI 运营商在部署前配置支持新代码所需的基础设施。确保 CI 作业的正确配置和部署位置所需的手动输入既有优点也有缺点。对基础架构的控制允许两个部署具有不同的基础架构配置。主要缺点是所需的手动干预为错误留下了空间，并且将组织的 CI 工具链接到其 API 的需要会带来安全风险。

CIOps 更适合自动化不便的小型开发环境。虽然 CIOps 为开发人员提供了配置云基础设施的灵活性，但这种方法需要时间和精力才能顺利运行。

**ITOps:自动化 IT 管理管道**

它被宽泛地定义为最终用户或员工系统的管理。它们是运行核心业务所需的系统。最根本的区别是几乎没有发展。如果您的企业不需要软件工程团队，并且可以单独运行预构建的应用程序，那么 ITOps 就是您的选择。

ITOps 负责交付和维护业务运营所需的服务、应用程序和底层技术，包括系统管理员和网络管理员等工作。ITOps 专注于更新笔记本电脑和办公网络，以及用于构建基础设施的经过高度测试的商业软件和解决方案，这使得更新基础设施组件变得更加复杂。当使用 ITOps 时，组织的首席信息官将专注于通过自动化来管理系统。

**为组织选择最佳运营方案**

在各种 DevOps 之间进行选择，就是找到一种最适合组织正在开发的应用类型和他们使用的云基础架构的方法。尽管上述所有运营战略都基于独特的理念，但它们往往侧重于技术层面。相反，DevOps 以其全面改进工作流和简化技术流程的能力脱颖而出，同时在组织中培养创新和协作文化。
