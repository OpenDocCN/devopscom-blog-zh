# 桥接 AppSec 和 DevOps 断开

> 原文：<https://devops.com/bridging-the-appsec-and-devops-disconnect/>

[研究估计](https://cybersecurityventures.com/hackerpocalypse-cybercrime-report-2016/) 到 2025 年，网络犯罪将使全球每年损失 10.5 万亿美元，因此网络安全成为企业领导人的首要任务也就不足为奇了。

如今，安全团队正努力强化其系统以抵御[网络攻击](https://securityboulevard.com/log4j-threatens-well-everything)，并通过更多的员工安全培训、事件响应计划以及实施监控和检测威胁的技术来提高其弹性。

许多组织也采用了左移安全策略，即在应用程序开发生命周期的早期解决安全性问题。这些组织意识到，在软件发布后发现其安全问题风险更大，成本更高。毕竟，漏洞存在的时间越长，攻击者找到并利用它的时间就越长。通过实施左移安全策略，如果在开发阶段就解决了安全问题，攻击者的机会窗口就会减少或完全消失，同时还能为组织带来显著的安全优势和成本节约。

左移是一种安全测试策略，开发人员和安全团队协同工作，在早期开发阶段识别和解决漏洞。这一过程极大地提高了安全性，因为它意味着在将漏洞引入企业环境之前就识别并缓解了漏洞。

然而，组织在采用左移安全策略( [DevSecOps](https://devops.com/?s=devsecops) )时面临的最大挑战之一是鼓励安全团队和开发团队合作。

## 鼓励 AppSec 和 DevOps 合作

直到最近，DevOps 团队和 AppSec 团队还是完全孤立的。DevOps 团队致力于加快应用程序发布到企业中的速度；应用程序安全团队只是在后期才参与进来。然而，这个过程已经引起了安全问题，特别是当更多的组织转向敏捷开发的时候。

当安全团队最终获得新的应用程序和更新时，不可避免地会发现安全缺陷，这导致了开发运维团队的延迟和摩擦。

然而，随着组织开始接受 DevSecOps 和左移战略，这两个以前孤立的团队被迫更加紧密地合作。那么，怎样才能减少摩擦，消除隔阂呢？

在文化上，一些根深蒂固的态度和行为挑战着任何开发合作努力的成功。安全团队看到开发运维流程加快了软件交付的速度，但没有考虑安全因素，而开发运维团队则经历了安全问题导致流程变慢，并给出不一致的结果和关于安全问题的反馈。每一方都有自己要讨好的经理；他们自己的一套衡量标准和一个优先级列表。这两个团队遵循不同的过程，关键的是，使用不同的工具。开发人员无法回避安全工具的复杂性，并且缺乏与现有工具集的集成，安全团队无法控制 CI 渠道以最佳方式实施安全保障。

克服这种摩擦的最佳方式之一是通过更好的技术、流程和文化来实现团队间的协作。首先，DevOps 团队*确实*关心安全性，但它可能在他们的优先级列表中较低。安全团队必须理解 DevOps 团队关心代码、质量和效率。他们不是安全专家，仅仅因为一个组织正在左移并不意味着他们需要成为一名安全专家。相反，组织应该为 DevOps 实现工具，在 devo PS 中可以很容易地嵌入安全检查。

那么，组织可以做些什么来改善 AppSec 和 DevOps 团队之间的协作以实现 DevSecOps 呢？

DevSecOps 是一种文化，它需要思维的转变。安全团队和开发团队需要了解彼此的优先事项，并明白通过合作，他们正在提高组织的整体安全性，并降低网络攻击的风险。DevOps 团队是否了解漏洞可能给整个组织带来的潜在风险？如果没有，安全团队需要教育他们。

当组织转向 DevSecOps 时，安全自动化工具提供了强大的支持，因为它们允许在开发应用程序时自动标记错误和漏洞。这将允许在服务上线之前解决这些问题；但是，由于它们会频繁生成警报，这可能会增加开发运维本已繁忙的工作负载。通过部署可与 CI/CD 管道同步的工具并提供定制咨询，开发运维团队可以自己解决问题，而无需求助于应用安全团队，这可以显著加快流程、减少摩擦并减轻工作负载。
