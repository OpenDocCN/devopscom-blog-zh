# 驾驭 MLOps 革命:要避免的三个失误

> 原文：<https://devops.com/navigating-the-mlops-revolution-three-missteps-to-avoid/>

到 2024 年，将近 60%的企业将采用 MLOps，这一功能已经迅速成为企业技术领域不可或缺的一部分。随着数字化转型工作的继续，计算需求迅速扩大，MLOps 市场的增长不太可能很快停止。

也就是说，与任何新兴技术一样——尤其是像 MLOps 这样强大的技术——如果企业及其技术团队行动过快，他们可能会陷入许多潜在的陷阱。更重要的是，这些失误会对公司创收和优化绩效的能力产生重大影响。

考虑到这一点，以下是公司在采用 MLOps 时容易犯的三个最大错误。

### 没有试水就跳进来了

如果您怀疑自己的 MLOps 成熟度，您可能应该等待。破坏 MLOps 采用和长期成功的最大因素之一是，公司在实际准备好采用该技术之前就跳了进去。这不仅导致 MLOps 感觉像一个负担，而且经常将客户-供应商关系转变为交易销售关系，而不是它应该成为的伙伴关系。Onboarding 技术需要是一个客户和供应商之间不断对话的交换过程。否则，供应商——即使是最有道德的供应商——也不得不依靠局部的情景情报和猜测来提供解决方案，这些解决方案可能会也可能不会解决问题并交付结果。克服这一点的最佳方法是，公司明确指出他们面临的 MLOps 挑战，存在的障碍，以及他们期望的结果。通过提前了解并同意这些关键组件，公司不仅可以确保获得最佳的供应商合作伙伴，而且供应商还可以提供更加量身定制的指导。

### 误解了什么是 MLOps

新采用者倾向于将 MLOps 归结为由两个组件组成的理念:存储和计算。是的，这是 MLOps 等式的两个重要元素，但这种观点过于简单化了。此外，如果没有在一开始就澄清，这种误解可能会导致组织对 MLOps 感到不知所措，从而降低采用的“粘性”。

事实是，MLOps 由许多部分组成，从实验到部署，每一部分都有其特定的细微差别。这些部分中的每一个都与下一个部分相互关联，如果没有正确的计划和维护，都可能破坏 MLOps 的结果。此外，确保每个必要的组件都以应有的方式运行，以满足每个客户的需求和期望，这需要时间。因此，如果企业及其更广泛的 DevOps 团队希望以一种面向长期可持续结果的方式成功采用 MLOps，他们需要做好耐心的准备。

### 没有意识到 MLOps 的迭代次数

当从[传统 DevOps 的角度](https://devops.com/mlops-vs-devops-whats-the-difference/)来看项目时，人们普遍认为如果一个项目足够稳定，可以进入 DevOps，那么它就可以投入生产了。然而，对于 MLOps 项目，情况就不一样了。与 DevOps 相比，MLOps 更倾向于发展——这意味着迭代和重复非常普遍。例如，在 MLOps 过程中，公司想要迭代现有的过程，有时比他们的软件发布更频繁。这意味着 MLOps 团队和其他团队之间有更多的持续联系和协作。这种互动与传统的 DevOps 项目形成了鲜明的对比，在传统的 devo PS 项目中，一旦生产完成，开发人员就交出钥匙，基本上就不管了。企业及其传统 DevOps 团队需要适应这种新的业务方式，并建立灵活性和回旋余地，以确保他们的 MLOps 能够正常运行和发展，并以适合他们的方式发展。

企业有理由对采用 MLOps 的前景感到兴奋。然而，为了立即投入运营并在短期、中期和长期产生影响，公司必须部署一个全面的战略来缓解入职难题，并使流程尽可能无缝。记住这些关键的失误，企业就能做到。
