# 长期优化 AWS 性能

> 原文：<https://devops.com/optimizing-aws-performance-over-the-long-term/>

问大多数人如何在 AWS 上优化性能，他们会谈到自动缩放和合理调整之类的事情。

这些工具当然是有效的 AWS 性能优化策略的一部分。然而，它们本身不足以保证长期的性能和可靠性。

这是因为自动扩展和合理调整等工具和流程只能帮助您优化当前正在运行或即将部署的应用的性能。他们几乎无法确保性能保持最佳，尤其是在云服务、成本结构和架构不断发展的情况下。

考虑到这一挑战，我们来讨论一下如何从长期来看充分利用 AWS 的性能。

## 什么是 AWS 性能？

在深入研究长期 AWS 性能策略之前，让我们先弄清楚我们所说的性能是什么意思。高性能应用程序是指:

*   满足或超过用户对响应能力和速度的期望。
*   满足或超过 SLA 对可用性的要求。
*   能够随着性能需求的变化而无缝扩展。
*   以经济高效的方式完成以上所有工作。

强调高性能 AWS 应用程序的这些方面很重要，因为它们反映了性能的整体方法。有时候，性能的定义很狭隘，就应用程序响应性和/或可用性而言。这些是性能等式的一部分，但是为了长期最大化性能，您需要考虑性能的其他维度，例如可扩展性和云支出所扮演的角色。

## 长期 AWS 性能考虑因素

为了长期优化 AWS 性能，您必须考虑各种因素。

**处理 AWS 服务变更和升级**

AWS 在不断发展。如果 AWS 引入了新的服务或者改变了现有服务的特性，那么今天有效的性能策略明天可能就不再有效了。

这里的挑战不仅仅在于更新您的应用程序和云配置，因为 AWS 改变了特定服务的工作方式。它还需要不断评估您正在使用的服务是否最适合您的应用程序。

例如，回想一下五年前 AWS Lambda (AWS 的无服务器计算服务)出现的时候。当时，如果您在虚拟服务器中运行您的应用程序，您可能能够通过将一些工作负载转移到无服务器功能来提高整体性能，这不是因为您的虚拟服务器中的任何东西变得效率低下，而是因为随着新 AWS 服务的增加，更高的效率机会出现了。

因此，不断了解 AWS 中的各种服务及其提供的机会是优化长期性能的关键。

**改变成本结构的计划**

AWS 不仅经常改变其服务项目，还会更新定价。由于成本优化是性能优化的一个关键组成部分，因此即使成本结构发生变化，您也必须确保您的云工作负载以最经济高效的方式运行。

当新的云服务(如低成本的 S3 存储层)出现时，保持警觉是实现这一目标的一种方式。但是，您还需要了解极其复杂的 AWS 定价世界的所有地区特定细节。

**为 OSS 依赖关系制定长期计划**

开源软件是另外一种不断变化的东西。如果您依赖开源代码来帮助运行您的 AWS 应用程序，您需要掌握您所依赖的项目的变化。

如果您使用的是第一次部署时构建到应用程序中的相同版本的开源应用程序或模块，您可能会错过此后出现的性能优化机会。

同样，对复杂的开源生态系统的持续了解和可见性对于确保您不会错过优化机会至关重要。

**特定应用性能监控**

每个应用程序都是一片特殊的雪花，在跟踪和优化应用程序性能时，您需要记住这一点。

用于优化一个应用程序性能的指标或 KPI 对于另一个应用程序可能并不理想。您用于应用程序的部署前测试例程也可能有所不同。

因此，请确保根据您的应用定制您的监控策略。这样做是确保您对应用程序性能有最大、最相关的长期可见性的唯一方法。

**备份和灾难恢复**

无论您的 AWS 工作负载执行得多好，迟早会出错，它们会失败。故障是会导致严重的可用性中断还是仅仅是一个小问题，这在很大程度上取决于您是否事先制定了备份和灾难恢复计划。

对于 AWS 应用程序而言，有效的备份和恢复计划不仅仅是将数据备份到存储桶并下载以重建服务器。它还应该利用这样的机会:创建虚拟服务器的映像级备份，并在灾难发生时自动恢复到 EC2。

当然，随着 AWS 的变化，您需要保持备份和恢复策略同步。

**改变 SLA 要求**

最后但同样重要的是，记住你的 SLA。您现在必须支持的 SLA 在未来可能会发生变化，因此，您的用户所期望的可用性和响应级别也会发生变化。

因此，请确保您的 AWS 性能策略不仅能够满足您当前已有的 SLA，而且能够随着 SLA 要求变得更加严格而进行调整和扩展。

## 结论

虽然我们倾向于在当前的背景下考虑绩效，但它实际上是一个长尾游戏。当然，您可以自动调整您的工作负载，或者使用合适的调整工具来帮助选择最佳类型的 EC2 实例。但是，真正实现性能最大化需要您进行长期思考，并准备好在新的性能挑战和机遇出现时灵活应对。
