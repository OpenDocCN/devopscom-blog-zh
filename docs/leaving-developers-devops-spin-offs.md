# 我们是否将开发者排除在 DevOps 衍生产品之外？

> 原文：<https://devops.com/leaving-developers-devops-spin-offs/>

秘密警察。数据操作。网络运营。阅读这些术语，您会觉得 IT 效率的关键是让 IT 运营与其他所有人合作。但这是一个错误，因为它将开发人员排除在外。

众所周知，DevOps 运动催生了许多其他*Ops 计划，包括上面列出的那些。所有这些背后的驱动理念是，IT 运营团队应该与安全团队(对于 SecOps)、数据分析团队(对于 DataOps)或网络团队(对于 NetOps)更紧密地合作。这些只是*Ops 活动的一部分；这个名单还可以继续下去。

可以肯定的是，让 IT 运营部门与组织的其他部门更紧密地合作会带来好处。但是和开发人员做同样的事情也有同样多的好处。

为了优化安全运营，您需要开发人员、IT 运营人员和安全团队通力合作。如果 Dev 被排除在外，这个等式的其他两个部分就很难确保组织部署和管理的代码是安全的。

出于类似的原因，只有当开发人员也参与时，数据操作、网络操作和几乎所有其他*操作才能正常工作。实际设计和编写应用程序的人很重要。

## 重新思考运营

为什么这很重要？我的观点不仅仅是像 SecOps 和 DataOps 这样的术语容易引起误解，因为它们没有提到开发人员。

相反，给予开发者同等的重视是很重要的，因为如果你不这样做，你就有孤立开发者的风险。有些人已经开始抱怨 DevOps 正在“杀死”他们。定义新的*Ops 实践而不提及开发人员的趋势于事无补。

因此，当我们讨论如何将 DevOps 理念扩展到 IT 组织的其他部分时，也许我们应该开始更突出地提及和考虑开发人员。这已经在 SecOps 的案例中做了一些，你有时会听到它被称为 SecDevOps。但谷歌列出的“SecOps”搜索结果是“SecDevOps”的近 13.5 倍。如果您搜索“NetOps”与“NetDevOps”，数字是相似的我也从来没听过有人说“数据开发包”

要点是:对于开发人员来说，开发人员和 IT 运营人员一样重要。不要漏掉他们。
