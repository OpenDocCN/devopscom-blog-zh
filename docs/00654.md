# 从基础架构自动化转向真正的开发运维

> 原文:[https://devo PS . com/moving-from-infra structure-automation-to-true-devo PS/](https://devops.com/moving-from-infrastructure-automation-to-true-devops/)

*决定两个独立的团队，还是一个完整的端到端团队*

虽然关于开发运维的“正确方法”已经说了很多，但我们在实践中看到的是两种不同的方法，这两种方法试图使开发和运营作为一个团队一起工作。一方面，有一种创建端到端的集成开发团队的方法，您可以称之为“Devops”。另一方面，还有“DevOps ”,由两个开发团队组成，一个团队开发应用程序代码，另一个团队在虚拟化环境中使用基础架构作为代码技术开发平台。

**DevOps:提供平台**

也许最显而易见的方法是有一个平台团队来提供一个“能够正常工作”的运行时环境，并有一个独立的应用程序团队来开发在平台上运行的业务逻辑。

对于一些组织来说，尤其是其中一个团队是外包的，这种分离会更有意义。您可能会购买现成的产品，在内部构建自己的平台(这似乎是许多组织目前正在寻求用 Docker 做的事情——警告:这比看起来要难！)…或者与 PaaS 供应商签约。

无论在什么情况下，平台都应该被版本化，就像任何其他软件一样。

这种方法对开发人员来说可能是理想的，因为他们可以自由地处理代码并专注于业务逻辑，他们真的不需要担心它在生产中如何运行。运营团队负责确保平台稳定，并交付开发人员需要的东西。

**沟通仍然是关键**

即使在物理或组织上分离，两个团队之间有效对话的重要性也不会被夸大。如果这种关系是敌对的，这是行不通的。必须真正意识到共同的目标。如果运营团队不明白开发团队要去哪里，以及他们需要从平台得到什么，那么就会有麻烦。一个平台可以满足所有开发需求的想法通常是不现实的，因此版本控制和并行运行多个平台版本的能力，或者满足特定应用程序可能需要自己的堆栈的特殊情况，是非常重要的。

必须有一个明确定义的边界，它应该靠近应用程序代码和配置层。应用程序处理业务逻辑，平台处理部署、日志记录、监控、故障转移和自动扩展。

开发团队可以专注于将这些新功能和改进融入到实际产品中，而运营团队则负责实际的推广和流量管理。要做到这一点，平台必须让您能够快速确定谁应该对突然出现的任何问题负责。值得在设置中投入时间，以确保错误是显式的，这样就不会浪费时间去追踪它们。

两个队之间的争论不利于进步和士气。

**Devops:全栈开发团队**

对于一些企业来说，将集成开发团队放在一起更有意义，他们直接从共享代码库交付完整的堆栈，从虚拟机到应用程序代码。这允许您以考虑应用程序的特定视角和需求的方式来设计和优化每个系统，如果您的应用程序在操作系统、中间件等方面存在很大差异，这可能是一个特别好的选择。，他们运行。

这种方法只有在整个团队很小(亚马逊的“两个披萨”团队就是一个很好的例子)并且有足够的知识和经验来看到整个系统的全貌时才能奏效。

不可能每个人都详细了解整个堆栈的每个部分，但是有一些重叠的领域，在这些领域中，更好的协作将提高共享知识的水平，并允许团队构建更好的整体系统。

**齐心协力**

Devops 方法的共享基础建立了相互尊重，并实现了更自由的交流，但它必须基于共享的目标和对系统总体目的的愿景。这种集成的方法可以给你更多的灵活性，但也意味着让整个团队负责保持一切正常运行。共享代码等于共享责任。

**正确的方法**

您需要设定一个基线，然后制定度量标准来跟踪您的进展。你心中有明确的目标，并有一个衡量目标的系统。那么，如何找到实现这些目标的正确方法呢？

说一种方法比另一种好是错误的。你需要做的是评估什么将在你的组织中起作用。两者都是实现相似目标的手段，但是找到正确的方法是关于理解你的公司想要实现什么。你可以选择共同的目标，比如减少开支，或者减少错误，或者你可以选择更具体的行业目标。一开始就确立你的目标，并确保它们是可以衡量的。

正确的方法将是为您获得结果的方法——这就是为什么设置基线和指标如此重要。你可能会发现这两种方法的混合将会是正确的选择，但是在你有一些硬数据来支持你的期望之前，你不能确定。
