# Linux 基金会提供软件清单工具

> 原文：<https://devops.com/linux-foundation-makes-available-software-manifest-tool/>

Linux 基金会正在提供一套免费的工具来构建基于它所管理的软件数据包交换(SPDX)文件格式的软件材料清单。

在 20 多个组织的支持下，SPDX 致力于标准化描述软件包内容的元数据的描述方式。

SPDX 规范是去年秋天由联合开发基金会(Linux 基金会的一个分支)提交给国际标准化组织(ISO)的。

作为努力的一部分，Linux 基金会还提供了一个免费的在线培训课程，为 IT 团队提供创建和维护 SBOMs 所需的基础知识。

Linux 基金会的可靠嵌入式系统副总裁 Kate Stewart 说，SBOMs 在维护软件供应链的安全方面起着一个关键的，但经常被忽视的作用。当今组织所面临的一个主要挑战是 SBOM 并不经常被创建，这使得发现软件包中包含了哪些软件组件变得更加困难。Stewart 指出，这使得组织很难确定他们使用的软件组件版本是否存在最近刚刚报告的严重漏洞。

即使有一个 SBOM，它的结构也会因团队或组织的不同而不同。Stewart 补充说，Linux 基金会提供的工具旨在标准化用于描述 SBOM 的字段的定义方式。

Stewart 补充说，由 Linux 基金会开发的 SBOM 工具在某种程度上是对拜登总统关于软件供应链安全的行政命令的回应。因此，国家电信和信息管理局(NTIA)正在广泛征求反馈意见，以确定 SBOM 的最低要求。

达成一致的 SBOM 标准可能需要时间，但随着基于容器的微服务变得越来越流行，这个问题显然会变得更加紧迫。此外，在没有清单的情况下在企业中部署更小的软件包只会进一步加剧应用程序安全危机，这一危机在最近高调的软件供应链违规事件之后已经加深。

最终，SBOM 应该作为任何 DevOps 工作流的一部分自动生成。挑战在于，在任何正式的 DevOps 工作流程之外，仍然有大量的软件被创建。最重要的是，企业 IT 环境中已经部署了大量缺乏文档记录的遗留应用软件，这表明缺少简单的 SBOM 是一个多么大的问题。

还不清楚组织内部谁应该负责标准化 SBOMs。挑战在于，在许多组织中，这不是那种开发人员会得到回报，更没有动力去解决的问题。然而，从应用程序安全问题到源于软件开发项目重复的大量技术债务都可以追溯到缺乏一个简单的清单。

当然，现在许多组织开始意识到他们是多么依赖软件，更多的问题是关于软件是如何构建的。许多问这些问题的企业高管将要发现(带着一些怀疑)的事情是，组织所依赖的软件中有多少仍然是使用中世纪工艺而不是生产线来构建的。
