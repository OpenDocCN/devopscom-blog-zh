# 通过独一无二的生产途径复兴 CMDB

> 原文：<https://devops.com/devops-resurrecting-cmdb-thru-singular-path-prod/>

变更一致性(或如何部署变更的可预测性)以及变更透明度(或在软件开发生命周期[SDLC]的任何阶段对变更状态的可见性)是采用高功能企业级 DevOps 服务集的两个主要优势。大多数组织都试图将自己从种类繁多的手动交付系统转变为 DevOps 所依赖的基于模板的自动化交付模型。

在这个过程中，一个重要的事实很快浮出水面:我们不能允许对我们的环境进行“手动”调整，并保持我们想要的可预测性、可见性或稳定性。要做到真正的一致，流程必须完全自动化。一个依赖于人的交互的过程(无论是计划内的还是计划外的)，会受到影响其成功的人的可变性的影响。简而言之，我们不希望部署过程中有人跟着来“清理”

因此，DevOps 的一个关键原则是，它成为“生产的唯一途径”我们希望生产中的所有变更都经过我们的 DevOps 系统和流程。我们的目标是在每一次互动中减少和消除人类的依赖。例如，我们自动化测试执行，然后测试评估，然后测试门控变更进展。我们使用“智能”部署技术来覆盖目标环境中的不同之处，迫使它符合我们声明的“应该”存在的变更版本。

实际上，我们“踩”变体，每次我们启动部署时都覆盖它们。这很快就教会了我们的自由式工程师，他们在运行中做出改变，他们的工作会随着每个新的部署而被“加强”(可能是每天或每小时)。大多数工程师很快就意识到，为了保存他们的工作，他们需要将它包含在部署过程本身中(更新适当的配置文件，或者在部署执行自动化本身中添加正确的步骤系列)。否则，他们将面临每天重复任务的冲击，这不可避免地会压垮他们。

从这个意义上说，一个正确实现的 DevOps 系统是“自清洁的”与通过开发运维将所有变革投入生产的行政命令相结合，另一个好处出现了，可能会让首席信息官感到高兴。就在几年前，IT 组织还热衷于通过配置管理数据库(CMDB)来记录变更(从而更好地理解变更)。这些项目往往规模庞大，成本高昂，注定收效甚微。通常，CMDB 的成果在第一次实例化后的 90 天内都很棒，然后超过 6 个月，它的数据变得“可疑”，一两年后，它的数据变得陈旧，几乎无用。

简单的事实是，工程师(几乎各种各样的)对文档过敏。要求工程师遵守人类遵守文档的纪律就像要求太阳不要发光一样。无论提供什么样的诱因，或威胁惩罚，工程师只是对文档及其维护(可能是 DNA 的东西)保持有毒的过敏。

工业界试图通过生产“发现工具”来解决这个问题，这些工具可以自动确定我们在不同环境中的变化状态。这个想法是，如果我们定期运行发现工具，我们至少可以不时地更新我们的 CMDB 基线。这是一种改进，但是发现仍然缺乏变化的环境，并且通常不够彻底，不能产生超出资产位置和基本配置的真正价值。

进入 DevOps，它有一个固有的工件数据库功能，可以随着时间的推移跟踪构建、部署和发布的版本。DevOps 了解对工件识别、表达、编辑和历史的需求。大多数早期的 DevOps 工具都在产品中内置了某种形式的 it。但这是完美风暴成为可能的地方。

如果我们的 DevOps 系统实现了透明；也就是说，如果我们能够将从构思(在业务需求中)到产品发布(包括测试的历史，等等)概述的变更的上下文联系起来。)根据版本，我们现在有大量的财富可以塞进一个复活的 CMDB。我们可以自动维护 CMDB(不需要工程师手工操作来保持更新)。我们的 CMDB 变成了一个仓库，或者说是我们通过 DevOps 实现的变更工件的主仓库。

这不仅仅是基线或建立变革“黄金拷贝”的能力。它是通过开发版本建立“相对”黄金拷贝的能力，同时它在甚至高度复杂的 SDLC 系统的环境中前进。因此，我们将 CMDB 的价值以指数方式提升，超出了其在生产中的用途，但可以将其集成到 SDLC 流程中，并实时观察变化，无论我们移动变化的速度有多快或多频繁。

这应该是开创性的。即使首席信息官在一个结果不尽如人意的前 CMDB 项目上耗尽了所有的资金和政治资本，他或她也可以挽救剩下的部分，悄悄地使用 DevOps 来复兴 CMDB，使之成为推动变革的一个完整步骤。一旦首席信息官证明成功做到了这一点，他或她就可以向企业提供这一回收项目，并开始探索为企业创造 CMDB 估值的方法(超越资产位置和估值)。

出现的是一个以 DevOps 为中心的 CMDB。DevOps 已经成为生产的唯一途径，它对所有的变更进程都有洞察力和背景知识，并根据需要自动更新 CMDB(主工件库)。这样，集成的 CMDB 成为 DevOps 连续体的一部分。也许最大的好处或胜利是同时认识到工程师对这个可持续解决方案中已经包含的文档的过敏。我认为这是双赢——或者至少是对现实的承认。
