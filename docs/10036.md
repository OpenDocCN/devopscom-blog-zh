# 可判定性:衡量质量属性的 SLO

> 原文:[https://devo PS . com/dec ability-slos-to-measure-quality-attributes/](https://devops.com/decidability-slos-to-measure-quality-attributes/)

你可能对“特性”很熟悉——定义系统中所谓的“非功能性”部分的质量属性。列表中没有的是可决策性，这是最终的质量属性，因为它关注的是对你的团队和客户真正重要的东西。

你不能什么都做。又快又便宜又好？在现实世界中，我们有权衡。软件是不停吃饭的嘴，但它不是数字化的免费午餐。当我们决定*不*做什么时，我们给团队带来了更大的清晰度和生产力。

当然，这个世界充满了“应该”。我们知道，我们应该让我们的产品变得容易接近。它应该是安全的；应该是靠谱的。当然了。但是你如何决定你应该做的限度呢？要实现可决策性，您需要上下文。为了获得对你最关键的品质属性的上下文理解，你必须与真实的人交谈，并对你的情况进行推理。没有别的办法了。

### 新的设计，错误的决定

我有一次大开眼界的经历，亲身证明了质量属性并不一定能带来业务成果。六年前，我在为公立学校开发电子学习软件。主要问题之一是用户管理。我们向大型机构销售，这些机构通常只有一名管理员负责管理数百所学校的所有用户。当学校放假时，他们必须为学校输入数据，创建班级和分配教师。

一旦我们有了一个改进工作流程的设计(幸运的是，在实现它之前)，我们就把模型拿给客户看，展示我们所建立的奇特想法。他们向我们展示了“对他们来说是什么样子”——一个电子表格，让我们大吃一惊！他们用手复制/粘贴所有的东西。

所以我们问他们，“如果我们给你们一个上传电子表格的方法会怎么样？这能行吗？”当然，答案是肯定的！因此，我们放弃了构建一个漂亮表单的计划，而是创建了一个拖放输入字段来验证和上传表单。决定是明确的。

### 测量可判定性

我们如何衡量关键业务驱动因素并将其转化为可决策性？我发现最接近的答案是[服务水平目标](https://devops.com/how-it-ops-can-exceed-service-level-objectives-in-digital-transformations/) (SLOs)。这些方法将模糊性转化为指标和目标，从而快速做出决策。

SLO 是一种通用的工具，用于从一个指标中测量质量属性，该指标表示为给定质量(如可靠性或可伸缩性)的“好”事件的比例。随着时间的推移，可决策性来自于跨您的服务、地理位置、用户组和质量属性的 SLO 数据的汇总。

将一个指标与一个特定的目标(随着时间的推移，随着业务和工程环境的变化而调整)进行比较，我们可以得出一个允许的“坏”时间或事件的分配——一个误差预算。为可靠性、性能、可扩展性等设定误差预算。，允许您决定您的服务可以运行得多差，但仍能让客户满意。更进一步，你可以找到用户行为的突破点，主动捍卫你的服务不违反你所期望的质量属性。

鉴于你的情况，技术债务不可避免地来源于你应该关注但不能完全触及的所有方面。有些[技术债](https://martinfowler.com/bliki/TechnicalDebtQuadrant.html)是刻意的，甚至是谨慎的。为了解决技术债务，您必须决定暂停哪些新特性，以便在路线图中腾出空间。SLO 提供了一个数据驱动的基线，通过告诉您何时您的能力在误差预算内或超出正常范围，来快速准确地做出决策。

### 果断缩放

你可能开始于一个很少使用的小的、单一目的的项目。但如果情况发生变化，你必须决定如何投资、重新设计和维护它。突然间，您可能会关心性能、可伸缩性、安全性等等。首先，你需要果断，这样你就可以优先考虑你的工作。所以你把关心服务的人聚集在一起，你开始思考和讨论，并进行排名。您使用您的工具来构建指标，设定明确的目标，并尽一切努力保持系统在预期范围内运行。

基于错误预算的自动化行动是设置 SLO 的神奇之处。它们不仅仅是复杂的、高度可用的服务。当你不确定该做什么时，错误预算在系统中非常有价值。SLO 在你精心构建的应用或服务的所有质量属性和功能需求的所有权衡中创造决策性。

如果你把决策放在第一位，其余的就会水到渠成。
