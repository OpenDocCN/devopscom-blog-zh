# devo PS——运营贡献是否同样重要？

> 原文：<https://devops.com/devops-operations-ops-contribution-equally-important/>

在投身于 **DEVOPS** 潮流之前，我从关于**服务交付卓越**的两部分博客(一部分和两部分)开始了我的技术写作之旅，但现在当我阅读所有相关讨论时，我最终觉得好孩子总是开发人员，而运营人员是不必要的跟屁虫。

我们总是谈论最终目标是如何更快地发布，专注于通过持续部署增加价值，暗示运营是实现该目标的障碍，或者协作是游戏的名称，运营应该停止对抗，等等。

今天，我站在围栏的另一边，将对 OPS 团队是否需要任何改变来融入 DEVOPS 文化的问题进行判断。

由 ITIL、COBIT 或 [MOF](https://en.wikipedia.org/wiki/Microsoft_Operations_Framework) 定义的生命周期框架(在我担任 GRC 的最初几天后，以新的兴趣查看“微软运营框架”)在 IT 服务交付的所有阶段都非常相似，并且与 DEVOPS 的稳定、可扩展、精益和更快地向用户社区交付价值的原则一致。

ITSM 的广泛引用

*   关注服务而不是系统
*   面向过程的管理方法
*   持续改进作为其交付、支持和运营生命周期的一部分，

于是，问题出现了:

*   **服务交付框架(SDF)** 是否发生变化？
*   当我们继续与**的发展目标**保持一致时，我们是否必须重新审视运行良好的 **3P 的**(政策、流程&程序)？

从历史上看，在几乎所有大型组织中，运营[OPS]总是在孤岛中工作，对开发和测试团队交付高质量变更的信任度极低，对所有变更都持怀疑态度，因为可以理解的是，他们负责生产环境的稳定性和可用性最大化，并提供一致的用户体验。

另一方面，由于许多原因，开发人员面临着更大的压力，要以更快的速度交付特性和修复缺陷。更短的编码窗口，测试周期中的偷工减料，延迟性能/回归问题(是的，它发生了！)，当然，你的团队的努力不仅仅是衡量，而是生存取决于生产运行的数量，而不是质量、创新或改进。

“把一只死猫扔过墙”是有人如何描述新代码移交给 OPS 的，我不同意这种类比，因为 OPS 至少会期望它有三条腿，一个 [DOA](https://en.wikipedia.org/wiki/Dead_on_arrival) 会同样不受惩罚地被扔回来！

在可用性和对客户的支持方面，运营增加的业务价值远远超过了在“速度需求”中增加不完整功能的风险。另一方面，为了保持竞争力，我们必须努力实现一个目标，即以协作的方式稳定、持续、快速地交付功能。

我将 DEVOPS 定义为 ***“将相互依赖的变量(如人员、过程和工具)的正确组合结合起来，以实现快速和高质量的软件交付”*** ，运营对这一努力的贡献跨越了所有变量，为成功的计划增加了巨大的价值，对改进过程和交付更好的软件的挑战和共同关注有了新的见解。

DEVOPS 不仅仅是动态创建可扩展的基础设施，或实施更新的工具来实现快速和自动化部署，甚至是围绕监控创建 SWAT 响应。它同样是关于设想一个协作模型，隐藏在已被证实的原则和实践中，以便通过快速交付高质量的软件来增加商业价值。

那么，为了成为开发运维团队的有效贡献者，运维应该做哪些不同的事情呢？

首先，OPS 应该抓住在计划的生命周期之初就参与的优势，提供建设性的意见并利用他们在以下方面的专业知识:

*   评估基础设施影响和容量审查。
*   影响设计决策和运营需求。
*   审查所有服务承诺和 SLA
*   安全性和权限管理
*   强调操作程序的任何潜在变化。
*   更新监控和警报机制
*   提高自动化程度。
*   升级操作手册；服务台的培训，分享知识。

OPS 还可以对该版本的部署/测试过程中面临的问题保持严格的概述，并根据他们自己的操作就绪阈值进行衡量。

在我看来，这种合作的最佳结果是开发团队不需要“保修”或“初始支持”期。OPS 将完全了解这一过程，并与合作伙伴一起确保“卡特彼勒”通过了最基本的健康检查，并且从第一天起就完全适应支持模式的变化。

事实上，这只“猫”将不再被扔出长城，而是将正式通过移民检查！
