# DevOps 的“最后一英里”取得了巨大的收益

> 原文:[https://devo PS . com/big-gains-is-made-in-the-last-mile-of-devo PS/](https://devops.com/big-gains-are-made-in-the-last-mile-of-devops/)

mabl 的一项新研究发现，完全自动化的测试覆盖率与客户满意度和 DevOps 成熟度的显著提高之间存在关联。相比之下，那些具有中等 DevOps 成熟度和几乎自动化的测试覆盖率的人往往会经历最终用户满意度的急剧下降。

Mabl 最近发布了 DevOps 报告中的 [2021 测试状态，该研究强调了整个软件行业中 DevOps 采用和质量测试的状态。据 mabl 的联合创始人 Dan Belcher 称，研究结果清楚地表明，真正的业务回报是在采用 DevOps 的最后一英里产生的。](https://www.mabl.com/blog/defining-and-redefining-devops-success-mabl)

我最近会见了贝尔彻，听听他对主要外卖的看法。下面，我们将探讨 DevOps 报告中 2021 年测试状态的这些和其他见解。

## 债务成熟度状态

近四分之一的团队拥有非常成熟的 DevOps 方法和自动化流程。尽管如此，仍有相当一部分人(52%)仍渴望或努力完全采用成熟的 DevOps 模型。一小部分(11%)拥有完全自动化的 DevOps 软件开发渠道。大部分团队仍然使用手动部署流程和不同的框架，限制了完全自动化的潜力。

有趣的是，报告发现，非技术壁垒仍然是实现完全自动化开发管道的最常见障碍。在这些障碍中，不愿改变是最常见的障碍，有 29%的受访者提到了这一点。采用 DevOps 的其他挑战包括预算限制、DevOps 不是公司的优先事项以及内部政治。

合计起来，非技术障碍占阻碍完全采用 DevOps 的问题的 82%。一方面，这描述了与在组织中引入 DevOps 相关的文化挑战。另一方面，这可能意味着大量的 DevOps 包和开源工具可用于管理自动化部署和版本控制等领域。该报告发现 GitHub、Jenkins、Gitlab、Azure Pipelines 和 BitBucket 是最常用的五种 DevOps 工具。

虽然增长迅速，但[持续交付](https://devops.com/continuous-delivery-vs-continuous-deployment/)仍是一个新兴趋势；几乎一半的开发人员仍然每月不到一次部署代码。只有 15%的团队每天或者每天多次部署代码。这与 CDF 的[持续交付状态报告](https://devops.com/the-state-of-continuous-delivery-in-2021/)的发现相呼应，该报告将这一数字定为 11%。

## 全自动测试

测试是软件生命周期的重要组成部分。由于需要确保功能应用程序和高等级的安全性，自动化测试正在帮助团队在开发过程的早期发现更多的错误。根据该报告，最受欢迎的测试类型包括 UI 或功能测试、单元测试、回归测试、集成测试和端到端测试。非性能测试和视觉测试排名较低。自然地，已经实现全自动化管道的团队正在自动化更多类型的测试。

总的来说，测试自动化似乎与 DevOps 成熟度一致。该报告发现，那些采用全自动化测试的人获得了显著的净客户体验收益。例如，72%自动化回归测试的团队表示他们的客户满意度非常好或令人惊讶，而在没有自动化回归测试的公司中，这个数字下降到 6%。80%的高测试覆盖率团队报告高客户满意度，而只有 30%的低测试覆盖率团队报告高客户满意度。尽管如此，许多还没有完全自动化——类似于持续部署的状态，大部分团队依赖于自动和手动测试方法的组合。

就工具偏好而言，Selenium 是功能测试中最受欢迎的选项，有 41%的开发人员使用它。该报告还发现了向 Cypress、Smartbear 和其他功能测试工具的迁移。有趣的是，该报告还揭示了自动化 API 测试的明显缺乏——只有大约三分之一的人在自动化他们的 API 测试。Postman、SoapUI、internal tools 和 ReadyAPI 是运行手工 API 测试最流行的工具。

正如我之前所报道的，API 在[数字化转型项目](https://devops.com/the-role-of-apis-in-digital-transformation/)中的使用激增。Web APIs 是一种越来越常见的向合作伙伴交付数据和使用第三方软件即服务(SaaS)的机制。全面的 API 测试覆盖对于审计整个端到端的数字体验至关重要。[低代码 API 测试](https://devops.com/low-code-makes-api-testing-more-accessible/)是减少审计高技术集成领域障碍的一种方式。

## 关键要点

**开发成功是一个“最后一英里”的问题**。DevOps 的成熟度和测试覆盖率有很大的差异。直到你到达最后一点，当突破性的结果出现时，它是相当平坦的。“在 DevOps 的成功是一个‘最后一英里’的问题，”贝尔彻说。" 90%和 100%之间的差别是巨大的."

质量真的很重要。尽管很多技术讨论都集中在提高交付速度上，但是代码质量也同样重要。“这不是不惜一切代价的速度，而是速度和质量的创新，”贝尔彻说。"质量工程师和编写代码的开发人员一样是成功的关键."

**DevOps 的成功与其说是技术，不如说是人**。正如报告所发现的，DevOps 成熟的大部分障碍都是非技术性的。“作为一个行业，阻碍我们发展的更多的是领导力和解决我们的加工问题，而不是引入更多的技术，”贝尔彻说。

**打破角色孤岛仍然是 DevOps 的基础**。有些人认为 DevOps 有助于开放角色之间的协作。同时，有一种想法是新的工具仪表板将武装个人开发人员，使他们拥有开发、测试和部署软件的完全所有权(被称为[全周期开发](https://containerjournal.com/features/can-great-dx-enable-full-cycle-development/))。

贝尔彻认为，整个周期的势头被误解为开发商独自承担全部责任。相反，他把开发人员比作棒球队的投手；领导比赛，但是没有团队的其他成员，他完全无能为力。贝尔彻说，同样，不同的角色应该一起工作。到达 DevOps 的“最后一英里”不仅仅是关于 CI/CD 或全周期开发；而是作为一个团队一起工作。

我只是从 DevOps 的测试状态中发现了一些皮毛。该报告询问了 600 名质量工程师和软件开发专业人员，并提供了大量关于开发运维、测试、CI/CD 等状况的额外见解。您可以在此查看全部结果[。](https://www.mabl.com/offers/2021-testing-in-devops-survey-report)
