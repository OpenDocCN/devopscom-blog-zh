# 现代应用安全和供应链攻击——三大挑战

> 原文:[https://devo PS . com/modern-app sec-and-supply-chain-attack-three-challenges/](https://devops.com/modern-appsec-and-supply-chain-attacks-three-challenges/)

最近关于[网络安全管理软件产品漏洞](https://www.zdnet.com/article/sec-filings-solarwinds-says-18000-customers-are-impacted-by-recent-hack/)的新闻聚焦于供应链攻击带来的困难和挑战。在[微软称之为“solorigate](https://www.microsoft.com/security/blog/2020/12/18/analyzing-solorigate-the-compromised-dll-file-that-started-a-sophisticated-cyberattack-and-how-microsoft-defender-helps-protect/) 的案例中，攻击者在一个受信任的应用程序内部修改了一个 dll，然后被部署到超过 18，000 个企业和政府组织，然后在那里创建一个活动的后门供攻击者利用。[微软威胁情报中心(MSTIC)最近公布了关于网络安全管理软件产品攻击是如何工作并试图避免检测的细节](https://www.microsoft.com/security/blog/2020/12/18/analyzing-solorigate-the-compromised-dll-file-that-started-a-sophisticated-cyberattack-and-how-microsoft-defender-helps-protect/)。

## **供应链攻击和 API 安全**

在这个特定的案例中，漏洞攻击的目标是作为 it 库存管理和监控应用程序网络安全管理软件产品猎户座的一部分的传统遗留应用程序。这次攻击是广泛的计划和资源的结果，这似乎是一个国家行为者的工作。当务之急是[必须缓解当前的攻击](https://cyber.dhs.gov/ed/21-01/)。尽管如此，随着现代云原生应用程序成为规范，未来应该考虑像这样的供应链攻击，这一点也很重要。

[云原生](https://www.redhat.com/en/topics/cloud-native-apps)和现代应用的发展(其中功能被分解为微服务，每个微服务彼此独立运行)为安全和 IT 专业人员提出了几个需要解决的新挑战。具体来说，底层应用程序编程接口(API)引入了新的攻击面，其中 API 安全性至关重要。 [OWASP API 项目列举了 10 个关键的 API 级威胁](https://owasp.org/www-project-api-security/),这些威胁在现代云原生应用的时代更加重要。

这三种趋势——微服务激增、应用程序变化和漏洞百出的外围环境——创造了一个攻击猖獗的环境，IT 和安全团队需要考虑重新审视他们的应用程序安全实践和控制。

### **微服务扩散**

采用云原生架构所固有的一个特征是将[整体架构](https://devops.com/?s=monolithic+architecture)分解为原子服务。微服务使业务应用团队能够更高效地运营和创新。这种方法还导致独立服务数量的增加，其中每个微服务都是供应链攻击的潜在来源。在微服务的情况下，攻击者有更多的机会，而不是危及整个系统。

维护者面临的挑战是如何管理数百或数千个新微服务的激增，每个微服务都有其独特的代码库和 API 调用，服务之间？一方面，传统的 web 应用防火墙(waf)可以发挥作用，试图保护边缘，但另一方面，waf 没有考虑到 web 应用和微服务的底层复杂性。要问的一个关键问题是，他们基于规则的方法能否充分检测和防止这种威胁？

### **连续申请变更**

随着团队采用持续交付和持续部署的 DevOps 实践，应用程序改进和增强的速度加快了。对更快交付和减少周期时间的业务期望自然地导致 IT 团队接受并采用 CI/CD 的 DevOps 实践，当采用时，会显著缩短变更之间的周期时间。该行业在交付速度方面取得了显著进步；组织已经从每季度发布一次转变为每周甚至每小时一次，有时甚至更快。例如，[高盛的一个团队每隔几分钟发布一次](https://about.gitlab.com/customers/goldman-sachs/)，而网飞[每天发布数千次。](https://techbeacon.com/devops/10-companies-killing-it-devops)
‍
在一个新代码缓慢交付的传统应用中，跟上变化的步伐并减少安全威胁是可能的。然而，在微服务和云原生应用的时代，保持对应用的状态和相关微服务的完全了解是不现实的。变化发生得太快，在太多的内部点上，安全领导者无法充分管理他们的风险状况。

### **相对多孔的周界**

因为应用程序不再是位于纵深防御堡垒后面的整体安装，所以它们更难保护。事实上，在设计和管理现代应用程序时，[零信任](https://www.traceable.ai/blog-post/what-threat-vectors-get-addressed-with-zero-trust-application-security)的原则是必不可少的。事实是，对于每个微服务(内部和外部)，都有一个接口库，使微服务能够通过已建立的 UI 相互通信(内部)、与第三方服务提供商通信(外部)以及与最终用户通信。

此外，您的应用程序中有多少具有“呼叫总部”功能，可以与供应商交流运行状况和使用数据？虽然许多供应商使用这些信息来提高他们的服务质量，但这也是另一个需要考虑的潜在威胁。

web 应用程序防火墙和传统的安全方法只能提供一定程度的保护，因为现代应用程序的性质鼓励使用 SaaS 服务、第三方服务和其他资源来加快交付速度。这使得交付安全应用程序并在运行时对其进行监控的挑战变得更加复杂。

## **解决 API 安全性的建议**

我们仍处于网络安全管理软件产品泄露事件的早期阶段，其后果将是广泛而昂贵的，从损失评估中吸取的教训也是如此。然而，有一件事是真实的。威胁在不断演变，这意味着安全专业人员(strike that) *IT 专业人员*(因为安全是每个人的责任)需要重新关注并致力于提供安全、可靠和值得信赖的应用程序。

没有完美的保护或 100%安全的方法；相反，这是一个如何建立多层防御的问题，每一层安全都会降低整体风险暴露。最终，您希望能够:

*   **首先防范风险**(软件供应链和零信任架构)以及
*   **检测并应对**可能试图利用您系统的敌人。

具体来说，这里有几个建议可以改善您的安全状况并降低您的暴露程度:

1.  投资流程和工具，帮助验证您的供应链。“信任，但验证”的方法在开发阶段至关重要。你应该看看像 Sonatype 这样的工具和其他可以帮助你降低软件供应链风险的工具。
2.  采用[零信任](https://www.crowdstrike.com/epp-101/zero-trust-security/)方法来设计和实现微服务。
3.  保持对 API 当前状态和当前风险的了解和洞察力(太多的组织跟不上开发运维变化的步伐)。
4.  用流程和实践扩大您的边界，在生产中实现“零信任，但要验证”。您需要观察微服务*的每个*的*行为，以及*检测微服务之间的异常行为。

*   微服务 A 是否应该与微服务 B 进行通信？
*   微服务 A 和微服务 B 之间的数据流是否典型？
*   是否有可疑行为或模式表明异常行为？
