# 秘密管理的秘密:减少源代码漏洞

> 原文:[https://devo PS . com/the-secret-to-secrets-management-reduce-source-code-vulnerabilities/](https://devops.com/the-secret-to-secrets-management-reduce-source-code-vulnerabilities/)

我们正处于网络攻击时代。从微软到 Crypto.com，没有一家公司能完全免受恶意攻击者的攻击，似乎每周都会有新的重大网络攻击发生，甚至更频繁。随着越来越多的公司不仅走向数字化，而且扩展他们的数字化产品和计划，网络攻击者有了更多的机会。IT 组织中的安全团队正在努力工作，以确保他们的公司不会成为下一个头条新闻。然而，随着技术世界转向一切即服务(IaaS、PaaS、[、SaaS](https://devops.com/understanding-saas-security-for-devops/) 等)。)，一切按代码进行的趋势正在同步发生。因此，许多开发人员在不知不觉中保留了敏感数据(秘密、密码、证书、密钥等)。)存储在 Git 源代码库中。这些信息经常被推送到生产环境中，这使得敏感信息面临更大的暴露风险。

好消息是，有一种方法可以确保秘密被紧紧地锁在源代码中。让我们探讨一下当今保护源代码的复杂性和现实性，以及如何消除猜测，找到有风险的代码，使您的公司登上明天的新闻头版。

### 机密管理极具挑战性且成本高昂

老练的网络攻击者会不择手段地破坏宝贵的数据资产。从源代码中泄露的秘密使得威胁参与者可以很容易地使用用于合法目的的相同凭证访问应用程序。

按照持续软件开发的速度，开发人员必须快速构建应用程序。然而，期望防止违规的安全团队通常并不精通软件开发生命周期(SDLC)的所有方面。随着开发的应用程序数量的增加，代码行数(以及其中保存的秘密)也在增加，这就成了一个问题。根据 1Password 最近的一份报告，“三分之二(65%)的 IT 和 DevOps 员工估计他们的公司有超过 500 个秘密。”根据同一份报告，“一个秘密泄露的平均成本是 120 万美元。糟糕的机密管理会导致组织每年损失 85 亿美元。”秘密对于构建任何连接的应用程序或流程都至关重要。如果在源代码中硬编码或不受保护，众所周知的通往王国的钥匙就有暴露的风险——就像您的忠实客户一样。

很明显，源代码安全应该是安全团队和 DevSecOps 专业人员的首要任务。让我们看看他们如何在不牺牲开发人员生产力的情况下实现这一点。

### 如何通过机密管理最佳实践减少威胁

手动搜索源代码库中易受攻击的秘密几乎是不可能的。但是，有一些最佳实践可以遵循，以确保尽早且经常地发现漏洞:

● **识别并移除代码中的秘密:**你不能保护你看不到的东西。了解源代码以便能够发现哪里可能存在漏洞是非常重要的。有几种潜在的方式来揭示漏洞。一种是通过手动扫描源代码，但这占用了开发人员可以花在更具创新性的任务上的宝贵时间。也有一些开源解决方案可以帮助扫描源代码库，但就像所有开源的东西一样，这意味着开发人员必须维护它(保持它最新，集成它，等等)。).相反，安全团队应该考虑寻找一个统一的工具链平台来帮助识别和修复源代码漏洞。
**●代码库中的警告和补救:**在将代码发布到产品中之前，向开发人员或 DevOps 工程师提供快速反馈循环。补救应该在提交时进行。这里的想法是首先阻止秘密提交到您的源代码库中。
**●采用秘密管理工具:**不要在源代码中存储秘密或二进制文件——在你的源代码库中，除了代码什么都不应该有。将秘密移至秘密管理器(即 HashiCorp 保险库)、密钥和证书管理器(即 Azure 密钥保险库)等。将这些管理人员模块化隔离，但要确保他们之间仍有沟通，这样才不会暴露秘密。

随着网络攻击的不断增加，安全团队必须仔细检查他们的源代码，以确保它们被牢牢锁定。从理论上讲，将秘密完全隐藏在源代码之外是值得推荐的，但在实践中很难保证。遵循上述最佳实践将有助于降低暴露敏感数据的风险，并降低失去忠诚客户和收入的风险。
