# 人工智能如何改进软件开发

> 原文:[https://devops.com/how-ai-can-improve-software-development/](https://devops.com/how-ai-can-improve-software-development/)

变化是持续的，不可避免的。虽然有些变化是受欢迎的，但其他变化则是不可避免的。

对数字化的持续需求与疫情推动的对转变业务运营的重视相结合，迫使各行业的组织对其遗留系统进行现代化改造。云计算的出现缩小了组织的内部 IT 足迹。然而，由于全球最大的 100 家银行中的 96 家、全球最大的保险公司中的 90%、美国最大的 25 家零售商中的 23 家以及财富 500 强中 71%的 [仍在使用大型机系统](https://www.forbes.com/sites/ibm/2014/04/08/from-shopping-to-space-travel-how-the-mainframe-changed-our-world/) ，任何导致停机或缺陷风险增加的现代化努力都将经常导致灾难性的后果。

因此，虽然做出改变可能会很昂贵、不确定，并导致客户流失和/或在市场上尴尬，但现代企业要求系统改变以继续提供价值。然而，大型机现代化的问题是，当涉及到减少与维护和改进系统相关的风险时，今天的代码搜索工具、linters 和程序分析工具是有缺陷的。

### 现代化:一个知识问题

随着 [10，000 台大型机](https://www.bloomberg.com/news/articles/2010-08-03/big-tech-problem-as-mainframes-outlast-workforcebusinessweek-business-news-stock-market-and-financial-advice) 在全球范围内的活跃使用，一般组织每年花费从 [60%到 80%的 IT 预算](https://marketing.dell.com/Global/FileLib/Services/legacy-systems-am-wp.pdf) 在维护上。但是，随着经验丰富和熟练的程序员退休或继续工作，许多公司意识到，那些开发人员用来创建和更新这些复杂的关键系统的行业和机构的专业领域知识正在随之消失。T9】

由于最初编写代码的开发人员已经离开很久了，并且文档留下了许多需要改进的地方，新接触系统的开发人员无法以足够详细或足够快的水平获得系统功能的全面知识，从而变得足够高效和熟练。对于任何组织来说，缺乏速度都会带来巨大的风险和成本。

识别用代码编写的功能的完整和真实意图并不简单。今天的开发人员花费 [大约 75%的时间](https://coralogix.com/blog/this-is-what-your-developers-are-doing-75-of-the-time-and-this-is-the-cost-you-pay/) 搜索源代码，以识别代表需要改变的功能的代码。更复杂的是，仅仅理解代码是不够的。为了有效和高效地维护和支持任何系统，软件开发人员必须准确地知道应用程序实际上做什么——以及在系统的一个部分更改代码如何影响整个系统。但是由于代表需要改变的行为的代码可能散布在整个系统中，开发人员可能认为他们正在进行简单的调整，而实际上他们可能正在破坏整个系统(甚至不知道)。变革导致意想不到的后果的风险是真实存在的。

话虽如此，开发者如何提高他们的效率和效果呢？

### **当今工具的缺点**

随着时间的推移，代码库不受控制地增长到非常大的规模，那些负责维护和保护系统功能的人说，在没有机器帮助的情况下找到代码中的错误变得越来越具有挑战性[](https://www.wsj.com/articles/ais-role-in-debugging-software-code-is-expected-to-grow-11625563802)。由于调试是一项艰巨、耗时的工作，开发人员越来越多地转向代码搜索工具、linters 以及静态和动态分析工具来分析数百万行代码，标记错误并提出修复建议。

无论该工具擅长于错误定位、代码可视化还是“错误”检测，当涉及到实际识别需要注意的特定代码行时，太多的工具都是不够的。程序分析工具只是以开发人员仍然必须解释(可能不正确)并得出他们自己的结论的方式来说明代码。虽然代码搜索工具可能会加快开发人员构建代码心智模型的速度，但这种工具因误报而臭名昭著，开发人员仍然必须经历将心智模型拼凑在一起以识别代码，然后安全地进行更改的精神挑战。更糟糕的是，代码完成工具——缺乏经验的开发人员不成比例地依赖的工具——实际上可以提出不精确的更改(一个著名的工具 [在 71%的情况下提供不正确的代码](https://www.fast.ai/2021/07/19/copilot/) ，这引入了风险。

这个有问题的圣代上的樱桃是，今天的工具没有任何方法来有效地验证一个提议的变更的范围和准确性。建议变更的影响可能很难识别，因此进行全面验证的测试几乎是不可能的。

理想情况下，在对遗留系统进行更改时，必须有更好的方法来降低风险。进入人工智能。通过采用一种独特的人工智能方法，代码库真正成为一个知识库，实现了许多以前没有实现的功能。

### **开发者的人工智能支持的可访问知识库**

通过利用人工智能来自动识别需要关注的特定代码行，开发人员可以简单地询问这个人工智能驱动的知识库行为来自哪里——并快速识别与该行为相关的代码。这将人工智能置于智能增强的位置，这是利用其能力的关键。

这种人工智能的新方法重新解释了计算代表的内容，并将其转换为概念，因此以与人类相同的方式“思考”代码。结果是，软件开发人员不再需要挖掘编码在软件中的以前开发人员的意图来寻找潜在的错误。更好的是，开发人员能够通过使用人工智能来验证他们在编译或签入代码之前没有破坏系统，从而克服自动测试 的不足。人工智能将向前模拟变化，并确定它是否独立于变化下的行为。结果是变化的界限被限制在变化下的行为，这样就不会出现意外的结果。

虽然人工智能还不够先进，不足以安全地独自进行更改，但它可以作为一个知识库，具有额外的能力，可以精确地指导开发人员在哪里进行任何需要的更改。有了人工智能，程序员——即使是在工作的第一天——也可以自信地做出任何必要的修复，而不会破坏整个系统。

与代码搜索、代码导航、代码完成和程序分析工具不同，AI 从根本上增强了开发人员安全、高效和有效地进行任何必要更改的能力。有了人工智能——它得到了一个知识库的加强，这个知识库像人类一样理解源代码——开发人员可以真正做到事半功倍。
