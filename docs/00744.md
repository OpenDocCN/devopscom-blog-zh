# 企业在实施自动化时会犯的五个错误

> 原文:[https://devo PS . com/five-errors-enterprises-make-while-implementation-automation/](https://devops.com/five-mistakes-enterprises-make-while-implementing-automation/)

如果操作得当，自动化可以为 IT 组织增加大量价值。好处包括速度、质量和提高生产率。但是只有少数 IT 组织能够真正获得它的好处。这是为什么呢？是什么让这些成功的公司与众不同，而其他公司仍在苦苦挣扎？

以下是我们在组织尝试实现自动化时看到的 5 个最常见的错误。

**1。自动化不常用的活动**

当手动任务定义明确、可预测且高度重复时，自动化效果最佳。前两个维度是决定一个活动是否可以自动化的关键。第三有助于确定自动化是否值得。然而，当组织决定自动化时，他们只关注前两个，很少考虑第三个。

例如，自动化代码构建是一种流行的自动化活动。它显然符合前两个维度。但是如果项目计划只要求每月构建…那还有什么意义呢？你所做的最多只是为开发人员节省了几个小时的工作。然而，如果同样的自动化被用于需要每日/每周代码构建的项目。现在你实际上是在减少工作时间。把它转化为每年节省的努力，你就可以结束这一过程。

**2。筒仓自动化**

移交是效率的无声杀手。然而，大多数 IT 组织被安排成一系列的移交(筒仓)——从需求到开发到测试到运营。像大多数孤岛一样，关注点是向内的，导致优化的孤岛，但端到端流程次优化。难怪一个新代码需要几个星期才能进入生产阶段，而且在很多情况下，质量都很差。

当您将各种活动整合到一个流程中时，自动化的真正价值就体现出来了，这可以最大限度地减少/消除移交，并跨孤岛进行协调。本质上，您不仅自动化了活动，也自动化了移交。这实质上是整个 DevOps 运动的核心。打破孤岛，重新设计和自动化流程。

**3。忘记重新设计流程**

如果你看看数据，大多数过程遵循经典的 80/20 法则。20%的过程时间花在执行实际的“工作”上，80%的时间浪费在运动上。一个很好的例子…过去，我们需要几天(有时几周)才能打开一个简单的 IP 端口。执行工作的实际时间只需要大约 5-10 分钟，但是将工作交给 SME 却需要几天时间(创建工单、路由、审批、寻找 SME 等。).

尽管如此，许多组织花费了无数的时间来寻找、折磨和优化“完美”的工具(提示:没有一个工具),而在过程上花费很少甚至没有时间。本质上，如果你只关注工具，那么你所做的就是优化那 20%。难怪组织在投入自动化而没有得到预期的结果时会挠头。自动化和流程再造就像一枚硬币的两面。如果你真的想改变，你不能没有一个。

**4。没有规划自动化的长尾**

嘘…让我告诉你一个秘密。创建自动化脚本很容易…毕竟，这是自动化的迷人之处。一旦你创建了它，真正的重担就开始了，现在你必须不断地调整和维护它。

无论好坏，应用程序及其组件都在不断变化。这意味着脚本必须不断调整。然而很少有人考虑后数学。事实上，我们的客户有如此多的脚本，以至于他们根本不知道如何处理。更好的解决方案是提前为长尾做好计划，建立一个治理模型，负责更新、维护和修剪脚本，并反过来保护他们已经做出的投资。

**5。害怕自动化会夺走他们的工作**

事实——我们的许多客户对自动化持怀疑态度。不是因为他们觉得它行不通，而是害怕它会运行得太好。这种成功最终会导致优秀的人失业。

这种思路有三个关键问题。首先，这个假设是不正确的。虽然自动化会消除某些类型的任务，但也会创造新的工作类型。毕竟，你需要有人来管理自动化。其次，妖怪已经从瓶子里出来了。自动化将会继续存在。忽视它只会让你更难赶上它。第三，不实施自动化可能会产生相反的效果，加速失业。看看网飞(自动化的典型代表)和百视达。一个更好的策略是，趁行业还在摸索的时候，现在就从小处着手，努力走在时代的前面。
