# 解决软件工程中的文化难题

> 原文:[https://devo PS . com/solving-the-culture-conundrum-in-software-engineering/](https://devops.com/solving-the-culture-conundrum-in-software-engineering/)

支撑我们向数字优先的经济转型的一个关键因素是软件。这就是为什么软件工程已经成为世界上最受欢迎的职业之一。仅在英国，该职位的平均工资同比增长了 13%，因为企业希望在竞争中保持领先地位，并吸引最优秀的 IT 人才。但是高薪通常与软件开发人员在行业中的地位不一致；许多组织仍然将他们视为供应商，而不是成功的关键角色。根据当前的转变速度，我们知道对软件工程的需求只会增加。因此，商业和技术领导者需要紧急考虑重要的文化变化，以帮助更好地整合更广泛组织内的软件开发团队。但是它们到底应该是什么呢？

### 发展困境

随着企业努力快速有效地满足客户期望，敏捷性迅速成为一种竞争优势。这预示着 DevOps 的崛起，devo PS 旨在更有效地编写高质量的数字解决方案，并能够以安全和低风险的方式持续更新。

这种做法已经在大范围内加速了数字解决方案软件开发中的反馈循环，这意味着新软件将尽快投入生产，同时降低风险。因此，今天成功的企业现在每天发布多次软件更新。对于亚马逊和网飞这样的科技巨头来说，这个数字甚至高达数万。然而，其他企业正在努力跟上市场需求的速度，这主要是因为软件开发商和更广泛的企业之间缺乏清晰的沟通、工作方式和集成。如果企业要保持竞争优势，这是一个需要立即关注的两难问题。

开发人员经常被留下来处理庞大的工作，这需要他们连续几个月编写代码，而没有经过测试。这不可避免地导致错误，这些错误只有在更新已经发布到市场上时才被发现，从而在整个组织中产生连锁反应。这是一个太常见的场景，表明企业领导人明显缺乏对采用 DevOps 文化可以为更广泛的业务(包括其客户)带来的巨大价值的清晰和理解。

举个例子，金融机构希望推出一种新产品，比如一种新的抵押贷款产品。如果没有一个共同的开发方法，为一个产品建立和运行后端系统可能需要长达六个月的时间。但是模块化选择和分解功能可以将时间缩短到几周。

企业领导人必须寻求建立统一的、有凝聚力的和跨职能的团队，在项目上合作，这些项目被分解成简化的和本地化的组件，这些组件可以很容易地集成在一起，而不会对彼此产生负面影响。这似乎是工作实践中的一个小的战略变化，但是它对软件开发的速度和效率的影响是巨大的。

### 重新审视个性和流程

软件工程师的角色已经改变；它不再是孤立地编写代码，而不太考虑或了解它如何给业务带来好处。当开发人员清楚他们的工作对实现业务目标和底线的直接影响时，他们会工作得更好。传达这些挑战和目标(换句话说，理解“为什么”)以帮助软件开发人员理解他们试图实现的是什么，这是业务领导者的责任。但是以一种向更好的工作文化发展的方式这样做需要一种新的方法来建立和管理软件开发团队。

第一步是抛开许多人对软件工程师的负面刻板印象，赞美他们团队中知识和文化的多样性。人员的多样性带来了性格的多样性，这对于创造更具包容性的文化，张开双臂接纳和欢迎所有角色至关重要。虽然这似乎是显而易见的，但往往被忽视的是多样性对刺激和增加创新的影响。

提高开发团队的多样性有助于开启应对和分析商业领袖提出的挑战的多种方式。毕竟，每个人都有一系列基于其学习、发展和经验的不同软硬技能。因此，鼓励个人将他们的性格和个性带到他们的角色中是至关重要的；它确保开发过程中的每一个利益相关者都有能力做出贡献，感受到内在的价值，并能够挑战现有的假设，帮助交付更强大的最终产品。

认识和利用这些技能也有助于消除企业内部存在的许多发展障碍。例如，一些软件工程师可能很难以更广泛的业务可以理解的方式传达技术更新，但利用另一个人强大的沟通技能可以帮助团队——以及整个业务——团结在一个整体的数字战略周围。

### 引导变革以交付更好的成果

显然，文化变革是将软件工程嵌入现代商业核心所必需的。但是，重新审视项目管理实践对于持续交付为业务提供价值的软件也是必不可少的，科学管理和瀑布交付方法的世界不再适合数字解决方案的开发。在如此依赖数字服务的情况下，企业——以及随之而来的开发者——需要能够以最小的影响做出改变。因此，避免单一项目，鼓励跨职能团队是至关重要的。它不仅减少了对大规模软件更新的需求，而且消除了项目无法满足最初概要的风险，这种风险最终会影响整个业务。

通过审查组织流程，并将更广泛的业务和软件开发人员聚集在一起，我们可以创建团队来有效、快速地交付数字解决方案，以便每个用户(无论是内部还是外部)都能从更复杂的软件工程方法中获益。
