# 邮递员调查:API 使用的增加导致了更多的安全事故

> 原文：<https://devops.com/postman-survey-increased-usage-of-apis-results-in-more-security-incidents/>

Postman 是一家开发和管理应用编程接口(API)的平台提供商，该公司本周公布了一项针对 37，000 多名开发人员和应用编程接口(API)专业人员的调查,调查显示，随着越来越多的 API 被部署，越来越多的组织看到网络安全事件的增加。

调查发现，超过一半的受访者(51%)报告说，他们组织一半以上的开发工作都花在了 API 上。大约 20%的人还表示，他们每月至少对其组织的 API 安全事件或违规事件做出一次响应。总的来说，超过一半(52%)的人说 API 安全事件一年发生不到一次。

Postman 的首席布道者 Kin Lane 表示，虽然对 API 安全的许多担忧可以归类为 API 安全平台提供商提出的“戏剧”，但很明显，需要一种零信任的方法来构建、部署和维护 API。

大部分 API 主要用于互连面向内部的应用程序和系统，一小部分用于集成合作伙伴。只有相对较小比例的 API 面向外部。然而，Lane 指出，许多组织未能意识到一个面向内部的 API 在其生命周期中变得面向外部的速度有多快。因此，随着 API 的发展，IT 团队需要管理和保护 API 所需的工具和平台，他指出。

事实上，近四分之三的受访者(72%)表示，能够远程构建 API 对他们来说非常重要。这意味着最好的 DevSecOp 实践需要扩展到经常在家工作的 API 开发团队。

然而，尽管存在这些担忧，但整整 89%的受访者表示，在未来 12 个月内，对 API 的投资将会增加或保持不变，以推动数字业务转型计划等。然而，调查也表明大多数组织仍然在 API 开发和部署方面妥协。只有 8%的受访者认为自己是 API 优先的领导者，Postman 认为这是一个精英群体，他们生产 API 更快，部署更频繁，故障更少，当故障发生时恢复得更快。

Lane 说，这项调查清楚地表明，虽然 API 在推进业务目标方面发挥着至关重要的作用，但随着越来越多的 API 在整个企业中部署，仍有必要提高可见性。这一点尤其重要，因为越来越多类型的 API 也在不断扩展。今天使用的大多数 API 都是基于 REST 格式的，但是现在也有基于 [GraphQL](https://devops.com/graph-cdn-for-graphql-apis/) 、事件驱动架构和各种消息平台的 API。这些较新的 API 都不会很快取代 REST APIs。Lane 指出，相反，它们正被用来驱动 REST APIs 之外的其他进程。

目前还不清楚有多少组织在编写应用程序代码之前采用了设计优先的方法来构建 API，但是随着更多 API 的部署，管理 API 的方式正在迅速改变。只要用于向其他应用程序公开应用程序服务的 API 保持稳定，应用程序服务本质上变得更加可任意使用。事实上，现在大多数组织完全意识到他们的整个 it 策略围绕着他们的 API 的程度只是时间问题。
