# 事件管理:让噪音哀号而不聋

> 原文：<https://devops.com/event-management-let-the-noise-wail-without-going-deaf/>

在詹姆斯·马龙的博客系列“[将运营置于开发阶段](https://devops.com/operations-management-puts-ops-devops/)”的第 1 部分中，我们讨论了运营管理的意识和预期，以及可以在应用生命周期的每个阶段改进应用的最佳实践。在这篇博客中，我想介绍一个事件管理规程的成熟度，并分享一些最佳实践。

如果您是团队中的一名运营专业人员，开始采用 DevOps 方法，您可能会面临一些重大挑战。

您知道企业正在施加压力以获得持续的集成/交付，在这种情况下，发布周期可以用几天甚至几小时而不是几周或几个月交付的价值来衡量。您的开发团队已经接受了敏捷开发，并且正在短时间的迭代中生产(应该是)生产就绪的代码。您的联合组织正致力于实现一个集成的交付工具链，只需按一下按钮，就可以构建、测试和部署一个新的发布版本。您的开发团队正在利用云技术和架构模式来增强他们的敏捷性，以前所未有的规模和模块化自动扩展微服务架构。作为一种最佳实践，他们正在生成高度插装的代码，这些代码提供了与部署在不断报告自身状态的运行时上的应用程序的状态相关的密集数据踪迹。您可能负责物理和虚拟计算、网络和存储基础架构，而这些基础架构本身是由工具监控的，这些工具会产生连续的状态信息流。

换句话说，你发现自己面临着持续的变化(自动的和人为的)和一连串潜在的噪音。如果你是一名经验丰富的运营专家，痛苦的经历可能会让你将变化视为敌人——随着变化趋于零，运营风险也趋于零。现在，你可能不得不应对持续的变化，这是推进公司业务目标的不可避免的力量。你将如何从服务状态的噪音中分离出服务状态的信号？成功地做到这一点将是在用户注意到之前迅速知道应用程序或服务有问题和被用户告知有问题的区别。这也可能是睡个好觉和不必要地响应指示良性应用程序状态的页面之间的区别。

进入事件和事故管理。几十年来，作为一种从状态*确定受管环境元素状态的机制，事件管理的规则已经成熟，并继续成熟。*活动管理的最佳实践包括:

*   过滤掉不太可能影响服务的事件。来自应用程序日志的跟踪消息不值得您关注，除非您需要将其作为诊断过程的一部分来查看。合成用户事务超时可能需要立即调查。
*   关联可能相关的事件，因此每个真实事件您会收到一个通知，而不是 20 个。
*   用上下文丰富事件，因此，如果五个实例组成的冗余阵列中有一个服务实例失败，请将事件路由到操作控制台，但不要唤醒任何人，除非服务受到影响。
*   实施 X-in-Y 策略。在大型复杂系统中，单个微服务到微服务的 HTTP 超时可能不是一个大问题。但是，如果您开始获得每秒 20 次，您可能需要进行调查。
*   针对常见故障实施操作手册。与开发部门合作定义这些操作手册。当事故发生时，将这些操作手册与事故联系起来，以便第一响应者有一个流程，他们可以恢复服务或在第一时间防止停机。
*   如果可能的话，自动化这些操作手册。如果问题是一个失败的过程，然后重新启动它。磁盘已满？释放一些空间。
*   利用分析和机器学习从工具收集的大量数据中获得洞察力。您能否从事件历史中吸取经验教训，以提出运营效率的相关性或改进领域？

随着新特性和功能的快速交付，开发团队应该已经注意到他们生产的软件的可操作性。将对事件管理工具和逻辑的认识转移到左边。让开发部门使用与运营部门在生产中使用的相同的管理工具，但要在生产前使用。然后从他们在 pre-prod 中看到的东西中学习以进行改进。这应该结合系统故障测试来完成。执行测试并查看操作工具如何响应。问问你自己，“如果这发生在生产中，我能解决吗？我会知道该怎么做吗？”

现在显而易见的是，在现代企业中成功运营应用程序或服务需要开发和运营团队之间前所未有的合作。事件管理作为一门学科和支持这门学科的工具现在和以往一样重要，甚至更重要。
