# 可观测性的机遇和挑战

> 原文:[https://devo PS . com/可观察性的机遇和挑战/](https://devops.com/opportunities-and-challenges-of-observability/)

[可观察性](https://devops.com/?s=observability)跨应用和 IT 基础设施统一指标、日志和跟踪的平台正在改变 IT 管理方式。如今，大多数开发运维团队能够使用跟踪一组预定义指标的工具来持续监控 IT 环境；可观察性平台使得聚合数据成为可能，以便开发运维团队可以发起查询，帮助他们发现 it 问题的根本原因。 DevOps 团队利用可观察性让他们更有效和高效地理解他们的应用程序中正在发生的事情。

从历史上看，应用程序是黑匣子——这意味着很难(在许多情况下仍然如此)知道应用程序的内部发生了什么。

随着 DevOps 领域中可观察性工具和技术的出现和随后的发展，团队从努力收集和理解可用数据(这些数据可能从强大到几乎不存在)到轻松聚合来自众多来源的数据并能够理解这些数据意味着什么，再到完全被数据[淹没](https://devops.com/devsecops-deluge-choosing-the-right-tools/)——如此多的数据很难跟上。

## 收集数据已经不够了

IT 基础设施监控专家 Netreo 的战略和创新副总裁 Josh Chessman 解释说，随着应用程序变得越来越复杂(例如，在不同的位置使用不同的技术在多个不同的环境中运行)，从这些不同的环境中收集数据已经不够了。

他说:“相反，DevOps 工程师需要能够收集数据并有效地分析数据，以了解应用程序中正在发生的事情。”。"可观察性是驱动这一点的概念."

Chessman 指出开发团队面临着许多可观察性问题，但是这些问题因组织而异(有时也因组织内的团队而异)。

他说:“问题的一部分在于，许多组织根本缺乏工具来提供数据收集和分析，而这些数据收集和分析是迁移到可观察性范例所必需的。”。

这可能有很多原因，包括预算(如果您已经有了一个工具，您能负担得起购买一个新工具吗？)、功能性(并不是所有声称提供可观察性的工具都是如此)、实现(这并不容易，一个实现不好的工具实际上会使事情变得更糟)等等。

Chessman 说:“我见过的一个更大的挑战是，组织将走出去，获得一个‘可观察性’工具，并假设这就是他们要做的一切。“不幸的是，事实并非如此。获得工具可以让您获得技术，但是团队也必须结构化并准备好利用工具提供的概念和功能。”

GitLab 的产品管理高级总监肯尼·约翰斯顿解释说，平均来说，DevOps 团队使用五种不同的可观察性工具，这些工具往往不能充分地将开发人员与可观察性联系起来。

反过来，开发人员在跨越多种工具和事实来源时努力快速响应事件，并努力将服务水平目标(SLO)和[错误预算](https://www.atlassian.com/incident-management/kpis/error-budget#:~:text=An%20error%20budget%20is%20the,can%20fail%20without%20contractual%20consequences.)定义分配给负责的产品开发团队。

他说:“为了补救这一点，我们建议采用一种统一的软件开发解决方案，取代束缚开发者的产品 DIY 胶带’,帮助他们更安全、更快速、更大规模地交付软件。”

Johnston 表示，当今最灵活、最经得起未来考验的解决方案是开源工具，如 Prometheus 及其生态系统，以及用于追踪等更高级用例的 OpenTelemetry 工具集。

Prometheus 在开发领域已经变得非常普遍，与其他开源解决方案或库不同，它的格式会自动发出指标。

他说:“这允许简单且经得起未来考验的互操作性，因为随后最难改变的事情之一是代码运行。”。“使用开放标准减少了开发人员一遍又一遍地重写和整合相同类型代码的需要。”

## 可观察性需要理解你的目标

从棋手的角度来看，理解你的目标对观察是至关重要的。

“可观察性工具可以做很多事情，但它们的主要功能是收集和分析数据，”他说。“虽然这一切都很好，但如果您不从一开始就定义 KPI，所有收集和分析的数据都没有意义，因为您不知道什么是好的和坏的。”

试图从收集到的数据中定义好与坏很可能会导致失败，因为你是在看你在哪里，以此来确定你想去哪里。

他说:KPI 因应用而异，如果你在开始观察之前没有确定和很好地定义它们，那就更难了。

大多数工具(可观察性或其他)能够收集数百甚至数千个指标，但并不是所有——在许多情况下甚至不是大多数——都与组织相关。

Chessman 说:“与其从这些资料中筛选出有价值的数据，不如从识别这些有价值的数据开始，然后想办法找到它们。”。“如果您没有提前确定 KPI，最终可能会在错误的情况下使用错误的 KPI。”

Johnston 补充说，KPI 和 DevOps 一样，是一个迭代过程，从 KPI 开始可以帮助开发人员创建满足性能和可用性预期的系统。

他说:“DevOps 团队不断审查他们的 KPI，以便对成功和失败进行定期反馈，这一点很重要。

从他的角度来看，维护和提高可观察性的最佳起点是将可观察性定义、工具、SLO、阈值、事件响应模板和仪表板存储为代码。

他说:“这允许任何团队成员通过修改代码来快速改进，无论是站点可靠性工程师(SRE)、开发人员还是平台团队。”。
