# 建立全方位服务所有权文化

> 原文：<https://devops.com/building-a-culture-of-full-service-ownership/>

近年来，我们对软件的依赖呈爆炸式增长 。自从马克·安德森预言“软件正在吞噬世界”以来，已经过去了十多年今天，几乎所有的互动都是通过屏幕和软件来实现的。随着大规模 由于疫情而转向远程和数字交互，已经在适应数字优先的未来的企业不得不迅速加快步伐，以在当今日益复杂的技术环境中生存，这给 IT 团队带来了过度的压力。

显然，开发人员交付了[软件开发生命周期中最关键的部分；](https://devops.com/contrast-security-releases-new-capabilities-to-secure-enterprise-software-supply-chains/)然而，拥有他们开发的服务和应用程序也同样重要。“全方位服务所有权 的理念意味着在软件和服务生命周期的每一个阶段承担起支持它们的责任。持续交付和创新的卓越之处在于保持质量的同时提高速度。

服务 ow nership 的核心是将人类与技术和服务联系起来，并理解它们如何映射到关键的业务成果。实现这种级别的所有权需要了解什么和谁提供关键业务服务。清楚地理解给定服务的边界和依赖性以及它交付的价值是起点。一旦它投入生产，在任何给定的时间，谁对它负责，如果它没有以最佳方式运行，或者在最坏的情况下，完全失败，它的影响是什么，都有一个明确的定义。为开发人员提供这些信息使 DevOps 团队更接近他们的客户、业务和他们创造的价值，这反过来又导致更好的应用程序设计和开发。

围绕全服务所有权模型的文化让开发人员与他们构建的应用程序紧密相连，从而更接近他们提供的价值。在组织内部，这种所有权类型将长期以来建立的集中式和 孤岛式团队分解为跨职能的全方位服务团队。对于流程根深蒂固的企业来说，这可能说起来容易做起来难——但结果非常值得投资。遵循这些指导方针将允许遗留的 IT 团队创建一种全面服务所有权的文化。

### 让开发人员参与进来，明确所有权

随着创新的压力已经沉重地压在软件团队身上，你可能会担心更多的责任可能带来的负担。然而，最终目标是互利的:以实时的速度创造清晰的所有权。这不仅使工程团队能够更快地发布功能，还使他们能够在出现问题时保持参与和联系，从而减少移交和平均解决时间(MTTR)。换句话说，没有恐慌，没有不愉快 和最小的客户影响。

### 分散团队结构

改造团队结构和技术以获得全面服务所有权似乎势不可挡，但事实并非如此。传统的基于功能的筒仓是许多传统 IT 部门 的特征。供应基础设施和资源、开发代码、质量保证、拥有客户关系、支持生产中的软件等等，常常被分散的、集中的团队所委托。允许分散的团队 拥有他们服务的整个生命周期，让他们有能力以新的方式增加价值。关键是从小处着手，逐步实施变革。

### 实施战略部署

在开始全面服务所有权采用流程时，以小见大 将获得关键组织利益相关者的支持，他们随后可以推动更大规模的变革。从非关键生产系统或绿地应用开始。敏捷方法在适应动态开发 需求和不可预测事件方面也很有用。记住，在初始阶段要灵活掌握时间表，对团队要有耐心——掌握生命周期需要时间！

牢记长期目标，你的团队不仅要考虑“构建和部署 ”，还要考虑“运行”以全面展示整个基础设施的方式打造您的技术堆栈，利用机器学习来减少错误警报，并使用自动化来节省随叫随到的人力和资源，以支持生产中的 service。

### 利用正确的工具和技术

这里的一个关键要求是建立信任和一个准确的系统，以便在出现问题时确定谁应该负责。如果开发人员收到错误的 警报、过多的中断或不断升级到重复修复，调整到 DevOps 方法将会失败。构建和执行强大的服务目录并支持他们的首选时间表至关重要。确保当需要他们的时候，你通过他们期望的渠道 联系他们。如果做不到这一点，会导致时间和精力的浪费，更糟糕的是，会让开发人员感到沮丧和疲惫。 [最近的数据](https://www.pagerduty.com/state-of-digital-ops/) 显示，那些被叫来支持服务太频繁或不正确的开发人员更有可能离开他们的工作，这只会进一步拖延你的进度。

有了准确的服务目录，消除错误警报就变得非常重要。团队必须确保“警告和警报” 是相关的，并且在你打断你宝贵的时间紧张的开发人员之前确认一个事件。这涉及正确的[可观察性](https://devops.com/?s=observability)和监控工具，但也必须涉及强大的集中式机器学习增强，以减少噪音和错误警报。随着团队的成熟，自动化在这里扮演着至关重要的角色。首先，通过减少可能耗费几分钟甚至几小时的普通诊断和调查步骤。然后，通过将共同的解决方案交由第一响应者或甚至客户服务团队处理，减少开发人员的升级(超过 50%)。这些技术集合通常被称为 AIOps，随着数字业务规模的扩大，它们成为了一项基本要求。

### 跨职能团队是桌面赌注

今天的数字生态系统比以往任何时候都更加先进和复杂。保持高质量的数字服务不仅仅是一种竞争优势，也是企业声誉的支柱。最近高调的——停机和网络攻击已经将软件供应链的脆弱性推到了国际聚光灯下，这意味着出错的余地很小。在问题影响到客户之前主动实时解决问题至关重要。

透明的问责制和分散的所有权确保了当问题发生时，团队对问题有更深刻的认识，并能迅速采取行动。向技术团队和开发人员揭示他们对业务成果的直接影响，可以实现高质量、创新的代码，实现它的承诺。
