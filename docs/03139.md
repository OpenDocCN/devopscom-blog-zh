所以当我们采用这个名字或者把这个词改成名词的时候，我们想认真对待这个问题；这意味着我们如何有效地构建一个产品，可以在左侧测量另一个产品的安全状况，并从右侧获得信息，因为 DevOps 通常只是一个管道，一个自动化和高速运行的情感管道。DevOps 就是获取代码、构建、打包，然后部署到您的生产生态系统中。

因此，为了进行这个基准测试，我们提交了一个应用程序的两个实例——一个易受攻击的应用程序。所谓易受攻击，我的意思是它本质上包含了一组特定的漏洞，我们经常针对这些漏洞进行测试，以衡量我们平台的功效。

在本次测试中，我们采用了该应用程序，并在生产环境中部署了两个实例。一个实例是不可预测的:代表这些漏洞并且没有应用保护的典型实例。而另一个实例由 ShiftLeft 的一次性代理保护。

因此，这个测试的前提是让这些渗透测试人员对这两个应用程序进行测试，从而得出结果。这两个结果都有一定的关注度。结果是确定这些 pen 测试人员可以在不受 ShiftLeft 保护的应用程序上声明哪些漏洞，然后对受 ShiftLeft 保护的应用程序进行相同的测试，并返回来让我们知道他们是否可以渗透该应用程序，或者是否可以利用该应用程序。

这是一个为期 14 天的练习，我们定义了这个练习的范围——也就是说，我们基本上要做的是识别 OWASP 十大调用和攻击的最易受攻击或最相关的应用程序，然后向 Cobalt pen 测试人员开放这些应用程序。

现在，当他们开始这项工作时，Cobalt 已经雇佣了三名白帽黑客专家来攻击这两个应用程序。Cobalt 能够找到并利用未受保护的应用程序中的所有六个漏洞，这证明了这是一个测试版应用程序，它有漏洞，并且从 pen 测试人员的角度来看，这些练习是成功的。

现在，他们进行了同样的练习，并将其应用于受 ShiftLeft 保护的应用程序，结果是他们无法利用我们在应用程序中描述的任何范围内的漏洞。

因此，本练习的证据是确保 ShiftLeft 承担保护应用程序免受现实世界中潜在的白帽或黑帽应用的任何存在性破坏或利用的责任。

绝对的。问得好，艾伦。ShiftLeft 是业界第一个无缝集成到 DevOps pipeline 的转换解决方案，通过识别其攻击面来保护应用程序，而不对威胁做出反应。让我解释一下我刚才描述的。

过去几十年来蓬勃发展的安全解决方案通常是为了从本质上识别和理解现有的威胁，然后保护应用程序而构建的。我的意思是，当应用程序受到威胁或被利用时，该威胁通常会发布在公共数据库中，如 missed CBEDBs 和 NBDBs。

然后，在所有这些保护应用程序的工具验证应用于该应用程序的每个请求后，通知威胁情况。现在，威胁通常有一个生命周期，这意味着一个或多个攻击通过识别潜在的违规路径来攻击应用程序。现在，在他们攻击后，它被命名为未知漏洞，当它被发现时，它就成为零日漏洞，此后，当它在数据库中发布时，它就成为已知漏洞。

现在请注意，组织经常会在从未知到零天已知的旅程中暴露出来，因为它尚未发布。对于所有这些保护应用程序的工具，还需要更新其策略来保护应用程序。

因此，我们在 ShiftLeft 设计了一个转换解决方案，它首先理解在该服务上发展的应用程序。现在，在开发运维的世界里，我们每天、每周或每月部署应用多次。现在，随着应用程序不断发展以满足公司的业务需求，应用程序的攻击面也在不断发展。要么是正漂移，要么是负漂移。通常更多的时候，它存在于或穿越于负面，因为工程师正在高速编码，当他们编码时，他们没有将安全性作为一个功能来考虑。

因此，攻击者可以想出更有创意的方法来利用应用程序不断发展的攻击面。

因此，ShiftLeft 设计了一种技术，可以自动将字节码处理到 devo PS CICD 的 CI 部分，并提取我们所说的应用程序的攻击面。现在，我们从攻击面提取一个策略:该策略定义了如何一次性保护应用程序，因为我们不想将这些警报反馈给人类，并要求人类关联、解决和优先处理这些漏洞。

因此，本着尊重 DevOps 快速运行的精神，我们从这个攻击面提取了安全配置文件，并引导了一个代理程序，该代理程序配备了正在运行的应用程序。

现在代理人的工作就是表现得像一个全职员工。代理负责识别黑帽黑客对应用程序施加的潜在紧急威胁，从而更好地了解使应用程序进入不允许状态的特定途径，因为攻击者通常希望首先使应用程序进入他们希望的状态，然后从应用程序中泄漏数据，或者可能在主机上安装一些恶意负载。

因此，代理的工作是识别和隔离恶意意图，然后阻止特定意图的实现。

**Conikee:** 问得好艾伦。我要说，我们已经开始取得进展。你知道，这是一个旅程，我们必须踏上这个旅程，这个旅程意味着采取一种做法并嵌入到我们典型的管道中——一个我们如何构建应用程序、部署应用程序和保护应用程序的管道。

所以这是一个开始，但是一个伟大的开始。但是使这个开端成为可能的是灵感。你知道，我们经常向其他系统学习，而我感觉到的是安全性正在向 DevOps 学习。DevOps 完全是关于自动化的，这意味着在 Amazon 之前——Amazon 或任何像 Azure 或 GC 这样的公共云出现之前——供应主机的概念需要不止一个步骤:我们必须编写大量脚本来供应简单的物理设备。

随着像 Chef Puppet 和公共云这样的公司的出现，他们创造了这个被称为云形成的概念；也就是说，如何在几秒钟内调配虚拟设备，并使其在此后变得可预测？因为有一个记录您的配置脚本及其随后演变的系统，这是一部分。另一部分是可预测性:如何评估特定的脚本并理解其状态？从资源消耗的角度和资源供应的角度来看？

现在，如果您将这两个相同的原则应用到过时的安全领域，我使用“过时”一词是因为在过去，安全就是对威胁和警报做出反应，然后我们必须处理如何确定警报的优先级，因为警报可能包含真阳性、假阴性、假阳性等等。那么我们怎样才能从《德沃普斯之书》中偷出一页呢？我们如何实现安全自动化？也就是说，你正在创建代码，从这些代码中，我们如何自动提取它的安全状态？

因为在安全领域，我们用三种特殊的过滤器评估代码——或者他们称之为 Sosa、净化程序和同步——Sosa 是消费者与我们的应用程序交互的地方；杀毒程序是我们验证输入的地方；从应用程序的角度来看，同步都是出站通信点。

攻击者通常会在成功利用应用程序后尝试创建新的同步。那么，我们如何提取这个特定的结构或脚本，然后考虑安全功能呢？

所以这是我们在 ShiftLeft 的目标，我们希望这成为一种规范，其他像我们一样的公司也采用同样的原则。
