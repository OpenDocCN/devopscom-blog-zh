Junqueira: 是的，所以当你在开发软件的时候，知道软件在工作是非常重要的。你通常有非常大的团队在创建代码，你不希望一个开发人员互相踩着对方的脚。所以你需要大量的测试来确保当你对软件进行修改时，它仍然可以工作。但是当你添加新的特性，你添加新的测试，你添加新的特性，你添加新的测试，随着时间的推移，你的代码库中的测试数量会增长得非常非常快，这对于开发人员的生产力和开发人员的快乐来说会成为一个非常大的问题，因为实际测试代码可能需要很长时间。如果你在一个大的代码库中做了很小的改变，要运行所有的测试，需要花费几个小时来确保这个改变是有效的。这就是我们要帮助的。

Shimel: 所以你在自动化[测试](https://devops.com/?s=testing)。这公平吗？

我们自动化了测试的测试选择。所以，是的，我们帮助运行测试，但最重要的是，我们帮助选择什么是正确的测试是安全的，但必须为每个变化运行。我们不需要测试整个世界，而是使用一个叫做——我们使用代码跟踪来构建一个依赖图，然后选择正确的测试来运行，然后让开发人员的工作效率更高。

Shimel: 当然可以。当然，我们的观众是非常技术性的观众，熟悉 QA 和测试和开发。所以，当代码进入 Git 时，您正在对其进行跟踪，或者——因为它是预部署代码，对吗？

**朱奎拉:**没错。是的，这是预部署代码。

**Shimel:** 因此，当代码被放入 Git 或其他某种存储库时，您选择要运行的正确测试并在代码上运行它。所以这一切都像 Git-ops 之类的事情一样发生在存储库中？

朱奎拉:是的。它可以发生在与 CI 连接的存储库中，使用持续集成工具，或者也可以发生在本地开发过程中。您可以让多个开发人员同时在同一个代码基础上工作，但可能会对代码进行细微的不同更改。在你的基础上，他们测试他们的代码，同时也使用他们同事的测试——这样他们就不必重复他们的同事已经为相同的代码基础运行的测试。因此，他们只需要测试他们对代码所做的微小改动。

Shimel: 我很喜欢。听起来不错。然后，我不想使用时髦的词语，但它是某种人工智能/人工智能在挑选什么是正确的代码吗——对于这些特定的变化，对这个特定的代码运行什么样的测试是正确的？

这是一个重要的问题。不是 AI。我们只使用简单的数据结构和算法，这些数据结构和算法是我们在进行跟踪时收集的。这实际上很重要，因为如果我们只使用人工智能或任何一种统计方法，我们只能得到有限的信心，即我们正在测试正确的东西。想象一下，你是一家金融机构，或者对冲基金，测试非常重要。如果您没有捕捉到某些错误，它可能会导致数百万美元的损失，例如，收入损失。因此，在代码中检测到正确的回归是非常重要的，如果我们只是使用简单的人工智能方法，我们只能有 90%的信心在代码中运行正确的测试。这还不够。所以我们的客户需要保证。他们需要充分的信心，所以我们使用一种更具侵入性的方法，比如说，因为它检测测试过程中发生的每一个被调用的函数。因此，在初始测试期间，它会对性能产生一点影响。但另一方面，它只是提供了完全的信心，而不是人工智能方法应有的 90%的信心。
