# 2021 年最佳——devo PS 的下一次发展

> 原文:[https://devops.com/the-next-evolutions-of-devops/](https://devops.com/the-next-evolutions-of-devops/)

随着 2021 年的临近，我们 DevOps.com 想要突出今年最受欢迎的文章。以下是我们 2021 年最佳系列的第七部。

随着社会继续从全球新冠肺炎疫情中恢复，一些正常的表象开始回归，技术继续向前发展。疫情无疑改变了正常的方式，人员、流程和技术都在适应和发展，以应对这些挑战。对于许多从业者来说，疫情对您的组织的数字化转型的叙述是真实的。随着 2022 年即将到来，DevOps 正处于以前最前沿范式成熟的尖端，并专注于工程效率。这里有五个 DevOps 发展，你应该继续关注。

## 工程效率是前端和中心

工程效率是一个非常宽泛的术语。在基础层面上，工程效率侧重于使某人(例如，工程师)更有生产力。从组织设计到工程/开发者经验，有许多学科是交叉的。工程效率对当今的组织越来越重要。疫情在资源和人员配备方面对技术部门产生了两大影响。第一，在疫情开始的时候，资源是前所未有的和不可预测的；由于疫情的医疗条件恶劣，以及“被公共汽车撞了”的因素在现实生活中不断上演，一些物理位置被关闭。第二，在复苏期间，大辞职开始了，随之而来的是一场日益激烈的资源争夺战。这两者都意味着资源的可用性，尤其是倾斜资源，是稀缺的。

Spotify 的工程资源部落/公会模式是一个有前途的发展；人们可以频繁地移动，减少工程资源面临的繁重工作，使工程资源不仅可以更快地增加，还可以提高参与度和保留率。该行业继续朝着通常定制的领域的标准化前进，例如发布过程，我认为我们也会看到这种情况继续发展。

## **Git 无处不在**

利用过去为软件工程师保留的源代码管理(SCM)解决方案。但是，随着“ *something-as-code* ”的扩散已经触及了技术堆栈的多个层面，从网络、存储，到最终的开发管道，专注于运营的工程师们正在继续采用软件工程师的许多特质。在您的基础架构堆栈中迭代更短暂/一次性的基础架构正成为常态。保存和打包“代码化”源代码控制的大量配置是进化的一个自然领域。T3】

基于源代码控制的是包管理器——您的 Docker 注册表、Helm chart 存储库等。—对于更可部署的工件；也就是说，将这些工件投入生产是一个过程。因为 Git 在整个技术领域变得无处不在，这导致了 GitOps 的持续采用。由于 Weaveworks 在 2017 年撰写了开创性的文章，定义了 [GitOps 范式](https://www.weave.works/technologies/gitops/)，五年后，利用 GitOps 被视为许多组织的一个可行范式，特别是当开始实施绿地计划时。T3】

## Kubernetes 不再是流血的边缘

2022 年，Kubernetes 将在 GitHub 上庆祝其首次提交的第八个生日。回顾一下技术记忆，在 Kubernetes 之前的八年(2006 年)，VMware 仍是一家私营公司，甚至过了几年 vSphere 才发布(还记得 2014 年在虚拟机上运行工作负载的感受吗？).Kubernetes 生态系统仍然在快速发展，但是将工作负载放在 Kubernetes 上已经不是一个新概念了。由于应用基础设施和架构已经采用了 Kubernetes 的方式(换句话说，是幂等的和短暂的)，在 Kubernetes 上运行合适的工作负载已经成熟。

Kubernetes 的设计是高度可插拔的。如果你不喜欢 Kubernetes 内部的观点或实现，你可以替换那个观点。不喜欢 Kubernetes 处理入口流量的方式？然后从当今众多可用的入口控制器中选择一个。这只是许多可插拔区域中的一个例子。因此，Kubernetes 被视为一种动态的、非静态的资源。使用 Kubernetes 需要反复试验，让您的集群变得健壮、高性能和可靠是一个需要不断迭代的持续过程。

## 作者可以是执行者

毫无疑问，在现代软件栈中，我们被数据淹没了。度量汤以及对可观测性的持续科学和探索(度量、轨迹、日志)为我们提供了很多工作。基于这些数据做出决策，甚至是自动化决策，仍然至关重要。随着站点可靠性工程(SRE)实践的兴起，对使用、故障甚至安全相关项目的高峰做出反应或预测是现代团队的常态。

这些决策可以被写入特定平台理解的策略中，例如云供应商的自动伸缩规则或类似 Kubernetes 生态系统中的开放策略代理。随着如此多的项目向开发团队左移，找到合适的资源、技能或机构知识来编写这些规则可能需要几个团队的投入。由于“ *something-as-code* ”的兴起，不管这些规则的作者是软件工程师、开发工程师、平台工程师还是介于两者之间的任何人，作者都有能力*执行*。

## **向左移动…但是复杂性也向左移动**

古老的格言是，复杂性就像算盘:你可以改变复杂性，但它永远不会真正消失。随着将责任转移给开发团队的趋势，这也意味着相关的复杂性正在转移给开发团队。现代平台工程团队为团队提供基础设施(兼容 Kubernetes 集群),在这些集群上运行的任何工作负载都由拥有它的开发团队负责。通常，开发团队会关注特性和功能。管理大量非功能性需求——甚至是网络等核心基础设施需求——可能是一种负担；考虑您的组织将如何处理服务网格。

如果你是一个开发人员或平台工程师，让你的内部客户——你的开发团队——成功是一个伟大的目标。对此至关重要的是传播专业知识。这可以是自动化和教育的形式。DevSecOps 运动的一个常见实践是将某种扫描步骤作为构建或部署过程的一部分，传播内部信息，包括如何执行扫描、如果发现了什么会发生什么等等。获得内部采用是一个旅程，围绕清晰和稳定建立良好的开发人员体验非常重要。

## 2022 年的进化

尽管过去的一年给了我们所有的挑战、坎坷和教训，但技术仍在继续发展和进步，甚至降低了准入门槛，不断变得更加包容。专注于提高工程效率和减少辛劳将允许更多的参与和降低准入门槛。
