# 在 API 管理的路上发生了一件有趣的事情

> 原文：<https://devops.com/a-funny-thing-happened-on-the-way-to-api-management/>

随着大大小小的 IT 部门对 API 的使用呈爆炸式增长，将 API 作为独立实体进行管理的需求——包括 API 本身和对它们的调用——迅速变得显而易见。我们以前已经经历过几次这样的循环——ORBs、 [DCOM、](https://www.varonis.com/blog/dcom-distributed-component-object-model/#:~:text=DCOM%20is%20an%20acronym%20that,seeing%20that%20in%20Windows%203.1.))、对象库……不胜枚举。

在以前按需远程调用中引入一些管理结构时，有几个相互竞争的目标。我们希望发布和宣传连接，希望我们可以避免(或至少最小化)重新发明轮子，同时我们也希望让试图使用我们 API 的应用程序开发人员更容易访问。同时，我们希望有一个集中的位置，在这里，运营部门可以监控 [API](https://devops.com/?s=API) 的使用情况，安全部门可以审查 API 的合规性，并且可以公布甚至托管“批准”的 API。最后，我们希望让内部开发人员能够访问经批准的外部 API，这些 API 被认为对组织来说是合适和安全的。

与该过程的先前迭代一样，这些目标加在一起被证明是巨大的。在不断变化的 API 环境中，有很多东西需要尝试和覆盖。在这次迭代中，情况变得更糟，仅仅是因为外部 API 的激增。在 REST 之前，每一次迭代都比前一次迭代有更多的 API，包括内部的和外部的，REST 保持了这种模式，尽管现在我们可以回头看，外部位是一条双曲线，而不是直线。

因此，看到市场出现一点点分化，既有趣又酷。整个包是一个巨大的订单，几个公司已经采取了；我将让您来决定他们是否为您的组织处理得足够好。其他人采取了两种方法中的一种——将 API 集中到已经过供应商审查和接口的外部服务，或者支持内部 API 开发人员和用户。

现在，我们有几个可管理的模块要研究。我甚至可以说，大多数供应商都有这样或那样的世界观，在购买 API 管理产品之前，您应该将他们的“倾向”与您组织的需求进行比较。我更希望我们将这些市场分为“API 管理”和“API 集成”，前者做(或试图做)所有的事情，后者简化和支持对 Salesforce 或 ADP 等已获批准的 API 的访问，以及“API 支持”，后者使编写、发布、保护和宣传内部 API 更容易。但我不能打这些电话，所以我们只能看着它展开。

如果您需要简化和保护您的组织对外部平台的访问，并且您使用许多流行的服务，那么从 API 集成开始。如果您需要处理您正在内部创建和使用的 API，请从 API 启用开始。如果你既需要这两者，又需要一个经过批准的 API 的集中存储库，而不考虑其来源，那么可以考虑 API 管理。API 已经成为我们开发工作的核心，将来只会有更多的 API，所以决定您的组织在哪里可以使用帮助并着手研究您的选择是值得的。

当然，你在摇摆。DevOPs 没有让 DEVops 在防火墙上戳太多的洞；DEVops 正在推出解决方案，devOPs 正在让它们继续存在。随着越来越多的组织开发是基于 API 的，这只是保持事情平稳的另一个工具。
