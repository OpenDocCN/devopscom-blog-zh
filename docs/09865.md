# 云:为什么一定要迷惑？

> 原文:[https://devops.com/cloud-why-must-we-be-confusing/](https://devops.com/cloud-why-must-we-be-confusing/)

当我还是一个学习玩[地下城&龙(D & D)](https://dnd.wizards.com/) 的孩子的时候，等级的概念从来没有混淆过——尽管作者应该已经看到它很可能会混淆。对于那些不熟悉的人来说，D & D 这个词有很多等级的用法——角色等级，施法者等级，法术等级，地牢等级，随着时间的推移，越来越多的是怪物等级。我几乎可以保证我在游戏中漏掉了一些 level 这个词的用法，但是你明白了。事实上，如果我错过了你最喜欢的，随时添加评论。

我们使用上下文来确定它的意思，我们没有任何(游戏)世界震惊的后果。尽管有些人非常困惑。

(附注:我曾与《D&D》的联合创作者加里·吉盖克斯共事过一段时间，他知道这种过度使用的情况，但不认为在 D&D 人气飙升后，这种情况可以得到解决。他后来的 RPG 创作确实追求了更高的清晰度和更少的单词重用，但是今天大多数可用的 RPG 系统仍然有许多单词级的用法。)

我很高兴地宣布，我们这些在 RPG 上长大的人现在正在经营供应商公司并做分析。否则，为什么我们会有这么多独特的词的用途？我是说，我们一拿到 DevOps 就把你说的“CD”搞得一塌糊涂。是持续部署吗？[连续发货](https://devops.com/?s=continuous+delivery)？两者都有？类似地，我们畸形的持续集成。让我们称之为 CI——然后让围绕它的整个市场真正成为持续的测试，而它仍然被用来指持续的集成，因为这一点也不令人困惑。

但后来我们跳到了云，并花了一些时间试图让每个人对“云”的真正含义感到困惑。是只有公有云，还是包括私有云？如果是私有云，是仅包括虚拟私有云，还是包括联合主机或数据中心中的云？SaaS 实际上是云吗，还是一种完全不同的动物？

这就引出了我来这里写这篇博文的原因。我们现在被“云”包围了，包括托管在 Kubernetes 上的 Docker。我认为这类似于“水平”这个词用得太多了。

Kubernetes 不是云。它以较小的格式提供类似的基础设施，但 Kubernetes 的关键是它取代了云。一旦实现了云，Kubernetes 就不会关心你的云(或缺少云)。部署 Kubernetes 后，云就变成了基础设施即服务(IaaS)。这一点非常重要，原因有很多，尤其是如果你把 Kubernetes 放到一个云提供商那里，你就有两套基础设施需要锁定。有很多互联网曝光率的两套——因为它不是白叫“公有云”的。

我确实理解将托管在 Kubernetes 上的 Docker 称为“云”的由来；有许多相似之处。但是 Kubernetes 需要一个平台——硬件或虚拟平台——来运行。挥舞着这种嵌套关系的含义是自找麻烦。当云要么已经成为过去，要么已经完全商品化，以至于人们不再考虑它时，Kubernetes 或它的继任者很可能就是原因。因此，让我们轻松地重复使用云。

不过，从短期来看，我的建议是，如果业内有人在他们的产品或分析中提到云，问问他们指的是“级别”的哪种用法。因为老实说，Kubernetes，如果在硬件或 IaaS 上实现，可以被拾起并移动到任何需要它的地方。而那不是“云”。

继续踢后面。您的应用程序在哪里运行并不重要，重要的是您保持它们运行并锁定它们。坚持下去，实事求是——容器编排器不是云平台，无论它们共享哪个功能。我所有的 RPG 爱好者，继续享受乐趣。总会有新的冒险等待着你…不管是什么级别。
