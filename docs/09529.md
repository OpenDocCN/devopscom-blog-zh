# 快！定义 DevSecOps:姑且称之为开发安全性

> 原文：<https://devops.com/quick-define-devsecops-lets-call-it-development-security/>

在很长一段时间里， [DevSecOps](https://devops.com/?s=DevSecOps) 特指像 Veracode 这样的供应商，他们进行静态应用程序安全扫描、动态应用程序安全扫描、软件组合分析和某种形式的运行时监控(通常是交互式扫描)。然后，我们意识到 DevSecOps 的潜在意义远不止于此，像 DevOps 一样，我们将这个词扩展到包含所有将安全性左移的事物。

这就是我们现在的处境。但这让很多人对 DevSecOps 市场感到困惑。我建议我们将专门关注代码质量的工具重新命名为开发安全——尽管这个名字并不完美且包罗万象。它比其他领域的术语更合适，如密码管理，其中包括双因素身份验证(2FA)、多因素身份验证(MFA)，以及越来越多的[无密码](https://securityboulevard.com/?s=passwordless)和秘密管理，实际上*没有一个使用*密码作为其功能的重点。

这样，我们都可以理解 DevSecOps 是一种思维方式和方法，而开发安全性是实现 DevSecOps 的一部分。

可能有比我更聪明的人已经得出了这个(或替代)结论，我也很高兴采用他们的解决方案——这只是试图让那些不生活在安全和 DevOps 市场中的人不那么困惑。如果一个新的人说，“我认为我们需要 DevSecOps”，他们实际上是在寻找他们的意思，以及他们是否真正需要 DevSecOps。我们应该努力澄清这一点，因此，在他们的研究开始时，很明显 DevSecOps 就像 DevOps 一样，是政策、程序、思维模式和产品的总括术语，而可以帮助 DevSecOps 的工具，如构成总括 DevOps 工具链的工具，属于一个名为 x 的细分市场。在我的情况下，我提出开发安全性，因为它非常适合。我们实际上叫它什么并不重要，只要新人们理解它是 DevSecOps 的子集。

当您继续保持系统运转和组织在线时，当您研究这些产品时，使用这样的短语:“我正在寻找适合 DevOps 环境的开发安全工具。”市场很大程度上是由供应商和专家命名的，但最终，所有这些人都想用他们的客户群认可的术语来说话。因此，如果你们中有足够多的人要求开发安全性，他们都会开始这么称呼它。立即生效，我是这么说的。这是一组独特的功能，所有做内部开发的组织都需要，所以让我们给它一个独特的名字。是的，如果你有更好的名字，给我写信。我致力于让它变得更容易，而不是某个特定的术语。
