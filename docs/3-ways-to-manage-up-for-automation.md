# 自动化管理的 4 种方式

> 原文：<https://devops.com/3-ways-to-manage-up-for-automation/>

IT 决策者了解流程自动化在大流行后的恢复中可以发挥多么重要的作用。在[](https://camunda.com/state-of-process-automation-report-thank-you/?submissionGuid=1d672d2a-bf5c-4120-91b1-dec02c22768b)最近的一项调查中，68%的领导者表示远程工作导致他们组织中的关键业务流程瘫痪。在同一项调查中，97%的人认为流程[自动化](https://devops.com/?s=automation)对于数字化转型至关重要，是实现业务成果的重要因素。

所以，全速前进？不完全是。虽然流程自动化的优势已被充分证明，但许多组织仍然对自动化太多的任务犹豫不决，因为他们担心这会使已经复杂的 IT 环境进一步复杂化。

这是开发人员可以帮助流程自动化计划起步的地方。他们无法控制其组织的财务，而且他们通常不会带头规划新的 IT 计划。但是开发人员是负责设计和构建驱动真正商业结果的软件和工具的人。在很大程度上，他们在数字化转型的核心中扮演着关键角色。

为了让组织获得过程自动化的全部好处，开发人员需要更密切地参与自动化项目。他们需要在这些计划如何构建、如何实施以及如何改进方面有更多的发言权。

自动化计划不是一劳永逸的项目。它们必须与端到端的组织过程相结合，这将随着时间的推移而发展。组织不想在两个月或两年后的过程中烘烤不适合他们的东西。考虑到这一点，流程自动化需要灵活、适应性强和智能——而开发人员需要具备推动这些计划向前发展所需的技能和洞察力，从而影响更广泛的组织成功。

然而，为了承担这个更大的角色，开发者必须被给予适当的资源、工具和访问权，以让他们的声音被听到。其中一部分是让他们能够管理高层领导(包括 IT 和业务部门),倡导能够推动真正数字化转型的自动化项目类型。

这里有四种方法。

### **构建自动化演示**

过程自动化项目最大的敌人是时间。最高管理层和高级 IT 领导不想花几个月甚至几年的时间来讨论自动化一系列任务的优点和可行性。他们想快速知道一个项目是否可行，是否应该投入资源。

开发人员可以通过创建一个基本的演示来将项目设置在正确的路线上。它不需要集成为完成的项目计划的复杂的依赖集；它只需要向当权者展示这个项目将会成功。演示手动计算项目如何被季度截止日期触发，并估计它将持续为部门节省的时间。向领导展示员工在反馈报告中提到的棘手问题是可以解决的。

### **展示经过验证的例子**

虽然一个演示可视化了一个流程*如何能够*工作，但是一个用例可以说明这样一个流程*如何在另一个设置中工作。例子可以在任何地方找到——在正式的案例研究、技术文章、GitHub 帖子、开发者留言板或技术合作伙伴提供的材料中。开发人员应该利用他们的网络来获得建议，并运行同行的场景。我们的目标是找到一种情况，在这种情况下，有人用一套可管理的程序解决了类似的问题——并且成功了。拥有一个坚实的例子可以将主动权从理论领域中解放出来，并给开发者更多的可信度。*

### **让企业架构师加入**

一旦确定了演示和用例，下一步就是接受自动化计划的愿景。这是企业架构师的领域。对于开发者来说，让这个人成为你的盟友是很重要的。他们应该与企业架构师合作，以确保任务集适合公司的架构——无论是现在还是将来。如果公司追求微服务战略，开发者需要展示如何在这种技术基础设施内实现一系列产品质量检查的自动化。

### **保持事情简单**

虽然组织努力以敏捷的方式运营，但有时说起来容易做起来难。开发人员报告的人喜欢计划。企业架构师喜欢定义完美的、经得起未来考验的架构。首席执行官们喜欢设想各种情景，并就结果展开辩论。开发人员可以通过鼓励他人避免沉溺于细节的诱惑来打破僵局。他们应该熟悉概念、演示、用例以及愿景——按照这个顺序——然后推广到更大的组织。

虽然流程自动化被广泛认为对业务转型至关重要，但构建一个项目往往比它需要的更具挑战性。鼓励开发人员承担更大的角色可以帮助组织将自动化计划从起点转移到数字化转型的道路上。
