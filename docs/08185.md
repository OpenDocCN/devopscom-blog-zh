# 基于支持 API 的组件保护可组合架构应用

> 原文:[https://devo PS . com/safeguring-composable-architecture-applications-based-on-API-enabled-components/](https://devops.com/safeguarding-composable-architecture-applications-based-on-api-enabled-components/)

应用程序开发中可组合方法的使用正在快速增长。原因是:它提供了许多好处，可以加速开发过程，并向所有技能水平的开发人员开放应用程序的创建。但是，通过重用预建组件获得的速度和易用性优势可能会带来传统安全管理解决方案无法应对的新安全风险。

对可组合架构日益增长的兴趣和采用似乎是在正确的时间采用正确的方法的问题。如今，企业面临着创新、转型和提供新产品和服务的巨大压力。压力来自组织内部和客户。内部人员需要应用程序、工具和服务来提高生产率、效率和削减成本。客户需要快速的服务、更好的参与以及对任何查询和问题的快速响应。

应用程序开发的可组合方法有助于企业按照现代业务变化的步伐交付所需的解决方案。它通过使用提供自包含业务功能的可组合实体来实现这一点。这些元素利用 API，并且可以被组装和组合成更大的应用程序。

**受到其他行业趋势的支持**

可组合应用程序开发流行的另一个原因是，它是其他应用程序开发趋势的良好匹配和补充。

例如，低代码/无代码开发方法增强了它的有用性。在疫情期间，低代码/无代码得到了广泛采用。企业意识到他们需要对快速的市场变化做出更快的响应，他们的家庭员工需要易于使用的解决方案。

今年，它们的使用继续激增。低代码开发技术市场预计今年将比去年增长 23%。

公司提到的许多转向低代码的原因与他们使用可组合应用架构的原因相似。

也就是说，专业开发人员的工作速度更快，他们可以花更多的时间在更高层次的开发问题上。低代码和可组合方法都允许开发专业知识有限或没有开发专业知识的个人和商业团体创建他们自己的强大应用程序。

因此，在许多方面，这两种技术在许多组织中是齐头并进的。

类似地，使用可组合性的许多基本原理和原则与其他云原生方法一致。具体来说，使用云原生方法，应用程序被分解为松散耦合的微服务和容器元素，然后将它们组装成更大的应用程序。云原生应用中的特定元素，如前端应用或业务流程，可以标准化，并作为云原生应用中的可组合组件提供。

**安全性需要什么？**

使用可组合元素进行开发以 API 为中心。具体来说，关于可组合性， [Gartner 指出](https://www.gartner.com/en/conferences/na/applications-us/featured-topics/applications)企业“寻求启动将组织转变为可组合企业的努力”应该:

*   通过确保应用程序开发团队使用可用的外部和内部 API 产品，实施可组合的应用程序设计
*   教育团队如何通过使用应用程序 API 和低代码开发和集成平台来利用现有的应用程序和平台的可组合性。

因此，安全工作必须从传统方法转变为更加以 API 为中心的方法。解决方案必须观察所有用户活动和 API 交互，然后对意外的用户和 API 行为采取行动。

解决可组合应用程序安全性的解决方案必须在单一平台中提供基础设施、服务和代码级威胁分析，以促进协作并加快威胁解决，从而满足现代 DevSecOps 应用程序开发和安全性方法的需求。

鉴于现代云原生应用的复杂性，此类功能至关重要。在许多情况下，当创建具有可组合元素的应用程序时，可能会出现隐藏的相互依赖。传统的安全解决方案通常无法检测到此类问题，或者将问题的根源追溯到可组合元素级别。

此外，虽然应用程序的每个元素都可能是安全的，但是可组合元素之间的相互作用以及它们之间交换的数据可能会带来风险，这就需要一个更加以 API 为中心的解决方案来理解更大的应用程序上下文。
