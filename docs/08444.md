# 发布管理，开发运维的催化剂

> 原文:[https://devops.com/release-management-catalyst-devops/](https://devops.com/release-management-catalyst-devops/)

我们在我的上一篇文章“[devo PS and Continuous Change](https://devops.com/2015/02/24/devops-continuous-change/)”中讨论了缩短和加快运行时间的好处，而不是管理几个“恐龙”版本。我们还提到了通过识别组件/功能依赖性来进行多个发布的方法，而不影响功能和预期的时间、资源和成本的节省，达到软件快速交付的目标。

在大型组织中很容易找到正式发布管理(RM)团队，在 suite 之后有更多的中型和一些小型团队。我们不会触及让核心 RM 团队帮助规划和管理复杂的变更之旅的好处，从应用程序团队移交代码，到运营部门跨测试环境打包和部署，验证测试签署，风险分析，影响评估，最终确保成功的生产部署。

相反，让我们更深入地了解这个**混淆或称为 RM 的第三个变量**如何适合 DEVOPS。在我看来，一个非常重要的演员既不属于开发也不属于运营！

我认为发布管理是一个真正意义上的催化剂，它能够在不经历变化的情况下实现变化，是一个中立的团队，致力于通过在开发和运营之间安排正式的移交来管理变化。

RM 团队传统上是最被低估的，开发和运营占据了每一次成功运行到生产的功劳。它们将对新功能的业务理解或修复功能中断与开发团队的新版本代码发布或基础设施升级的运营预期相融合，这也将提供对业务和开发有利的增强用户体验。

那么，回到我的问题，RM team 在惬意的 DEVOPS 自拍中的位置？

从 DEVOPS 方程中忽略 RM 岂不是不正确？

我冒昧地声明，组织中的 DEVOPS 运动应该由 RM 拥有、推动和决定，将变更/配置和发布管理专业知识添加到 DEVOPS 团队的现有跨职能组合中。

RM 应该推动这种快速交付的协作方法，将双方的人员适当组合在一起，并确保资源的有效利用，获得授权，并在行政监督下支持艰难的决策。

他们将能够在这个快速的交付过程中增加价值，不仅仅是分配他们管理发布周期的任务，而是授权和增强他们的能力。DEVOPS 旗下的 RM 团队也应参与以下主要内容:

*   区分工作簿的优先级，在业务需求和开发交付之间建立平衡。
*   理解复杂的业务需求，遵守地理和地缘政治需求和规则。
*   分布式应用程序集、体系结构复杂性、相互依赖和独立的组件和依赖关系，
*   完全了解基础架构环境、可扩展性、容量规划和迁移。
*   发布计划、日历管理、流程遵守、真正的敏捷冲刺、最佳资源利用、应急或回滚策略计划。
*   部署/发布后的经验教训，在每次运行后回顾、简化或整合流程。
*   确保传递一致的信息，即在软件交付的协作方法中，开发人员和运营人员不是对手，而是合作伙伴。
*   管理报告以及评估的任何风险或强调的不足之处。

这一观点基于我遇到的许多 DEVOPS 讨论，其中没有提到 RM，或者被假定为“开发的一部分并与运营部门合作”。

我对这个思考过程的不同意见不是关于它本质上属于哪个组或者将成为哪个组的一部分，而是在整个发布过程中给 RM 分配一个无关紧要的角色，而事实上他们应该是驱动者！
