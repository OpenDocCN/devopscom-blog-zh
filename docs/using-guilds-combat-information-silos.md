# 使用行会对抗信息孤岛

> 原文：<https://devops.com/using-guilds-combat-information-silos/>

在 xMatters，我们由 100 名左右的天才工程师组成的团队被细分为团队，处理从前端工作到发布自动化，再到用户界面(UI)和用户体验(UX)的所有事情。在一个有敏捷思维的组织中，我们的工程团队成员是跨职能的，他们的知识和专业技能延伸到他们主要关注的领域之外。这种跨功能的优势是增强的、自组织的协作形式，以及对项目需求和目标的更有背景的理解——所有这些最终导致更快的创新。

七年前我加入公司时，我们有三个工程团队。我们已经在利用敏捷方法消除尽可能多的低效，并简化持续交付流程。然而，当我们发展到六个、七个甚至八个以上的团队时，我们意识到敏捷原则不足以衡量收益。仍然有一个临界点，关键信息开始储存，团队之间的沟通障碍开始蔓延。

尽管我们的平台取得了卓越的性能和令人难以置信的进步，但我们仍有发展的空间，可以变得更聪明、更快、更高效。我们需要在团队层面上解决这个问题，为此我们发现另一种文化转型是必要的。

事情是这样开始的:我们观察到我们的许多工程师对他们通常范围之外的领域有浓厚的兴趣，我们可以将他们的兴趣作为一种宝贵的资源加以利用。有可能出现另一种团队——不那么官方，更灵活——让我们能够捕捉新的想法和问题的解决方案，并让我们的工程师有机会从事他们感兴趣的工作。

因此，我们采用了最初由 Spotify 开发团队首创的“公会”概念。尽管 Spotify 的结构更复杂，因为它有近 2000 名工程师(它利用团队、部落、分会和公会)，但前提是相同的:允许不同团队的人流畅地走到一起，为共同的目的进行合作。我们的团队已经利用 Slack 进行内部交流，所以我们也为公会创建了相关的渠道。

例如，UX 公会独立于我们专门的 UX 团队，包括来自前端和其他团队的成员。我们还有一个测试协会，主要由 QA 团队成员组成，他们每月面对面或通过视频会议会面。我们的架构协会的成员来自几乎每个工程团队，他们对讨论我们构建平台和构建代码库的方式感兴趣。每当人们有一个与架构相关的问题时，他们会将它发布到架构公会，其成员会设计、记录并审核一个解决方案。

在一个案例中，我们需要一种方法来配置我们客户的所有环境，并在我们遍布全球的数据中心中复制这些配置。关于这个问题的讨论出现在架构公会的渠道中，通过它我们收集了所有必要的信息和专业知识来完成一个 spike 并编写一个完整的架构文档。

如果你在一个敏捷思想的组织中，并且正在考虑实现行会，最重要的事情是创建一个开放的、友好的环境。不要试图将它们结构化或使它们过于正式；原则上，它们应该是几乎没有等级制度的自治实体。关于公会最酷的事情之一是他们会被你的工程师社区自我审查，所以一旦一个不再服务于一个功能或者变得多余，它自然会失败。我们在一个自动测试协会中看到了这种情况(针对自动化测试和手动测试)——它基本上被测试协会吸收了。

记住，除了汇集不同领域的专家，你还邀请了对学习感兴趣的新手。这种方法有很多优点:你不仅可以从免费的用户测试中获益——以 UX 公会为例——还可以利用各种技能和创造性思维来解决问题。正如爱因斯坦曾经说过的，“我们不能用创造问题时的思维来解决问题。”美国宇航局将这一想法铭记于心，长期以来一直使用公共众包来解决人类在太空中的问题，并创造新的创新项目。

最终制定解决方案的人通常在与问题本身相同的领域没有任何传统经验或专业知识。有时，需要那些离问题远近的人的合作来克服它。
