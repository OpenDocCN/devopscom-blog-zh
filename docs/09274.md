# DevSecOps 团队如何升级

> 原文:[https://devops.com/how-devsecops-teams-can-level-up/](https://devops.com/how-devsecops-teams-can-level-up/)

2019 年，我写了一篇文章，详细介绍了 [DevSecOps](https://devops.com/6-traits-that-define-devsecops/) 的特点——这种做法可以定义为安全自动化的过程，其中 IT 和安全更加孤立。DevSecOps 引入了一种更有意识的文化，在这种文化中，设计安全原则在早期就被融入到软件开发过程中。总而言之，我们的目标是让安全性支持开发，而不是阻碍开发和降低 SDLC 的速度。

那么，几年后，DevSecOps 怎么样了？它实现了这个理想吗？我最近会见了对比安全公司的首席技术官兼联合创始人杰夫·威廉姆斯，了解了现代开发人员的现状。据 Williams 称，大多数大公司在应用程序安全性和软件依赖性方面仍然存在实际问题，普罗维登斯经常受到质疑。我们最近已经看到这一现实在像 [Log4j](https://devops.com/?s=Log4j) 和 [Spring4Shell](https://devops.com/?s=Spring4Shell) 这样的框架中暴露出来。

大型组织的工作清单中可能有成千上万的漏洞。然而，作为消费者，我们很高兴地信任关键应用软件，却不知道其背后隐藏着什么。“这个行业有一个真正的透明度问题，”威廉姆斯说。威廉姆斯补充说，随着应用程序越来越相互关联，它们变得越来越复杂，也越来越重要。“这三个 c——关联、复杂和关键——增加了安全性的难度。”这些条件最终形成了他所谓的“完美风暴”

## DevSecOps 团队应采取的 6 种方式

那么，DevSecOps 团队应该如何应对似乎永无止境的新漏洞和攻击呢？Williams 分享了一些团队可以增强其安全态势的一般方法。

### 1.在阳光下练习安全

当出现问题时，很容易指责他人，尤其是当问题是严重的漏洞或攻击时。相反，威廉姆斯鼓励团队实践他所谓的“阳光下的安全”这种态度使得内部的安全问题更加透明。拥抱一种无可指责的回顾文化可以确保员工乐意在团队中提出问题。

阳光中的安全也可以采用面向外部的方式。威廉姆斯将其比作在[产品制造](https://www.thebalancesmb.com/bill-of-materials-2221363)中，与贸易伙伴共享物料清单(BOM)是一种标准做法。Williams 建议说，我们可能会看到类似的标签标准以软件材料清单(SBOMs)的形式出现，这样外部消费者就能更好地意识到依赖性和潜在威胁。

网络安全意识当然需要公共努力，需要透明的沟通。我们每天都在 bug bounty 程序中看到这一点，CVE 报告并修复了开源软件。将潜在威胁暴露在光天化日之下是在漏洞被利用之前将其粉碎的必要条件。

### 2.翻译高级需求

必须牢记 NIST 和 ISO 等机构发布的安全指南。实施这些指南推荐的概念，如最小特权原则、多因素身份验证和[零信任架构](https://containerjournal.com/topics/container-security/zero-trust-architecture-zta-to-secure-a-nation/)，被大多数安全专业人员视为最佳实践。然而，这些高层次的需求通常不会转化为底层的开发人员。“开发人员需要将这些需求分解成四个层次，转化成他们正在使用的具体技术，”Williams 说。

程序员需要更现实的建议，比如这些指导方针如何影响 Spring Boot 应用程序，或者他们应该如何在 AWS 中配置权限。根据 Williams 的说法，团队必须用更具体的注释来解释这些抽象的建议，以满足他们正在工作的技术堆栈的细微差别。需要相同的过程来帮助开发人员创建符合数据法规的项目，因为很难将这种高层次的建议付诸实践。将这些需求翻译成可读性更强的术语是解除安全和开发团队孤岛的一个很好的方法。

### 3.换挡*智能*

近年来，有很多关于左移的讨论。这个想法是为了减少以后的安全威胁，组织需要在软件开发生命周期的早期更紧密地掌握安全性。通过使用自动化分析让代码对潜在威胁负责的工具，通常可以实现安全性的左移。

然而，Williams 认为这是组织需要小心的领域。并非所有的安全工具都是开发人员友好的，可能会导致大规模的低效率。他不主张左移，而是提倡“聪明地移动”,在最有效的地方引入安全性。“你不会想把所有的安全措施都向左转移——你也想向右转移，”威廉姆斯解释道。"把它分散在整个生命周期中."

### 4.避免额外步骤

烤蛋糕的方法不止一种。类似地，有不止一种方法可以创建一个优秀的软件应用程序。因此，大多数组织都在处理多种工具，这使得在所有堆栈中一致地应用安全性变得更加困难。

Williams 解释说，安全性应该迎合正在使用的特定工具和过程。但是这需要回答某些问题，比如软件开发团队在他们的旅程中处于什么位置？他们是瀑布型还是敏捷型？每个 CI/CD 管道的行为如何？

显然，有许多独特的因素需要考虑。Williams 说，理想情况下，工具应该足够灵活，能够将其自身附加到现有的基础设施上，而不是向已经庞大的工具生态系统添加额外的步骤。

### 5.认真对待安全性

当然，还有文化因素。“现在是 2022 年，公司仍然没有认真对待安全问题——他们需要改变，”Williams 说。“他们如何到达那里并不重要，最重要的是他们到达那里。”

诚然，文化变革是一个缓慢的演变过程，最好的前进方式是随着时间的推移逐步增加实践和技术。因此，Williams 建议在旅程的每一步反复插入片段，测量结果并证明价值。

### 6.更新过时的工具

怀特哈特安全部门发现，2021 年春天修复关键漏洞的平均时间是 205 天。但是如此长的时间与新威胁的速度和快速修补它们的需要不一致。

Williams 说，行业正在改变他们使用 AppSec 的方式。他说，传统的 DevSecOps 安全工具，如静态分析工具、动态分析工具和 web 应用程序防火墙，与交互式应用程序安全测试(IAST)和运行时应用程序自我保护(RASP)等新技术相比，已经过时且不够准确。准确性、速度和可扩展性将是安全自动化工具的重要特征，这些工具不需要额外的人工干预，也不会产生太多的误报。

## DevSecOps 仍在不断成熟

更健康的 DevSecOps 实践旨在降低 Williams 所说的“漏洞逃逸率”。希望通过对设计提供即时反馈，工程师将学会不重复同样的错误，从而减少他们引入的漏洞数量。

总的来说，DevSecOps 还在不断成熟。展望未来，更强大的 DevSecOps 方法将需要更开放的态度来减轻威胁。它还需要实际的指导，因为并不是所有自顶向下的指导方针都能很好地转化为活跃的开发环境。除了对生产环境进行主动监控之外，应用程序需要适当的代码卫生，软件供应链本身也必须得到保护。
