# 将你的团队从释放思维中解放出来

> 原文:[https://devops.com/releasing-team-release-thinking/](https://devops.com/releasing-team-release-thinking/)

您在优化交付渠道上花费了多少时间？我猜是没有。许多团队已经找到了自动化他们的发布的好方法，一些甚至自动化了测试和监控。但是他们不经常采取下一步措施，并从这些结果中学习来改进交付过程。

如今，自动化负载测试和应用程序监控非常流行。我确信这是通向更好、更快的软件的正确道路，我很高兴看到许多组织正在拥抱自动化工具。但是许多人没有利用这些工具提供的巨大机会。他们能做的不仅仅是提高发布的质量；他们提供的反馈可以改进整个发布过程。

因为当组织开始使用交付自动化工具时，他们通常不考虑他们的端到端过程，他们不构建他们的工具链来提供将驱动任何有意义的变化的测试结果。即使他们花了一些时间研究结果，通常也不过是看看漂亮的图表，指出一些警告信号。不要误会我；这总比什么都没有好，但它错过了一个巨大的机会。如果组织准备花费时间来安装代理，加载跟踪脚本，并运行测试，他们应该准备好考虑这些时间如何帮助改进过程，并最终改进产品。

应用程序监控和负载测试不应该被认为是发布过程的结束。这只是开始。随着时间的推移，来自生产中的多个测试和发布的结果将显示您的整体部署过程运行得有多好。如果它是经过深思熟虑构建的，结果将开始显示出漏洞可能在哪里。

很快，团队就可以发现发布中的趋势，并将这些趋势与发布周期中的活动联系起来。例如，一个真正快速的负载测试与应用程序监控相结合，表明某些页面返回 404，这意味着您不仅显著提高了应用程序的性能，实际上您只是非常快速地提供了错误页面。或者您可能会发现您的效率低得多，即使产品团队增加了与新人数成比例的积压工作。无论是直觉还是反直觉，这些见解都是基于证据的，很容易在您的测试组织中应用。

**超越个人发布的思考**

很难不陷入具体化的发布思维，在这种思维中，整个团队生活在个体发布周期的潮起潮落之中。甚至有人会说，有了这种心态，团队实际上是在做一个真正快速的瀑布方法——不是敏捷，当然也不是 DevOps。

虽然组织利用先进的技术来衡量生产中应用程序的质量，但他们也是在逐个版本的基础上这样做的。这对于个人的发布有很大的帮助，但是对于告知团队整个过程的效率没有任何帮助。

DevOps 不仅试图提高每个版本的速度和质量，还试图建立所有版本之间的关系。这有点像元发布管理。如果一个组织有效地观察发布之间的关系，他们可以优化他们的部署管道，迭代它，并像改进一段代码一样改进它。

有效地做到这一点的唯一方法是收集大量的数据，在每个版本结束时进行自动化测试，并关联所有以前的版本。但是它不仅仅是漂亮的图形和错误页面警告。如果你从整体上来看这些数据，你会对你的过程以及单个版本有一个健康的认识。

**哦，是的……这需要计划**

我知道这不容易。分散的开发工具市场面临的最大挑战之一是让两个工具像一个工具一样协同工作。在构建同类最佳工具时，很少会考虑到其他同类最佳工具。因此，通常由实施组织来规划他们的综合视图。

在最好的情况下，应该有可能跟踪一整年的所有发布，并且在年底知道你的团队/管道在发布软件时是最有效的。希望趋势显示出全年的改进，但是更有可能的是你会发现一些季节性，与积压的大小的相关性，或者与发布的功能类型的关系。

这不是偶然发生的事情；需要深思熟虑。更多的工具和更多的“开发人员”不会单独解决这个问题。事实上，它们可能会增加更多的问题。例如，在单个负载测试应用程序中，组合多个版本的负载测试结果可能很容易。但是一旦第二个工具进入，就很难了:例如，很难将多个负载测试与来自生产中被监控的应用程序的时间序列数据相关联。而且，通常情况下，更多的人员意味着更多的分工:一个人负责负载测试，另一个人“拥有”应用程序监控。即使有优秀的人员和优秀的工具，复杂性也可能先于全面而简洁的规划出现。

最终，除了单个版本之外，更明智地了解整个渠道也很重要。应用程序监控和负载测试的结果是一个很好的起点。

有一个元素完全不受你的控制。这就是 ISV 产品决定合作的程度。大多数 ISV 关注于他们各自的用例，并且只关注与其他工具的集成作为业务开发工作。但这是可能的。

**你可以今天就做，只要提前考虑**

例如，一个名为 [Blaze Meter](http://blazemeter.com/) 的很酷的移动后端负载测试工具是一个非常容易设置负载测试、运行负载测试和检查负载测试结果的地方。测试的输出通常与特定的版本相关联，并且每个测试的结果都是个性化的。所以你可以深入了解每个特定的测试，但不是所有的测试。

将其与一个流行的应用程序监控工具(如 NewRelic 的新洞察产品)相结合。它进一步扩展了其流行的监控功能，允许用户编写应用程序测试，就像用近乎自然的语言编写问题一样。将 BlazeMeter 结果输入 NewRelic Insights 将所有负载测试与所有应用程序监控结合在一个界面中。

我选择这个特别的例子是因为我知道它是可行的。实际上，您可以非常轻松地从 Blaze Meter 中获取测试结果，并将它们注入 Insights，并在结果上构建一个仪表板。还因为这两种工具通常被认为是在应用程序发布流程的最后阶段以及进入运营和监控阶段的工具。

在规划阶段，开发、QA 和运营团队需要对他们希望能够完成的问题和分析的类型有所了解。整个团队应该能够看到发布管道的质量，并且能够指出成功和弱点。这样，他们可以选择工具并与目标进行整合。

最终结果是一个真正的 DevOps 过程。该团队不仅将快速迭代应用程序修复，还将迭代应用程序周围的流程，提供高度可见的持续改进——管道的*和*为什么。
