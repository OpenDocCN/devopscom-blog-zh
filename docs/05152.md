# 无服务器和 GraphQL:以思考的速度构建和启动

> 原文:[https://devo PS . com/server less-and-graphql-build-and-launch-in-the-speed of the-think/](https://devops.com/serverless-and-graphql-build-and-launch-at-the-speed-of-thought/)

现在，网站和应用程序通常是同一个东西。传统的在线或可下载应用程序开发方法可能需要几个月的时间，因为需要管理后端开发方面——基础架构供应和管理、安全性、可扩展性、一致性等，尤其是对于用户遍布全球且潜在使用需求非常高的公司而言。

此外，一旦应用程序构建完成，可能需要几天时间才能启动并确保全球可用性。一旦启动，该应用程序可能需要持续的维护和监督，以确保可用性和性能，并不断微调过度配置和配置不足之间的平衡，以管理成本。

当与 GraphQL APIs 结合时，无服务器计算是应用程序开发的圣杯。这些技术消除了服务器供应和管理方面的顾虑，使开发人员能够以极快的速度构建和启动应用。开发人员只需要担心如何构建一个具有可用性特征的引人注目的应用程序，以优化参与度。此外，使用无服务器和 GraphQL，应用程序可以在几小时甚至几分钟内启动到全球用户群，而不是几天，同时避免了管理服务器的需要。让我们来看看。

## GraphQL 是什么？

GraphQL 是一个 API 标准，用于构建基于众所周知的 HTTP 协议的现代数据驱动应用程序。GraphQL 最初由脸书在 2012 年创建，是社交媒体巨头将其移动应用从 HTML5 转移到原生移动应用的关键。基于其成功，脸书在 2015 年将 GraphQL 作为开源项目向公众开放。

使用 GraphQL，您可以围绕您的数据结构在以下方面定义一种不可知的领域特定语言(DSL ):

*   你使用的编程语言。作为一个 API 标准，GraphQL 对很多语言的实现都是开放的，比如 JavaScript、Java 和 C#。本质上，您可以将标准与您选择的语言一起使用。
*   您使用的数据库引擎。GraphQL 只关心准备好数据，而不是数据的来源和它来自哪里，所以您可以自由地用几乎所有的数据库引擎实现 GraphQL API。

## 无服务器有什么大惊小怪的？

无服务器代表了我们创建和运行 web 服务和应用的方式的重大转变。无服务器计算在云中运行，将开发人员从考虑后端任务的负担中解放出来，例如管理基础架构、扩展服务器以及配置容量和资源。

下面，我列出了无服务器提供的一些具体优势，特别是无服务器数据库——许多这些优势也是其他无服务器产品的一部分。总体影响是，使用无服务器技术，开发人员只需为他们使用的内容付费，或者根据用户需求为应用程序使用的内容付费。当流量达到峰值时，开发人员可以在几秒钟内扩展到几乎无限的磁盘空间和计算负载，并在用户离线时缩减规模。无服务器的 NoOps 方面意味着消除了维护成本和负担。此外，全局可用性区域允许在不同位置之间进行复制，并允许您将数据保存在需要的位置—靠近用户以最大限度地减少延迟。内置加密意味着开发者只需掌握基本的安全知识就能保护他们的应用。

*   基于消费的计费。
*   自动缩放。
*   没有运营开销。
*   全局低延迟。
*   默认安全。

当这些好处与 GraphQL 的优点结合在一起时，人们会注意到这两种技术之间的完美协同。不仅过程变得更加自动化，标准还促进了内部团队之间的交流，使客户更容易与您的软件集成，通过:

*   API 可发现性和自动化文档。
*   数据整合。
*   与语言和数据源无关。
*   类型安全。

## 无服务器加 GraphQL 等于开发者天堂

有许多公司成功利用无服务器加快建设速度并降低成本的例子。最令人印象深刻的演示之一是澳大利亚的学生成功地重建了人口普查应用程序，并处理了官方 900 万美元的系统无法处理的负载，而成本只是使用无服务器技术的一小部分。GraphQL 进一步发挥了这些优势。

Airbnb，一家已经以采用无服务器而闻名的公司，声称由于 GraphQL，移动速度[快了 10 倍](https://medium.com/airbnb-engineering/how-airbnb-is-moving-10x-faster-at-scale-with-graphql-and-apollo-aa4ec92d69e2)，而 Coursera 注意到，随着 API 的发展，他们的前端开发人员发现[发现和查询新数据](https://blog.apollographql.com/courseras-journey-to-graphql-a5ad3b77f39a)更容易了。这些好处不仅仅局限于开发商自己的公司。例如，由于基于 GraphQL 的面向公众的 API，GitHub 客户端现在可以更高效更容易地获取数据。GraphQL 甚至在与 SQL 竞争——当像 [FaunaDB](https://fauna.com/faunadb) 这样的数据库提供原生的 GraphQL API 时，用户会迅速转向 GraphQL。

鉴于无服务器在消除服务器供应和维护方面的优势，以及 GraphQL 在使用编程语言和数据库引擎方面给予开发人员的自由，开发人员基本上能够以他们想到的速度在全球范围内构建和启动应用程序。而最终，我们不都希望客户有更好的体验吗？
