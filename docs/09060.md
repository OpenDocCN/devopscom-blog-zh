# 无论事情如何变化，都要有一个计划

> 原文：<https://devops.com/no-matter-how-things-change-have-a-plan/>

有足够的零钱了吗？很有趣的是，我和其他一些技术人员十年前就有过这样的对话，从那以后，一切都没有慢下来。我们的结论是，在不久的将来，变化将是不变的。这个结论是正确的；我不得不联系其他人，看看他们怎么想，但是“不久的将来”比我预期的要长。

然而，不可避免的是，变化有时需要休息一下。看起来，高科技公司终于准备好承认让你的员工筋疲力尽不是一个可行的长期商业模式，并且正在放缓……然后，新冠肺炎来了。现在，我听到有人抱怨说要回到无休止的变革中去，就好像过去几年是一个假期，员工们都休息好了，准备更换大量的基础设施。

然而，变革需要间歇的原因并不是 100%与员工和[精疲力竭](https://devops.com/?s=burnout)有关。这只有在你不停地做出一吨*吨*的改变时才会显现出来。在员工精疲力竭成为问题之前，恢复和业务连续性计划就会过时。以我们一直以来的速度，我甚至可以说它们可能毫无用处。

是的，所有这些变化给我们带来了一些很酷的东西。当我们变得更忙时，一些忙碌的工作变得更容易，甚至不存在了——但这并不能改变你需要呼吸的事实。这是一个很好的时间来回顾计划，以保持业务在 IT 灾难的情况下，或至少一些重大的系统灾难。说“我们将只是部署一个新版本”是很酷的，但它只在某些情况下有效。我谈到了这一点，并讨论了一个在【2021 年 10 月不起作用的案例。但是还有很多。每个环境都是不同的，但硬件故障并没有消失，破坏性写入仍然会发生，这只是另外两个例子。你的信息安全小组可以给你更多的例子。

知道最坏的情况发生时你要做什么是非常重要的。对于那些从未经历过的人来说，在之前知道*人们会有压力，并向你询问时间表，会让你省去很多麻烦。*

对于那些从未有过灾难恢复或业务连续性计划的人，可以考虑一下。至少要有一个基本的灾难恢复计划，列出关键系统、最有可能出错的地方，以及恢复和恢复正常运行的步骤。制作这样一个列表的过程通常会发现应该做但没有做的事情，比如备份或备份验证。

对于那些有灾难恢复或业务连续性计划的人，我大胆地说，“它已经过时了。”组织倾向于创建它们，然后忘记它们。这些计划应该涉及每个项目的一部分，所以它是以小块的方式处理的，但是，可能因为灾难恢复或业务连续性的第一次实施通常是一个大型项目，所以它往往会被搁置起来，每隔几年重新审查一次。不要那样做。

正如我以前说过的，你和你的员工已经构建了现代企业的核心。别忘了保护它。使用这些可爱的新技术进步来实施灾难恢复程序并不需要花费太多时间。对每个主要计划进行灾难恢复审查/更新，无论是新产品还是对现有软件的重大更改。继续摇摆。如果你在你的组织中有发言权，考虑放慢一点节奏，甚至更新博士。如果你没有发言权，希望你的雇主足够明智地这样做。
