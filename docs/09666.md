# 避免安全审查延迟

> 原文:[https://devops.com/avoiding-security-review-delays/](https://devops.com/avoiding-security-review-delays/)

2021 年夏天，我与西雅图一家领先科技公司的高级安全开发人员共进午餐。尽管我们在太平洋西北部阳光明媚、凉爽的下午很放松，但毫无疑问，我的朋友很沮丧。他的部分工作是通过流行的扫描产品运行代码，然后评估结果。这是一个艰难的角色，因为他坐在开发代码的团队之外，并且他经常在压力下在期限的限制内运行他的分析。

很多时候，他不得不传达扫描发现安全问题的坏消息。但这并不是他沮丧的唯一原因。他的问题是，他使用的工具需要几个小时才能完成扫描。他在周末开始扫描，并希望在周一早上完成。本该让他的工作变得更轻松的工具实际上是一件令人头疼的事情。

这是分析软件系统[安全漏洞](https://devops.com/?s=DevSecOps)的基本挑战之一。开发人员和利益相关者经常将他们的项目时间表集中在功能规范上。团队狂热地工作，从原型到成品快速迭代。T3】

安全问题从来不会被忽视，但是“让我们先看看我们是否能让它按照我们想要的方式工作，然后再关闭它”的文化最终会花费更多的时间，因为评估和测试周期会更长。更不用说，如果发现实质性的安全问题，在不破坏本应已经完成的工作的情况下，很难修复这些问题。这个问题在 SaaS 世界变得越来越严重，在那里，解决方案的构建速度比以往任何时候都快，而几乎没有安全监督。

## 正确的工具和软件至关重要

首先，经理和执行官需要规定整个软件开发生命周期和每个环境都应该有安全性。特别是对于基于 web 的 SaaS 工具，任何存在的漏洞都是潜在的攻击媒介。攻击者知道追求一个开发版本可能不会带来什么好处，但它可能会给他们提供暴露在以后版本中工作的凭证的权限，或者识别出一个在生产中不为人知的漏洞。

第二，自动化和工具化只是游戏的名字。使用带有[代码库](https://securityboulevard.com/?s=GitHub)的工具，管道管理和测试框架有助于确保消除尽可能多的潜在错误。安检应该也是一样的。我们不能总是依赖开发人员或管理员的技能来遵循安全最佳实践。T3】

我们需要提供可靠和严格检查的工具。而且，这些工具需要经常运行。SAST 扫描应该足够快，开发人员可以在每次保存或至少每次提交时运行它。将 SAST 和 IAST 扫描耦合在一起是安全开发的黄金标准。每次将代码编译到开发或测试环境中时，都应该对其进行测试，以确保漏洞没有悄悄进入，并且系统的权限和体系结构不会相互冲突而暴露漏洞。

最后，每个经理办公室都必须解决一个基本问题。代码质量并不等同于安全性。当然，可能会有一些重叠，但是有 200 条代码质量规则和 50 条安全性规则是没有意义的。这两件事是不相等的，期望拥有高质量的代码而不经常进行安全检查最终会导致花在安全审查上的时间更少是一种谬误。不要只见树木不见森林。确保安全是团队的首要任务。

## 倡导 SaaS 的安全

我的朋友认为让团队对他们写的代码负责没有问题。事实上，他喜欢能够坚持高标准，并为他的组织赋予他这样做的权力而自豪。这正是我们在软件行业所需要的:敢于向同龄人要求最好的个人。但是，我们必须为他们提供有效的工具，让他们能够依靠这些工具取得成功。

安全测试延迟的问题对于软件开发来说并不新鲜。有充分的证据表明，修复生产中的错误可能需要 640 倍的资金，但由于缺乏有效的工具，这个问题在 SaaS 开发中被放大了。SaaS 开发环境带来了独特的挑战，仅仅通过对 SSPM 供应商进行代码分析或配置测试是无法应对的。它需要对环境有一个全面的了解，并且能够使用定制来有效地识别和补救漏洞。

纵观全局，负责安全的人和实施安全的人会将安全视为快速创新的必需品，而不是推迟到最后才希望可以避免的必要之恶。也许下次我们见面时，我的朋友就不会这么沮丧了。
