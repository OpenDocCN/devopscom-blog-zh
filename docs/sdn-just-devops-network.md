# SDN 只是网络的 DevOps 吗？

> 原文：<https://devops.com/sdn-just-devops-network/>

*答案取决于你对 SDN 的定义。*

最近通过 Twitter 对 SDN 和 DevOps 之间的关系进行了几次热烈的讨论，一些人呼吁将网络和 DevOps 的婚姻称为“网络运营”，其他人认为这是不必要的，各方都充满热情。

此外，Network Instruments 最近调查的近 40%的网络专业人士[已经确定 SDN 是“不可定义的”,难怪我们无法确定 SDN 和 DevOps 之间是否存在关系。](http://www.networkinstruments.com/about/press-releases/2014/2014-05-12.php/)

正如维兹尼在《公主新娘》中告诉伊尼戈·蒙托亚的那样，是时候“回到最初”，从 SDN 最初试图解决的问题的角度来看待这段关系了。不是厂商提出的由 SDN 解决的问题，而是市场希望 SDN 解决的问题。

#### SDN 试图解决什么问题

开放网络用户组(ONUG)及其最近由 ONUG 董事会撰写的白皮书《[开放网络的挑战和机遇](http://lippisreport.com/2014/07/open-networking-challenges-and-opportunities/)》是一个开始探索的好地方作者清楚地陈述了他们对“开放网络”(包括 SDN)的期望，然后顺利地与开发运维及其他以运营为中心的概念联系起来:

> “开放网络必须解决的关键难题是当前网络的配置、操作和管理挑战。研究表明，高达 80%的网络工程师时间花在执行手动配置上。从 IT 的角度来看，网络已经成为问题；例如，配置一台虚拟机(VM)需要几秒到几分钟，但配置网络和相关的第 4-7 层设备需要几天到几周。使用 SDN 软件的开放式网络可以开发一个编排堆栈，并通过利用开放式 DevOps(例如 Puppet、Chef)和其他基于 Linux 的功能来显著缩短网络部署时间。"

虽然整篇文章中有一些元素体现了 SDN 的原始本质，即控制平面和数据平面在网络中的分离，但该群体的大部分要求都是为了解决一个特定的棘手问题:加快网络服务供应。基于 OpenFlow 的 SDN 不仅关注第 2-3 层(路由和交换)的网络，还关注网络的其他部分:防火墙、负载平衡器、身份和应用访问所在的第 4-7 层。对成功部署应用程序至关重要的服务，占据了部署生命周期的大部分。

毫无疑问，DevOps 与 SDN 一样容易被误解(就像早期的云一样)，但无可争议的是，它的目标之一是通过各种措施来改善应用部署，包括上市时间和部署频率。这正是上面所要求的–通过减少配置网络服务所需的时间来缩短上市时间。*所有*网络服务，而不仅仅是子集。

根据 ONUG 的论文，如果我们看看实际市场希望 SDN 在哪里，它不是关于数据包处理和 OpenFlow 的。市场真正想要的是一种 DevOps 方法来配置网络服务，以减少部署所有这些第 4-7 层服务的时间以及执行手动配置所花费的时间。现在，有两个新兴的 SDN 阵营:一个专注于**数据包**，另一个专注于**流程**。前者紧紧抓住 SDN 的原始(据称是官方的)定义，并通过专注于动态、可编程的**数据包**处理，对其进行了字面上的理解。后者将可持续发展网络视为实施该网络的机会；通过专注于优化和协调**流程**，将一致性、可预测性和可重复性的概念应用于网络服务供应。

DevOps 与 ONUG 提议的不同之处在于，DevOps 专注于运营**应用**部署，而 SDN 专注于运营**网络**部署。双方都认识到需要将基础设施视为代码，在通用平台上实现标准化，以减少部分导致错误率上升的变化，并实现配置自动化，以降低与应用程序部署相关的运营成本。

那么 SDN 对于网络来说是不是相当于 DevOps？

如果不是，那就应该是。
