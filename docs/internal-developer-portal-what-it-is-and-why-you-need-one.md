# 内部开发人员门户:它是什么以及为什么需要它

> 原文：<https://devops.com/internal-developer-portal-what-it-is-and-why-you-need-one/>

云的一个主要优势是它为[开发者](https://devops.com/?s=developers)提供了灵活性和自助式解决方案。每当开发团队需要新的服务器或集群时，不必联系中央基础设施团队，开发团队可以通过简单的 web UI 自己完成。这一变化极大地提高了生产率。但是有一个警告:大多数公司实际上并不想使用云提供商提供的默认门户。这有几个原因:安全性、简单性或对定制、公司特定选项的需求，等等。这就是为什么我们现在看到内部开发人员门户的趋势。在本帖中，您将了解什么是内部开发人员门户，以及它们如何使您的开发人员的生活更轻松。

## 什么是内部开发者门户？

内部开发人员门户是一个自助式应用程序和数据存储，允许开发人员和管理人员跟踪和组织他们的工程团队构建和操作的一切。

假设您需要创建一个新的微服务。您的开发人员首先需要为其提供基础设施；为此，他们将使用云提供商门户。接下来，他们需要创建新的 CI/CD 管道或修改现有管道。然后，他们可能需要对 API 网关和数据库等现有组件进行一些更改。最后，他们可能需要添加一些文档，或者让他们的新微服务对公司的其他人可见。

总的来说，他们需要接触一些不同的 ui 或门户。现在，想象一下他们可以在一个地方做所有的事情。这就是我们所说的内部开发者门户。这是一个自助服务门户，允许开发人员分类和理解关系、所有权、资源、外部服务数据以及他们理解和操作软件所需的任何其他内容。

## 为什么内部开发者门户如此重要？

现在我们知道了什么是内部开发者门户，让我们后退一步，讨论一下为什么内部开发者平台(IDPs)正在兴起。主要原因是因为软件和基础设施的复杂性增加了。特别是当我们谈论基于云的微服务环境时，即使是高级工程师也很难跟踪所有依赖关系和每次需要更改时必须发生的事情。微服务架构是高度分散的。虽然你仍然会看到典型的[](https://www.ibm.com/nl-en/cloud/learn/three-tier-architecture)三层软件架构，但在大多数情况下，仅仅通过名称或服务器位置来查看一个随机的组件，不再容易判断它在更大的画面中的位置。

## 自然进化

第二个原因是，这只是自助服务门户发展过程中的一个自然的下一步。老式的内部环境几乎没有任何自助服务功能。你还记得那有多难吗？要申请新服务器，您必须向中央基础架构团队发送电子邮件。如果你幸运的话，你会在几个小时或几天内得到你需要的东西。但更多的时候，需要一两个星期。如果你在最初的请求中犯了一些错误，那会让你多等一周(或者更久)。更先进的公司？即使他们创建了一些定制的自助服务解决方案，这些解决方案也是有限的。通常，它只不过是一个由基础设施团队处理的在线表单，这常常成为所有开发团队的瓶颈。

当大多数公司开始向云迁移时，他们开始看到自助服务平台的好处。在几分钟内创建服务器的能力提高了开发人员的工作效率，加快了上市时间，提高了软件质量。但是同样，云提供商平台适合管理云基础设施；要获得完整的解决方案，您需要的不止这些。您的开发人员可以更有效地创建和更改云基础架构，但其他所有事情仍然是单独完成的。这就是我们如何自然地演变成对内部开发者门户的需求。为了在提高开发团队生产力的同时带来更大的灵活性和清晰度，您需要采取自助服务平台发展的下一步，将开发人员需要的所有东西合并到一个内部开发人员门户中。

## 清晰度

此外，从业务角度来看，内部开发人员门户带来了清晰性，并提供了所有基础设施和软件开发工作的良好概览。即使您在云中，并且您的开发团队可以按照他们的选择管理基础设施，您仍然需要一些控制和监督——例如，避免在开发人员忘记删除的测试服务器上浪费金钱。当然，您可以从您的标准云提供商门户获得简单的报告，但是如果您使用多个云提供商或者有一个混合云环境呢？内部开发人员门户对开发人员和他们的经理都有好处。长话短说:IDP 门户是软件开发的未来。

## 国内流离失所者实际上能做什么？

您如何为您的公司创建 IDP？你可以从零开始构建自己的软件，使用一些 OSS 产品或者使用现成的产品。

### 建立你自己的

第一个选项将为您提供最大的灵活性和能力，以完美地定制解决方案来满足您和您的开发人员的需求。然而，从零开始构建自己的 IDP 几乎就像从零开始构建自己的操作系统一样。这需要大量的工作，可能会抵消国内流离失所者应该带来的主要好处。您希望实施 IDP 来提高开发人员的工作效率并缩短上市时间，但是您需要花费数月时间来构建它，并花费数年时间来维护它，以保持它与不断变化的云 API 一起正常工作。

### 开源 IDP

下一个选择是开源的 IDPs。因为 IDP 通常是相当复杂的软件，所以没有太多的选项可供选择。最受欢迎的两个是后台和 Gimlet 。Backstage 更像是一个构建开发者门户的 DIY 平台，这意味着你仍然需要花费相当多的时间来创建一些实际上对你的公司有益的东西。Gimlet 也很受欢迎，但是它侧重于基于 Kubernetes 的环境。两者都是很好的工具——不要误解我的意思——但是如果你正在寻找一些现成的、功能丰富的工具，可以考虑第三个选项。

### 现成的解决方案

最后一个选择是使用一个[现成的产品](https://configure8.io)。现成的解决方案将 IDP 功能与事件响应能力以及指标、依赖关系可视化和跨多个云的过滤相结合。

内部开发人员门户越来越受欢迎是有原因的。事实上，许多公司多年来一直试图建立 IDPs，甚至不知道该解决方案有一个名称。这是因为类似 IDP 的解决方案会在任何关心开发人员体验和更快发展的公司中自然地发展。问题是，通常需要数年时间才能找到满足他们所有需求的单一解决方案。另一个问题是，这些有机创建的内部开发人员门户大多只关注技术方面，而忽略了业务需求。这就是为什么专门构建的工具可以更容易、更快地实现，同时还能更好地完成工作。
