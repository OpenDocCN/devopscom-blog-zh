# 金融服务开发中最薄弱的环节

> 原文:[https://devo PS . com/devo PS-for-financial-services 中最薄弱的环节/](https://devops.com/the-weakest-link-in-devops-for-financial-services/)

如今，金融服务领域的许多公司正在对 IT 转型进行巨额投资。他们正在[采用 DevOps 和敏捷最佳实践，实现遗留应用的现代化，并将其基础设施迁移到云中](https://fincondx.com/)。

然而，在许多情况下，这种时间和资源的重大投入并没有产生对更广泛的金融服务组织来说重要的可衡量的结果。开发人员可能编码更快，新的操作效率可能实现，但其他方面几乎没有变化。团队根本无法在支持决定市场成败的核心业务战略方面发挥作用。

为什么会脱节？通常，那些被分配到 IT 转型项目的人是孤立工作的。他们无法知道他们是否提供了客户需要的东西，何时以及如何提供。此外，他们无法交付产品开发和战略团队所需的指标和反馈，以指导持续改进和创新。

在艰难时期，转型团队往往是金融服务领域最先被解雇的人，这有什么奇怪的吗？他们只是无法证明他们工作中的重大投资是值得的。

## 两步解决方案

如果您的 DevOps 团队致力于现代化，可以考虑这个两步走的方法来解决您所做的工作和您所服务的核心业务之间的脱节。

首先，不要戴着眼罩工作。拓宽思维，让所有关键利益相关者参与进来，从业务战略家和开发人员到运营专家和面向客户的支持团队。让这一更广泛的群体参与进来，可以帮助你从多个角度了解成功是什么样的，以及你需要在哪些方面投入时间和精力。您可以确定什么结果对您的组织真正重要，以及您需要监控什么标准来展示您产生的价值。

其次，是时候重新思考测试以及它在您的组织中所扮演的角色了。采用一个自动化的、与技术无关的平台，该平台可以向扩展团队中的每个人提供持续的反馈，而不仅仅是编写代码的开发人员。

一家全球 500 强银行和金融公司的经历表明了拓宽测试方法的重要性。该公司将测试视为一个事件，是线性应用程序开发过程的一部分。项目被划分优先级，代码被编写，测试被执行以确定工作是否没有错误。

测试的创建和维护依赖于手工操作，这使得它不可能跟上敏捷团队的持续代码开发。此外，专有测试工具很难使用。在开发的早期，不可能向左移动来持续集成经过充分测试的代码。而且，不可能将权利转移到监控和学习应用程序性能和客户体验上。

因此，该公司缺乏一个连续的预测性见解和反馈循环，多个地理位置的利益相关者可以访问和使用这些信息来推进业务的优先级。

该银行重新思考了其方法，并采用了基于开源标准的新人工智能平台。现在，所有的利益相关者都可以从任何地方运行测试和监控服务，使用任何人都容易使用的工具。他们可以做到这一点，而不会让敏感数据离开他们的网络。他们拥有创造焦点和交付重要结果所需的洞察力。

## 做一个明智的投资

如果您准备拓宽您的 DevOps 思维模式，并向持续测试转移，请记住以下提示。寻找一个开源的、企业就绪的测试自动化环境，它易于使用并与您现有的测试工具无缝集成。此外，坚持以下特点，这将有助于您扩大测试和监控计划的战略影响:

*   **单一平台业务洞察:**选择一种解决方案，帮助您让所有人都保持一致，实现预测性洞察和反馈的持续流动，并且易于访问和理解。确保您可以跟踪来自多个构建的趋势，并且可以与您的所有团队成员和利益相关者共享交互式测试结果和图表。
*   **On**–**需求可伸缩性:**许多遗留工具的负载生成能力有限。寻找一个可以帮助你克服障碍的解决方案。您应该能够跨多个地理位置按需上传测试和模拟网络条件，同时模拟数百万虚拟用户。
*   分散化:您的新测试环境应该允许您在编排新版本时通过左右移动来扩展您的测试程序。确保您可以轻松地代理新的自动化工具进行持续监控，并授权所有利益相关方使用它们。这种民主的方法将帮助您克服手工测试的瓶颈，并在问题出现时促进更快的干预。
*   **自助式测试创建:**确保新平台的用户界面是“加载并运行”的，不需要专有语言或特殊技能。任何人都应该能够从本地机器创建小规模的自助式测试，然后通过云无缝地重用相同的测试，以加速测试自动化，作为您的持续交付管道的一部分。

## 金融服务公司的收益

是时候确保您的转型努力朝着正确的方向前进，并且您的团队没有脱离更广泛的业务孤立工作。扩展您的思维模式和工具集，以涵盖关键利益相关者，并与您组织的业务战略保持一致。

将测试自动化嵌入到软件交付生命周期的每一个活动中，并创建一个连续的反馈循环，通知规划、开发和操作。当您这样做时，您可以提高速度、改善质量、提高效率并与客户期望保持同步，同时支持对您企业的成功最重要的事情。
