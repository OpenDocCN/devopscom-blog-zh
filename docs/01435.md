艾伦·希梅尔:嗨。我是 DevOps.com 的艾伦·希梅尔，来这里参加另一个 DevOps 聊天。我很高兴 Sumo Logic 的首席技术官兼联合创始人 Christian Beedgen 能加入我们。克里斯蒂安，我希望我没有把你的名字毁得太惨。

克里斯蒂安·比根:不，很好。

艾伦·希梅尔:欢迎。Christian，我们之前已经安排好了，所以我想采访你，因为相扑逻辑有太多的事情要做，但是我意识到也许我们的一些听众可能还不熟悉这家公司。所以在我们深入一些细节之前，你能给我们一个高层次的，50，000 英尺的相扑概述和你们在那里做什么吗？

克里斯汀·比根:绝对是。Sumo 是一家机器数据分析公司。这是我们在做日志管理的一种高调的说法。我们从 2010 年就有了。我们为市场带来的一大独特优势是，我们实际上提供了这种服务。所以没什么好装的。重点是实际使用该产品来理解日志、机器数据，以便进行故障排除和应用程序日志分析，以及堆栈上下的所有内容。

*艾伦·希梅尔:*优秀。所以日志作为一种服务。你是对的。这个想法是一个非常 50，000 英尺的视角。Christian，开发-运营市场正在迅猛发展，这是工具供应商的热门领域。然而，特别是相扑逻辑，在开发-运营社区中的认知度方面总是处于或接近顶端。你认为这是什么原因？

Christian Beedgen: 我认为我们处于一个非常幸运的位置，我们的产品基本上符合市场需求。我来自这个日志管理领域，我的联合创始人也是如此。我们可能知道的不多，但我们肯定知道一些如何构建帮助人们管理日志的产品。

日志和任何与监控相关的东西都是任何操作活动的核心。因此，开发人员对 Sumo 正在做的事情感到兴奋对我来说并不奇怪，因为这是他们必须能够做的事情的一部分，我认为我们真的很好地支持它，并且我们经常会出现在他们的工具箱中。

*艾伦·希梅尔:*优秀。Christian，Sumo 人员现在似乎正在传播的新消息或新功能是将日志与度量标准统一起来的想法。你能给我们的观众一个想法吗？你这话是什么意思？

Christian Beedgen: 事实上，我非常高兴能够谈论这个话题，因为这个话题已经酝酿了很长时间，从 4 月份开始，我们可以公开谈论这个话题了。这让我想起了 2010 年。我早就想这么做了。

从我自己的观察来看，当然在 Sumo 我们正在运行一个相当大规模的分布式系统，所以我们有很多自己的见解，比如我们需要什么来运行一个大规模系统，以及与客户交谈并观察他们实际需要什么来使他们的应用程序运行得更好，更好地解决他们的问题并让他们更好地回到正轨。

日志当然是其中非常重要的一部分，但是对于机器数据来说，另一方面是指标。那么什么是度量呢？指标本质上是时间序列数据。它通常来自于宿主。它从 CPU、I/O 指标、网络指标开始，然后一直到应用程序，您知道，每秒登录多少次，您的 API 响应时间是多少，以及诸如此类的事情。

虽然这些可以潜在地在日志中捕获，但是日志不一定是处理指标的最有效的方式。但是指标非常重要，我认为它们可以和日志一起给你一个全面的概述，让你知道你的系统在做什么。

显然，我们从我们的内部使用和内部运营使用案例中了解到这一需求，我们也从客户那里听说，他们不太喜欢我们所说的那种转椅分析，在这种分析中，您必须在不同的工具之间来回切换，一个用于度量，一个用于日志，一个用于其他工具。我们认为我们处于一个很好的位置来统一这些东西，这就是为什么我们现在基本上将一个额外的指标功能作为 Sumo Logic 的一部分推向市场，而不是作为第二个单独的产品来构建，我们决定，嘿，它实际上是同一高级用例的一部分，也是我们独特的主张、价值主张的一部分，我们可以在同一工具中做到这一点，我们可以合并两个不同的数据源，并从中驱动集成的用例，如果这有任何意义的话。

艾伦·希梅尔:太好了。我只是想让我们的观众明白。这还没有正式发布，或者它还处于测试或早期访问形式，或者它是什么？

克里斯蒂安·比根:没错。我们已经宣布了。我们在四月宣布了它。显然，我们已经与设计伙伴合作了一段时间。正如我们所说的，我们的许多客户实际上正在使用它，我们继续从他们那里得到很好的反馈，而且，我们现在意识到我们可能仍然需要修复的部分内容，你知道，就像你决定推出新功能时一样。

我们目前正计划在 7 月底全面推出这款产品。所以我们会祈祷我们能赶上那个日期。所以实际上它很快就会出现在市场上。与此同时，我们基本上允许人们尽早注册，我们也看到了人们对此的极大兴趣。

*艾伦·希梅尔:*优秀。现在，如果我们的听众中有谁目前不是相扑的客户，对此感兴趣，我当然可以理解他们为什么会感兴趣，Christian，有没有什么地方可以让他们注册，或者至少获得更多的信息，如果不是真的成为早期用户的话？

克里斯汀·比根:绝对是。去 SumoLogic.com，它会带你去那里。

艾伦·希梅尔:太好了。Christian，如果你不介意，我们还有几分钟，我想稍微转一下话题。基本上是我自己的事。我实际上刚刚开始写——或者写完一份关于容器和容器技术的调查。似乎每个人都在谈论这件事。你在做什么?你对此有什么策略？我想问一点关于:这对相扑来说是一个挑战吗？还是一根木头是一根木头？

克里斯蒂安·比根:这实际上是一个非常好的问题。事实证明，为了保持领先地位，我们必须做几件事情，我想这是从 2014 年开始的。我记得那是码头工人破产的一年。然后，我们开始考虑这到底意味着什么，当然，即使对于我们自己的运营也是如此。但是，我们的客户发出了一个重要的声音，他们当然马上投入了进来。

事实证明，当有了容器的额外抽象层时，收集日志的方式实际上会有一点点变化。因此，我们基本上在 2015 年全年都在思考这个问题，并想出了如何从集装箱化的环境中收集日志。在这方面已经有了几次迭代，当然进展非常快，而且就 Docker 而言，他们在日志工作方式上也取得了一些进展。

但我们现在有了第一党的支持，我称之为对 Docker 的全面监控策略。这包括日志、指标或 Docker 世界中所谓的统计数据，以及这类 Docker 事件和可以在 Docker 之外查询的附加元数据。所以你不仅知道你的容器的想法是什么，还知道它运行的图像是什么，等等。我们可以将所有这些整合到相扑中，除此之外，我们还有一个非常好的相扑扩展，我们称之为 app。这是我们的客户正在使用的 Docker 应用程序。

所以确实变了。主要的变化实际上来自于——从技术上来说，主要的变化是我们必须做几件事情才能收集日志和指标，因为这基本上是最大的底层变化。

从更高的层面来说，关于 Docker 的好消息是，我认为它提供了大量的元数据，如果你知道如何获取的话，而且主要是通过 Docker Inspect API 来进行环境分析，当然现在这种分析更加高度抽象了。您甚至可能不知道哪个容器运行在哪个源上，等等。所以有很多额外的元数据使它变得非常漂亮。所以我喜欢这样。

艾伦·希梅尔:非常酷。克里斯蒂安，我们只剩下几分钟了，就一两分钟，我还有两件事，我的清单上还有两个问题要问你。集装箱肯定是一个非常热门的项目。Dev-ops 被大肆宣传。你认为下一步是什么？下一件大事是什么？

克里斯蒂安·比根:我想这是每个人都在思考的问题。当我意识到 AWS Lambda 实际上是什么的时候，我就想，“哇。这真的非常非常酷。”我还不知道它将走向何方，但我认为直觉告诉我，在许多方面，亚马逊实际上已经发明了容器之后的未来，与此同时，他们发布了弹性容器。所以我以为那是 _____。

无论如何，我认为已经出现的术语是无服务器，无服务器计算。考虑到堆栈已经变得如此复杂，我认为这实际上非常有趣。它在某种程度上增加了另一个抽象层，但基本上破坏了堆栈的所有其他底层。我认为这是目前正在发生的一件非常有趣的事情。

艾伦·希梅尔:是的。嘿，一旦我们有了 t 恤，就会有风投来投资。

Christian Beedgen: 我最近读过的最好的一本书是——哦，天哪，这是一本很沉重的书。我们实际上正在就此举办一个内部读书会，它让人们大吃一惊。这里到处都是物质。一个叫 Reinertsen 的人把它叫做*产品开发流程的原则*。这是应用于产品开发的第二代精益。这家伙是个学者。我喜欢那本书。我认为这是一种思维拓展，所以*产品开发流程的原则*。那是本很棒的书。
