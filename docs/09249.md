# 新遗迹为观测平台增加了 SLO 工具

> 原文:[https://devo PS . com/new-relic-adds-SLO-tools-to-observability-platform/](https://devops.com/new-relic-adds-slo-tools-to-observability-platform/)

新遗迹本周使[成为新遗迹一号可观测性平台](https://www.businesswire.com/news/home/20220405005460/en/New-Relic-Launches-Service-Level-Management)中普遍可用的服务水平指标(SLI)和服务水平目标(SLO)能力。新功能使开发运维团队能够定制指标、统一的运行状况报告和警报。

New Relic 总经理兼产品工程高级副总裁 Alex Kroman 表示，目标是在越来越多的组织依赖软件来推动数字业务转型计划的时候，让 DevOps 团队能够更轻松地优化跨多个应用程序和软件组件的服务。

当然，几十年来，IT 团队一直在跟踪 SLO。然而，随着应用程序环境变得更加分散(部分原因是微服务的兴起),实现和维护 SLO 变得更具挑战性。Kroman 指出，DevOps 团队通常缺乏对哪些指标与业务最相关的理解，即使他们了解，他们也一直依赖于手动过程和专用工具。

他补充说，相比之下，New Relic 提供这些功能的方式使管理服务级别变得更简单，而无需额外成本。例如，开发运维团队可以通过一次点击创建 sli 来自动建立所需性能和可靠性的基线。该平台还提供基于历史数据的建议，以建立基准并定制和配置 sli 和 SLO。

这些功能使基于基准、标签和报告(包括为服务所有者和业务负责人定制的视图)跨团队设置服务边界和跟踪可靠性变得更加简单。科尔曼说，实际上，通过 New Relic 公开的应用程序编程接口(API ),使用仪表板来获得整个业务流程的可见性成为可能。

就 SLO 的现代化而言，现在还为时尚早，但很明显，随着时间的推移，它们将作为代码与 DevOps 工作流中的其他东西一起被管理。组织越依赖应用程序来推动数字业务计划，开发运维团队跟踪 SLO 就变得越重要。必须做出的决定是，手动实现某种形式的 SLO 即代码与依赖可作为云服务访问的可观察性平台相比，在多大程度上是有意义的。

当然，可观察性一直是 DevOps 的核心原则。然而，很少有组织能够真正实现它，因为大多数 it 团队一直在监控一组预定义的指标。现代的可观察性平台使得站点可靠性工程师(SREs)更容易针对度量、日志和分布式跟踪的聚合实例发起查询，以确定问题的根本原因。接下来的问题是，如何将这些查询的结果与企业期望从一组应用程序中获得的服务级别相关联，这些应用程序可能会推动订单到现金的流程。毕竟，了解 IT 环境中发生的事情并不总是容易转化为普通业务主管能够容易理解的报告。
