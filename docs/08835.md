# 支持开发人员实现基础设施自动化

**支持开发人员实现基础设施自动化**

率先向市场推出新的和改进的应用程序和服务是成功组织发展和区分业务产品的方式。持续的软件交付意味着一切都是流动的和不断变化的，以实现所需的速度和敏捷性，确保应用程序交付越来越高的消费者满意度期望。

随着需求的变化，发布和推出的所有产品都需要经常评估和更新，并添加新的功能和特性。开发运维、持续集成和交付实践在简化软件管道和加速软件发布方面发挥着越来越大的作用，但是对于企业来说，支持管道的基础设施仍然是孤立的，在软件交付管道的每个阶段都要进行(可能是多次)供应。

虽然[基础设施](https://devops.com/beyond-iaas-the-benefits-of-using-eaas/)仍然是一项技能密集型、基于阶段的实践，但对于管道执行速度和敏捷性以及扩展开发运维以满足业务需求的需求来说，没有比这更大的阻碍了。

结果是 IT 组织无法跟上步伐，难以为开发和测试团队提供他们需要的基础架构环境，同时保持安全性、合规性和效率。为了帮助实现速度和控制，许多专注于开发运维的公司开始转向基础设施自动化。这种方法的目标是让开发运维团队能够及时访问他们需要的环境，同时保持对成本和治理的控制和监督。

按照当今的业务运营方式，DevOps 需要立即可用的生产环境。当开始新项目时，开发者不能等待数天或更长时间。一些企业试图通过构建自动化脚本来设置环境来应对这一挑战。

问题是自动化脚本通常是定制的、手工编码的，并且需要越来越稀缺的技能。当需要对环境进行更改时，定制脚本需要同样高水平的技能来理解、评估、更改和测试环境，这占用了管道执行的关键时间。

开发运维团队需要即时访问基础架构。解决方案必须以应用管道为中心，自动化，按需和自助服务。它必须是一个平台，不仅能在整个管道中交付和管理环境，还能与现有的基础设施工具集成，包含它们所创建的内容，以确保企业满足业务需求，同时不抑制应用软件开发创新。

**不可变基础设施的作用**

为开发人员提供基础设施自动化是可重用自动化和不可变基础设施的概念。不可变基础设施的概念是按照一组精确的规范来构建基础设施组件。如果需要对规范进行更改，则根据更新的要求提供一套全新的基础架构，而以前的基础架构由于过时而停止使用。

这对开发运维团队的自动化基础架构有何影响？连续交付利用自动化使部署变得常规和可重复。为了实现这一点，交付建立在版本化和自动化的概念之上，以将部署推向开发管道上的可变环境。

不可变的基础设施开始发挥作用，因为消除了 IT 繁重的基础设施杂务，如实现更新和安装补丁。如果基础设施元素需要更新或必须被替换，则旧元素被已经做出这些改变并被合并、测试和验证的新实体替换。

一句话:可重用的不可变构建块使得交付开发人员真正需要的资源变得更加容易。

**解决治理问题**

应该使用基础设施自动化来确保治理。然而，传统的云治理方法受到功能的限制，通过复杂的脚本实现基础架构自动化。这导致自动化成为主要的维护挑战和时间承诺。对于多云环境，复杂性成倍增加。

明智的方法是将基础架构自动化的重点放在管理跨公共云资源的治理上。将基础架构自动化交给 DevOps 实践者并不意味着放弃治理控制，而是确保通过控制使用、安全性和成本的策略来维护和执行企业合规性。这可以防止云蔓延，而不会减缓开发周期。

**总结**

对于 IT 组织来说，满足当今不断增长的开发运维基础架构需求是一项非常耗时的任务。它减慢了应用程序开发流程的每个阶段。需要一种解决方案，让开发人员和 DevOps 实践者能够按需访问他们需要的云基础设施和应用程序环境。这是使用环境即服务(EaaS)平台实现的，可通过 GUI 门户、命令行界面和现有 CI/CD 工具进行访问。

必须为 DevOps 实践的所有团队成员设计现代环境自动化平台，以自动化基础架构环境的请求、供应和生命周期管理。平台必须在不影响效率或限制创新的情况下集成。It 必须提供具有自助服务自动化功能的按需环境，并监督和实施治理合规性。该解决方案是寻求简化应用程序开发、测试和发布到生产的连续软件交付管道的组织的答案。
