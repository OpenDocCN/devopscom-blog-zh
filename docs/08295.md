# 报告:持续交付改进的空间很大

> 原文:[https://devo PS . com/report-lots-room-for-continuous-delivery-improvement/](https://devops.com/report-lots-of-room-for-continuous-delivery-improvement/)

持续交付(CD)基金会发布的一份报告表明，就实现这一目标而言，大多数组织还有很长的路要走。根据对 SlashData 收集的开发人员国家调查数据的分析，近三分之二的开发人员表示，从代码提交到代码在生产中成功运行至少需要一周时间。只有十分之一的开发人员每天在生产环境中多次部署代码。

另一方面，一半的开发人员报告说，他们在不到一天的时间内就从计划外停机中恢复了服务。

该调查确定了被用作 shell 脚本语言的顶级编程语言，Go/Golang、JavaScript、PHP、Scala、Dart、C#、Ruby、Python 和 Java。

令人惊讶的是，在实现[连续交付](https://devops.com/?s=continuous+delivery)方面走得最远的垂直行业是已经接受数字化业务转型的零售行业，而落后最远的是电信行业。

裁谈会基金会执行主任 Tracy Miranda 指出，即使经过 10 多年的倡导，开发计划署对裁谈会最佳做法的采用仍处于萌芽状态。许多组织已经采用了所谓的持续集成/持续交付(CI/CD)平台，但它们主要用于驱动一组 CI 最佳流程。相比之下，CD 被证明是一个更难实现的目标，主要是因为部署应用程序的每个平台都有独特的属性。她指出，因此，跨多个平台的应用程序交付自动化很难实现和维护。

然而，随着基于云原生容器的应用程序的到来，这些应用程序注定要部署在 Kubernetes 集群上，这些集群公开了一组一致的应用程序编程接口(API)，更广泛地实现 CD 的前景正在改善。运行 Kubernetes 集群的每个平台将向 CD 平台公开相同的 API。

然而，从短期来看，Miranda 指出，许多 IT 专业人员仍然对 Kubernetes 和 CD 最佳实践有些熟悉。她补充说，太多的组织仍然倾向于将 CD 视为一个获取的平台，而不是一套需要定义和培养的流程和技能。

然而，无论如何，随着越来越多的组织意识到他们是多么依赖软件来提供差异化的客户体验，CD 挑战将需要得到解决。客户体验改善的速度将决定组织能够获得和保持的客户亲和力水平。Miranda 补充说，这并不意味着每个组织在实施 DevOps 最佳实践时都必须像网飞那样精通。然而，她说，已经很明显，大多数组织需要改进他们交付软件的方式，以推动多种数字业务转型计划。她说，现在的机会在于回顾这些过程，着眼于创造一种直接专注于不断改善客户体验的文化。

在大多数组织真正掌握 CD 之前，可能还有一段时间，但是，正如大多数旅程的情况一样，最重要的事情是简单地迈出第一大步。
