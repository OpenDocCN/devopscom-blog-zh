# COBOL:完全过时但无处不在的语言

> 原文：<https://devops.com/cobol-completely-obsolete-omnipresent-language/>

*编程语言 COBOL 是个讨厌的东西，是每个人都想摆脱的东西；但是，我建议你去学。这可能是一座金矿。*

IT 世界是一个快节奏的世界，过时是规则，事物发展如此之快，任何想要了解它的人都必须不断学习和培训。语言、方法和技术一直在变化，不断发展，以使编程更容易、更安全、更健壮，最重要的是，更快。每条规则都有一个例外。

1959 年，确信英语比机器码或汇编语言更适合编程，格蕾丝·赫柏创造了第一种人类可读的编程语言:COBOL

COBOL，55 岁了，还在年轻的时候，从来没有想过退休，已经成了一个讨厌的东西，每个人都想摆脱但没有人有勇气去尝试。

我们准备好取代 COBOL 了吗？

如今，80%的金融交易都是通过 COBOL 程序完成的，而且大多数程序都是几十年前编写的，已经被证明是可靠、有效和高效的。最重要的是，大多数仍在使用它的公司已经在这些系统上投入了大量资金，如果它没有坏，为什么要修复它呢？

那么，为什么要改变它们呢？嗯，不是 Java。COBOL 是在一个没有今天意义的时代产生的。程序的设计主要是为了取代简单的重复性任务，如从数据库中检索数据，执行一些操作，生成报告或更新一些其他数据。今天的程序必须将前端呈现给成千上万的用户，在访问后端以最终改变数据并将一些信息发送回用户之前，要经过不同的安全、逻辑和快速数据检索层。或者，一个程序可以成为设计物体和建筑的工具，与你的朋友交流，粉碎你的对手，增强你的现实，你能想到的…程序概念背后的整个哲学在过去的二十年里发生了如此大的变化，以至于拥有这样一个几乎没有人知道的过时技术代表着一种潜在的危险。现在只有很少的人学习 COBOL，即使这些系统不应该有太多的发展，但总是有维护要做。但是我们准备好取代它了吗？

**快速、健壮、面向未来的语言**

COBOL 多年来一直处于死亡的边缘，但它的存在总体上没有改变。不仅从一种技术转移到另一种技术的成本巨大，而且替代技术本身的选择也是一个百万美元的问题。有什么比 COBOL 更好的方法来执行典型的 COBOL 操作呢？尽管 COBOL 并不是像 Java 或 C#那样被设计成面向对象的语言，但它已经慢慢地采用了它们的一些方面，扩展了它自己和它的功能，达到了几乎一切皆有可能的程度。除了创建用户友好或好看的人机界面。在这个问题上，自霍普的时代以来，什么都没有改变。我们中的许多人仍然记得花费数小时获得一个带有几个可修改字段的简单屏幕的沮丧，那些仍然使用它们的人肯定每天都在梦想用简单的按钮取代 F5 和 F6…

但这从来都不是重点；关键是要有一种快速、健壮、经得起未来考验的语言。

**COBOLs 主要竞争者**

关于哪种语言可以取代 COBOL 的讨论还没有结束，但是下面是一些主要竞争者的优缺点:

–Java 易于学习、广泛传播，并且已经有了一个庞大的开发人员社区，但是与其他语言相比，它的速度相当慢，尽管存在一些专门的编译器，并且在最近几个版本中有了很大的改进。也可以说它还处于年轻阶段，每年的演变和语言的变化对它不利。更别提最近的安全问题了。

–c++快速、灵活且健壮，但它需要时间来学习和理解，并且有可能是它最大的缺点，即潜在的内存泄漏。COBOL 和 Java 程序员几乎从未听说过的东西。从概念上来说，它为开发者提供了很大的自由，但这也意味着几乎没有现成的组件，因此它不太划算。

假设有一天我们设法找到了一个明确的赢家，另一个主要问题就会突然出现。COBOL 是为了在一个块中执行所有类型的操作而设计的，但是多年来我们已经认识到模块化是成功的关键。因此，如果我们用另一种语言取代 COBOL 本身，我们还需要有数据库和 SQL，前端和 HTML，很可能是一个业务流程管理软件来具体化工作流…基本上，取代 COBOL 程序意味着创建全新的程序。

**终极建议**

虽然有可能取代这种老化的语言，但这需要极高的成本，我们可以预计它至少在未来二十年内会出现，这给我们带来了本文的最终建议:如果你有机会学习它，并且有足够的动机在黑色界面上写白色，那就去做吧。有能力的 COBOL 程序员越来越少，而需求没有变化，这意味着这种个人投资可能会发现自己是一座金矿。
