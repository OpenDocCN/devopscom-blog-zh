# 保护开源应用

> 原文:[https://devops.com/securing-open-source-apps/](https://devops.com/securing-open-source-apps/)

**通过 API 扫描和可观察性保护开源应用**

构建应用程序的企业总是大量使用开源软件。向云原生应用的转移、低代码开发的使用以及可组合架构的出现使得开源应用更加普及。因此，企业需要一种方法来确保复合应用程序可靠、性能良好并满足强大的安全需求。

测试并确保开源应用程序具有这些企业级特征需要思维的转变。传统工具通常只评估特定程序的特征，或者产生太多的遥测、日志和跟踪信息，以至于很难发现问题。使传统方法更加困难的是，现代应用程序通常通过连接松散耦合的元素来构建，掩盖了可能导致性能或安全问题的相互依赖性。

**从 API 扫描和发现开始**

现代企业在一个以 API 为中心的世界中运营。无论是使用微服务、可组合元素还是低代码平台，开源应用都是通过 API 向其他元素公开数据和流程来构建的。

一个问题是，在过去，API 的设计使得应用程序和应用程序组件可以通过安全的私有(公司拥有和管理的)网络相互通信。现在的趋势是让公司构建的应用程序和服务通过 API 在公共网络上提供给第三方(如客户、合作伙伴和供应商)。

随后，如果公司的应用程序和数据没有得到适当的保护，API 会为外部人员提供一个直接的入口点。虽然简单地保护所有 API 似乎是一件简单的事情，但问题很容易出现。首先，公司可能会处理成百上千个不同的 API，这些 API 都是经过多年开发的。每一个都是为了特定的目的而构建的，并且每一个都针对其运行的环境进行了保护。在公共网络的开放环境中使用相同的 API 会带来不同的安全问题。

对一家公司来说，掌握全局的最好方法是首先了解自己在做什么。这可以通过使用工具或者发现复杂分布式应用程序中所有 API 的解决方案来实现。接下来，开发人员需要深入了解这些 API 是如何工作的。他们必须在开源应用程序开发生命周期的每个阶段定期进行 API 扫描。因此，一个合适的解决方案将有助于发现 API，知道它们何时改变，并提供对哪些服务相互通信的洞察。

**深入了解 API 交互**

多年来，开发人员和 IT 人员一直依赖监控作为处理应用程序性能和安全问题的常用方法。然而，对于现代商业环境来说，监控有时有点被动。

在采用监控时，DevOps 收集和分析数据以保持应用程序平稳运行。监控基本上是在问题发生后才发现问题。在典型的设置中，监控解决方案会在出现问题时发送警报。例如，当应用程序超过低性能阈值或存在可疑活动(如大量数据被共享)时。

相反，[可观察性](https://devops.com/fixing-risk-sharing-with-observability/)使用通过监控应用程序收集的日志、指标和分布式跟踪来帮助理解正在发生的事情。可观察性更侧重于理解问题发生的原因，而不是识别问题存在的监控。

可观察性解决方案通常使用复杂的分析，在某些情况下，使用人工智能对他们收集的数据进行推断。理想情况下，开源应用程序的可观察性解决方案将有助于解释系统内发生的事情，并提供可操作的见解，以维护或改善应用程序的性能或安全性。

帮助保护开源应用程序的另一个武器是可追溯性，这是可观察性的一个关键部分。具体来说，我们需要的是分布式跟踪，这是一种监控应用程序的方法。分布式跟踪让开发人员在处理应用程序元素时跟踪事务或请求的结果。

一个理想的解决方案应该能够实现分布式跟踪，收集 API 网关、负载平衡器、代理、Kubernetes 集群、服务网格、VM 和容器之间以及应用程序组件内部的所有应用程序通信。这些功能可以显著减少 IT 或开发人员在应用程序和系统中解决问题所花费的时间。

**以 API 为中心的开源应用安全方法**

在当今以 API 为中心的开源应用程序世界中，开发人员需要了解他们的应用程序的各种元素是如何协同工作的。从安全的角度来看，这意味着关注 API 本身。我们需要一个解决方案来帮助开发者发现所有的 API，包括影子 API。这些信息可以帮助开发人员了解变化和风险。理想情况下，开发人员将在他们的应用程序中使用这些 treats 知识来提高代码和 API 的安全性。
