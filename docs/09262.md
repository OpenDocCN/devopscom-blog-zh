# DevSecOps 工具:前方热气腾腾

> 原文：<https://devops.com/devsecops-tools-hot-air-ahead/>

我和你们大多数人一样，不喜欢硬核营销，这种营销把它的陈述建立在“嗯，从技术上来说是真的……”我也不喜欢谈论未来的人。当然，他们有时是对的；这是因为我们有如此多的人自以为是，随机分布意味着一些人将是正确的。我提出这一切是为了警告你，我(基本上是一个高科技发言人)将要谈论工具、工具市场的未来(特别是一个)以及我认为应该/将要发生的事情。

有时候，谈论某项技术的未来是为了让自己出名。我不玩那个游戏。我的目标就是简单地说，“这是我想看到的，也是我为什么想看到它。”

说够了——让我们来谈谈 [DevSecOps](https://devops.com/?s=DevSecOps) 工具。不久前，当我简要接触这个话题时，我记下了这个话题，在最近的一个项目中，我的想法得到了巩固。

让我们从简单的开始。如果您的 DevOps 工具链中没有使用我们目前称之为 DevSecOps 的工具，请安装它们。去吧，我们等着。这是一个显而易见的事情，我们会坐在这里，让您选择、购买和安装您的产品。说真的。“这是一个让开发更安全的工具*和*让开发人员更有效率。”现在就去安装。选择您想要使用的功能，并将其纳入您的工具链。

但是不要和你的供应商结婚太久。因为同样的原因，我刚才告诉你去安装它们，我会说，我不认为他们应该是一个独立的市场。这里有一个工具集，一端插入 IDE，另一端独立运行，与 CI/CD 集成。It *在很多方面都是* DevOps。尽管代码扫描无疑是一项安全功能，但 DevOps 的总体精神是“谁在乎？”使用这些工具会留下相当多的安全功能，还会警告不良的编码实践。注意，我并没有特别提到*不安全的*编码实践，尽管这些都包括在内。我说*糟糕的*编码方式。这些供应商中最大的一个会让你知道这样的事情，“嘿，你把那个变量发送给了这个函数，然后从来没有使用过它。你确定吗？”或者“这个循环看起来永远不会存在。你确定吗？”

它应该是，而且我预测会是，DevOps 过程的一部分。在我的梦想世界中，功能会被分割，一些会在 ide 中结束，而其余的会在测试工具或 CI 工具中结束。但当有知名厂商时，市场通常不会这样分割。即使你能感觉到功能上的差异(一堆插件直接插入 IDE 其余的没有)，它不会被分割，因为现有市场通常通过收购合并。所以我们必须看看事情会如何发展，但我相信 CI/CD 供应商或全系统安全供应商会进入这个市场。事实上，拥有 [SOC](https://securityboulevard.com/?s=SOC) 的公司可能会发现这些供应商很有吸引力，因为他们中的大多数也为系统的动态扫描部分运营 SOC。

还没有安装 DevSecOps？现在就去。选择供应商。

因为我强烈要求您安装这些工具集之一，所以我要明确声明:在撰写本文时，我与该领域的任何供应商都没有任何关系。我这样说是因为我使用这些工具，并且坚信它们在开发过程中作为一个层所带来的棘手问题在很大程度上被它们给应用程序带来的好处所弥补。

继续摇摆。这只会让你的系统更加稳定，合并(我认为这是不可避免的，但实际上只是一种可能性)出现时你可以担心。
