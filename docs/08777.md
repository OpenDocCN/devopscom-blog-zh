# 选择正确的软件测试指标

> 原文：<https://devops.com/choosing-the-right-software-testing-metrics/>

软件测试度量是跟踪质量保证工作有效性的标准。首先，在整个规划阶段建立成功指标。然后，在完成该过程后，将它们与接收到的度量进行比较。

然而，许多软件 QA 和测试专家倾向于关注如何执行测试，而不是测试产生的实际信息。我的意思是测试人员通常只关注完成所有测试的满足感。但是这总是一件好事吗？仪表板上的所有绿色指示灯都显示为 100%通过，但您的测试仍有可能不够强大。

这篇文章将讨论五个软件测试度量标准，它们可以帮助 QA 专业人员评估他们的成功。

## 一个“好的”测试标准的特征

让我们来讨论一下一个指标在理想情况下应该具有的特性。

### 与业务目标相关

关键 KPI 应反映企业的主要使命和目的；比如月收入增长或者新用户数量。每家公司都根据他们的产品想要达到的目标来选择衡量标准。虽然在所有测试中取得成功看起来很有吸引力，但专注于错误的目标可能具有欺骗性。这可能会影响应用程序的工作和整个复杂的系统，如无头商业架构。

### 允许增长

每个指标都应该允许改进。如果你达到了 100%的成功率呢？目标可能是将指标保持在这一水平或进一步提高。

### 鼓励制定战略

当一个指标给团队一个目标时，它也激励他们提出问题来制定计划。假设你需要增加收入。考虑产品是否需要新的特性来鼓励更多的购买。有必要创建新的获取渠道吗？竞争对手是否推出了吸引新买家的新产品或新功能？

### 可追踪和可理解

好的度量标准易于理解和遵循。否则，召集他们的人如何做出明智的决定？员工必须明白他们可以做些什么来改善结果。

## 选择和测量软件测试度量的三个技巧

### 1.从提问开始

你的问题应该涵盖三个主题:
1。您正在测量的内容
2。衡量它的策略和工具
3。跟踪它的原因

为了避免分析无用的指标，请注意指标定义过程。有时候，少量的 backlog bugs 意味着你的 QA 团队正在做它的工作。然而，当您将这些错误分解成高/中/低优先级问题时，您将能够更好地看到整体程序质量，并做出必要的调整。

### 2.计算 QA 指标时不要忽视自动化

自动化节省了您手动收集数据的时间，并有助于确保您的指标始终相关。让我们假设你用吉拉。如果你需要每个 sprint 的关键 bug 的数据，在你的 Confluence 页面上设置一个吉拉查询语言( [JQL](https://support.atlassian.com/jira-service-management-cloud/docs/use-advanced-search-with-jira-query-language-jql/) )请求。会经常更新。或者您可以使用基于您偏好的测试管理/任务跟踪系统的其他工具。

### 3.收集意见并逐步改进指标

一旦你建立并收集了所有的指标，反馈和改进过程就开始了。注意反馈，以提高您的指标和报告的效率和清晰度。

## 要跟踪的五个软件测试指标

现在我们来看一些具体的例子。请注意，不同的质量方面因环境不同而有不同程度的影响。

### 1.用户满意度

在这里，你会希望看到客户对产品的反应。您使用用户满意度调查和支持票来揭示错误。如果您跟踪这些质量指标并努力改进它们，业务将会增长，因为您会看到更多满意的客户和回头客。如果出了问题，你必须做一个因果问题分析并排除障碍。

### 2.过程度量

这些是对您的产品质量有重大影响的内部测量。例如，您可以跟踪前置时间以及设置任务与代码部署和生产之间所用的时间。

您可以使用的另一个指标是周期时间。这意味着在被批准开始工作之后，是时候构建一个特性了。最后，你可以追踪解决困难所花的时间。这可能是指一旦报告了故障单或故障，解决故障单或故障的速度。

由于这些度量标准很难测量，另一种提高流程效率的方法是检测队列中未完成的工作从哪里开始堆积。它可以突出瓶颈，如果消除了瓶颈，可以帮助您的团队变得更有生产力。

### 3.覆盖率指标

测试质量的另一个指标是测试覆盖率。它告诉我们被测试代码的数量。这是一种确保您的测试检查代码以及它们运行多少的方法。在这种情况下，最好使用自上而下的策略。第一步是分析模块覆盖率。然后考虑功能，最后是每个功能中的数据覆盖。它意味着你用测试覆盖的潜在数据输入有多少种不同的组合。

这组指标包括:
●需求覆盖百分比
●单元测试覆盖
●手工或探索性测试覆盖
●按需求类别划分的测试用例
● UI 测试覆盖
●集成和 API 测试覆盖

### 4.代码质量度量

评估代码质量意味着将代码的价值分为两类:好的和坏的。没有单一的质量概念，因为实际上每个开发人员都为自己定义了什么是好的代码。如何评估代码质量？像 SonarQube 这样的工具可以让你揭示一个系统中有多少技术债务。您需要对问题和漏洞进行分类，按优先级进行组织，并选择您要关注的内容。

### 5.Bug 或事件指标

每个问题的严重性不同，所以不要给所有问题同等的权重。有些问题仅仅是改进的建议。确定质量的哪些组成部分对你的公司来说比其他的更重要。也就是说，在分析您将使用的度量标准时，不要仅仅局限于缺陷的数量。

你能从事故报告中提取什么？这些结果可能包括:

bugs 总数
●未解决的缺陷
●已解决的缺陷
●关闭每个事件报告的时间
●自上次发布以来的变化

## 测量软件测试度量的规则

评估软件测试中的度量并估计它们的成功可能是令人沮丧和模糊的。以下是您可以使用的一些提示和建议:

1.将您的指标与项目、流程和产品目标相关联。请记住，单个指标不足以完整地反映软件质量。
2。跟踪整个过程中的进展(或退步)。通过自动化简化数据收集流程，将数据存储在 Wiki/Confluence 等协作资源中，并定期审查结果。
3。向客户和团队报告统计数据，展示您的进展。报告应该易于理解，所以要有帮助性和用户友好性。
4。检查以确保指标是否有效。跟踪不相关的指标并显示不准确的数据是不可能的。

度量是软件测试中的一项重要活动，例如确定成功测试的数量与失败测试的数量。你得到的所有信息都是利益相关者得到的。因此，他们可以做出明智的决定，如何时发布应用程序。

你如何监控你的测试活动？您需要确定相关的软件测试指标。选择正确的测试指标可能很困难。团队经常选择与整体业务不同步的度量标准。

缺乏足够的基准会导致什么？涉众未能衡量进展，识别开发的可能性，或者控制哪种测试策略具有最积极的影响。考虑到所有的事情，QA 团队需要跟踪个人的进展、技能水平和成功，以及代码质量、bug 和覆盖率。
