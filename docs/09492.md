# code 公证人增加了 SLSA 框架支持，以提高应用程序的安全性

> 原文:[https://devo PS . com/code 公证人-adds-slsa-framework-support-to-advance-app-security/](https://devops.com/codenotary-adds-slsa-framework-support-to-advance-app-security/)

code 公证人本周宣布，在其免费的公证和验证服务中，它已经集成了对软件工件供应链级别(SLSA)框架的支持，以确保代码的完整性。

Code 公证人首席执行官 Moshe Bar 表示，作为第一个获得 SLSA 合规性的应用安全平台，该公司正在使组织更容易保护其软件供应链。

SLSA 是 Google 定义的一个框架，用来给代码附加二进制授权。在其当前形式下，SLSA 是一组指导原则，确定了如何获得四个不同级别的应用程序安全性。长期目标是创建一个框架，该框架将自动生成可审计的元数据，这些元数据可以输入到策略引擎中以创建 SLSA 认证。

code 公证人提供了用于编目和验证软件开发生命周期中使用的组件的真实性的工具。该平台的核心是一个不可变的开源 immudb 数据库，它以加密的方式为每个软件工件附加一个身份。这种能力也可以用来动态生成软件材料清单(SBOM)。

code 公证人证明服务还可以与持续集成/持续交付(CI/CD)平台集成，DevOps 团队使用这些工件来构建和部署应用程序。Bar 指出，理想情况下，开发运维团队应该构建一个与现有开发运维管道并行的安全管道。

在最近备受瞩目的软件供应链违规事件之后，对软件构造方式的审查水平显著提高。至少在理论上，开发人员承担了更多的应用程序安全责任，这是整体转变的一部分，为他们提供了对 IT 环境更多的编程控制。然而，网络犯罪分子现在越来越擅长损害开发者的凭据，并在开发应用程序时插入恶意软件。

当然，DevSecOps 最佳实践正被用来教导开发人员如何更好地保护他们的应用程序环境。问题是没有足够多的开发人员学会如何足够快地保护应用程序。针对软件供应链的攻击数量继续以比组织仅仅通过教育开发人员所能阻止的速度更快的速度增长。

Bar 指出，现在很明显需要在 DevOps 工作流中嵌入更多的网络安全护栏。巴尔说，case 公证人不是试图以零敲碎打的方式实现这一目标，而是为一个集成平台做准备，该平台可以作为 DevOps 工作流的自然扩展来调用。

总的来说，Bar 说，现在对于软件供应链安全有了更大的紧迫感。可以说，对这些进程的许多审查早就应该进行了。不管怎样，将会有更多关于软件实际上是如何开发的问题。随着对软件的整体依赖不断增加，使用这种软件的组织开始意识到他们是多么的脆弱。不幸的是，现在唯一真正的问题是，在彻底解决这些软件安全问题之前，还会有多少漏洞。
