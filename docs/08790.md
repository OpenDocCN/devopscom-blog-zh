# 组织架构

> 原文：<https://devops.com/organized-architecture/>

我以前多次提到过钟摆，但今天我将使用它们来探索一个不同的角度:容器和微服务的时代。是时候问问自己，当前的整体 It 产品体系结构设计是否是您所能提供的最佳设计。

敏捷和整个“开发者为王”的运动给我们带来了很多独立的应用团队，他们各自为战。虽然这场运动的核心是短暂的，但我们从中获得了许多好的和坏的东西。好处包括打破了一些 IT 部门传统的“我们以极慢的速度改变”的过程——因为一个团队绝对想要让他们的工作变得更容易的工具，而不是那些由委员会预先批准的工具或给定组织使用的任何过程。坏的方面包括我们中那些已经看到这种事情发生的人之前被警告过的事情。突然间，我们有了关键的应用程序和基础设施，它们依赖于一个开发人员在他的爱好开发中使用的工具。有时候这很好，当然这发生在敏捷出现之前，但是现在我们遇到了雪崩。没有通过安检的东西；通常团队之外的人(或个人)不知道的代码/库/应用程序正在组织中运行。有些事情仍在发生。

然后，我们真正了解了[微服务](https://containerjournal.com/?s=microservices)。这驱使我们用非传统的方式打破东西。这个过程带来了很多好处，但也带来了很多复杂性。随着 API 调用和操作的激增以及部署的流程和路由的激增，复杂性在应用程序间共享。也许这两者结合起来最糟糕的事情就是“谁在乎它为什么失败？再转一个！”那场运动基本上已经死了(谢天谢地)，但它之所以死了，只是因为它引起了许多令人瞩目的“哎呀”问题。它让攻击者不被注意；它导致付费用户在网络上传播安全细节；它导致用户遭受崩溃，这在任何正常的开发环境中都是可以修复的…其中一些仍然存在。有一个脚本每 X 小时重启一个应用或服务，因为它随机崩溃。不可接受。尽你所能找借口，但如果你不知道是什么导致了崩溃，你就不该让应用或服务继续运行。

我们引入了大量的复杂性。现在，我们拥有两倍的[基础架构](https://devops.com/?s=infrastructure)—一个物理基础架构，一个虚拟基础架构(容器或云)。我们的应用程序分布在更多的系统上(无论是实例还是物理系统),而且通常是跨平台的；甚至跨数据中心。我们生活在一个非常酷的时代，一些美好的事情正在发生。但是复杂性是我们需要解决的一个非常现实的问题。一个应用程序/应用程序组合可能具有有限的复杂性，但我们仍然允许团队在定义应用程序所处的环境方面有太多的余地。我们需要提供一个快乐的媒介——一个有选择的世界，但这些选择是明确的、可管理的和安全的。

因此，我们可能会再次看到企业架构团队重要性的复苏，他们的重点是组织我们制造的混乱。事实上，我知道在我接触的几家财富 500 强公司中，这种情况正在发生。我知道有一个组织没有这样做，他们通过一个容器为多个应用程序运行所有的 UI 功能。这并不比各自使用完全不同的 UI 工具集更好的架构。因此，我预计他们也会想要协调的解决方案。这样类似的事情就不会再发生了。

你们都在做，日复一日。正如我常说的，不要让这一点拖累你。直面它，用“是的，在它好的地方；不，这不是你的信条。安全性是一个大问题，但可维护性是另一个问题。即使有 Ansible 这样的工具可供使用，管理 50 个不同的环境对 ops 来说也有点多，如果用 5 个就可以做到的话。继续摇摆。通过疫情，满足日益增长的需求，给予用户越来越多。再次感谢所有不知道感谢你的用户。
