# DevOps 死了！DevOps 万岁！

> 原文:[https://devops.com/devops-dead-long-live-devops/](https://devops.com/devops-dead-long-live-devops/)

有些人在寻找一个真正的德沃普斯。他们被误导了。我在这里说:放弃吧。无论你在旅程的终点发现什么都不是。那个真正的德沃普斯已经死了。死了，埋了。搜索毫无意义，或者更糟:搜索完全没有抓住重点。

DevOps 最初是一种生活哲学:关于包容而不是排斥，提出而不是责备，增加弹性而不是指责，变得更聪明和更棒而不是定义过程和建造掩体。无论如何，这也是故意从来没有严格的定义。

2010 年，似乎主要是关于[应用发布自动化(ARA)](https://en.wikipedia.org/wiki/Application_release_automation) 、监控、配置管理以及许多关于文化和团队的初步讨论。到 2015 年，增加了很多 CI/CD、容器和 API。日期是粗略的，但我的观点是:这些都还在，但现在 DevOps 今天的讨论*也*包括服务设计和微服务。哦，还有新的闪亮的“无服务器”这是关于它的一切，因为它是自然适应的。

这种新兴的、包容性的内容是可能的，因为人们总是在谈论实践，分享如何以最佳方式做事。Infra 作为代码，“牛，不是宠物”，webscale，版本和自动化所有的东西，马不只是独角兽，等等。大量的对话、见解分享、演示。

美好时光。

2017 年全球自然发生的 [DevOps 日](http://www.devopsdays.org)比以往任何时候都多。注意，我写的是“自然”——devo PS 是**一直是**一个有机的运动，一个对增加的应用发布压力的群体(以及，哇，全球)反应，以及选择团队合作而不是地盘之争的愿望。

所以我们从 DevOpsDays 开始，它遍地开花。然后我们有[凤凰计划](https://www.amazon.com/Phoenix-Project-DevOps-Helping-Business/dp/0988262592)，DevOps 企业峰会(现在在[美国](http://events.itrevolution.com/us/)和[欧盟](http://events.itrevolution.com/eu/)格式！)，以及更多伟大的书籍，包括"[基础设施代码](http://shop.oreilly.com/product/0636920039297.do)、" "[系统可靠性工程](https://play.google.com/store/books/details?id=tYrPCwAAQBAJ&source=productsearch&utm_source=HA_Desktop_US&utm_medium=SEM&utm_campaign=PLA&pcampaignid=MKTAD0930BO1&gclid=CMjvs8fKvdICFdCiNwodpscPeQ&gclsrc=ds)"以及最后的"[devo PS 手册](https://www.amazon.com/DevOps-Handbook-World-Class-Reliability-Organizations/dp/1942788002)"这些都是可靠的、信息丰富的东西，我相信作为一个社区，我们会因此变得更聪明、更快乐、更有效。

甚至那些唱反调的分析师——你知道你是谁——也开始接受 DevOps 的优点。

当我们不断适应管理突发属性，在我们的 IT 和业务中产生系统级反馈循环时，DevOps 就存在了。做更好、更快、更高质量的事情来取悦顾客。

任何关于 DevOps 的讨论实际上都是关于持续的 IT 创新适应，因为 IT 变化的速度也在变化。越来越快。所有这一切都意味着从蛤蜊这样的根开始，从容器开始，拥抱快速交付更好的客户体验的旅程。

因此，DevOps 的概念在不断发展，正如我们对需求的理解也在不断发展。

克里斯托弗·利特尔
