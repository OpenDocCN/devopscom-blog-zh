# 迎接新的(ish)管道

> 原文：<https://devops.com/meet-the-newish-pipeline/>

我的公司几乎一直在运行一个样本项目。我们目前的项目(使用 Java 和 Spring 的多目标 web/Android 应用程序，由 Gradle 和 Ant 构建，我们在这里称之为“Finder ”)有点过时了，所以我们在过去的几个月里一直在寻找新的项目。从非工作来源，我们想到了用 C#作为一个可行的辅助项目的多目标 Godot 的想法。转移到一套完全不同的工具是关键；它让我们伸展，这是我们正在进行的项目的想法。这些项目的既定目标是:

1.  扩展关于不同语言、开发工具和平台的知识。
2.  继续做你们正在做的事情。太多的专家已经“不再”从事实际工作，这一点也很明显。
3.  尝试新的想法/架构。
4.  避免“厨房水槽”我们不可能有一个包含所有可用技术的当前测试应用程序，所以要保持精简。把它当作我们用于测试的生产应用，而不是测试应用。

这就是为什么我们希望替换我们的 Java/Spring 项目。我们已经关闭了管道，我们已经调整了应用程序，直到我们对核心感到满意，它没有改变了很长一段时间。但改变是 DevOps 的一部分，所以我们要走了。

# 明显的事实

让我们马上关注一下工具链中的明显元素:

1.  饭桶。我们将使用 GitHub，但是某种形式的 Git 将保存我们的代码是一个假设。
2.  Visual Studio 代码。我们在做 C#，所以这将是代码编辑最有用的选择。内置的 Godot 编辑器很好，会被使用，但是 VS-Code 提供了 C#功能插件。
3.  Antsle 是首选的部署工具，尽管这一点在过去几年中已经发生了变化。我们将使用它来为客户端设置安装集。
4.  詹金斯或者哈德森。我们将只使用这个*来进行测试管理。*

# 不太明显的

我们想要(或必须)满足几个约束条件。首先，我们希望避免使用 Jenkins 或 ant 来控制整个构建过程，主要是因为我们在以前的项目中已经使用了这两种方法，并且大多数 DevOps 团队不再为新项目使用这些方法。其次，这是我们保留的样本，而不是投资，所以我们希望尽可能使用开源技术，以降低费用。第三，我们使用游戏引擎进行应用程序开发。这有一些影响——像大多数游戏引擎一样，它在运行时受到控制，我们需要在最终应用程序中捆绑引擎。

5.  戈多发动机。Godot 是作为游戏平台编写的，但它的图形/跨平台能力使它对像我们这样的图形密集型应用程序很有用。但是我们必须在管道中包含引擎及其配置文件。
6.  代码扫描。静态分析是 Finder 的新增功能，它是用 Java 编写的。这个项目是在 C#和 Godot 脚本。我们还没有找到支持 Godot 脚本的工具，但是 C#我们可以得到；只是增加了限制。所以我们将使用静态分析工具，但我们仍在决定使用哪一个。
7.  构建自动化。在过去的不同时间，我们已经用 Jenkins，Gradle，Maven 和 Ant 来处理这一步，但一些更新的东西将是一个好主意。我们仍在评估这一步。我们最终可能会回到这些工具中的一个，因为它们今天仍在使用。Godot 对我们来说是一个新元素，所以我们也在评估需要什么来挂钩它，并在一个可重复的过程中制作功能应用程序。

# 推动创新

我们玩得很开心——前几次冲刺让我们运行了一个工作原型，所以我们对此很高兴。在应用程序和管道中还有很多要实现，所以我们无疑很快会有更多要说的。我们正在讨论放入一个服务器组件——不是因为最初的概念需要它，而是因为我们的输出应用程序——Finder——有服务器端和客户端，我们对诸如 Postman 之类的东西的需求离开了这个架构。然而，API 安全、测试等等。是 DevOps 架构的重要组成部分。如你所见，我们仍在迭代。

至于 Finder，我们通常会在新的游乐场项目启动时取消之前的项目，但 Finder 花费我们很少。有五个容器(三个永久的应用程序托管；两个用于构建/测试)和一些处理能力，所以我们可能会保留它，因为它使用了一组不同的管道工具(和语言——主要是 Java/JavaScript)。

同时，你们继续摇摆。我们正在开发一款(其他)应用，除了我们之外，没有人会看到它——但你让企业保持运转。向你致敬！
