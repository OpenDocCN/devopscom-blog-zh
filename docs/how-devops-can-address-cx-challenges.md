# DevOps 如何应对 CX 挑战

> 原文：<https://devops.com/how-devops-can-address-cx-challenges/>

交付无缝的客户体验(CX)已经成为正在进行数字化转型计划的公司的首要任务，因为 85%的购买者愿意为出色的客户体验支付更多费用。然而，客户不仅愿意为正面的体验支付更多的钱，如果他们的体验不正面，他们还会去别的地方。事实上， [33%的客户](https://www.pwc.com/us/en/advisory-services/publications/consumer-intelligence-series/pwc-consumer-intelligence-series-customer-experience.pdf)会在一次糟糕的经历后离开他们之前喜欢的品牌， [92%的客户](https://www.pwc.com/us/en/advisory-services/publications/consumer-intelligence-series/pwc-consumer-intelligence-series-customer-experience.pdf)会在两三次负面经历后彻底放弃一家公司。因此，对于那些希望留住现有客户，同时从竞争对手那里赢得新客户的公司来说，实现积极的 CX 已经成为首要任务。

为了确保为客户服务的技术产品和服务的 [CX](https://devops.com/?s=CX) 质量，既定的 DevOps 流程和策略应适用于客户体验软件的独特方面。面向 CX 的 DevOps 支持在客户体验至关重要且质量至关重要的情况下交付软件。然而，在为 CX 部署 DevOps 时，客户服务组织必须克服六个主要缺陷。

## CX 的 DevOps 是什么？

为了理解这些挑战，我们必须首先理解 DevOps 对于 CX 的目的。

面向 CX 的开发运维将开发运维软件方法重新聚焦于 CX 软件的独特方面，以实现更高的效率和质量。通过新的软件开发和功能不断提供扩展的 CX 功能，已成为正在进行数字化转型的组织的优先事项。但是对于 CX 和客户服务的软件产品来说，不能为了快速发展而牺牲质量。与此同时，公司仍然必须满足快速创新的需求和软件交付的紧迫期限。

客户的期望比以往任何时候都高。如今，顾客体验已经没有第二次机会了。社交媒体让客户能够向全世界传播 CX 的失败。由于 CX 事故和负面客户互动的可见性，一个品牌在 CX 的声誉取决于客户第一印象的完美执行。

顾客永远是对的；因此，在软件开发和测试中，客户的观点必须是首要考虑的因素。这就对开发运维解决方案和用户体验(UX)设计流程以及 CX 测试和监控提出了很高的要求。软件必须实时设计、开发、持续测试和监控。公司需要了解客户可能使用软件来实现其目标的所有不同方式，并在客户体验之前快速识别和修复任何潜在的问题。

## CX devo PS 的 6 个潜在陷阱

### 高管赞助投资不足与文化进化

CX 转型计划要想取得成功，就必须有一位能够支持推动文化变革和战略投资需求的执行领导者来授权。执行领导是必要的，因为 CX 计划通常跨越多个部门:例如，运营，开发，营销，销售和服务。

承担这一职责的领导者是现有的高管还是全新的职位，将取决于公司的组织结构。一些公司将这一职责分配给首席运营官、首席信息官或 CMO，而其他公司则创建了一个全新的首席 CX 官角色，以便领导者可以将全部注意力放在手头的 CX 转型计划上。不管 CX 冠军头衔的开发人员是谁，他们都必须在预算、人员和流程方面拥有广泛的权力，以使他们的角色有效。

### 竞争和不一致的激励

DevOps 是一个多团队的努力。确保团队的目标和激励一致是确保 DevOps 项目和谐执行的关键。每个团队都应该有适合其团队功能的目标，但不反对其他团队的目标，例如开发方面的速度或低缺陷率，以及操作方面的高可靠性。这些职能团队的目标应该围绕各自的客户满意度业务目标。

### 关键绩效指标(KPI)不明确

采用 DevOps 迫使组织采取更加协作的心态。由于 DevOps for CX 计划的多团队协作性质，关键绩效指标在团队间得到清晰传达、广泛理解和监控至关重要。KPI 还应该广泛分布，以便可以在操作层面查看 CX。

### 缺乏 DevSecOps 优先级

许多组织未能从开发运维的不同方面认识到开发运维的重要性。不仅客户数据必须得到保护，全球运营的大型组织也必须确保遵守地区数据隐私法规。不遵守法规要求的组织面临罚款和其他惩罚性挫折的风险。

您必须能够在生产中测试安全特性，以保护客户和您的组织本身。不幸的是，安全性常常是 DevOps 最后要讨论的方面之一。

### CX 交付环境中的不同配置

开发、质量保证、试运行和生产之间的转换是 DevOps 团队感到沮丧的主要原因，因为环境通常配置不同。随着代码在软件生命周期和不同的环境中移动，通过部署自动化对解决方案和环境采用一致的方法将防止浪费时间来修复代码移动时突然出现的错误。

### 手动或过时流程引入的错误

手动流程引入了人为错误的机会，导致延误和缺陷，从而导致计划外的工作。手动执行软件测试是一个缓慢的过程，不仅会导致质量问题和交付延迟，还会减少开发人员执行任何计划外的工作或必须在测试后完成的返工的时间。CX 系统测试的自动化提高了质量和可靠性，加快了速度，为开发人员提供了更多的后测试时间，以确保产品符合质量要求。

面向 CX 的 DevOps 通过交付高质量的软件和保持卓越的客户互动，使公司能够快速创新 CX。随着 CX 领导人通过软件创新新产品和服务，DevOps for CX 实践将在未来几年继续得到完善。同时，遵循此处概述的最佳实践来避免 CX devo PS 的六个主要陷阱将确保客户体验保持在较高水平。
