# 变化的速度需要更好的测试

> 原文:[https://devo PS . com/the-pace-of-change-requires-better-testing/](https://devops.com/the-pace-of-change-requires-better-testing/)

2020 年的新冠肺炎疫情创造了一种迫切的需求，即通过移动设备为人们提供更多的工作和处理日常生活任务的方式，并迫使应用创新以前所未有的速度增长。得益于新的应用程序开发技术和[敏捷流程](https://devops.com/?s=agile)以及开源和嵌入式库，组织正在为多种设备(比如 web、移动设备、平板电脑等)更快地交付更强大的功能。).虽然源源不断的新应用和更新对消费者来说非常好，但事实证明，这是许多企业的一个痛点，他们的测试流程没有跟上速度，无论是字面上还是象征性的。

尽管测试自动化是敏捷开发的重要组成部分，但在已发布的敏捷方法中，如 [Scrum](https://scrum.org) 和看板，并没有涉及到它。当然，测试工具和方法已经随着现代开发实践而改变，但是它们并不总是跟得上今天 AppDev 快速发展的步伐，过去的 18 个月只是提升了对改进测试的需求。对于公司来说，要跟上并超越他们的竞争对手，测试必须与开发同时进行。

## 自动化和连续测试:类似，而非同义

有许多方法来测试软件，包括手动、用户界面(UI)、自动化和连续测试(CT)，但是今天大多数企业采用某种形式的自动化测试，并且越来越多地采用连续测试。人们经常把自动化和连续这两个术语当作同义词来使用，但事实并非如此。因此，在我们深入研究其中一个的优点之前，清楚地理解每个术语的含义是很重要的。

自动化测试指的是以编程方式进行软件测试，很少或没有人工干预。它测量与预测结果相关的实际结果，并且，由于它比手工测试更省时，已经成为现代企业的标准。同时，持续测试使用自动化测试在软件开发生命周期的每个阶段提供不间断的反馈，因此，在 DevOps 中起着关键的作用。

自动化和持续的测试携手工作，以确保软件质量、节约成本和加快上市时间，并且应该与开发同步进行。虽然可以缺一不可，但这并不明智。

## 持续测试的好处

持续测试将自动化带到了下一个层次，因此，它比单独的自动化测试有几个优点。

*   在每个开发阶段的早期和可操作的反馈意味着问题可以更快地得到解决，以免演变成更大、更复杂的问题。
*   敏捷团队的速度是通过他们在一次冲刺中能完成多少来衡量的。正因为如此，团队的测试焦点被缩小到他们所参与的 sprint，这意味着影响先前部署的功能的问题通常直到开发生命周期的后期才被发现，或者在部署之后才被用户发现。
*   自动化和持续的方法意味着在部署之前识别和解决更多的问题。
*   随着组织能够更快、实时地响应市场变化，发布周期也加快了。
*   尽早纠正问题可以减少返工，从而节约成本和时间。
*   持续集成需要持续测试。
*   CT 不仅测试应用程序中发生变化的部分，还测试这些更新在应用程序的整个功能中是如何工作的。
*   与自动化测试不同，CT 不需要人工干预。

## 为什么耽搁了？

尽管有许多好处，CT 并不像人们希望的那样广泛。脱节的原因之一归结于投资，无论是雇佣合适的人、培训还是工具。

也许最大的绊脚石出现在招聘阶段。CT 需要一套特定的技术技能，加上测试开发的天赋和专注于弹性的心态。除了投资员工，还有前期(和持续)自动化成本需要考虑。不幸的是，这并不是说您可以实施 CT，然后再也不用碰它—CT 成为了它自己的软件交付管道，必须对其进行开发、维护和增强，以确保它不会过时。

## 持续测试:什么会出错？

随着 AppDev 步伐的加快，将它集成到 DevOps 中的要求也越来越高。如果做不到这一点，将会产生严重的后果，其中最明显的是，如果没有 CT，公司将无法尽可能多、尽可能频繁地进行测试。例如，回归测试做得不够深入，所以当一个新的应用程序或更新发布时，更有可能会出现问题。此外，没有正确的技能组合，团队倾向于检查任何新的东西，而看不到它如何融入整体。此外，如果没有适当的测试，您可能会无意中引入错误，或者发现功能没有按计划工作时为时已晚。依赖手动的、缓慢的、脆弱的 UI 测试只会阻碍不适应的组织的进步。

负面后果超出了敏捷团队或企业的范围，因为糟糕的软件设计和实现会影响到客户满意度这一至关重要的问题。众所周知，不满意的客户不是忠诚的客户，这意味着更低的底线。另一个同样严重(可能更严重，取决于您的观点)的后果是对数据安全的潜在影响，使组织面临非常不同但重大的风险。

多亏了 DevOps，想法正以闪电般的速度从部署走向商业价值。显而易见，测试也必须比以往更快。

当你考虑到今天你所看到的是你将要经历的最少的变化，并且明天肯定会带来更多的变化时，就应该非常清楚为什么持续测试如此重要:你做的测试越少，下游会出现越多的问题。

企业开发软件是有原因的——获得商业价值。如果你的应用不工作，你就没有收获它的全部价值。因此，不要问你自己是否应该实现持续测试，你需要问你自己为什么还没有这样做。现在不投资 CT，你以后肯定会为此付出代价。
