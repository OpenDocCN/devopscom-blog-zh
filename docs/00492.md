# 关于 DevOps 的五本书

> 原文:[https://devops.com/five-great-books-on-devops/](https://devops.com/five-great-books-on-devops/)

虽然 DevOps 运动于 2009 年正式开始，但关于 DevOps 的含义、如何实现以及据称 DevOps 的好处是什么，仍有许多问题。很大的问题，当然。没有快速的答案。事实上，要彻底回答每个问题需要大量的书籍。

这给我们带来了一个问题，什么是定义 DevOps，描述如何在现实世界中操作它，并显示人们可以预期的有益结果的最佳书籍？

以下是我选择的五本关于 DevOps 的巨著，排名不分先后:

**本书:** *[凤凰计划](https://www.amazon.com/Phoenix-Project-DevOps-Helping-Business-ebook/dp/B00AZRBLHO/ref=sr_1_1?s=books&ie=UTF8&qid=1423609719&sr=1-1&keywords=the+phonex+project)*

作者:吉恩·金，凯文·贝尔，乔治·斯帕福德

这是一个虚构的故事，以引人入胜的小说风格写成，讲述了如何应用 DevOps 来释放其潜力的故事。这个故事是关于虚拟公司 Parts Unlimited 的 IT 经理 Bill 的。这不是一个令人兴奋的企业名称，但冒险真正开始时，在开车上班的路上，比尔接到首席执行官的电话，故事开始了。

这本书的封面描述最能说明问题:

该公司代号为凤凰计划的新 IT 计划对 Parts Unlimited 的未来至关重要，但该项目严重超出预算，而且进展非常缓慢。首席执行官希望比尔直接向他汇报，并在 90 天内解决这个烂摊子，否则比尔的整个部门将被外包。

在一位未来的董事会成员和他神秘的三种方式哲学的帮助下，Bill 开始发现 IT 工作和制造工厂的工作比他想象的更相似。随着时间的推移，Bill 必须组织工作流程，简化部门间的沟通，并有效地为 Parts Unlimited 的其他业务职能部门提供服务。

DevOps 运动的三位杰出人物以一种快节奏、有趣的方式讲述了一个故事，任何在其中工作的人都会认识这个故事。读者不仅会学到如何改进他们自己的 IT 组织，他们再也不会以同样的方式看待它。

**正如一位亚马逊评论者所说:**“这个关于如何应用 DevOps 来摆脱延迟和缺陷的 it 死亡螺旋的“寓言”是一个令人愉快的阅读，有很好的参考和支持材料来理解 DevOps 背后的原则。这还不足以让你一路走到那一步，但它会激励你沿着这条路走下去。”

**书:** *[连续交付:通过构建、测试和部署自动化进行可靠的软件发布](https://www.amazon.com/Continuous-Delivery-Deployment-Automation-Addison-Wesley/dp/0321601912/ref=cm_cr_pr_product_top?ie=UTF8)*

作者:杰斯·亨布尔和大卫·法利

这本开创性的读物是关于如何快速地将新的高功能软件交付给你的用户。封底:“通过构建、部署和测试过程的自动化，以及开发人员、测试人员和操作人员之间改进的协作，交付团队可以在几个小时内——有时甚至是几分钟内——发布变更，无论项目的大小或代码库的复杂性如何。”

该书涵盖了自动化基础设施管理和数据迁移，虚拟化的使用，自动化软件的创建、集成、测试和部署，以及团队内部协作的改进。

**正如一位亚马逊评论者所说，**“这是任何软件或应用开发人员的必读之作。令人惊讶的是，作者很好地抓住了真正的难点，并为您提供了更有效的 It 实践所需的所有信息。”

**书:** *[【可见作战手册】](https://www.amazon.com/Visible-Ops-Handbook-Implementing-Practical-ebook/dp/B002BWQBEE/ref=sr_1_1?s=books&ie=UTF8&qid=1423613789&sr=1-1&keywords=the+visible+ops+handbook)*

作者: Kevin Behr，Gene Kim 和 George Spafford，IT 流程研究所

这是我读过的第一本关于 IT 过程改进的书，它让我大开眼界。这本书在 DevOps 这个术语被创造出来的 4 年前出版，但是它抓住了高性能的本质。这是今天的必读书目:

Visible Ops 整理了这些[高性能企业]组织如何实现从优秀到卓越的转型，展示了感兴趣的组织如何通过四个步骤复制这些高绩效组织的关键流程:

1.  稳定患者，修改第一反应–几乎 80%的停机都是自己造成的。第一步是通过处理如何管理变更和如何解决问题来控制有风险的变更并减少 MTTR。

2.  捕捉并释放，找到脆弱的工件–通常，存在不可重复复制的基础架构。在这一步中，我们清点资产、配置和服务，以确定变更成功率最低、MTTR 最高和业务停机成本最高的资产、配置和服务。

3.  建立可重复的构建库——投资的最高回报是实现有效的发布管理过程。这一步为最关键的资产和服务创建可重复的构建，使其“重建比修复更便宜”

4.  实现持续改进–前面的步骤逐步在发布、控制和解决流程之间建立了一个闭环。这一步实现了度量标准，以允许所有这些过程领域的持续改进，从而最好地确保满足业务目标。

正如这位亚马逊书评人今年秋天对这本书所说的:“这是我的团队的第一本也是最重要的必读书籍！如果你是一名 IT 专业人士，你必须阅读这篇文章。这是如何重振您的运营并成为顶级 It 组织的简明而有力的表述。”

**书:***[devo PS for Developers](http://www.apress.com/9781430245698)*

**作者:**迈克尔·胡特曼

这本书是为软件开发人员和工程师准备的，旨在通过使用精益实践将应用程序和软件投入生产来帮助改进软件交付。

从其描述来看:

面向开发人员的 DevOps】是面向有积极性的软件工程师，特别是程序员、测试人员、QA、系统管理员、数据库管理员，包括初学者和专家，他们希望改进他们的软件交付过程。对于那些想通过集成他们的软件开发和交付方法来更进一步的工程师来说，这是一个完美的选择。这本书是为那些想要塑造他们的过程，决定和集成开源工具，并寻求如何在高级真实世界用例中集成标准工具的指导的工程师而写的。

正如一位亚马逊评论者所说:“开发者倾向于在短周期内进行更新，而管理员更喜欢稳定的系统。如果发生了不好的事情，他们通常会指责对方。一个解决方案是拆除开发和运营的筒仓。换句话说，建立整合的团队。

这是 Michael Hüttermann 想要强调的核心信息。敏捷方法和工具可以用于开发，也可以用于部署和操作。在他的书中，Michael 解释了如何做以及使用哪些(免费)工具。"

平心而论，这本书认为丁斯不是一本编程书，并且重复了它的一些概念。但本书的价值在于它的真实例子，以及对开发和运营团队中不同角色如何更好地合作的解释。它还深入探讨了组织在实施 DevOps 的过程中遇到的常见陷阱。

**本书:** *[目标:一个持续改进的过程](https://www.amazon.com/Goal-Process-Ongoing-Improvement-ebook/dp/B002LHRM2O)*

作者:艾利·M·高德拉特，杰夫·考克斯作者，大卫·惠特福德(其他撰稿人)

《目标》是这个列表中的第二部虚构的书。写于 30 多年前，可能是昨天写的。当讨论概念和系统时，很多时候故事是一种方式，因为不同角色的动机和思维过程可以被深入而优雅地讨论。《目标》在这一点上取得了成功，对于任何寻求帮助成为组织改进催化剂的人来说，这是一本必读的书。

从书上描述:

Eli Goldratt 的信念之一是个人或组织的目标不应该用绝对的术语来定义。一个好的目标定义是让我们走上持续改进的道路。

追求这样一个目标需要不止一个突破。事实上它需要很多。为了能够识别这些突破，我们应该对我们环境的潜在规则有深刻的理解。在写完《目标》25 年后，戈德拉特博士站在巨人的肩膀上写道。在这篇文章中，他提供了基本的操作规则。

这篇文章出现在这本书的末尾。

以下是一位亚马逊评论者所说的:“我经常听到自己抱怨公司把最简单的活动复杂化了。度量并不真正衡量最终结果，而且常常受到高管的高度重视。这些测量导致进一步的损失，并掩盖了真正的问题。这本书很吸引人，从某种意义上说非常“简单”,这使得它更加吸引人。我学到了很多关于我的“部门”的生产方面的知识，这个部门在字面上通常不被视为生产部门。发人深省和鼓舞人心，每个领导人都可以从讲述的故事中受益。必读。”

我很想听听你的意见，你读过的关于 DevOps 的最重要的书是什么？
