# 为什么 MTTR 是 DevOps 团队的重要指标

> 原文:[https://devo PS . com/why-MTTR-is-a-vital-metric-for-devo PS-teams/](https://devops.com/why-mttr-is-a-vital-metric-for-devops-teams/)

在 DevOps，分秒必争。系统故障意味着您的工程师失去了改进软件和开发新功能的宝贵时间。那么，如何有效地检测和管理这些问题呢？ 平均解决时间(MTTR)可以让您深入了解[开发团队](https://devops.com/your-devops-teams-can-do-better-heres-how/)对事件的响应效率以及软件的可靠性。如果没有数据来分析您的事件管理工作流，就很难知道您的团队如何改进或确定性能基准。在本文中，您将了解这一有价值的指标，解释它并利用这一信息来最大限度地减少停机时间。

### 为什么 MTTR 如此重要

无论你是一家正在努力实现现代化的传统企业，还是一家希望获得优势的初创企业，或者介于两者之间，MTTR 都是你所在企业的一个重要指标。它可以表明您的流程是否标准化，并可以突出优化的关键领域。最重要的是，mmean-time-to-resolve提供了对你的软件有多稳定以及你的团队随着时间的推移有多灵敏的洞察。

### 平均解决时间以及如何解释它

MTTR 可以代表许多不同的东西:平均修复时间、平均恢复时间、平均恢复时间或平均解决时间只是这个通用缩写词的几种解释。这些指标中的每一个都从不同的方面考察了事件处理。例如，平均修复时间是工程师解决一个问题所花费的平均时间，从他们开始工作到将一个变化推向生产的时刻开始计算。但是这一指标并没有考虑到从收到警报到工作开始之间的时间。

平均解决时间是这些指标中最高级别的，它考虑了从收到警报到事件得到解决的平均时间，包括事后分析和实施流程更改以避免将来出现同样的问题。虽然该指标的标准会因行业而异，但表现最佳的开发运维团队会看到一天的 MTTR 低于。

由于这是一个如此全面的指标，高平均解决时间测量值可能表明警报存在问题，或者您的工程师在维修上花费了大量时间。因此，有必要长期观察 MTTR，并分析您的事件管理工作流程的每个组成部分:提醒工程师、诊断问题、测试修复、交付生产、进行审查以及从事件中吸取教训的时间。

结合其他指标 检查平均解决时间也可能有所帮助。要确定您的 DevOps 团队是否面临生产挑战，请评估您的[变更失败率](https://cloud.google.com/blog/products/devops-sre/using-the-four-keys-to-measure-your-devops-performance) (CFR ),看看有多少版本导致服务降级。其他 [DORA 指标](https://devops.com/how-dora-metrics-can-measure-and-improve-performance/) ，如部署频率和交付周期，是平均解决时间的完美伴侣。

为了确定软件的可靠性，除了平均故障间隔时间(MTBF)之外，您还可以查看平均解决时间，MTBF 计算的是两次事故之间的平均时间。如果你经常更新你的软件，比较平均解决时间和平均无故障时间(MTTF ),它测量程序需要重新设计功能之前的持续时间。要深入了解您的警报流程，请检查平均检测时间(MTTD ),它评估您的团队识别问题存在所需的时间。

### 如何改善平均解决时间

警报是响应事件的第一阶段，在努力缩短平均解决时间时，应该是首先关注的领域之一。确保警报是可操作的，并且 DevOps 团队成员拥有立即响应所需的工具。直截了当的升级过程是必不可少的:定义每个成员的职责，并在彼此的角色中培训团队，这样，如果有人不在，该过程永远不会停止。

预先监控可以帮助您在问题出现之前就采取措施，通过主动检查潜在事件，您可以避免意外停机。

改善 MTTR 的最好方法是用操作手册来规范你的操作程序。没有操作手册，DevOps 团队必须在没有明确方向的情况下做出响应，并花时间相互发送信息，他们无法立即采取行动。但是，使用 runbooks，您组织的知识库是集中的，所有团队成员都可以访问，使他们能够在问题出现时立即做出响应。

如果你已经在使用 runbooks，考虑自动回复。自动化不仅改善了您的平均解决时间，还会让您的 DevOps 团队有更多的时间来实施长期的变更，从而提高您服务的稳定性。

正确实施时，平均解决时间是一个经过验证的指标，可以表明您的 DevOps 团队应对挑战的效率以及您的软件的可靠性。但是要小心，不要在追求最小化这个指标的过程中鼓励错误的行为。例如，您可以通过取消所有警报协议来快速减少 MTTR，但这也意味着为您的最终用户提供较差的服务，并可能导致团队士气低落。随着时间的推移检查这一指标，并以缓慢和稳定的增长为目标，而不是快速修复。
