# 三个常见的测试错误和解决方案

> 原文：<https://devops.com/three-common-testing-mistakes-and-solutions/>

几周前，在 SauceCon 上，DevOps.com 与 SauceLabs 的首席客户官马特·怀曼进行了交谈。今年的主题是重新想象测试，强调了在测试中优先考虑用户体验的必要性。正如 Wyman 所说，“我们真正关注的是客户体验，在正确的时间使用正确类型的测试工具。”会议结束后，我们找到他，更多地讨论了一个关键点——许多团队不知道如何测试用户体验，因为这不是他们的专长。

Wyman 就三个典型的测试错误以及开发人员如何提高测试环境的准确性和广度提供了建议。

### 1.臃肿的测试套件

这里有一个常见的场景:脆弱的测试被放入大型测试套件中，开发人员忽略它们，而不是纠正或删除它们。随着人们添加越来越多的测试，套件变得越来越臃肿，这可能会模糊他们的结果。怀曼说:“从外部来看，当你做出改变以通过考试时，很难判断哪些考试是最重要的。”

他解释说，如果两个人在代码库的同一个部分工作，A 建立了一系列集中的测试，B 可以做出改变，并确保当这些测试通过时它会工作。但是当测试套件中塞满了不相关的测试时，情况就不同了。

“相反，如果我创建一个两小时的套件，里面有数千个测试，你不会知道哪一个最适合你正在做的改变，”Wyman 指出。“当人们对构建一个好的测试套件或测试实践知之甚少时，这是更重要的复杂因素之一。”

### 2.使用像锤子这样的单一工具

Wyman 强调了在正确的时间为正确的目的使用正确的测试工具的必要性——这是许多开发人员没有做到的。他说，在这种常见的错误场景中，开发人员采用功能性用户界面测试框架，并使用它来验证他们的集成、API、视觉结果和其他一切。

“这是一个典型的例子，你有一把锤子，所有东西看起来都像钉子，”他说。“这使得您的整体测试实践非常繁重和漫长。通常情况下，如果您没有为正确的目的使用正确的工具，就需要花费大量精力来维护它。”

### 3.用户盲点

你不知道自己不知道什么——或者，正如怀曼所说，“经验错误很难发现，甚至更难发现。”

Wyman 讨论了一个假设的场景，其中工程师构建并测试一个只接受数字输入的系统。测试运行完美——因为工程师只测试数字的输入。这里的解决方法是模糊测试:超越简单用例的思维，通过测试随机和不可预见的输入(如输入字母的用户)引入“如果”。将这种远见融入编程语言本身，可以让开发人员更早地参与到质量保证过程中，避免错误的发生。

他还强调了测试中可观察性的需要，使用仪器可以帮助团队预测用户可能遇到的限制。“开发者必须关注用户分析和使用分析，而不仅仅是系统分析，”他说。“通常情况下，产品经理关注的用户分析和可观察性是从外部看到的，这对开发人员来说是有用的数据，因此他们可以尽早看到这些失败案例。”

### 超越测试限制

有没有完美无缺的测试环境？大概不会。但是通过避开上面的三个错误，开发人员可以超越根深蒂固的测试范式，提升质量和体验。
