# CircleCI 将 CI/CD 平台与 GitLab 集成

> 原文:[https://devo PS . com/circle ci-integrates-ci-CD-platform-with-git lab/](https://devops.com/circleci-integrates-ci-cd-platform-with-gitlab/)

CircleCI 今天宣布[已经将其持续集成/持续交付(CI/CD)平台与 GitLab](https://www.businesswire.com/news/home/20220726005124/en/CircleCI-Launches-Native-Support-for-GitLab-SaaS-Enhancing-Flexibility-and-Speed-for-Developers) 的竞争对手软件即服务(SaaS)产品相集成。

例如，除了合并请求和提交特定分支之外，DevOps 团队现在还可以从 GitLab 分叉合并请求中触发 CircleCI 中的管道。其他功能包括访问 CircleCI 平台上的测试功能，以及集成权限以确保安全性。

CircleCI 首席执行官 Jim Rose 表示，该公司经常将其 CI/CD 平台与其他产品集成，因为组织通常有多个团队在不同时间使用不同的版本控制平台。我们的目标是让这些团队获得能力和最佳实践，而不管他们使用的是哪种特定的平台。

![](../Images/bf26fe59caca18210c9c6a8f8cfefef7.png)

总的来说，Rose 说，组织正在试图在 DevOps 平台的标准化和通过添加新工具使开发团队能够创新的需求之间取得平衡。标准化往往会创造更多的机会，以降低成本的方式自动化更多的任务。然而，Rose 指出，开发人员最终会反抗任何限制他们用来提高生产力的工具选择的努力。他补充说，太多的高级 IT 领导很快就认为过多的工具是问题的迹象，而不是正在进行的实验的结果。

随着全球经济衰退，许多组织也在重新审视其现有的开发运维工作流，试图通过自动化更多任务来提高整体效率。Rose 补充说，这些组织意识到，他们无法承受继续在手工开发任务上投入人力。

当然，挑战在于构建和部署现代软件正变得越来越复杂。随着组织意识到他们对软件的依赖性，他们开始采用微服务来使应用程序更具弹性。Rose 说，问题是随着微服务数量的不断增加，传统的手动流程显然无法扩展。因此，自动化开发运维流程的动力变得更大了，他补充道。不太清楚的是，已经采用 DevOps 平台的组织会在多大程度上愿意替换他们当前的 CI/CD 平台来实现这一目标。

与此同时，越来越多的组织开始采用 DevOps 来管理软件开发项目，这些项目推动了广泛的[数字业务转型](https://digitalcxo.com)计划。驱动这些流程的应用程序通常需要随着业务需求的变化而不断更新。当使用围绕持续开发和[部署](https://devops.com/?s=continuous+deployment)优化的流程时，这变得更加容易实现。

无论组织选择哪种底层 CI/CD 平台，他们都应该假设最终这些平台之间的某种程度的集成是必要的。毕竟，举例来说，在收购的情况下，两家公司不太可能在同一个 DevOps 平台上实现标准化。考虑到从一个 DevOps 平台迁移到另一个平台所需的时间，互操作性所需的时间不可避免地将以月计，如果不是以年计的话。
