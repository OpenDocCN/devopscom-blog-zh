# AWS 中断和应用弹性:Roomba 取代了金丝雀吗？

> 原文：<https://devops.com/aws-outage-and-app-resiliency-did-a-roomba-replace-the-canary/>

金丝雀曾经被送进煤矿，作为危险的预警信号——对我来说，是我的 Roomba 没有自动搜索出狗毛，没有清理我儿子餐厅椅子下的地板。支持云的应用也表现得很“怪异”。有些下降了，而另一些只是缓慢——尽管在我看来，缓慢也可能下降；谁有耐心等待应用程序加载？

是什么导致了这种头痛的感觉？AWS 中断。

自 12 月 7 日停机以来的几天里，我与首席信息官、[同事](https://techstrongresearch.com)和其他行业分析师进行了多次交谈，我们共同关心的许多问题仍未得到解答。以下是对最近美东 1 号航班停运的一些快速思考。

### 了解 AWS SLAs

重要的是要记住，AWS 需要跨两个或更多区域部署实例/应用程序，以实现冗余和弹性。SLA 不仅是服务消费者衡量他们所花的钱的价值的一种方式，也是云服务提供商(CSP)和客户之间的合同协议。它概述了云服务将遵守的正常运行时间、可用性和性能标准。它还可以指定服务提供商的帮助台将在设定的时间内对断电做出响应。亚马逊有非常明确的服务级别协议。如果客户在某个月的正常运行时间少于 99.99%，公司将向其账户提供信用。

然而，有三件事要记住:

99.99%的正常运行时间协议仍然意味着您愿意容忍服务每月停机近 4.5 分钟(每年 53 分钟)，并且您必须以满足 AWS SLAs 的方式设计您的 AWS 实例和应用程序。第三点是，在 AWS 看来，停电不是他们的错。

### 停机不仅仅影响规划实践不佳的创业公司和组织

云允许组织定期做出预算和弹性决策。然而，最近的 AWS 故障不仅仅影响了诞生于网络的创业公司。迪士尼、麦当劳和亚马逊等成熟企业也受到了停电的影响。这些以技术为先的公司不会为了弹性而牺牲预算。他们知道他们是技术公司，他们有大量的 IT 预算来代表这一优先事项，但仍然被蒙在鼓里。

一个反复出现的问题是，为什么所有这些组织都因为一个亚马逊地区的衰落而陷入瘫痪？是糟糕的建筑吗？这些 AWS 客户没有完全理解 AWS 的 SLA 吗？他们是否选择了预算而不是弹性？

### 作为企业平台的亚马逊

亚当·塞利普斯基[的主题演讲和 re:Invent 2021](https://www.youtube.com/watch?v=WGA2P_oH5Xc) 的行业分析师 Q & A 明确表示，AWS 完全专注于企业。Amazon 理解 DevOps 社区的需求，但他们仍在努力开发他们的企业方法。谷歌云认识到了自己组织中的同样缺点，并从甲骨文公司请来托马斯·库里安来建立公司的企业和 IT 行业形象。

科技行业的每个人都知道会发生停机和性能下降，但他们希望服务提供商提供最佳实践并避免停机。在回应 Selipsky 的主题演讲时， 联合航空公司技术执行副总裁兼首席数字官 Linda Jojo 表示，该公司决定采用“一个云”的方法。联合航空肯定不能容忍定期停机。

### 我的观点

2021 年 12 月 7 日 不会为 AWS 活在骂名里；我们已经看到了许多其他的中断。

[应用弹性](https://devops.com/?s=application+resiliency)仍然是一个新兴的焦点领域。过去，关注的焦点是平台和数据中心的正常运行时间；然而，在许多情况下，开发人员无法控制基础设施。因此，团队需要关注应用程序的弹性，并假设底层 IaaS(云基础设施)会因为这样或那样的原因而失败。

此外，亚马逊需要提供更多自以为是的产品。灾难恢复和故障转移应该内置于 AWS 的每个级别中。软件公司不应该被迫签署多区域部署，但是 AWS 至少应该提供建议这种做法的最佳实践。
