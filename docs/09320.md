# 为什么我们仍然需要专家

> 原文:[https://devops.com/why-we-still-need-specialists/](https://devops.com/why-we-still-need-specialists/)

几年前，我在这里写道，专家——尤其是基础设施专家——不会去任何地方。

我认为这是值得更新的预测，并提醒你们所有人，大部分 DevOps 子域的专家仍然是必要的——即使你已经打破了他们的筒仓。

花时间去 DevOps 的人们经常去的地方，听听/看看他们到底在说什么。当他们把情绪表达为实时活动时，我们让一个团队负责大部分(如果不是全部的话)应用程序生命周期的想法似乎相当残酷。

在大多数行业中，随着复杂性的增加，我们的专业化程度越来越高。即使这是允许的，大多数木匠没有资格做管道或电气工作。哦，大多数人知道基本知识，必要时可以做，但他们不像水管工或电工那样是专家。在过去的许多年里，它一直试图反其道而行之，并取得了相当高的成功率。但是实用性战胜了僵化，从而取得了成功。

我们仍然有专家。你应该忽略那些建议他们离开的人。这种情况已经够糟了，没有人可以帮助你解决这种语言中的棘手问题，或者是网络设备上的问题。虽然一些语言专家(简单的例子是 T0 语言专家)被合并到 DevOps 团队中，但是他们中的一些人(例如网络管理员)只是部分被合并。在[容器](https://containerjournal.com/?s=container+networking)内部/之间的联网通常由 DevOps 团队处理(使用一套专门技术)，而允许组织所有部分运行的联网仍然由运营部门处理。但是我想不出一个我经常打交道的组织没有这些专业化的地方。

这是传统上阻碍低代码/无代码解决方案的事情之一——它们假设这些专门化，并且当事情不可避免地出错时，将需要底层技术领域的专家。DevOps、AI 和自动化也是如此(程度较轻)。有人必须知道下面正在发生什么——他们必须拥有专业知识——以便在出现问题时解决问题。

请记住，您是在要求您的 DevOps 团队既精通所有行业，又精通某些行业。这是失败的秘诀，但 DevOps 旨在承受或在短时间内克服失败，所以我只提到这一点，以便你理解为什么新员工有时看起来很惊讶，而老手似乎很疲惫——他们正在掌握和重新掌握很多东西。

所以，如果你是“逆风航行”人群中的一员，请站起来。为你能在一个有大量专业零件的环境中茁壮成长而感到自豪，并确保你知道在出现问题和力不从心时该向谁寻求帮助。即使“我可以向谁寻求帮助？”最后变成“为你自己学习，然后准备好帮助别人”，你会让它摇摆不定，而我们这些挣扎过的人会感受到你的痛苦，尊重你的工作。所以，感谢那些从未经历过，但依赖于你的工作的人。
