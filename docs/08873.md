# 敏捷与开发的职业足球方法

> 原文：<https://devops.com/a-pro-football-approach-to-agile-vs-devops/>

企业和[职业足球队](https://www.pro-football-reference.com/)有很多共同点。无论在场上还是场下，人们都能感受到在确保表现的同时变得更加敏捷、更有弹性的压力。对于首席信息官来说，生存和发展需要认真关注与业务授权和增长相关的运营。

我经常被问及关于基础设施和运营(I&O)的问题，这些问题涉及希望解决 [ITOps](https://devops.com/?s=ITOps) 、DevOps 和 [DevSecOps](https://devops.com/?s=devsecops) 挑战的组织。

*   我们如何从传统的 ITOps 模式转向变革性的数字化运营？
*   我们如何调整流程、人员和工具，以实现更紧密的集成、整合和关联？
*   我们是如何跨职能部门(ITOps、DevOps、SecOps)构建团队的？
*   我们如何快速高效地向分析驱动型运营转型？

从历史上看，敏捷团队和开发团队在 IT 领域是对立的；双方都想推进他们的计划，执行战术，到达终点区域。当团队不一致、目标不明确、资源受限时，就会出现脱节。

为了有效地提供数字服务，团队必须专注于清晰的沟通、统一的文化和措施的要素，以确保弹性、性能和可用性，如果他们是有效的。

传统的 I&O 团队通常独立运作，跨职能团队无法共享或获得知识。尝试响应和解决关键事件可能需要更长时间。对于 DevOps 甚至 DevSecOps 来说，理想的情况是通过打破工具和职责的孤岛，让团队跨职能部门保持一致。

在混合运营环境中，开发运维角色之间缺乏集成和协调很容易导致工作和优先级错位。无论是 it 环境方面的混合还是支持模式方面的混合，都有必须解决的独特挑战。当实施适当的分析开始发挥作用时，在对事件进行故障排除时，不能挤在相同的数据集周围会带来巨大的挑战。传统 IT 运营管理(ITOM)工具根本无法支持开发运维人员使用云原生服务，也无法支持远程 IT 团队在全球范围内开展工作。

为了成功的开发运维与传统的敏捷团队共存并取得成功，IT 领导者需要改变现有的思维模式，并实施新的实践以进行扩展。

*   DevOps 和 ITOps 团队必须感受到整个 IT 组织的支持。

DevOps 通常是从一小组人开发一个应用程序的实验开始的。IT 中的每个人，从领导者到 IT 工作者，都需要理解集成新方法的最终目标是什么。当 DevOps 完成一个项目并推向生产时，深度实现就开始了，这在传统的 ITOps 模型中仍然得到支持。很明显，运营需要适应频繁的变化。

*   计划并实施开发运维实践以避免中断。

由于传统系统仍在交付数字服务，开发运维团队无法在不考虑迁移因素的情况下，通过推动新的支持模式来中断关键的日常运营。相反，团队将需要使用应用程序开发交付方法在混合模型中工作。

*   协作和规划工具是必不可少的。

在混合组织中，使用协作和规划工具以及自动化功能是必不可少的。从自动化警报开始，为适当的团队生成要处理的票证，而不是让 1 级支持独立处理所有警报。

*   **建立或购买可扩展的数据分析系统**。

这些系统将聚合和关联跨基础架构和数字/云原生技术的数据，以提供对问题的上下文洞察，并提供解决问题的纠正步骤。

扩展和现代化运营以及创建敏捷实践是一个持续的过程，需要不断的规划、测试和调整。使用自动化来简化 DevOps 和 ITOps 流程启动了该流程，因为团队开始一起工作，而不是在传统的孤岛中工作。

为了赢得足球特许权，王朝不是在一个赛季中建立的；您的数字化运营转型也是如此。这需要挑战现状——认真审视 ITOps 一直以来根深蒂固的运作方式——并重新想象以数据驱动的数字化运营为重点的整个流程。
