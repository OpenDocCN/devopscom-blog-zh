# 将安全验收标准写入 DevOps 故事

> 原文：<https://devops.com/writing-security-acceptance-criteria-devops-story/>

当组织为开发编写用户故事时，应该包括验收标准。验收标准是对故事应该交付的预期特性和功能的详细描述。开发人员和产品经理经常没有将信息安全需求作为验收标准的一部分。在每个故事中包含安全标准是 SecDevOps 和 rugged 软件开发的一个要求。我经常被问到用户故事中应该包含什么样的安全标准。

虽然并不是每个用户故事都需要包含一系列的安全标准，但是有一些基本的规则是应该遵循的，这些规则对于构建一个更安全、更健壮的应用程序大有帮助。下面是几个需要考虑安全性接受标准的故事类别示例。

## 输入验证

大多数开发人员都理解验证任何类型的数据输入的重要性。与任何其他方法相比，未能正确清理和验证数据是更多类型攻击的根源。缓冲区溢出、跨站点脚本和 SQL 注入只是由于输入验证失败而导致的一些常见攻击方法。处理任何类型的输入数据，尤其是来自最终用户的数据时，最重要的是永远不要相信它——永远不要。任何处理输入数据的用户故事都需要包含验收标准，说明数据至少需要根据预期进行净化和验证。

## 私人信息的存储或传输

在组织能够对受限信息进行控制之前，他们首先必须定义什么被组织视为隐私。从定义什么被视为受限开始。数据可能分为 PII、PHI、IP、客户数据等类别。任何涉及这些数据类别的故事都需要有信息安全验收标准。根据公司选择的保护、存储或传输这些数据的方式，将决定故事的发展和接受标准。

## 密码学的任何用途

谈到加密，有一些基本但重要的规则。第一条也是最重要的一条规则是，不要发明自己的加密技术。第二条规则是使用可信的、有信誉的加密库提供商。第三条规则是只使用根据公司安全政策批准的加密功能。第四条也是最难的一条规则是确保您将加密技术用于其预期目的。在创作与密码学有关的故事时，请记住这四条规则，以及如何使用它们来制定验收标准。

## 认证、授权、记账

认证、授权和记账(通常简称为 AAA)是任何访问控制系统的核心。AAA 是控制对资源的访问、策略实施和对所述事件的审计背后的智能。更确切地说，AAA 中的一切都与信息安全相关。当开发一个处理 AAA 的故事时，应该确保测试标准包括期望访问矩阵的允许和拒绝情况。例如，简单地测试用户可以使用正确的密码登录并不意味着错误的密码尝试会拒绝访问。要考虑的另一个要点是，出于审计目的，AAA 操作通常应该包括一个日志事件。

致力于开发加固软件是 SecDevOps 的一个关键组成部分。确保故事包含安全验收标准对于将安全性移到开发管道的左侧大有帮助。
