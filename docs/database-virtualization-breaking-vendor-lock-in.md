# 数据库虚拟化:打破供应商锁定

> 原文：<https://devops.com/database-virtualization-breaking-vendor-lock-in/>

每个企业都在不断地与供应商锁定做斗争。如果问任何一位 IT 领导是什么让他们夜不能寐，那么供应商锁定肯定是最重要的。虽然网络、存储和计算早已被虚拟化大众化，但有一个领域一直是供应商锁定的最后堡垒:数据库。

能有多糟？以亚马逊为例。他们兴高采烈地庆祝最后一个内部 Oracle 数据库的退役。知情人士表示，取代甲骨文至少需要十年时间，甚至可能更长。谁也说不准完成这一壮举需要多少努力。事实上，一场盛大的庆祝活动已经准备就绪。

现在，让我们看一下数据库虚拟化。作为一个相对较新的学科，虚拟化像数据仓库这样的关键系统的想法使 [最近成为一个大热门](https://www.datanami.com/2021/11/23/cloud-data-warehouse-migration-a-group-effort-for-uks-co-op/) 。但是它适用于您的系统吗？还有一个普遍存在的问题:它会带来性能损失吗？

## 什么是数据库虚拟化？

数据库虚拟化是将应用程序和数据库相互抽象的概念。使用虚拟化，为 Oracle 数据库编写的应用程序可以在 PostgreSQL 等开源数据库上运行，而无需对 SQL 或相关的[API](https://devops.com/?s=APIs)进行更改。此外，不要将这与*数据*虚拟化相混淆，后者实际上是一种数据集成手段。

数据库虚拟化平台实时翻译查询和数据。通过仿真，它弥合了源系统和目标系统之间的功能差距，因此应用程序在移动到不同的数据库时保持不变。数据库虚拟化并没有使迁移变得更容易；这使得它们实际上已经过时了。

这个想法相当有说服力。以至于亚马逊最近开发并发布了他们自己版本的数据库虚拟化产品，目标是小型独立数据库系统。

## 为什么您需要数据库虚拟化

这里有一个简单的经验法则。在没有虚拟化的情况下，迁移典型数据仓库的成本大约是数据库未来五年的总成本。因此，如果您当前的数据仓库运行成本为每年 500 万美元(这是中端系统的典型成本)，那么您的迁移成本可能接近 2500 万美元。此外，该项目可能需要长达五年的时间。

但是等等；还有更多。传统数据库迁移的真正缺点实际上是它们糟糕的失败率。分析师认为这一比例通常在 80%以上。对于大型数据仓库系统，这个数字似乎更高。这就是数据库成为供应商锁定的最终堡垒的原因。

作为一名 IT 领导者，您自然没有时间、预算或胃口来承担这样一项高风险的任务；你可能更愿意将这笔钱用于创造新的收入。数据库虚拟化让您做到了这一点:避开这些风险，转而关注真正相关的东西。

## 何时使用数据库虚拟化

数据库虚拟化特别适合于数据仓库工作负载的平台化。这些工作负载因其复杂性和重写工作量而臭名昭著。通常，这些工作负载已经增长和管理了多年，有时长达几十年。在重新平台化的同时维护工作负载具有经济意义:这意味着保留和保护长期投资。

然而，要使数据库虚拟化成功，源系统和目标系统都需要具有相当的性能和规模。这应该是显而易见的:将工作负载从一个 Teradata 系统转移到一个 Postgres 实例不太可能满足您的业务用户。

但是有好消息。在过去几年中，云数据仓库越来越多地达到了有竞争力的性能目标。它们已经成为传统内部设备的可行替代品。从 IT 领导者的角度来看，这是一个绝佳的时机。如今，大多数 IT 领导都面临着在未来几年淘汰这些设备的任务。数据库虚拟化再及时不过了。

## 再见，供应商锁定

作为一个通用概念，虚拟化在过去 20 年中彻底改变了 IT 体系。服务器虚拟化，一开始是 QA 和开发者的小众，后来成为云计算的基础；虚拟化随后扩展到网络和存储领域。到目前为止，数据库是少数几个逆虚拟化趋势而动的学科之一。

毫不奇怪，数据库虚拟化的前提引起了从业者的极大共鸣。尽管在数据仓库领域的最初成功还有待在其他领域复制，但这一概念变得无处不在似乎只是时间问题。
