# API 在大型机现代化中的作用

> 原文:[https://devo PS . com/the-role-of-APIs-in-mainframe-modernization/](https://devops.com/the-role-of-apis-in-mainframe-modernization/)

大型机历来都是笨重的系统。它们非常安全，可以处理令人印象深刻的计算，但管理起来很复杂，从中提取数据也很困难。这一现实与强调分布式计算和便携式微服务的现代云原生基础设施不一致。那么，现有的大型机如何适应这种互联的、数据驱动的经济？

我最近会见了 Adaptigent 的总裁 Alex Heublein 博士，讨论了遗产现代化的现状。根据 Heublein 的说法，在可预见的未来，许多大型机都将继续存在。然而，企业通过在这些系统上添加一个集成层，继续从这些系统中获得价值。通过在大型机之上使用 REST API 层，企业最终可以开放这些系统。这可以促进与较新的云原生服务的双向通信，实现阶段迁移，并更接近实时处理。

## 大型机现代化的现状

软件业每天都在变化。“科技领域没有什么是永恒的，”Heublein 说，然而一些传统技术将在可预见的未来继续存在。就大型机的采用而言，Heublein 看到了几类用户。首先是大型金融服务组织，如大型银行和保险公司，它们在自己的[大型机体系结构](https://devops.com/old-man-yells-at-cloud/)上进行了大量投资，并且可能会继续依赖大型机环境来进行策略管理和核心基础架构。“我认为有很多很好的理由，”休伯林解释道。“如果你想一天处理数十亿笔交易，[大型机]仍然是一个不错的选择。”

也就是说，监管和市场压力正在影响全球银行业走向开放。例如，PSD2 在欧洲要求开放银行业务，而金融数据交换(FDX)等行业联盟正在指导它在美国市场的发展。这些计划鼓励金融服务转变其传统环境，并刺激云迁移，特别是对于非关键任务的应用程序。

Heublein 指出了企业大型机现代化的几种不同模式。一种方法是，企业重写或重新设计旧的 COBOL 应用程序，并将它们保留在内部。或者他们正在移植和重新平台化应用程序；例如，将 COBOL 应用程序迁移到 Windows 或 Linux 系统。最后，一些公司直接转向云计算。对于不需要在大型机上运行的解决方案，企业可能会转向云本机以释放容量。

## API 支持传统现代化

分布式云技术正在帮助企业扩展到合适的水平。然而，Heublein 指出，完全跳出大型机是很困难的。“你很少能一次把所有的东西都搬走，”他说。"这是一颗难以下咽的药丸。"将核心银行基础设施迁移到分布式环境尤其困难，因为瞬间的迁移可能会带来安全风险。出于这个原因，他推荐了一种“阶段迁移”方法，在这种方法中，各个组件被缓慢地迁移。例如，一家保险提供商可能会将他们的索赔引擎、保单引擎和评级引擎一次一个地转移到云中。

然而，由于阶段迁移，工程师必须同时支持内部和基于云的应用程序。它们还必须确保双向通信，以便这些系统可以交换数据。Heublein 解释说，这就是 REST API 事务层的必要性所在。这种层可以实现大型机和云原生应用程序之间的双向通信，在调用现代世界时抽象复杂性，或者拦截云数据并将其过滤为大型机技术可以处理的格式。

为了提高开发人员的可用性，从大型机集成过程中抽象出复杂性将是构建连通性的关键。Heublein 说，这可能涉及一种用 COBOL 子程序调用外部系统的无代码开发方法。为了实现流畅的双向交流，数据的自动重组也是必要的。

## 大型机集成的四个业务驱动因素

那么，在大型机系统之上放置一个集成层的最终结果是什么呢？根据 Heublein 的说法，使用这样一个抽象层有四个关键的商业好处:

1.  **编排**:软件生态系统经常需要多次调用不同的应用程序来收集和转换数据。然而，大型机环境非常复杂，很难集成。更多可用的大型机系统集成结构能够以对开发人员友好的方式更好地协调工作流。通过使用 REST 接口，所有应用程序都可以调用同一个接口，从而大大简化不同服务之间的编排。
2.  **持续的适应性:**世界正在快速加速，企业需要非常快速地维护和修改这些 API。赫布莱因说，组织需要“持续的适应性”来适应数字世界并在其中生存。然而，大型机系统不是为 21 世纪设计的。因此，低代码集成层对于快速开发至关重要。
3.  **数据转型:**数据对于业务来说是强大的。数据不仅本身是一种商品，而且是人工智能和机器学习所必需的。然而，大型机有一种完全不同的数据编码方式。因此，需要将这些数据转换成企业可以消费的现代格式。这些系统之间的集成层可以提供改进的数据消费模型。
4.  **实时处理:**你可能认为在线支付是实时发生的；然而，像 Venmo 这样的支付应用程序实际上只是银行交易的一个抽象层，可能一天只清算两次。简而言之，大型机在实时处理方面并不出色。然而，对这种基础设施的更多可用的集成可以使面向用户的应用更接近实时地交互。

另一个问题是安全性。开放承载核心财务基础设施的大型机环境必须非常小心——尤其是在 [API 攻击](https://securityboulevard.com/2021/07/api-attack-traffic-grew-300-in-the-last-six-months/)仍然是软件领域日益增长的头号威胁的情况下。从出站的角度来看，所有构建新集成的组织都需要一个安全层。Heublein 说，这通常需要在集成层之前放置一个 API 管理层或网关。该系统支持授权和功能，就像内部和外部世界之间的防火墙。他解释说，它还可以启用日志记录并防止拒绝服务攻击。

## 提高遗留系统的敏捷性

遗留系统在旧技术中根深蒂固，并背负大量技术债务，迫使运营以更慢的速度前进。维护人员在文化上也与新一代的云原生工程师有所不同。“大型机人员并不…快，”Heublein 说。

DevOps 人员希望非常快速地构建、测试和推广项目，可以说，这种敏捷性对于满足日益增长的[数字创新需求](https://devops.com/the-state-of-digital-innovation-one-year-into-the-pandemic/)是必要的。提高内部系统的集成能力可以使企业更好地利用现有投资，同时提高敏捷性。

Heublein 补充说，低代码/无代码能力也可以增加协作潜力，这可以弥合内部工程师和快速移动的业务人员之间的差距。当采用低代码/无代码工具进行大型机集成时，他告诫不要使用通用的公民开发者框架。尽管通用集成层在许多情况下都能很好地工作，但他强调，非常具体的边界框最适合为关键任务应用程序构建 API。
