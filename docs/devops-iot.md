# DevOps +物联网

> 原文：<https://devops.com/devops-iot/>

物联网(IoT)一部分是营销炒作，一部分是真正的运动。在低功耗 WiFi 芯片允许小型设备长时间连接网络之后。还记得 2005 年成立的 Eye-Fi 吗？

但不可否认它很大。根据 IDC 最近的一份报告，基于微控制器的物联网的服务收入预计到 2017 年将达到 7.3 万亿美元，DevOps 的全新范式转变将导致物联网采用的完美风暴。

物联网将会颠覆那些构建和连接事物的方式。它推动了一场转型，这场转型在很大程度上是由开发、It 运营和质量保证实践的集成推动的，这些实践是在面向开发运维的世界中的单一自动化保护伞下交付的。

**导致突破性成功的复杂解决方案**

智能企业和住宅 IT 网络中的物联网实施为 DevOps 带来了独特的挑战，因为需求远远超出了软件开发生命周期，并包含复杂的质量保证和强大的后端支持阶段。主要要求、考虑事项和挑战包括:

**可审计性和可追溯性**

虽然物联网在很大程度上是由消费者驱动的，但该技术在企业市场中同样普遍。在这种情况下，DevOps 工程师必须解决所有物联网固件操作系统开发的可追溯性和可审计性，以确保合规性成功。在整个开发过程中与硬件产品专家和供应商的合作也确保了软件的健壮性，从而能够简化与现有 IT 网络的集成，同时避免供应商锁定。

**兼容性和互操作性**

随着物联网终端数量的爆炸式增长，容量有限的世界网络基础设施正在达到极限。这进而推动了影响更广泛 IT 网络的互操作性、联网和连接性问题，而专注于网络环境、协议和标准的物联网发展可以帮助消除这些问题。

鉴于全球物联网生产和部署的规模，维护一个强大的后端架构来自动化测试和升级需要全面了解开发周期以及单一存储库来跟踪设备部署后的变化。

当数十亿台“哑”设备相互交互时，互操作性问题自然会出现。在开发物联网时，考虑到 API 的发展，以展示硬件的独特功能，确保轻松推出升级，解决可能出现的集成、连接和互操作性问题。

**彻底测试**

物联网硬件的性能和行为属性给 DevOps 工程师带来了独特的挑战，他们必须在复杂的现实环境和用例中测试物联网软件。例如，天气条件和硬件的耐用性会影响软件性能，尤其是当技术是为响应环境条件而设计时，如联网的自动火警。

**复杂的质量保证和持续支持**

质量保证本质上是复杂和专门化的，架构的负担几乎完全落在后端。借助这种服务模式，DevOps 工程师可以频繁推送更新，因为应用商店的缓慢审批流程不适用于物联网软件。物联网生态系统的庞大也导致了这些设备的应用需求的不可预测性。因此，即使在发布后，对物联网应用的理解也会发生变化，促使定期进行重大更新以纳入所需的变化。使用 DevOps 方法，这些更新通过持续交付服务模型从后端直接推送。

然而，这种便利是有代价的。随着我们转向新的开发方式和软件版本，为易受攻击的设备提供智能以在无人监督的情况下执行自动化操作，这为代价高昂的安全性、可访问性和稳定性威胁打开了大门。在发布时将安全性有机地构建到物联网设备中，并根据用户反馈推动以安全为中心的更新，可确保用户数据隐私和网络安全。

DevOps 方法还应该解决 IT 现实和管理期望之间的脱节，从而导致企业客户对互操作性和生产力的担忧。因此，在企业环境中，使用创新的软件解决方案来升级昂贵的硬件是必要的，因为在这种环境中，不断主动寻求具有兼容功能的新物联网产品。专注于标准化的物联网环境，只需最少的定制即可成功实施，这极大地减少了兼容性问题。根据物联网设备的处理能力对数据进行加密，并限制物联网应用的功能，也可以在第一时间防止易受攻击的物联网端点执行不必要的操作。

在这种情况下，传统的 DevOps 方法“构建一点点/失败一点点/修复一点点”将不适用于物联网设备，这些设备在运行失控时可能会被证明是致命的—[想想医疗物联网](http://readwrite.com/2013/09/18/internet-of-things-security-disaster-terrorism-war)！

至少可以说，正确地完成它，回报将是不可言喻的。
