# Dynatrace 在可观察性平台中嵌入了 DX 监控

> 原文:[https://devo PS . com/dyna trace-embeds-dx-monitoring-in-observability-platform/](https://devops.com/dynatrace-embeds-dx-monitoring-in-observability-platform/)

Dynatrace】增强了其分析能力，以支持数字体验监控，包括基于其通过可观察性平台收集的日志、指标和跟踪的会话回放。

Dynatrace 产品管理高级副总裁 Steve Tack 表示，这一更新使 DevOps 团队能够更简单地跨构成现代应用环境的多个组件和微服务跟踪用户旅程。例如，Tack 表示，Dynatrace 的分析功能将自动启动会话回放，使 DevOps 团队能够查看所遇到的应用程序问题的确切性质，而不是试图通过在没有任何上下文的情况下解析日志数据来确定问题的影响。

以前，开发运维团队需要单独部署 Dynatrace 会话重放产品。他补充说，总的目标是通过以一种更容易理解的方式提供上下文，消除 DevOps 团队在试图修复应用程序问题时遇到的许多摩擦。

![](../Images/bf451f8d9c6d276c4d02db712a002330.png)

Tack 说，总的来说，随着[可观察性](https://devops.com/s?s=observability)的不断提高，现在可以收集的数据的数量和类型变得越来越少。他补充说，这些数据是实现可观测性的关键。他说，现在需要关注的是，一旦收集到数据，可操作的分析就会浮出水面。

Tack 指出，事实上，随着越来越多的组织开始基于高度相互依赖的微服务构建和部署应用，分析的丰富性现在比以往任何时候都更加重要。他补充说，虽然基于微服务的应用程序比单片应用程序更具弹性，但每当应用程序性能下降时，对它们进行故障排除更具挑战性。

目前还不清楚 DevOps 团队接受可观测性的速度。这个概念一直是 DevOps 的核心原则，但是，在大多数情况下，更成熟的 DevOps 团队只能实现对预定义指标的持续监控。可观察性平台承诺在 IT 问题升级之前发现异常情况；然后，开发运维团队可以启动查询，以更好地确定其根本原因和潜在的严重性。

Dynatrace 正在为一个可观察性平台提供案例，该平台作为云服务访问，采用该公司的戴维斯人工智能引擎和机器学习算法来揭示应用程序问题的根本原因。

随着可观察性平台的不断发展，IT 团队通过排除过程来确定 IT 问题根本原因的日子终于要结束了。开发运维团队将获得更多对其应用环境的可见性，从而获得更有意义的可操作情报。这种智能对于推动数字业务转型计划的面向外部的应用程序尤为重要。

当然，当谈到可观测性平台时，并不缺乏选择。在某些情况下，IT 组织选择依赖云服务，而其他组织则扩展他们自己构建并继续维护的 DevOps 平台。不管采用哪种方法，问题不再是如何收集数据，而是收集到数据后如何处理。
