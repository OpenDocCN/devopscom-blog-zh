## **测试自动化**

定义测试自动化要求。根据产品或服务的测试策略，为每个新功能创建新的自动化测试。自动化测试用例被审查和测试，以确保它们满足测试的目的。
自动化测试用例符合产品或服务的测试用例标准。
自动化测试用例被设计成彼此独立运行。

## **测试工具**

管道中的所有测试工具都集成到一个通用框架中，该框架支持测试资源的编排和测试任务的自动化。测试框架支持测试所需的一切自动化，包括测试用例、测试工具、应用程序、测试基础设施和测试数据。
可以通过[应用编程接口](https://devops.com/?s=APIs) (API)调用、控制和发布一个测试框架，这样它就可以集成到 DevOps 连续交付工具链中。
测试框架提供的功能包括编排测试基础设施、选择测试、控制测试的执行、监控正在进行的测试以及报告测试结果、日志和结论数据的能力。
为测试工具定义了选择标准。该标准包括对覆盖率、自动化特性、支持、与自动化框架的兼容性、与测试进度报告系统的兼容性的要求。
测试工具软件版本保存在源版本管理系统中。为整个组织定义了一个通用的测试自动化框架，以确保所有的测试工具能够与通用的测试自动化实践相集成和协调。
测试自动化工具可用于测试可自动化应用的所有特性。
手动测试工具可用于测试必须手动测试的功能。这些工具可以向一个通用的测试管理报告系统报告结果。
·测试创作工具支持输入错误检查和特性，有助于鼓励根据最佳实践开发脚本。这种特性的例子包括创建和使用已知良好的测试实用程序的存储库的能力，用于被测系统的控制和分析。
采用 DevOps 就绪测试工具，可轻松集成到测试框架和工具链中，并可根据需求进行纵向或横向弹性扩展，以满足不断交付的软件变更测试的不同容量和工作负载需求。
devo PS 就绪型工具可通过 API 进行编排、扩展、调用、控制和监控。

## **测试基础设施管理**

记录了描述测试配置的资源和拓扑的标准。使用加速测试执行时间的技术，如测试活动的垂直和水平扩展。
测试资源和拓扑文件保存在软件版本管理系统中。
进行流程编排和自动化测试，以验证支持流程编排和自动化层的软件是否满足其评估标准
。用于流水线每个阶段的测试基础设施是生产配置变化的复制品，尽可能符合实际。
用于测试的基础设施是有弹性的，能够根据特定测试的需要建立和发布(即编排)基础设施配置。用于编排测试环境的典型技术包括动态基础设施配置工具、基础设施即代码(IaC)策略、云服务、测试即服务(TaaS)提供商，以及使用工具来支持和发布应用程序以及测试封装在不可变容器中的资源。
还需要测试数据库迁移、复制和恢复工具，以确保它们能够足够快地运行，跟上 DevOps 持续部署管道节奏的速度。
测试环境的健康检查用于减少测试环境故障导致的假阴性测试的机会。

## **测试脚本**

测试脚本保存在软件版本管理系统中。测试脚本和测试套件实践的指导方针被记录在案。
·测试实用程序、测试和测试套件遵循文档、测试结果报告、重用和维护实践的指南。
进行测试脚本同行评审，以鼓励协作，并确保测试脚本最佳实践得到使用。
·每个测试脚本都有一个唯一的标识。
确定脚本的测试类型(例如，健全性测试、构建测试、冒烟测试、安全性、验收测试、一致性测试、功能测试、单元测试、补丁测试、缺陷修复测试、集成测试、回归测试、性能测试、系统测试等)。
包括执行日志在内的证据证明测试的每个结论都有效。
测试至少确定一个结论，可能是“通过”、“失败”或“不确定”
·记录或报告裁决和主张的优先级。
测试是数据驱动的。数据输入控制测试的执行。每个测试脚本都标有编码的风险等级。
脚本报告的失败判断包含一个风险优先级标签。

## **测试结果**

使用技术确保测试结果分析跟上加速测试。例如，测试设计技术、测试框架中内置的分析工具、测试结果仪表板和/或与测试并行运行的测试结果分析器。质量保证目标包括发布前测试覆盖范围的具体阈值，如运行到计划(RTP)和通过到计划(PTP)的最低阈值。RTP 是对所有计划测试中已经完成的测试数量的度量。PTP 是对有多少已完成的测试通过的一种度量。例如，95%的 RTP 和 90%的 PTP。
·质量目标包括可靠性的测量和阈值。例如，当通过率在一段时间内稳定时，通过率的 S 曲线可以指示可靠性。例如:连续三个测试周期 95%的通过率。
对故障服务的恢复时间进行例行测试，并用于计算平均恢复时间(MTTR)。
进行监测和管理测试，以验证管道的监测和管理能力满足其评估标准。
技术和业务指标旨在监控 DevOps 测试在技术持续改进方面的成功，并证明实施这些改进所需的业务案例。
价值流中的每个阶段都有输入和输出指标以及验收标准。

## **这意味着什么**

持续测试(CT)实践对于开发运维的成功至关重要。许多组织低估了工程化持续测试的重要性和实践。本文是由三部分组成的系列文章的第二部分，它提供了一个全面的实践列表，可以用来理解和评估一个组织的持续测试能力。要了解更多关于我的持续测试蓝图，请参考我的书[工程开发](http://mybook.to/engineeringdevops)。
