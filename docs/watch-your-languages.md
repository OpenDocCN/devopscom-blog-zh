# 注意你的语言

> 原文：<https://devops.com/watch-your-languages/>

如今，我所熟悉的每一个组织都在使用更多的语言、框架、库等等。比十年前更好。当时，大多数组织都在做大量的 Java 工作，可能还会做一点 PHP 或。NET(几乎都是 VB。网)。现在我们有 Java，Python，Ruby，Go，Swift 等。等等。等。

问题是，员工不会事先得到你选择语言的指示。大多数开发人员会使用 Java 或 Python，并且大多数人至少精通一门其他语言。他们都可以学习(稍后会有更多)，但是在招聘时，组织需要关注招聘团队正在使用的一种或两种语言，而不是组织可能要求开发人员在雇佣过程中学习的所有语言。毕竟，20 年前招聘时，大型机团队的工具通常不被列为桌面开发人员的要求。虽然存在差异(只有当有经验的大型 iron 开发人员非常缺乏时，桌面开发人员才被要求在大型机上工作)，但今天的情况类似。

你可以着眼于培训来招聘，大多数开发人员会非常愿意学习一门新语言并了解它的来龙去脉。但是去问任何一个密切关注的经理，他们会让你知道高效时间和熟练时间之间的巨大差异。一位经理告诉我，任何有经验的程序员都可以相对较快地开始使用 Ruby，但是做重要的事情——比如知道哪种数组访问方法更快，以及 Ruby 和其他语言之间最有效的通信——需要更长的时间。在我看来，根据我的经验，他的数字似乎太高了，所以我四处打听。我认识的所有与 Ruby 一起工作的经理(因为在很大程度上，这是语言相关的，所以我不想草率地概括，尽管事后看来，Python 中的相同数据点会给我更多的回答)都同意有一条曲线，范围在六个月到两年之间。这完全是轶事，但似乎问题团队的技术含量越高，经理认为熟练需要的时间就越长。

但是，即使较低的数字是准确的，并且更普遍地适用，这也是保持你的语言倒计时的一个理由。是的，有些语言在 X、Y 或 Z 方面比其他语言更好，但是大多数现代语言可以做一般企业团队要求它们做的任何事情。因此，将语言保持在一个更小的预定义集合中仍然允许团队选择工作的最佳工具，同时将组织中开发人员可能需要的技能数量保持在可管理的水平。

继续让他们惊叹。无论您使用何种语言，您的用户群每天都在登录并使用结果，或者您的设备每天都在报告或处理结果。正如我们的数字世界所展示的，无论你如何交付，你就是。继续摇摆吧，尽你所能减少你家里的语言数量。
