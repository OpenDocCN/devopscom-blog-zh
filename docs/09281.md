# 左移只是安全软件交付的一部分

> 原文:[https://devo PS . com/shift-left-is-only-part-of-secure-software-delivery/](https://devops.com/shift-left-is-only-part-of-secure-software-delivery/)

我们生活在一个加速消费和交付的时代。你可以在两天内从数千英里之外免费获得送货上门的看似无限的产品选择。你可以在几秒钟内在线获得各种各样的服务:电影、音乐、游戏、教育甚至医疗保健。这些现代奇迹是通过软件开发、交付和运营方面相对较新的进展提供给我们的。

这些进步包括采用 DevOps 实践、持续集成和持续部署(CI/CD)管道，以及自动化从基础架构供应到测试和质量保证的一切。这些进步使开发人员能够更快地交付软件，并在设计和开发的早期考虑安全性，以交付更安全的软件。

## 什么是“左移”,为什么它很重要？

" [Shift left](https://devops.com/?s=shift+left) "是一种解决传统瀑布式开发陷阱的哲学。在瀑布式开发中，许多 IT 团队经过数周或数月的工作，最终开发出一个大而漂亮的软件版本。当然，人是会犯错，会忘事，可能相互沟通不好。当发现应用程序元素不能很好地互操作时，团队疯狂地试图修复可能的东西，抛弃不可能的东西，一个非常严峻的期限就摆在他们面前。产品不可避免地会延迟交付，仍然有些破损，安全性是事后才加上去的。

向左移动意味着将安全过程和工具更早地引入到设计和开发阶段。安全性应该与功能和质量一样重要。这是一个很好的主意，因为当你已经在一个关键的依赖项上构建了几十个功能之后，在其中发现一个关键的、导致构建失败的漏洞，要比在你构建任何东西之前就发现它所带来的伤害要大得多。也许修复这个问题就像使用一个新版本的依赖项一样简单，但是现在你必须确保你定制的所有东西仍然能按预期工作。

有许多形式的应用程序安全测试(AST)旨在系统开发生命周期(SDLC)的早期检测代码级问题。向左移动时，AST 的两种常见类型是软件组合分析(SCA)和静态应用程序安全测试(SAST)。简而言之:SAST 测试开发人员编写的定制代码；SCA 测试代码中包含的依赖关系。

评估现代应用程序的安全性需要两种测试类型。如果你认为左移不仅意味着发现问题，还意味着解决问题，那么开发人员友好的工具是必要的。这到底意味着什么可能是主观的，取决于组织中的开发人员工作流，但至少，开发人员需要自助服务，无缝集成到他们现有的 CI/CD 工具链和可操作的结果中。因为大多数开发人员不是安全专家，所以安全发现必须优先考虑最危险的问题，指出问题的根源，并提供上下文相关的修复，而不是让开发人员自己去寻找它们。

## 用运行时智能增强 AST

进行大量早期测试——尤其是安全测试——的一个挑战是，测试越多，发现的问题就越多。不幸的是，一些结果会是假阳性，而另一些可能是真阳性，造成相对较低的风险。具有高度安全专业知识的开发团队必须筛选和审查发现，以优先考虑严重的和可操作的缺陷，但是这样的从业者很少，并且这种类型的工作很乏味。

解决安全测试的陷阱并不简单，它涉及收集尽可能多的上下文信息，以开始推理与每个发现相关的风险，通常由实际执行的可利用代码来表示。

对漏洞风险进行推理并不容易。我们试图对问题的关键程度进行排序，以决定采取何种行动。我们可以把问题分成两部分:

*   漏洞本身的背景包括严重性指标，如 CVSS 分数，以及威胁情报，如漏洞是否公开或攻击是否普遍。这些信息很容易从扫描工具和第三方来源，如 NVD，米特和 CISA。
*   然而，受影响资产的环境背景是特定于您的应用程序和基础设施的。只有你的组织才能真正填补这一部分。例如，受影响的应用程序是否受到缓解安全控制的保护？是暴露在网络上吗？易受攻击的组件会被使用吗？在应用程序在其预期的生产环境中运行之前，很难或者不可能知道这一点。

左移是现代网络安全的一种前进方式，但这种方式可以通过引入运行时智能来大大丰富。这有助于确定优先修复的内容，消除摩擦，节省大量 IT 团队的周期，并降低组织的漏洞风险。

## 发布后会怎样？

左移的全部意义在于更快地发布安全软件，但接下来会发生什么呢？您完美无瑕的代码以其惊人的特性集和令人印象深刻的性能让您的客户欣喜不已。然后，在某个无所事事的周五下午，就在本该是一个长假周末的前一天，你遭遇了可怕的零日[，这是你即使努力也无法预测的。](https://sysdig.com/blog/exploit-detect-mitigate-log4j-cve/)

发生这种情况时，您需要快速识别环境中运行的所有受影响的工作负载，并对它们进行优先修复。您可能需要将这些工作负载离线以立即进行修补，限制对它们的网络访问，或者在修补程序不可用或部署时间过长的情况下缓解问题。重点是，在现实生活中，左移只是成功的安全计划的一部分，而针对右侧生产环境的保护、检测和补救活动对于您组织的安全和成功也至关重要。

## 结论

单靠 Shift left 永远无法解决所有的安全问题，但它确实为可靠、快速和安全的软件交付管道奠定了基础。安全和开发团队经常遭受噪音泛滥和很少有用信号的困扰。他们真正需要的是一致的、规范性的方法，以便对风险做出合理的决策。通过堆栈的多个层从多个来源收集上下文，包括运行时信号，极大地提高了 DevSecOps 团队从他们的工具接收到的信息的质量，并且必须对这些信息采取行动。
