# DevOps 的数据存储问题

> 原文:[https://devops.com/devops-data-storage-problem/](https://devops.com/devops-data-storage-problem/)

技术领域一直是解决问题的地方。当[大数据](https://devops.com/?s=big%20data)的价值最终被接受时，得益于 90 年代末和 90 年代初开发的新分析功能，行业通过投资本地数据中心来帮助存储数据，从而调整其对存储的态度，以推动更好的业务决策。当这种数据的增长加速了软件开发和部署，在开发人员和运营人员之间产生了组织孤岛时，行业通过创建 DevOps 来改善协作。

现在，DevOps 彻底改变了公司的工作方式，消除了孤岛，提高了灵活性，增强了可见性，从而实现了更快的部署和更好的整体服务。这些新的工作方式只会增加产生的数据量，并且随时随地访问数据的必要性使得迁移到云成为任何成功的 DevOps 团队唯一真正可行的解决方案。不幸的是，这催生了科技行业需要解决的新挑战:云。

## 数据和云困境

早期的 DevOps 采用者迅速利用云的优势，通过提高数据可访问性来增加协作。然而，这些早期采用者通常与三大云供应商签订协议，提供更大的存储容量和看起来更灵活、更易访问的格式。在大多数情况下，这些协议会立即带来满足感，但随着开发运维的不断扩展，随着服务和出口费用的不断增加，新的棘手问题也随之出现，团队不得不增加预算以满足存储需求，或者冒着达到限制和数据丢失的风险。

这种情况造成了云存储的困境:公司要么支付比他们需要的更多的云存储，要么必须选择保留什么数据和删除什么数据。这些不可持续的供应商协议带来了扩大或缩小存储空间的财务难题，超过了保持每天产生和收集所有有价值的数据的好处。因此，大多数人都在考虑存储多少和存储什么，而不是使用所有数据来推动业务发展。

这种形式的供应商锁定对开发运维团队来说是一种损害，因为他们依赖看似无穷无尽的数据来进行实验、进行维护以及开发和部署新的应用。

## 用无止境的心态加速开发

IDC 最近的一份报告指出，仅今年一年，存储的数据量就有望达到 59 兆字节，未来三年的创建和消耗几乎超过了过去 30 年的总和。虽然 DevOps 肯定对这一激增有所贡献，但我们数字世界的现实是数据为王，一家公司无缝存储、访问和利用数据的能力将使其在竞争中脱颖而出。

这种增长的问题是找到可持续地将这些数据存储在云中的方法，特别是当团队被耗尽预算的现有存储协议所束缚时。这从 DevOps 团队重新思考他们的方法开始。云战略不需要将云存储和可访问性视为账单和服务限制通知的循环，而是需要关注只为任何给定时间所需的空间付费的能力，而不必担心访问归档数据的额外费用。需要扩大存储容量吗？太好了！你想加多少就加多少。出于特殊原因需要缩小规模？不要因为闲置在供应商数据中心的不动产而紧张。需要随时访问存档数据？不要因为害怕高额的出口费而犹豫不决。

随着这一转变，DevOps 团队可以从保存、归档和销毁数据转向收集和利用所有数据，以推动当今数字经济所需的数据驱动型洞察力。这种向“无底”云转变的思维方式消除了限制，并使创新能力以更快、更可持续的速度发展。

## 开启 DevOps 创新

人们经常认为开发运维与数据管理是独立的实体，但随着软件和应用开发不断发展，包括预测分析等功能，很明显这两个实体之间需要联姻。通过重新思考其数据战略并采用无止境的方法，DevOps 团队可以释放更高的效率，引领下一次技术创新革命。
