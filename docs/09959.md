# 保护 DevOps 的 5 个技巧:你希望你能早点知道的事情

基础和框架，混凝土和钢铁——不令人兴奋。但这是几乎所有现代建筑的基础和框架。建筑的其他部分——地板、电线、照明、房间布置等等——都是通过基础和框架实现的。如果您希望 DevOps 在您的组织中取得成功，您必须从构建一个坚实的基础和框架开始，包括安全性。以下是 DevOps 混凝土和钢的建议方法。

### 三个基本需求

**1。领导和治理**
必须有人做出决策，负责领导团队，给出方向并提供可操作的治理。保护 DevOps 不仅仅是遵从一组安全概念；这也是为工作场所设定正确的基调。领导力和治理是发展成功的关键因素。

更快、更频繁地部署软件的压力是真实存在的，但安全性、监管和许多其他围绕自动化的问题也是真实存在的，这些问题可能会在软件投入生产之前发现软件中的错误或缺陷。在面临更快发布代码压力的环境中，如何平衡这些相互竞争的优先级？速度和质量哪个更重要？速度和质量之间的矛盾仍然是当今采用 DevOps 实践的组织所面临的主要挑战之一。

关于人员和数据的组织领导和治理必须到位，以保持人们在正确的事情上工作，保持士气，提供关于项目优先级的方向，并为完成工作提供正确的工具。

治理的一个重要子类是数据分类。数据驻留在哪里？它流经哪些渠道？是什么样的数据？对数据进行分类会对资源采购产生长期影响，例如使用什么存储，是否需要加密，对环境的访问控制需要有多严格，以及各种技术的互操作性。

**2。法规遵从性**
就在不久前，如果一家企业仍然只是全国性的，那么就不一定有太多的法规要求。目前，如果一个企业有一个网站(谁没有呢？)，那么国际和跨境数据传输是必然的，除非采取地理限制的过程，这本身可能是一项具有影响的重大事业。

确定监管需求。人们或其他服务从哪里访问数据？涉及哪些跨境转移？套用 20 世纪 60 年代末至 80 年代的公共服务公告，“现在是晚上 10 点，你知道你的数据在哪里吗？”相应地设计和实现。

**3。风险管理**
技术管理的一个基石是[一切都与风险有关](https://securityintelligence.com/articles/journey-organizational-resilience-security-by-design/)。如果发生严重事件，如数据入侵或失窃，组织会付出什么代价？公司是否是问题的一部分(例如，疏忽)？

产品或服务存在哪些技术漏洞、威胁和风险？评估和解决风险需要什么样的测试(如回归、渗透、漏洞、CTI)？补丁抑扬顿挫是什么样子的？

在人事方面，关键人物离开的风险有多大？在文化和薪酬方面，团队的士气如何？如何应对这些风险？包括第三方风险、合同风险和隐私风险(还记得上面的数据分类吗？).

NPM 包裹作为一种剥削手段有日益增长的趋势；正常运行时间是履行义务的常见要求，许多人认为隐私不仅仅是一种特权，而是一种人权，不能掉以轻心。

### 两个框架活动

基础活动支撑着 DevOps 的整个结构，但也要考虑是什么塑造了这个结构。

**1。**
软件开发生命周期(SDLC)SDLC 至关重要。没有它，所有的人员(团队、承包商、顾问)哪里知道如何进行产品的设计和实现？使用什么模型，谁负责什么？这些都在 SDLC 中。当然，像 API 架构细节和必然应用程序文档的参考资料这样的东西会让人感觉像在国会图书馆里闲逛(并且让开发人员想把它烧掉，就像古埃及亚历山大图书馆一样！).

说到 API:由于 API 在所有行业(包括医疗保健和零售)中的使用都在大幅增加，任何 API 的使用都必须是 SDLC 的一部分。

不仅公共使用和交互的 API 在增加，而且组织内微服务的使用也在急剧增加。API 对于企业内部和外部都是必要的，以达到、保持和加速创新的速度，因此将它们作为关键资源是至关重要的。每家公司的平均 API 使用率增加了 221%，因此开发者需要通过确保适当的活动(尤其是安全性)来关注该生态系统。

SDLC 通过将公司和客户都需要做的事情编成法典，使工作流和生产一致(并随之变得更容易)，而不是通过呈现令人作呕的乏味来使工作变得更加困难。像任何公司政策一样，它需要保持有用，同时不被削弱。

以下是一些需要包括的具体方面:

●需要包括某种形式的威胁建模。这不一定是一个巨大的努力，但威胁的现实将在发展中发挥重要作用。
●测试。回归、压力、pentest、安全——这些和其他的将有助于指导一个人需要什么样的测试环境，并导致任何增加诸如容量和可伸缩性等项目的需求。
●安全编码。如果已经完成，采取措施改进。如果还没有完成，那就开始吧。回过头来修复不安全的代码总是很难，但由于向公众提供软件涉及许多风险和法规，这是一种必要。

**2。培训**
培训是必需的，因为企业必须适应技术变革带来的新机遇，从而不断改变客户需求。

训练无处不在！没有一个正确的地方或正确的方法-只需根据需要进行培训和改变(具体细节应由领导层传达)。

在严格和低预算的情况下？一种选择是 OWASP 会员资格(每人每年 50 美元)，包括几种类型的 AppSec 和 DevSecOps 培训。这一小小的投资可以帮助提高干净和安全编码的水平、威胁和安全意识以及专业声誉。

### 设定路线并始终保持运动状态

在商业中设置任何课程的一个问题是，它是半永久性的。DevOps 模型成为基础的一部分，这可能使人不愿做出承诺。证明所选方法的唯一方法是将它们固定下来，并长期坚持使用。模型的某些方面可能会随着时间的推移而改变，但是一旦这个基础建立起来，就需要大量的工作、时间和金钱来改变它。这种改变不是一种改造，而是一种重建，所以模型在一开始就考虑到了正确的路线。

虽然 DevOps 需要一个坚实的基础，但它是一个不断发展的模型。有新技术、新人、业务变化、客户需求以及社会和文化的转变。这些概念可以包括在计算中，但这些变化将是不可估量的。雇佣和培训合适的人，必要的步骤和调整将成为可能。
