# 谁控制你的构建过程？

> 原文:[https://devops.com/who-controls-your-build-process/](https://devops.com/who-controls-your-build-process/)

我们在短时间内取得了很大进步，目前正处于“消化”状态，我们需要稍事休息，为下一波变革做准备。好吧，不是真正的突破，只是我们所看到的变化。

在过去，CPU、内存速度、网络吞吐量和磁盘访问时间会不断变化；一个是服务器性能瓶颈，当这个问题解决后，另一个就是瓶颈了。我们现在就是那种关系。大多数组织现在都在使用 DevOps，我们目前的重点是 DevSecOps 和云计算。一路走来，通过不成文的同意，供应商们已经开始称 [Kubernetes](https://containerjournal.com/?s=Kubernetes) 为“云”。作为一名学者，我觉得这很有趣，因为 Kubernetes 真的是一个“云杀手”

所有公共云都基于这样一个前提，即它们将被大规模锁定。而 Kubernetes 打破了这个前提——除非一个组织使用云厂商定制版的 Kubernetes；然后他们还有其他问题。不，我不盲目，我很清楚发生云锁定的其他原因，但“我们选择了这一点自定义功能，很清楚它将我们锁定在我们选择的供应商中”和“使用该供应商的行为将我们锁定”是有区别的如果没有像 Kubernetes 这样的工具，多云和遣返是不会被广泛讨论的事情。

在我看来，下一点是，一旦云可移植性在大多数组织中成为既定事实，我们将回到构建管理的另一个迭代。许多组织仍然处于 CI/CD 1.0 的迭代阶段，并且使用 Jenkins 来管理他们的构建过程。虽然 GitLab 和 Atlassian 等公司已经从 Jenkins-as-build-management 现象中分得一杯羹，但它们也通过增值牢牢锁定了市场，而且这两家供应商(以及 DevOps 为中心的构建管理的 wave 2.0 中的大多数其他供应商)都在云领域进行了大量投资。虽然在云中构建的能力是一件非常好的事情，但是对您的工具链至关重要的工具从云中运行的要求却不是。

这意味着我们将再次回到这个问题上来。而且可能越早越好。不可否认，未来几个月，我们大多数人的预算都会减少，我不知道预算何时会反弹。这意味着经常性开支——比如强制性的月订阅费——将会像在 B2C 一样对 B2B 产生冲击。不要回避这些评估。实际上，客观地看待您的每月付款计划，并确定您的组织使用的每个订阅的投资回报率。几年前，组织集中并减少了他们管理的开发工具订阅的数量；这只是那个过程在现代的延续。把你的组织的钱花在能获得最大回报的地方，而不是“我们总是有的地方”

您还希望确保您信任负责您的构建的公司，并且您相信他们是稳定的。这些年来，我们已经看到了一些公司看起来很稳定，但实际上并不稳定的例子；实际上，做出评估是很重要的。

您正在构建和维护使组织保持在线和高效的系统。不要犹豫去削减脂肪。不要犹豫保持控制。如果工具和它的锁定迫使组织以不适合文化的方式工作，那么就寻找替代方案。

简而言之:保留你的选择。我们知道在不久的将来资金会更加紧张，所以让我们确定我们已经为此做好了准备，并且保留了对我们构建过程的控制。继续摇摆。
