# CircleCI 首席执行官认为对开发人员体验工程师的需求越来越大

> 原文:[https://devo PS . com/circle ci-CEO-sees-growing-need-for-developer-experience-engineers/](https://devops.com/circleci-ceo-sees-growing-need-for-developer-experience-engineers/)

构成 DevOps 工作流的所有任务都不缺少头衔，但鉴于开发人员现在在任何组织中扮演的重要角色，组织可能应该开始在 DevOps 团队中创建开发人员体验工程师(DXE)职能，专门负责提高[开发人员](https://devops.com/?s=developer)的工作效率。

CircleCI 首席执行官 Jim Rose 表示，随着更复杂的应用程序的构建，应用程序开发和部署过程中的摩擦比以往任何时候都多，这在一定程度上要归功于微服务的兴起。

CircleCI 发布的[报告](https://www2.circleci.com/DevEx.html)中概述的 DXE 将负责实施一套通用的 DevOps 原则，维护正确的工具，并实施一套有凝聚力的标准流程，以确保开发人员的成功。Rose 说，今天，在同一个组织中使用了太多的语言、框架、工程风格和过程，这共同降低了开发人员的整体生产力。

虽然软件工程师今天正在解决这些问题的子集，但 Rose 表示，是时候将这些任务分配给一个工程团队了，该团队通过提供更具响应性的基础设施，将自己视为开发人员的倡导者。

随着各种规模的组织越来越意识到他们如何依赖软件来推动数字业务转型计划，Rose 指出，开发人员的生产力正在成为一个主要的企业问题。收入预测和利润率现在直接受到软件交付和更新速度的影响。

Rose 指出，今天的软件工程师通常为开发人员构建 DevOps 管道，但没有太多的时间和精力用于优化开发人员的体验。Rose 说，这不仅仅是一个主要的生产力问题，它对组织吸引和留住开发人员的能力有着实质性的影响。许多开发人员更喜欢为小型创业公司而不是大型企业工作的原因之一是，他们的日常工作体验更好。Rose 说，如果大型组织想要争夺同样的人才，他们需要减少开发人员经历的摩擦。

Rose 指出，DXE 也更倾向于征求开发人员对工具选择的意见，以最佳地优化生产率。今天，许多软件工程师在构建管道时，很少或根本不考虑开发人员使用工具的简单或困难程度。

当然，今天组织开始越来越痴迷于测量应用程序开发速度。然而，在许多情况下，简单地让某人负责实现一个目标会以更低的成本获得更多的收益。分析工具可以发挥作用，但专注于关键绩效指标(KPI)可能不会像简单地给工程师一个具体的命令那样迅速产生相同的结果。

随着应用程序管理的责任继续向开发人员转移，人们对底层基础设施的自动化越来越感兴趣也就不足为奇了。在这一点上，问题是确定软件工程师将在多大程度上实现这一目标，而不是迫使开发人员寻找另一个承诺代表他们实现这一目标的平台。
