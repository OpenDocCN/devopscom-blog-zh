# DevOps Chat: BlueOcean 让每个团队都能接触到 CD 管道

> 原文:[https://devo PS . com/devo PS-chat-blue ocean-makes-CD-pipelines-in-reach-every one/](https://devops.com/devops-chat-blueocean-makes-cd-pipelines-within-reach-everyone/)

蓝海为每个人带来了易于使用的团队持续交付。经过多年的酝酿，蓝海是詹金斯管道公司从头开始的一个重大项目。我有机会与 CloudBees 和 Jenkins 社区的两位主要贡献者 James Dumay 和 R. Tyler 克罗伊进行了交谈。泰勒是詹金斯管理委员会的成员。

蓝海提供:

*   **CD 管道的复杂可视化**，允许快速直观地了解管道的状态。
*   **Pipeline editor** 通过直观、可视化的流程指导用户创建管道，使管道的创建变得触手可及。
*   **个性化**满足团队每个成员基于角色的需求。
*   **当需要干预和/或出现问题时，精确定位**。蓝海显示了管道中需要注意的地方，促进了异常处理并提高了生产率。
*   **分支和拉取请求的原生集成**在 GitHub 和 Bitbucket 中与他人进行代码协作时，可以最大限度地提高开发人员的工作效率。

因此，对于蓝海，我们有点专注于开发运维实践和持续交付。我们给自己设定了一个目标。如果我们能让人们真正容易地采用连续交付，那会怎么样？蓝海是我们的答案。

我想说的是，这个行业的转变让蓝海变得非常重要——大约 10 年前，当我第一次在北加州开始工作时，持续集成是一件非常重要的事情。我不断地去找那些没有持续集成服务器或者没有实践的公司——除了开发人员没有人知道有持续集成服务器。随着行业转向这种越来越快的节奏“让我们练习连续交付。让我们尽快把产品和价值带给我们的客户，”仅仅让开发人员获得流程中正在发生的事情的反馈已经不再被接受。

所以我认为对蓝海来说重要的是，我想说更广泛的软件行业，持续交付不是开发人员的事情。这是整个团队的事情。这是组织上的事。蓝海是从头开始设计的，以确保那些可能不会每天都编写代码的人，或者他们可能是设计师，或者他们可能是产品经理，或者软件开发过程中某种形式或时尚的利益相关者。他们需要了解正在发生的事情，蓝海将为他们提供一个窗口，让他们了解交付渠道中正在发生的事情。这与 10 年前只为开发人员开发的 Jenkins 相比是一个巨大的转变。
