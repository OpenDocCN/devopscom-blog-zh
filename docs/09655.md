# MLOps Vs. DevOps:有什么区别？

> 原文:[https://devops.com/mlops-vs-devops-whats-the-difference/](https://devops.com/mlops-vs-devops-whats-the-difference/)

机器学习操作(Machine learning operations，简称 MLOps)是机器学习(ML)工程的一个关键方面，它专注于简化和加速将 ML 模型交付到生产以及维护和监控它们的过程。MLOps 涉及不同团队之间的协作，包括数据科学家、开发运维工程师、IT 专家等。

[MLOps 可以帮助组织](https://www.run.ai/guides/machine-learning-operations/) 创建和提高他们的 AI 和机器学习解决方案的质量。采用 MLOps 允许 [机器学习工程师](https://www.run.ai/guides/machine-learning-engineering/) 和数据科学家合作，通过实施持续集成和持续部署(CI/CD)实践来提高模型性能。它通过整合适当的监控、治理和验证 ML 模型，加速了 ML 模型的开发过程。

## DevOps 是什么？

DevOps 结合了开发和运营的概念，描述了一种协作方法来执行通常与独立的应用程序开发和 IT 运营团队相关的任务。从最广泛的意义上来说，DevOps 是一种哲学，它鼓励组织内这些(以及其他)团队之间改进的沟通和合作。

从最狭隘的意义上来说，DevOps 指的是采用支持迭代应用程序开发、自动化和可编程基础设施的部署和维护的实践。它还包括工作场所文化的变化，比如在开发人员、系统管理员和其他团队成员之间建立信任和联系。DevOps 将技术与业务目标结合起来，可以转变软件交付链、工作职能、服务、工具和最佳实践。

## MLOps 与 DevOps:主要区别

以下是 [MLOps](https://devops.com/?s=MLOps) 和传统 DevOps 的一些主要区别。

### 发展

开发的概念指的是每个模型中不同的东西，具有稍微不同的 CI/CD 管道。

**DevOps:**

*   通常，代码会创建一个接口或应用程序。
*   代码在被部署和用一组检查测试之前被包装到可执行文件或工件中。
*   理想情况下，这种自动化循环将持续到最终产品准备就绪。

mlops:

*   代码使团队能够建立或训练机器学习模型。
*   输出工件包括可以接收数据输入以生成推论的序列化文件。
*   验证包括根据测试数据检查训练模型的性能。
*   这个循环还应该继续，直到模型达到指定的性能阈值。

### 版本控制

**DevOps:**

*   版本控制通常只跟踪代码和工件的变更。
*   需要跟踪的指标很少。

**MLOps:**

*   MLOps 管道通常需要跟踪更多因素。构建和训练一个 ML 模型涉及到一个迭代的实验周期，需要跟踪每个实验的各种度量和组件(这对以后的审计是必不可少的)。
*   要跟踪的其他组件包括训练数据集、模型构建代码和模型工件。
*   度量包括超参数和模型性能指标，如错误率。

### 复用性

**DevOps:**

*   DevOps 管道专注于可重复的流程。
*   团队可以混合和匹配流程，而无需遵循特定的工作流。

**MLOps:**

*   MLOps 管道重复应用相同的工作流。跨项目的公共框架有助于提高一致性，并允许团队更快地进步，因为他们从熟悉的过程开始。
*   项目模板提供了结构，使定制能够解决每个用例的独特需求。
*   使用集中式数据管理来整合组织的数据，以加快发现和培训流程。常见的集中化方法包括单一来源的事实和数据仓库。T3】

### 连续监视

对于 DevOps 和 MLOps 来说，监控都是必不可少的，但原因略有不同。

**DevOps:**

*   站点可靠性工程(SRE)在过去几年中一直是一个趋势，强调从开发到生产部署都需要监控软件。
*   软件不会像 ML 模型那样退化。

mlops:

*   机器学习模型可能会很快退化，需要不断的监控和更新。
*   生产环境中的条件会影响模型的准确性。部署到生产环境后，该模型开始根据现实世界中的新数据生成预测。这些数据不断变化和调整，降低了模型性能。
*   MLOps 通过整合程序来促进持续监控和模型再培训，从而确保算法保持生产就绪状态。

### 基础设施

DevOps 和 MLOps 都非常依赖云技术，但有不同的运营要求。

**DevOps** 依赖于基础设施，例如:

*   基础设施代码(IaC)
*   构建服务器
*   CI/CD 自动化工具

**MLOps** 依赖于基础设施，例如:

*   深度学习和机器学习框架
*   大型数据集的云存储
*   [深度学习的 GPU](https://www.run.ai/guides/gpu-deep-learning/)和计算密集型的 ML 模型

## DevOps 和 MLOps 趋势

以下是推动 DevOps 和 MLOps 发展的一些主要趋势。

### GitOps

DevOps 工作流的新发展， [GitOps 是用于控制和自动化基础架构的新范例](https://codefresh.io/learn/gitops/) 。面向 Kubernetes 的范例使开发人员和运营团队能够管理 Kubernetes 集群，并使用 Git 交付容器化的应用程序。为运营和开发团队实现 Git 工作流允许开发人员利用 Git 拉请求来管理软件部署和基础设施。

GitOps 整合了现有的开发工具，通过 CI/CD 管理云原生和基于集群的应用。它使用 Git 存储库作为唯一的事实来源，自动部署、监控和维护云原生应用程序。

GitOps 是在 Kubernetes 中实现和维护集群的一种方式。持续交付和部署允许开发人员通过增量发布更快地构建、测试和部署软件。Kubernetes 持续集成和运行时管道必须能够读写文件，更新容器库，并从 Git 加载容器。GitOps 通过版本控制、实时监控和配置变更警报来帮助企业管理其基础设施。

### 综合数据

合成数据是任何人工生成的信息，而不是从真实事件中收集的信息。算法生成合成数据，用作运营和生产测试数据集的替代品。合成数据集对于验证数学模型和训练机器学习模型也很有用。

合成数据的优势包括:

*   最大限度地减少与使用敏感和受管制数据相关的限制。
*   根据现实数据中没有的特定要求和条件定制数据。
*   为开发运维团队生成测试软件质量和性能的数据。

### 无代码机器学习和人工智能

机器学习通常涉及计算机代码来设置和处理模型训练，但情况并非总是如此。无代码机器学习是一种编程方法，它消除了 ML 应用程序经历耗时过程的需要。

无代码 ML 消除了专家开发系统软件的需要。它的部署和实现也更简单、更便宜。在机器学习过程中使用拖放输入可以通过以下方式简化训练工作:

*   评估结果。
*   拖放训练数据。
*   创建预测报告。
*   使用纯文本查询。

无代码 ML 使开发人员能够轻松访问机器学习应用程序，但它不能替代高级的细致入微的项目。这种方法适合缺乏资金来维持内部数据科学团队的小型企业。

### TinyML

TinyML 是机器学习和人工智能模型开发的新方法。它包括在有硬件限制的设备上运行模型，例如为智能汽车、冰箱和电表供电的微控制器。这种策略最适合这些用例，因为它加快了算法的速度，不需要数据在服务器之间来回传输。这在大型服务器上尤其重要，可以加速整个 ML 开发过程。

在 IoT 边缘设备上运行 TinyML 程序有很多好处:

*   能耗更低。
*   减少延迟。
*   用户隐私保证。
*   降低带宽要求。

使用 TinyML 提供了更大的私密性，因为计算过程完全是本地的。它消耗更少的功率和带宽，从而降低延迟，因为它不需要将数据发送到中央位置进行处理。利用这一创新的行业包括农业和医疗保健。他们通常使用嵌入了 TinyML 算法的物联网设备，利用收集的数据来监控和预测现实世界的事件。

## 结论

在本文中，我介绍了 MLOps 和 DevOps 之间的主要区别:

*   **开发** —DevOps pipelines 专注于开发软件产品的新版本，而 MLOps 专注于提供工作机器学习模型。
*   **版本控制** —DevOps 主要关注跟踪二进制文件和软件工件，而 MLOps 则跟踪额外的因素，如超参数和模型性能。
*   **可重用性** —DevOps 和 MLOps 都努力创建可重用的流程和管道，但是使用不同的策略来实现可重复性。T3】
*   **持续监控**—监控对 DevOps 很重要，但对 MLOps 更重要，因为模型和数据漂移会降低模型的性能。

最后，我介绍了将在不久的将来改变 DevOps 和 MLOps 的几个关键趋势。我希望这将有助于您在新的激动人心的开发生态系统中找到自己的位置。
