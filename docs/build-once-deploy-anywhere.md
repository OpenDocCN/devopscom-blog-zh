# 一次构建，随处部署

> 原文：<https://devops.com/build-once-deploy-anywhere/>

在游戏的后期，我不得不写下这些，这对我来说似乎令人震惊，但你们中有很多人还没有到那一步——所以让我们尝试一种不同的方法。

DevOps 的很大一部分是减少冗余。我们已经削减了脂肪，直到它受伤，然后重新调整和削减更多。这里有一个关于倦怠的博客，但是我们会把它留到下一天。关键是，我们不会一遍又一遍地做同样的事情，除非有可量化的好处——比如重建和测试新的迭代，或者旋转基础映像来启动新的项目。

我们坚持认为，我们的应用程序应该在尽可能标准化的环境中运行，使用来自存储库的映像并安装标准化的软件。

然后，我们立即将其放入一个独特的环境中，在那里部署是完全不同的。是的，我说的是哪个云或者哪个数据中心；如果是通过数据中心部署，那么是硬件、虚拟机还是容器？标准化重要部分并将影响较小的部分留给团队来决定的工作在 DevOps 的开发方面令人震惊(对于新来的人，我称之为 *Dev* Ops)，但在开发 *Ops* 方面却远没有这么有效。即使是自称纯云的公司，一般也有两个或两个以上的云厂商，每个厂商做的事情都不一样。

所以，说了这么多，我可以说我应该说的话，而不是这么多年后不得不说的话:

使[容器](https://containerjournal.com)标准化。我不在乎您使用哪种风格和管理工具，但是要添加可移植性层。它将保护您的首选托管环境(无论是云、本地还是托管)在未来变得不友好或过于昂贵，或者干脆不复存在。它还将标准化您的 DevOps 流程的其余部分。是的，你首选的云供应商提供 Kubernetes，不，这还不够。今天的 Kubernetes 产品就像 20 世纪 90 年代的微软。拥抱并延伸，以建立更高的锁定屏障。你*确实*从我关注的云供应商那里获得了一些很棒的功能，但是流程变得不标准。最好将特定于平台的工作限制在“启动服务器”上，让其他所有东西在您部署的任何地方都可以重用。这里的好处是深远的；当您从云迁移到云或从云迁移到数据中心或从数据中心迁移到云时，会有不同的配置和安全问题，但开发*运营*的核心将保持标准化。有一些工具可以帮助基线云安全和风险管理，因此除了使用 Kubernetes 作为可移植性平台之外，这些工具还可以减轻当前工作的繁重程度，并大幅提高部署位置之间的可移植性。

你缺人手。不需要火箭科学家来预测这种情况不会改变。采用那些会减少你工作量的东西——比如 Kubernetes 上的开发*操作*标准化(或者 T2 的 open shift 或者其他什么)作为部署平台——并且继续摇摆。你不能挥挥手就获得更多的员工或时间，但是你*可以*将容器作为唯一的部署选项来节省时间。利用这些时间解决公司的其他问题，因为这是我们的工作，为企业解决问题。保持微笑，因为你在摇摆。如果你不是，他们会找一个可以的人。
