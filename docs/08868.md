# 有些改变是不好的

> 原文:[https://devops.com/some-change-is-bad/](https://devops.com/some-change-is-bad/)

在年底的某个时候，我开始考虑我们，作为从业者，已经经历了…几年，甚至几十年的巨大变化。我开始试着想办法让你们不要淹没在变化中。然后我写了一些东西，希望能帮助你们所有人关注好的改变，并不断推动你们的环境向前发展。

今年我要说的是“考虑改变的原因。”我将尝试寻找不需要提及具体产品或供应商的例子，因此如果这些例子读起来有点复杂，我很抱歉，因为我试图在不进行指责的情况下说明这一点。

市场或营销人员希望你考虑改变有三个基本原因。当然，有各种各样的情况，其中有一点这种原因，也有一点那种原因，但大多数要求我们改变做事方式的要求都可以归为“其中一种”。它们是:

1.  这是一个伟大的想法，对于某些细分市场来说，这确实是一个游戏规则的改变者。
2.  谁说了算的转变。
3.  锁定客户或扩大收入。

所有这些都发生在供应商/项目层面，所有这些都发生在市场层面。

第一个最好的例子是已经出现的任何一波[服务器虚拟化](https://devops.com/?s=server+virtualization)。从虚拟机到容器，每一个都解决了问题，改变了我们做生意的方式。值得注意的是，我*只*在这里谈论服务器虚拟化的努力。大多数其他虚拟化尝试都不符合“改变游戏规则的伟大想法”这一类别，尽管有些有用性有限。就我个人而言，我认为新版本的存储虚拟化将属于游戏规则改变者类别，但我正在等待看到它的使用。

第二点是要小心的，当知名的 OSS 项目被分叉时，这一点最明显。它很少与正在实现的技术有关(毕竟，一个分叉是原始的副本；起初，这根本不是技术问题，而是谁在发号施令的问题。每个公共云供应商都在实施他们自己版本的容器管理(尽管在我看来，这个例子更多的是关于锁定)。

第三点是多管齐下。如果你的供应商从向你出售硬件或软件转向向你出售服务，他们想要的是经常性收入，他们希望你与该服务的联系如此之深，以至于你无法离开。备受诟病的 20 世纪 90 年代口号“拥抱和扩展”就是这种变化的典型代表。在某个地方可能会有附加值，有时增加的功能会使迁移变得不需要动脑筋。不过，通常情况下，这种变化只是对供应商有利，而不是对他们的客户有利。正如大多数主流 Linux 发行版所显示的那样，开源软件也不能幸免于此，因为大多数发行版都让 OSS 用户成为他们付费产品事实上的 beta 测试者。

这并不是说你面前的变化是不好的——即使它最初产生的原因是不好的。它*是*提醒你，在持续不断的变化之后，我们比我们应该的更加开放，你应该记住*关于新变化的唯一*重要的事情——**它们让我们的环境更安全、更快速、更敏捷吗？其他一切都只是噪音。是的，所有的酷孩子都在转向用[insert flavor du jour]编码，但是它给你的组织带来了什么新的东西吗？是的，似乎每个人都在使用[insert current CI/CD flavor]，但它是否解决了您当前工具链的弱点？**

你们都在*钉它*。坚持做下去。不要害怕变化，但要确保你知道新的 OSS 数据库系统是否真的是新的，或者只是在稍微不同的名字下安装相同的代码。因为有人不喜欢新主人。
