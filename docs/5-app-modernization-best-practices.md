# 5 个应用现代化最佳实践

> 原文：<https://devops.com/5-app-modernization-best-practices/>

如果一个组织不能从其应用程序中挖掘出最佳价值，那么它的数字化转型将是无效的。这几乎是一个普遍的担忧，因为大多数企业仍然拥有主要在本地专用环境中的传统 IT 环境中工作的核心应用程序，这降低了 IT 和业务性能，阻碍了创新，并增加了对[风险](https://securityboulevard.com/?s=risk)的脆弱性。通过放弃灵活性、适应性和竞争力，企业正在为他们的[技术债务](https://martinfowler.com/bliki/TechnicalDebt.html)付出代价。这在正常时期已经够糟了。在大流行后的情况下，这可能会产生灾难性的后果。

收购一些未与现有系统集成的云原生应用程序价值有限，充其量只能提供部分答案。另一方面，彻底改变遗留应用程序环境通常是不可能的。那么，唯一合理的解决方案就是应用程序现代化。但是企业从哪里开始呢？

## **拾取并移动**

一个很好的起点是检查现有的应用程序。了解每个应用程序的功能、性能和问题，以及它可以从现代化中获得什么，将有助于建立一个可供选择的“通用集”。可以根据技术因素(例如，当架构增加成本、漏洞或复杂性时)以及业务考虑因素(例如，折衷的敏捷性或价值潜力)来确定现代化候选对象的优先级。最后，企业需要权衡潜在的好处，尤其是 ROI，以及现代化的难度。

通常，企业核心中最大、最重要的应用程序是最难现代化的。但是没有他们的参与，转变通常是不完整的。企业可以将这些单一的应用程序及其庞大的数据封装在一个 API 层中，将它们连接到正在迁移的所有其他应用程序以及部署的任何新应用程序，而不是将它们提升和转移到云中。使用 API 来公开遗留核心系统(如 ERP)的功能和数据，允许它被广泛使用、重用和修改，以及用于构建新的解决方案和服务。

## **拆分成微服务**

重构是使遗留应用程序现代化的一种有用方式，包括将它们分解成微服务，然后连接到云基础设施，如 Docker 或 Kubernetes。 这些松散耦合的微服务可以被单独‘调用’和扩展。 由于重构需要对代码进行大量的重写和重组，所以最好通过多次迭代来逐步完成，以确保一切正常运行。

## **提供全栈可见性**

即使应用程序被现代化为云上的微服务，它也需要与企业中的其他遗留系统进行通信。问题是，现在有更多的服务和环境需要监控和保护，更不用说单点故障了，这不仅增加了复杂性，还剥夺了企业[全栈可见性](https://devops.com/?s=full-stack+visibility)。但是跨环境(从边缘到核心(内部)到云)的可见性是成功的必要条件。这是通过确保现代化的应用程序适合组织的数据结构来实现的。企业需要 从端到端捕获整个堆栈中的数据，并对其进行实时处理，以便不间断地查看其应用程序的性能和可用性  。

## **实现流程的现代化**

如果没有满足周围的要求(例如，规划和满足质量标准)，一个完美执行的应用程序现代化计划可能无法交付结果。遗留流程可能会拖累现代化工作。因此，有必要通过重新架构流程来解决这个问题，以适应更加自动化的设置，方法是为代码开发人员配备理解业务需求、采用最佳实践并以成功的数字原生者为榜样的团队成员。

## **跟踪绩效指标**

回到能见度的问题上来。企业需要建立关键指标来跟踪，以便他们可以监控应用程序的性能，直到代码级别。不仅仅是应用程序，从云到数据库再到网络的整个基础架构都需要持续跟踪，以便在问题爆发前发现问题。应用程序的性能需要与相应的现代化前的值进行比较，以了解优势并确定需要优化的领域。

## **平衡和行动**

应用程序现代化方法的范围从重新托管和重构到重新架构和替换，这些方法越来越昂贵和有风险，但也越来越有影响力。企业需要基于对努力、成本、风险和结果的考虑来决定什么对他们最有利。虽然有些企业可能比其他企业走得慢，但最终他们将不得不更新他们的所有应用程序，以具有竞争力并以最有效的方式满足他们客户的需求。
