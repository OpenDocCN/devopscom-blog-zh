# 将 DevOps 融入您的公司

> 原文:[https://devops.com/embracing-devops-into-your-company/](https://devops.com/embracing-devops-into-your-company/)

一些公司正在 DevOps 开始新的冒险。当组织开始考虑实现 DevOps 的可能性时，他们需要考虑它能解决什么问题以及它将在他们的组织中扮演什么角色。

## **公司文化和发展目标**

DevOps 是企业文化和技术的转变，提供了从想法到生产的无缝路径。因为它涉及开发、运营、it 和业务的其余部分，许多组织已经看到了他们的组织孤岛的崩溃。

公司文化和价值观是采用 DevOps 的核心，领导层鼓励更多的合作。随着角色和职责的转变，对这些变化可能会影响工作的担忧可能会导致员工之间的一些内部摩擦。在整个过程中，沟通正在进行的和即将发生的变化对于保持员工的积极性至关重要。因此，如果一个组织具有开放的沟通风格和实验新技术的理念，那么采用 DevOps 将会更容易。

## **德沃普斯的安全** 

安全性通过降低早期风险的可能性，为开发过程带来更大的灵活性，从而使开发运维受益。考虑到安全漏洞的平均成本(2018 年迄今为止在 [、220 万美元到 690 万美元](https://securityintelligence.com/ponemon-cost-of-a-data-breach-2018/) 之间)及其对客户和品牌声誉的影响，拥有一个 DevSecOps 战略至关重要。

DevOps 方法强调安全和速度的重要性。虽然传统模型为手动测试留出了时间，但在 DevOps 中却没有这样的时间，在 devo PS 中，持续部署是常态。团队必须在开发过程中更早地集成安全问题的测试和解决(左移),以防止将来出现更大的问题。

## **入门建议**

对于如何在您的组织中开始使用 DevOps，我们有五条建议:

1.  **从流程开始**:了解 DevOps 流程是否成功的唯一方法是开始。通过让团队合作来构建创新，充分利用这个机会。做好实验和失败的准备。最终，正确的过程将会发展。
2.  **定义指标**。为了使转换生效，您需要理解重要的关键指标。使用这些指标作为起点:
    1.  平均生产时间:新提交的源代码投入生产所需的时间长度。
    2.  部署频率:发布部署到产品中的频率。
    3.  平均交付时间:一个新特性被开发、测试和部署到生产中需要多长时间。
    4.  部署速度:将新版本部署到生产环境中需要多长时间。
    5.  生产故障率:生产故障的频率。
    6.  平均恢复时间(MTTR):从故障中恢复所需的时间。
3.  **将工作流程可视化**:只有付诸行动，你才会真正知道 DevOps 会有多有效。通过创建可视化模型，团队成员将更好地理解他们的工作如何影响整个 DevOps 过程。
4.  **创建连续的流程** : DevOps 需要自动化，而且是大量的自动化。只有通过实施 CI 和 CD 才能实现高速稳健释放。在 CI 中，团队成员经常集成他们的工作以更快地构建软件，而 CD 提供了将自动化测试集成到发布过程中的机会。
5.  **改善沟通**:团队整合后，下一步是确保他们能够相互沟通。首先在业务和工程团队之间建立一种统一的语言，来表达两个团队期望的验收标准。

随着组织采用 DevOps，在成功过渡的过程中，文化将与技术一样重要。确保您的 DevOps 行动手册阐述了您选择的 DevOps 工具和技术将如何与人员和流程协同工作。
