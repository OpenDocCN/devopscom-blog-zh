# 为死者说话:“敏捷”和“单一”真的好吗？

> 原文:[https://devo PS . com/speaking-dead-agile-monolithic-actually-good/](https://devops.com/speaking-dead-agile-monolithic-actually-good/)

如果敏捷是厌倦工作的产物，微服务是我们需要挑战的结果，而 scrums 是一个痴迷的项目经理的结果，那会怎样？难道我们都认为瀑布开发是一只恐龙，它的灭绝是活该，因为我们被告知的，而不是因为现实？我们认为敏捷更好是因为它听起来更好吗？

[迈克尔·德汉](https://www.michaeldehaanllc.com/) ( [@laserllama](https://www.twitter.com/laserllama) )， [Ansible](https://www.ansible.com/) 的创造者，在 [2016 全天 DevOps](https://www.alldaydevops.com/) 会议上发言，该会议拥护 DevOps 的好处，以及与之相关的敏捷和微服务。它本质上是说，“让我们退后一步，评估过去。也许瀑布开发和整体架构实际上是一件好事？”

Michael 是一名软件工程师，他在 IT 自动化工具箱中创建了一个关键工具:Ansible。他的观点值得一看，因为重新评估可以让你不犯错误，并改进前进的过程。

Michael 在他的演讲中首先指出，在这个时代，这是瀑布与敏捷的较量，是整体架构与微服务的较量。假设是，如果您仍然使用瀑布方法开发软件和/或开发单片应用程序，您还不如骑着马车去工作，用您的旋转式电话给同事打电话。

在他的整个演讲中，他问了很多问题来阐述他的观点，从“预先的大设计不好吗？”有了结论，“我不这么认为。”他的观点是，在流程开始时的设计和要求能够提高生产效率。营销和工程师根据需求、能力和预算敲定了需求。有足够的时间来研究和设计解决方案。双方共同努力，提出问题和意见，并记录双方的协议。这提供了不同团队成员的优势以及任务和任务顺序之间的依赖关系的准确视图。也许最重要的是，工程就是工程——工程师有更大的责任和更少的参与。

与他对敏捷的看法相反，预先的大设计被贴上不好或适得其反的标签，工程师被认为是可互换的，范围更不可预测。鼓励不要基于时间进行评估，这通常会导致持续的时间紧张。最后，会议可能会消耗团队成员很大一部分时间，并且经常包括“让人想起幼儿园的游戏，比如五拳或规划扑克。”即使在轻量级敏捷系统中，测试也会受到影响，需求更加模糊，并且没有双方的签字同意。

迈克尔认为，事实上，团队可以有一个中间地带:

*   分享设计
*   有一个容易接近的、有效的产品负责人
*   了解硬需求和软需求
*   将测试驱动开发和测试自动化视为超级重要
*   理解敏捷和迭代开发可能比我们想象的更接近

敏捷和开发运维也经常与微服务和整体应用联系在一起，但是根据 Michael 的说法，“整体应用有什么不好？”

一个示例微服务商店有 150 项服务，每天有数千个 Jenkins 构建，八个 Amazon 区域，包括开发、准备和生产，以及数千个 EC2 实例。这可能导致 Jenkins 配置和安全执行偏差、次优代码共享以及开发人员编写大量定制代码来管理它。

迈克尔将其与巨石柱进行了对比，后者具有:

*   增强了在一台笔记本电脑上进行开发的能力
*   受益于静态类型而非契约式设计的接口
*   生产中非常少量的不同服务器/映像类型
*   比 RPC 快得多的本地函数调用
*   利用率为 5%的实例很少
*   不需要“容器”来解决上述利用率问题
*   大大简化了调试
*   简单的代码共享
*   非常简单的版本升级

他没有主张所有的应用程序都应该是单片的，而是指出，许多导致微服务的问题可以通过改善通信来解决。他认为微服务是解决人的问题的技术方案。

迈克尔总结道:

*   标签掩盖了真实的含义，
*   单片结构更有效，
*   敏捷可能效率较低，
*   微服务试图解决人的问题，但是引入了新的技术和人的问题
*   我们应该试着把标签和事物的真实含义分开。
