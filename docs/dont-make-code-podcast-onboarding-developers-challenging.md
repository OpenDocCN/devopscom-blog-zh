# 不要让我写代码播客:开发人员入职

> 原文：<https://devops.com/dont-make-code-podcast-onboarding-developers-challenging/>

感谢收听“不要让我写代码”,这是一个讨论开发者体验设计和构建面向开发者的产品的独特挑战的播客。《不要让我写代码》由 [Opsee](https://opsee.com/) 的联合创始人兼首席采购官[史蒂夫·博克](https://twitter.com/sboak)和 [Convox](https://convox.com/) 的联合创始人兼首席执行官[大卫·多勒](https://heavybit-blog.squarespace.com/config#)主持。

在这一集的《不要让我写代码》中，Steve 和 David 讨论了入职开发人员面临的一些挑战，以及如何让这种体验变得更好。

这个播客是由 heavy bit(T1)带给你的，这是一个帮助以开发者为中心的公司将他们的产品推向市场的项目。

Steve: 所以，今天我们来谈谈入职，以及这对开发工具公司来说是多么痛苦的事情。对我来说很奇怪的是，在消费者的世界里，我们可以在几秒钟内愉快地获得产品，而在开发工具的世界里，我们只需要几分钟，你知道，5 分钟或 10 分钟感觉很好。

**大卫:**是的，我们正在处理复杂的事情，人们正在将这些复杂的事情整合到他们已经建立的其他事情中，这通常不像“只需输入你的电子邮件地址并点击一个按钮”那么简单你知道，你必须给你的代码添加一些片段，或者添加一些依赖项，或者开始检测一些东西——你可能需要做所有这些事情来开始。

史蒂夫:这些要求有时感觉不合理，我的意思是我无法忍受经历这些，不得不编写代码以装载到产品中。

我记得第一次停止使用 New Relic 时，在开始使用该产品之前，我不得不实际编写代码，这令人沮丧，但却是必需的。这只是一种无限的挫折感，而我们在我们的产品中，我们试图尽可能地消除这种挫折感，但是，这是我们这个世界特有的挫折感。

**David:** 当然，我的意思是，我们正在尝试将应用程序部署到互联网上，你知道，这些应用程序的变化是如此之大，所以我们花了很多时间专注于让它开箱即用，而不必一开始就进行大量的更改。我认为这真的很重要。

Steve: 你看，你和我都在早期公司，我们现在和我们的测试客户打交道，这是一个非常高接触的场景，我们和他们非常密切地互动，一次一个。但是，我们经常不得不处理个人电子邮件地址之类的事情，他们在那里注册公司的产品，但使用他们的个人电子邮件，所以，我花了大约 10 或 15 分钟才发现我正在处理的人是谁。

**大卫:**是的，我的意思是，我们肯定经常看到这种情况，而且实际上不仅仅是个人电子邮件地址，你知道，他们实际上可能会在个人帐户的背景下开始试用我们，或者个人，你知道，也许他们会带来一个个人项目，比如他们的博客，或者类似的东西，开始时，他们想，“是的，这东西会工作吗，在我实际开始在工作中尝试和介绍它之前？”

**Steve:** 是的，你试图在一个环境中展示价值，第一，这不是预期的环境，第二，周围可能有完全不同的需求。例如，你知道，对于我们亚马逊的人来说，我们处理的是像 EC2 Classic 支持这样的个人帐户，这种支持已经被否决了，他们工作中的生产环境不太可能有这种支持，但这是我们作为产品的制造商必须考虑的事情，因为人们可能会如何加入它。

大卫:是的，当然。

**Steve:** 即使我们足够幸运地获得了某人的工作电子邮件地址，我们仍然必须通过整个测试流程，他们将我们放在他们的试运行环境中，让我们试一试。这是一个很大的限制，在这个地方，你的价值会明显减少，但仍然要证明你对客户有价值。

大卫:是的，他们想知道这个工具对我有用吗。

> 不管你的文档有多好，不管你的例子有多好，大多数人都想亲自动手，看看它是如何工作的，看看它的缺点在哪里。对于我们的目标客户来说，任何实验都有点像游戏的名字。

所以我认为为此提供一些支持，让你的产品变得更容易使用是非常重要的。

史蒂夫:是的，真正考虑你将如何在这些你可能受到限制的情况下提供价值是非常重要的。虽然，我不得不说，今天早上，我们有一个人注册了这个产品，把我们放在一个什么都没有的环境里。这是一个有 0 个实例和 0 个组的 AWS 环境，我们是一个对环境中的其他东西进行运行状况检查的产品，没有任何东西需要运行状况检查。那是一次艰难的推销。

大卫:我想，现在他知道你不会注销他的亚马逊账户了。

史蒂夫:是的，还有安全检查，对，就像，一半的战斗是说服人们，一，你不会打破，二，你不会打破他们的东西。

大卫:是的，我想，这肯定是人们首先尝试使用个人账户的另一个原因，对吗？就像是，‘我不会把这个工具安装到我一无所知的工作中的生产账户上。’

**史蒂夫:**是的，看这个真的很有趣，因为我们通过视频聊天亲自做了很多工作，甚至只是看着他们的笔记本电脑屏幕，我们在和非常聪明的人打交道。他们观察，他们可以看到我们的入职流程对他们的环境产生了什么影响，他们去审计一切，他们检查偏好的每一个变化，他们确切地知道发生了什么。他们检查我们的一切，你知道，这是令人担忧的，但也是一种很酷的事情，看看他们在做什么。

大卫:我们基本上得到了同样的东西，人们试图把我们安装到他们实际的亚马逊账户，这是一个相当艰难的销售，我们很幸运地让早期用户要么创建一个新的亚马逊账户，然后在那里安装所有东西。或者我们实际上，我们将创建一大堆，我们预先创建一大堆亚马逊账户，然后把它们分发给人们，让他们开始尝试。我认为，我们能做的任何事情都是非常有价值的，可以减少他们为了进入实验室进行实验而不得不做的事情。

史蒂夫:对，这是个很好的观点。我们现在的公司还没有做的事情，但我们过去已经做了，就是创建这些真实的演示环境，在那里，你知道，即使没有证书，人们也可以在网上试用产品，并在没有注册的情况下感受一下它会是什么样子，这在过去对我来说是一个非常有用的工具。

**David:** 是的，用开发者工具来做这件事是很困难的，因为，你知道，在很多情况下你都在谈论复杂的集成，但是，是的，我认为当你做对了的时候，这是非常有价值的。如果我只需点击几下鼠标，就能真正使用该产品，甚至再点击几下鼠标，就能使用我一直在使用的产品，在这一过程中你可以减少的任何摩擦都会有所帮助。

**Steve:** 是的，这实际上提出了一个有趣的观点，即在过去几年中，企业软件公司从过去的销售方式转变为现在的销售方式，这就是透明度，现在，像定价这样的事情，以及对产品功能的真实感受。这听起来很基本，但是，这些是你在 5 年或 10 年前想不到的。销售 B-to-B 产品——你会有一个联系销售人员的按钮，你可能甚至不知道该产品在主页上做了什么。

大卫:是的，完全正确，我的意思是，你今天仍然会看到很多这样的情况，作为一个软件购买者，遇到这样的情况是非常令人沮丧的。我们努力记住这一点，从另一方面来说，知道它的成本是多少，到确切地知道它将为你提供的价值，以及它将如何让你的生活变得更好，这是非常重要的。

史蒂夫:是的，至少设计你的市场营销是显而易见的，这样当人们访问你的网站时，他们就知道产品是做什么的。我无法想象一家消费公司会处理其中的一些问题，比如多种环境的问题，我们让人们注册个人账户，或者注册根本不是产品预期用途的分期环境，但为了赢得这些客户，我们必须展示它们的价值。

大卫:是的，这很难，我的意思是，那些环境基本上也完全不同，对吗？就像，我的个人 id 帐户可能与我公司的没有任何共同之处，你知道，这就像有人想测试你的 iPhone 应用程序，首先在家里安装在他们的 Android 手机上。它可以是完全不同的。

**Steve:** 是的，实际上我们前几天讨论过有一个“清理”按钮，因为我们知道很多人都是以这种方式加入的，首先将我们放在一个暂存环境或其他环境中，他们将不得不在转移到另一个环境之前停用一些东西并删除他们在第一个环境中创建的一些东西。因此，作为我们入职培训的一部分，我们也在努力让他们的流程变得更简单。这看起来很可笑，但是说起来很有趣。

**David:** 是的，这几乎值得一试，是否有某种方法可以通过权限集或其他方式，以可信的方式将它们慢慢引入生产环境，而不是让它们先经历这个实验阶段。

也许如果它被很好的引导，它可能会成功。所以，你提到了在一对一的基础上，以一种非常高接触的方式与客户打交道，这是我们已经取得了很多成功的事情，我们实际上认为这非常重要，你知道。我们有一个公共的 slack 频道，我们邀请人们进来，我们一直与人们交谈，他们问关于产品的问题，你知道，如果他们有问题，或者他们，你知道，甚至是关于亚马逊的问题，甚至是与我们没有直接关系的事情，并就这些事情展开对话。

拥有这种实时性是非常有价值的，你知道，我可以通过 ping 某人来了解我们的客户，了解他们在构建什么，他们有什么问题，这真的很有帮助。

**史蒂夫:**是的，不仅我们可以与他们实时互动，而且我们有一种接触他们的方式，我认为，作为一家早期公司，我们面临的一个问题是，人们正在尝试产品，他们不一定参与到产品中，他们可能不会告诉我们这些。

因此，我们有各种可能的方式联系他们，但也是我们入职流程的一部分，在此过程中积极联系他们，确保他们获得价值，查看他们是否发现了问题，并尽我们所能确保他们开心和参与。

**大卫:**是的，我认为肯定是这样，我的意思是要主动接触，保持高参与度，并且不断主动接触，就像，你几乎总是要征求反馈。我的意思是，你会得到一些非常棒的客户，他们会过来告诉你你的产品的所有问题，这些都很好，但是，对大多数人来说，你必须征求他们的反馈。我们在入职过程中所做的许多最好的学习，以及找到摩擦所在，只是与某人交谈，并看到他们问的问题，你知道，有时他们会像这样，完全错了，或者甚至不在你期望他们问的问题的范围内，你知道，你期望他们问，“某个东西是如何工作的”，他们会问，“这个东西到底是做什么的？”

史蒂夫:是的，我认为你提出了两个非常有趣的观点:一个是，当我刚接触这一领域时，无论是初创公司还是设计技术产品，我过去都很反感这种反馈。你竟敢批评我的作品。但是，正如你所说的，

> 最直言不讳的顾客是最好的顾客。那些将要告诉你每一个 bug 的人，我现在很珍惜。这就是我们在测试版客户中寻找的。

大卫:两个原因:第一，它展示了所有你没见过的东西，第二，这些客户实际上是你让他们成功的最大机会。如果他们和你接触得足够多，告诉你出了什么问题，他们非常希望你能帮助他们一点点，他们就会变得很棒。

**Steve:** 是的，完全正确，如果没有其他原因，如果他们抱怨，这意味着他们足够关心抱怨，就像我们之前谈到的那样，部分原因是我们的入职流程往往会询问很多客户，因为这可能会花费大量时间，所以我们真的需要与客户接触，并找出我们如何帮助他们度过难关，以及我们如何让他们更容易，因此，由于他们的负担， 对我们来说，这是一件大事，我们不仅要学习如何做得更好，还要尽我们所能帮助他们帮助客户度过难关。

**David:** 我们看到的一件有趣的事情是，在我们的入职流程中，不仅有真正的技术人员。我们有一些非常技术化的开发人员，他们了解正在发生的事情的基本原理，他们有一系列的问题。

我们还有其他人，他们可能是开发经理，或者，在很多情况下，我有一个运营人员，他现在擅离职守，我无法联系到他，我只是需要帮助，当他们来找我们谈话时，他们需要的是完全不同的东西。能够与所有这些团体交谈，并了解他们在寻找什么，这真的很有启发性，因为这非常不同。

Steve: 是的，在与使用该产品的不同类型的人交谈时，开发团队的组建方式似乎发生了转变。因此，我们有时与开发人员打交道，有时与运营人员打交道，有时与开发人员打交道，这就好像我们必须先了解组织的一些情况，他们的团队是如何建立的，他们如何部署基础架构，以便为我们正在进行的对话提供基础。知识水平——是的，就像你说的，可能是经理中的任何人，我甚至有几个产品人员联系过我。所以，是的，仅仅是建立团队是如何建立的，对于弄清楚如何与我们的客户沟通就非常重要。

**大卫:**是的，了解谁在这类事情上做决定，了解新技术是如何在他们的公司部署的，你知道，这对每个人来说都是不同的。就像，有些人非常狂热，他们只是接受新的东西，并一直为之疯狂，而有些公司非常保守，为他们推出新的闪亮的东西是非常不同的。

**Steve:** 我认为，我们之前讨论过，广度与深度，开源与 SaaS 的对比，所以我认为，Convox 的案例非常有趣，因为你既有开源产品，又有 SaaS 产品。你能谈谈设计这两个产品的对比和优先顺序吗？

大卫:当然，是的，我的意思是，从最开始说起。就像你提到的，我们有开源软件，我们有 SaaS，我们的 SaaS 产品不是开源的，所以，我们在这些项目上的工作方式完全不同，即使是开源软件，我们也是公开工作的。在我们做出改变之前，我们与社区进行了大量的讨论，你知道，拉请求，并试图让许多人参与到改变中来，并确保我们考虑了各种各样的观点，这将对广泛的人群有用。

然后，对于 SaaS 层，我们基本上是说，“好吧，我们知道有这种固执己见，这就是你应该如何使用它，这就是我们要编纂的快乐之路，你的方式，简单的，几次点击的方式进入产品，使用我们已经建立的所有这些疯狂的技术。”

史蒂夫:是的，这种对比真的很有趣，因为开源工具从定义上来说必须吸引广泛的受众，你希望尽可能多的人采用它们并为它们做出贡献。我认为他们如何发展吸引力的很大一部分是你得到一个愿意提供帮助的开发人员的参与社区，而 SaaS 似乎恰恰相反，一个真正常见的创业口号是:“让少数客户喜欢你，而不是很多人喜欢你，”因此，为更少的人提供优秀的体验对于发展 SaaS 业务来说似乎非常重要。

在 Opsee，我们也在处理这个问题，我们只是 SaaS，我们没有开源组件，所以我们真的试图专注于一个狭窄的用例集，然后尽可能地使它们完善和完美，我们正在与开源工具竞争。所以，有时我们会看到这样的反应，“哦，当我可以免费获得开源软件时，我为什么要为你的工具付费？”但是定制它所需要的开发时间，以及这些工具的不同性质，都是一个来回。

大卫:运行它，你知道，它可能是免费的，但必须有人运行它，必须有人为它写一个页面，很容易忽略这些东西。
