# 连续开发周期中缺失的环节

> 原文：<https://devops.com/the-missing-link-in-continuous-devops-cycles/>

无论哪个行业，快速、可靠和安全地交付软件都是技术变革和组织绩效的核心。作为整体技术转型的一部分，许多公司正在认识到将 DevOps 方法应用于软件开发之外的领域的好处，例如一般产品开发、IT 扩展和扩展以及新的业务流程。然而，一些组织——甚至软件开发团队——可能缺少产品开发生命周期中的一个关键步骤。

## 缺失的 DevOps 链接

DevOps 方法并不新鲜。事实上，组织在软件开发之外的计划中采用 DevOps 实践的部分原因是因为它在开发和部署新的过程和特性时经过测试和证明的成功。DevOps 过程中涉及的确切步骤可以由理性的人来讨论，但有三个普遍同意的因素:[持续集成](https://devops.com/?s=continuous+integration) (CI)、持续部署(CD)和持续发布(CR)。

***持续集成***

CI 指的是定期用新特性和代码不断更新产品的策略。更新的频率从每周到每月都有所不同，但是策略是相同的:在问题滚雪球并影响未来开发之前尽早发现问题。

***连续交货***

CD 与持续集成相关，但是集中在交付阶段。CD 在新功能和开发更新部署到生产环境时执行。CD 最关键的因素是对产品最终环境的一致管理。为了在最终用户环境中保持产品功能的一致性，必须定期进行部署和测试，以确保产品更新和新功能不会影响性能。

***连续释放***

CR 是产品开发生命周期重新开始前的最后一步。CR 是指不断向最终用户发布新产品更新、特性和功能的实践。CR 的棘手之处在于，您必须完美地执行 CI 和 CD，以确保最终用户不会遇到由更新引起的任何错误或问题。

但是少了什么呢？答案在于不断探索(CE)。

## 为什么 CE 对产品开发的成功至关重要

CI、CD 和 CR 是经常讨论的产品开发策略，但是它们没有告诉组织从哪里开始 DevOps 生命周期。要开始开发新产品或新功能，组织必须首先确定他们到底在为谁构建什么。正如我经常对我的产品团队说的，你需要在做一顿大餐之前准备好厨房，但是在你开始做错误的事情之前，你需要知道你在做什么饭。

CE 指的是对公司市场、目标受众和客户的持续探索，以了解他们的需求和偏好。这可能听起来像是营销部门的工作，而不是产品开发人员的工作，但是这两者应该以协调的方式一起工作，以帮助实现两个团队的目标。例如，产品开发团队不知道他们的客户喜欢哪些新产品功能，讨厌哪些功能，或者真正需要哪些功能。另一方面，如果营销团队不了解新功能的来龙去脉以及它们如何为客户服务，他们就不知道如何营销产品。

即使组织中不开发产品的部门也可以从 CE 中受益。例如，如果一个组织正在寻求内部软件的云迁移，他们可能会聘请专家顾问或供应商来帮助他们实现云愿景，而不是开发自己的云环境。

然而，简单地将所有内部系统和数据一次性迁移到云中可能会是一场灾难。如果您将系统和流程迁移到云中，而它们突然停止工作，该怎么办？除非您按照 CI、CD、CR 和 ce 策略循序渐进地执行迁移，否则您不知道具体是什么导致了云中的系统错误。

同样，通过在内部执行 CE，组织可以确定哪些流程和内部系统将从云功能中受益最多，并相应地执行迁移。这不仅可以根据业务需求确定技术转型计划的优先级，还可以让业务部门知道在云环境中可能不再需要哪些内部系统。

许多有兴趣通过利用 DevOps 策略改善[客户体验](https://www.hotjar.com/customer-experience/) (CX)和内部流程的组织将从利用 ce 中受益匪浅。产品和业务流程的设计应满足客户的需求并推动底线收入。因此，满足客户需求的第一步是理解这些需求，这是显而易见的。没有 CE 作为产品开发生命周期的基础构件，这是不可能的。
