# 何时、如何以及为何建立开发人员关系团队

> 原文:[https://devo PS . com/when-how-and-why-to-build-a-developer-relations-team/](https://devops.com/when-how-and-why-to-build-a-developer-relations-team/)

开发者的角色正在发生变化。越来越多的时候，他们被正确地视为关键的利益相关者，在我们生活的这个以产品为主导的世界里，他们帮助设定公司的方向。对于许多技术供应商来说，赢得开发人员的信任对他们的短期和长期成功至关重要，但这也带来了挑战。开发商有太多的工作要做，传统的营销策略无法吸引他们的注意力。需要一种新的、更可信的方法。这导致了开发者关系(DevRel)的出现，这个术语对不同的人有不同的含义。

我认为 DevRel 的角色是负责在技术公司、其产品和外部开发者之间建立良好的相互关系。成熟的 DevRel 程序通常位于营销、工程和产品团队的交汇处，因为在这些团队中通常都有开发人员。因此，DevRel 团队由介于开发人员世界和营销世界之间的一群独特的人组成。然而，DevRel 不是一个放之四海而皆准的想法。对于早期创业公司，DevRel 团队的构成往往更加有机。

### 早期公司应该在什么时候建立一个 DevRel 团队？

对于许多初创公司来说，过早创建 DevRel 团队可能是一个错误。对于一家初创公司来说，从一开始就想投资 DevRel 可能是令人兴奋的，但事实是，过早组建这个团队(甚至在知道产品是否适合市场之前)可能是浪费金钱。开发人员记性很好，给他们带来有缺陷或未完成的产品可能会对你的品牌造成长期损害。

在投资 DevRel 之前，你需要首先证明你在市场中的价值，并经历增长。这样，你可以确保你的假设是正确的，你的产品有被证明的价值。一旦你证明了你的假设，并且彻底测试了你的产品，下一步就是为你想要创建的 DevRel 团队建立短期和长期目标。了解你的市场，准备好执行计划。

### 早期的 DevRel 团队是什么样子的？

例如，在 Algolia 的早期阶段，我们的 DevRel 团队甚至不是正式的 DevRel 团队。它融合了 DevRel 和客户支持。我们的首要任务是创造尽可能好的开发者体验(DX)。作为开发商，我们主动与市场互动。我们亲自与注册了这项服务的人接触，为他们提供帮助。

当您的客户是开发人员时，他们会发现与那些不懂代码的人交谈令人沮丧；因此，在处理一对一的支持时，拥抱开发者对开发者的连接是有帮助的。这对于帮助理解客户的痛点和快速迭代产品是非常宝贵的。想象一下，听到一个 bug，并能够近乎实时地修复它！

我给任何开始早期 DevRel 团队的人的建议是，你的第一批布道者应该是你的核心开发者——他们对你的产品了如指掌。然而，要确保你的核心开发者在使用你的产品——“吃你自己的狗粮吧！”这是一个非常好的方法来确保你的核心开发人员与外部开发人员有共鸣。事实上，即使在今天，我们也要求核心开发人员每月提供几天的支持。当我们接近 150 名员工时，我们才创建了一个完全专注的 DevRel 团队。只有到那时，我们才能扩展到专业技能集，以推动更大规模的活动。

### **如何让你自己的开发者成为布道者**

一旦建立了强大的开发者体验，DevRel 应该专注于最大化入站活动，而不是具体的收入。这可以是免费用户、试用、学生兴趣等形式。这是一项长期增长投资。试图将 DevRel 视为一种销售策略，并以影响收入为目标是失败的秘诀。您希望您的 DevRel 团队中的开发人员是可信的。他们不是销售人员；他们不应该成为销售人员，如果他们试图成为销售人员，那将是不真实的。

您的开发人员也不一定需要过度社交或外向来接受 DevRel——这是一个常见的误区。他们可以保持真实的自我，真诚地尽力发挥自己的优势。提供内容就是一个很好的例子。让开发人员完成他们构建的特性的文档。将文档视为产品的核心部分。对很多人来说，文档是事后才想到的，事实上，它是产品最重要的方面之一。然后，文档可以用作其他内容(博客、白皮书等)的起点。

也就是说，如果你的开发者真的对 DevRel 的社交方面感兴趣，为他们提供公开演讲和培训工具。拥有这些技能的开发人员可以在网络研讨会和小组讨论中成为公司的代言人，代表公司参加活动等等。

最好的 DevRel 程序了解开发人员社区，他们使用什么类型的技术，然后去开发人员所在的地方建立联系。DevRel 应该通过展示用例以及问题的解决方案来扩大开发者社区正在做的事情。作为开发者和产品团队之间的粘合剂，它应该给你的用户社区一个声音。

### 为什么公司需要 DevRel？

API 经济使得开发者和 DevRel 的角色比以前更加重要。争夺开发者注意力的竞争已经很激烈了；随着越来越多的组织认识到，在制定战略决策时，开发人员需要一个席位，这个数字只会越来越大。

许多公司将通过传统的营销视角来看待 DevRel 项目，并在非个性化的普遍认知上投入大量预算。这将会失败。相反，拥有可信的、可衡量的 DevRel 项目的公司将是最成功的，这些项目是围绕建立有意义的联系而建立的。接受这种思维的初创公司将比那些困于传统思维方式的大型竞争对手更有优势。
