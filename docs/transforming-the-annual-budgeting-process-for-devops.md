# 转变 DevOps 的年度预算流程

> 原文：<https://devops.com/transforming-the-annual-budgeting-process-for-devops/>

年度预算和计划周期是一个涉及所有这一切的过程。它对如何做出决策、组织团队、委派工作、开始(和/或结束)项目以及分配责任都有影响。但是，很少有人考虑调整和转变开发运维的预算流程。

## 当前预算流程的问题

年度预算过程存在固有的缺陷。它要求高管预测未来(有时提前 12 个月)，据此分配资源并锁定资金。如果这还不够糟糕的话，一旦预算确定下来，他们就会花剩下的时间在今天的需求和昨天的预测之间进行调和。

再加上个人议程和组织政治，你最终会得到一个耗费时间、精力和雄心的系统。对于一个希望变得更加敏捷、反应更加迅速的组织来说，我们当前的预算流程产生了完全相反的效果。

## devops 的预算流程:3 个选项

一些组织正在采取一种截然不同的方法来帮助应对不断变化的需求。

**1。更短的周期和频繁的审查**

向我们的欧洲朋友学习，一些组织正在转向更短的预算和审查周期。每年资助项目的传统结构仍然存在，但计划和分配按季度或滚动进行重新审查。业务部门和 IT 部门都在不断地重新确定资助哪些功能的优先级，并共同协作做出可行/不可行的决策。

这种系统的好处是，它允许团队在项目中动态地分配资金——它迫使团队从最小可行产品的角度来考虑，并且分配中的错误可以很快得到纠正。然而，有两个主要缺点。首先，资金不能在项目之间重新分配，迫使高管们仍然要等待年度周期。其次，如果季度评估流程没有得到简化，可能会造成一种持续预算的氛围，经理们最终会花更多的时间去争取资金，而不是做真正的工作。

**2。产品型资助**

其他人一直在尝试基于产品的融资。在这里，应用程序和相关的资源被分组到产品团队中，并得到总预算的一部分。该预算主要由首席产品负责人(由业务/IT 代表)管理，然后再分配给她的产品负责人。就像在最后一个模型中一样，首席产品负责人与业务部门合作来决定进行/不进行的活动。此外，它还允许首席产品所有者在他们投资组合中的产品和项目之间重新分配资金。产品所有者负责在他们分配的资金范围内执行最少的可行功能集。

这种模式的好处在于简单。首席财务官不必在项目上下赌注，他们只需决定如何在各个产品组之间分配预算。首席产品所有者和产品所有者可以自由分配他们认为合适的份额。因此，预算决策被尽可能地推向第一线，允许更大的灵活性和改进的市场响应时间。

这种方法的一个先决条件是要求它从功能组织模型转移到产品模型——这不是一个容易的任务。至于缺点，产品所有者倾向于投资新的/性感的工作，而忽略日常维护工作。为了解决这个问题，一些组织强迫团队保留一部分预算用于维护和偿还技术债务。

**3。风险投资**

虽然不是我最喜欢的，但风险投资是少数公司正在尝试的一种模式。就像硅谷式的风险资本家一样，执行董事会为最小可行产品的创意提供资金。根据产品的有效性，向团队提供额外的资金。

这种模式的承诺是播种大量的想法，只资助那些可行的想法。好处是更大的透明度和降低金融风险。缺点是大多数公司不像风险投资公司那样。“不成功便成仁”的文化会引发其他问题。当生存能力至关重要时，员工为什么要遵守公司规则、安全政策、最佳实践或基础设施要求等呢？如果这个想法是好的，那么是什么阻止了开发者寻找实际的种子资金并创造一个新的竞争对手呢？

改变开发运维的预算流程并不是一件容易的事情。随着新流程的实施，需要应对重大风险和短期中断。但有一点是肯定的，DevOps 转型最终也必须包括预算转型。
