# 监管审计人员讨厌连续交割吗？

> 原文：<https://devops.com/do-regulatory-auditors-hate-continuous-delivery/>

在这个世界上，卓越的客户参与和其他市场需求依赖于尽早将正确的代码投入生产，持续的软件交付是一种竞争的需要

但是，除非您正确地测试代码，否则您无法将代码投入生产。没有适当的测试数据，你就不能正确地测试代码。

这就是监管者介入的地方。

**解决个人身份信息问题**
生产数据中充斥着个人身份信息(PII)和其他合规敏感内容。这些数据通常在生产环境本身中受到各种安全措施和访问控制的良好保护。但是，将这些数据(或者更准确地说，这些数据的一个样本集)放在测试环境中会使其面临巨大的安全和法规遵从性风险。

首先，测试/QA 工程师不是客户经理或临床医生。因此，从监管的角度来看，有些数据是不允许他们看到的。另一方面，测试/QA 环境通常不像生产环境那样受到保护。更糟糕的是，测试/质量保证通常由承包商执行，甚至可能在海上进行。这造成了不可接受的安全风险，并给监管审计人员带来了巨大的危险。

为了保护 PII，同时实现对新代码的全面有效的测试，公司使用合成数据和其他屏蔽/混淆技术。这是监管者的要求和期望。

又一个路障？
不幸的是，大多数公司都是在非常特殊的基础上生成测试数据的。在每个项目的某个时候，有人意识到他们将需要一些测试数据——并且他们开始创建它。但是，如果没有一个全局机制来确定需要屏蔽哪些内容并实际执行屏蔽，这项工作就会变得不确定且耗时。并非所有需要屏蔽的数据都会被屏蔽。几个开发人员可能会独立决定最好的办法是编写一个一次性的脚本。如果脚本运行得不好，团队可能会祈祷并选择完全放弃屏蔽。

因此，这个博客的标题。当然，监管者实际上并不讨厌持续交付。他们中的大多数人可能甚至不知道这是什么。但是，他们的使命——加强对敏感数据的尽职保护——对于那些在测试数据管理方面没有统一行动的公司来说，可能会成为持续交付的严重障碍。

**解决方案:持续的合规交付**
协调监管审计员的要求和业务对持续交付的需求的唯一方法是采用一种管理良好的企业范围的机制来创建测试数据。这种机制加速了跨每个项目的每个代码提升工作流，同时还提供了一种实施 PII 屏蔽/模糊策略的可审计方法。企业测试数据管理还通过消除在每个项目上重新发明数据屏蔽轮的需要来降低开发运维成本。

所以，不，监管者不讨厌持续投放。但是他们可以——如果公司为了速度而牺牲合规的话。避免这种牺牲的最佳方式是对测试数据采用企业级屏蔽原则。
