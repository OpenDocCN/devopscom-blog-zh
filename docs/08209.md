# 嘘！这是个秘密

> 原文:[https://devops.com/shhhh-its-a-secret/](https://devops.com/shhhh-its-a-secret/)

一两个星期前，在我们一次定期的往返休息中， [Lori](https://www.linkedin.com/in/lmacvittie/) 和我聊起了工作，她和我分享了一篇由 [GitGuardian](https://www.gitguardian.com/) 撰写的[优秀报告](https://blog.gitguardian.com/state-of-secrets-sprawl-2021/)。坦率地说，报告中的信息没有一个让我感到惊讶——Git 充满了公司可能不希望出现的信息，但这几乎是不可能主动跟踪的。这就是 GitGuardian 试图为你做的，但老实说，我还没有深入研究它们，所以*买者自负*。或者，阅读其他人的意见来帮助你决定他们是否是一个好主意——我只是想谈谈这份报告。

关键是你要求你的开发团队越来越快地前进。将他们正在工作的一部分代码放入公共存储库是一种简单的“带回家”或共享工作的方式，尤其是在锁定期间。这也是将关键信息传递给任何搜索它的人的好方法——关于您的基础设施，甚至像 SSL 密钥或凭证这样的关键数据。鉴于目前几乎每个行业都受到网络攻击，我想说，无论你身处哪个行业，都有人在寻找这些信息。

这实际上不是 DevOps 特有的问题——自从我在这个领域工作以来，凭证和代码中的这类东西一直是 IT 的一个问题。在某些时候，任何自动化系统都需要执行 AAA，这意味着所需的数据存储在某个地方。已经尝试了多种方法来从技术上解决该问题，但是没有一种方法是灵丹妙药。还没有。最终，有人会想出办法让它工作；然后，一个全新的攻击载体将会到来，很可能是针对用来保护这些数据的工具。

这是一个人事/政策问题，敏捷和开发运维的速度加剧了这一问题。此时，您的最佳解决方案是制定一个清晰的政策，包含将机密数据放入公共存储库的后果。我们现在缺少“后果”，更愿意说，“好吧，让我们确保这种情况不会再次发生，”但在公开发布[秘密](https://devops.com/?s=secrets)的情况下，这是与被检入的编码错误不同级别的“坏”。这些秘密可能会被用来攻击组织，如果不可共享的数据被明确界定，任何违规行为都可能被视为故意的，无论是出于便利还是渎职。

根据这份报告，有数百万条保密信息。并非所有这些都来自企业 IT 部门，但有些确实如此。有些可能来自您的组织。设定期望，获得反馈。明确表示您不希望停止系统交付的进展，但是您也不希望实现一个带有供攻击者使用的内置后门的系统。很容易对这些少量的数据使用替代的交付方法，并将大部分的源放在需要的地方。

继续摇摆。《走向数据科学》杂志[称](https://towardsdatascience.com/githubs-path-to-128m-public-repositories-f6f656ab56b1)目前有 1.28 亿个公共存储库(Q1，2021)。想想那些仓库里有多少行代码，还有多少潜在的秘密。确保你的不在列表中。因为您的关键数据泄露而阻止您的下一个伟大解决方案的运行是没有意义的。
