# 数字客户体验:未来是模块化的

> 原文：<https://devops.com/digital-customer-experiences-the-future-is-modular/>

随着微服务和 API 的激增，ui 也会模块化吗？一些技术专家注意到围绕更小的、基于意图的、更直接反映 API 端点的客户体验的上升趋势。

我最近会见了 Airkit 的联合创始人兼首席技术官亚当·埃文斯(Adam Evans)，以更好地了解当今数字客户之旅的中断之处以及他们需要如何发展。他预测单一目的的体验在未来会变得更加普遍。在这个过程中，我们可能会遇到更少的本地应用和更少的下载。为了实现这一切，他相信低代码可以为更多的开发人员带来更容易的集成和自动化模式。

## 数字体验的状态

[数字客户体验](https://devops.com/the-role-of-apis-in-digital-transformation/)变得越来越互联和基于网络，有时与核心服务脱节。在[无头环境](https://devops.com/more-and-more-enterprises-go-headless/)，终端用户通过越来越多的抽象层与公司互动，如聊天、短信、语音或电子邮件通知。埃文斯指出，这些互动越来越模块化，越来越受事件驱动。

越来越多的情况下，用户接触是在需要用户操作时激活的基于触发器的模式的结果。不仅通信方式在改变，而且新的设备和平台也出现了，这进一步证明了分离开发方法的合理性。B2C 应用在这一转变中变得更加模块化，呈现围绕特定任务构建的体验，并密切反映驱动它们的专门微服务或 API。以单页应用程序(SPA)风格或 AWS Lambda 的兴起为例。

随着越来越多的自动化和订阅模式对商业软件采取“一劳永逸”的态度，Evans 预测我们可能会看到更多由意图和事件驱动的工作流出现。他说，这些重新想象的用户界面将掩盖平台的复杂性，而是专注于单一的专门行动，如提交保险索赔，检查包裹的状态或更新你的咖啡订阅。

## 破碎的客户之旅

那么，为什么终端用户的数字体验会进化成更多意图驱动的应用呢？嗯，复杂的数字平台会很快变得功能臃肿。具有许多菜单的庞大应用程序难以导航，从而影响用户体验。

正如 Evans 所描述的，通用界面经常显示大量与手头任务无关的信息。“不要让我通过通用接口来帮助我做一些特定的事情，”他说。

笨重的界面与注意力持续时间不超过 20 秒的小型数字任务格格不入。Evans 将这些描述为“我们日常生活中的微型接触点，在这里你不需要传统的大型应用程序。”他说，这些行动可能是，例如，查找新冠肺炎感染率，检查你的信用卡余额或更新流媒体订阅。

组织也有改善用户体验的其他必要条件。例如，如果一个平台不是自助服务，它可能会增加支持呼叫，这会耗费公司大量资源。此外，如果服务不是由事件驱动的，用户必须定期自己登记，这可能会浪费时间，并加剧技术焦虑。

## 单页应用程序

在模块化微服务的时代，笨重的应用程序和“手动轮询”的日子可能会受到限制。对 Evans 来说，解决方案在于表达“表示该接口事务的最小方式”

"我们如何将更小的部门从后端转移到界面构建的方式上？"埃文斯问道。类似于微服务架构分离域的方式，Evans 注意到 ui 沿着 API 端点分割注意力。他设想构建更多单一用途的体验，用用户界面扩展 API 和微服务。

从技术上来说，这可能会导致更多的[单页面应用程序](https://devops.com/the-evolution-to-cloud-native-applications-and-apis/)被开发出来。Evans 描述说，映射 API 的利基 UI 层可能会携带一个数据模型，这样微型应用程序就可以提供数据，以便在 API 调用中输入字段。由于它们是基于网络的，异步的单一用途体验也可以超越应用商店和第三方市场的障碍。

## 模块化的潜在好处

更多基于意图的工作流意味着用户的开销更少，从而更快、更方便地解决他们想要解决的问题。当然，用户仍然需要注册一项核心服务，比如他们的银行、咖啡订阅服务或保险代理，以便进行集中账户管理。但是，随着用户周围的互动不断增加，用户更少经常光顾客户中心。

“应用程序的未来将看起来更像个人的、基于意图的微型应用程序，”埃文斯重申。这将反映出埃文斯称为“快乐的小瞬间”的接触点节拍器

只在需要用户采取行动的时候对其进行 ins 操作可以减少签到次数，从而减少数字焦虑。Evans 说，除了对用户有利之外，小型应用程序的部署和维护可能更容易支持。最后，他补充说，更多孤立的有意客户体验可以提高参与度、回应率和销售额。

## 使用低代码实现客户体验

[互联数字体验](https://devops.com/the-role-of-apis-in-digital-transformation/)越来越依赖 API。然而， [API 集成可能很困难](https://devops.com/low-code-opens-api-integration-potential-for-citizen-developers/)，即使对于有经验的程序员来说也是如此。那么，我们如何让人们更容易地构建接口来访问这些 API 呢？Evans 说，在 API 之上构建面向用户的层将需要优秀的[开发者体验](https://containerjournal.com/features/can-great-dx-enable-full-cycle-development/)。

最近，[低代码](https://devops.com/help-us-low-code-youre-our-only-hope/)一直主导着这些讨论。随着更加强调视觉驱动的开发，低代码正在赋予非技术员工权力。对埃文斯来说，低代码能力对于影响未来客户旅程的构建至关重要。

埃文斯指出，低代码“改变了应用程序构建的整个游戏”例如，品牌化的体验通常需要可重用的代码和公共库，这些都是低代码平台擅长的。此外，当使用 API 时，“REST 之上的层设计对于维护状态和实现长期处理是必不可少的，”他说。

## 新的数字之旅会走向微观吗？

复杂的界面似乎适用于像亚马逊这样的企业巨头。但是其他场景可能受益于更单一目的的工作流。随着平台变得更加广泛，用户交互需要变得更加精简，以避免增加压力。因此，埃文斯预测，在未来几年，基于触发器的微型 B2C 接触点将变得更加普遍。在这种范式中，更好的通用 APIs 扩展可以帮助人们融入其中。

“未来十年，我们会看到很多很酷的东西，”他说。“未来的应用不会像过去那样。”随着它转向支持新的多种渠道，如聊天、语音、网络和 VR/AR，我们对应用程序的认知也将不可避免地发生改变。

在未来的“应用程序的大拆分”中，正如埃文斯所说，“如果你不是每天都使用它，它很有可能不需要存在。”
