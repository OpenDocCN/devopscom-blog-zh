# Stytch 推出新的灵活性第一的 SDK

> 原文:[https://devo PS . com/s tytch-launchs-new-flexibility-first-SDK/](https://devops.com/stytch-launches-new-flexibility-first-sdk/)

无密码解决方案已经成为一种趋势有一段时间了，它改善了用户体验(UX ),同时减少了暴露于常见攻击媒介的风险。现在， [Stytch 的新 JavaScript SDK](https://stytch.com/blog/stytchs-new-javascript-sdk%ef%bf%bc/) 旨在通过增加定制和灵活性来使无密码身份验证变得更容易，同时提供开箱即用的完全烘焙的身份验证。

Stytch 首席技术官 Julianna Lamb 表示，该解决方案提供了双重优势，一方面完成了繁重的实施工作，另一方面提供了从客户的第一次登录体验中创建强大品牌 UX 的灵活性。Stytch JavaScript SDK 可以提供一体化的体验，通过预构建的前端组件处理整个用户身份验证过程。但它也可以让团队完全控制设计复杂和定制的身份认证之旅。兰姆说，在一个科技公司将最终客户体验视为圣杯的时代，这应该会吸引那些希望改善客户对其品牌的第一印象的团队。

## 智能前门的多种选择

像许多认证工具一样，Stytch 的产品套件最初提供了两种选择:一种是处理前端组件的标准小部件，而后端 API 提供登录。另一个选项提供直接的 API 集成，开发人员构建他们的前端，在前端处理会话管理和设置 cookies 等任务，并使用 API 做一些事情，如发送一次性密码或通过电子邮件发送神奇的链接。

“但我们发现，当涉及到他们正在使用的任何解决方案时，人们真的希望有灵活性，”Lamb 说。“他们也不想承担像会话管理这样的繁重工作。这些东西可能很难理解，打破常规真的很有吸引力。”

新的 [SDK](https://devops.com/?s=SDK) 将嵌入式 ui 从助手功能中分离出来，为团队提供了多种选择。他们可以将 Stytch 的 UI 组件放在产品中任何他们想要的地方，按照他们的规范来塑造认证流。作为一个无头 SDK，它可以嵌入到应用程序的客户端来处理对 API 的所有调用，进行身份验证，创建会话并检查其有效性。结果是:需要编写的后端代码更少，需要实现的逻辑更少。

可定制的 UI 流意味着团队可以完全拥有 UX 体验，并构建一个完全按照他们想要的样子定制的 UI——他们可以这样做，而不必构建一个内部 API 来将身份验证请求从客户端路由到服务器。寻找在一开始就产生品牌影响的方法的团队可以通过创造独特的用户体验来利用这种能力。

“认证是你的应用程序的前门，”Lamb 说。“注册是用户对你的应用程序的第一印象，能够根据你的品牌外观和感觉定制应用程序，使用户界面符合你的审美，这一点非常重要。”

## 身份验证的无摩擦灵活性

因为团队在一个产品中可能有不同的身份验证需求，所以 SDK 可以与核心的 Stytch API 配合使用，以满足不断变化的需求。团队可以通过 API 生成嵌入式神奇链接，同时通过 SDK 提供 Google OAuth 和 Google One-Tap 登录选项。

该解决方案还减少了摩擦，不仅可以更容易地验证用户，还可以在回访时验证他们是否继续相同的会话。通过在前端加载 SDK，团队可以自动将 Stytch 的会话产品连接到浏览器 cookie，将用户会话数据直接输入 cookie。

兰姆说，Stytch 团队目前正在开发更多的 SDK 功能和前端组件。考虑到满足复杂需求的简单身份验证工具的不断发展，观察创新身份验证的下一步走向将是一件有趣的事情。
