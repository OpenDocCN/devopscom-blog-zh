# Devs 如何提高企业中的开源安全性

> 原文:[https://devo PS . com/how-devs-can-improve-open-source-security-in-the-enterprise/](https://devops.com/how-devs-can-improve-open-source-security-in-the-enterprise/)

现代应用程序是动态的。它们是分布式的，通常诞生在云中。这些应用程序可以动态开发、快速升级和扩展，以满足不断变化的用户和市场需求，从而实现一定程度的业务灵活性，使用户能够实时做出快速、明智的决策，并在机会出现时加以利用。

支持这种开发速度的是数以万计的开源库——开发人员可以将代码片段包含在他们的应用程序中，以执行特定的功能，而不必从头开始编写代码。开源库使开发人员不必重新发明轮子，并允许他们专注于自己的差异化业务逻辑。虽然高效和节省大量时间，开源软件可能是当今软件供应链的最大威胁。开发人员只是过于相信自由、开源代码的完整性，他们随意地将这些代码包含在他们的应用程序中并随其一起发布。

尽管有操作上的好处，但是快速开发的压力不应该使组织面临更大的安全风险。开发团队需要重新思考他们如何使用开源软件，并采取措施确保它们符合与其他企业应用程序相同的安全标准。

## 开源安全问题

问题是大多数开源项目都是由少数开发人员维护的，他们贡献自己的时间和专业知识来构建可以被整个社区重用的软件组件。然而，这些开发人员是志愿者，他们的时间和努力没有报酬，他们也没有企业开发人员那样的压力来确保他们的代码中没有漏洞。他们还可能缺乏必要的企业安全专业知识来保护应用程序免受日益复杂的攻击。如果这意味着降低性能或改变用户体验，一些人甚至会反对更好地保护他们的代码。

结果是不安全的代码越来越多地进入使用这些开源库的企业应用程序。事实上，去年的 [Log4j](https://www.ftc.gov/policy/advocacy-research/tech-at-ftc/2022/01/ftc-warns-companies-remediate-log4j-security-vulnerability) 漏洞是由于一个[开源 Java 日志库](https://devops.com/?s=Log4j)中的漏洞。认识到开源软件对国家安全的重要性，拜登政府最近发布了一项 [行政命令，并会见了网络安全专业人士](https://www.whitehouse.gov/briefing-room/statements-releases/2022/01/13/readout-of-white-house-meeting-on-software-security/) ，以更好地保护联邦政府的计算机系统。由此产生的建议包括一些策略，这些策略将有助于防止开放源代码包中的安全漏洞，改进识别和修复缺陷的过程，并减少分发和实施修复的补救时间。T11】

私营部门必须注意到，并在全行业范围内做出更多努力来保护开源软件，这些软件构成了当今企业最关键的应用程序。

## 在企业中实现更好的开源安全性的三个步骤

底线是企业自身需要为保护企业应用承担更多的责任——尤其是当这些应用更加依赖开源库的时候。他们需要采取更加主动的方法，跨越整个软件生命周期，将安全工具直接嵌入到开发平台中，并对构成其最关键应用的内容提供完全的透明度。

组织可以做以下三件事来确保他们的开源软件是安全的:

**1。持续进行端到端的应用测试**

现代应用程序在不断发展，这使得在整个软件生命周期中不断测试漏洞变得比以往任何时候都更加迫切。从规划阶段开始，安全性应该嵌入到整个过程中——在通过测试之前，为开发人员提供一个构建安全代码的可靠框架。一旦推出，应用程序安全性测试应该继续作为将代码推向生产时的一个自动的内置步骤。开发人员和安全团队应该共同努力，确保安全不再是特性开发的事后想法。

**2。将企业安全功能直接嵌入开发平台**

当开发人员已经面临快速推送代码的压力时，强迫他们采取额外的步骤来确保代码的安全是失败的原因。安全团队必须通过将安全控制直接集成到现有的开发过程中来使其尽可能简单。在开发过程中标准化安全测试和控制使策略更容易被识别、理解和遵循。

**3。确保你使用的每一段代码都是透明的**

复杂性是良好应用程序安全性的敌人。DevOps 团队需要确切地知道他们的应用程序是由什么组成的——包括开源库及其依赖关系。联邦政府正在推动开源社区标准化软件材料清单(SBOM)。类似于要求硬件制造商知道每个部件来源的法规，这将迫使开源开发者和企业开发者为他们从其他来源获得的代码承担更多责任。这样，如果在库中发现了漏洞，企业可以轻松地确定他们是否面临风险，并采取适当的措施尽快完全缓解问题。

## 是时候承担更大的责任了

开源软件实现了使企业蓬勃发展的那种开发灵活性，但开源也极大地扩大了企业威胁面——使许多组织面临违规风险。组织需要对他们用来加速应用程序开发的开源库的安全性承担更大的责任。这需要在整个软件开发生命周期中进行端到端的应用程序测试；开发平台和流程中的嵌入式安全工具和控制，以及对构成关键企业应用程序的开源软件的完全透明。这种主动的方法将确保在威胁者能够利用漏洞和危害应用程序之前，快速识别和修补 Log4j 之类的漏洞。
