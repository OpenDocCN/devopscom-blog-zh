# 为什么数据沿袭如此重要，为什么它如此具有挑战性

> 原文:[https://devo PS . com/why-data-lineage-matters-and-why-its-so-challenge/](https://devops.com/why-data-lineage-matters-and-why-its-so-challenging/)

变化会带来风险。这是软件开发的基本原则之一，我们大多数人在职业生涯的早期就学会了。然而，尽管有那些重复的生活教训，它似乎总是不断出现。我们无法预见变化的影响，即使是很小的变化，也经常导致负面的结果。

随着互联 IT 系统复杂性的增加，这个问题变得更加严重。越来越难预测一个小的修改会如何影响上游或下游的系统。[数据血统](https://www.datafold.com/)实现了这一点，但是要做好数据血统非常困难。

### 为什么数据沿袭很重要

在旧世界中，数据驻留在孤岛中，它们之间的连接点相对较少。今天，数据在多个系统之间多向流动。许多这样的系统与外部组织共享数据，这意味着信息随后也会流向内部系统。因此，比以往任何时候都更容易忽略一个变更对多个其他系统的影响。

例如，想象一下，您的公司提供了一个移动应用程序，使客户能够快速方便地请求新的服务呼叫。过去，您的客户可以指定他们拥有的产品变型。您的设计团队已经决定，既然您的 CRM 数据库中已经有了这些信息，您就不再需要从移动应用程序的用户那里收集这些信息。

您的任务是简化应用程序，因此您删除了该输入字段，并用一个从 CRM 中提取客户产品信息的查询来替换它。从理论上讲，生成的工作票应包含现场服务人员完成工作所需的所有信息。

不幸的是，有一个问题:您的 CRM 系统中的一些客户产品记录包含空值。您的移动应用程序正在查询这些信息，但它会向您的票务系统发送无效结果。您的现场服务人员不再知道如何准备他们的下一次服务呼叫。

这听起来可能是一个相对简单的数据质量问题——如果有人更彻底地审查 CRM 数据，就可能预见到这个问题。但是，如果 CRM 数据库中的空值直到后来才被引入，即使这样也可能无法解决问题。例如，如果来自新收购公司的客户数据被合并到您的 CRM 系统中，您可能没有与这些新记录相关联的产品信息。

有时，这些问题更难预测，而且可能不会马上显现出来。想象一下，当您的营销团队决定重新定义其客户细分标准时会发生什么。添加了新的数据字段来指定客户类别；然后用所有客户的值填充它，旧字段被弃用。

然而，市场部不知道的是，该公司的高层管理人员仍在查看旧数据。他们使用依赖于废弃字段数据的仪表板。每天都在添加新客户，但是没有人在 deprecated 字段中输入任何值。因此，高管团队看到的是扭曲的数据。突然之间，首席管理人员跟踪的 KPI 与组织中的其他人截然不同。更糟糕的是，直到事情发生几周或几个月后，才有人注意到这个问题。

接下来，想象一个类似的场景，但不是填充一个执行仪表板，而是源系统将数据输入到人工智能算法，该算法在你的电子商务网站上驱动客户推荐。当人工智能在有缺陷的数据上接受训练时——或者如果数据以某种方式发生了重大变化——它可能会极大地影响该技术的有效性。在这种情况下，它扭曲了你的客户推荐引擎。

### 用数据血统应对挑战

数据沿袭工具通过展示上游和下游潜在影响的完整路线图，提供了一种理解变更影响的系统方法。

这相当于降低风险和控制成本。如果我们能够预测变化会如何影响其他系统，我们就可以避免产生我们之前描述的问题。这将减少浪费的精力，加快上市时间，降低成本。我们都熟悉这样一个原则，即在开发过程的早期发现问题，解决问题会更快、更便宜；数据沿袭提供了一个关键的缺失环节，有可能在问题到达 QA 之前就发现它们。

数据治理和法规遵从性也有好处。当欧盟在 2016 年通过《通用数据保护条例》(GDPR)时，它对存储个人身份信息的系统提出了一系列新要求(PII)。如果客户要求你删除他们的数据，你在法律上有义务答应他们的要求。但是这种变化的上游和下游会发生什么呢？会打碎什么东西吗？或者，客户的一些 PII 会留在您系统的其他地方，使您不符合法律规定吗？

数据沿袭通过自动化检测上游和下游影响的过程来解决这些问题。它提供了清晰的可审核性，有助于确定更改源自何处以及数据如何达到其当前状态。

列级数据沿袭尤其困难，因为它要求解析 SQL 查询来准确辨别哪些内容发生了更改以及如何更改。这意味着利用数据库日志，但是由于有如此多的数据库供应商——每个供应商都有自己独特的 SQL 方言——解析各种数据库的信息可能特别具有挑战性。随着 SQL 方言的不断发展，数据沿袭工具必须不断改进以适应这些变化。
