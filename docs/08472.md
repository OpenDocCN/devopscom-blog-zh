# 白盒 PaaS

> 原文:[https://devops.com/whitebox-paas/](https://devops.com/whitebox-paas/)

我最喜欢的关于 PaaS 的博客文章之一是 James Urquhart 的“[你的 PaaS 是可组合的还是有上下文关系的？(提示:答案很重要)](https://gigaom.com/2013/02/16/devops-complexity-and-anti-fragility-in-it-context-and-composition/)’。在这篇文章中，James 区分了两种不同类型的 PaaS:上下文 PaaS 和可组合 PaaS。他还暗示了第三种 PaaS(尽管他将其归类为“可组合的”):

然而，也许最重要的是，有一些开源的“构建”工具链被直接部署到基础设施服务中，这些服务展示了一种交付和操作应用程序的纯粹可组合的方法。将 GitHub 与 Jenkins、Gradle 与 AWS CloudFormation 和 Autoscaling 等相结合，可以为应用程序开发和运营提供一个完全自动化、灵活的“平台”——您希望从 PaaS 获得的一切。当然，问题是，随着时间的推移，您将需要组装和维护这个工具链(而不是让 PaaS 供应商为您做这件事)。”

**白盒平台即服务:**

当我们开始 VisualOps 时，起初我们没有意识到我们正在构建一种新型的 PaaS，或者说是一种可定制的 PaaS，这部分是由于我们的 DevOps 背景。正如 James 所指出的，经典的“上下文相关的”PaaS 的价值可以通过使用 DevOps 工具链来实现。“可定制”比“可组合”意味着更多的透明度和控制，因为环境的所有方面都在您的控制之下。每个积木的整体构成都是由你决定的，而不仅仅是安排好的。黑盒 PaaS 消除了这种定义自己的规则的能力。

虽然目前这类工具链大多是自主开发的，但这里的关键点是 PaaS 不必是一个黑盒。白盒 PaaS 公开底层基础设施的 100%原生接口，提供终极控制和透明性。将基础设施即代码和配置管理功能相结合，作为构建、自动化和编排应用程序的快捷方式。将白盒平台即服务与上下文平台即服务进行比较，优势很明显，白盒平台即服务不会对应用程序造成预先确定的限制，因此您的应用程序将是通用的，不会被锁定。

**好处:**

这种方法的一个关键优势是，白盒 PaaS 能够自行部署和管理。通过 aWhitebox PaaS 服务，客户可以点击部署私有克隆，并且不需要任何管理工作，因为克隆由“源”监管。这消除了约翰·E·文森特在他的博客中所说的“不成熟操作”:

“需要说明的是，虽然我觉得大多数组织还没有准备好应对维护 PaaS 的运营挑战，但是 PaaS 软件让这项工作变得更加困难。在这两种情况下，产品本身的运营成熟度根本不存在。”

另一个价值是，在白盒中 PaaS 不是应用程序运行时。相反，您的代码将在独立的基础设施(VM)中运行，PaaS 只需要帮助自动化和编排应用程序。因此，即使 PaaS 关闭，您的应用程序仍将不受影响地运行。这完全回避了“他们的停机时间[就是]您的停机时间”的问题，以及公共平台即服务的多租户所带来的安全问题。

**异议:**

对于 WhiteboxPaaS 的必要学习曲线和内在复杂性，可能存在一些顾虑，因为它需要您参与应用程序设置和系统架构。诚然，这绝不是最容易的选择。然而，如果你从商业的角度来看，拥有这些知识会给你带来巨大的竞争优势。如果你不知道如何做到这一点，你可能会失去构建一个更好、更快、更健壮的应用程序的机会。此外，如果人们使用 PaaS 来实现“NoOps ”,那么“无差别的繁重工作”的真正含义似乎是保持应用程序运行所必需的日常操作，而不是它所基于的知识。在这种情况下，使用白盒 PaaS 免除了您所有的繁重工作，但仍让您有机会和能力进行改进，它是“可定制的”。

这并不是说 PaaS 必须永远是白盒。詹姆斯说得好:“这不是一个非此即彼的选择，很像稳定性和弹性”。不同类型的 PaaS 服务于不同的市场和不同的目的。根据我们的经验，我们已经看到一些客户开始使用黑盒子 PaaS，如 Heroku，并随着它快速增长，直到他们想要甚至需要更灵活、更强大的东西，然后他们选择迁移到 AWS+白盒子 PaaS，如 [VisualOps](http://www.visualops.io) 。这种进展允许您在开始时将时间和精力完全集中在业务本身上，然后随着业务的增长升级到更加熟练和灵活的平台。
