GitOps 2.0 的目标是提供改进软件交付和可靠性的模式和标准。GitOps 是一种已被证明的健壮的软件部署模式，它已经围绕一些核心原则发展起来，但还有几个问题需要解决:处理多种环境、秘密，以及最紧迫的可观察性。Codefresh 正在与 DevOps 社区合作定义这些模式，并使交付软件的方法主流化和标准化，无论是否在 Kubernetes 上。

沙斯塔风险投资公司(Shasta Ventures)的投资者、最成功的 Kubernetes 产品 OpenShift 背后的关键人物伊萨克·罗斯(Issac Roth)表示:“DevOps 自动化至关重要。“GitOps 2.0 是将实践主流化的关键举措，有助于公司更快地发布代码，并始终相信他们的更改会奏效。”

Codefresh 的首席技术传道者丹加菲尔德(Dan Garfield)表示，微服务的兴起导致了可观察性的困难。“在迁移到大型微服务堆栈后，关于整体式服务的难以回答的问题可能无法回答。这就是为什么我们的 GitOps 2.0 的第一个支柱是可观测性，”他说。“今天，我们发布了更好的工具，不仅可以在 Kubernetes 上部署软件，还增加了可观察性，这样你不仅可以看到部署了什么代码，还可以看到部署了什么问题。”

除了让开发人员看到与发布和控制回滚相关的所有内容的详细视图之外，Codefresh 还提供了更大的聚合视图，可以很容易地判断所有应用程序中正在发生的事情。拥有这种可见性和控制可以给团队在他们的发布过程中更多的信心，并导致更快、更可靠的部署。

Veeva 的 Codefresh 用户 Victor Cuascut 说:“将 GitOps 直接嵌入 Codefresh，只需为我们的开发人员点击一下就可以了。”“Codefresh 了解 DevOps 的动向。GitOps 支持是游戏规则的改变者。”

Codefresh 现在支持的另一个重要领域是构建关于如何以及何时进行部署的逻辑。指导原则是基础设施应该反映 Git 的状态。在实践中，推广有时需要在许多地理区域之间进行，或者对于某些区域来说，可能无法将人从批准循环中解放出来。为了解决这个问题，Codefresh 为管道引入了不同的部署步骤，可以基于管道逻辑触发 Git 同步。这允许更高级的流程，例如部署、将 PRs 开放到其他基础设施回购上，或者跨不同集群和部门进行部署，以及通过回滚对已部署版本进行自动测试。

“采用 GitOps 2.0 并不是孤注一掷的努力。在这个版本中，我们为团队提供 onramps，让他们在组织允许的情况下尽快采用这些实践，并最终支持理想的工作流，”Garfield 说。
