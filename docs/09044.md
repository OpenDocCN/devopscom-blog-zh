# Linux 基金会调查发现 SBOM 使用增加

> 原文：<https://devops.com/linux-foundation-survey-sees-rise-in-sbom-use/>

Linux 基金会今天发布的一项针对 412 个组织的全球调查发现，近一半的受访者(47%)生产或消费软件材料清单(SBOMs ),超过四分之三的人表示他们预计在 2022 年生产或消费软件材料清单。

在一系列高调的[软件供应链](https://devops.com/?s=software+supply+chain)相关违规事件之后，SBOMs 已经成为一个更大的关注领域。这些违规行为涉及被恶意软件破坏的应用程序组件，这些恶意软件是在应用程序开发生命周期中偷偷插入的。拜登政府甚至发布了一项行政命令，要求所有机构为他们使用的所有商业和定制软件提供 SBOMs，但截至目前，遵守该命令的最后期限尚未设定。

总的来说，Linux 基金会的调查发现，很大一部分受访者(82%)已经熟悉了术语 SBOM。受访者列举了拥有 SBOM 的各种好处，包括让开发人员更容易理解应用程序中组件之间的依赖关系(51%)，更容易监控组件的漏洞(49%)，以及更容易管理许可证合规性(44%)。

然而，调查也清楚地表明，SBOM 还有许多工作要做。例如，整整 62%的受访者希望在如何将 SBOMs 的生产/消费整合到他们的 DevOps 实践中达成更好的行业共识。受访者还提到需要就 SBOMs 整合到其风险和合规流程中达成共识(58%)。

超过一半(53%)的人还表示，关于 SBOMs 将如何发展和改进，需要更好的行业共识。

该调查还表明，组织更加意识到 SBOMs 在保护供应链中的作用。整整 80%的组织知道白宫关于改善网络安全的行政命令，超过四分之三(76%)的组织考虑做出改变，作为该命令的直接后果。总的来说，调查发现组织在评估开源软件时最关注的是安全性。

Linux 基金会研究副总裁斯蒂芬·亨德里克表示，2022 年很可能会作为 SBOM 年被人们记住。构建 SBOMs 有几个标准，包括 Linux 基金会和联合开发基金会支持的开放源代码软件包数据交换(SPDX)规范，该规范现在被认为是 ISO/IEC 5962:2021 国际标准。还有 CycloneDX，这是一个轻量级的 SBOM 标准，基于一个开源项目，该项目起源于开放 Web 应用程序安全项目(OWASP)社区，以及一些组织用来创建 sbom 的标签的软件识别(SWID)标准。

本质上，SBOMs 是一种列出组成应用程序的组件的工具，但使用起来并不一致。但是，当发现关键漏洞时，他们可以大大减少发现所有需要修复的软件组件所需的时间。挑战在于，随着容器的兴起，以一种可能并不总是在 SBOM 中得到反映的方式来撕毁和替换软件组件已经变得简单得多。亨德里克在《时代》杂志上说，机器学习算法等技术的进步应该更容易在 SBOM 中即时反映这些变化。

不管采用哪种方法，显而易见的是，无论喜欢与否，DevOps 团队都将对其整个应用程序组合中使用的每个组件的来源承担更多责任。
