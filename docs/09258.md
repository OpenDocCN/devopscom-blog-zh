# 面向 SAP 的 DevSecOps 是什么样子的

> 原文:[https://devops.com/what-devsecops-for-sap-looks-like/](https://devops.com/what-devsecops-for-sap-looks-like/)

在过去的几年中，针对业务关键型应用程序及其内部数据的网络攻击不断增加，因为出售或交易这些数据特别有利可图。 运行 SAP 软件的组织尤其困难，因为 SAP 系统复杂且难以修补，更糟糕的是，信息安全和网络安全团队往往不了解 SAP 系统以及如何保护它们。传统的安全方法不足以保护 SAP 系统，但针对 SAP 的 DevSecOps 可以引入一种效果更好的新方法。

### 什么是 DevSecOps ？

传统的安全性经常成为加速软件交付的障碍。 DevSecOps 引入了一种新的安全第一的思维模式，强调软件开发生命周期每一步的安全性，并让组织中的每个人都对安全性负责。

DevSecOps 包括一个组织的所有部门，并将安全思想嵌入到从执行委员会到各个团队的整个流程中。为了取得成功，业务运营者需要工具和流程来支持:

*   连续监控
*   扫描安全缺陷
*   攻击检测
*   变革管理和治理
*   定期评估

### 为什么 SAP 的 DevSecOps 会改变游戏规则？

过去，在实施新项目或新功能发布时，通常会忽略安全性，因为信息安全团队会在现有系统中解决安全缺陷。然而，这种方法带来了额外的复杂性、成本和业务中断。

如果在项目的早期阶段就引入了安全考虑因素，那么安全缺陷就可以更便宜、更快速地得到识别和修复。仅这一点就使得解决潜在的可利用漏洞变得更加有效和容易。有了好的工具，这个变得非常有效，并允许组织识别和解决在他们的变更过程的早期迭代中引入的漏洞。

然而，最大的好处是，任何新的流程或功能都是通过安全设计来实现的，从而形成一个弹性环境，可以更轻松地抵御网络攻击。

### 为 SAP 开发软件T2

这里有一个例子:假设一个新的业务项目开始了，目的是改变 SAP 应用程序或流程以引入新的功能。像任何项目一样，时间、预算和可用资源是关键要素。为了让 DevSecOps 正常工作，必须在项目的早期阶段考虑重要的安全问题。 现实中，每一个 单项工程都是安全工程。这意味着业务需求和目标不能优先于安全问题。需要过程和工具来使开发和安全团队一起工作来回答重要的问题:项目会对包含的数据和已建立的过程带来安全影响吗？类似地，是否需要额外的软件和安全架构，或者是否需要将特定的技能集合加入到项目中？

在敏捷环境中，一旦所有的史诗和用户故事都写好了，设计阶段就可以开始了。有了嵌入到项目中的安全思维，这将自动导致设计安全的解决方案。

在实现阶段，开发人员需要工具来扫描潜在易受攻击的源代码。

在开发过程的早期识别允许 SQL 注入、跨站点脚本或遗漏授权检查的漏洞，可以更容易地修复它们。

这里的挑战是 SAP 没有提供开发人员验证源代码安全缺陷所需的工具。SAP 运输管理系统易受软件供应链攻击，除非安装了适当的 安全补丁。由于这个原因，组织需要一个可以集成到 SAP 标准开发 IDE 中的代码漏洞分析器。

此外，SAP 运输管理系统中启用的质量关可以非常有助于开发人员避免缺乏适当安全验证的源代码。

在用户验收测试(UAT)阶段发现的功能问题也必须触发验证周期的重启。只有满足所有安全和功能要求，才能开始生产部署。 在生命周期的这个阶段， DevSecOps for SAP 侧重于监控，以实现攻击检测、定期(或更好的持续)漏洞评估和准确的安全补丁。

### 为 SAP 引入 DevSecOps 需要哪些工具？

虽然许多组织已经利用了变革管理和 IT 服务管理解决方案，但他们还不具备踏上 SAPDevSecOps之旅所需的所有工具。

请记住，市场上有一些平台提供开放的 API，允许集成到已建立的 SAP 解决方案中，并提供变更管理和安全事件管理之间缺失的链接。与用于代码扫描和漏洞管理的独立工具相比，这些解决方案提供了保护 SAP 的一站式服务，并降低了 [TCO](https://securitybridge.com/business-case-calculator-for-sap-security/) 。
