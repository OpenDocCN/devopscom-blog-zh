# 发展战略的六大支柱:支柱一----集体责任

> 原文：<https://devops.com/the-6-pillars-of-devsecops-pillar-one-collective-responsibility/>

随着对 DevSecOps 兴趣的增加，[云安全联盟](https://cloudsecurityalliance.org/) (CSA)和软件保障卓越代码论坛(SAFECode)组建了一个 DevSecOps 工作组，以确定并分享开发和维护 DevSecOps 计划的最佳实践。该工作组确定并定义了对将 DevSecOps 整合到组织中至关重要的六个重点领域；本文将关注支柱 one⁠—collective 的责任

### 安全的背景

从历史的角度来看，我们发现与其他主要目标(如快速向市场发布功能)相比，安全性常常被视为次要目标。基本原理是，如果我们不快速发布，我们的竞争对手将会超过我们。不幸的是，这种心态充满了安全风险，我们软件中不断增加的漏洞就是明证。恶意行为者可以利用这些漏洞破坏关键系统并泄露敏感数据。

然而，我们不要对自己太苛刻。DevOps 世界的安全问题不是一个容易解决的问题。当我们继续修补易受攻击的代码时，新的缺陷也会很快出现。从本质上讲，安全形势是动态的，并且会继续发展。解决这样一个复杂的问题需要思维的多样性。换句话说，我们需要让具有不同观点的人参与安全对话。本质上，我们需要建立一种植根于安全文化的集体思维。这样，安全就成了集体的责任，而不是少数人的责任。

既然我们已经就集体责任的重要性达成一致，那么合乎逻辑的问题就是，“我如何实施集体责任？”第一支柱解决了这个问题。我们需要将我们的安全和开发文化融合在一起。我们再也不能谈论安全*或* DevOps。必须两者兼而有之。这意味着在软件开发和操作生命周期的每一点都必须考虑和解决安全性问题。将这一点进一步扩展到技术领域之外，安全性从根本上与业务目标联系在一起，而不是作为一种事后的想法或仅仅被认为是一种成本消耗。

建立支持安全的文化包括三个主要阶段:

*   第一阶段:获得管理层的支持和参与
*   第二阶段:设计有效的计划
*   第三阶段:维持计划

## 支柱一对 DevSecOps 实践意味着什么？

支柱一强调了安全责任不能局限于少数人的观点。正如软件开发不再是一项专门的技术活动，安全性也不再是一项法规遵从性活动。这对 DevSecOps 实践有着深远的影响。以下是受这种心态和责任转变影响的几个领域:

*   从事软件开发和运营活动的人现在必须考虑安全需求
*   安全工具不再能独立于 DevOps 管道运行
*   DevOps 团队需要了解安全性以及它如何应用于他们的角色
*   安全团队需要将高级策略转化为清晰、原子的安全需求，以便实施和测试

## 第一支柱是如何实施的？

支柱一的实施涉及旨在消除筒仓的结构性问题。建立跨职能团队有许多可能的方法。一种方法是在 DevOps 团队中嵌入安全性。另一种方法是将安全性集成为 DevOps 团队的支持结构。在这两种情况下，鉴于安全专业人员的数量有限，安全冠军可以帮助弥补开发运维团队和安全团队之间的差距。安全倡导者确保正在构建和部署的软件考虑到了安全性。为了实现这一点，建立一个成功的安全拥护者非常重要:

*   安全必须是他们的首要责任
*   他们必须被视为工程团队中的安全专家
*   他们负责指导和培训开发人员
*   他们可以拥有威胁和隐私模型
*   他们必须有与业务一致的明确目标

在没有明确方向感的情况下设立安全冠军计划可能会导致大量时间和精力的浪费。集体责任取决于团队是否能够跨职能部门进行沟通，安全冠军就是这种心态的典范。

## 速度与安全

我们生活在一个软件开发和部署速度超过传统安全实践的世界。这是有代价的。平衡速度和安全性是新的标准。正确的平衡取决于每个组织。然而，有一点是不变的，那就是文化。我们需要一种支持安全并将其视为集体责任的文化。这需要意向性。组织需要让他们的管理人员参与进来，并确保该计划具有推动安全文化的相关元素，并具有实现商业价值的清晰路径。
