# 是时候保护 Git 了

> 原文:[https://devops.com/it-is-time-to-secure-git/](https://devops.com/it-is-time-to-secure-git/)

在这一点上，我们已经有了大量使用 Git 的经验。锁定 Git 是有据可查的。好吧，这是有记录的。至少我们可以这么说。有趣的是，很多“如何保护 Git”信息实际上是“如何将关键信息隔离在 Git 之外”，这就是问题所在。“关键信息”总是包含“不要将配置信息放入 Git”的警告。永远不会。”这几乎是 Git 存在以来的默认锁定方式。

然后出现了 [GitOps](https://devops.com/?s=GitOps) 。简而言之，它的目的是什么？*将关键信息放入 Git。*所以我们一直在跳华尔兹；我们中只有一小部分人真正锁定了我们的实现，因为我们要确保 Git 中的数据虽然有价值，但不会充满对危害系统有用的信息。但现在会了。事实上，在 GitOps 中，几乎所有东西都是对攻击者有用的信息。

所有这些都意味着是时候认真对待 Git 安全性了。不是*内容*，而是实际 Git *安全*。在这里，我不会疯狂地使用锁定文档的链接——锁定本地 Git 存储库和远程 Git 存储库是完全不同的过程，当您锁定 GitHub 或 GitLab 这样的服务时(实际上，有些工作是由 SaaS 供应商为您完成的)…因此，您必须自己根据您的 Git 实现来查找。但是*有*文件可以带你浏览；您只需要确保您找到的是“如何锁定 Git 存储库”而不是“如何将特权数据排除在 Git 存储库之外”

但是无论你做什么，不要再拖延了。现在，如果一个不务正业的人进入了你的存储库，泄露的将不仅仅是各种应用的源代码——这当然足够糟糕，即使你的组织不是一个软件组织——也将是可以被使用和滥用的基础设施信息。因此，确保信息以各种可能的方式得到保护。指出 Git 的安全性并告诉他们，“我们知道你们非常忙，但是如果我们要实现 GitOps，这是最重要的事情。”因为它确实是。如果数据库配置、网络配置，甚至可能还有登录信息都要存储在 Git 中，那么必须将其锁定(包括本地和远程副本)。你不会将这些信息以纯文本的形式保存在一个集中的数据存储中，并指望现有的保护措施。所以不要用 Git 做等价的事情。

如果 GitOps 不在你的眼前，那么你的源代码仍然充满了漏洞广告，所以至少让你的安全团队检查一下你的 Git install/policies/config。即使作为一名开发人员，我对源代码库的关注也不如对配置库的关注。有几个原因，但最主要的是你们所有人。大多数组织似乎并不关心源代码存储库的安全性，但是当运营部门被问及将防火墙中的漏洞列表移入 Git，并且可以从 Git 进行配置时，情况就变了(这只是一个例子)。

所以，继续踢吧！你们都在制造、交付和保护大量的软件，在远程工作浪潮袭来之后，我们环境的复杂性并没有降低。然而，你仍然摇摆不定。愿你有足够的员工来做这份工作，也愿你有足够的应聘者来与你合作，而不是竞争对手。继续向世界展示软件是可管理的。但是保护好你的饭桶。
