# 谷歌提出保护软件供应链的 SLSA 框架

> 原文:[https://devo PS . com/Google-proposes-slsa-framework-to-secure-software-supply-chains/](https://devops.com/google-proposes-slsa-framework-to-secure-software-supply-chains/)

Google 提议组织采用一个框架来保护整个软件供应链中软件工件的完整性。

谷歌开源软件安全产品经理金·莱万多夫斯基(Kim Lewandowski)表示，软件产品 (SLSA)的[供应链级别是基于一个内部框架，即所谓的 Borg 二进制授权，该公司已经使用了八年多来保护其软件。Borg 是 Google 为构建和部署 Kubernetes 所基于的应用程序的平台起的名字。](https://security.googleblog.com/2021/06/introducing-slsa-end-to-end-framework.html)

在当前形式下，SLSA 是一套指导原则，它确定了如何获得四个不同级别的应用程序安全性。但是，我们的目标是创建一个框架，该框架将自动生成可审计的元数据，这些元数据可以输入到策略引擎中，以创建 SLSA 认证。

最低级别的 SLSA 要求构建过程完全脚本化或自动化，并生成支持创建出处的元数据。在最高级别上，SLSA 要求在一个封闭的、可重复的构建过程中对所有变更进行两人审查。Lewandowski 说，在最高的 SLSA 安全级别上，组织应该能够向其他人保证他们的软件供应链没有受到损害。

在最近一系列引人注目的软件供应链违规事件之后，对软件构造方式的审查水平显著提高。至少在理论上，开发人员承担了更多的应用程序安全责任，这是整体转变的一部分，为他们提供了对 IT 环境更多的编程控制。因此，网络犯罪分子现在专注于损害开发人员的凭据，作为在开发应用程序时将恶意软件插入应用程序的努力的一部分，或者他们试图利用开发人员通过编程错误配置基础架构而无意中创建的漏洞。

当然，最佳实践被用来教导开发人员如何更好地保护他们的应用环境。问题是没有足够多的开发人员学会如何足够快地保护应用程序。针对软件供应链的攻击数量继续以比组织仅仅通过教育开发人员所能阻止的速度更快的速度增长。在某些情况下，组织再次授权网络安全团队确保软件供应链的安全，而不管对应用程序开发速度的潜在影响。

莱万多夫斯基说，谷歌正试图推动采用 SLSA 作为保护开源和专有应用程序代码的框架。Lewandowski 指出，保护软件没有灵丹妙药，但一个最终在 DevOps 工作流环境中尽可能多地自动化安全流程的框架将大大有助于强化软件供应链。

SLSA 是对拜登总统发布的网络安全行政命令的回应，该命令要求美国商务部下属的国家标准与技术研究所(NIST)与联邦机构、私营部门、学术界和其他利益相关者协商，确定标准、工具、最佳实践和其他准则，以增强软件供应链的安全性。考虑到这些软件供应链之间存在的依赖程度，该行政命令现在在不同程度上适用于几乎所有为某个在某种程度上与联邦政府互动的组织开发软件的实体。
