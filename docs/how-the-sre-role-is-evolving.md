# SRE 的角色是如何演变的

> 原文：<https://devops.com/how-the-sre-role-is-evolving/>

近年来，场地可靠性工程(SRE)引起了人们极大的兴趣。在 [2019](https://blog.linkedin.com/2019/january/10/linkedins-most-promising-jobs-of-2019) 年，LinkedIn 将网站可靠性工程师列为美国第二大最有前途的工作。现在，在 2021 年，这一角色在许多组织中继续成长和发展。

该战略最初由谷歌牵头，并归功于工程师本·特雷诺(Ben Treynor)，旨在为应用程序的可靠性寻求更智能的问责制。SRE 团队设置服务水平指标，为新功能制定错误预算，并使用应用性能监控(APM)等工具来可视化性能洞察，以及其他任务。sre 正迅速成为增加业务产出以应对新的数字创新的关键手段。

我最近会见了 AppDynamics 地区 CTO Gregg Ostrowski，讨论了 SRE 的出现以及该方法如何发展。简而言之，SREs 在越来越多的公司中的重要性似乎只会继续增长，采用混沌工程这样的新策略，并用新的领域知识使他们的团队多样化，以应对日益增长的技术复杂性。

## 什么是 SRE？

传统上，组织雇佣系统管理员或系统管理员来维护大型计算系统和服务的运行。然而，这通常会产生二分法:产品开发人员希望推出新功能，而运营团队(系统管理员)希望确保现有服务不会中断。开发人员和运营人员之间的分裂可能是不健康的，并会引起摩擦。

系统管理员的职位与开发无关，并且涉及大量的手工工作，而 SRE 的方法，从另一方面来说，采取了一种更加关注工程的方法来自动化操作。已经成为各种圣经的谷歌 SRE 书籍将这个角色定义为“当你要求一个软件工程师设计一个运营团队时会发生什么”

那么，SRE 是做什么的呢？在谷歌，他们负责他们运营的服务的可用性、延迟、性能、效率、变更管理、监控、应急响应和容量规划。在谷歌，他们花在传统“运营”上的时间不超过 50%，如门票、随叫随到和手动任务，他们被鼓励用剩余的 50%时间来开发自动化，以取代人工操作。

## SRE 方法发展的 5 种方式

谷歌引入 SRE 概念已经有几年了，自然地，其他企业中也出现了独特的迭代。但是，总的来说，这一概念将走向何方？Ostrowski 认为 SREs 在几个关键领域不断发展:增长和成熟，领域专家的多样化和新的监控策略。

### 1.采用率提高

首先，并非所有公司都接受 SRE 模式。[brambless](https://www.blameless.com/blog/top-predictions-for-sre-2021)最近的一项研究发现“… 50%的受访者采用 SRE 模式，让专门的工程师专注于基础设施和工具，或者采用嵌入式模式，将全职 sre 分配给一项服务。”

SRE 模式的势头越来越大，但仍有更大的采用空间。还有内部成长的空间。奥斯特罗夫斯基认为一个 SRE 团队就是一个失败点。“它需要成为一个完整的部门，”他说。

此外，sre 在会议上的发言权越来越大，影响着功能的推广。“有了适当和成熟的 SRE 的参与，团队不能随意部署，”他说。Ostrowski 认为这些团队在商业风险和引入新技术之间保持着关键的平衡。

### 2.更大、更多样化的 SRE 部门

许多公司都面临着不断增长的用户需求，因此必须快速扩展其应用网络。与此同时，部署类型出现了寒武纪大爆发——系统可以使用任何种类的传统基础架构、大型机、微服务、云环境和[多家云供应商](https://devops.com/from-cloud-zombies-to-multi-cloud-experts/)。Ostrowski 说:“IT 领域的复杂性和拓扑结构已经大幅增长，存在许多相互依存关系。”

由于日益复杂的技术堆栈，SRE 现在必须覆盖许多领域。因此，SRE 部门可能需要一个更加多样化的领域专家团队。奥斯特罗夫斯基将 SREs 比作“IT 团队的海军海豹突击队队员”例如，让他们有在谷歌云平台(GCP)或微软 Azure 背后导航的经验，可以保持各自云中的服务可靠性。

奥斯特罗夫斯基说，个人的性格对 SRE 的适用性也很重要。这个角色需要特定类型的创新者，他们考虑代码的影响，并且喜欢改进过程和性能。

### 3.新的测试策略将会出现

Ostrowski 还预测 SRE 部门引入新的监测和测试方法，以保持可靠性。其中之一是[混沌工程](https://devops.com/chaos-engineering-for-itops/)，它支持故意破坏应用系统的想法。不同的策略无疑会出现，以帮助推动用户体验(UX)，并确保性能始终是第一位的。

### 4.企业依靠 sre 来降低风险

所有的事情都不可避免地会失败，在某些时候。sre 接受失败并学习管理失败，设计可重复的操作缓解结构。“随着公司越来越依赖于 IT 的消费化，业务也依赖于 SREs 来推动业务，”Ostrowski 说。随着企业利用 SREs 来维持稳定，这种依赖可能会增加。

无论是减少平均修复时间(MTTR)，规划服务水平指标以监控网站加载时间，还是预测新功能引入的错误预算，SREs 都将越来越多地被用来保持业务稳定性和高性能。

奥斯特罗夫斯基说:“SRE 人的心态是接受一个无可指责的环境。”。他们可以帮助“平衡业务和应用团队之间的风险，”Ostrowski 说。

### 5.SREs Steer UX

Ostrowski 说，在这个纯数字环境中，“应用已经成为业务”。如果应用程序是业务的核心，那么监控用户的旅程对于改进它是必要的。奥斯特罗夫斯基认为，SREs 监管着一个独特的领域，也能产生有价值的商业见解。

除了监控正常运行时间以确保 URL 响应时间符合 SLA 之外，SREs 还可以跟踪与 UX 相关的洞察，如转换率或购物车放弃百分比。跟踪此类分析并设定标准基线有助于查明影响 UX 的问题。它还可以帮助产品开发人员设计性能更好的软件。

## **最终想法**

"任何你做了两次以上的事情都必须被自动化."D-Tools 首席执行官亚当·斯通

传统的运营团队通常与产品开发无关，文化差异很大。然而，在 SRE 方法中，操作的目标是运行系统，使通常由系统管理员手动执行的工作自动化。

在这个简短的介绍中，我们仅仅触及了主题的表面。当然，每家公司在混合产品开发和运营方面都是独一无二的。DevOps 租户与 SRE 关系密切——他们通常是[混杂在一起，或者是同一个](https://techbeacon.com/enterprise-it/5-ways-site-reliability-engineering-transforms-it-ops)。

最终，真正从这种方法中收获商业价值将取决于打破孤岛并在不同的单位之间展开对话——或者至少是自动通知。只有这样，运营和开发才能以最高效的方式共存。
