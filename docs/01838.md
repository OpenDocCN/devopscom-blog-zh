# devo PS Chat:plu tora 企业中的价值流管理

> 原文：<https://devops.com/devops-chat-value-stream-management-in-the-enterprise-with-plutora/>

在企业范围内做任何事情都会带来一系列全新的需求。由于数百个应用程序和版本中潜在的数千个需求，将大型企业中的所有开发运维活动整合在一起可能是一项艰巨的挑战。这就产生了新的术语价值流管理(VSM)。)

就像管理销售渠道的 Salesforce 一样，VSM 关注的是贯穿其所有计划、团队和项目的 IT。高价值需求进入生产系统了吗？DevOps 是否在企业范围内产生了预期的影响？VSM 工具是否很好地集成到 DevOps 工具链中？或者团队被额外或不必要的工作拖慢了？你必须控制住一切，但我们不希望药物比疾病更糟糕。

在这一集的 DevOps Chat 中，我们和我们的嘉宾鲍伯·戴维斯·CMO 在 [Plutora](https://www.plutora.com/) 探索 VSM。Plutora 在一些最大的银行、金融机构、电信和移动运营商、保险、酒店和互联网技术公司应对 VSM 挑战。

## 副本

米奇·阿什利:大家好。我是 DevOps.com 的米奇·阿什利，您正在收听的是另一个 DevOps 聊天播客。今天，我邀请到了 Plutora 首席营销官鲍伯·戴维斯。我们的话题很有趣。我们都在谈论组织中的开发运维，但是如何实现高性能的开发运维呢？鲍勃，欢迎来到 DevOps 聊天。

鲍伯·戴维斯:嗨，米奇。很高兴来到这里。感谢您将我纳入 DevOps 会谈的议程。

阿什莉:这是我的荣幸。非常感谢你能来，鲍勃。你能开始介绍你自己吗？请告诉我们一些你作为首席营销官的工作，以及一些关于 Plutora 的信息。

戴维斯:绝对是。我在 Plutora 工作了两年半，在 Plutora 获得第一轮融资后不久就加入了进来，这让我们所有人都非常兴奋，我们真的走上了为转型到 DevOps 的人们提供更好结果的道路，这就是 Plutora 所做的。Plutora 是一家提供 SaaS 平台、大型企业、过去的典型实体、银行、保险公司等的公司。

这些人来找我们，我们与他们合作，提供可视化他们的整个管道的能力，以最终成为更好的软件开发人员，并捕捉他们在转型中组装的工具的泥沼，并真正理解它们，使它们高效并使组织成功，这就是我们所做的。

**Ashley:** 我特别有兴趣与您谈论这个话题，因为当您查看您的客户名单时，您会发现一些在《财富》500 强和《财富》1000 强中非常知名的公司。它不仅仅是一群其他的创业公司。其他人——

戴维斯:没错。

阿什利:–公司和其他人会认识到。所以处理金融交易，很多，电信，等等。因此，你有一个独特的视角，当我们想到 DevOps 时，我们想到的是将开发人员和运营人员聚集在一起的核心团队，但要真正在企业级的组织中扩展它，那是一个非常不同的问题。您认为在与客户打交道时面临的挑战是什么？为什么他们来找你说，“我们可以做到这一步，但我们需要走得更远，这就是我们需要解决的问题”？

**Davis:** 毫无疑问，潜在客户和客户的这一特殊要求是我们听到的第一件事，即他们已经开始向开发运维转型。他们可能从自动化测试开始。他们从提供 CI 服务器和自动部署功能开始。这是他们开始旅程的地方，他们从自动化开始，他们立即发现，虽然他们在开发管道的这一部分有所收获，但整体性能并没有像预期的那样提高。他们基本上站在那里说，“我做错了什么？”

人们对可见性这个词有很大的疑问。缺乏可见性会在整个开发过程中导致巨大的问题，不仅仅是对开发人员，不仅仅是对运营人员，而是对每一个人，从项目办公室到安全人员到治理和审计人员等等，因为我们与这些事情非常重要的人打交道。可追溯性、可审计性，这些都是——软件运行着非常敏感的业务，这些东西非常重要。

因此，如果没有可见性，为了在中构建这些东西，会打乱我的 DevOps 流程。我沿着小路走。作为一名开发人员，我准备进入一个测试周期，突然安全人员说，“嘿，我需要检查一下安全问题，”他来晚了，突然有两个星期的延迟。人们感到沮丧，首席信息官说，“我们在工具上投资了数百万，在组织变革上也投资了数百万，以适应敏捷和开发运维。我们越来越差，而不是越来越好，为什么？”这就是问题所在。

**Ashley:** 当你说可见性时，对我来说，如果我把这个词分开，我想你真正想说的是，它不一定总是由 DevOps 团队本身可见，而是与组织的所有其他部分，公司的政策部分，公司的审计部分，公司的安全部分的交集。如果你愿意的话，是所有其他人都参与到这个商业工具链中，把产品推出去。你不能一个人去做。也许是的，如果你是一个在地下室创业的三个人，你当然可以。但是，当您谈论在企业级扩展时，他们有停止和检查的流程。它准备好投入生产了吗？检查过安全了吗？所有这些事情。这就是你所说的能见度吗？

**Davis:** 了解范围的一部分是了解这些企业的规模。我们采访了中西部一家大型保险公司的一个团队，他们有 35 份申请，仅此而已。听起来不多。

阿什莉:是的。并没有。

戴维斯:但是那一周，也就是我在那里的那一周，他们有 1600 个版本要在那一周发布。当你往下看，在 35 个应用程序中，1600 个版本是什么？35 个应用程序，一个应用程序可能需要管理一个完整的发布组合。因此，这种对我的应用程序在开发周期中所处位置的理解的可见性，从它开始成为一个想法的时间到它发布的时间。

因此，这一过程需要了解发布了哪些功能，在哪里进行了测试，测试的版本是什么，他们何时准备好进行用户测试，何时准备好进行预生产最终测试，所有这些步骤都需要不同的工具，不同的团队，通常需要共享资源，如环境，他们依赖于团队中的其他团队以及这些团队中其他人的努力，以便在同一时间达到终点，因为正如我所说，一个特定的发布将有多个不同的团队，不同的项目团队。

因此，可见性是跨整个管道的，首先是从需求开始到发布的时间，但可见性也是管道之间的可见性，以了解依赖关系在哪里，了解可能影响我发布能力的其他事情，如安全性、审计，以及这些事情如何发挥作用。我必须对所有这些都有可见性，这样我才能参与并预测什么时候会有事情发生。那真的很重要。如果我正在开发——例如，如果我正在开发一个支票兑现应用程序，我是一家主要的金融机构，我的开发团队的一部分是一个 iPhone 应用程序，我有一个团队正在开发这个应用程序，他们遵循苹果的规则，他们遵循 UI 的规则，他们正在研究我如何做表格，他们正在做所有的事情。

在后端有一个数据库，它可能位于一个主机上，周围有令人难以置信的安全性，它必须升级才能提供处理自动移动现金支票的能力。正如我所说，这可能是在一个具有不同开发节奏的主框架上完成的。也许它的发布时间表是每六个月一次，这个应用程序应该在两周内发布。我如何协调这一点，如何让安全人员来做正确的事情？

所以为了做到这一点，我必须有一个单一的记录系统。我已经有了自动失能和仪表盘来提醒人们什么时候会有事情发生。我必须能够从多个不同的地方、多个不同的应用程序、多种不同的应用程序方法中获取信息，并将它们关联起来，从而为我提供简洁的单页仪表板视图，让我能够回答这样一个问题:“这个版本什么时候发布？”这就是我们做的所有事情。

**Ashley:** 好吧，让我们把可见性这个词再转 90 度，从这个角度来看，顺便说一句，我认为大多数企业都想拥有 36 个应用程序，你说什么？这是一个侧面的观点。无论如何，如果你考虑能见度——

**戴维斯:**嗯，那是一个小组——我想说清楚，那是一个小组。那是一个团体。

**阿什利:**哦，一组。好吧。我以为你指的是整个企业。我就像哇。我知道很多公司会说“你是怎么做到的？这方面的课程在哪里？给我报名。”无论如何，以可见性这个词为例。我认为你所说的很大一部分不仅仅是 CICD 从开发到部署的工具链。它也是发布的计划、安排和协调。这是把信息传递给人们的协调，也许不仅仅是信息，而是例外，因为你经常发布那么多的产品。

您不想每天阅读 20 份关于即将发布的每个版本的安全合规性的报告。你想知道我需要注意的五件事是什么吗？所以有很多协调计划和安排，我认为这是你们所做的更独特的事情。

戴维斯:我很高兴你提到了这一点，米奇，因为这是一个很快就会出现的问题，但通常没有被考虑到。当人们谈论开发运维，甚至是敏捷和开发运维时，当他们谈论软件交付管道时，通常他们实际上谈论的是生产的检入。当我开始将我的代码签入我的源代码库时，时钟开始计时。所有的开发工具，无论是自动测试 CI 服务器、CICD 管道、管理系统等等，它们都是围绕过程的后半部分和后端构建的。

对我来说，真正的问题要追溯到最初的需求规划。在企业敏捷规划方面，已经出现了一些非常成熟的非常好的工具。PPM 市场本身已经存在了很长一段时间，有一些非常有趣和出色的产品，允许我确定开发工作的优先级，与我的开发团队一起确定优先级，并根据公司的战略、公司的预算分配、相关应用和项目的预期回报值(无论是收入还是成本节约)来完成这些工作。

我把它摆好，然后说，“这是我的投资组合。这是当务之急。我有一个 _ _ _ _ _ 应用程序，我将驾驶它。这是我的首要任务。去吧。”所有团队都在努力，他们有信心按照公司的优先顺序进行建设。他们进入管道，我看不见了。嘣。它不见了。当产品被开发时，事情就发生了。范围必须改变。从开发的角度来看，为了满足我的时间表，我可能不得不去掉一些特别棘手的特性。必须做出决定。

我应该把它拿出来还是留着，因为它太重要了，我接受延期？如果我不能把它和史诗、战略、主题联系起来，如果你愿意的话，我怎么回答这些问题呢？除非我能明确地理解这与应用程序的最初想法有什么联系，否则我该如何回答这个问题呢？输入是清楚的。我把它放在了正确的优先位置。我真的 ____？我变好了吗？人们在使用该产品吗？NPS 分数如何？这些分数有什么影响？

你可以问各种各样的问题来确定我是否做得更好，因为为了快而快并不有趣。快速实现价值，重要的是，理解价值。这就是为什么我在一年前 Forrester 开始谈论价值街管理，成功的衡量标准最终是企业从努力中受益。如果没有我们所说的那种可见性，整个努力就会大打折扣。

**Ashley:** 我以不同的身份与一群首席信息官一起工作。我和开发者社区一起工作。一个共同的主题是，我认为 DevOps 正在消除组织的障碍，我们都在真正的价值链或真正的管道链中开始更紧密地合作，所有社区都在努力或想知道更多–他们如何更有效地与业务部门和 C 套件沟通。

这不是针对我，但我最近在这里的一个播客上看到了这个问题，我说，“好吧，如果你是一名开发人员，你想引起你的首席执行官的注意，说‘我有一个想法，可以增加收入，为我们带来新的收入流，降低成本，让我们更快地上市’，这些商业术语中的任何一个，他们都会立即把你放在日历上。”我认为这就是你要去的地方，也是价值流管理分析师要去的地方。

戴维斯:总共。这正是他们要去的地方，也是人们想要回答的关键问题。它涉及到速度。这涉及到质量。它涉及人们想谈论的关于开发运维的所有事情，但必须在意识到对业务的影响的背景下进行。从很多方面来说，当我第一次来到 Plutora，并与创始人兼首席执行官谈论 Plutora 的工作时，我的第一反应是，“这听起来非常像 It 销售人员。”这个问题和那个问题非常相似。

记录系统，就像销售人员提供的，在这个平台内，例如我们的平台，它连接所有的点，并自动提供 C 套件想要了解开发人员发生了什么而不必改变他们正在工作的工具的那种指标。如果他们使用吉拉，如果他们使用 Jenkins，如果他们使用 Selenium，无论他们使用什么工具，他们都可以继续使用该工具，但是平台、管理平台将获取所有信息并将其关联起来，并向管理层提供关于所提供的确切价值的可见性。以及发布周期的度量标准。

当我有一个 bug 时，修复它需要多快？如果是旧的，我需要多长时间来修复？我是否在质量等方面变得更好了？这些都是人们想知道的事情，他们想比较进入生产的发布范围和进入计划的发布范围，看看他们在各种级别上做得如何。为了能够让信息自然地流向 C 套件和他们之间的每个人，而不需要从业者在他们的日常工作中做任何不寻常的事情，你有了这个完美的场景。

如果你愿意的话，你可以在整个软件制造过程中进行展示，让管理层能够看到它，而不必改变开发人员的方式和可以提供的信息，我可以建立利益相关者。我可以有一个安全涉众，审计涉众，无论我的组织的独特需求是什么。让我们明确一点，软件是当今组织的竞争优势。

阿什莉:它统治世界，对吗？

戴维斯:它统治着世界，它是——是的。不管是好是坏，随着它变得越来越重要，你不能只是一家碰巧使用软件的银行。你必须是一家选择进入银行业的软件公司。

阿什利:是的。科技公司。

戴维斯:如果你——是的。你会失业的。

阿什莉:嗯，鲍勃，我很感谢你花时间在播客上。我知道我们可以继续讨论这个问题。也许某个时候在半月湾的芭芭拉那里，我们可以找个时间吃点东西。

戴维斯:听起来是个很棒的计划，伙计。任何时候。任何时候你都可以打我那里。

阿什莉:食物和健康饮料，真好吃。是啊。当然了。

戴维斯:这就对了。
