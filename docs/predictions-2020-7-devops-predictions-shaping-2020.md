# 2020 年预测:7 个 DevOps 预测塑造 2020 年

> 原文：<https://devops.com/predictions-2020-7-devops-predictions-shaping-2020/>

2019 年对于 DevOps 从业者来说是令人兴奋的一年，DevOps 的采用继续快速增长。大多数组织正在执行或评估他们的 DevOps 策略。那么，2020 年将给开发运维、基础设施和应用自动化带来什么？以下是 DevOps 对新一年新兴趋势的七个预测。

## 客户体验

CX 将成为 DevOps 旅程的指路明灯，而不仅仅是技术。这与早期纯技术专家的热情相比是一个巨大的思维转变。随着 DevOps 的成熟，我们必须提醒自己，这一切都是为了让业务更快，而不仅仅是酷技术。

*立即利用:*

*   将 DevOps 成就与业务需求和结果联系起来，以获得管理层的认同和预算。
*   成功的 QA 团队将开始把他们的角色视为客户体验的守护者，而不仅仅是测试员。

## **信任**

每个人都会谈论信任——不仅在零信任网络的背景下，而且在使开发人员和运营人员能够相互信任作为开发运维基础的背景下。

*立即利用:*

*   积极投资创造信任文化。找出对你的价值流缺乏信任的原因，并把信任作为 2020 年的目标。
*   在价值流管理、应用和基础设施自动化方面，鼓励产品层面的对话并定义方法和所有权。这是每个人都在努力的基础。
*   投资于试运行环境—需要试运行环境来确保在投入生产之前一切正常。对于我们中的许多人来说，直接跳到自动部署到生产是一个很大的进步。投资于试运行环境自动化可以降低风险，这将是部署到生产环境的第一步。

## 价值流

DevOps 工具链即将消失。2020 年，我们将思考价值流。这个行业正在走向成熟，不再是将几十个开源工具链接在一起并试图让它们一起工作的概念。前几年，许多人会很坦然地说，如果他们有一个 CI 系统，这意味着他们“做 DevOps”现在情况不再是这样了。DevOps 不仅仅是 CI，DevOps 不能建立在孤岛上。成熟的开发运维需要考虑整个价值流——从规划新功能和业务需求，一直到客户获得更多价值和美妙体验。

2019 年，Gartner 宣布这是 ARO MQ 的最后一年，因为这不是正确的视角。在 2020 年，我们最终获得了价值流的系统级视图。更多的参与者正在宣布端到端价值流管理，包括 Atlassian 的新 Forge 平台、GitLab 和 CloudBees。公司正在走向成熟并采用系统级解决方案。

*趁现在:*

*   规划平台运营。系统级的价值流视角需要在整个价值流中而不仅仅是在小仓库中拥有编排和自动化的角色。

## SRE 的角色将继续成熟

始于谷歌将软件开发实践引入运营的 SRE 角色将继续演变。不断增长的数据量、速度和多样性将使 AIOps 解决方案蓬勃发展。此外，自动化水平的提高将需要在不失去控制的情况下处理动态基础设施和应用程序的智能方法，并能够跟踪变化直到编码人员。

*立即利用:*

*   如果您尚未实施 SRE 角色，请考虑一下 2020 年 SREs 如何成为您团队的一部分。定义 SRE 的职责范围和 KPI，以及 sre 如何与团队沟通。这是一个集中的职位还是 SREs 是团队的一部分？
*   确保为 sre 提供他们成功所需的工具，包括:
    *   你使用的可观察性解决方案是什么？今天谁在使用它们？你是怎么分析的，又是怎么沟通的？
    *   您目前使用的生产环境协调、部署和配置工具有哪些？如何将它们与生产前的应用和基础架构自动化联系起来？

## 无缝基础设施

我们必须为我们的团队提供无缝的基础设施。内部客户也是客户，他们的体验也同样重要。自助服务和授权团队将是需要在 Dev 和 ITOps 之间建立的桥梁的主题。公共云、Kubernetes 和[无服务器技术](https://devops.com/infrastructure-as-code-and-six-key-automation-concepts/)将会增长。虽然它使自动化比以往任何时候都更加可行，但重点将转向自助服务和最终用户。随着越来越多的公司采用类似的方法，开发内部专业知识的需求将会减少，我们将会看到更多现成的解决方案。

*立即利用:*

*   绘制基础设施在整个价值流中的使用方式，并在产品层面统一方法。
*   确保您构建的自助服务解决方案高度关注最终用户，以实现速度和控制的平衡。在确定基础设施自动化的方法时，确保开发人员和 ITOps 具有同等的代表性。

## 基础设施自动化

随着大量人工智能和人工智能的出现，基础设施自动化将变得更加智能。确保自动化得到适当的治理也将变得越来越重要。考虑安全性、合规性以及成本。

*立即利用:*

*   评估整个价值流中的基础设施自动化实践。谁构建自动化？有哪些团队和系统消耗它？您如何实施安全最佳实践(例如，公共存储库中没有密码，使用正确的数据集进行测试，自动化蓝图的合规性)？您如何实施成本控制(例如标记、审查云支出报告)？
*   在失去控制之前投资控制机制，尤其是在公共云自动化方面。如果您没有合适的治理，可以考虑将治理自动化，而不是采用传统的云管理方法。投资于基础设施的自动停用、自动标记、基础设施消费数据分析，并将其与业务需求、机密和密钥管理以及访问控制联系起来。

## SaaS

开源很好。SaaS 很棒。虽然大型开源项目引领潮流(K8S、Github ),并得到行业巨头的大力支持和资助，但较小的项目越来越难以扩展和整合，尤其是在试图确保企业就绪性时。“自己动手”的需求正在逐渐消失，因为越来越多的供应商提供产品和服务来支持开发运维，这些产品和服务具有清晰的业务和支持模型，同样重要的是，具有与生态系统的 OOB 集成。仅仅 Rest API 已经不够了。

*立即利用:*

*   如果您的企业就绪性要求使您难以扩展 DevOps 计划，请考虑评估新的 SaaS 产品，这些产品消除了早期技术密集型实施中常见的大量繁重工作和粘合工作。如果您关心安全性、成本和技能要求，今天您可以找到专门构建的工具来支持大规模企业开发运维。好消息是，大多数供应商都很清楚集成到您的生态系统中的重要性，从而更容易实现您的端到端价值流愿景。
