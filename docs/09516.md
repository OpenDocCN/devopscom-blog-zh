# TechStrong Con:开源软件社区需要安全帮助

> 原文:[https://devo PS . com/tech strong-con-开源-软件-社区-需求-安全-帮助/](https://devops.com/techstrong-con-open-source-software-community-needs-security-help/)

显著改善开源安全状态的唯一方法是，如果更多从开源项目中受益的组织承诺提供更多资源来实现这一目标。

在虚拟的 techs tong Con 活动上，开源和 DevOps 小组的高管们呼吁企业 IT 组织做出更多贡献，以确保开源软件的完整性。

各种规模的组织现在都直接或间接地从开源软件中受益。医疗转录服务提供商 Remote Operations 的高级软件工程师杰弗里·巴恩斯(Jeffrey Barnes)说，这些组织有责任以任何方式参与该软件的开发。“我们需要回馈社会，”他说。

Barnes 补充道，这些努力不一定需要贡献实际的代码。他指出，许多开源项目需要文档和测试帮助，因为项目的实际维护者相对较少。

工业制冷专家首席信息官 Tom Sweet 指出，挑战在于许多企业 IT 组织不了解如何参与开源软件开发过程。一个组织可能会下载一个开源软件，然后扩展它。Sweet 说，组织不应该继续维护他们自己的分支，而应该将代码贡献给项目，以减轻维护者的总体 IT 负担。

Sweet 说:“许多组织都在努力回馈社会。“这是一种进化。”

组织显然需要将贡献代码返回到开源软件项目，使其成为 DevOps 工作流的自然扩展。与此同时，Linux 基金会的一个分支——开源安全基金会(OpenSSF)正在[关注 10 个投资流](https://securityboulevard.com/2022/05/openssf-seeks-150m-to-address-open-source-software-security/?),总共需要超过 1 . 5 亿美元的资金来推动开源软件项目的维护者更多地采用 DevSecOps 最佳实践。

核心问题是，许多项目是由少数程序员维护的，这些程序员自愿贡献他们的时间和精力来构建其他人可以免费使用的组件。像任何其他开发人员一样，这些人拥有的安全专业知识是有限的。确保项目和软件安全的责任在于决定部署该软件的组织。

不幸的是，许多 IT 供应商和大型企业 IT 组织重用这些代码，而没有为项目做出任何有意义的贡献——无论是在融资方面还是帮助开源维护者发现和修复漏洞方面。

目前还不清楚商业和 It 领导意识到他们的组织实际上是多么依赖开源软件。他们中的许多人认为开源项目是由 IT 供应商支持的，而实际上，有许多项目依赖于一小撮自愿贡献他们的时间和技能的开发人员的努力。

当然，如果没有足够数量的开发人员来开发开源软件，许多组织可能会很快决定不采用开源软件。然而，在一个 DevOps 团队和开发人员经常下载软件的时代，实施这些政策会有问题。企业 IT 组织能做的最好的事情是更多地参与开源软件社区，而不是更少。
