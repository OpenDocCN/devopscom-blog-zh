# Rezilion 为动态 SBOM 工具添加了 Windows 支持

> 原文:[https://devo PS . com/rezi lion-adds-windows-support-to-dynamic-sbom-tool/](https://devops.com/rezilion-adds-windows-support-to-dynamic-sbom-tool/)

Rezilion 在其动态生成软件材料清单(SBOMs)的工具中增加了对 Windows 应用程序的支持[。](https://www.prnewswire.com/news-releases/rezilion-expands-dynamic-sbom-capability-to-support-windows-environments-301671749.html)

Rezilion 首席执行官 Liran Tancman 表示，除了现有的对 Linux 应用程序的支持，现在还可以实时分析构成 Windows 应用程序运行时环境的所有组件。

他补充说，这种能力还使得在数百万个文件和数千个主机、容器和应用程序中搜索和定位组件(如 Log4j 漏洞)成为可能。

Rezilion 创建了一个可搜索的专有下一代漏洞数据库(NGVDB ),可以识别哪些问题会影响特定的类和功能。除了提出补救建议之外，该功能还减少了 IT 团队不得不追踪的误报数量。

该公司提供的 [SBOM](https://devops.com/?s=SBOM) 工具基于这种能力，能够洞察可能被网络犯罪分子利用的应用程序的完整攻击面。

一般来说，Windows 应用程序的组件更难跟踪和分析，因为在过去几十年中创建了许多不同的框架来构建它们。然而，随着超过一半的应用程序在 Windows 上运行，为这些应用程序创建 sbom 的需求已经成为一个更紧迫的问题，特别是在拜登政府发布行政命令要求联邦机构为他们使用的每个应用程序提供 sbom 之后。

目前还不清楚有多少企业 It 组织会效仿，但普遍的预期是 SBOMs 将很快比以前更加流行。在一系列高调的网络安全违规事件之后，现在对软件供应链的审查比以往任何时候都多。

当然，挑战在于一旦在应用程序中发现漏洞，就要确定首先修复哪些漏洞。有太多的漏洞需要一次修补，因此 DevOps 团队需要能够识别哪些漏洞是最严重的。多年来，网络安全团队一直在创建充满漏洞长列表的电子表格，但通常没有提供漏洞严重程度的上下文。因此，开发人员会分配一小部分时间来应用补丁，这些补丁是基于直觉和复杂程度而不是严重性来划分优先级的。

在大多数构建软件的组织能够一致地创建准确的 SBOMs 之前，可能还需要一段时间。消费这些标准业务手册的组织可能需要更长的时间才能拥有实施这些手册的流程。理论上，如果有必要，组织应该能够根据 SBOM 中包含的组件列表阻止软件被部署。然而，这个决定并不总是直截了当的。在应用程序中包含潜在易受攻击的组件通常有一系列需要考虑的原因。

不管怎样，应用程序的整体安全性将会稳步提高。唯一的问题是需要经历多少痛苦才能实现这个目标。
