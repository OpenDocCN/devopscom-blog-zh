# 如果你是第一个 SRE 雇员，给你 5 个建议

> 原文：<https://devops.com/5-tips-if-youre-the-first-sre-hire/>

现场可靠性工程师(SREs)有相当多的 [任务](https://rootly.com/blog/what-is-an-sre) 要兼顾，不管他们在哪里工作，也不管他们的公司有多长时间的 SRE 实践。但是如果你是第一个加入一个组织的人——考虑到趋势是 [渗透到越来越小的公司，现在有很多 SREs 你会面临一组特殊的挑战。](https://rootly.com/blog/what-is-an-sre)

你可能会发现很难得到其他技术团队对 SRE 的认可。你可能很难知道如何让你的老板相信公司雇佣 SRE 的决定是值得的。你可能担心你会被过度扩张，因为你是唯一的 SRE。诸如此类。

这五个技巧是给那些发现自己处于公司先锋位置的现场可靠性工程师的。无论你是一名经验丰富的 SRE，刚刚被一家以前没有经验的公司聘用，还是你刚到 SRE，你的第一份工作恰好是在一家对 SRE 这个角色来说也是全新的公司，以下最佳实践将帮助你尽可能平稳地完成过渡。

### 尽早并经常收集 SRE 指标

首先，一旦老板或工程师同事提出要求，你需要准备好证明公司对你新角色的投资是合理的。

尽可能早地收集有关 SREs 的技术和业务 影响的指标。你的目标应该是量化你是如何使组织受益的，这样你就可以鼓励更多的支持你自己和其他可能在你之后被雇佣的人。

除此之外，收集指标对于建立早期反馈也很重要，反馈是关于哪些 SRE 战略是成功的，哪些可以改进。因为你是从零开始建立一个 SRE 实践，所以尽早并经常获得这种洞察力是至关重要的。

### 平等对待开发人员和 IT 工程师

当你开始这个角色时，你可能会发现现有的工程师对你站在哪一边有些不确定。你是去支持开发者还是 IT 工程师？

答案当然是两者皆有。尽管相对于 IT 运营团队，软件工程师更倾向于借鉴软件工程师的剧本，但 SRE 的目标应该是授权并支持这两个团队。因此，除非你得到明确的指示，只与一个组或另一个组打交道，否则让自己成为 it 组织中每个人的伙伴。

### 向非技术团队介绍自己

sre 大部分时间都在用技术工具做技术上的事情。但最终，在大多数情况下，他们这样做的唯一原因是通过提高非技术用户使用的系统的可靠性来支持他们。

作为公司的第一个 SRE，你应该花时间去了解这些非技术利益相关者，并询问他们认为组织中最大的可靠性问题是什么。尽早建立这些关系将有助于你在整个组织中传达你的角色价值。在[事件管理](https://devops.com/the-evolution-of-incident-management/)日常工作中，当你需要与非技术利益相关者以及其他工程师交流时，它也可能派上用场。

### 构建您的 SRE 工具集

对于任何从零开始建立业务的 SRE 来说，决定使用哪些工具显然是第一步。但是我们没有把它放在这个列表的第一位，因为在使用一组特定的工具之前，了解组织的其他部分是很重要的。

当你会见组织的其他成员时，详细询问他们使用哪些工具来满足可观察性、安全性和事故管理等需求。然后，通过考虑关于哪些工具工作得好，哪些不好的反馈，选择您将部署哪些工具来支持 SRE。做好准备，这些工具不会完全符合你的个人喜好。您可能需要优先考虑其他组织的偏好。

### 计划壮大团队

除非你已经被一家非常小的公司雇佣，否则随着时间的推移，这家公司很可能会雇佣更多的销售代表(当然，前提是你证明了这个职位的价值)。

为了这个目标，计划一下团队将如何扩展。考虑是否使用 [嵌入式 SREs](https://rootly.com/blog/the-pros-and-cons-of-embedded-sres) 或者创建一个专门的 SRE 团队。还要考虑如何记录你公司的 SRE 实践，并确保它在单个工程师来来去去时保持平稳运行。

### 结论

新入职的 SRE 的工作并不轻松，尤其是如果你是公司里第一个接替这个职位的人。但是，通过对你的工作方式、你收集的数据以及你如何向 SRE 介绍公司进行深思熟虑和战略性的思考，你就为自己和你未来的同事走向成功做好了准备。
