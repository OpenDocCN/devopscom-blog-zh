# BMC 为大型机工具组合增加了对 DORA 指标的支持

> 原文:[https://devo PS . com/BMC-adds-support-for-Dora-metrics-to-mainframe-tools-portfolio/](https://devops.com/bmc-adds-support-for-dora-metrics-to-mainframe-tools-portfolio/)

BMC 本周宣布在其面向大型机环境的 DevOps 工具组合中增加了对 DevOps 研究和评估(DORA)指标的支持。

BMC 高级副总裁兼智能 Z 优化和转型总经理 John McKenny 表示，最新版的 BMC Compuware zAdviser 现在包括一个 DORA KPI 仪表板，该仪表板与谷歌旗下一个部门定义的广泛采用的衡量 DevOps 效率的框架相一致。

[DORA 指标](https://devops.com/?s=DORA+metrics)跟踪部署频率、失败率、变更前置时间和平均恢复时间(MTTR)。McKenny 指出，跟踪这些指标对于拥有大型机的 IT 团队来说至关重要，因为他们需要确保应用程序继续部署在该平台上，而不是被另一个平台侵占，该平台被认为提供了更快的构建和部署替代方案。

BMC 还在其 BMC Compuware Abend-AID 产品中增加了对自动 webhook 通知的支持。这包括诊断和根本原因分析，以更快地解决测试和生产中的缺陷。此外，BMC 还提供了一个到由 Venafi 创建的机器身份管理平台的连接器。这种集成有望消除与证书相关的中断，这种中断会破坏应用程序开发计划。

总的来说，DORA 指标被用来识别 DevOps 团队可以改进的地方。由每个组织来决定这些度量标准在多大程度上被用来指导团队而不是强制执行生产力指令。然而，考虑到可用开发运维人才的整体短缺，很少有组织采用指标来迫使开发运维团队改进。事实上，没有一套通用的 DevOps 目标是每个组织都在努力实现的。相反，考虑到他们自己的目标和内部文化，大多数组织试图确定哪一套 DevOps 最佳实践最有意义。McKenny 指出，在大型机的情况下，大多数组织在开始更广泛地使用 DevOps 之前，先从几个项目开始。

当然，DORA 框架定义的指标只是 DevOps 团队应该跟踪的全部指标的一个子集。McKenny 指出，例如，BMC 可以跟踪工具的使用情况，为开发运维团队提供关于哪些工具可能不再相关或不再需要的见解。

大型机环境使用基于瀑布的方法来构建和部署应用程序也有很长的历史，根据所构建和部署的应用程序的类型，许多大型机环境将继续与 DevOps 实践一起使用这些方法。在其他情况下，使用 DevOps 实践构建的分布式应用程序正在与大型机应用程序集成，这些应用程序现在需要使用相同的工作流进行更新。大型机不再是一个与分布式计算环境完全隔离的平台。

McKenny 说，主要的挑战是许多使用大型机的组织经常高估采用 DevOps 最佳实践的难度。不管平台是什么，毫无疑问，所有应用程序的构建和部署的整体速度将继续增长。事实证明，大型机也不例外。
