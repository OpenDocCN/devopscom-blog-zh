## DevOps 应用于企业应用

企业应用程序帮助组织解决业务问题。企业应用程序通常被设计为与组织内使用的其他企业应用程序接口或集成，并被部署在各种网络(internet、intranet 和公司网络)上，同时满足安全和行政管理的严格要求。

专有企业应用程序通常由组织内的专业 IT 开发团队在内部设计和部署。然而，企业可能会将应用程序的部分或全部开发工作外包出去，并将其带回内部进行部署。

大型企业应用程序的规模和复杂性，尤其是没有采用模块设计的老式整体架构应用程序，在实现 DevOps 时是一个挑战。鉴于企业应用程序的人员、流程和技术组件的大规模和分布式性质，建议企业聘请经验丰富的 DevOps 顾问来确定 DevOps 的战略要求、路线图和计划，充分考虑所有三个维度(人员、流程和技术)以及 [DevOps](https://devops.com/nine-pillars-of-devops-best-practices/) 的[九个](https://devops.com/nine-pillars-of-devops-best-practices/) [支柱](https://devops.com/nine-pillars-of-devops-best-practices/) [，然后再开始主要的 DevOps 转型之旅。](https://devops.com/nine-pillars-of-devops-best-practices/)

尽管很复杂，但通过正确的策略和计划，DevOps 可以为企业应用程序带来巨大的投资回报。亚马逊和 Adobe 只是众多大型企业中的两个例子，它们从在整个企业中应用 DevOps 中受益匪浅。

## **应用于软件服务的 DevOps】**

我经常听到有人问:“DevOps 是否像应用于产品一样适用于软件服务？”这些人通常只在以产品为导向的组织中工作过，比如生产嵌入软件的产品的公司。

具有讽刺意味的是，我经常听到人们问相反的问题，“DevOps 适用于带有嵌入式软件的产品吗，就像它适用于服务一样？”这些人通常只在这样的组织中工作过，在这些组织中，软件作为服务通过内部网或互联网有偿或无偿使用。这些组织的例子有 IT 部门(甚至是制造商的 IT 部门！)、银行、保险公司、医院、金融贸易公司等政府机关和企业。

如果你读过这个博客系列的前几部分，你已经知道这两个问题的答案了。这两个问题的答案显然都是“是的！”云服务对 DevOps 来说有双重意义，因为正如下一章所述，DevOps 不仅将云服务作为首选基础设施，而且云服务还在创建和部署软件以实现云组件的过程中使用 DevOps。

当我们谈到开发运维与服务的话题时，我想帮助澄清一些困惑，即“较新的”开发运维是否会使“较旧的”IT 服务管理(ITSM)过时或变得不那么重要。这种混乱之所以存在，是因为 ITSM 经常被认为是由 Ops 管理的看门人，以防止 Dev 失控。事实上，虽然有些组织可能已经这样实施了 ITSM，但这不是 ITSM 的目的。

“DevOps 倾向于关注软件交付生命周期以及构建和交付软件，”ITSM 学院院长 Jayne Groll 说。就其本身而言，ITSM 倾向于更广泛的关注；它实际上关注客户参与的所有方面，从履行标准服务到持续的支持和运营活动。所以，ITSM“几乎就像 DevOps 的左右两端”

## 通过 DevOps 使用 ITSM

以下是将 ITSM 与 DevOps 结合使用的一些推荐工程实践:

帮助 DevOps 专业人员了解 ITSM 的培训，反之亦然，将有助于两个团队找到结合点——不仅在技术上，而且在流程上。
:找到自动执行服务变更批准的方法，实施已知的、好的变更，以实现敏捷性、迭代和创新。同时，阻止高风险、高影响的变更，这些变更在提交给客户之前可能需要不止一个贡献者的签字。
·自助服务请求、自动化流程和 IT 流程编排现在是 IT 服务管理行动的大门。这些可以并且应该在后台发生，由我们的 DevOps 工具自动创建、维护和测量。
拥抱敏捷服务管理，采纳敏捷价值观，将类似 Scrum 的开发实践应用到流程设计和改进中。旨在进行更频繁的增量更改，而不是以传统方式设计和实施完整的端到端流程。
