# 当自助 DevOps 忘记它为谁服务时

> 原文:[https://devo PS . com/when-self-service-devo PS-forgets-who-it-served/](https://devops.com/when-self-service-devops-forgets-who-it-serves/)

多年来，自助服务一直被吹捧为 DevOps 的救星。为什么？好吧，您组织中的每个部门都有自己的理由将自助开发置于其愿望清单的首位:

●运营团队需要确保高可用性、性能和安全性
●开发团队需要速度、自给自足和透明度
●安全团队需要护栏、合规性和问责制
●当然，除此之外，高管团队还需要投资回报、战略和洞察力。

这些目标通常会导致关系紧张，尤其是在开发人员和运营人员之间。验证安全性需要花费开发人员通常没有的时间。一旦规模化，即使最快的手动流程也会崩溃。例如，美国国防部拥有近 20 万名内部和外部开发人员。

自助式开发运维平台承诺为软件交付提供虚拟自助式服务亭，团结团队，为独立开发人员的访问和配置创造顺畅的途径。根据来自 Puppet 的[DevOps 状态](https://puppet.com/resources/report/2021-state-of-devops-report/)报告，自助式 DevOps 是“高度发展的”devo PS 组织的标志。

许多自助服务解决方案都是可用的，但是有人真正弄明白了吗？毕竟，如果每个人都对他们的自助方式感到满意，为什么他们还在寻找答案呢？

### 自助开发的方法

真正的自助式开发运维承诺了广泛的好处:它让运维不再急于灭火，而是转向全局战略；它通过自动化、安全护栏和清晰的问责制简化了合规性。此外，还有巨大的资本效率的承诺。

毫不奇怪，如此多的供应商正在涌入这个领域，提供如此多的方法，让芭斯罗缤嫉妒不已:

●内部开发者门户
●共享服务平台
●工作流自动化
●聊天平台/聊天机器人
●服务目录
● API 市场
●云能力中心

但让我们面对现实:在现实中，这些方法大多都失败了。当自助服务没有发挥应有的作用，公司在其服务请求队列中落后时，许多 DevOps 团队默认采用旨在加快运营但实际上使公司面临攻击的高风险做法:

●过度特权和[过度许可的账户](https://devops.com/devops-and-security-in-a-cloud-native-world/)，给予任何人广泛的访问权限，就像 2020 年的网络安全管理软件产品泄密事件
一样●不活跃的账户为网络犯罪分子敞开大门，就像我们在 2021 年的殖民地管道泄密事件中看到的那样。

您可能没有意识到的是，这两种风险如此普遍，以至于 IBM Security X-Force 渗透测试人员成功地利用它们侵入了 99%的客户端云环境。因此，当我们说自助式开发运维失败时，这是一次大失败。

那么解决办法是什么呢？为什么真正的自助开发如此难以实现？

### 他们失败的原因

实际上，今天的自助式开发运维平台失败的原因有很多:

●它们会破坏最终用户的工作流程，迫使用户改变行为并学习新的流程。他们强迫开发人员切换环境，这会降低工作效率。
●他们需要领域专业知识来运营，减少了可以使用他们的目标受众。
●他们严重依赖[人在回路](https://devops.com/5-ways-to-transform-dataops-with-human-in-the-loop-automation/)的参与，经常创造更多的工作。
●他们需要付出巨大的努力来更新和维护工作流，并满足或超过 SLA。

最重要的是，使用当前的自助式 DevOps 解决方案，用户经常会盲目行动。他们不知道自己不知道的事；例如，如何定位、搜索、识别、创建和使用资源和工作流。

例如，想象一下，选择自助结账来避免在杂货店收银台排长队。但是后来…

a)你被堵在某人后面，疯狂地试图扫描一张皱巴巴的过期优惠券。你被一个不懂条形码的人堵在后面，并且扫描了他们所有的物品两次。
c)你忘记了酒精需要年龄验证，导致警报铃和闪光灯开始在收银台上方响起(经验之谈；尴尬)。
d)以上皆是。

在所有这些场景中，服务员必须跑过来修理东西，这就破坏了自助服务的全部意义。如果自助服务即使在出错时也易于使用，那么它才是自助服务。

对于大多数所谓的自助式 DevOps 解决方案，如果出现问题，您需要重新等待，将操作员拉回到循环中，就像旧系统一样。

### 真正的自助式开发运维是什么样的

要实现真正的自助式开发运维，您需要具备一些要素。

**无头 UI**
大多数开发者都是(喘气！)人类。所以最好的界面就是根本没有界面。为什么不让开发人员使用现有工具，如 Slack、微软团队或命令行界面，用自然语言自由表达他们的需求呢？
**工作流程**
当流程没有意义或者采取了多余的步骤时，开发人员会感到沮丧。让我们采用实际处理所有细节的工作流——这样所有开发人员需要做的就是请求(用他们自己的话说)被授权访问资源、信息和工作流。
**智能**
对于开发者来说，没有什么比不知道去哪里找你需要的信息更令人沮丧的了。由人工智能提供支持的真正的自助式 DevOps 将使用您现有的数据来提供一个真实的来源，其中包含开发人员需要的所有答案和洞察力。
**最终用户反馈循环**
理想的自助式 DevOps 平台会要求开发人员提供反馈，然后利用这些反馈来改进平台本身。首先，因为每个人都喜欢听到自己的声音，其次，因为这加强了平台的作用，即通过改善最终用户体验来服务于 DevOps。Waze 等应用程序也使用了类似的反馈循环，该应用程序使用社交导航来不断提高其最终用户体验的质量。自助开发运维是下一个目标。

一旦您完成了所有这些，您就可以更高效地工作，提供即时的按需协助，这样您的开发和运营团队就可以不间断地工作。

您如何知道何时实现了真正的 DevOps 自助服务？当你忘记你曾经以其他方式做过事情——并且不会为了一百万美元而回头。当你有一天醒来，意识到你的平台不仅让你建立了信任和责任感，还让你的所有团队一起工作…而且——(喘息)我敢说吗？—玩得开心。

这将是你购买的最后一个 DevOps 平台。
