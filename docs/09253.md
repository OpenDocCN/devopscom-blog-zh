# 谷歌与 GitHub 结盟保护软件供应链

> 原文：<https://devops.com/google-allies-with-github-to-secure-software-supply-chains/>

谷歌今天透露，它一直在与 GitHub 合作创造一种防伪方法来签署源代码，这是正在进行的更好地保护 T2 软件供应链的努力的一部分。

谷歌开源软件供应链安全技术负责人鲍勃·卡拉威(Bob Callaway)表示，这种方法的原型是用 Go 编程语言编写的，使用 GitHub Actions workflows 进行隔离，使用 Sigstore 提供的代码签名工具进行验证，从而生成不可伪造的出处。他说，利用 GitHub runners 运行作业的项目现在可以实现软件工件供应链级别(SLSA)框架定义的第三级安全，以确保软件工件是真实可信的。

可重用的工作流还可以防止维护人员可能的干扰，否则他们会试图以干扰构建者的方式定义工作流。与可重用工作流交互的唯一方式是通过它向调用工作流公开的输入参数，这可以防止维护人员通过变量、步骤、服务和默认值来更改信息。

Callaway 说，目标不仅是确保单个软件工件的出处，而且使识别哪个构建系统被用来创建那个工件成为可能。除了与 GitHub 合作，Callaway 补充说，谷歌希望将这种方法应用于保护组织用来构建应用程序的其他构建系统。

总体而言，Callaway 说，除了教育开发人员如何构建更安全的软件，谷歌还在努力将安全功能嵌入 DevOps 平台，以确保软件供应链的完整性。

为了防止一个作业篡改另一个作业使用的工件的可能性，Google 的方法创建了一个可信通道。作业输出发送哈希，然后使用哈希来验证通过不可信工件注册中心接收的二进制文件。GitHub 刚刚选择支持的 OpenID Connect (OIDC)协议随后被用于向外部服务证明工作流的身份，例如 [Sigstore](https://www.sigstore.dev) ，方法是为每个运行者附加一个唯一的 JSON web 令牌(JWT)。令牌包含关于工作流身份的可验证信息，例如调用者存储库、提交散列、触发器以及当前工作流路径和引用。

这使得工作流可以向 Sigstore 提供的 Fulcio 根证书颁发机构证明其身份。Fulcio 签署了一个短期证书，证明了在 runner 中生成的临时签名密钥。签名记录保存在 Sigstore 的透明日志 Rekor 中。然后，用户可以依靠签名证书以经过验证且不可伪造的方式来验证出处。Callaway 指出，这种方法意味着维护者不需要管理或分发用于签名的加密密钥，直到现在，这种方法仍使大规模管理代码签名变得具有挑战性。

在一系列涉及开源软件的高调安全违规事件之后，谷歌和其他公司在一套 DevSecOps 最佳实践的背景下，显著增加了用于保护开源和专有软件的资源。开发人员可能还需要一段时间才能从这项工作中普遍受益，但很明显，从开发人员使用的工具到开发团队管理的构建系统，一切都在取得实质性的进展。
