## 副本

*史蒂夫·伯顿

*史蒂夫是新的持续交付即服务平台 Harness 的产品营销副总裁。在 Harness 之前，Steve 曾在 AppDynamics、Moogsoft 和 Glassdoor 负责产品营销。早在 2004 年，Steve 就在 Sapient 开始了他的 Java 开发职业生涯。当他不玩技术的时候，他通常会看 f1 或者在网上研究汽车。推特:@BurtonSays** 

Alan Shimel: 大家好，我是 Alan Shimel，这里是 DevOps.com 的另一位 DevOps Chat。今天的聊天对象是史蒂夫·伯顿，他是一家令人兴奋的新公司的产品副总裁，这家公司今天刚刚成立，名为“Harness”史蒂夫，欢迎。

史蒂夫·伯顿:谢谢你，艾伦。很高兴来到这里。

Shimel: 很高兴你能来，嘿，今天是开公司的好日子，Steve。

**伯顿:**是的。当然可以。

当然，你知道，我想起了当我卖掉我创办的第一家公司时有人告诉我的话，他们说:“公司就像船，最好的两天是你买下它们的那一天和你卖掉它们的那一天。“所以这是一个好消息——对你来说这是一个重要的日子。说到这里，史蒂夫，在我们进入什么是 Harness 和市场之类的话题之前，你实际上是那里的创始团队之一，我想，这个团队实际上是当年帮助创建 AppDynamics 的许多人的聚会。正确吗？

**Burton:** 是的，当然，这里有几个 AppDynamics 的人。我们也有来自苹果和谷歌的人，所以这是一个很好的组合。但是，是的，肯定的。有 Jyoti，有我自己，Greg Howard，和 Randy Boysen，所以有–是的，这就是我们喜欢做的，建立新的公司。

希梅尔:是啊。让乐队重聚。那么，Steve，关于 Harness，你们已经采取了一种有趣的方法，你知道，你看到了 CICD 市场的巨大机会。正确吗？

伯顿:是的。是的。我的意思是，Harness 主要是为作为服务的连续交付而构建的，所以是 CICD 的 CD 部分，我们认为那里有很大的机会。有很多令人兴奋的事情，有很多项目和客户，有很多复杂的问题需要解决，所以，是的，这就是我们成立公司的目的，帮助客户解决问题。

Shimel: 当然可以。Steve，我的意思是，在 DevOps.com，我们显然非常密切地关注 CD 领域，当然也不乏公司参与进来，对，没有指名道姓，从资金雄厚的初创公司到传统的 800 磅重的大公司，但没有人真正占据主导市场份额的事实可能表明，这是因为没有人真正推出杀手级 CD 应用程序，如果你愿意的话，或者如果不是应用程序，可能是交付方式和其他一切。这是促使你们走上这条路的原因之一吗？

伯顿:是的，我是说，当然。我的意思是，如果你和 Jyoti 交谈，他创建公司的一个关键原因是，很明显，当他是 AppDynamics 的首席执行官和创始人时，他会见了许多企业和大客户，他们告诉他，“嘿，看，我们有数百名 DevOps 工程师，但他们中的许多人都在花时间编写脚本和构建自动化平台。因此，没有真正的解决方案来帮助我们像脸书或谷歌一样，可以持续交付。”所以这真的是 Jyoti 后退一步思考的动力，“这个问题真的还没有解决。”

希梅尔:是啊。不，我认为你击中了一些东西，因为有一些-不是反弹，但在一些 DevOps 频道上有一些谈话，“你知道吗？DevOps 并没有像我们想象的那样成功，许多组织也没有获得他们希望的成功。”坦率地说，很多都是主观的，而不是客观的，也许其中一些也与永远无法实现的被夸大的期望有关。

但是，也就是说，你发现了一些我认为很重要的东西，很多时候，当我们谈论 DevOps 团队和实施 DevOps 时，我们花了很多时间来搞清楚自动化，对吧，以及编写自动化脚本。无论是自动化测试、自动化集成、自动化配置，还是其他什么，都有点像打基础，而不是实际做工作。你可能需要先做一些基础工作，对吗，史蒂夫？我的意思是，没有人——我不是说，但是，有了像 Harness 这样的 SAS 解决方案，您可以不作弊，但您可以在这方面领先一步。

**Burton:** 是的，当然，我认为我们已经尝试做的一件事是简化持续交付，所以你可能需要 DevOps 工程师团队做的许多脚本和自动化，我们觉得我们可以给这些团队一个坚实的平台，他们可以在其上构建，以进行持续交付。所以我真的很同情，因为如果你看看持续交付工作的复杂性和所涉及的工作，它不仅仅是供应基础设施和配置管理。我的意思是，今天，你有发布策略，你有金丝雀部署，你有蓝绿色。您认为这将验证这些部署如何在生产中获得成功。您有回滚功能，因此，如果出现故障，您能以多快的速度回滚？

我也认为，从与客户的交谈中，我们经常听到的一件事就是 DevOps 的合规性和安全性，因此 DevOps 最终是为了增强团队的能力，并给予开发人员和运营人员自由，但与此同时，这些公司需要控制和审计，并确保事情得到正确处理。因此，许多无聊的东西，如凭证和机密管理，以及某种可见性和报告，这是许多很酷的东西一样多的难题。

史梅尔:是的，你说的和我一样，史蒂夫。当然，安全和法规遵从性，我倾向于认为法规遵从性是安全引擎的一部分。你知道，仅仅遵从并不能让你安全。但是，话虽如此，安全性和法规遵从性是主要的——在持续交付游戏中，它们可能是您最好的朋友，也可能是您最大的敌人，对吗？如果你做得好，你可以自动化很多，并把它向左移动。这确实有助于加快成功的裁谈会进程，一次成功的经历。但是，如果您不重视它，并且与您的安全性和法规遵从性发生冲突，我的意思是，这就像回到“去”，不要收取 200 美元，您就要从头开始。所以这肯定是拼图中重要的一块。

Steve，给我们介绍一下背景，Harness 看到了这一点，团队也看到了 CD 带来的机会。这是一个基于 SAS 的解决方案。你还能告诉我们的观众们你们在做什么吗？

伯顿:是的。因此，主要是 SAS，我们还提供内部部署软件，因此您可以部署任何一种模式。我认为我们正在努力做的是三件事，一是自动化实际部署方面，因此，不需要编写脚本或编码大量的配置和连接您的所有生态系统，所以您可以在您的存储库工具中使用 CI，如 Jenkins、Bamboo、JFrog、CircleCI 然后让这些工作与亚马逊云平台、Kubernetes、Docker 同步；最后，所有的配置。因此，我们试图为生态系统中的所有技术建立可重复使用的连接器和模板，以便您可以在几分钟内将它们连接在一起并建立 CD 管道。

这听起来完全陌生，听起来好得难以置信。我们已经设法做到了，但这是一项艰巨的工作，并且已经与许多客户进行了合作。因此，在过去的 3、4 个月里，我想我们实际上已经与客户举行了 270 次验证会议。我们已经部署了超过 20 架贝塔机。因此，它一直在理解我们今天如何简化 CD 管道的复杂性。所以，是的，还有很多工作要做，但我认为我们走在正确的道路上。

**希梅尔:**优秀。当然，史蒂夫，你提到了一些我也想提出来的东西，那就是，你知道，这有点像 CD 的“需要一个村庄”，对吗？没有一个工具，一应俱全，将为您做所有的事情，正如您所提到的，Harness 已经与许多最常见的工具集成在一起，如 Jenkins 等。

伯顿:是的。是的，我认为，如果您看一看市场上的许多解决方案，它们要求您重写您的 CD 流程，改变您进行部署的方式，并有可能替换您已经拥有的工具，但这种方式永远不会真正奏效，因为大多数客户已经在这些工具、这些流程上投入了大量的时间、金钱和资源。因此，我们要做的是，我们可以适应您现有的生态系统和工具，以及您的流程，因此，许多经历过我们的测试计划的客户，在最初的三四个小时内，他们实际上已经能够启动和运行 CD 管道，并重复使用 Jenkins 或重复使用所有的云技术和平台，以及监控工具。

**希梅尔:**优秀。所以，你知道，虽然今天是发布日，史蒂夫，你们实际上已经为此投入了数月甚至数年的时间，但你认为接下来会发生什么？

伯顿:是的。是的，所以，在介绍中，我谈到了我们为自动化构建模块并构建 CD 管道。我们还设计了 Harness 来进行持续验证。这意味着，一旦您在生产中部署了代码或工件，我们实际上将使用机器学习来了解该部署对业务产生了积极还是消极的影响。因此，我们将验证诸如性能、响应时间、错误、吞吐量之类的东西，此外还会查看应用程序日志，以查看是否有任何回归或部署过程中引入的任何东西。

这就是我们兴奋的地方。我们认为，这是独一无二的，即验证生产部署的能力。如果我们发现故障或异常，我们可以自动回滚到客户的上一个工作版本，不会中断他们的业务。所以，对我来说，未来不仅仅是部署。如果您不知道您在生产中的部署对业务的影响，那么您可以说已经做了大量的工作，花费了大量的时间和精力，但可能什么也没有。

我记得，当我还是一名开发人员时，我在项目上工作了几个月，我们的成功达到了上线的那一天。一旦我们进入上线日，部署它就是运营部门的责任，而我们从来不知道我们是有积极影响还是消极影响。我认为，在当今世界，随着不断的交付和开发，以及尽可能快地从想法到资金，你真的必须知道每一个变化的业务影响。如果你不这样做，你会让你的公司失望。

**Shimel:** 同意。同意。Steve，这听起来–你知道，这是我一直想知道的关于 DevOps 的事情，我们在自动化方面投入了如此多的努力，并做得更好，你知道，获得更好的代码，自动化所有的东西，自动化 CI，自动化 CD 到我们实际部署的点，然后，突然，在部署后，我们开始下一件事情，或者我们正在寻找我们的反馈循环，以便我们可以开发它的下一个迭代。我想 APM 在某种程度上填补了下一个部署后监控的空白，但听起来 Harness 想要填补这个空白。这样说公平吗？

伯顿:是的。绝对的。我认为我们希望自动化客户如何验证他们的部署，因此 APM 市场现在是一个价值 50 亿美元的市场，因为从字面上看，每个公司都在构建软件，而他们的软件不断出问题。一个统计数据是 30%的应用程序部署失败。这是 Wired.com 写的一篇文章——我想是几个月前——但这是一个惊人的失败。应对故障的被动方法是购买 APM 产品，在您进行部署时帮助您在生产中排除故障。

如今，更明智的做法是在生产环境中开始进行 canary 部署，将应用程序部署到少量节点或用户，然后在应用程序环境的其余部分部署之前进行验证。因此，您可以将线束视为一个安全网，我们将让您在生产中部署，我们将帮助您部署金丝雀，如果我们发现有问题或异常，您可以将其回滚，几乎不会对业务产生影响

**Shimel:** 明白了。明白了。好吧，史蒂夫，信不信由你，我们可能已经超时了，但是，嘿，我们尽力了。史蒂夫，在我们结束之前，对于那些想了解更多关于马具的人来说，他们可以去哪里？

伯顿:是的，他们可以去 harness.io，这是我们的网址，我们有很多视频，很快就会提供免费试用。如果你想展示你的兴趣，你也可以订阅早期采用计划。

哦，那太好了。非常好。我相信网站上有很多东西，人们可以在那里找到所有这些东西的更多信息。Steve，祝贺你、Jyoti 和团队的其他成员。听起来你们正在进行下一次伟大的冒险，我相信这将是一次令人惊叹的旅程。我们希望您能尽快回到这里，我们将密切关注 Harness，看看它会给市场带来什么。恭喜你。

伯顿:是的。感激不尽。谢谢艾伦。
