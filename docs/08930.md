# AWS 中断暴露了 DevOps 弹性的弱点

> 原文:[https://devo PS . com/AWS-outage-exposes-vulnerability-of-devo PS-resilience/](https://devops.com/aws-outage-exposes-weaknesses-of-devops-resilience/)

2021 年 12 月 7 日[亚马逊网络服务(AWS)中断](https://www.cnbc.com/2021/12/09/how-the-aws-outage-wreaked-havoc-across-the-us.html)严重中断了许多企业的服务超过五个小时，[凸显了企业对互联网交付服务的依赖程度](https://techstrongresearch.com/aws-outage-and-app-resiliency-did-a-roomba-replace-the-canary/)。此次中断主要影响了美国东部的网络服务，但其影响是普遍的:它提醒人们，许多企业盲目忽视了将所有鸡蛋放在一个篮子里的古老公理，而是依赖于存在单点故障的提供商。

从机票预订系统到流媒体视频再到电子商务的各种服务都在停电期间中断，导致数百万美元的收入损失和无数小时的工作效率。此次中断更有趣的一个方面是它对 Slack、Trello、Asana 和 Smartsheet 等协作供应商的服务的影响，这些工具是许多开发和 DevOps 团队所依赖的。

此外，核心 [AWS](https://devops.com/aws-outage-outrage-rusty-linux-arm-latest/) 服务，如该公司的弹性计算和 DynamoDB 云工具也受到了影响，中断了许多第三方服务，并严重阻碍了使用这些服务的业务流程。虽然停机的明显受害者众所周知，就像亚马逊自己的电子商务运营一样，但有一股令人不安的暗流:对 DevOps 框架和使用它们的人的破坏。

到目前为止，AWS 对中断的根本原因守口如瓶；然而，对于 DevOps 社区来说，仍然有许多教训需要学习，并且必须提出一些问题，例如“在 IaaS/SaaS 中断期间，DevOps 流程能否继续存在？”以及“能否将多云故障转移解决方案融入开发运维构建的应用中？”

这些问题没有简单的答案，但是这次中断凸显了理解部署的 DevOps 系统的底层架构和框架的需要。举例来说，DevOps 社区中有多少开发人员已经采用 SaaS 工具来加速开发过程并支持 CI/CD 管道。代码扫描器、流水线编排甚至 ide 等 SaaS 应用在 DevOps 领域已经变得很常见。但是，有没有人费心问一下，如果这些工具中有一个失败了会发生什么？

更重要的是，在开发过程中对 SaaS 工具的依赖导致了 DevOps 开发人员开发的应用程序中潜在的责任。DevOps 应用程序已经开始依赖 API，通常由微服务驱动，并经常部署到在 SaaS 上运行的容器中。如果这些元素中的任何一个变得不起作用，许多应用程序可能会失败，最终将责任推给开发人员，让他们解释为什么他们要创建具有单点故障的应用程序。

展望未来，DevOps 社区需要认真审视他们框架的组件，并确定是否存在任何单点故障，包括他们的 IaaS/SaaS 提供商。虽然不可能补救每一个问题，但是如果没有人费心建立所用工具的清单，并且考虑到这些工具中的任何一个的失败会如何影响工作流，那么开发过程会变得多么脆弱，这是一个教训。单个组件的外部故障如何影响应用程序的功能，而根本原因的发现过程需要几天甚至几周时间，这样的例子不胜枚举。这主要是因为不仅缺乏知识，而且缺乏对所用组件的了解。

这些经验教训可以扩展到开发过程本身，而根除单点故障的最佳实践可以扩展到应用程序本身。利用这种智能从理解软件材料清单(SBoM)的概念开始，SBoM 是对应用程序供应商越来越重要的支持文档。正确定义的 SBoM 揭示了所有的组件(库、API 等)。)嵌入到应用程序中，并可用作定义弱点所在的地图。

对于 DevOps 社区来说，最近的 AWS 中断已经成为一个号角，让他们向内看，并发现他们正在构建的应用程序可能是问题的一部分。随着连续性和弹性成为 IT 和业务领域的主要话题，是时候让 DevOps 从业者开始考虑如何支持这两种关键业务需求了。相互推诿责任的日子必须结束，如果依赖软件的企业想要发展，就需要有人负责在停机时提供答案，并从这些停机中吸取教训，以创建更具弹性的应用程序。
