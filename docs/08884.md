# 努力保守秘密的开发团队

> 原文:[https://devops.com/devops-teams-struggling-to-keep-secrets/](https://devops.com/devops-teams-struggling-to-keep-secrets/)

根据最近的一份报告，越来越多的组织正在遭受与开发人员 CI/CD 管道中暴露的秘密相关的安全事件。

这项研究描绘了一幅令人不安的画面:只有 5%的受访者说他们的大多数开发团队使用相同的[秘密](https://devops.com/?s=secrets)管理过程和工具。

这些事件无所不包，从公开发布在公共云代码库中的秘密到不安全的第三方代码，再到组织自己的代码或配置中的漏洞。

由于开发团队拥有自己的云基础设施或在安全团队不知情的情况下直接创建面向互联网的应用程序的能力，安全团队不知道要保护什么，这通常会使问题变得更糟。

ThycoticCentrify 首席安全科学家兼 CISO 顾问 Joseph Carson 表示:“我们在 DevOps 中看到如此多与暴露机密相关的安全事件的原因是，使用硬编码的密码和密钥是让事情正常运行的简单途径。”。“我们需要让开发人员能够轻松使用机密库，这样即使机密遭到泄露，它们也不再可用、有效或过期。”

卡森说，从持久特权转移到即时、按需特权将减少这些类型的错误配置和攻击。

“不幸的是，许多开发人员继续使用硬编码的密码或静态密钥，一旦遭到破坏，攻击者就可以控制应用程序，从而导致数据泄露、更改应用程序配置或在网络中横向移动，”他补充道。

Netenrich 的首席威胁猎人 John Bambenek 表示，根本问题是，当组织转向敏捷和开发运维时，他们消除了快速移动代码的所有摩擦，这几乎总是包括消除安全性。

“这意味着安全本质上是反应性的，而不是嵌入到工程中，这是假设安全甚至知道首先要保护什么，”他说。“这是将安全性从等式中移除并期望它只在后端工作的自然结果。这种方法已经行不通几十年了；JIRA 董事会不会突然解决这个问题。”

从卡森的角度来看，秘密管理是一种从硬编码密码或静态密钥转移到即时权限或一次性密码的能力，即使密码被破解也无法使用。

他说:“许多多年来保护特权访问的特权访问管理解决方案已经向开发人员扩展了功能，以帮助将价值转移到 DevOps 中，以便他们可以管理应用程序、数据库、CI/CD 工具和服务的凭据，而不会在开发过程中造成摩擦。”

特权访问安全等方法有助于实现 API 即服务，并提供机密、SSH 密钥、证书、API 密钥和令牌的即时可用性。

## 机密管理

Bambenek 补充说，问题不在于选择秘密管理流程或工具，而在于它们根本不存在。

他说:“挑选一些东西，将密钥和秘密保留在公共云储存库中，开发人员将使用这些东西，以便在需要时快速轻松地轮换密钥。”“最重要的是，它在任何地方都是相同的工具，以一种对开发人员来说几乎不可见的过程运行。四十年的软件漏洞告诉我，如果你依赖开发者来做安全的事情，你会一直失望。”

## 走错一步

Valtix 的首席安全研究员 Davis McCarthy 指出，开发人员的职业不是安全从业者。他说:“开发者正在创造新的想法，没有实验就没有好的想法。”。“试验新的技术堆栈可能会导致错误配置，从而无意中暴露数据。”

McCarthy 说，用户之间共享 SSH 密钥会产生更大的攻击面，用户会以不同的方式存储秘密，从而产生多种攻击媒介。

如果一个用户的终端受到威胁，他们的 SSH 密钥被泄露，很难知道第一个缓解步骤是什么。在内部威胁和恶意软件之间，走错一步就可能导致违规。

“尽管一个软件工程师可能掌握了大量关于安全编码方法、网络应用攻击和加密的知识，但他们仍然专注于执行他们的工艺，”他说。“没有在开发生命周期中构建安全性的组织会产生系统性风险，从而导致最佳实践的失败。”

McCarthy 补充说，组织可以通过创建清单、审计清单和频繁地将机密转移到敏感环境，将机密管理集中并嵌入到 CI/CD 管道中已在使用的工具中。

Bambenek 说，简单的答案是确保 CI/CD 管道的基础设施得到集中管理和控制，而工程团队没有能力“自己动手”

“如果有人可以在没有任何真正治理的情况下选择自己的工具，任何安全控制都不会保持完整，”他说。“CI/CD 流程不需要工程团队来管理部署管道的后端，这意味着有人可以确保这些功能从一开始就到位。”
