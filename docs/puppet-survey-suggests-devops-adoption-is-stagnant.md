# 木偶调查表明 DevOps 的采用停滞不前

> 原文：<https://devops.com/puppet-survey-suggests-devops-adoption-is-stagnant/>

Puppet 今天发布了一份年度 DevOps 状态报告，表明 DevOps 实践的整体采用已经停滞。

对 2，650 名 IT、开发和信息安全专业人员的调查发现，83%的人在一定程度上实施了 DevOps 实践。然而，只有 18%的受访者在拥有高度发展的自动化开发运维流程的组织中工作。超过四分之三的受访者(78%)正在朝着这个目标前进。

调查发现，整整 90%高度发展的 DevOps 团队已经自动化了大多数重复性任务，相比之下，只有 67%的中级组织实现了自动化。此外，91%的高度进化的团队报告对他们的职责有清晰的理解。

毫不奇怪，最大的障碍与其说是技术上的，不如说是文化上的。中层组织中最常见的文化障碍包括不鼓励风险的文化(21%)、职责不明确(20%)、不重视快速流程优化(18%)和反馈回路不足(17%)。相比之下，Puppet 报告发现，不到 2%的高度进化的组织报告从执行层面抵制 DevOps。

Puppet 的现场 Nigel Kersten 在一次在线讨论中表示，经过十多年的 DevOps 倡导，人们仍然是采用时最大的限制因素。Kersten 指出，具体来说，从薪酬的角度来看，与开发人员相比，许多 IT 运营人员往往觉得自己被低估了。

企业面临的另一个挑战是试图平衡创建可一致管理和保护的标准化 IT 环境的需求与开发运维团队在他们认为合适时采用新工具的愿望。高级 IT 主管通常倾向于根据成本标准做出平台决策，而这些成本标准并不总是考虑到支持敏捷开发方法的需求。这些平台中的大多数被设计为由 IT 管理员使用图形工具来管理，而不是通过命令行界面(CLI)和对开发运维团队有吸引力的应用编程接口(API)来管理。

因此，在企业 IT 组织中，选择和维护自己选择的平台的开发运维团队的个人群体正在涌现。这些团队通常认为他们选择的平台是提高开发人员生产力的更好的选择。例如，寻找和留住站点可靠性工程师(sre)来管理这些 DevOps 平台是大多数企业 IT 组织面临的主要挑战。大多数 it 管理员缺乏成为 SRE 所需的编程技能，也不清楚有多少 IT 管理员倾向于获得这些技能。

相反，不清楚有多少程序员愿意以牺牲他们本来会用来编写代码的时间为代价，通过编程来管理 it 基础设施。只能雇佣有限数量开发人员的组织可能不希望将管理 IT 运营的责任转移给众所周知的左派，因为他们需要更多的代码来更快地编写。在许多情况下，寻找和留住开发人员与雇佣和留住 sre 一样具有挑战性。

自动化和人工智能(AI)的进步应该会使组织在未来的几个月和几年中更容易采用 DevOps 最佳实践。然而，对所有人来说显而易见的是，在整个企业中采用那些最佳实践仍然需要大量的时间和努力，并且将会持续很多年。
