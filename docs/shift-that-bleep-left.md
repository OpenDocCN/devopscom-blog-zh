# 他妈的向左移动

> 原文：<https://devops.com/shift-that-bleep-left/>

说真的。你越是能够[将安全和测试之类的事情转移到左边](https://devops.com/?s=shift+left)，DevOps 过程的响应就越快。有些事情向右移动更好……但只是在解决左移工作的问题时。一个很好的例子是在开发过程修复代码以永久阻止攻击的同时，主动阻止零日攻击。

我们 IT 人员最担心的一件事是跟进“修复”部分——因为我们非常忙。一旦一个问题被堵住，我们倾向于去扑灭其他的火，因为我们有太多的事情在进行。我们必须在这方面做得更好。但我不想跑题。今天的主题是关于修复产生的错误/漏洞。

我们过去处理测试和安全的方式的一个问题是，当我们得到“这是一个我们需要修复的问题”的反馈时，开发人员已经在做其他事情了。但是随着 DevOps 过程中安全性和测试的增加，我们现在有能力更快地提供反馈。

我在 DevOps 领域最喜欢的创新不是一些疯狂的新 bug/漏洞检测功能，也不是人工智能的猖獗使用(尽管人工智能的一些使用确实接近于我最喜欢的)。就是不起眼的 [IDE 插件](https://en.wikipedia.org/wiki/Integrated_development_environment)。作为一名开发人员，编写有错误或不安全的代码从来都不是目标，但这每天都在发生。集成到测试和安全软件中的 IDE 插件允许开发人员在编写代码时得到可能有问题的提示。这不仅解决了开发人员记忆犹新的问题，还让开发人员有机会通过了解他们经常产生的问题并学习如何避免它们来改进他们的技术。

我们都写过糟糕的代码。我曾经为一家银行重写过数据库；对于一个年轻的高手来说，这是一个很好的谦卑和学习的经历，他需要被提醒他在编码时并不完美。毫无疑问，其他人也有类似的故事，尽管没有那么灾难性。在编写代码时，得到问题警告的能力——但是质量和安全性——不能被高估。一些开发人员会讨厌它，因为它比老派的“好了，所有代码都写好了，现在可以测试了”过程更具侵入性，但它的强大恰恰是因为它更具侵入性和主动性。

对于那些在测试和开发安全领域的人来说，挑战将是把误报保持在最低限度。我想说零，但我宁愿偶尔听到“哦，那个。是的，在这种情况下不是问题”，而不是在整个开发过程中让它告诉我一些事情，最后仍然是错误的/不安全的代码。

你们都还在摇摆。全世界每天使用的软件数量是惊人的，无论您是在开发内部应用程序还是商业软件包，您都在努力生产并维护它。你成功了。看看 QA 和安全 IDE 插件，诚实地评估它们是否能让你的工作更安全和/或更高质量。我认为对我们几乎所有人来说都是如此。并且他们可以通过在开发过程中更早地做这件事来节省时间。也许你会因为摇摆而被认出来。好吧，可能不会，但有可能。
