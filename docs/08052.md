# 获得低代码的好处而没有风险

> 原文:[https://devo PS . com/get-the-benefits of-low-code-with-the-risks/](https://devops.com/get-the-benefits-of-low-code-without-the-risks/)

今天，超过 300 家厂商和各种平台提供不同风格的[低代码](https://devops.com/category/blogs/low-code-no-code/)解决方案。然而，这些低代码工具中的大部分是真正的无代码工具，有益于试图解决特定程序的个人或团体。在您的环境中，低代码工具的好处有一长串，令人信服；然而，在使用这些强大的工具时，企业应该意识到存在安全风险。

当评估或使用低代码工具时，重要的是要记住，虽然低代码工具抽象了编写代码的过程，但代码仍然在被编写。使用低代码工具创建的应用程序与任何其他 web 应用程序一样容易受到攻击。然而，公司无法控制由低代码工具生成的代码，因此开发安全应用程序的负担在很大程度上转移到了低代码工具供应商身上。因此，在使用低代码供应商时，最小化风险的最佳方式是正确评估该供应商的安全实践，并像对待使用传统/亲代码方法开发的任何应用程序一样，积极地测试使用这些工具创建的应用程序。

要考虑的风险包括访问控制和身份验证(权限、多因素身份验证)、数据安全性(传输中和静态加密)以及针对漏洞和攻击的应用程序安全性(网络安全和入侵检测)。其他要考虑的风险是风险缓解，包括 DDOS(分布式拒绝服务)攻击、备份和灾难恢复以及安全事件处理等领域。管理这些风险的责任属于低代码供应商、托管提供商和使用低代码工具的公司。

在您的环境中引入新的低代码工具时，应该考虑以下提示，以最大限度地降低安全风险。请记住，对供应商进行尽职调查至关重要。要考虑的项目包括:

托管平台:确保供应商使用的是声誉良好的托管提供商，最好是亚马逊网络服务(AWS)、谷歌云平台、Salesforce 或微软 Azure。领先企业将提供安全、高度可用的平台来应对上述风险，并将完成第三方审核和认证，以证明其对安全性的承诺。

**供应商基础设施管理:**低代码供应商通常在一定程度上负责管理整个堆栈的上层；这通常被称为共享安全模型。审查供应商，确保他们有可靠的控制和流程来正确管理和保护基础架构。理想情况下，供应商定期接受某种形式的认可审核，如 SOC 2 Type 2 或 ISO-27001。这有助于确保在安全性、可用性、保密性、隐私性和处理完整性等领域的控制和流程到位，以解决供应商负责的托管平台领域的上述风险。

**正式的 SDLC:** 供应商应遵循用于开发其低代码工具的文档化软件开发生命周期(SDLC)和变更管理流程。良好的 SDLC 流程包括使用源代码管理系统、依赖性检查流程、静态代码分析、自动化单元和回归测试、同行代码审查、安全编码实践培训、应用程序漏洞扫描、质量保证测试流程和第三方渗透测试。

**政策和流程:**供应商应制定涵盖信息安全、业务连续性、灾难恢复、安全事件处理、基础设施加固等内容的书面政策和流程。这有助于证明供应商致力于提供安全、高可用性和高质量的产品。

采用低代码工具的公司本身也应该考虑内部流程以及如何使用低代码工具。要考虑的项目包括:

**正式的 SDLC:** 即使公司使用低代码工具，仍然需要遵循正式的 SDLC。对公司来说，强调并接受安全的编程实践是很重要的，即使对于低代码开发人员来说也是如此。低代码工具能够实现客户开发的代码片段或组件，建议公司遵循开放 Web 应用安全项目( [OWASP](https://owasp.org/) )基金会概述的编程标准和安全编程实践。另一个有时会被忽视的重点领域是测试需求，包括功能和面向安全的测试。

**安全测试:**考虑对完成的低代码应用程序执行渗透测试。虽然理想情况下，供应商已经在工具本身上执行了这些测试，但是在完整的低代码应用程序上也执行一次测试也是一个好主意。

**访问控制和认证:**根据最小特权模型配置访问和权限。尽可能利用多因素身份验证和单点登录等高级身份验证机制。确保访问权限移除流程到位，以移除离职员工的访问权限。

**数据安全性:**验证应用程序被配置为使用 HTTPS，并且所有数据库都被加密。低代码供应商可能提供也可能不提供数据库，不管是哪种情况，公司都有一定程度的责任来保护数据，不管是谁在托管数据。

**政策和流程:**针对团队成员的成文政策、流程和相关培训有助于降低风险。对于 SDLC、业务连续性规划、灾难恢复和安全事件处理的文档和培训，建议采用某种形式。例如，负责低代码应用程序的团队需要知道如何处理发生的安全事件，无论是由于低代码应用程序、低代码工具还是托管基础设施中的问题。

低代码工具的好处远远超过在您的环境中引入这些工具的风险，但是只需预先做一点努力，您就可以省去很多麻烦。
