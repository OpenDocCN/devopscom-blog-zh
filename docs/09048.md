# 前端框架支持更好的开发协作

> 原文:[https://devo PS . com/front-end-frameworks-enable-better-dev-collaboration/](https://devops.com/front-end-frameworks-enable-better-dev-collaboration/)

普通大众通常认为软件开发是一个孤独的软件开发者孜孜不倦地敲击键盘。当然，这充其量是一个时代错误。真实的情况是，软件工程师之间的协作，尤其是与他们在 IT 运营部门的同事的协作，已经成为更快推出更好代码的标准，IT 运营部门的 已经发展成为 DevOps。

随着软件开发变得比以往任何时候都更像是一项“团队运动”,我们正处于下一次大变革的风口浪尖。现在，协作正在将 延伸到 IT(开发人员和运营)之外，延伸到跨不同业务团队的跨职能领域。产品开发、市场营销和其他方面也为软件开发过程提供输入和反馈，最重要的是，成为 过程的一部分。

## 前端框架正在推动协作

从历史上看，软件编程，包括前端编程，都发生在本地 [IDEs](https://en.wikipedia.org/wiki/Integrated_development_environment) 。然而，现代前端 web 框架的兴起带来了一整类开发人员工具，它们加快了构建过程，并允许更快地部署应用程序。

这样做的前端框架有很多: [Netlify](https://netlify.com/) ，[Vercel](http://vercel.com/)和[app Smith](http://appsmith.com/)，一个开源的内部 app builder，就是三个例子。这些工具正在被广泛采用，因为在前端变得越来越复杂的世界中，它们使托管、部署、认证和无服务器功能变得更加简单。采用的另一个原因也是因为这些工具中的许多都是建立在流行的开源框架之上的，比如 React，它已经拥有了大量的用户基础。

这促成了一种新的协作方式，为所有相关利益方释放了大量价值。受 Figma [在 2016 年通过允许实时评论重塑设计流程](https://techcrunch.com/2016/09/28/design-tool-figma-launches-multiplayer-collaboration-mode-for-interface-designers/) 或谷歌套件如何允许实时编辑的启发， 去年软件工程领域也出现了类似的现象。

这也意味着协作发生在应用程序上，而不是通过电子邮件、Slack 和吉拉等工具，从而获得有针对性的反馈，而不必在多种工具之间切换。

## 协作和反馈循环变得更加包容 积极

虽然开发人员经常在代码级别进行协作，但非 开发人员的 反馈经常在设计级别给出。内部工具(或应用程序)更是如此，它们通常占组织中工程带宽的 20%到 40%。

例如， Appsmith 如今有数千名开发人员为用户自行托管和部署内部应用，这些用户通常是业务或运营团队，例如支持、销售和/或财务团队。通常，这些业务用户甚至不知道谁构建了 app lication。通过前端框架的实时评论促进了软件工程团队和他们的业务同行之间的协作，他们现在可以直接交流，提供 反馈 并与他们的工程同行协作。这种紧密的合作 n 增强了各方的能力，提高了 效率 并节省了大量时间，同时提高了软件的整体质量并确保其符合预期。

## 协作是趋势

伟大的软件是团队将协作扩展到开发人员之外的故事，有很多好的工具可以帮助实现这一点。这种趋势在未来几年应该会继续发展，因为它利用了软件开发领域的许多其他潜在趋势。

首先，后端即服务 服务领域替代方案的爆炸式增长(Heroku 和 Firebase 是这一领域的早期先驱)意味着 BaaS 被更多的组织所接受，用于各种各样的用例。 这一趋势加速了 的发展，因为它们呈现了一个更加紧密集成的服务生态系统，组织不必 担心部署、托管、内容交付网络(CDN)和/或认证等问题。

其次，API 的激增推动了基于各种数据源的更复杂应用的出现。一切都是相互联系的。当然，这种 als o 意味着更多的数据可用于支持决策，这导致了许多不同种类的 应用程序被创建出来，既用于外部也用于内部目的。

最后，规模更大的[低代码/无代码](https://devops.com/category/blogs/low-code-no-code/)运动正在创造 数百万新的公民开发者，并模糊了开发者和非开发者之间的界限。

### 摘要

DevOps 标志着软件开发向团队运动发展的开始，协作现在正加速超越 IT 的界限。像 D evOps 一样，软件开发过程中的这种合作的结果将是更好的软件，推动企业以前所未有的速度向前发展。
