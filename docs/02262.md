## 副本

艾伦·希梅尔:嘿，大家好，我是艾伦·希梅尔，我是 DevOps.com，您正在收听的是另一个 DevOps 聊天节目。我很高兴今天能邀请到马克·施瓦茨作为我的 DevOps 聊天嘉宾。Marc，如果你不知道的话，是 AWS 的高管，是你将要阅读的关于 DevOps 的一些最伟大书籍的多位作者——《餐桌上的座位》仍然是我一直最喜欢的书籍之一——我认为正如他在关于 IT 革命的最新简介中所描述的那样，他是一位打破传统的首席信息官。

马克，欢迎来到 DevOps 聊天。

马克·施瓦茨: *【笑声】*谢谢。我很高兴来到这里。

**Shimel:** 那么，Marc，什么是打破传统的首席信息官？

**施瓦茨:** *【笑声】*我喜欢质疑假设。你知道，我发现，很多时候，人们都很沮丧。他们试图实现的目标存在障碍，问题是他们受到各种各样的限制，如果你不质疑这些假设并消除其中的一些限制，他们就会继续感到沮丧——所以，这就是我要说的。

是的。我同意。所以，我不想花太多时间，但是 Marc，我们提到你来自 AWS，但是我没有提到一个角色。我们的观众可能不太熟悉——您目前在 AWS 中的角色是什么？

**Schwartz:** 他们称我为企业战略家，意思是说，我是一个由前 IT 领导者组成的小团队的一部分，我们中的许多人是大企业的前首席信息官或首席技术官，他们进行了数字化转型，转向了云，我们现在所做的是，我们与大企业的高管合作，帮助他们克服他们在尝试转型时面临的一些挑战。

所以，通常情况下，这是像文化变革，或组织变革，或组织结构，融资模式，人际技能，诸如此类的事情。我们在会议上发言，我们写书和写博客，我们和个人客户一起工作。

非常酷。不，我们跳进来吧，马克。你有一本新书，最近刚出版。顺便说一下，这本书在亚马逊上有售，以防有人感兴趣，还有 Barnes and Noble、800-CEO-READ、Indiebound、BAM 和其他任何你能找到像样的书的地方。书名是《战争与和平和信息技术:数字时代的商业领导、技术和成功》

马克，和我们的读者分享一下——这本书是关于什么的？

嗯，这本书本质上是对人们读完《餐桌上的座位》后给我的一个挑战的回答因为，在《餐桌上的一个座位》中，它主要是为 it 领导者而写的，在整本书中，我提出了我们需要摆脱业务与 IT 的区别的论点，你知道吗？我们谈论业务和 it 的方式就好像这是两件独立的事情，我们认为 IT 是从业务中获取需求并执行这些需求，但这种思维方式不再有效。我们需要找到一种方法将它引入业务本身。

人们对“餐桌上的座位”中的信息反应很好，但他们说，“除非有人告诉商界人士同样的事情，否则这不起作用，你知道吗？*【笑声】*我们不能自己改变关系。因此，应该有人为企业高管写本书，向他们解释你在谈论什么。”我意识到，如果我能写这本书的话，这本书也能帮助很多高管，这些高管是我工作的一部分，他们面临着转型的障碍。

因此，我写了一本书，主要是说，“你转型的一大障碍是业务和 IT 之间的关系不正确，这里是你如何改变它，这里是为什么它会为你带来巨大的变化。”

**希梅尔:**优秀。所以，你知道，我觉得你说得很好，马克，为了大家。你知道，我刚刚回来——我们正在离线交谈，我告诉你我上周在巴塞罗那参加了一个 KubeCon + CloudNativeCon，这是另一个很棒的活动，7500 人参加。

真正让我震惊的是，尽管这是一个相当技术性的事件，对吗？我们正在谈论云原生和许多开源软件程序和运营程序，it 的业务角度，有多少业务人员试图了解 IT 领域的最新趋势和潮流，以及他们如何利用无服务器或 Kubernetes 等新东西来改善他们的业务，而不一定是因为它很酷，或者你知道，闪亮的饰品综合症或类似的东西。

但是说真的，橡胶是如何——何时与这项技术相遇的？你知道，我忍不住——我的意思是，我很受鼓舞，马克，但我仍然不知道企业高管是否都相信这一点。

施瓦茨:是啊。

**Shimel:** 你的想法？

嗯，我不认为这是关于云，本质上，或 DevOps 或其他什么。真正重要的是云和 DevOps 以及新的工作方式如何让企业能够做以前做不到的事情。我认为这是非 IT 业务主管需要了解的，即以前不可能做到的事情现在可能做到了。

他们必须这么做的原因是，为什么这是一件紧急的事情，因为他们的竞争对手是，你知道吗？无论你有什么工具，你的竞争对手也有，如果你不用，而你的竞争对手在用，那么你就处于商业劣势。我认为主要是速度的问题，不管怎样。因此，善用云计算，善用 DevOps，可以让您以更快的速度获得 IT 能力来改变您的业务。

这个速度很重要，不仅仅是因为显而易见的原因，你需要时间来营销，你需要获得先发优势，当你的竞争对手扰乱你时，你需要迅速做出反应，还因为更微妙的原因，因为我们现在使用速度作为一种方法来快速测试想法，看看它们是否可行，并在我们完全投入之前通过测试想法来管理我们的风险， 我们利用速度来获得客户和用户的反馈，并根据我们所了解的情况来调整我们的产品，我们利用速度来降低风险，以改善我们的安全状况，提高我们的系统可用性。

因此，速度成为一个非常重要的因素，云、DevOps，你知道，他们的技术实现，但他们碰巧帮助公司实现速度，我认为这是高管需要带走的重要信息。

I don’t think it’s about the cloud, per se, or DevOps or whatever it is. What really is important is how the cloud and DevOps and new ways of doing IT empower businesses to do new things that they couldn’t do before.**Shimel:** I agree. But, you know, there’s an old saying—speed can kill. Great speed kills, too, right?

**施瓦茨:** *【笑声】*嗯，如果你使用正确的速度模型，它不会致命，原因是使用速度的模型通常也使用自动控制或护栏。所以，你想要的，也是你的想法是，你想要能够真正快速地移动，但在强制合规、强制安全、强制质量的护栏的边界内，并且两者都可以做到。在过去，我们一直认为速度和稳定性或速度和质量之间存在权衡，事实证明，在这个 DevOps 世界，在这个云世界，速度——提高的速度实际上提高了质量、弹性、可用性、安全性和所有这些东西。这是一个学习如何将速度用于这些目的的问题。

希梅尔:是啊。这真的是——我的意思是，开玩笑，速度是致命的。我真正想说的是，仅仅为了速度而速度，你知道，速度必须被运用——伴随着巨大的力量而来的是巨大的责任，对吗？

**史华兹:** *【笑声】*是的。正如我之前所说，出于竞争和商业模式的原因，你必须达到这个速度。所以，这不是一个真正的选择，但你必须同时学会如何利用这种速度来降低风险和提高你所提供的质量。

**Shimel:** 正是，正是。那么，招待会怎么样，马克？你从人们那里得到了什么样的反馈？

嗯，这本书几周前才出版，真的。我们在哪里？是的，5 月 14 日是出版日期，所以几个星期。人们似乎乐在其中。我写这本书的时候有点好玩，我认为它的某些方面很有趣，这让我感觉很好。

但是从我得到的反馈来看，人们把它看作是前一本书的后续，正如我所希望的那样。你知道，这是看待类似信息的方式，但很大程度上是从非 IT 业务主管的角度。你知道，首席财务官应该如何看待新的世界，新的数字世界，首席执行官应该如何看待它，董事会应该如何看待它？我很高兴人们认为它在这方面很有价值，很多人说，“好吧，我会让我的整个执行团队阅读这一点，这将激发一些伟大的讨论，”所以我对此非常高兴。

Shimel: 太棒了，很棒的东西。让我介绍一下它的另一个方面，听听你的想法，Marc。你知道，我们在过去听到了很多，我可能在过去一年或 18 个月中听到过——DevXOps，对吗？我们有 DevSecOps、DevBusOps、DevHROps——dev whatever ops，对吗？DevXOps。我认为这里有一个逆向方程。正如你所说，企业领导人必须更多地参与其中，必须学会如何利用它来推进他们的业务目标和取悦客户，对吗？

与此同时，我们可以从 DevOps 等工具中吸取经验教训，而不仅仅是 DevOps。但在相对较短的时间内，它经历了一段相当痛苦的自我完善和学习新事物的道路，并进化成我们所看到的 DevOps 之类的东西。企业能做什么——本质上不是利用它，而是节省一些白痴税？

**施瓦茨:** *【笑声】*

**Shimel:** 你知道，从错误中吸取教训，并将其应用到一般业务中？

施瓦茨:是啊。我想——我也开玩笑说 DevOps，DevSecOps，DevSecBizOps，DevSecFinBizOps——你知道吗？你可以继续补充。

真正的重点是跨职能团队，对吗？DevOps 意味着我们将拥有一个拥有开发技能和运营技能的跨职能团队，在某种程度上，我们意识到这些技能还不够，他们还需要拥有安全技能，并且您可以无限地增加这些技能，因为底线是功能孤岛在快速发展的环境中不能很好地工作，就像我们今天在数字世界中一样。功能孤岛会减缓你的反应，它们会减缓你创新和抓住新机遇以及应对变化等的能力。

这就是 DevOps、DevXOps 概念背后的真正原因。我认为企业通常会发现这一点，当您有功能孤岛并且它们作为孤岛运行时，当您将东西从一个孤岛传递到另一个孤岛时，您会引入等待时间。你减少了创新，因为每个筒仓认为自己有自己的责任，而不是对结果的共同责任。

因此，我认为每个人都可以从 IT 中所发生的事情中学到的经验是，要在需要快速响应的环境中快速响应，您必须消除孤岛，转而专注于以*【背景噪音】*为目标的跨职能团队。至少，那是我最大的收获。

Shimel: 当然，当然。*【背景噪音】*对不起，我的机器——哦，又来了。你知道，我在我的，我不会给你我在表面的品牌。只有一家公司生产表面。

你能给我们先睹为快吗，还是现在还为时过早？

你知道，我认为这符合我作为一个反传统者的角色。我认为官僚主义是人们非常反感的词汇之一。但是，当然，在 AWS 之前担任政府首席信息官期间，我见证、经历了许多官僚主义，并制定了应对策略，我认为这些策略将对广大读者有用。所以，我想我可能会写一些与此有关的东西。

当我深入研究时，我意识到人们并没有真正地解读它，在很多情况下，没有以最好的方式解读，如果你对此思考得更深入一点，就会改变你练习的方式。

所以，这就是第一本书的驱动力，在那本书里，我提出了一个问题:首席信息官与计算什么是商业价值有什么关系？首席信息官如何确保他或她正在交付商业价值？实际上，我意识到这是第二本书的好主题。*【笑声】*一些我真的需要更深入探究的东西。所以，这就产生了“餐桌上的一个座位”
