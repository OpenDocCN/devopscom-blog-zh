# 持续交付的进步是以测试成本为代价的

> 原文:[https://devo PS . com/continuous-delivery-advances-come-at-cost-of-testing/](https://devops.com/continuous-delivery-advances-come-at-cost-of-testing/)

应用程序测试平台提供商 Applitools 对 820 名应用程序开发专业人员进行了一项调查,发现超过三分之一的受访者(38%)就职于现在对生产环境进行日常变更的组织。

挑战在于应用程序测试无法跟上持续部署的速度。调查发现，组织看到的测试套件覆盖不到 50%的应用程序。

Applitools 首席运营官·摩西·米尔曼说，测试覆盖率如此之低的一个主要原因是，超过 40%的受访者表示，他们的持续集成(CI)测试套件可能需要一个多小时才能运行，这往往导致测试要么被推迟，要么可能根本不运行，他指出。

Milman 补充说，大多数组织仍然依赖于手动测试过程，这些过程跟不上应用程序部署和后续更新的速度。根据调查，超过一半的组织(55%)自动化了不到一半的测试。调查发现，超过三分之二的受访者(67%)正在采用测试自动化来提高他们的测试覆盖率。

Milman 说，总的来说，现在采用 CD 过程来加快应用程序部署速度的组织的百分比正在稳步增长。然而，手动测试过程降低了组织部署代码的速度。许多组织经常在应用程序质量和更快部署代码的需求之间进行权衡。Milman 指出，不可避免地，会有一些问题可以追溯到在部署之前没有充分测试代码的决策。

总体而言，调查发现 90%的受访者构建和部署了 web 应用程序，其次是 52%的受访者构建和部署了移动应用程序。只有 30%在构建桌面应用程序。然而，应用程序的测试往往变化很大。近三分之二的受访者(65%)表示，他们在端到端的基础上测试应用程序，其中 55%测试集成，其次是单元测试(53%)和应用程序编程接口(API)测试(51%)。

然而，调查发现，只有 26%的人测试了应用程序性能，13%的人测试了安全性，12%的人测试了负载。

当然，问题的一部分是，并不总是清楚谁负责应用程序测试。理论上，开发人员正在接受更多的应用程序测试责任，因为他们承担了更多的端到端管理应用程序的责任。在实践中，开发人员经常赶着制定应用程序开发截止日期。开发人员以生产力的名义跳过或限制测试并不罕见。他们中的许多人也接受了敏捷开发方法，至少在理论上，允许他们通过下一次应用程序更新来解决任何可能出现的问题。

出现的问题是，最终用户对实验的容忍度越来越低。如果一个应用程序最初没有像预期的那样工作，那么重新获得用户的信任是非常困难的。许多人会简单地评估另一个类似的应用。当然，具有讽刺意味的是，在更快交付代码的竞赛中，由于在部署应用程序之前没有时间对其进行适当的测试，应用程序体验最终被牺牲了。
