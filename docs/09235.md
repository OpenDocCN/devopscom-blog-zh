# 安全债务:速度与常识

> 原文:[https://devops.com/security-debt-speed-vs-common-sense/](https://devops.com/security-debt-speed-vs-common-sense/)

几年前，我们遇到了一些引人注目的安全事件，涉及 DevOps 和 Kubernetes，每当一个容器崩溃时，管理团队只需重新部署容器。事实证明，许多组织都在做同样的事情，更糟糕的是，他们没有谈论它，因为他们知道这不是崩溃问题的长期解决方案。

当时，我认为这些引人注目的课程将教会人们实际修复代码问题，即使是在高吞吐量的敏捷开发车间。我承认我也笑了，再一次，以凤凰计划为代价，想，“是的，但是 CISO 没有通过要求得到修复来阻止事情！”我希望它会变老，但它不会。

看来我错了。或者至少部分如此。一些——也许是许多——商店似乎忽略了这些问题所传达的信息。有趣的是，我们听说这种情况仍在发生，所以我们开始提问。没错。有些商店(因为我们是从“轶事”开始的，所以我不会假装用数字来衡量它)仍然抱着“如果它没有造成核心系统的中断，就不要修复它”的心态。

哎哟。我敢肯定，将来当它被列为某个地方的漏洞或违规时，我们会重新审视其中的一些。

我不是安全专家。因为绝大多数企业没有专门的安全人员，甚至更没有足够的安全人员，所以我几乎把安全作为我所做的每份工作的一部分，因为这是最需要做的事情。你们中的许多人，如果不是大多数人，都在同一条船上。

也就是说，当你公然要求安全事故时，我能看出来。运行已知有错误的代码或者[错误配置](https://devops.com/?s=misconfigurations)到反复崩溃的程度，是*故意向*要求安全事故。“但它没有暴露在互联网上！”只是一个有效的反驳，直到攻击者进入边界，利用您的错误进行特权升级之类的事情。然后没人管它有没有联网；他们想知道攻击者是如何做到的。这是你的责任。

因此，对于那些速度已经超越了常识的组织，让我们非常明确和中肯。

最好不要运行崩溃级别的故障软件。

因为你一定没有，所以写一个政策来解释这一点，并列出解决这些问题的过程和/或时间表。然后把它开回家。确保人们知道这项政策将得到执行。对于我的绝大多数读者来说，他们没有“创建和颁布政策”所暗示的那种权威，说服那些有这种权威的人。以书面形式向他们指出这些问题。“以书面形式通知可能会伤害公司的错误做法”，往往会让最不友好的管理层点击进入 CYA 模式，并对此采取行动。

当我们看到那些壮观的失败时，我认为这都是成长的烦恼；你可以用敏捷/DevOps/cloud 和/或 Kubernetes 做更多的事情，这类事情会突然出现，并最终得到解决。我仍然相信。你们都还在测试边界，挑战极限，让一切看起来都很好。有时候到达那里会很痛苦。修复应用程序，制定政策，继续努力，知道你正在扼杀它(作为一个行业。如果你不是，就不要嘲笑我的总括赞美，修复你自己。
