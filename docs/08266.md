# 远离单一应用程序开发

> 原文：<https://devops.com/the-move-away-from-monolithic-application-development/>

**从单片应用开发转移**

企业需要敏捷性来提供更好的服务，满足不断变化的需求，并抓住新出现的机会。许多工作的核心是基于现代开发策略和技术的应用程序。具体来说，许多企业正在摆脱过去那种难以修改且需要很长时间才能推向市场的单一应用程序。

为什么要转变，为什么是现在？[超过 80%](https://www.gartner.com/en/doc/3874972-realizing-the-benefits-of-superior-customer-experience-a-gartner-trend-insight-report) 的组织期望主要基于客户体验进行竞争(CX)。因此，零售、银行、保险、金融服务和旅游/酒店等竞争最激烈的市场中的企业无法承受等待数月或数年来发布新的服务、应用或功能。

底线是软件开发团队面临着在更短的时间内交付更多东西(新的应用程序、增强的特性、更多的访问选项)的压力。一份被广泛引用的过去的行业研究让我们看到了这个问题。领先的公司每天例行执行多次部署。相比之下，表现不佳的员工报告称，他们一个月最多只部署一次。(在极端情况下，像亚马逊和谷歌这样的公司一天要做数千次部署。)

单一的应用程序开发在这种快节奏的环境中是行不通的。现有应用程序中的简单更改，如合并新数据集、在前端应用程序中添加新功能或使用不同的分析方法，都需要修改、重新测试和重新部署整个应用程序。这都需要时间。很多时间。

更好的方法是使用现代应用程序开发方法。特别是，正在转向使用基于松散连接的微服务和元素的云原生架构。

云原生方法特别适合现代应用程序，原因有几个。例如，云原生方法可以帮助组织解决需要不断更新和新服务的客户体验驱动因素。云原生应用的每个组件都可以作为独立的微服务进行部署和维护。如果一个广泛的应用程序的任何元素需要更新或更改，只需要接触那个元素。集团应用程序的其余部分保持不变。

**z/OS 与云原生系统配合良好**

许多行业在加速开发和交付更加定制化、个性化的客户体验的巨大压力下使用 z/OS 系统。如上所述，它们是零售、银行、保险、金融服务和旅游/酒店。政府也可以被添加到列表中。它是 z/OS 系统的大用户，并不断尝试通过新的或改进的 web、移动、聊天、社交和电话应用程序来改善对其服务的访问。

有两种方法可以增强和开发这些使用现代开发方法的系统的应用程序。一个是让运行在 z/OS 系统上的数据、应用程序和服务作为 API 和微服务供大型云原生应用程序使用。

另一种越来越受欢迎的方法是为 z/OS 系统本身提供现代开发工具。具体来说，远离整体架构的现代应用程序开发方法需要某些基本元素。许多依赖于开源开发技术、工作流自动化以及持续集成和持续交付(CI/CD)工具。

将这些常用工具引入 z/OS 环境有助于企业加快新应用程序的开发，以满足客户不断变化的需求。它允许企业利用其 z/OS 系统上的数据和应用程序，并创建创新的新应用程序。

在一个越来越难找到具有本地 z/OS 编程专业知识的开发人员的时代，将熟悉的 DevOps 工具引入 z/OS 环境有几个好处。那些拥有原生 z/OS 技能的人可以专注于核心应用程序，而将支持新前端或增强 web 界面等任务留给其他人。毕竟，有更多拥有现代开发技能的开发人员。

将 DevOps 工具引入 z/OS 环境还可以让企业保留这些系统的所有优点。也就是说，企业可以确保同样的可靠性、卓越的性能、严格控制的安全性和巨大的可扩展性得以保留，并且可以与更广泛的云原生应用集成使用。
