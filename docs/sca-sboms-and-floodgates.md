# SCA、SBOMs 和闸门

> 原文：<https://devops.com/sca-sboms-and-floodgates/>

有两个标准用来确定一个新思想的普及性。提供易于理解的解决方案和客户需求。考虑到这两个因素，在单个 IT/DevOps 市场中可能成为市场差异化特征的功能在多个市场中变成了一波可供选择的浪潮，组织可以(也应该)从中进行选择。

这是怎么开始的？嗯，我一直在涉及开源广泛使用的领域工作。在这一点上，这相当于说，“我一直在从事这项工作”，但这是一个不同的博客。问题是，今天，使用开源意味着一个复杂的软件基础设施，传统上，没有人真正掌握它。那个项目直接包括什么？那个项目间接包含了什么？

但是后来我们遇到了一些与常用开源软件相关的非常严重的安全问题。我们有从成千上万的应用程序中提取出来的常用编程语言模块。然后我们就开始关注了。或者，更准确地说，然后我们终于着手解决一个我们一直忽视的问题。

这个市场对于不同的用例以及有两个工具集来实现它所认为的一项工作这一事实感到有点困惑。为了清楚起见，我们来谈谈复杂性。首先，软件材料清单是软件项目中包含的内容的列表。这里的复杂性是双重的。从编程的角度来看——意味着在开发过程中为程序员设计的工具——“包含”意味着库、模块等。应用程序调用它来做组织需要的事情。这些都“包含”在构建过程中。从系统的角度来看，“包含”意味着在生产时安装在映像上的所有东西。这些是相似但非常不同的列表。开发者列表是生产列表的子集，因为开发列表中的所有项目都包含在生产中，但是生产列表将具有用于支持所开发的应用的其他软件。设想应用程序拥有所有的库和模块，但是假设 SSL 终端将会存在。因为会的。在图像中。这就是区别。多年来，许多产品都有一个内部化的 SBOM 版本——开发者安全工具和发布扫描工具几乎都需要一个他们需要扫描的所有东西的列表。但是很少有人公开这些列表，许多工具也没有这些列表。

软件组成分析(SCA)采用 [SBOM](https://www.cisa.gov/sbom) 并增加价值。虽然包含所有内容的列表很有用，但是列出实际使用的所有内容的子集更切题。在包含的库/包中发现的缺陷列表以及如何解决它们的建议也是如此。这可以从源或生产 SBoM 端完成，具体取决于实施情况。

一旦这两种工具的价值变得清晰——并且实现展示了如何实现它们以及好处是什么——闸门就打开了。现在，很少有 DevOps 或安全工具不提供这些功能中的一个或两个。这给我们带来了财富的尴尬。我们使用哪些工具进行合规性和安全性审查？正如我们处理数据一样，我将建议选择一个单一的产品作为“真实的来源”,并与。在这个时候，我没有关于哪个*更适合这个目的的建议，只是建议，如果你为不同的目的使用不同的，你可能会得到不同的结果，这根本不会有助于简化事情。因此，看看您已经可以访问的产品—它们存在于足够多的产品中，您现有的基础架构中可能有不止一个可用的产品—并使其成为您的标准。*

然后，使用该标准来帮助监控开源的使用。监控有助于开源许可、应用和基础设施安全、映像简化和合规性。因此，将未使用的包从映像中剥离出来，并清理构建过程中不再调用的库。更快地响应问题，并让您精心打造的应用基础架构保持运转。并且不停的踢；你的用户*应该*欣赏它，即使他们不欣赏。
