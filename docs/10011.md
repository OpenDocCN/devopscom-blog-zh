# MACH 架构介绍

> 原文:[https://devops.com/introduction-to-mach-architecture/](https://devops.com/introduction-to-mach-architecture/)

近年来，构建端到端数字体验的需求变得越来越重要。但是，为了迅速交付新的功能和服务，组织必须加倍提高敏捷性和可重用性。作为这一趋势的一部分，应用程序变得越来越模块化，通常使用预构建的微服务进行构建，并集成通用 SaaS 以避免重新发明轮子。

MACH 是一种非常适合这些趋势的建筑风格。它代表**M**microservices-based、 **A** PI-first、 **C** loud-native、 **H** eadless。这场运动的中心是马赫联盟，一群遵守这些原则的 API 优先公司。在撰写本文时，随着越来越多的 API 第一公司进入市场，以及对云原生技术的需求增长，MACH 正在赢得更多的关注。

我最近与 Cloudinary 技术合作伙伴副总裁 Gary Ballabio 聊了聊 MACH 的发展趋势。据 Ballabio 称，MACH 提供了“同类最佳”的技术，避免了大型平台的膨胀。然而，对于软件消费者来说，MACH 风格的构建需要将各个组件缝合在一起，并有效地协调它们。下面，我们将探索 MACH，并考虑使用符合这种方法的工具的优点和缺点。

## 马赫是什么？

MACH 是一种新兴的软件架构风格，被许多商业和数字体验工具所采用。为了更深入地理解 MACH，有必要理解微服务、API 优先、云原生和无头的底层概念。所以，让我们先把这些熟悉的术语分开:

*   **微服务**:与整体开发不同，微服务架构将功能组织成独立的、专门的组件。这提高了可重用性，并允许更频繁、更有针对性的软件更新。
*   **API-first** : API-first 将[设计-first](https://nordicapis.com/what-it-means-to-go-api-first/) 更进一步。这意味着将消费模型(API)置于其他关注点(如应用程序或 UI)之前。[Web API 已经变得无处不在](https://devops.com/api-sprawl-a-looming-threat-to-digital-economy/)用于连接微服务生态系统，并向合作伙伴和公众消费者公开数据和功能。
*   **原生云**:专为云的弹性和可伸缩性而构建的项目的总称。
*   **无头**:无头的意思是将前端或“头”与后端分开。通过这种方式分离开发，网站和应用程序可以更好地满足多种平台和设备的需求。

根据 Ballabio 的说法，公司可以通过集成采用 MACH 原则的工具来利用云就绪的最佳功能。Ballabio 说，整合一个单一的 SaaS 来满足特定的需求也可以被视为使用大型、全面的企业商务套件(如 Salesforce、Adobe 或 SAP)的替代方案。

马赫风格的专业 SaaS 服务有很多例子。其中包括用于搜索和推荐的 Algolia、用于媒体管理的 Cloudinary、用于移动体验的 Builder.io 或作为无头 CMS 的 Contentstack。所有这些工具都是 MACH 联盟的成员，该联盟由 70 多家供应商、合作伙伴、初创公司、系统集成商和其他致力于促进 MACH 方法的大使组成。尽管重点一直放在商业和数字体验上，MACH 正在向其他领域扩展。

> MACH technologies 支持一个可组合的企业，其中每个组件都是可插拔、可伸缩和可替换的。

一些读者可能会问，“这与 [Jamstack](https://devops.com/jamstack-boards-the-enterprise/) (JavaScript、API 和 markdown)有什么不同呢？在某种程度上，MACH 类似于 [Jamstack](https://devops.com/jamstack-will-revolutionize-how-websites-are-built/) ，因为这两种方法都致力于无头前端体验和由 API 支持的模块化后端。但是 Ballabio 说，一个关键的区别是 MACH 并没有规定特定编程语言的使用。

## 马赫的优点和缺点

MACH 结合了我们在软件领域看到的许多发展趋势。数字应用变得越来越可组合，[云优先正在兴起](https://accelerationeconomy.com/cybersecurity/hybrid-multi-cloud-is-the-new-normal/)，无头模型可以带来定制化的好处。然而，引入这种风格也有一些潜在的缺点。所以，让我们来权衡一下采用 MACH 架构的利弊。

### 利益

*   Ballabio 说:**灵活性**: MACH 最大的好处是它提供的灵活性。“能够根据自己的独特需求选择想要使用的服务是有价值的。”为电子商务、支付、CMS 或电子邮件等应用循环特定的 API 优先工具，可以实现大量定制。
*   **减少浪费**:这也可以减少浪费，因为你可以通过只购买每个项目所需的东西来减少足迹，而[免费增值模式](https://nordicapis.com/when-api-monetization-does-and-doesnt-make-sense/)可以让消费者只为他们使用的东西付费。当您想要单个组件，但不需要更全面的平台的附加功能时，这很有帮助。
*   同类最佳:利基 API 优先产品专注于做好一件事，这种特别的专注可以提高质量和敏捷性。它还将工程师从一个广泛的产品开发生命周期的突发奇想中解放出来。
*   开发者优先:最后，大多数数字体验都有共同的功能，开发者不必重新发明。马赫支持开发者优先，巴拉比奥说，这对投资于 T2 开发者体验的公司来说是一笔财富。

### 缺点

*   **增加维护**:消费很多第三方云服务，必然带来新的维护障碍。这些包括跟踪、监控和保护所有集成点。它还要求对新版本更新做出响应。
*   **缝合在一起**:自然地，当你添加一个基于云的抽象层到你消费的服务中时，消费者就有责任将它们缝合在一起，使组件能够相互交流。
*   **启动成本**:balla bio 承认，由于服务之间的协调，初始成本和工程时间可能超过实施一个整体套件。“但你会获得更多长期利益，”他说。
*   **可能需要帮助**:那些不习惯微服务开发和 API 集成的人可能需要帮助。这可能需要系统集成商或中间件的支持来协调连接。
*   **缺乏标准** : MACH 更多的是一种通用架构，而不是标准规范。因此，具体的技术和具体的行为可能因服务而异。

## 马赫的未来

MACH 联盟最近的一项研究发现，79%的技术领导者计划在下一年加大对 MACH 技术的投资。尽管这种风格主要在欧洲流行，巴拉比奥预测美国将很快赶上它。MACH 联盟提出并倡导一个开放的最佳企业技术生态系统。

随着[可组合架构](https://www.technologyreview.com/2022/09/28/1059666/growth-means-composability-brick-by-brick/)的兴起，MACH 方法正引起云原生领导者的兴趣，他们看到了使用预制构件组装应用程序的优势。MACH 特别适合绿地开发，在这里可以干净地集成新技术。

MACH 代表了一类共同成长的新公司。巴拉比奥评论说，他感觉自己是一场运动的一部分，是一个行业的一部分——可以说，这场运动可能会破坏更大的商业生态系统。

但是尽管马赫给一张脸起了个名字，大多数技术人员还是不知道这个缩写。虽然这个概念目前可能还停留在供应商的领域，但 Ballabio 预见它将成为消费者和最终开发者更常用的说法。
