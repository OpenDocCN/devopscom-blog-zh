# Armory 旨在将持续交付转变为一项服务

> 原文:[https://devo PS . com/armory-aims-to-turn-continuous-delivery-into-a-service/](https://devops.com/armory-aims-to-turn-continuous-delivery-into-a-service/)

Armory 本周普遍提供了一种连续交付即服务(CDaaS)产品，该产品有望使更广泛的组织以编程方式部署应用程序变得更简单。

从三月份开始，[军械库持续部署即服务](https://www.armory.io/blog/armory-continuous-deployments-as-a-service-is-now-available/)平台已经进入测试阶段。它不同于 Armory 现有的开源 Spinnaker 持续交付(CD)平台，因为它基于一个专有的声明性平台，该平台旨在与第三方持续集成平台集成。

Armory 首席执行官 Jim Douglas 表示，目标是满足已投资 CI 的组织的需求，而不是要求他们淘汰并更换整个 DevOps 平台。他补充说，该公司仍致力于提供 Spinnaker 的策划实例，但显然有许多组织希望体验更具主见的 CD 平台的好处，这种平台更容易通过云服务访问。

Douglas 指出,“军械库持续部署即服务”产品应用了许多针对 [Spinnaker CD](https://devops.com/?s=spinnaker+CD) 平台的最佳实践，而不需要组织有专门的团队来管理 CD 平台。他指出，IT 团队可以在不到 30 分钟的时间内开始使用该服务部署应用程序。

Douglas 说，Armory 连续部署即服务的第一版针对在 Kubernetes 集群上部署基于微服务的应用程序进行了优化，但 Armory 将很快将服务范围扩展到运行单片应用程序的平台。他补充说，该公司还计划提供内部版本。

Douglas 说，在这两种情况下，IT 团队将能够参与从部署的金丝雀测试到应用程序出现故障时的自动回滚的所有工作。

经常使用光盘平台的组织数量仍然相对较少。历史上，用于运行应用程序的每个平台都是独一无二的，因此自动化交付一直是个问题。在某些情况下，CI 平台已经使用定制脚本进行了扩展，以自动化应用交付，但这些工作通常不会扩展到整个扩展企业。

Douglas 说，随着越来越多的 it 团队开始构建和部署基于微服务的应用程序，CD 成为一个更独特的 IT 学科只是时间问题。面临的挑战是使 IT 组织能够实现这一目标，而不必雇用一小群现场可靠性工程师(sre)。仍然会有很多 DevOps 团队倾向于部署 Spinnaker，但是 Douglas 说，也有很多情况下，不太复杂的 CD 方法就足够了。

很难说是否很快会有不同级别的 DevOps 平台服务可供组织调用，这取决于项目的复杂性和他们自己的内部文化偏好。然而，随着应用程序开发速度的不断加快，自动化 CD 的需求将成为一个更加紧迫的问题。实现这一目标的方式可能有很多，但如果历史有所启示，阻力最小的路径可能是通过云。
