# IT 管理如何治理一切运营？

> 原文：<https://devops.com/how-does-it-management-govern-everythingops/>

工程师用来在云中提供软件架构的工具数量开始变得有点失控。做一件事有许多相互竞争的方法。大多数情况下，人们只是采用一种新工具来解决当前的 DevOps 问题，而不总是考虑解决方案的寿命或其他部门如何处理它。这可能会导致碎片化和技术债务，使公司所有部门很难找到一致性并同时提高速度。

我最近采访了 CloudBolt Software 的首席执行官 Jeff Kukowski，探讨为什么这些问题持续存在，以及高管应该如何在他们的组织内实现云管理。根据 Kukowski 的说法，许多类型的 Ops 已经渗透到大公司中。不同的方法通常存在于每个部门或部门间，团队使用不同的软件栈。

根据 Kukowski 的说法，在新的云秩序中保持敏捷并解决这种状态需要一种新的思维框架。这样做将要求组织消除围绕所有权和 [IT 治理](https://accelerationeconomy.com/low-code-no-code/it-governance-is-broken-heres-how-we-can-fix-it/)的错误二分法。

## 一切顺利吗

在这一点上，大多数大型数字服务套件都采用了复杂的混合和多云模式。许多团队已经采用了他们自己独特的 DevOps 形式，以应对交付软件和维护微服务架构可靠性的管理困难。Kukowski 解释说:“不同的群体朝着不同的方向努力，这就形成了一场持续的战斗。”。

您可以将其视为围绕不同业务职能出现的独立运营方法。例如，FinOps 是为了控制支出，ITOps 是为了提供服务，DevOps 是为了提高发布的流动性，而 [DevSecOps](https://devops.com/6-traits-that-define-devsecops/) 的目标是将安全性放在发布过程的最前面。最后， [BizDevOps](https://devops.com/seeing-the-bigger-picture-with-bizdevsecops/) 是关于提高业务成果的可观察性。

总的来说，首席信息官们致力于加快数字化转型的步伐。Kukowski 说，现在的问题是“不同运营部门之间的争斗很容易导致 IT 治理的崩溃”。他解释说，这种紧张引入了几个错误的二分法。Kukowski 说，除了降低灵活性之外，缺乏 DevOps unity 还会阻碍可重用性，导致浪费和增加 IT 开支。

## 解决方案:向左转移治理

人们相信，如果你只给工程师使用他们想要的任何东西的钥匙，真正的创新就会到来。Kukowski 警告说，虽然这可能是真的，但如果在这个新的云秩序中没有更好的领导，一些事情可能会很快失控。相反，他认为供应工具的内置护栏是必要的。“我们有太多的工具——我们需要更好的框架来将它们结合在一起。”

增加云治理不仅有助于将不同的工具决策联系在一起，而且 Kukowski 认为它还可以减少每个工具上必需的认知负荷。“向左移动会提高速度——它不会让人们慢下来，”他说。与普遍的看法相反，他说向左移动护栏可以提高速度，这可能大大超过将开发人员束缚在单一方法上的负面影响。

CloudBolt 预测，孤岛式管理将在未来一年转向围绕工具的组织治理。但不仅仅是工具决策需要深谋远虑，“这是领导和文化的方法，”Kukowski 说。他说，需要更大领导力的一个常见领域是为应用程序标记工作负载。“任何地方都没有做好。”

或者，考虑集装箱化。Kukowski 解释说，容器已经成为大多数现代云技术栈的基础，然而，部署、配置和运行容器所需的认知负荷仍然很高。您是否允许每个开发人员对每个容器进行定制编码，或者您是否支持自助式可重复供应流程？他解释说，如果标准代码没有被重用，你就不会获得速度或治理。此外，随着绿色 IT 计划成为优先事项，公司应该寻找方法来重用流程并尽可能减少浪费。

## 最后的想法:左移，自动化和重用

对于云原生公司来说，重点一直是速度和增长。工程师绝对是这种专注和创新的驱动力。然而，这种创新不能以牺牲所有部门的创新为代价。Kukowski 说:“云中的创新内容不能与治理、ESG 和安全性分开。

未来云管理的一大障碍将是识别和统一这些自动化孤岛，这些不同部门的工具和偏好。这也需要不同部门有不同的心态。“为任务解决问题和为重用和速度解决问题是有区别的，”Kukowski 解释道。他说，交付重用和速度将需要更少的定制 DevOps 自动化和解决未来问题的更好基础。

“左移，自动化和重用—人们不应该接触它。这就是需要应用到新的云秩序中的想法，”Kukowski 说。
