# 我们不知道的东西是危险的

> 原文：<https://devops.com/what-we-dont-know-is-dangerous/>

突击测验:在您的整个组织中，有多少数据库管理系统(无论是 RDBMS 还是 [NoSQL](https://www.mongodb.com/nosql-explained) 或者甚至是平面文件)正在运行？不是实际的数据库，而是管理系统？

您有多少[个容器](https://containerjournal.com)直接或间接地暴露在互联网上？不是容器管理系统，因为你只运行其中的几个，而且它们都相对较新，但是实际的容器呢？

这些问题可以继续下去。不久前，由于存在可识别的相关成本，云库存管理成为一件流行的事情——查找和删除不再需要的云实例、服务和存储成为拥有中等规模云存在的组织的正常运营的一部分。

但是现在，我们已经到处蔓延。因此，让我们将[库存管理](https://devops.com/?s=inventory+management)扩展到所有软件，去掉那些不经常需要的……因为它确实会很快建立起来。

是的，有一些公司正在尝试这样做，我鼓励你去看看他们。我没有，因为我们在根深蒂固的技术库存是可以管理的，没有管理费用。我们的开发和运营时间是有限的，所以我们对现有的产品和所需的维护保持严格的控制。

但是对于较大的组织，您拥有的资源越多，攻击面和潜在的数据筛选就越大。所以要确保如果不需要，就把它拿走。在容器时代，这实际上需要相当多的调查；首先，一个正在进行的项目的所有容器的列表(即使在一个中等规模的公司中，“每个应用程序都在一起”，不会创建这个列表，因为随机的容器也在那里)。然后是与已知项目不相关联的所有容器的列表，这些容器可以被检查并确定用途。

一点也不光鲜。有点吃力。然而，这符合组织的最大利益，这种艰苦的工作应该激励您的团队制定管理容器生命周期的策略。就像 Docker 文件中的注释，说明何时不再需要容器，或者它的用途，这样将来你就可以轻松地处理掉它，而不会有压力。

如果你在一个集装箱化的环境中，这个问题几乎每天都在增长。说真的。在我们的上一个项目中，我们有 12 个活动的容器，然后我们将它减少到我们长期需要的 5 个。其他的是我们在关注实现相同目标的不同方法之前所使用的方法或工具中的 R&D。由于我们是一个小团队，我们只是在项目进入焦点时进行清理，但一个更大的组织需要更加专注于清理工作。

与此同时，这些资产让客户接触到您的组织，并允许员工管理系统。继续摇摆它，记住要捡起来，特别是在尖端的新项目之后，但也偶尔作为一个正常的清理步骤。
