# 在 SDLC 中嵌入可访问性的 5 种方法

> 原文:[https://devo PS . com/5-ways-to-embed-accessibility-in-your-SDLC/](https://devops.com/5-ways-to-embed-accessibility-in-your-sdlc/)

开发一个可访问的软件开发生命周期(SDLC)需要每个人在每个项目开始的时候都参与进来。通过致力于开发可访问的产品，您将避免在产品路线图结束时部署之前进行昂贵且及时的补救。

## 创建可访问的软件开发生命周期(SDLC)

可访问的 SDLC 是指从规划到维护的每个阶段都集成了可访问性的 SDLC。这包括可访问性策略、针对该策略的自动化测试、整合到您的 QA 流程中，以及验证和记录一致性要求，如自愿创建产品可访问性模板(VPAT)。

集成可访问性与测试任何其他质量标准并没有什么不同，比如跨浏览器兼容性或其他缺陷。以下是支持您的努力所需的五个关键要素:

**政策—**从建立您的可访问性目标开始，然后根据重要性进行优先排序(这有助于未来的 sprint 规划)。在您的策略中定义这些目标和优先级，以便您可以使用它们来衡量您的成功并了解您需要改进的地方。

**自动和手动测试—**自动测试通过使用软件扫描您的数字资产(无论是在生产、开发还是准备阶段)来工作，并使用可访问性规则引擎来检查错误。它可以快速有效地捕捉机器可检测的 web 内容可访问性指南(WCAG)故障(可访问性标准)，工程师可以在开发过程中修复这些故障。自动化测试可以通过多种方式进行；然而，最适合您的方法将取决于您的 DevOps 团队能够促进的集成水平。手动评估远比自动扫描更全面。它们还考虑到了基本的用户体验，并且它们可以捕捉远远超出今天机器可检测的错误。

**集成用户故事开发或持续集成/持续部署(CI/CD)–**一旦你发现了失败，你会怎么做？它们应该像任何其他阻止客户访问或使用你的网站或应用程序的错误或故障一样被处理。您需要在您的票证跟踪系统中捕获这些信息。许多组织使用敏捷 sprints 和项目管理软件来创建标签，然后将它们输入到开发管道中，并进行优先级排序、分配和完成。找到一个解决方案，让您能够轻松地将 WCAG 的失败转换成用户故事(手动地，或者更好的是，通过自动化)，这是为要做的工作提供优先化管道的关键。

**知识和资源—**有专门的开发技术用于解决可访问性失败。如果不能访问全面的可访问代码技术库，开发人员将花费大量时间在网上寻找解决方案。是的，如果你观察足够长的时间，你可以找到好的解决方案，但即使如此，你怎么知道那个解决方案对你的情况来说是合法的还是最优的呢？让某个问题的相关解决方案唾手可得(消除了进行大量在线研究的需要)可以让您的 DevOps 团队更顺利地工作，减少中断。

**问责制—**可访问性分析建立问责制。您的 DevOps 团队在减少 WCAG 故障方面的效率如何？按照规定的节奏跟踪数字资产的故障，可以让您看到并记录一段时间内的趋势。您还可以跟踪活动，例如标记为可访问性补救或缺陷的票据数量，或者表示积极努力的新特性或功能。

无论您是在任何规模的组织中的小型或大型开发团队中，通过在开发周期的每个阶段实施连续的可访问性检查来采用可访问的 SDLC 是确保您的网站或应用程序符合 WCAG 和 ADA 的最有效的策略。
