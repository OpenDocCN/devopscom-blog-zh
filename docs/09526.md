# Dynatrace 增加了综合测试来验证应用体验

> 原文：<https://devops.com/dynatrace-adds-synthetic-tests-to-validate-app-experiences/>

Dynatrace 今天将其提供的应用发布管理功能扩展到[包括验证和确保用户体验的综合测试](https://www.businesswire.com/news/home/20220628005441/en/Dynatrace-Extends-Automatic-Release-Validation-Capabilities-to-Improve-Software-Quality-and-Resiliency)。

Dynatrace 产品营销总监 Saif Gunja 表示，用户体验验证和用户体验保证(UXA)功能涵盖了从应用可用性和性能到基于 DevOps 团队定义的服务水平目标(SLO)的特定功能的实际参与的所有内容。

Gunja 说，Dynatrace 将应用发布管理视为其[可观察性](https://devops.com/?s=observability)平台的扩展。虽然应用程序发布管理在历史上被视为连续交付(CD)平台的扩展，但 Gunja 表示，可观察性平台——如 Dynatrace 平台，其中注入了戴维斯人工智能(AI)引擎——提供了更好地自动化该过程所需的可见性。

Gunja 说，事实上，随着 DevOps 的不断发展，很明显，随着 DevOps 过程的成熟，CD 和持续集成(CI)变得越来越松散。大多数组织还没有实施 CD，只是因为它太难实现了。应用程序部署的每个平台都是独一无二的。Gunja 说，Dynatrace 通过它提供给仪器应用程序的代理软件，为自动化应用程序发布管理提供了基础。

目前还不清楚 DevOps 团队是否正在重新评估他们的应用交付方法，但 Gunja 指出，对 Dynatrace 方法的兴趣主要来自已经安装了 [Jenkins](https://devops.com/?s=Jenkins) CI/CD 平台的组织。他指出，已经构建的和需要支持的扩展的数量已经产生了一定程度的技术债务，这对许多 DevOps 团队来说是不可持续的。

Gunja 补充说，许多 IT 领导者认为他们宁愿分配更多的资源来构建应用程序，而不是维护 DevOps 平台。

大多数 DevOps 团队致力于无情地自动化尽可能多的 IT 流程。可观察性一直是 DevOps 追求这一目标的核心原则。未检测的应用程序可以通过编程方式进行管理。Gunja 说，问题是提供可观测性是不够的——devo PS 团队需要能够根据可观测性平台收集的数据，在整个软件开发生命周期中采取行动。

显然，就可观测性而言，现在还为时尚早。虽然应用性能管理(APM)平台已经使用了几十年，但是直到最近才出现能够关联应用和 it 基础架构数据的可观察性平台。由于开源代理软件的兴起，检测 it 环境的成本也越来越低。当然，一如既往，挑战在于部署和维护所有代理软件。

不管可观察性和 DevOps 的方法如何，很明显 it 团队很快就会对他们的 IT 环境有更多的可见性。IT 团队通过排除过程来确定 IT 问题根本原因的日子终于要结束了。然而，可观察性本身并不是目的，而是前一个 DevOps 时代开始的结束。
