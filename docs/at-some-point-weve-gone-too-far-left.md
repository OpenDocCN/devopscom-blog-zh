# 在某种程度上，我们已经走得太左了

> 原文：<https://devops.com/at-some-point-weve-gone-too-far-left/>

我们这些参与 DevOps 的人有一种带着眼罩看世界的倾向。很容易陷入“如果你只有一把锤子，所有东西看起来都像钉子”的陷阱。我们使用短语“[左移](https://devops.com/?s=shift+left)”的态度是，这是对其中每个问题的解决方案。

为了清楚起见，它不是。这是我所能做出的最直率、最明确的声明。DevOps 很棒，在许多领域都有帮助，但它不是唯一的解决方案，也不是每个问题都是 DevOps 工具链的问题。

最后，IT 中的许多关键工作都与开发运维无关，也不会受到开发运维的影响。我将提到两个显而易见的领域，并深入讨论其中之一。

首先，不管你在敏捷/DevOps 的道路上走了多远，你乘坐的马车都是独立的，与 DevOps 过程截然不同的。如果您有一个数据中心，所有的硬件最终仍然是必须配置和管理的硬件。对于“但是我们 100%在云中！”人群(这些人大声而有力地说，我怀疑他们试图说服自己这是一件好事)——最终，你仍然使用网络和计算机来连接。由于封锁造成的大规模在家工作的迁移非常清楚地表明，建立和支持这两者是必要的。ops 和 DevOps 社交媒体上的一些恐怖故事是…丑陋的。员工不想也不应该成为 IT 专家来完成他们的工作。

另一个简单的问题是安全性。是的，将安全性融入开发和开发运维工作流。是的，让操作人员更加注意安全。是的，尽可能实现安全自动化，因为我们长期缺乏安全人员。但是，最终，很多安全性都是在 DevOps 之外的。虽然 [Kubernetes](https://containerjournal.com/?s=Kubernetes) 安全绝对是 DevOps，比如密码管理，绝对不是。这只是众多例子中的一个。

所以我要问(但不回答)这个问题:界限在哪里？向左多远才算太远？答案取决于您的组织以及组织中存在的工具/环境。在以云为主的环境中，这个问题的答案与在以数据中心为主的环境中截然不同。在这个时间点上，任何组织都很少是纯粹的非此即彼，所以有相似之处，但我们很少能说出确切的解决方案。

安全性类似于成本中心。这实际上是一种保险，但很少有组织会这样认为。即使他们这样做了，也不会有太大的不同。正如我父亲曾经宣称他和我母亲是“保险穷人”，并取消了一半的保单，大多数机构都希望控制安全成本，不管他们承认与否。当人们听到“安全性与开发业务相结合”的说法，试图尽可能扩大预算时，糟糕的事情往往会发生。

DevSecOps 在开发阶段就实现了安全性，但它并没有取代其他安全性，只是让我们的应用程序得到了更多的保护。在某些情况下，安全测试时间会大大减少，但由于在人员紧张的情况下，所有形式的测试都是首先要做的事情，所以这并不是取代正在发生的事情，而是使我们有可能做我们一直应该做的事情。

我们的工具链已经有了自己的生命——在许多组织中，员工需要维护它们——这并没有帮助，因为它们也是 DevOps 努力帮助的应用程序。一旦你投入时间和金钱来建造这座大厦，你就会想尽可能地利用它。因此，许多组织希望将 IT 中的所有东西都转移到工具链章鱼中。

你比任何人都了解你的组织。在你们所有辉煌复杂的环境中，是你们让它日复一日地工作。开始说什么叫过分；当在已经很复杂的工具链中增加一个步骤需要调整时。抵制一切需要左移的东西(在适当的时候，为了清楚起见，我要再说一遍，DevOps 为我们做了很多了不起的事情，但太多的好事是我们生活的噩梦，从历史上看，是明天的技术债务)。继续摇摆，确保你在正确的地方摇摆正确的东西。
