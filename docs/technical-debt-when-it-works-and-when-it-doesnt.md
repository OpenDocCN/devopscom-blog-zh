# 技术债务:何时有效，何时无效

> 原文：<https://devops.com/technical-debt-when-it-works-and-when-it-doesnt/>

每个软件团队都可能面临技术债务——需要返工的小开发缺陷的累积。就像金融债务一样，它们随着积累而产生利息，表现为困难的软件修改或增加新功能的挑战。时间通常是罪魁祸首；您的开发团队有一个创建的时间框架，但是您的领导团队希望将软件更快地推向市场。当涉及到在构建一个干净和完美的软件或者按时交付之间做出选择时，你通常需要向来自高层的压力低头。在这一点上，您的协作可能会偏离轨道，不完整的代码片段会混入其中。

没有仔细的监控，[技术债务](https://devops.com/?s=technical+debt)最终会减缓您的开发和部署过程，降低您的产品质量，并限制您在不断变化的市场中的创新能力。但是技术债是单方面不好吗？不总是。有时候这是为了一个更大的战略。让我们来看看不同之处。

## 什么时候债务是好策略的一部分？

Citrix 产品营销高级总监 Pankaj Gupta 表示，当技术债务阻碍了一个组织的运作能力时，这是一个明确的信号，他们需要清除债务，从头开始。

“但在一些情况下，让债务累积是合理的，”他说。“例如，如果解决技术债务的成本现在明显高于未来；如果债务没有影响您当前和短期的业务需求，或者您有一个紧急版本，如重大安全漏洞修复程序。”

CI&T 的首席技术官丹尼尔·维韦罗斯(Daniel Viveiros)建议团队在做出正确的权衡时要牢记大局。“对我来说，这是关于把业务放在第一位，并理解不断激活价值是成功的必要条件，”他说。“管理良好的技术债务可以成为缩短交付周期的有效工具。为了优先部署而推迟重构、现代化工作和其他常见债务，是工程团队每天都要处理的一个经常性的权衡。”

## 技术债太多有什么征兆？

古普塔为技术性债务超载提供了一些警示信号。“当解决技术债务的成本和/或时间高于重新开始时，你知道你正在积累太多的技术债务，”他说。"每次发布和部署所花费的时间都在不断增加."

一个不太明显的结果是:德沃普人大批离去。开发团队将最先感受到过多技术债务的影响，例如完成任务更加困难，长时间工作以修复频繁的故障。古普塔说:“员工离开公司是因为对遗留系统的工作感到失望，他们把大部分时间花在解决临时问题上，经常性地失踪和随时待命。”债务达到那个水平，辞职开始增加。

Viveiros 指出，编译后的债务将影响[系统弹性](https://securityboulevard.com/?s=resilience):“bug 产生的频率越来越高，变更失败率也越来越高。速度被认为是有风险的，而不是对数字交付过程有益的东西。”

他说，债务过多的最终标志是糟糕的业务表现和收入损失。“如果停机等于失去机会，技术债务将成为金融债务，”Viveiros 说。“过多的技术债务可能会影响产品的感知质量。因此，产品和品牌的形象将受到损害。”

## 管理债务的最好方法是什么？

未能处理技术债务意味着将它踢给另一个开发人员去处理——没有人想成为那个人。那么，控制和回报的最佳方式是什么呢？首先，将它作为一项协作工作来对待，并实施这四项实践。

1.  **确定债务的种类**。并非所有的技术债务都是平等的。区分你当时接受的债务和你无意中发现的债务，并将其分为代码债务、设计债务、构建债务和其他形式，以确定起源模式。
2.  **分析并自动化。**分析你债务的来源，寻找收紧工作流程或自动化某些测试和流程的方法。通过减少常见错误和隐藏的 bug，您可以阻止它们滚雪球般变成技术债务。
3.  制定新的政策和标准。这些应该澄清什么时候债务有意义，什么时候债务会造成不可接受的损害。例如，一个安全补丁的立即发布会被认为是可接受的，而最终会导致相当长的停机时间的错误是不可接受的。
4.  **向 IT 和业务领导传达成本。**当另一个超高速截止日期到来时，确保你的领导层了解产品质量和开发人员保留的风险。有了明确的价格标签，他们更有可能改变交付日期或为更多的开发者提供资金。

像大多数失控的支出一样，有害的技术债务始于糟糕的决策和一定程度的否认。通过更清晰和更有远见地处理开发和运营方面的决策，团队可以给自己更多的喘息空间和空间来开发优秀的软件。
