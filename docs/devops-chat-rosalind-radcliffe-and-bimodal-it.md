*Alan Shimel:* 大家好，我是另一位 DevOps Chats 的 Alan Shimel，我的嘉宾是——我的一位朋友，IBM 的 Rosalind Radcliffe。罗莎琳德，欢迎你。

罗莎琳德·拉德克利夫:谢谢。**

*Alan Shimel:* Rosalind，我想你以前也在 DevOps 的聊天室出现过，当然我们也一起参加过网络研讨会和视频会议，还有很多其他的事情。但是对于那些可能不熟悉你的观众来说。你想简单介绍一下背景吗？

罗莎琳德·拉德克利夫:当然。我是 Rosalind Radcliffe，我是 IBM 的一名杰出工程师，负责我们企业系统的 DevOps 战略。因此，我不仅负责推动我们的产品方向，而且我真的花了很多时间与客户在一起，帮助他们实现企业系统(主要是大型机和记录系统)的 DevOps 转型。

*艾伦·希梅尔:*优秀。Rosalind，当然，我们今天要谈一谈这个术语，它是一种新兴的术语，即双模 it，然后我想花点时间和你谈谈即将到来的互连会议。也许我们可以把双峰 IT 和互联结合起来；我不知道我们会叫它什么，但是三模态。无论如何，罗莎琳德在我们跳到这里之前的离线谈话中，我想我们都同意我们不是双峰 IT 这个术语的忠实粉丝，所有这些似乎都需要现在。你想和我们的观众分享为什么你可能不是一个粉丝吗？

罗莎琳德·雷德克里夫:是的。我认为这个术语和它的使用方式并不意味着记录系统有一个确定的慢度。我认为重要的是，不要说双峰，而要说更多的变速 it 或暗示系统可能以不同速度移动的东西。我不会不同意这一点，但这并不是说一个天生更快，一个天生更慢。希望你的记录系统已经构建成一套 API 或服务，这样你就不必经常改变服务，因为你需要改变参与系统来吸引用户保持联系。

因此，它真的应该是关于记录系统和参与系统如何通过松散耦合和适当的服务一起工作。我确实认为这将会发生。不同的时间会有不同的速度。有时候，老实说，记录系统可能比参与系统发展得更快，因为需要改变——法规改变等。但是，我们需要从这个角度关注它，这就是我不喜欢双模 it 的原因。真的是变速。

艾伦·希梅尔:是的。让我着迷的另一点是双模 it 的整个概念，它几乎需要我们有两条以不同速度移动的轨道，或者一条轨道上以非常不同的速度移动的两列火车。但他们从不互动；这几乎——这几乎让我想起了民权，独立但平等。我们有供大型机人员使用的饮水机，也有供云人员使用的饮水机，它们都提供水，所以没关系。但是主机的人不能喝云伙计的饮水机。

它没有为民权服务，而且——我认为它也没有为民权服务。我认为，当我们谈论记录系统、参与系统以及整个世界时，一个更好的模型是——我称之为瑞士手表模型。齿轮以不同的速度转动，但它们都啮合在一起，形成一个微调的计时器。对我来说，如果你愿意的话，这是一个比分离轨道更好的模型。你觉得那个罗莎琳德怎么样？

罗莎琳德·拉德克利夫:实际上，我认为这是一种很好的表达方式。因为我认为——齿轮和瑞士表是更好的解释，而瑞士表的计时系统总是很好；原来如此。团队需要一起工作，他们需要使用相同的工具，相同的实践，相同的过程，并且每个系统需要以他们需要做出的改变所需的速度运行。但他们显然在互动。没有你的记录系统，你的参与系统什么都不是。因此，他们需要恰当地沟通，需要共同努力为最终用户提供这种端到端的解决方案。

所以他们必须合作，老实说，我认为那些将团队聚集在一起的组织，那些将参与系统和记录系统整合到一个团队中的组织，实际上做得最好。它们是最有效率的。所以它们在不同部位的传递速度可能不同。但是他们都在一起工作，他们都在合作，他们都是一个团队；不是这种分离的想法。我觉得你举的例子很好，不是分开，而是平等，是一个团队。

*艾伦·希梅尔:*我完全同意。我并不是在这里指责 Gartner 推动了整个双峰 IT 名称和模式。但是，罗莎琳德，我记得在去年的互联大会上，我与你、Sanjeev Sharma 和 Carmen Diardo 进行了一系列对话，我们使用了“变速 IT”这个术语。对吗？

罗莎琳德·拉德克利夫:是的，绝对的。

*Alan Shimel:* 谈论它——类似的事情，当然，我——我们听到其他人用两个速度这个词。但是我认为在更大的组织中，实际上有两种以上的速度；有很多不同的部分以不同的速度运动。

罗莎琳德·拉德克利夫:这确实是正确答案。我真的认为，能够以业务所需的速度前进是非常重要的，这也是业务需要的原因。我认为有很多关于双峰 IT 的思考过程，因为仍然有或似乎有一些人仍然认为大型机不能快速移动，它不能使用 DevOps 实践。我认为这实际上是问题的核心。我们必须打破这种思维过程，记录系统——大型机无法快速移动，无法开发。确实可以，而且这是一个现实——我想从某种意义上说，这是一个热点问题。过去两周，我一直与两个不同的客户在一起，本周与其中一个客户在一起——他们提出的第一个原则是，大型机并不意味着它不应该是 DevOps。第一个原则是它需要开发运维，它需要利用相同的管道、相同的功能和相同的能力，以便能够以可靠、一致的方式根据需要快速移动，同时具备自动化测试和自动化的所有优势。

艾伦·希梅尔:当然。Rosalind，让我们稍微谈一谈今年的互连，无论您是想称之为双模 it 还是可变 IT。在 200 或 300 首歌曲中，我忘了有多少首与 DevOps 相关的歌曲，将会有一些很棒的歌曲，我想和大家谈一点这方面的内容。

罗莎琳德·雷德克里夫:是的，我也很期待这场比赛，尽管我还没有准备好。将会有很多讨论和会议，客户和 IBM 人都在谈论变速 IT 或应用程序开发的不同部分需要交互的不同方式，并且需要一起工作。我本周与之交谈的一个客户将在那里谈论他们的 DevOps 开发，特别是他们如何使用 DevOps 原则在 COBOL CICS 应用程序上构建全新的应用程序。因此，我期待着在那里举行的所有不同客户的讨论，将他们的团队聚集在一起，我认为你的瑞士齿轮解释是一个很好的方式。因为这是他们将要谈论的，他们如何让团队一起工作，以提供公司需要的价值。

艾伦·希梅尔:当然。罗莎琳德，你是否被允许——我们能向我们的观众提及正在展示的公司吗？在这一点上你可以说还是不可以？

*罗莎琳德·拉德克利夫:*因此，在这种情况下，它是荷兰合作银行，因为他们是作为会议发言人发表的；所以他们会说话。其实也不少。荷兰合作银行的一家，瑞银是另一家；我们安排了许多演讲者——客户演讲者，他们将讲述他们的努力和转变。

艾伦·希梅尔:罗莎琳德，你呢？我确定你在说话。

*Rosalind Radcliffe:* 是的，我有好几个会议要讨论 DevOps 的整体情况，与客户及其转型合作，可用的功能，以及更多关于 DevOps 运营方面的讨论。我认为一个有趣的挑战是，既然我们真的有运营组织，我们如何真正将运营纳入开发运维中。这听起来真的很奇怪，但是当我们想到 DevOps 时，很多讨论都集中在自动化和交付上，而不是集中在运行系统上。因此，我试图帮助一些讨论带来更多的运营方面。我们如何真正实现运营协作？从完整的端到端的角度来看，我们如何将团队联系在一起？

*艾伦·希梅尔:*优秀。罗莎琳德，很难在众多作品中做出选择，但除了你刚才提到的那些，你还能给我们的观众什么呢？如果他们想知道我该不该去，不确定；你认为人们应该知道的两三个最大的学习环节或学习要点是什么？

Rosalind Radcliffe: 老实说，我认为要命名的会话太多了，但可以这样想，要命名的会话太多了。有很多客户会议，我们将讨论他们的经历。因此，像这样的会议的价值，其中一个关键的价值是能够听到你的同行，并与你的同行交流，以了解他们是如何做到这一点的，他们是如何获得价值的。这是向已经有经验或正在经历的人学习的最好方法。你将向向你走来的人学习，你将帮助在你后面的人。所以这是网络，这是机会——听听这些会议，有机会认识在这个领域工作的所有不同的人。

我认为这就是像这样的会议的价值；建立关系网并加以利用的机会。我在 prior InterConnect 中遇到的许多人都说，是协作、是对话、是人们的会面给了他们最大的价值。参加会议也很重要，但正是会议给了他们接触的机会。所以他们在会议中学习，然后他们有这种接触，允许他们继续讨论。你总能得到乐趣——学习新的东西，听一些有趣的故事，得到一些你真正能够带回去并为你的组织提供价值的东西。

艾伦·希梅尔:当然。罗莎琳德，信不信由你，我们快没时间了；我们可能已经检查过了。像往常一样，每当我们谈话时发生这种情况，我都会道歉。但是，我很期待在 InterConnect 上看到您的一些会议，我应该提到，这是一个快速插件，我自己主持了一个关于 DevOps 在未来三到五年内将会是什么样子的会议，我的小组成员不是别人，正是 Gene Kim，这应该会很有趣。因此，如果有人正在收听互联，请下来参加。

但是，罗莎琳德，这应该是一场精彩的演出，我期待着整个双峰技术，变速技术，以及它的发展。我们以前说过，大型机不会去任何地方，像 DevOps 和敏捷等新的做事方式将影响我们使用参与系统的方式，但不会否定它们。与您交谈总是令人愉快的，期待 2 月 21 日至 25 日在拉斯维加斯与您见面，对吗？

罗莎琳德·拉德克利夫:正确。我也很期待，正如你提到的小组讨论，那总是很棒的。我希望我能赶上你和金·金的会议，如果我自己没有其他会议的话。所以我很期待，这将是很棒的一周。
