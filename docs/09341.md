# 也更新那些操作工具

> 原文：<https://devops.com/update-those-ops-tools-too/>

我们都非常认同这样一种观点，即随着时间的推移，IT 的变化率一直在增加。从虚拟机到敏捷再到开发运维再到[容器](https://containerjournal.com/?s=containers)的出现，发展速度越来越快——内部开发也是如此。

我将从三个角度而不是通常的两个角度来探讨这个话题。有点像打破电影中的第四堵墙。我们将从我最喜欢的两个角度来讨论这一运动，这两个角度分别是*开发*运营和开发*运营*。但是在这篇博客中，我们还将提到 devo PS——支持其他两者的基础设施、工具和流程。

对于 *Dev* Ops 来说，语言——既包括对当前使用的语言的更新，也包括新语言的添加——带来了对所用工具的近乎自动的更新。最喜欢的库/模块、ide、构建工具——据我估计，这些在今天比历史上任何时候都要被关注和更新得更频繁。对于不经常更改语言或库的团队来说，源代码分析(SCA)扫描越来越有助于仔细查看所有包含的工具，并根据影响(通常是安全影响，但有时也有其他影响——如项目活动和稳定性)过滤它们。

对于 DevOps 来说，工具链仍然是足够新的，以至于进化正在引起常规的变化，而这种变化带来了新的工具和实用程序来解决现有基础设施的问题。事实上，我个人一直认为 DevOps 部分在变化和新工具/实用程序方面需要一点冷却。

最后，Dev *Ops* 在这方面有点偏离。每个负责对机器和应用程序进行故障排除的人(无论您称他们为运营、管理还是开发人员)都有一套工具。但是平台的分歧造成了工具的分歧。没有一套专用的工具也可以工作——事实上，许多 DevOps 工程师就是这样做的。其他人有一个文件夹。与此同时，还有另一组专门的 DevOps 工程师，他们总是带着一个装满有用工具的 PCMCIA 或 USB。有些两者兼而有之。当然，所有这些都包括哪些工具也千差万别。在我们的 It 发展中，一个有趣的现象是，开发人员倾向于在整个组织中共享或借用最好的库/模块/API，但是开发*运营*更加独立。只要开发*运营*团队成员在完成工作，就没人会担心他们用什么工具来进行故障诊断。如果被问及，他们确实倾向于分享，但由于他们使用了什么工具的实物证据很少，这种情况并不经常出现。

那么，我在说什么？寻找持续(或至少定期)支持开发*运营*工具共享的方法。一个内部社交媒体渠道，专门用于你的团队交流的供应商，棕色袋子，一个维护的评级系统“为什么我认为 fdisk 摇滚”之类的东西，等等。每个发展*运营*成员的指尖掌握的权力越多越好。鉴于我们的历史，建立一个“批准的”名单可能是一个坏主意，但有一个中央通信位置，安全也可以仔细检查寻找危险信号不是一个坏主意。当然，他们可能太忙了，但最糟糕的情况是，这样的谈话可以帮助以后的法医分析。不是愤世嫉俗，只是生活在 21 世纪的 T4。

继续踢它。更多更好的[工具](https://devops.com/?s=tools)将有助于保持所有这些可爱的应用程序和服务器，所以这里真的只有好处。为您的团队提供工具，尽情享受您可观的正常运行时间。
