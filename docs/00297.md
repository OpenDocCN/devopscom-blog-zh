# 持续交付管道——它是什么，为什么它在软件开发中如此重要

> 原文：<https://devops.com/continuous-delivery-pipeline/>

连续交付(CD)是一种软件策略，它使组织能够尽可能快速有效地向用户交付新功能。CD 的核心思想是创建一个可重复的、可靠的、增量改进的过程，将软件从概念带到客户手中。连续交付的目标是通过自动化的软件生产线，使变更持续流入生产。持续的交付渠道是实现这一切的关键。

管道将软件交付过程分成几个阶段。每个阶段都旨在从不同的角度验证新功能的质量，以验证新功能并防止错误影响用户。管道应该向团队提供反馈，并向参与交付新功能的每个人提供变更流程的可见性。

没有所谓的标准管道，但一个典型的 CD 管道将包括以下阶段:构建自动化和持续集成；测试自动化；和部署自动化。

*   *构建自动化和持续集成*

管道从构建二进制文件开始，以创建将被传递到后续阶段的可交付内容。由开发人员实现的新特性被持续地集成到中央代码库中，进行构建和单元测试。这是最直接的反馈周期，告知开发团队他们的应用程序代码的健康状况。

*   *测试自动化*

在整个阶段，应用程序的新版本都经过严格的测试，以确保它满足所有期望的系统质量。重要的是，所有相关方面——无论是功能性、安全性、性能还是合规性——都要经过管道验证。该阶段可能涉及不同类型的自动化或(至少最初)手工活动。

*   *部署自动化*

每次在测试环境中安装应用程序时都需要部署，但是部署自动化最关键的时刻是部署时间。因为前面的阶段已经验证了系统的整体质量，所以这是一个低风险的步骤。部署可以分阶段进行，新版本最初发布到生产环境的一个子集，并在完全推出之前进行监控。部署是自动化的，如果需要，可以在几分钟内将新功能可靠地交付给用户。

**您的管道需要平台供应和配置管理**

平台供应和系统配置管理支持部署管道，允许团队自动或按下按钮创建、维护和拆除完整的环境。

自动化平台配置可确保您的候选应用部署到正确配置且可重现的环境中，并针对这些环境进行测试。它还促进了水平可伸缩性，并允许企业随时在沙盒环境中试验新产品。

**协调一切:发布和管道协调**

部署管道中的多个阶段涉及不同的团队，他们协作并监督应用程序新版本的发布。发布和管道编排提供了整个管道的顶层视图，允许您定义和控制阶段，并深入了解整个软件交付流程。

通过在您的版本上执行价值流映射，您可以突出任何剩余的低效和热点，并查明改进您的管道的机会。

不要添加新的功能，直到你得到正确的质量！

持续交付就是让您的组织能够快速、可靠地将新特性一个接一个地引入生产。这意味着每个单独的特性都需要在展示之前进行测试，以确保该特性满足整个系统的质量要求。

在传统的环境中，开发团队通常试图一次实现一个全新的版本，只有在项目接近完成时才处理软件质量属性(如健壮性、可扩展性、可维护性)。然而，随着截止日期的临近和预算压力的增加，质量往往是第一个受到损害的。

糟糕的系统质量、低用户满意度和无休止的“质量创可贴”可以通过采用在获得正确的质量之前不添加新功能的原则来避免。您应该总是首先满足并保持您的质量水平，然后才考虑逐渐向系统添加功能。

对于 CD，每个新特性都需要从一开始就满足系统整体的质量要求。只有达到这一质量水平，该特性才能进入生产阶段。

**开始连续交付**

显然，组织不能也不应该匆忙地在他们所有的业务单元中一次采用连续交付。最好的方法是专注于改善你最大的交付瓶颈。CD 会自动显示下一个瓶颈是什么。这让你走上了一条可衡量的改进之路。

使用连续交付的主要目标是推出比以前的迭代更好的新特性和功能——在整个组织中逐渐合并和精炼 CD 原则。慢慢地走，平稳地走——观察改进！
