# 开源可观察性将在 2022 年成熟

> 原文：<https://devops.com/open-source-observability-comes-of-age-in-2022/>

可观察性对于实现预想的业务目标越来越重要。尽管有性能数据的好处，组织在实现基于实现的可观测性的全部潜力方面面临挑战，分布式软件工具和其他技术障碍。但是随着新的创新，真正可观察的时代即将到来。

## 创新变革

特别是开源可观察性领域，正在经历一个创新阶段。由分布式跟踪“时代的到来”和 [OpenTelemetry](https://opentelemetry.io/) 以及 eBPF 作为下一个前沿 [的可观察性和 Linux](https://ebpf.io/) 上的监控在重组方面占据中心位置所推动，创新达到了前所未有的高度。

## 核心技术

extended Berkeley Packet Filter(eBPF)提供了一种 Linux 内核特有的迷人技术。eBPF 的初衷是以一种可编程的方式处理网络，现在它已经发展成为一种相当通用和高性能的工具，可以安全、动态地访问 Linux 内核，除了其他功能之外，它还可以提取遥测数据以实现可观察性和安全性。eBPF 很强大，并且越来越受欢迎，微软一直在开发 WindowsT3 的移植。

## 多才多艺

就可观测性而言，eBPF 在 2021 年被证明是非常通用的。例如，有一些工具使用 eBPF 来执行分布式跟踪，如 New Relic 在 2021 年年中开源的 [Pixie 项目](https://docs.newrelic.com/whats-new/2021/05/pixie/) 。但是 2022 年 eBPF 最有希望的方面是，有了 eBPF，在生产中实现应用程序的连续分析是可能的。这方面的例子有前面提到的 Pixie 项目和 [Parca](https://www.parca.dev/) 。正如 Google 的 [Cloud Profiler](https://cloud.google.com/profiler) 和各种商业观察工具所证明的那样，在生产环境中拥有一个低开销、始终在线的分析器是解决生产环境中延迟和内存问题的一种强大方法，众所周知，在测试实验室中重现这些问题非常困难和耗时。

## 消除差异

虽然现在使用 eBPF 进行产品概要分析，但它还不是大多数应用程序的通用功能。在不同的运行时和编程语言中，概要分析的工作方式惊人地不同。虽然几乎所有语言在使用 CPU 和分配内存方面的行为都是相似的(即使在这样的基本计算问题上，也有差异)，但进程并发的方式却明显不同，几乎是全面的。

例如，Java 正在使用 Java 虚拟机内部的线程，这些线程映射到操作系统中的线程(公平地说，也有其他选项，如 NIO 和像 RxJava、Reactor、Vert.x 这样的库，加上 [Project Loom](https://cr.openjdk.java.net/~rpressler/loom/Loom-Proposal.html) 最终将会发布)。另一方面，在 Node.js 中，并发性主要由 [事件循环](https://blog.logrocket.com/a-complete-guide-to-threads-in-node-js-4fa3898fe74f/) 处理，操作系统中的“真正的”线程从开发人员那里抽象出来。像这样的差异对于探查器来说很重要，因为向排除问题的人显示的数据必须“翻译”成编程语言使用的并发模型，这需要探查器对特定运行时的显式支持。

## 开源可观察性将在 2022 年成熟

幸运的是，2022 年将为我们带来功能全面、开源、基于 eBPF 的生产分析工具，这些工具可以理解当今云原生应用程序中流行的许多运行时的细节，如 Java、Node.js、Python 和。NET，以及像 Go 和 Rust 这样的编译语言。对于所有开发人员、SRE 和运营商来说，这将是改进软件和更快解决问题的游戏规则改变者。

当我们展望未来的一年时，像 eBPF 这样的革命性技术意味着操作系统层面的创新速度正在加快，并为真正可观察的一年奠定了基础。
