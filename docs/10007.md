# 我们正在做更多更少的开发

> 原文:[https://devops.com/were-doing-more-less-development/](https://devops.com/were-doing-more-less-development/)

我的根在[软件开发](https://devops.com/?s=software+development)。我写过税务软件、 [SmartGrid](https://www.smartgrid.gov/files/documents/Field_Meter_Programming_d_Meter_Firmware_Upgrades_V3.pdf) 固件和服务器、许多开发和购买的服务器的企业集成代码、地图软件、手机固件——不胜枚举。关键是发展是我的天性。当我看一些影响软件开发的东西时，我从一个专业人员的角度看，就像从一个分析师的角度看一样。在软件开发领域，有很多变化正在发生*。*

很长一段时间以来，都有减少我们开发的软件数量的选择。这些选择通常很受欢迎。曾经有一段时间，企业 IT 中最常见的短语是“开发核心系统，购买支持系统”，这个信息已经深入人心，我们不需要再说了。曾经有一段时间，企业拼凑出像电子邮件一样重要的系统——一些开发自己的系统，但大多数努力集成可用的产品。我们已经完全颠覆了这个等式，现在几乎没有人会考虑在电子邮件上花费大量的 IT 资源。不是发展，不是融合。当然，我们不得不面对一些棘手的问题——我们的行业可以被定义为我们还没有解决的尖锐问题的集合——但是很少有人，如果阅读这篇博客的人曾经担心过这些问题的话。所以，第一个变化是:购买 IT 解决方案极大地减少了我们的时间。从 CRM 到电子邮件再到安全，我们正在使用更多由他人编写的工具。

代码生成工具也一直存在。虽然历史上最著名的都是以 UI 为中心的，但是 4gl 通常试图用更少的代码来解决开发问题。同样，时间、失败和迭代给了我们有用的代码缩减技术。从我们只是插入变量的预格式化‘if’语句，到通过(有时是人工智能驱动的)初始程序生成引擎生成 API，这些引擎为我们提供启动项目的代码。对每个项目来说，编写代码的要求更少。

最重要的是:开源。整个软件市场不值得开发解决方案，因为市场上有很好的开源解决方案。如果它很受欢迎，但即使是供应商也无力提供解决方案，因为每个人都在使用 OSS 解决方案——那么，这就减少了开发的代码量。有几家公司的整个商业模式仅仅是为企业提供 OSS 服务和支持。不需要开发，即使是“供应商”如今，我们开发的任何给定应用程序的很大一部分都是基于开源的，我们加入开源是为了解决一部分问题。基于这种模式，我们遇到过一些令人惊叹的黑客攻击和失败，但不可否认的是，它为更多的解决方案创造了条件。

随着所有这些(可能还有更多——比如 API 优先和 API 标准化)减少了我们正在开发的代码量，你会认为开发人员不会那么忙了，对吗？

DevOps 和 Agile 简化了我们的开发，缩短了给定项目子集的交付时间，但是没有——也没有被设计成——减少软件开发的数量。事实上，恰恰相反。通过更快的迭代，可以尝试更多。

但是，认为开发工作较少是错误的。这有几个原因，但我将把重点放在一个大的原因上。投资于公司的人——商业领袖等——总是有一种巨大的被压抑的愿望，即改进软件以提高效率、竞争力等。这是瓶颈。Agile/DevOps 并不完美，但是他们为被压抑的需求提供了一个压力释放阀。我们已经做了足够长的时间，知道响应这些需求的快速迭代激发更多的新想法，创造另一个迭代…所以它不太可能停止。事实上，我们都知道什么是技术债务，但是直觉地知道更多的代码意味着更多的维护开销形式的债务，这两者之间似乎是脱节的。因此，开发人员正在做更多的工作，因为组织有需要实现的好想法，即使我们有大量的工具来减少开发工作。一些组织已经减少了开发人员，但不是很多。这可能会放缓，但不太可能。在即将到来的高度竞争的环境中，人们需要更多的功能和解决方案，而不是更少。

你会在那里，继续摇滚。注意可以进一步减少代码工作量的方法。这是一个长期的过程，但目标应该是降低工作人员时间的增长率。它不太可能整体下降——因为维护费用加上新项目是一个不断增长的数字——但增长放缓有助于开发商和在职经理保持理智，并花时间找到好的雇员。

继续多做少开发。
