# 约束下的非传统

> 原文:[https://devops.com/unconventionality-within-constraints/](https://devops.com/unconventionality-within-constraints/)

依赖性解决、情境意识和超级英雄有什么共同点？见见 Chris Corriere，Autotrader 的一名开发人员/软件工程师，他讲述了最大限度利用上述所有方法的创造性方法。Sonatype 的社区倡导者和资深讲故事者 Mark Miller 采访了 Chris，了解他的团队正在做什么。

![Chris Corriere](../Images/23acdc1325103b5e87d771b1b51ea0ac.png)

我是[克里斯·科里耶](https://www.twitter.com/ccorriere)，我是 [AutoTrader](http://www.autotrader.com/) 的开发工程师。

Miller: 你能给我们介绍一下你是如何使用 Nexus 的吗？

Corriere: 我们使用 Nexus 进行依赖解析。其中一部分是将我们的企业基础设施与一般的第三方依赖隔离开来。我们能够通过我们的内部回购进行过滤，并确保我们的开发人员使用预先批准的东西，我们可以跟踪和更新这些东西。我们还使用它来转移内部项目，这些项目在内部应用程序之间具有交叉依赖性。

**Miller:** 您如何实施预先批准的依赖关系？

**Corriere:** 他们通常围绕谁有权进入 [Nexus](http://www.sonatype.org/nexus/) 以及谁被允许改进和将东西拉进 Maven Central。但是有办法解决这个问题。有很多方法可以把东西偷偷放进构建中。如果一个开发者真的想得到一些东西，有时当你排除故障时，你会发现一些聪明的解决方案。在这两者之间肯定有一个取得平衡的元素，即确保他们能够进行实验并获得完成工作所需的工具，但不要因为引入不可信的东西而在过程中暴露任何责任或风险。

**Miller:** 如果有人没有使用二进制库，你会告诉他们什么才能让他们使用呢？

构建环境中的一致性是吸引我的主要因素之一，那就是代码的可重用性。我认为这又回到了情景意识上——取决于他们如何衡量他们的组织，他们在改进方面寻求什么，以及哪里有优化的空间。在构建中可能会有很多中断，很多不同的排列。我要说的是，在不同的解决方案中，它可以更加标准化，在团队之间更加一致。没有它你也能过。但是你可能要为同样的最终结果做更多的工作。没有回购，会有更多的戏剧，更多的心痛。

“如果没有回购，会有更多的戏剧，更多的心痛。”

米勒:我认为开发人员越多，你就越需要它。

Corriere: 我认为我见过的最有创意的解决方案是团队通过使用源代码控制和特定版本的构建获得相当一致的结果。但是在这种情况下，你必须有成功的构建。你必须完成构建，即使它没有崩溃也需要时间。您的存储库正在将该步骤之后的所有内容放在准备好的地方。你知道它是建的。你知道它是安全的。它没有进入重启程序。它没有通过过滤器。

米勒:你们用 Nexus 做过什么很酷的事情吗？

科里奥利:我不会说太出格的话。我们如何利用它有一些重叠，看起来更像配置管理，这是非传统的。根据项目的不同，你可以检查 Nexus 的一些东西，以及它们将如何站起来[对抗]你在工作申请之外的许多环境依赖。它可以接触到你运行的 Java 版本，不同的英寸柄，以及其他相互嵌套的东西。这让你开始启动我们的许多标准内部应用程序。再一次，回到情景意识，考虑到我们正在工作的团队的一些限制，这是我遇到的一些事情，起初感觉有点左撇子。但是当你停下来看它时，他们能够在给定的限制内把球移动到球场上。在一天结束时，它的工作。

Miller: 你使用了持续集成、持续交付吗？

科里埃:是的。我不认为这是一个布尔值。在生命周期中，我们有很多 it 终结，并且我们正在运行持续的构建。我们有一些内部工具，我们也运行 Jenkins。总是有加速、改进和优化它的空间。我们正转向更持续的交付。我们真的很想开始让产品更接近 prod。但你总是从左边开始，然后从那里离开，并尝试从风险较低的第一步开始讨论质量。在过去的几年里，我看到 AutoTrader 在这方面取得了很大的进步。

“总有加速、改进和优化它的空间。”

米勒:最后一个问题，如果你想成为超级英雄，你会选择开发、运营还是安全领域？

**Corriere:** 我会选择安全，因为我认为如果你做得对，会涉及到一些开发和运营。我觉得你不应该是蓝队或者红队。我认为紫队是最棒的。你真的需要从两方面着手才能完成今天的工作。
