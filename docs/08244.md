# 为什么你应该拥抱事件，抛弃 MTTR

> 原文:[https://devo PS . com/why-you-should-embrace-incidents-and-ditch-MTTR/](https://devops.com/why-you-should-embrace-incidents-and-ditch-mttr/)

老生常谈的是，IT 中的每个人都讨厌事故，当收集事故响应指标时，自然的反应是寻找您可以随着时间的推移*降低*的数字。我们认为，更少的事故和更短的事故响应时间*肯定*会更好。您可能已经熟悉与这些目标相关的通用指标，包括平均解决时间(MTTR)和总事故数。

然而，事实证明，这些指标实际上与公司的弹性目标背道而驰。要了解原因，让我们来看看诞生了大多数现代事故管理实践并最了解如何正确应用它们的行业:航空业。

## 航空公司能教给我们什么

从 20 世纪 50 年代到 20 世纪 90 年代，航空业建立并完善了事故管理的最佳实践，此后我们对这些实践进行了简化，并在整个技术行业广泛采用。通常所说的“现代事件管理”只是指处理意外灾难的可预测和可重复的方法，以及在构建 DevOps 或 [SRE](https://devops.com/?s=SRE) 实践时所珍视的学习驱动的补救步骤。

这项研究得出的最重要的发现之一是，有些令人惊讶的事实是，事故没有被足够经常地宣布。相反，只有在发生灾难性事件后，才会创建事件报告，并遵循精心组织的响应和编目流程，这意味着错过了在不太严重的事件中发现潜在问题的早期机会。

那么，为什么一个风险如此之高的行业花了这么长时间才意识到更多的事故实际上是一件好事呢？你猜对了:减少事故数量的错误愿望。在追求“降低数量”这一目标的过程中，飞行员和机械师通常不愿意对非最严重的情况提交事故报告，这意味着强大的、以学习为中心的响应和补救流程仅在响应相对罕见和严重的情况时才会触发。

这是古德哈特定律发挥作用的典型例子。换句话说，Charles Goodhart 假定任何创建的度量标准最终都将成为目标。换句话说，如果激励措施是为了减少事故，无论你的反应和学习过程有多好，都只会在最可怕和不可避免的情况下执行。发现这一点后，行业迅速将注意力转向扩大事故流程的使用，努力扩大可归类为事故的情况范围，目标是更频繁地触发正式的事故响应流程，为更多相对较低严重性的事故传播经验，是的，最终重新设定目标以增加事故数量。

这种方法有效地将古德哈特定律转化为航空公司的优势——任何试图利用度量标准的努力都意味着更多的学习，而不是更少。备案事件变得直接和严重激励，这些指标从来没有被用作惩罚行动的理由。这导致了商业航空史上最安全的十年，超过 120 亿乘客出行，没有致命事故。

## IT 可以从成功的事件响应中学到什么

作为技术公司，我们可以从这一实践中学到很多东西，以实现弹性，而不是追求“更低的数字”主要的经验是:通过鼓励对更多非关键事件(可能是 SEV2s 及以下)使用事件管理流程来增加您的事件数量，并将您的指标和激励集中在该目标上，这可能是减少灾难性故障总数的最佳方式。

对于大多数公司来说，这意味着将 MTTR 放在次要位置，而是专注于使事故过程本身在整个公司和整个组织中更易访问。这意味着领导者应该公开谈论有影响的和最近的尸检，并且尸检报告应该是优先考虑的；它也应该是可以衡量的东西。最重要的是，这意味着更广泛地开放事件管理流程。理想情况下，您公司的每个人都应该能够访问允许他们归档事故、读取正在发生的事故以及查看以前事故的事后分析的系统。

我喜欢引用工厂车间的“大红色按钮”哲学，在这里，宣布事故的决策权掌握在每个员工手中。一个不安全的情况不应该在采取行动之前进行讨论、辩论或批准，即使后来确定情况没有最初想象的那么严重。这一事件被归档的背景——那个红色大按钮被按下的原因——可能会防止未来更大、更具灾难性的事故发生。

因此，如果您刚刚开始在您的公司建立事件管理实践，或者您一直在追求 MTTR 指标，并专注于将降低事件计数作为首要目标，请考虑将这些经验铭记在心，并创建一种积极的事件文化，采用积极的指标来鼓励和奖励事件响应者和报告者。你公司的未来取决于此。
