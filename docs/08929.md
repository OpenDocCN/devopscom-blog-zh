# 克服自动化开发团队面临的挑战

> 原文：<https://devops.com/overcoming-challenges-to-automating-devsecops/>

在过去的几年中，DevSecOps 已被希望主动提高安全性的组织广泛采用。传统上，开发团队会不断地在企业中实现和部署新的应用程序，安全性是最后的附加部分。

然而，随着网络犯罪造成的威胁增加，组织已经将其安全性“左移”,并将其融入开发生命周期的开始。

这提供了显著增加的安全优势，并意味着软件漏洞可以在暴露给企业之前得到识别和补救。一个缺点是这需要额外的时间。因此，许多组织也在寻求自动化他们的 [DevSecOps](https://devops.com/?s=DevSecOps) 流程。

自动化是 DevSecOps 的一项重要功能，因为它可以提高效率、削减成本并简化操作。然而，为了最大限度地利用 DevSecOps 自动化，从而为组织交付真正的价值，有几个挑战必须首先考虑。 但是这些挑战是什么，如何克服它们？

## **如何实现 DevOps 安全自动化**

当您第一次转向安全自动化时，您会发现有些安全任务很容易实现自动化，而有些则很难适应。DevSecOps 团队将需要使用许多工具来覆盖所有的基础，但是没有人希望团队检查许多不同的工具输出。通过将安全工具和结果整合到一个中央平台中，可以更容易地了解正在发生的事情和风险所在。这使得生活变得更加轻松，因为它提供了一个单一的窗口。大家普遍认同 DevOps 团队必须采用网络安全最佳实践，但是测试需要时间。开发团队没有时间在他们现有的工作量上学习、使用和检查大量的安全工具。自动化解决方案无需任何维护或管理即可投入使用。这些解决方案根据发现的结果触发工具在正确的时间工作，无需任何手动输入。这给了你和你的团队宝贵的时间去关注其他重要的事情。

### 将 DevSecOps 安装到安全的 CI/CD 管道中

CI/CD 管道的目标是快速交付构建和发布步骤，通常是通过自动化。DevSecOps 的一个关键目标是在自动化过程中尽可能早地提醒人们注意新的问题。您需要开发不会淹没 CI/CD 管道的解决方案，同时为不同的技术堆栈、安全工具和环境提供灵活性。您需要问的是，最新的更改是否导致了任何新的和重要的安全问题，应该在上线之前报告？

### 在不经常更改 CI/CD 的情况下进行开发备份

每天尝试在您的整个应用程序源代码上手动运行安全工具可能非常耗时，并且会妨碍您跟上日常变化的能力。为了使安全检查与 CI/CD 环境中的代码交付保持同步，安全自动化是必须的。安全产品需要集成到开发管道中，并使开发和安全团队能够一起工作，而不是互相扔东西。

### 关注提出的问题以加快补救

世界上所有的人工智能和机器学习(AI/ML)都不会比人工分诊更好地消除非问题，所以我们在这里建议认命吧。将一组安全工具的第一次运行视为正常的安全测试。预计会有数百个问题，并准备仔细检查它们，标记出误报、重复和现在不重要的问题。但是，这样做的前提是，您正在创建一个基线，可以根据该基线来衡量您未来的管道扫描，以便您只对新的重要问题发出警报。通过自动管理和记录每个 DevSecOps 管道运行中存在的实际问题的列表，我们可以专注于差异。这些新问题需要被标记出来，然后让所有团队都容易看到并采取行动。

### 在 CI/CD 中运行多个安全工具

DevSecOps 包括设置许多自动化安全工具，以涵盖需要检查的许多类型的安全问题。许多组织运行多种安全工具。然而，随着安全工具数量的增长和 DevSecOps 过程变得更加复杂，许多公司意识到最大的挑战是在短时间内理解所有的测试结果。对于一些公司来说，整合所有这些工具的结果是一项全职工作，而且自动化的时机已经成熟。

### 对安全问题进行分类

误报和无意义问题是使用安全工具的常见问题。移除这些会产生更好的结果，并增加开发团队与 DevSecOps 过程建立更好关系的机会。组织应该建立一个不需要采取行动的问题列表，并使用自动化安全工具来标记新问题，并将它们与之前积压的问题区分开来。自动化所有问题的“最佳修复”也加快了修复时间，同时减少了安全团队的工作量。

**将安全性映射到开发流程**

Dev 和 DevOps 团队承受着让管道平稳运行的压力，并且不希望安全性减慢事情的速度或使其复杂化。通过自动化安全测试，使其能够轻松处理 repos 和容器中的更改，组织可以在不损害安全性的情况下克服这一挑战。

**风险优先级**

DevSecOps 将增加发现问题的数量以及处理问题的速度。事实上，只有少数问题会给企业带来巨大风险。能够实时查明真正的大问题有助于安全部门了解正在覆盖的大风险，并允许开发人员专注于真正的问题，而不是大量的小错误。您还可以智能地确定 CI/CD 管道中快速和自动化决策的风险变化，而不会减慢开发团队的速度。
