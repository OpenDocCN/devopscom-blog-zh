# CI/CD 仍然完全是关于开源的

> 原文:[https://devops.com/ci-cd-is-still-all-about-open-source/](https://devops.com/ci-cd-is-still-all-about-open-source/)

在我执掌 DevOps.com 的这些年里，我学到的一个教训是，DevOps 的核心是 CI/CD。CI/CD 的核心是开源。虽然为了保护无辜的人(和过时的人)，这些名字已经被改变了，但这些教训在今天仍然是真实的。CI/CD 仍然都是关于开源的。

当我们刚开始研究 DevOps.com 时，人们说 DevOps 只是一种症状；CI/CD 是治愈的方法。对于你们中的许多人来说，这种解决方案是通过开源交付的——可能是使用 Jenkins。Jenkins，本身就是开源 Hudson 项目的一个分支，由 [川口幸介](https://digitalanarchist.com/videos/2020-devops-experience/kohsuke-kawaguchi-data-driven-devops-the-key-to-improving-speed-scale) 打造。

Jenkins 本质上是一个 CI 工具，后来演变成一个 CI/CD 工具。许多人认为，从长远来看，这个岔路口可能已经伤害了持续交付的持续发展。但这是另一篇 DevOps.com 文章的论点(或者甚至是即将到来的 DevOps 现场活动的小组讨论)。

不管你在这个问题上的立场如何，作为一个[开源项目](https://devops.com/7-popular-open-source-ci-cd-tools/)，很难否认 Jenkins 的成功。Jenkins 插件架构在很大程度上推动了这一成功。有数以千计的插件可以让 Jenkins 处理任何东西。没错，这就是为詹金斯提供动力的引擎。但是它的秘密超能力过去和现在都是开源的。

也就是说，这些年来詹金斯的年纪有点大了。这并不是说它不再像以前那样做了，而是我们所做的和我们做的方式已经改变了。微服务、Kubernetes 甚至云已经改变了詹金斯面前的挂毯的结构。支持 Jenkins 的开源社区在这里应该得到巨大的信任:它已经尽力跟上许多变化。

例如，詹金斯 X 试图将詹金斯带到库伯内特斯 T2。但最终，没有什么是永恒的，正如《圣经》中所说，“万物皆有季节，天下万物皆有定时:生有时，死有时。”现在，我并不是说詹金斯快死了，但我认为作为狼群领袖的时代已经过去了。

好消息是有许多新的 CI/CD 工具已经出现，填补了 Jenkins 留下的空白。其中一些是完全开源的；有些是基于开源的。例如 Buildkite、Harness、Codefresh、GitLab、GitHub、CircleCI、JFrog 和 Tekton 等解决方案。

另一个重大发展是 GitOps 的兴起。Weaveworks 在采用方面一直处于领先地位，更不用说创造了术语 GitOps 本身，但显然，他们并不孤单。然而，这些解决方案的共同点是开源。

我曾经认为当涉及到 CI/CD 和开源时，这是一个先有鸡还是先有蛋的问题。但我现在意识到事实并非如此。我坚信，如果没有开源，CI/CD 就不会有今天的力量。毫无疑问，它*是*真正的力量。CI/CD 已经成为软件开发的主导方法。

当然，你可以说“为什么在 CI/CD 停止呢？开源已经成为可观察性、操作系统、云等领域的主要软件开发基础！”是的，我同意——我们生活在开源时代。事实上，这种上升的趋势可能会抬高所有船只，包括 CI/CD。

这股浪潮正迫使所有厂商要么完全开源，要么至少支持开源。此外，在可预见的未来，我认为这种情况不会改变。开源也给市场带来了其他变化。包括 CI/CD 在内的软件工具是如何分发和消费的。谁还在雇佣企业软件销售人员？做大型 POC？不，开源让软件直接进入开发者和 DevOps 团队的手中，带来了快乐。如果你的产品有效，他们就会购买。如果不起作用，他们也不会——而且他们也会让你知道！

这种精英管理是开源的另一个好处，有助于加速市场上更好的下一代 CI/CD 工具的开发。最终，我们都会因此受益。
