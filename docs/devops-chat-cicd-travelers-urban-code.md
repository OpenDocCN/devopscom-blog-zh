是的。明白了。弗兰克，你来了，你知道你的任务是更新这个。对于那些在外面听着的人来说，天啊，那就是我刚才在的地方。三年前弗兰克在哪里，我就在哪里。你能给他们一点建议吗？你从哪里开始？

**Canihuante:** 当然可以。所以我以前把管道放在一起，但规模较小。所以我最大的担忧是，同时也是让我对这份工作感到非常兴奋的是，我如何在一个大企业中做到这一点。Travelers 有 30，000 名员工，其中 3，000 人从事 IT 工作。所以我知道这直接或间接地影响了成千上万的人。所以随之而来的是很多责任。所以我建议你做好功课。你必须有丰富的开发经验，才能熟悉开发人员的思维方式。因为归根结底，他们是你的顾客。他们要求很高。他们现在有了可以工作的东西，不管它们有多古怪，它都是可以工作的。你将会改变这一切。您必须确保无论您要用什么来替换旧工具，它都是您作为开发人员感到舒适的东西。这就是 DevOps 部分的 Devs 部分的用武之地。另一方面，一旦你把系统组装起来，你必须像运行一个产品一样运行它。这是有时我们在匆忙中得到的东西，人们可能不会想到系统的操作方面。但这是我们需要的东西。

我刚才说的是，你知道把它们组合在一起需要一系列技能，这些技能基本上需要你所有的开发经验。但是你不能忘记你将如何运行这些工具，你将如何操作它们。因此，试图让你的客户满意是另一个挑战。

**Canihuante:** 好的。当然可以。所以我们有一个过程，在这个过程中，我们引入了几个 CI 和 CD 工具供应商。我们真的让 UrbanCode Deploy 按照部署工具的速度进行部署。在我们的 RFP 中，我们确实有一个正式的 RFP 流程，有许多许多问题涉及到它实际上是如何工作的，以及它实际上是如何操作的。因此，在我们将该工具用于实际的概念验证之前，我们已经有了一个很好的想法。但是有了 POC，才真正让人们意识到它能做什么。它与我们更换的旧工具非常不同。实际上，一旦我们让用户参与进来，他们对新系统的焦虑和不安就会降低到正常水平。你知道，一件事是观看演示，另一件事是实际动手，通过这种新工具部署应用程序。

**Canihuante:** 绝对。有许多方法可以衡量一个项目是否成功。这类项目。在我们的案例中，旧系统中有 2，000 多个应用程序。衡量成功的一个方法是说一年后我们在旧系统中还有多少。所以在 2000 多个中，我们有大约 100 个仍然在旧系统中。在接下来的几周或几个月里，它们将会真正迁移到 UCD。但除此之外，还有其他一些事情让它成功了。只是和成百上千的人一起合作建造了一些东西，然后集思广益，把一条管道组合起来，让他们以后的生活更轻松。这是衡量成功的另一种方式。这很难衡量，但你可以从人们的谈话中看出。例如，我们不再谈论我们需要这个工具或那个工具的方式。我们正在谈论解决一个特定问题的两种不同的方法。之前，我们甚至不知道怎么解决。现在我们有多种方法。甚至连对话都不一样。它不是关于一个群体可能有的特殊问题。但是它一直在想怎么才能给大家解决这个问题。我们如何提出解决如何构建和部署应用程序的模式？不仅仅是我的团队或部门，而是整个公司。
