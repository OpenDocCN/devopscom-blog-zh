# GitHub 为 JavaScript 包管理器带来 2FA

> 原文:[https://devo PS . com/github-brings-2fa-to-JavaScript-package-manager/](https://devops.com/github-brings-2fa-to-javascript-package-manager/)

GitHub 已经为由 NPM 公司维护的 JavaScript 应用程序包管理器提供了一个双因素认证工具。

此外，所有 npm 软件包都已重新签名，现在有一个 npm 命令行界面(CLI)命令来审核软件包完整性。

最后，GitHub 增加了将 GitHub 和 Twitter 帐户连接到 npm 的能力，以及使用 npm CLI 的简化登录和发布体验。

GitHub 的开源产品经理 Myles Borins 表示，这些改进是正在进行的努力的一部分，旨在通过更好地保护凭据不被用来接管 GitHub 帐户，使组织能够更好地保护他们的软件供应链。

![](../Images/976d4e310e10cffb80df4a0433d3d92b.png)

Borins 指出，这种类型的[攻击](https://securityboulevard.com/2022/08/github-zero-day-from-35k-repos-compromised-to-false-alarm/)数量并不多，但很明显，网络犯罪分子正试图通过非法访问用于构建应用程序的平台来危及软件供应链的安全。一旦网络犯罪分子获得访问权限，他们就有可能在软件组件中插入恶意软件，这些恶意软件可以找到进入任意数量的下游应用程序的途径。例如，该恶意软件可能会被用于发起勒索攻击或只是泄漏数据。更麻烦的是，恶意软件可能开始在整个应用程序环境中横向传播。

GitHub 目前正在处理一份报告的余波，该报告称其存储库遭到了攻击；那原来是虚惊一场。被攻击的存储库要么是分叉的，要么是真实的、流行的存储库的克隆，其名称类似于 GitHub 平台上的合法存储库。没有恶意代码实际上被合并到任何原始存储库中；然而，很明显，网络罪犯正在使用一种称为欺骗提交元数据的技术，试图欺骗开发人员将代码提交给一个虚假的存储库。

Borins 说，GitHub 试图应对的挑战是，根据风险水平，一些安全控制应该默认启用到什么程度。他指出，举例来说，个人开发人员可能比企业 IT 组织对风险有更高的容忍度，他们构建的应用程序是推动业务发展的核心流程。Borins 补充说，在某些情况下，每当忘记用户名/密码时，组织可能会选择完全限制访问，而其他组织则会简单地限制访问以确保保持生产力。

总的来说，一般的企业 IT 组织可能对中断工作流有更高的容忍度，以确保关键的知识产权始终得到保护，他补充说。

当然，关键问题是没有人确切知道哪一天哪一个开源组件会出现在哪一个应用程序中。不管应用程序的类型是什么，开发人员现在都会重复使用可能包含任意数量的已知漏洞或尚未发现的零日漏洞的代码。Borins 说，为大多数组织构建最基本的安全 [DevSecOps](https://devops.com/?s=DevSecOps) 工作流的基础将从实施双因素身份认证开始。

不管 DevSecOps 方法如何，几乎每个人都认识到有必要更好地保护软件供应链。还有一件事需要确定，那就是实现这个目标实际上会对软件构建和交付的速度产生什么影响。
