# 为 DevOps 构建业务案例

> 原文:[https://devops.com/building-business-case-devops/](https://devops.com/building-business-case-devops/)

你可以随心所欲地快速调整你的 DevOps 汽车，但在你向董事会提供有效的商业案例之前，它将一直留在车库里。要想获得好评，你必须证明你的模式是高效的，对整个公司来说是有财务意义的。

## 您是否需要为 DevOps 制作商业案例？

在 DevOps 之旅的各个不同阶段，公司都需要连贯的业务案例，而不仅仅是在最开始的阶段。扪心自问，以下哪种情景最适合您的现状:

1.  您/您的首席信息官知道公司需要变得更加敏捷，但是您不知道如何量化收益。
2.  您的团队已经开始在本地实施 DevOps 实践，并且可以看到好处，但是您不知道如何量化成功以向管理层推销。
3.  您的组织看到了开发运维试点的成功，但需要证明投资的合理性，以便将开发运维扩展到企业规模。

DevOps 可以代表 IT 部门的一次大改组，必须清楚地从逻辑上证明这一点。预计的财务收益必须超过风险。至关重要的是，这需要用财务部门能够理解的语言进行量化。它被称为最低预期回报率:一个项目为补偿风险所必需的年回报或总回报。

商业案例应该包含什么？

1.  管理层简报
2.  推荐
3.  所需的决策
4.  业务驱动因素
5.  范围
6.  假设
7.  分析
8.  净现值
9.  战略选择

## DevOps 业务案例成功的标志

**敏捷度**

这是 DevOps 的主要好处:能够更快地交付应用程序，并允许在生产过程中频繁更改和采用。但是我们如何度量和量化敏捷性的好处呢？

**货币的时间价值**

金钱的时间价值表达了这样一个事实:今天的钱比明天的钱更值钱。你越是需要早期投资(而不是长期投资)，越是需要更长的时间来获得回报，你投资的现值就越低。

当计算一项投资的现值(收益减去成本)时，你得到的数字称为 NPV(净现值)。它告诉你今天你的未来投资值多少钱。

有了敏捷和 DevOps，你就能够开发一个应用程序，并以小块而不是大块的形式交付给你的最终用户；因此，即使投资和产出总量相同，你也会提高项目的净现值。

持续交付软件而不是每年交付软件会带来好处，就像你的储蓄账户每天而不是一年才收到利息一样。

金钱的确切时间价值取决于公司必须承担的平均资本成本。例如，一家大型上市公司的资本成本低于一家小型私有公司。你的财务部会知道的。

**浪费成本**

第二个方面是减少浪费，它以可量化的方式说明了敏捷性。

使用非敏捷软件工程实践交付的功能一旦到达最终用户手中，可能会因为时间差距和需求与期望的不匹配而变得不相关或过时。使用 DevOps 可以大幅降低此类废物的成本。

在应用程序没有变化的情况下，使用可用的平均功能百分比统计数据来计算减少浪费的好处。对于最近发布的两到三个版本来说，评估这个数字是值得的。

考虑到:

1.  开发和交付这些版本的总成本。
2.  由版本交付的不同功能的相对努力。
3.  在发布日期期间和之后到达的变更请求的成本。
4.  关于这些功能的需求和可用性的最终用户调查。

将减少浪费和频繁交付的好处结合起来，并将它们作为正的未来现金流添加到您的计算中。

您还需要计算开发运维计划的预测成本。考虑:

1.  在构建、测试、部署和配置方面的投资以及相关的运营费用。
2.  IT 基础设施投资。
3.  投资以补充开发和运营团队的技能。
4.  投资于组织变革。
5.  上述所有项目的运营费用。
6.  由于发布速度加快，测试、发布和部署中的非自动化活动产生额外成本。

作为您在业务案例中的战略选择的一部分，平衡您在[自动化](http://automic.com/devops-ara)上的投资与在给定敏捷性场景的构建、测试和部署上的更多资源上的投资。发布速度越高，需要的构建、测试和部署活动就越多。自动化程度越高，手动完成的活动就越少。在图表中形象化地展示随着发布速度的增加，对更多自动化的投资是如何开始得到回报的。

最后，将正现金流和负现金流相加，用你公司的预期回报率来计算你的净现值。
