# 改进您的流程编排，以便更快地向客户提供更多功能

> 原文:[https://devo PS . com/improve-orchestration-deliver-features-customers-faster/](https://devops.com/improve-orchestration-deliver-features-customers-faster/)

当今所有公司面临的巨大商业挑战是如何更快地向客户提供更多的功能。今天的最终用户的期望越来越多地受到网络和移动应用程序体验的影响，这些应用程序自动地或点击一个按钮就能“做更多的事情”。

随着竞争对手能够在几天甚至几小时内发布新功能，公司再也无法承受不可预测、冗长且低效的发布流程，这些流程只能勉强支持每两个月一次的更新。

不足为奇的是，公司正在加速和简化其交付渠道，以保持“领先地位”，并在当今竞争激烈的市场中取得成功。

**现在的情况——流程和人员的低效编排**

大多数公司目前的现实是，将一个完整的特性投入生产所需的所有步骤都是一个耗时费力的过程。

即使交付管道的第一部分，例如部署到测试或 QA，已经是自动化的，到生产的“最后一英里”通常是复杂的手工工作。通常，发布经理会和所有参与发布的团队开长时间的会议。然后，他们起草一份定义发布计划的电子表格或文档，随后根据计划团队的意见对其进行多次修改。

当执行发布的时候，技术团队将使用一整套工具来实际执行工作——不同于那些用于计划和跟踪发布的工具。

由于计划工具和用于执行发布的工具之间通常没有直接的联系，发布经理必须不断地追踪技术团队，以了解事情的进展情况。

“技术轨道”和“管理轨道”之间缺乏集成和共享的洞察力，这确保了企业环境中的发布通常是高度反应性和压力大的事件。发布经理很少或根本没有警告潜在的冲突、延迟或瓶颈，尤其是在发布之间，因此很少有能力提前采取行动以防止意外。

上一层，服务交付经理和业务所有者无法比较和分析整个服务组合中版本的规划和发展，以确定流程中需要改进的区域，从而加速和简化功能交付。

**简化和加速:企业发布路线图**

实现加速的、简化的发布过程的第一步是提供对技术和管理轨道的集成洞察——为整个发布团队提供一个单一的、共享的控制台。这个控制台支持跟踪和执行所有手动和自动活动。

一旦发布经理从追赶和救火的双重杂务中解脱出来，他/她就能够更好地提前识别和预防冲突和瓶颈。此外，整个团队受益于一个完整的审计跟踪，它跟踪版本的发展，可视化偏离原始计划的程度，并快速突出不可预测的应用程序或项目。

您认为有助于推动管道协调的任何技术都应促进流程和数据分析，并提供全面的指标和报告，让您能够跟踪工作的有效性。

**DevOps 和持续交付越来越受欢迎**

在大多数企业中，发布的技术轨道很少能够完美地进行。许多任务，尤其是应用程序部署、环境调配和配置仍然是手动或半脚本化的，因此速度慢且容易出错。这些任务是需要立即改进的。精明的公司通过 DevOps 和/或持续交付(CD)计划寻求改进。

这些计划通常包括引入持续集成和应用程序发布自动化、测试自动化措施，以及虚拟化环境的按需供应和配置。接下来的目标是将这些整合起来，创建一个“端到端”的管道，协调适当的构建、环境供应、部署和测试步骤。

可以使用持续集成工具中的链式作业或构建这样简单的东西来链接这些步骤，或者您可以使用许多专用自动化编排器中的一个。

如果您正在开始一个绿地项目或在一个小型的、高度灵活的组织中工作，端到端自动化一直到生产通常是可能的。然而，在大多数企业环境中，发布仍然需要在多个团队之间协调手动和自动任务的混合，以及以受控的、可审计的方式偏离原始计划的能力。

由于简化的、高度自动化的发布过程仍然是您的最终目标，所以您的编排工具支持完全自动化的 CD 风格的管道和 DevOps 风格的自动化平台是至关重要的。仅专注于管理轨道(规划和分析)或自动化轨道(无法处理手动任务或偏离计划)的技术无法帮助您实现这一目标。

为了开始加速和简化您的发布过程，确保所有的步骤——手动的和自动的——都得到整个团队的同意，并且每个人都从一个单一的、更新的发布计划开始工作。

接下来，确保您选择的技术支持这种协作规划和可见性，并且最重要的是，通过支持手动任务的分配和跟踪以及自动执行，可以统一您的发布的管理和技术跟踪。

使用您的技术的分析能力来识别您的发布过程中不可预测的和缓慢的元素，并启动改进项目来加速和简化这些元素。使用该工具提供的指标来跟踪这些项目的进展和有效性。
