# TidalScale 支持故障服务器的热插拔

> 原文：<https://devops.com/tidalscale-enables-hot-swapping-of-failed-servers/>

TidalScale 更新了其同名软件，用于自动化服务器管理，包括检测和预防即将发生的故障的能力，并使 IT 团队能够热插拔另一台服务器。

TidalScale 的 4.1 版还增加了服务器损耗平衡和预防性健康检查功能，可根据服务器的剩余预计寿命来确定应该更换的服务器。

TidalScale 首席执行官 Gary Smerdon 表示，鉴于 DRAM 内存故障率高，导致服务器故障导致的停机率高，这种能力尤为关键。他说，该公司软件定义的服务器管理软件的最新更新将内部或云计算环境中的故障率降低了 100 倍，以确保 99.999%的正常运行时间。

该公司还宣布，其软件现在可以在亚马逊网络服务(AWS)市场上购买。

TidalScale 采用机器学习算法来聚合和虚拟化跨服务器的 CPU 内核、内存和 I/O，以通过单个软件定义的服务器呈现操作系统。这使得热插拔单个服务器以确保可用性成为可能。

IT 环境中[停机](https://devops.com/?s=server+downtime)的最大原因是内存故障；斯默登指出，现在这种可能性是十年前的 100 倍。他补充说，在大多数情况下，问题的根本原因是在传统基础设施上运行的现代应用程序的需求，而传统基础设施并不打算支持这些应用程序的延迟要求。

Smerdon 说，这个问题在云计算环境中尤为突出，因为在云计算环境中，基础设施通常由多个应用程序共享。

尚不清楚云服务提供商更换传统服务器基础架构的速度，但确定何时更换基础架构的能力为 It 团队提供了对服务器的额外控制措施，无论服务器部署在何处。当然，最大的问题与其说是更换服务器，不如说是确定哪台服务器实际上发生了故障。识别故障服务器所需的时间通常比让新服务器上线所需的时间要长。

TidalScale 由云服务提供商和内部 it 环境中使用的服务器制造商提供，但需要由 IT 团队进行配置和部署。Smerdon 说，总体目标是通过在服务器的整个生命周期内更有效地管理基础设施，不仅提高系统可用性，而且降低 IT 总成本。一些 IT 团队定期更换服务器，不管实际使用情况如何，以避免停机。他指出，TidalScale 使 IT 团队能够根据自我修复服务器群环境中的实际磨损情况来决定更换哪些服务器。

考虑到应用程序环境中可能出现的所有问题，不必担心服务器故障应该能够让 DevOps 团队花更多时间来优化应用程序性能。

当然，理论上，基于微服务的现代应用应该能够在服务器不可用时将调用重新路由到应用编程接口(API)。然而，随着时间的推移，随着应用程序的更新，出现单点故障的情况并不少见。与此同时，今天运行的大多数应用程序都是基于遗留的整体架构，这些架构更容易受到服务器故障的影响，不幸的是，这种情况仍然太常见了。
