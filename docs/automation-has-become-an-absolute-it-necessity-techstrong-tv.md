任何关于云计算的东西，任何关于人工智能的东西，所有这些东西。所以，我们用一种非常不同的方式思考这个问题。我们认为——我们在思科内部做了一系列调查——我们意识到事件的全球性——我们知道一些事件正在发生——所有的地缘政治事件正在发生，所有的疫情，人们真的意识到生命太短暂了。

这不是一个精辟的说法，这实际上是人们认为生命太短暂，他们实际上想做有意义的工作。他们希望产生影响，不仅是对他们所在的团队或公司，甚至是对他们所做的工作产生全球性的影响。

但他们也想成长和学习，并确保自己走上一条有意义地改善自己生活的道路。因此，我认为这些是我们收集的非常重要的特征，作为组织和团队，我们需要利用这种愿望来产生本地影响和全球影响，并找出如何将意义融入每个人的个人生活——你的、我的、我们的团队的，等等。

因此，我们在思科内部做了很多事情，以确保这种意义或有意义的影响得到培养。我们一直在说的一件事是，是的，有一项业务需要运营，但我们也需要为我们的员工和团队提供一条创新的途径，以超越任何公司都需要做的普通业务的影响。

所以，通常我们最终会做的是，我们最终会把我们面对的问题陈述，比如说，作为一个团队或一个组织，我们开始把它分成几个桶。因此，考虑处理一般业务的铲斗，因为这是你绝对需要注意的。

但是，也有你需要创新或做有意义的工作的地方。因此，我们内部有一个投资组合管理系统，至少在我的团队中是这样的，它说让我们拿出 60%的时间，并确保这些时间被分配给常规的业务项目，然后可能有 30%的时间是前瞻性的，添加新功能或寻找新市场等等，可能有 10%的时间实际上是用于成长自己，培养自己，或在全球舞台上产生一些有意义的影响。这就是我们看待这个问题的方式。

迈克尔·维扎德:早期，我认为很多人被自动化吓倒了，有人担心人们可能会失业。现在，我想知道我们是否完全颠倒了这一点，人们会说，嘿，我真的不想为因为你的观点而没有自动化 IT 流程的组织工作，有些事情我不想再花时间去做了。那么，就自动化而言，整个对话是否已经翻转了呢？

我要说的是，在我们说话的时候，对话正在翻转，再次，是的，回到之前的评论，是的，人们不想花时间在可重复的任务上，在计算机能做得比你我更好的事情上。如今，我和我的孩子一直在纠结，既然有语法，为什么我还需要学习语法？但那是另一回事。

但我认为，这是一种类似的心态，贯穿于劳动力中的每一个人，至于我是否真的在工作场所增加了价值，我是否更有生产力，这反过来又转化为组织更有生产力，这反过来又转化为经济和国家更有生产力。

因此，我认为生产力的提高肯定是人们更关心的目标，你会听到另一面，正如你所指出的，工作场所的数字化是所有经济的数字化。正在进行中。

我们用各种短语称之为——软件蚕食世界，或 API 经济——无论你想用什么术语。但是，我们正在走向一个世界，在这个世界中，软件是驱动我们所做的一切的关键。就自动化而言，甚至超越自动化，软件绝对是我们做任何事情的关键。

迈克尔·维扎德:不久前，我们都认为现在我们会开着会飞的汽车到处跑，这很容易。我们两者都没见过，而且看起来它变得比以往任何时候都更加复杂。所以，我的问题是，它是否已经复杂到无法手动处理，我们需要自动化，因为我们看到了所有这些微服务、无服务器框架。

我们现有的所有单块应用程序，现在都分布到了边缘。我们是否已经到达了某种拐点？

*Vijoy Pandey:* 是的，我认为就你的观点而言，现在肯定比十年前复杂得多，我认为问题的关键在于，你构建应用程序的方式以及我们谈到的所有数字化实际上都是以一种非常分散的方式进行的。

因此，作为应用程序开发人员，首先，我们思科喜欢说，应用程序是您的品牌。因此，如果我是一家银行，我的客户如何与我开发的应用程序互动，就将我、我的公司、我的银行定义为品牌。因为如果 A 银行的软件、移动应用比 B 银行有更多的功能，我会向 A 银行迁移。

因此，我认为应用程序正在成为一切客户体验的前端。因此，如果你接受这个概念并付诸实践，这意味着开发这些应用程序的应用程序开发人员面临着极大的压力，他们必须提供功能背后的速度，提供客户数据的可信度，提供正常运行时间和安全性等等。

因此，这意味着他们正在挑选所有这些存在于任何地方的 API，以使他们的工作更快、更容易、更安全、更值得信赖。因此，如果你考虑这种环境，作为一名开发人员，我从云提供商、SaaS 提供商那里挑选 API，我有遗留系统、交易系统、边缘设备，以银行为例。它可能位于分公司。

我需要与所有这些 API 和系统进行交互，以构建一个特定的应用程序或功能。所以，在这种分布式环境中，想想 IT——更多的 IT 人员，对吗？就好像我正在努力构建一个平台来满足这些需求，所以它肯定会变得越来越复杂，越来越分散，分布式环境中的每个提供商都有自己的语义，在他们的 API 和基础架构背后有自己的生命周期管理。

所以，自动化——我的意思是，人类不可能不仅部署，甚至管理和观察这样一个高度分布式的——顺便说一下，也是安全的——这样一个高度分布式的系统。所以，自动化是实现这一切的关键，自动化的一个方向，对我来说，就是可发现性。

它是基于意图的，它是安全的，它不再是安全的边界，它是跨所有这些端点的可观察性，从 API 到大型机。这是前进的关键。

*迈克尔·维扎德:*你认为随着我们的发展，这些组织——至少是我与之交谈过的一些组织——几乎就像是鸡和蛋的问题。他们有太多遗留的东西不能自动化，他们不能投资自动化，他们也不一定明天就能购买所有的新平台。那么，我们如何克服自动化的障碍，从 A 点到达 B 点，而不需要打破 IT 预算呢？

维乔依·潘迪:是的，这是一个很好的观点。我想其中的一些，我在以前的生活中也处理过。要做到这一点的方法是——毫无疑问，在一段时间内，将会出现比之前或之后更多的混乱和更多的资金投入。

因此，我们必须承认，作为供应商、客户和组织，在一段时间内，将会出现混乱。至少我过去就是这样处理的，你说让我们建立里程碑，我们说我们可能会照顾组织的一部分，我们说我们将以高度自动化、分布式、云优先、API 优先的方式来构建它。

应用程序的这一部分、基础架构的这一部分出去做那件事。为此，我们将在这里执行一个全新的流程。因此，我们会考虑——我们不会让自己受限于组织界限、技能组合界限——这很难做到。大多数人的工作，人们的技能，组织结构，所有这些。

但我们必须在一些口袋里尝试，然后给自己严格的指导方针和标准，并努力达到这些标准。如果我们达到了这些标准，那么最终的结果就是我未来的花费会比现在少。

所以，我可能会多花一点钱去找 sre 或软件工程师，把他们和 it 人员或架构师结合起来，然后做点事情。我会给自己一个时间表，比如说一年或 18 个月；它不能比那个长，否则它是一个科学项目。

但是在那 18 个月之后，这些是我将要回来讨论的指标。比方说，我的生产率会提高 X %,员工数量会减少 Y%。我们应该用这个目标来衡量自己，我已经看到了这一点。

如果有效，从简单的事情开始。但是要一路走下去，不要百分之八十。让我们不要做 80%的自动化或 80%的用例。100%的自动化，100%的环境用例，然后随着时间的推移不断扩展。

迈克尔·维扎德:我们当然听说了很多关于人工智能和人工智能操作的事情。机器能拯救我们吗？AI 的现实是怎样的？

*Vijoy Pandey:* 对我来说，人工智能是一种工具，就像我们谈论的所有其他工具一样，我们已经谈论了自动化或作为代码的基础设施，我们已经谈论了云原生、无服务器和微服务。这些工具中的每一个，编程和脚本工具。

因此，所有这些工具都带来了一些东西，基础设施如代码带来了一致性和基于意图的基础设施，无服务器和微服务或云原生带来了不可变的架构。

事实上，您可以扩展、横向扩展并完成所有这些工作，可用性和速度都将突飞猛进。类似地，人工智能和人工智能带来了对表格的洞察力，这是人类不可能做到的，仅仅通过查看大量不同的数据和我们面前的海量数据。

如果我们能把所有这些都拿走，而且都是非结构化的。所以，如果我能把所有这些放在一个结构化的数据库中，并以某种方式把它理解为一个个体，当然，我们不需要人工智能或人工智能。但是，如果我需要获取所有的非结构化数据，并从中找出一些意义，没有其他办法，我确实需要人工智能和 ML。

我的意思是，我首先需要一些统计机制，来弄清楚并理解它。我确实需要人工智能/人工智能，也许是深度学习，来搞清楚其中的一些道理。但是，我需要将这个工具与主题专业知识结合起来，我需要将这个工具与我作为个人或人类在过去学到的东西结合起来，因为人工智能没有这种背景。

是的，我们可以学习，各种各样的学习，但它并不总是在各种环境下都有效。我们需要将它与主题专业知识和一些边界条件结合起来，然后我们需要将它与行动结合起来。因为人工智能/人工智能会给我们一些见解，但需要有人根据这些见解采取行动。

所以对我来说，它是工具箱中的一个工具，当它有意义时，我们需要利用它。当有意义的时候，我们需要依赖主题专业知识，最终事情会发展到我们需要进入下一个抽象层次的地步。

因此，这总是一个将抽象层次越来越高的游戏，这就是我们所追求的。所以，也许有一天会有会飞的汽车和机器人会取代我们所有的工作，但不是现在。

*Michael Vizard:* 我该如何弥合 IT 中的这一鸿沟，也许我们谈得不够多，但显然有一个非常以编程为中心的 DevOps 社区。使用 API。还有使用 ITSM 工具的传统 IT 管理员。它们本质上更倾向于图形化。随着我们向自动化迈进，你认为这两个阵营会融合吗，或者这一切会如何发展？

在我看来，我认为他们必须融合。我认为每一部分在事物的层次中都有自己的空间。在我之前的生活中，我很早就学会了一件事，如果你不能在仪表盘上采取行动，那它还有什么用？

因此，如果你想一想今天存在的所有仪表板，最终会有关于某个问题的数据以某种方式呈现，以便有人可以查看并采取行动。如果你把基础设施看作代码，那就是事实。

基本上，让我们移除控制面板，让我们弄清楚发生了什么，触发了什么，这允许我采取这些预设的操作。让我们试着去掉人类看着仪表板，思考它，并对它采取一些行动的行为。

现在，再一次，我在描述乌托邦，但是我认为作为工程师，作为操作人员，我们都需要向它前进。对我来说，当问题不太好理解时，仪表板是一个好的开始。随着我们对问题理解的加深，我们需要转向代码。

仪表板在帮助我们理解越来越复杂的问题方面仍然有其一席之地，但是随着问题被更好地理解，它们需要转向代码。至少我是这么想的。
