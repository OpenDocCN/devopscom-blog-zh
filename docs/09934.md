# 敏捷定价，敏捷吸收

> 原文:[https://devops.com/agile-pricing-agile-uptake/](https://devops.com/agile-pricing-agile-uptake/)

非常有趣的是，许多软件供应商向我们出售在我们的[敏捷](https://www.atlassian.com/agile/)和/或 DevOps 基础设施中工作的工具，但在他们的销售周期实施中却一点也不敏捷。

顶级供应商提示:如果你有一个标题为“定价”的页面，上面没有价格，那你就做错了。

让我们严肃点，这里。在我可以实现一个你声称可以使我的开发或部署生命周期更加敏捷的产品之前，我不应该放下一切，打电话给你，经历一个漫长而复杂的过程。

除了“定价页面上没有价格”的问题，还有一些供应商不遗余力地让我们不知道他们的成本。有些失败是因为他们有公共云映像，上面有定价，但有些人甚至通过包括“自带许可证”和廉价出售实例来绕过这个问题。

几乎在每种情况下，竞争对手的定价都是可用的、简单的，并且可以计算出 [TCO](https://devops.com/?s=TCO) 。如果你的不能，你就没有赢得别人的关注。

我可能在这里提到过，有一次，一个知名的数据库供应商试图使用“您必须与我们交谈”的模式向我们收取疯狂的费用。我升级到我的副总裁，他告诉销售人员，“我不买你的兰博基尼，”然后我们付费将产品移植到一个竞争数据库，价格开放、简单，并尽量减少在违规数据库上使用该产品。这个故事的寓意是，“保持简单，让它有价值。”我们会为那个项目多付几百万(已经很多了)。供应商通过制定复杂的定价方案一无所获，这些方案允许他们说每个端点都是用户，并坚持我们应该为他们支付全额许可费。

顶级供应商提示二:我们是专业人士。我们每天都在安装、配置和管理各种系统。如果您的系统需要服务来启动和运行，您应该考虑这是为什么。

我们不希望花费几个月的时间来安排安装和配置，因为您(供应商)认为服务是实施的一个必不可少的部分。我们可以部署整个基础设施，而不需要援助；您的产品有什么问题，我们不能以同样的方式部署它？

一般来说，组织愿意花高价购买能完成工作的产品。隐藏的定价、混乱的定价和使系统运行的强制性服务都使一个组织更难评估产品的价值。说“开始免费试用”而不解释价格可能是有史以来最滑稽的供应商举动。当然，在注册免费试用之前，我们想知道我们可能会遇到什么情况！

以清晰合理的价格卖给我们好的工具。我们会买下它们。因为我们有工作要做，我们需要工具来做。我们不想放弃一切去了解这样做的成本，也不想被迫购买服务来让它运行。
