# 打破软件开发的黑箱

> 原文:[https://devo PS . com/break-open-the-black-box-of-software-development/](https://devops.com/break-open-the-black-box-of-software-development/)

如果你问一位航空公司高管，就在几十年前，他们最大的支出是什么，他们会很容易地回答:“飞机燃料。”如今，你很难找到任何一个行业的高管不回答“软件”但区别就在这里:有了喷气燃料，那位高管会告诉你，他们每加仑/升燃料(或者磅燃料，如果我们是技术性的)可以飞 x 个座位英里数。随着软件开发的进行，费用变得越来越重要——也变得越来越不稳定——但是投资的回报很少是清晰的。在过去的十多年里，软件开发已经成为几乎每个行业中增长最快的(通常也是最大的)开支项目。所有的公司都在为他们的产品添加软件体验，他们需要人员和工具来构建、支持和持续优化这些体验。这要花钱——很多钱。

问题是，语言、度量标准和软件开发所处的世界并没有转化为业务领导可以容易理解的操作或财务术语。

“去年我们在软件开发上投入了 X 百万美元，”首席财务官可能会说。"我们有什么可以证明的呢？"

与此同时，技术领导者可能会回应说，“我们每天都在做 X 部署！”或者，“我们写了 X 千行代码！”

他们也可能在说不同的语言，因为实际上，他们是在说不同的语言。

问题是，你不能管理你不能测量的东西。继续我们的航空主题，软件开发仍然是一个“黑匣子”——这个隐藏在众目睽睽之下的信息来源——财务主管似乎无法窥视。

### 为什么是黑盒？

技术领导者不太可能有意回避商业领导者——更有可能的是，他们根本没有就说同一种语言所需做的事情达成一致。

实际上，眼下的问题是，“我们正在进行的这些投资会变成我们想要的业务成果吗？”但是技术领导者觉得必须回答的问题是，“你的人整天都在做什么？”

这些是敏捷和 DevOps 没有解决的问题——将敏捷或 DevOps 指标(也称为 [DORA 指标](https://devops.com/?s=DORA+metrics))转化为实际的、可操作的指标，以便业务领导理解。尽管这些度量对团队非常有用，但是用它们来指导业务是错误的。

随着软件开发成本的持续上升，商业领袖和技术领袖的目的不一致，使用不同的语言，互相扯皮。

但是软件开发的成本太高，商业领袖无法继续忍受这种不断上升的成本“黑箱”。理解软件投资的价值并弄清楚它们如何影响业务成果是财务主管的受托责任——否则他们的工作可能岌岌可危。

如果这看起来像一个遥远的威胁，它不是:我最近看到一个头条新闻，关于一位杰出的汽车高管失去了工作，因为他们无法获得必要的洞察力，以及从他们的巨大软件开发成本中获得业务成果，这影响了他们整个公司的竞争力。

它正在发生。商业和技术领袖都有责任打破这个黑匣子，开始说同一种语言。

当然，下一个问题是，“怎么做？”

### 如何弥合技术和商业领袖之间的沟通鸿沟

长久以来，商业领袖们已经适应了技术领袖们提供的活动度量标准——他们已经容忍了这种缺乏可见性，并找到了其他“战略性”的方法来证明不断上升的软件开发成本。也许他们一直在质疑 A 如何导致 B——例如，增加软件开发资金可能会导致更高的客户保留率或应用程序使用率——但他们没有切实的方法向自己证明，更不用说向投资者证明，这是否真的在发生。

为了理解和量化软件开发成本，我们必须开始对我们的技术领导者提出更多的要求:与他们一起建立现实的目标和关键结果(okr ),并让他们对与结果实际相关的业务度量负责。这意味着坚持业务结果——而不是接受活动指标。

### 稳定基金产品，而不是项目

对于金融业来说，这是一颗难以下咽的药丸:当谈到交付软件支持的客户体验时，它与优化成本无关。这比 X 美元进等于 X 美元出要微妙得多。按项目融资可能有助于使数字简洁，但它不能反映业务的现实。

基于项目的资助也倾向于让项目领导保持以成本为中心的心态:它让所有资源看起来都是可替代的；好像工作和执行工作的人是可以互换的。

敏捷的目标之一是为客户实现价值最大化。但是当我们按项目资助时，我们经常在一项工作真正开始产生价值之前就停止资助它—[宣布它完成了](https://devops.com/?s=definition+of+done)。我们有一个错误的结束日期，它不能反映我们出资创造的价值的实际结束。

相反，财务组织应该转向产品的稳定资助:资助一个成果，并给予团队停止、改变或扩展其工作的能力，以实现产品的业务成果。

这是一家著名的英国银行近年来正在经历的转变——从传统的投资组合管理到精益投资组合管理，包括向稳定融资项目的转变。你可以在[本次网络研讨会](https://blog.planview.com/royal-bank-of-scotland-the-journey-from-traditional-portfolio-management-toward-lean-portfolio-management/)中了解更多关于 NatWest 集团(前苏格兰皇家银行)如何实现这一转变的信息。

### 用流程度量可视化价值流程

最后，我们必须停止将技术指标和业务指标的收集视为两个独立的、不相关的工作。敏捷和 DevOps 在加速软件开发的吞吐量方面很棒……但是团队交付速度的虚荣心不足以证明软件开发中不断增长的物质投资。

财务领导者有机会成为技术领导者和商业领导者之间的通天鱼。如上所述，鼓励团队之间更清晰交流的一种方法是使用 OKRs 将软件开发目标自然地与有形的业务结果联系起来。我们越是练习在技术和业务指标之间建立联系——我们就越接近最终打开那个难以捉摸的黑匣子。

这其中的一个关键部分是以不同的方式交流操作度量:开发一组核心度量来跟踪软件交付中的业务价值流，并提供一种机制来将每个产品的价值流中的投资与该价值流的业务结果相关联。

可视化软件开发过程中的价值流可能不像在生产车间观察价值流那么容易——但这并不意味着这是不可能的。由于我们拥有大量的数据和可视化工具，使得软件产品及其价值流像生产线一样清晰可见成为可能。

由 Mik Kersten 博士创建的流程框架(Flow Framework)解决了打破软件开发的黑箱和弥合业务与技术之间的差距的关键挑战，以优化业务价值向客户的流动。

### 用价值流管理打开黑匣子

软件开发的黑盒本质上是一个沟通问题。商业和技术领导者都有责任弥合这一沟通鸿沟。业务领导者需要对技术领导者提出更多要求，并坚持将活动转化为业务指标。技术领导者需要练习将他们的努力与有形的业务成果联系起来，以帮助证明他们不断增长的开支是合理的。

当然，使用专门构建的工具来促进这种可见性和交流是有帮助的。
