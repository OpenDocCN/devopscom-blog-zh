# 模糊测试开发者指南

> 原文：<https://devops.com/developers-guide-to-fuzz-testing/>

模糊测试在开源社区的开发者中越来越受欢迎。 [谷歌的开源安全(OSS)团队](https://github.com/google/oss-fuzz) 最近报告称，通过使用模糊测试，在 650 个开源项目中发现了 4 万多个漏洞。但是如果你不知道什么是模糊测试，不要担心。你来对地方了。

## 什么是模糊测试？

模糊测试是一种动态测试方法，用于发现软件中的错误和安全问题。在模糊测试期间，测试中的程序或函数在无效、意外或随机输入的情况下执行，以发现不太可能或意外的边缘情况。

## DevSecOps 的模糊值

正如我们所知，DevSecOps 通过在软件开发生命周期(SDLC)的每个阶段添加合适的安全测试方法，扩展了 DevOps 和 Agile 的方法，创建了一个动态和连续的测试过程。

测试过程的一部分可以而且应该包括模糊化，以检测 SDLC 所有阶段的安全性和稳定性问题，并使开发人员能够快速安全地交付软件。例如，通过在每次代码变更时运行自动化模糊测试来防止回归生效。

特别是在汽车行业和复杂的嵌入式项目中，需要处理大量不可信的复杂数据，fuzzing 为 [DevSecOps](https://devops.com/?s=DevSecOps) 带来了巨大的好处。事实上，开发人员可以应用类似于单元测试的模糊化来自动测试他们的安全关键模块的潜在漏洞。但在集成层面上使用 fuzzing，它们也可以覆盖不同模块 之间的 [交互中可能出现的所有复杂和危险的边缘情况。](https://www.code-intelligence.com/blog/automating-embedded-security)

## 模糊测试的好处

Fuzzing 代码在一秒钟内生成数千个自动化测试用例，fuzzer 接收关于代码覆盖率和程序状态的深度反馈，以模拟源代码执行期间不同模块的交互。fuzzer 可以在接下来的迭代中根据运行时反馈调整和静音输入，以最大化代码覆盖率。一些好处是:

1.  **(几乎)没有误报:**如果发现崩溃，fuzzing 会为您提供错误来源和导致问题的恶意输入。所以，你可以确定每个发现都是一个真正的漏洞。
2.  **调试可重现的结果:**现代模糊测试工具将为您提供导致问题的输入，包括错误的来源。这样，您就已经有了许多有用的信息来重现 bug 并将其直接加载到调试器中。
3.  **在被遗忘的边缘案例中发现问题:**因为 fuzzing 正在执行一个带有无效或随机输入的程序，它还揭示了不可能和意外的边缘案例错误，否则这些错误不会被发现。

## 谁使用模糊测试？

随着网络安全法规和标准在全球范围内不断扩展，越来越多的组织正在扩展网络安全计划，并在交付软件之前运行自动化安全测试。整个行业，尤其是那些涉及高级质量和安全法规的行业，正在制定推荐模糊测试的安全测试标准，包括汽车、航空、金融、医疗保健、电信和能源。

例如，深入研究汽车行业，各种国际标准化组织(ISO)和其他组织标准现在推荐 fuzzing:

*   *[ISO/SAE DIS 21434:道路车辆-网络安全工程](https://www.iso.org/standard/70918.html)*
*   *[【NIST 软件验证安全指南(行政命令 14028)](https://www.nist.gov/itl/executive-order-14028-improving-nations-cybersecurity/recommended-minimum-standards-vendor-or)*
*   *[欧洲经委会 WP.29:联合国统一车辆法规世界论坛](https://en.wikipedia.org/wiki/World_Forum_for_Harmonization_of_Vehicle_Regulations)*
*   *[【ISO 26262】:公路车辆——功能安全](https://www.iso.org/standard/43464.html)*

## *最佳实践*

*进行安全测试(包括模糊化)的最有效方法是持续进行*。这意味着将 fuzzing 集成到您的 CI/CD 管道中。这样做可以创建快速的反馈周期，以便开发人员可以在代码发布之前修复安全漏洞。将 fuzzing 工具集成到像 GitHub 这样的代码托管系统中也很有价值。这样做允许一旦发现问题，就在开发运维团队中共享通信和警报，这有助于修复漏洞。**

## **能找到什么 bug？**

**虽然这种实践仍然相对较新，但 fuzzing 已经在不同的用例中发现了数千个 bug。发现的错误和漏洞类型包括:**

*   **远程代码执行**
*   **注射，可由不受信任的输入触发**
*   **内存泄漏:不正确的内存分配**
*   **敏感数据泄露:个人数据意外泄露**
*   **功能错误:没有被响应的输入**

**以及更多的错误，如未定义的行为、未捕获的异常和缓冲区溢出。 [了解更多](https://www.code-intelligence.com/blog/what-bugs-can-you-find-with-fuzzing) 。**

## **入门指南**

**你可以尝试一个开源的 fuzzer，比如 [Jazzer](https://github.com/CodeIntelligenceTesting/jazzer/) (用于 Java 测试)或者 [CI Fuzz CLI](https://github.com/CodeIntelligenceTesting/cifuzz) (用于 C/C++测试)。当您习惯于模糊化并希望在更复杂的环境中使用这种方法时，企业解决方案提供了一些附加功能，如 API 模糊化、报告和 CI/CD 集成。**

**所以试试吧，加入开源开发者社区，使用 fuzzing 快速安全地发布他们的软件。**
