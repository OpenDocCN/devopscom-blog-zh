Alan Shimel: 嗨，我是 DevOps.Com 的 Alan Shimel，又来参加 DevOps 聊天了。今天的聊天嘉宾是黑鸭软件公司的迈克·皮坦格。迈克，欢迎。

迈克·皮腾格:非常感谢，艾伦。很高兴来到这里。

Alan Shimel: 那么，Mike，我们的观众，你知道，我相信我们的观众中有很多人以前没有见过你，除非你邀请了你的亲戚。但是你能给我们介绍一下你的背景吗，迈克，还有你在黑鸭中的角色？

迈克·皮坦格:当然，当然。因此，我已经在软件安全领域工作了大约 16 年。我从@stake 开始，那时它还是一家领先的软件安全咨询公司。我在那里管理产品团队，当赛门铁克收购@stake 后，我将该团队剥离出来，成立了一家名为 Veracode 的公司。我在那里呆了几年，然后开始为 Bit9，Digital Guardian，BeyondTrust，Coralogix 做咨询工作。

黑鸭是我的一个客户，大约两年前，我们参与了一场关于产品改进等等的讨论。大约一年前我回到他们身边。他们叫我回去做另一个项目，我发现这个机会非常有趣，所以我全职负责他们的安全策略。

艾伦·希梅尔:非常好。迈克，信不信由你，我们的观众中可能有些人甚至不熟悉黑鸭。

迈克·皮坦格:这很难相信，艾伦。*【笑】*我们一直存在于-

艾伦·希梅尔:很久了，我知道。我已经写了你们很久了。不过话虽如此，以防万一还是给他们一点背景吧。

迈克·皮腾格:当然。该公司大约有 13 年的历史，我们非常擅长的是查看源代码或二进制文件，以确定其中使用的开源组件。一旦我们确定了这些，我们就可以告诉客户他们想知道的任何事情，因为我们跟踪开源项目，并且已经做了大约 13 年。

在公司生命的前 10 或 12 年，人们对此的主要兴趣是在内部运行我们的软件，以确定开源，但他们对它感兴趣是因为许可证合规性。他们想确保他们没有使用像 GLP 代码和 _____ 分发这样的互惠或限制性许可；一些可能会让他们自己的 IT 面临风险的事情。

大约五年前，我们开始从国家漏洞数据库中添加安全信息，一年前，我们推出了一款专门针对安全的产品。这是一件大事，因为尽管许可证合规性仍然是风险管理人员的一个问题，但开源软件，因为它变得如此普遍，也是安全人员的一个问题。

有——有超过 4000 个新的漏洞——对不起，仅在过去两年，国家漏洞数据库中就报告了超过 6000 个新的漏洞。因此，除非人们意识到他们正在使用的软件以及他们在该软件中使用的组件，否则他们无法保护自己免受这些漏洞的影响，这就是我们的切入点。

*艾伦·希梅尔:*优秀。Mike，今天我想和你谈谈开源。当然，开源漏洞和扫描是当今的大话题，尤其是当我们在容器的上下文中谈论它的时候。我知道黑鸭实际上正在围绕容器安全和漏洞扫描等方面做一些工作。但是我想再往上 50，000 英尺，谈谈整个开源——开源在整个 DevOps 革命/发展中的作用，如果你愿意的话。你能谈谈吗？

迈克·皮腾格:当然。是的，所以我的意思是，当然，最重要的是，随着容器的流行，许多容器都运行在轻量级版本的 Linux 上，当然，这是开源的。我们认为风险不在于使用 Linux，而在于开源操作系统或组件是否比商业组件更安全。这变成了一个宗教争论，你可以一直争论下去。

事实上，你只需要知道你在用什么，并密切监控。因此，我们对 Docker Hub 上的容器进行了非正式的研究，发现大多数容器都存在安全问题。很容易看出这是如何发生的，因为容器的构造就像软件的构造一样。人们通常会引入开源组件，他们会构建一个 LAMP 堆栈，并在其中添加他们的应用程序。

如果他们要部署一个新的应用程序，他们可能会剥离应用层，可能会剥离到 Linux 核心，然后重新构建。但是，如果他们重用这些组件，而某个漏洞恰好从一开始就存在，或者某个漏洞在您构建初始容器后被发现，它就会自我传播。所以人们有理由担心，我想这个词是对的，这些东西的安全性。

从我们的角度来看，更重要的是了解容器的卫生状况，了解您在使用什么，然后密切监控，因为两年内出现了 6，000 个新漏洞，很明显，您的安全状况可以在您不采取任何措施的情况下发生变化。

艾伦·希梅尔:当然。迈克，我们在——你知道，这很有趣。我刚刚看了一下我们即将发布的关于 DevOps 工具采用情况的调查结果。它被分成超过 10，000 个的大型组织，以及大约相同数量的不到 200 个的组织。所以中间没有太多东西。但是，在这些人带来的所有工具中，至少有 25%是开源的，不管他们是——让我们称之为合法的软件供应链，还是通过后门，如果你愿意的话，影子——25%或更多。

迈克·皮腾格:这实际上比我们看到的要低。

*艾伦·希梅尔:*真的吗？

Mike Pittenger: 所以我们有一部分业务，黑鸭点播。因此，我们经常在并购交易中被引入 T2。所以如果我要买你的软件公司，我会“黑鸭子”他们把它当作动词使用。我打算回避它，以了解相关的风险或可能存在的任何风险。这些都是商业应用。你认为他们拥有大量的所有权价值，因为有人对收购该公司感兴趣。

即使在这些情况下，我们也看到 35%的代码是开源的。如果你在谈论一个内部应用程序，比如说，一家银行正在运行，他们自己构建应用程序，自己使用它们，看到这些应用程序包含 70-80%的开源代码并不罕见。他们这么做是有道理的。

我的意思是，开源让他们不必从头开始构建这个功能，所以它提供了关键的功能，而没有购买成本。因此，它缩短了上市时间，降低了开发成本。他们这样做完全有道理。

*艾伦·希梅尔:*明白了。我很惊讶听到。我认为 25 是一个很大的数字。你是说这个数字很小，但意义重大。

迈克·皮腾格:是的。它很小——所以如果你看看那些商业应用——我们将很快发布一份我们自己的报告——但这些应用中的独立开源组件的平均数量超过 100 个。

*艾伦·希梅尔:*明白了。嗯……疯狂。Mike，我想——信不信由你，正如我在开始录音前告诉你的，我们在这里的时间过得很快，我们已经录了八九分钟了。我只是想快速进入整个容器，并围绕它开放源代码和安全性，以及黑鸭正在做的一些事情。再说一次，我们的观众可能不熟悉。你能给他们一点——只是一点点味道吗？

迈克·皮腾格:当然。所以我们与红帽和 Docker 都有合作关系。在 Red Hat 领域，我们与他们合作，在他们的 open shift 平台即服务和容器平台 Red Hat Atomic 上实现安全性。这里的想法就是能够在部署容器之前快速评估它的安全性。因此，我们可以将扫描客户端作为一个容器来运行，从存储库中获取 Docker 映像，并在部署之前对其进行扫描。很简单。

在我们的世界里，你可以把 Black Duck 指向一个容器，你可以把它指向一个典型的与构建服务器集成的开发过程。但是如果你想的话，你可以把它指向一个存储库，对我们来说，容器只是另一个代码位置 _____ 扫描。当我们看到这一点时，我们不仅仅是在看组件，我们还在看这些组件是由什么组成的，并且能够在修补程序级别识别这些组件。

因此，与其说您使用的是不良版本的 _____ 和 Linux，不如说我们实际上可以查看每个组件，并判断这些组件是否已针对报告的漏洞进行了修补，从而使开发运维经理的工作更快，了解他们需要担心什么，不需要担心什么。

*艾伦·希梅尔:*明白了。非常好。你知道，Mike，我们只剩下一两分钟了，但显然这是一个动态的情况-整个容器的事情，DevOps 和开源的所有事情。如果你能凝视一个水晶球，你认为我们在未来三到五年内会走向何方？你认为它将走向何方？

迈克·皮腾格:嗯，我认为集装箱显然会快速增长。我们看到一些——在部署方面犹豫不决，在一些世界中转向容器，但我认为这只是试图更好地理解它。一个类比可能是几年前人们转向云计算。他们希望在孤注一掷之前了解收益和风险。

因此，我们看到一些非常大的公司积极地接受它，但许多公司，比如说金融服务行业，仍然在测试它，玩它，看-试图决定他们什么时候跳进去，他们是从游泳池的浅水区开始还是直接进入深水区。

*艾伦·希梅尔:*明白了。明白了。信不信由你，迈克，我们的时间到了。我知道这似乎过得很快。但我想感谢你今天的出现，你知道吗？如果人们想了解更多关于黑鸭的信息，请快速登录他们的网站。

迈克·皮腾格:是的，当然。是 BlackDuckSoftware.com。我们确实提供了——有一个可以在应用程序或容器上运行的免费试用版，你可以直接在网站上注册。是 www.BlackDuckSoftware.com 的。

艾伦·希梅尔:完美。迈克·皮坦格，黑鸭软件公司。感谢您成为今天 DevOps 聊天的嘉宾。

迈克·皮腾格:非常感谢，艾伦。很高兴来到这里。

艾伦·希梅尔:好的。我是 DevOps.com 的艾伦·希梅尔。
