# 一切都是 DevOps

> 原文:[https://devops.com/everything-is-devops/](https://devops.com/everything-is-devops/)

有趣的是，我们在 It 界往往是绝对主义者，把所有东西都塞进当前的桶里。XML 将会消除编程——尽管它甚至不擅长数据表示。NoCode 将消除编程(是的，这是一个主题，我将跳过我们已经经历的其他 50 个左右的“结束编程”周期)，AI 将消除它，你*必须*有一个移动应用程序，即使没有人会使用它，我的操作系统/浏览器/语言比你的好，将会赢——你明白这个想法。

我们目前正接近“一切都是开发工作”阶段的尾声。DevOps 是一个伟大的方法论，它帮助解决了几个棘手的 IT 问题，但是，像所有事情一样，市场营销和一大堆想要“正确”的聪明人走得太远了。或者他们有吗？

DevOps 的问题在于，与那些试图从 DevSecOps 或 SRE 中分离出一个独立市场的人试图告诉你的相反，它被设计成真正包含所有人类 it 活动。我们可以原谅 DevSecOps 人群，因为 DevOps 的根有一个明确的“安全是一个不重要的瓶颈”的感觉。另一方面，SRE 人群似乎是“我的筒仓很特别”的另一个例子。或者，更好的是，一个跺脚的营销人员说“我们要迟到了！让我们发明类似的东西，宣称我们做得更广泛，并给它起我们自己的名字！”

对于那些持怀疑态度的人，让我们来看看它们各自做了什么。1)打破孤岛。检查。2)监控一切。检查。3)利用监测结果改进产品性能和稳定性。检查。4)设计和实现产品和产品特性。检查。还有更多的，但你得到的想法。

你知道这一切忽略了什么吗？你。

我们的绝对主义带来的一个问题是，营销人员可以决定什么是重要的。有时这正是企业 IT 所需要的。太多时候，这是市场营销人员被支付的命令。

我不在乎您称之为 DevOps 还是 SRE，我在乎的是您的组织觉得它的钱花得值——我要补充的是，大多数组织在 DevOps 之前都没有这种感觉。至少大部分我处理过的。我甚至有一份工作，他们创建了一个全新的部门，并雇佣我来管理它，以“避开 it 部门的那些人”我花在教育业务领导不要复制整个团队(如存储服务和 InfoSec)上的时间，与我构建和部署新产品的时间一样多。因此，避免这类问题是我最关心的事情。我们比以往任何时候都更“在那里”。

我很关心你是否称之为 DevSecOps。应该不会有什么不同。安全左移是解决安全人员短缺和对手能力增强这两把双刃剑的适当方法。因此，如果您的开发人员在他们的 IDE 中或每次构建开始时都获得安全的编码反馈，那就太好了！但叫它 DevOps。说这是 DevOps 流程的一部分。云配置管理之类的配置审查也是如此，这只是开发运维流程的一部分，而不是一个需要区别对待的安全步骤。

我可以继续，但希望我已经激发了你的想象力，不需要。

你真棒。精简你的思维来精简你的过程，忽略绝对主义，称之为通心粉。只要继续做需要做的事情——在最方便的地方做。继续开发、部署和监控维持公司运营的系统。
