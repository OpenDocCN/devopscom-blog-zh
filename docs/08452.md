## 对 DevOps 说不？

好了，现在开始:为什么 DevOps 不总是正确的选择？原因很简单，不是所有的制度都生来平等。对于后端主客户数据系统，稳定性胜过迭代。快速失败和前滚在当今许多最核心的业务应用程序中是不可持续的，例如银行、零售、媒体、制造或其他垂直行业。很明显，这些企业仍然需要创新，为他们的客户提供新的和令人兴奋的方式，通过 web 和移动来参与，但不能冒着后端终端业务失败的风险。

这些后端系统的交付能力需要加速，这样它们才不会成为整个业务的瓶颈。然而，重组团队和将开发与测试和发布一起改变成数十和数百个小的/微小的并行更新不能被遗留技术栈(比如后端的 Siebel 或 SAP)所支持。这甚至可能根本不可取。

这并不意味着我们应该接受上个世纪的现实，保持现状，把关键应用的发布周期定为一年一次或两次。慢慢地，我们肯定会失去任何竞争优势。

幸运的是有一个中间地带，这就是为什么我相信发布自动化比 DevOps 更重要。中间地带是发布自动化。我会解释为什么，但我需要先对 DevOps 多做一点检查。

## DevOps 是什么？

其核心是，DevOps 旨在将新的想法和功能快速传递到用户手中，并快速迭代。它通过修复软件交付生命周期中的两个传统弱点，即“开销”和“返工”来实现这一点

开销是开发人员所做的与开发新功能或改进现有功能没有直接关系的任何事情(手动部署肯定是开销栏中的一大项，调试生产部署问题或处理配置问题也是如此)。

每当开发人员不得不回顾他们已经工作过的代码时，返工就会发生。然而，这不包括错误修复，因此当测试周期花费很长时间，并且开发人员已经在开发新功能，现在需要返工以解决新发现的问题时，就会发生返工。

DevOps 通过尝试缩短周期来应对这些挑战——例如，通过添加开发人员和测试人员可以实时测试的较小更改，或者通过使用 Puppet 或 Chef 等工具来快速供应以进行测试，同时确保跨系统的一致性。

这依赖于开发人员在整个生命周期中拥有更新和部署，包括生产，这挑战了企业团队传统组织方式中的一切。它需要全新的流程，这些流程通常非常适合新开发的应用程序和现代架构，但对于许多其他核心系统来说却完全不适合(也是监管障碍)。

我相信你能明白我的意思。就像《皇帝的新装》这个故事一样，DevOps 并不是人们所认为的那样。

## 为什么发布自动化比开发运维更重要

然而，DevOps 的目标是合理的，这让我想到了我的最后一点:[发布自动化](http://automic.com/products/automic-release-automation)比 DevOps 更重要。如果做得好，发布自动化可以在这个新的、双模态的世界中适应不同的组织需求和不同类型的应用程序。它通过根除手工工作、改进周期时间、减少开销和返工来改进软件交付过程及其最终产品。

至关重要的是，发布自动化使现有的企业能够保持竞争力并确保生存，而不会迫使他们以“大爆炸”的方式完全放弃他们的 DNA。这样一个过程可能会危及一个大企业的生存。

在结束这篇博客之前，我想澄清最后一件事:我在这里提出的论点并不是试图淡化 DevOps 作为一种有效方法的好处。我也不谴责任何 DevOps 工具链技术，当然，这些都是受欢迎的进步。今年，2016 年，是(根据大多数行业专家的说法)DevOps 采用成为主流的一年，我从根本上相信是发布自动化推动了这种采用，而不仅仅是纯粹的 DevOps 转换。
