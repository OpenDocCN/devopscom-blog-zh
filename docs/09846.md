# 数据运营:开发运维加大数据

> 原文:[https://devops.com/dataops-devops-plus-big-data/](https://devops.com/dataops-devops-plus-big-data/)

在传统的开发运维中，有开发运维(我称之为开发运维)和开发运维(我称之为 T2 运维)的互补形式。在他们之间，他们自动化了工具链，并且将致力于将应用程序提供给用户的人带到了同一个团队中——不一定是同一个组织团队，但是可以协同使用他们的优势来满足业务需求。

DataOps 是 DevOps 的一个有趣扩展。您仍然需要所有的编码工作来将数据输入到系统中，并一致地维护查询。您仍然需要所有的操作端来启动和运行您的数据库/no SQL/无论什么。事实上，虽然 DataOps 环境中的开发方面可能(但不一定)更简单，但操作方面几乎总是更复杂。无论使用的大数据引擎是什么，在正常环境下，除了其他支持的系统之外，都是一个复杂的系统。我第一次安装大数据环境(碰巧是 Cloudera)是一次为期数周的学习之旅。只有在我完成之后，我才使用一个自动化工具(现在已经没有了)来简化它。我的第二轮是几个小时…但它假定了我在第一次尝试中获得的知识。

DevOps 的另一部分——持续监控和管理——在大数据环境中也更复杂，但我们一会儿会回来讨论。

一旦正常的 DevOps 系统就位，ETL/数据导入工具也需要得到支持。这些工具每天处理的数据量使得它们在数据密集型环境中的开发运维中的应用变得至关重要。如果数据获取缓慢或数据本身不准确，就会对组织产生影响。这一步还需要纳入数据科学家，即传统上未被纳入 DevOps 模型的人。但他们是能够衡量数据准确性的人，通常负责数据采集。

这将我们带到了数据操作的监控和管理阶段。通常，当我们谈论 DevOps 中的监控和管理时，我们谈论的是最终有助于可用性、响应性、自动扩展和恢复等方面的工具。像任何其他应用程序一样，DataOps 需要所有这些东西。我们已经提到，它在您的复杂环境中嵌入了一个复杂的环境。但是它也需要数据监控(一些人认为这更重要)。了解数据的状态是必要的。无论是一段代码出错还是数据中毒，监控大数据归档的健康状况对于防止结果失控至关重要。这种监控将再次涉及了解数据的人，即数据科学家，不管他的头衔是什么。这个迭代过程需要相对持续地运行，检查任何使数据有价值的东西，通常使用跨字段检查来验证数据没有以某种方式损坏，或者比较趋势平均值和当前平均值来监控结果的突然波动。

当然，对于一个系统(在研究中有很多)来说，监控部分是不太重要的，但对于任何不断访问和添加到数据池的东西来说，它是基于数据集的健康应用程序或应用程序组合的关键。

如果这一切听起来有点吓人，你并不孤单。“数据操作”这个词已经出现了几年，虽然数据科学家在使用它，但它还没有在企业中引起重视。以下是我作为首席信息官将如何处理它:鉴于 DevOps 正在运行其他项目，这是一个容易实现的结果。考虑到服务器的变化率(随着查询(尤其是长期运行的查询)的增加和减少，系统需求的变化就是一个例子)，传统的 DevOps 比许多其他企业系统对大数据有更大的好处。因此，首先推出 DevOps，除了 DevOps 如何影响 ETL 之外，不要认真考虑数据操作——不是 ETL 数据质量，而是 ETL 数据可用性/性能。只有当系统和团队就位，并且整个系统的软件/硬件/代码部分完全处于 DevOps 环境中时，才开始寻找将数据移入其中的方法。数据监控将是一个大问题，但是在 DevOps 实施的第一轮中，ETL 将显示出可以改进的地方，并且可以添加 ETL 数据完整性，因此也可以改进这些方面。建立在组织经验基础上的有分寸的、循序渐进的方法是这里的关键。数据监控和管理将需要不同于普通 DevOps 监控和管理的工具，因此让数据科学家研究可用的工具集以及它们如何在早期与现有工具相结合将有助于最终的实施。

毫无疑问，将大型数据驱动的环境迁移到 DevOps 是一项艰巨的任务。但好处是提高了对数据质量和数据质量问题的认识，以及 DevOps 为每个业务应用程序提供的更具响应性的系统和团队。随着物联网、开放数据集、合作伙伴关系和 ML 推动更大的池，任何给定组织可用的数据量目前都在大幅增加。尽早将数据操作的自动化和协作融入其中是值得的。
