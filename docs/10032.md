# 新遗迹将可观测性延伸至更多 DevOps 平台

> 原文：<https://devops.com/new-relic-extends-observability-reach-to-more-devops-platforms/>

[New Relic](https://newrelic.com/blog) 今天通过增加其可观测性平台与各种 DevOps 和网络安全工具之间的集成，扩展了其生态系统。新的遗迹可观测性平台现在与 Atlassian、亚马逊网络服务(AWS)、CircleCI、Confluent、GitHub、JFrog、Lacework 和 Snyk 以及开源的 Jenkins 持续集成/持续交付(CI/CD)平台的产品相集成。

New Relic 负责战略和体验的高级副总裁 Peter Pezaris 表示，其目标是让基于从 it 运营环境中收集的遥测数据的分析与组织赖以构建和部署应用程序的工具共享变得更加简单。

他指出，现在总共有 500 多个与新遗迹可观测性平台的集成。

在最近的一次新遗迹调查中，只有 27%的受访者报告说他们已经实现了 T2 的全叠加可观测性，只有 5%的人声称他们有成熟的可观测性实践。三分之一(33%)的受访者还表示，他们仍然主要是手动或根据最终用户的投诉来检测停机。超过一半(52%)的受访者表示，他们每周经历一次或更多的高业务影响停机，29%的受访者表示，他们需要一个多小时来解决这些停机问题。

从好的方面来看，近四分之三(72%)的受访者预计明年将维持或增加他们的可观测性预算，超过一半(52%)的受访者预计明年可观测性预算将会增加。

以这样或那样的形式，可观察性一直是任何 DevOps 实践的宗旨。在很大程度上，直到最近所能达到的最好水平是持续监控一组预定义的指标。随着可观察性平台的出现，不仅监控指标变得更加简单，而且在 it 服务被中断之前很久就启动表面问题的查询也变得更加简单。

一个可观察性平台实现了这个目标，它使得查询日志、度量和分布式跟踪成为可能，这种查询方式展现了没有预先定义的模式。实际上，可观察性平台使得调试应用程序环境变得更加简单，在这些环境中，服务之间存在的依赖关系变得过于复杂，以至于无法手动跟踪或破译。

当然，这些应用程序将需要使用能够生成平台所需的数据可观测性的代理软件。幸运的是，像 OpenTelemetry 这样的开源项目正在帮助降低仪器的总成本，让更多的应用程序开发人员能够更容易地使用代理软件。

Pezaris 指出，尚不清楚 It 团队将多快接受可观察性平台，但现在有一个明显的机会来缩小 IT 运营团队和应用程序开发人员之间的差距。他补充说，由于每个 DevOps 平台都使通过应用编程接口(API)收集数据变得更容易，因此可以收集更丰富的遥测数据。

与此同时，随着应用环境在云原生时代变得越来越复杂，大多数 DevOps 团队将需要额外的工具。现有的传统监控工具不便于发现，例如，由于呼叫被重新路由，哪个微服务可能会对应用的整体性能产生负面影响。像往常一样，令人沮丧的是，可能需要几天或几周才能确定一个可能只需要几分钟就能修复的问题。
