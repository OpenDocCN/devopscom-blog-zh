# 管理技术债务是一种平衡行为

> 原文：<https://devops.com/managing-technical-debt-is-a-balancing-act/>

技术债是产品经理的噩梦。它会减慢团队的速度，抑制向市场交付高质量产品的响应时间。然而，有时候技术债务是有益的；但是把它从头痛变成积极的是一个平衡的行为。

敏捷的项目经理和开发人员应该努力开发一种平衡，平衡创造、减少和接受技术债务。好消息是:这是可以实现的。

## 技术债务的形式

对于项目经理来说，理解技术债务是如何产生的是至关重要的，这样他们就可以识别其特征，并管理对团队有利的金额。

铺太多的砖和砂浆将花费太长的时间来开发软件特性。这就有可能使它们的推出远远超过到期日。但是专注于速度可能会忽略功能。随着时间的推移，这样的环境会削弱交付强大成果的能力。

同样，忽视第三方解决方案在满足客户需求和加快上市速度方面的力量也是过时的想法。例如，像嵌入式分析这样的软件可以快速、经济地完成一项架构任务，同时解放团队来应对其他挑战。保持敏捷，为团队提供所有的选择是至关重要的。

总的来说，项目经理的目标应该是识别技术债务的指标，并磨练他们处理每种类型的技能。

### 建筑建筑学

当团队承担交付新特性或功能的任务时，构建现有结构来支持新功能并不罕见。对实现该功能的必要条件缺乏远见或勤奋的规划可能是一个严重的疏忽，导致需求解决无效，即招致需要付款的债务。

### 速度与质量

以现代商业的速度交付通常意味着面临影响工作质量的时间限制。面对这样的限制，团队可能会走捷径。最终，这些捷径会导致技术债务的增加。

### 数量与速度

另一方面，急切地着手一个需要大量准备时间的项目会导致过度建设。这导致了一种不同形式的技术债务。较长的项目可能会受到技术变革的影响，这种变革会使一些以前完成的结构过时。债务以团队的形式出现，当他们应该留在下游时，被迫返回上游进行必要的改变。

### 近视

与上面描述的疏忽或过度建设不同，如果团队没有考虑到未来的需求，只关注当前的任务，那么它就有从零开始产生债务的风险。如果团队在开发过程中没有预料到下一个特性或为未来的功能做准备，他们可能会为这种短视行为付出代价。

### 建造或购买

开发者想要建造。这是他们的工作。但这并不总是符合组织的最佳利益。当一个团队面临复杂的需求，扩展超出了它的能力，或者完全超出了它的技能范围时，可以通过寻找满足其需求的现有平台来避免负债。在这种情况下，第三方解决方案可以节省大量时间和金钱。例如，采用最新技术构建的嵌入式分析工具和部署的全包式解决方案使开发团队能够专注于其专业知识并在其中发挥其才能。思考:团队需要发明一个新的轮子吗，或者已经有一个轮子了？选择前者会承担债务风险，而选择后者有望减少债务。

### **可管理的债务**

不过，技术债务也是一种好处。

考虑这个例子:你想推出一个平台，向客户和市场交付，但它是一个突破性的产品。对于最终用户将如何使用 it 以及他们将要求 it 做什么，您还不知道您想要或需要知道的一切。在这种情况下，将软件的功能框架交给市场是合理的，这样你就可以吸收大量的反馈，回到平台，然后构建或微调其特性和功能。像这样的行动是明智的。项目经理不会为了一袋魔豆而抵押农场——他们承担了合理数量的债务，将团队知识转化为最终会带来最大价值的产品。

认识到技术债务和它的方法，相应地避免或接受它，然后有意识地努力管理它，在可接受的限度内偿还它，这是项目经理想要做的。如果你投资支付它，或者寻求并接受第三方解决方案来完全避免它，你将保持团队对他们擅长的项目的关注和速度。
