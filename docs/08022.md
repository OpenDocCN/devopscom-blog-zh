# 机器学习改变事故管理的三种方式

> 原文:[https://devo PS . com/three-way-machine-learning-can-change-incident-management/](https://devops.com/three-ways-machine-learning-can-change-incident-management/)

重要应用程序的事件管理有许多内涵，其中许多是负面的。大多数都是当之无愧的。压力，恐慌，匆忙，紧急，放下一切，所有人都在甲板上，紧急情况，等等。通常，存在一个严重的问题，从完全失败到丧失某些功能或性能。问题或中断可能意味着收入损失、客户流失、负面社交媒体评论或差评等。时间是至关重要的，这意味着首要目标是[尽快解决事故](https://digitalanarchist.com/videos/featured-guests/ajay-singh-techstrong-tv)。通常，这意味着主要解决症状或部分原因，而不是根本原因。

如果不完全解决根本问题，公司几乎肯定会再次出现同样的问题——或者类似的问题。不确定根本原因通常会妨碍持久的修复。此外，公司失去了根据实际经验和问题主动改进应用程序代码或基础架构的机会。事后分析可能只会导致对监控和可观察性解决方案的审查以及对警报规则的不可避免的更新。大多数 DevOps 专业人士不仅理解，而且已经经历了这些挫折。于是，管理层经常想知道为什么他们的系统如此不稳定。

当务之急加上人手不足、团队超负荷工作，限制了事件管理模式的改变。尽管人工智能和机器学习被定位为几乎所有技术疾病的灵丹妙药，但这显然是一个“机器”可以从根本上增强人类改善状况的努力的例子。最优秀的故障诊断专家展现出直觉、经验和耐心的结合，仔细筛选大量数据，发现异常事件及其与不良结果的相关性。这对于机器学习来说是一个完美的应用。

通过摄取所有日志和指标来发现有意义的问题或异常，系统化的机器学习可以快速全面地筛选所有数据，以彻底了解真正的问题。使用这种方法大大加快了根本原因分析的速度。这样一个实用的解决方案为事件管理引入了三个改变游戏规则的因素。

## 事件管理的三大变革因素

首先，事件补救时间可以大大减少，而且不会出现人人都参与、无所不包的混乱局面。自动化方法可以更快、更彻底地完成任务，对专业人员的影响更小，使他们能够专注于其他职责，而不是让 sre 和工程师花费数小时挖掘仪表板、跟踪和数百万条日志线。这可以立即缓解紧张和恐慌，并减轻压力，因为可以更好、更快地补救事件，将应用程序恢复到其预期的服务级别。当然，它的价值是双重的:更快的解决问题的时间和更高的团队效率或生产力。结束或减轻专业人士必须处理的挫折是一个额外的奖励。

第二，除了更快的 MTTR，自动化的机器学习驱动的方法更全面，并能找到问题的根源。这意味着未来的事故可以完全避免。这也意味着警报规则“仓鼠轮”的潜在终结不断调整警报规则和设置以实现更好的“早期预警系统”的下意识倾向远没有必要。与其试图获得未来类似故障的早期迹象，不如主动找到并解决此类故障的原因。首先消除此类问题远胜于试图发展警报来捕捉未解决的原因。警报仍然很重要，但机器学习可以将它们的基础转移到更根本的问题上，并且它们可以由系统自动构建。

第三，基于机器学习的自动化事件管理有助于主动捕捉可能尚未直接成为事件一部分的静默 bug。它还可以在周期的早期通知开发人员。不久前，新版本在部署到生产环境之前经过了广泛的测试，允许创建特定的测试计划，进行压力测试，并有可能捕获可能在未来产生可怕后果的错误。现在有了更快、不间断的开发周期，测试主要发生在生产过程中。机器学习可以发现新的或不寻常的错误、事件模式和度量模式。这种方法可以在影响用户或导致大范围失败之前，提前发现细微的错误。

## 改变 DevOps 的生命周期

这三个结果中的每一个不仅显著地改变了事件管理生命周期，而且从整体上改变了 DevOps。研究日志和指标是机器学习的完美应用，并提供了从根本上改变事件管理的机会，将它从灾难和必要的负担转变为一种能力。应用于事件管理的自动化机器学习系统促进了发展，而不是阻碍了发展。
