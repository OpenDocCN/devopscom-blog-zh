# 企业发展中的 3 个实际变革要素

> 原文：<https://devops.com/the-3-practical-change-elements-in-enterprise-devops/>

> *文化、科技&生态系统*

大多数企业以单一的方式采用 DevOps:他们使用自下而上的方法，这种方法通常是由技术驱动的，并将所有考虑因素(如文化、指标、流程和沟通)分组到一个工作流中，该工作流通常由技术导向的拥护者单独领导和实施。

然而，这种单一的方法存在一些问题:

(1)首先，它导致对开发运维的非(也不应该是)技术驱动的变革方面的糟糕规划——尽管我们(开发运维传道者)不是变革专家，但我们相信我们能够恰当地评估影响并领导组织变革，因为我们是技术主题(开发运维)的主题专家。因此，我们忽略了会受到开发运维影响的更广泛的背景以及与非 IT 职能部门的关系。

(2)第二，它自然地将工作重点放在我们舒适区内的领域(技术方面),并可能将 DevOps 计划与企业优先事项分开。如果没有，它将 DevOps 限制在工具和自动化的技术变化上。

(3)第三，也是最重要的一点，它导致人们没有意识到 it 之外的领域的变化对 DevOps 的成功采用至关重要，这些领域可能需要专家(内部或外部)的帮助，更好的是，这些专家通常会成为变化的催化剂。

以上 3 点的后果是，在初始阶段之后，企业达到一个不完全的“债务均衡点”；这是企业 IT 在安装了一些自动化工具、建立了一个基本的实践并改变了一些行为之后所达到的状态。组织的 DevOps 传播者需要付出大量的努力、奉献和压力来进行这一初始发布，并将 DevOps 的开端带入他们的企业。当他们达到这一点时，精力已经耗尽，企业政治和生态系统的更大挑战变得明显，再推动更多团队、部门、工具和利益相关者加入进来的想法变成了负担。

那么问题就变成了，为什么这是一场斗争，为什么在企业中采用 DevOps 是困难的？原因是单一的、技术驱动的方法“考虑”了 DevOps 的软方面，未能通过将其转化为高管赞助和政治影响来利用其提供的价值；发生这种情况的原因是，高管们考虑的是一个更大的企业视角，它延伸到他们的部门和 IT 部门之外，但技术驱动的单一方法的视角更窄，无法解决围绕技术和文化的企业生态系统，因此也无法解决高管们的所有问题。

根据我的经验，企业中的 DevOps 鼓吹者未能超越他们的平衡点的主要原因之一是，他们将 DevOps 视为单一的技术变革，而不是企业变革。

我从多个企业开发运维之旅中看到的是，企业开发运维的采用应该分为 3 个主要的“变革元素”,开发运维倡导者应该在一个项目框架下分别为每个“变革元素”制定计划:

**第一个变化要素是“技术”**–企业 IT 环境通常因定制和打包产品而多样化。拥有了解哪些 DevOps 工具适合企业环境以及如何将它们集成到定制和打包产品中的技术专业知识是关键。此外，理解“实现 DevOps”对不同的 IT 系统意味着什么是至关重要的；例如，在速度不如质量重要的情况下，您是否需要自动发布收入有保证的系统，或者仅仅实现质量检查就足够了？此外，您从哪些系统开始您的旅程，您基于什么标准选择这些系统/项目？

**第二个变化元素是“文化”**–这是关于 IT 团队将如何改变他们的工作方式，例如，开发人员需要更频繁地签到，业务分析师需要学习编写行为驱动的需求，运营部门应该开始更多地了解开发人员的工作。另一个例子是开发人员(或测试人员)如何…等等。目标将会改变，变得对开发人员更加友好？(例如，目标应该关注团队之间的交接点，而不是团队特定的活动)。这更侧重于内部 IT，而不是企业的其他部分。当采用在没有变更计划的情况下分阶段进行时，挑战就出现了——当开发人员可能需要改变他们的方法，而测试人员和操作团队却不需要。在这种情况下，会产生摩擦。

**第三个变革要素是“生态系统”**–这是信息的核心，是关于如何在 it 之外规划组织变革。此元素是关于开发运维如何影响 IT 领域之外的企业，以及如何将开发运维定位于企业的其他部分。以下是在这一领域应该做些什么的几个例子:

(1)寻找背景:DevOps 布道者需要一个*背景*用于他们的计划，并且它需要补充组织中的其他工作。例如，传播者应该选择一个组织痛点，如“产品质量差导致的客户体验差”，并将 DevOps 定位为更广泛的企业计划中的 IT 解决方案，以解决该问题——这有助于传播者采取利益驱动的方法来采用 DevOps，这有助于驾驭企业的政治。

另一个例子是“精益倡议”——在我见过的大多数组织中，通常有一个过程改进或精益倡议在非 IT 功能中进行；传道者需要与这些相关的计划挂钩，展示 DevOps 与组织的精益方向一致，并将 DevOps 定位为精益之旅的 IT 部分。

(2)第二，传道者需要理解和评估开发运维在 IT 之外的影响。例如，当他们通过改变开发人员的目标来改变文化时(见上文)，这如何影响企业的整体人力资源战略，该战略将角色分等级，并有一个“明星”的明确定义，以及如何提升这个“明星”？与人力资源相似的是财务治理:DevOps 生命周期如何与大型企业的预算和财务治理流程一起工作，大型企业在通常不迭代的阶段/阶段发布预算？

另一个例子是布道者如何将他们的 DevOps 度量与更广泛的组织绩效 KPI 和业务目标联系起来？布道者如何与企业的托管服务提供商和系统集成商打交道，他们很可能位于监管部门，如何让他们在不破坏现有合同的情况下以 DevOps 的方式工作？

(3)最后，福音传道者需要了解政治形势，以及如何获得信息技术和非信息技术对他们的福音传道之旅的支持。这一领域的活动应该解决以下问题:传道者如何获得 IT 和非 IT 高管的赞助，以获得非 IT 团队的支持？

回答上述所有问题需要合作伙伴关系和其他非 IT 企业部门的帮助——但是传道者如何搭建这些桥梁呢？

由于所有这些生态系统挑战，我认为在企业中采用 DevOps 应该被视为一种*企业*变革，而不是*技术*变革。开发运维传道者应该在一个开发运维项目下为上述 3 个变革元素制定单独的计划，并且不应该羞于寻求非 IT 帮助(外部和内部)。

在接下来的几篇博客中，我将分别讨论每个变化元素。然后，我将回答你们大多数人现在可能正在问的问题(继续类比):那么，如果企业通过这个“DevOps 均衡”并达到“DevOps 逃逸速度”的一个点，会发生什么？当 DevOps 成为公认的企业标准时。企业会收获什么利益？为什么平衡点还不够？
