# GitBOM 工具自动识别软件工件组件

> 原文:[https://devo PS . com/Microsoft-tool-automatically-identified-software-artifact-components/](https://devops.com/microsoft-tool-automatically-identifies-software-artifact-components/)

一个开源的 GitBOM 工具，在本周的[开源峰会欧洲](https://events.linuxfoundation.org/open-source-summit-europe/)会议上讨论，可以自动跟踪每个构建工件中包含的每个源代码文件。

微软首席软件工程师 Nell Shamrell-Harrington 通过视频链接告诉与会者，基于压缩工件依赖图(ADG)技术的 [GitBOM](https://gitbom.dev/) 工具可以验证哪些工件正在跨任何编程语言、环境和打包格式使用，而无需开发人员做出任何努力。为此，GitBOM 重新利用了 Git 版本控制软件中可用的有向无环图。

所需要的就是在构建时将 GitBOM 标识符——一个惟一的、内容可寻址的引用——插入到工件中。Shamrell-Harrington 说，这个标识符然后创建一个可以被 ADG 使用的 Gitoid。然后，每个 Gitoid 将工件视为二进制大对象(BLOB ),通过它可以识别进入工件的所有组件，包括源文件、依赖项和目标文件。

还支持软件包数据交换 2.3 格式，用于共享 GitBOM 发现的软件材料清单(SBOM)信息。

GitBOM 还可以在运行时检测每个可能受到影响的软件工件中的潜在漏洞，无论其深度如何，从而使 it 团队修复问题变得更加简单。例如，这种方法使得查找应用程序环境中可能存在的每个 [Log4j](https://securityboulevard.com/?s=Log4j) 漏洞实例变得更加简单。GitBOM 仅包含创建指纹所需的最少信息，以便在运行时高效扫描已知易受攻击的工件。

Shamrell-Harrington 说，我们的目标不是取代 SBOM 的工具，而是让他们更容易以对开发者来说最无摩擦的方式收集关于工件的数据。GitBOM 允许将任何元数据链接到一组特定的相应软件工件，从而更容易发现它们是如何构造的。然后可以扩展工件 id 来生成 GitBOM 文档，供其他构建工具使用。

GitBOM 是在对保护软件供应链的关注达到前所未有的高度的时候被分享的。在 Log4j 漏洞披露之后，拜登政府要求每个联邦机构建立 SBOMs，以使开发人员能够找到易受攻击的软件组件的实例。大多数大型企业组织一般都在效仿。

然而，收集 SBOMs 只是更大挑战的第一部分。然后，组织需要获得工具来分析 SBOMs，并向应用程序开发人员提供反馈。实际上，最终，组织将能够根据用来创建软件的组件来接受或拒绝软件。有了这些见解，开发人员就可以选择升级软件组件，使应用程序符合组织定义的任何策略。

SBOMs 在 It 组织中扩散可能还需要一段时间，但眼前的问题是找到一种尽可能不打扰的方式来创建它们。
