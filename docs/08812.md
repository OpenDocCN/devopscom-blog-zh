# 我们能有多分散？

> 原文:[https://devops.com/how-dispersed-can-we-be/](https://devops.com/how-dispersed-can-we-be/)

从存储角度考虑 DevOps 很有意思。DevOps 总体上——特别是更新的开发和部署模型——试图假装长期存储不是软件交付的必要组成部分。简而言之，似乎因为在无状态开发模型中没有一个好的长期状态的答案，使用状态的存储和数据库被抛弃了。当然，失败了。容器部署模型做了同样的事情，同样悲惨地失败了。哦，*容器*没有失败，但是忽略持久存储的尝试失败了；从一开始，就很清楚这将会发生。我们正在进行早期的容器实现，并研究如何确保容器 ABC 能够访问相同的共享存储。这种需求从未消失，也永远不会消失。*应用程序可以在没有持久性的情况下运行，但是没有持久的数据，业务就无法运行。*

查看一波很酷的新产品，例如，io NIR(disclosure——我对它们了解更多，只是因为我是它们的 CMO 的朋友。其他供应商可能也有好的解决方案)让我意识到我们需要的答案就在我们面前。这不仅仅是持久存储的答案，也是存储可移植性的答案，存储可移植性从云诞生之初就是一个问题。[数据引力](https://www.dataversity.net/how-data-gravity-is-forcing-a-shift-to-a-data-centric-enterprise-architecture/)是指数据将应用保持在适当的位置。像 Ionir 这样的工具降低了数据的重要性。很多。他们无法摆脱数据引力，因为有些平台对数据存储、数据进出等收费。但这变成了一个 TCO 运营问题，而不是一个几乎无法克服的技术问题。

但同样有趣的是:如果缓慢但稳定的核心网速增长将允许我们从任何地方访问数据，那会怎样？我们*现在可以*做到这一点，但是如果你强迫高性能的应用程序离开云 A，到你的数据中心或云 B 去获取数据集，它们会很困难。然而我们经常使用[API](https://devops.com/?s=APIs)来做这件事。如果谷歌地图和谷歌地图不是简单的大型老数据库，那它们是什么样的*？然而，我们几乎在每个平台上都向他们发出呼吁。*

因此，值得一试，看看数据引力的概念是否对你来说已经过时了。对于一些真正需要超快的响应时间的应用来说，这个概念仍然适用。但是对于很多应用呢？你的应用相对于数据的位置可能没有你想象的那么重要。为关键数据集编写或获取一个 API 包装器，并在不同的平台上运行一个简单的测试应用程序。看看你得到了什么。如果它表现良好，考虑将应用程序放在它有意义的地方，并使用带有秘密的包装 API 来授权访问任何经过验证的应用程序的数据。

你们中的一些人已经这样做了。不多，但确实存在。出色的工作；继续拍！

其他人都在踢屁股和取名字，因为你们的集体工作，世界仍然每天在线。因此，继续摇摆它，但考虑一下这些天你是否真的需要应用程序靠近数据，如果你需要，数据移动工具可能如何促进将数据移动到应用程序——在去年左右之前，我们大多数人都不会考虑大型数据集。
