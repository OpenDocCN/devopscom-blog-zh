# DevOps 工作流已中断。以下是如何修复它

虽然在缩小开发人员和运营工程师之间的差距方面已经取得了很大的进步，但是角色和职责、一致性以及信息访问方面的根本差异阻碍了进一步的进步。

每个人都熟悉 DevOps infinity 循环:这是一个理想的工作流，软件开发生命周期的每个阶段都进入下一个阶段。虽然这一理念很好理解，但是组织已经致力于在 DevOps 的旗帜下将开发人员和 SREs 更紧密地联系在一起。现实是大部分时间，DevOps 无限循环都是坏的。开发人员继续专注于构建功能，而没有利用生产数据所能提供的数据驱动的洞察力。开发人员经常在没有完全理解他们的改变将如何影响性能和用户体验的情况下“祈祷和推动”。结果，公司错过了更快改进产品和更快取悦客户的机会。

根据 Gartner 的调查，90%的开发运维计划将无法完全达到预期。然而，全球 DevOps 市场在 2021 年达到 73 亿美元，预计到 2030 年将达到 370 亿美元。总而言之，我们都在花费数十亿美元来实施那些预计 90%的情况下会以某种有意义的方式失败的计划。问题是:什么是失败，对此能做些什么？T3】

随着 DevOps 工具链的不断扩展，跨工具的协作变得越来越困难，成功取决于您的组织在实现一致性的同时简化和自动化的能力。开发人员在日常工作中使用来自不同供应商的各种工具。使用最广泛的是微软的 VS Code、Atlassian 的吉拉和 Salesforce 的 Slack。最流行的代码主机/库包括 GitHub、GitLab 和 Bitbucket。弥补这一差距的是 CI/CD 供应商，如 Jenkins 和 CircleCI。Ops 工程师的日常工作通常依赖于其他供应商，如 New Relic、Datadog 和 Dynatrace。那么，我们如何确保 SDLC 中的所有利益相关者能够同时访问所有相关信息，理解目标并获得独立于所用工具的不同输入和工作流的综合视角呢？

DevOps 计划必须解决三个基本领域，以实现一致性、成功协作和简化工作流程。让我们分别解决这些问题。

1.  **协作。**第一步是良好的协作。大多数组织依靠 Slack 或微软团队来实现这一目的，但是当涉及到代码协作和指向代码的事件时，它们都不是最佳的。将代码和数据放在 DevOps 协作的中心可以确保每个人都可以访问解决问题所需的信息，减少甚至消除过程中的意见分歧。
2.  **能见度。**其次，运营工程师和开发人员必须了解整个技术体系中应用的性能。这三个障碍可能出现在组织运营的不同方面，它们的影响可能会限制团队向客户交付价值的能力，就像工作流效率低下一样。可见性有助于组织在共同目标上达成一致。
3.  **上下文切换。**第三，考虑到成功实施 DevOps 所涉及的流程和工具的数量，努力减少上下文切换并在已建立的工作流中集成工具非常重要。将协作、跟踪、历史和可观察性集成到单一平台中，可以大大提高清晰度和解决问题的时间。

对于 DevOps 团队来说，提高速度和透明度总是值得追求的。然而，重要的是要避免短视地关注工作流，因为其他恼人的问题可能隐藏在众目睽睽之下。

通过提交生产遥测，开发团队可以采用一种客观的、数据驱动的软件工程方法——节省时间并避免基于意见而非事实的潜在误导方法。可见性、透明度和速度可以帮助工程团队满足不断增长的需求，并为他们的组织提供最佳的业务成果。
