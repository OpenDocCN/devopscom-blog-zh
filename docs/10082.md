# 为什么 sre 对开发人员至关重要

> 原文:[https://devops.com/why-sres-are-critical-to-devops/](https://devops.com/why-sres-are-critical-to-devops/)

虽然站点可靠性工程师(sre)是一个相对较新的概念，但它已经成为开发运维团队的关键，帮助解决一系列运营问题，如网络可用性和用户体验。然而，在前几年，有人质疑这样一个角色的寿命。

本文将讨论为什么现场可靠性工程角色会一直存在，以及为什么他们对开发运营变得至关重要。

## 什么是现场可靠性工程师？

站点可靠性工程师通过完成之前分配给运营角色的任务，帮助连接开发团队和 IT 运营。SREs [使用一系列自动化工具](https://devops.com/5-tips-to-consider-when-introducing-infrastructure-automation/) 进行故障排除，创建可扩展且可靠的软件系统。

随着云系统变得越来越普遍，这个角色围绕着自动化和标准化展开。sre 在确保系统的用户体验达到最高标准方面也起着至关重要的作用。

## 为什么现场可靠性工程角色会一直存在

让我们来看看 SRE 角色继续存在的两个主要原因。

### 用户期望

用户体验是站点可靠性工程角色受欢迎程度上升和预期寿命延长的关键原因。在现代环境中，用户期望非常高，需要更多的专门角色来帮助提供不仅可以接受而且试图超出用户期望的服务。

即使回到 2010 年代，用户在网站加载速度[](https://blog.hubspot.com/marketing/page-load-time-conversion-rates)和他们访问信息的速度方面也更有耐心。即使用户不退出网站，也可以容忍崩溃和加载失败。

然而，由于市场的竞争以及远程工作的增长，这一切都变了，远程工作增加了对在线资源的依赖。这意味着，如果某项服务在某段时间内不可用，用户将无法工作。

SRE 的职责包括优化系统，确保系统在要求的水平上运行，并确保资源始终可供需要的人使用。这使得 SREs 对于可能没有能力充分关注满足用户期望的开发运维团队来说至关重要。

### 软件架构的复杂性

另一个为 SRE 的角色增加了很多价值的因素是，他们能够为软件环境和架构提供急需的支持，这些软件环境和架构在复杂性方面由 [继续开发](https://devops.com/introduction-to-mach-architecture/) 。Kubernetes 就是这样一个需要特别关注的架构。

跨 Kubernetes 和其他云原生平台等系统分布的应用程序需要具备必要知识和经验的专业人员。机器学习也变得越来越普遍，需要从性能角度特别关注。2020 年，仅 [就有超过 22%的公司](https://youngandtheinvested.com/machine-learning-statistics/) 有机器学习模型投入生产一到两年。

随着云原生系统成为常态，预计未来几年这种复杂程度只会进一步增加。这就是为什么必须创造出超越普通 IT 工程师的新角色。

## SRE 角色面临的挑战

与任何 IT 角色一样，站点可靠性工程师的角色总是面临挑战，这使得人们质疑 SREs 是否是一个长期的解决方案，或者它是否是一个利基专业化。

### 缺乏远离超大规模公司的机会

SRE 的角色是由像谷歌这样管理超大型 IT 系统的超大规模公司创造的。对于在全球范围内没有如此广泛和多样的 IT 基础设施的公司来说，这样的角色是否必要还存在疑问。

但是，如前所述，云原生技术的发展没有放缓的迹象，随着这种发展，小型组织中的软件资源企业可能会有机会。

### SRE 的角色被认为是模糊的

在某些情况下，SRE 角色的模糊性可以被认为是一个败笔，许多企业不确定与传统的系统工程师相比，这些专业人士能提供什么。在许多情况下，sre 被认为是软件工程师和 IT 运营角色的混合体。

幸运的是，DevOps 已经意识到 sre 的潜力，以及它们在开发人员和工程师之间搭建重要桥梁的能力，有助于实现高标准的专用任务。

## 先生们...。奉献精神

虽然这两个角色有联系，有一些相似之处，但也有一些明显的不同。

*   **流程**–devo PS 对 [开发环境](https://devops.com/aws-previews-amazon-codecatalyst-to-automatically-create-dev-environments/) 有全面的了解，这使得他们可以从最初的开发阶段到生产阶段进行变更。另一方面，sre 只能看到生产过程，允许他们提出建议以确保保持绩效水平。
*   实现–在系统上实现新特性的任务落在了 DevOps 团队的肩上。同时，sre 的任务是确保新功能不会导致任何系统故障或影响生产阶段的性能。
*   **主要关注点**–现场可靠性工程师的主要关注点是确保系统的可靠性、可用性和可扩展性。DevOps 关注产品开发的完成速度及其连续性。
*   **结构**——SRE 团队和 DevOps 团队的结构有明显的区别。 SRE 团队由在开发和运营方面拥有相似技能的个人组成。而 DevOps 团队由专业人员组成，他们的具体角色与项目的各个方面相关。这些角色可以包括团队领导、产品负责人、云架构师、软件开发人员、QA 工程师、系统管理员、发布经理等等。
*   工具–开发团队在开发产品时经常使用集成开发环境(ide)等工具。这些可以包括 Jenkins(持续集成和持续开发)、[](https://www.atlassian.com/software/jira)(变更管理)、Splunk(日志监控)和 GitHub(分布式版本控制)。 SREs 常用的工具有 Prometheus 和[Grafana](https://grafana.com/)(指标的收集和可视化，如 CPU 使用率和可用磁盘空间)、OP5 和 PagerDuty(事件警报)、Ansible 和 Kubernetes ( [容器编排](https://containerjournal.com))以及一系列云平台。
*   **错误报告**–每当最终产品中报告有错误时，开发调试代码。从开发的角度来看，sre 不参与调试，只有在出现生产中断、基础设施问题或者处理 [常见 AWS 错误配置](https://www.crowdstrike.com/cybersecurity-101/cloud-security/common-aws-misconfigurations/) 时，才需要执行此类任务。
*   **测量性能**–devo PS 用来测量性能的典型指标是部署频率和部署失败率。SREs 衡量服务水平目标(SLO)、服务水平指标(sli)、服务水平协议(SLA)和错误预算。
*   **事件处理**–在事件处理中，开发运维团队根据提供的反馈努力缓解问题。然后，SREs 分析事件后的问题，包括问题的根本原因。然后记录下来并提供给开发人员进行修复。

## 为什么 SRE 角色对开发团队至关重要–总结

正如您所看到的，这两个角色有很大的不同，但是 sre 在确保 DevOps 团队 [交付能够提供最佳性能的产品](https://devops.com/your-devops-teams-can-do-better-heres-how/) 方面发挥着关键作用。

雇用专业人员担任现场可靠性工程师的核心优势是高水平的产品性能和可靠性，以满足用户的期望，同时帮助管理复杂的 IT 架构。

在未来，如果没有这些专业人员，管理大量基于云的 IT 资产的组织可能很难交付高效的产品。因此，对 SRE 角色的需求可能会增长。
