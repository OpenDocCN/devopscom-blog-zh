# Dynatrace 调查认为未来将面临持续的交付挑战

> 原文：<https://devops.com/dynatrace-survey-sees-continuous-delivery-challenges-ahead/>

研究公司科尔曼·巴夏礼代表 Dynatrace 对 1300 名应用开发和 DevOps 负责人进行的全球调查发现，平均而言，企业希望在未来两年内将软件发布频率提高 58%。

可观察性平台提供商 Dynatrace 的战略合作伙伴关系总监 Andreas Grabner 表示，虽然很明显组织正在采用 DevOps 最佳实践来实现这一目标，但调查还发现，超过四分之一(27%)的 DevOps 团队的时间花在了手动持续集成/持续交付( [CI/CD](https://devops.com/?s=CI/CD) )任务上。近四分之一(22%)的人承认，他们在满足应用程序部署需求方面经常面临巨大压力，以至于他们必须牺牲代码质量。

作为回应，近三分之二的受访者(62%)正在投资自动化以消除手动任务，而类似比例的受访者正在寻求自动化事件响应流程。另有 45%的受访者正在投资可观察性工具，近四分之三(74%)的受访者指出，端到端可观察性对于未来的开发运维至关重要。

整整 98%的受访者还指出，将 DevOps 扩展到更多应用将在推动数字化转型和优化客户体验方面发挥至关重要的作用。共有 71%的受访者表示，无缝集成其工具链的统一平台对于扩展开发运维至关重要，而 79%的受访者表示，将人工智能扩展到 IT 运营(AIOps)领域对于他们未来的成功至关重要。

然而，调查受访者也认为孤立的团队文化(75%)和应用程序复杂性(70%)是实现 DevOps 成功的主要障碍。因此，组织正在投资手动 CI/CD 任务的自动化(62%)，消除手动事件响应(62%)和端到端可观察性(45%)。

Grabner 指出，随着越来越多的组织基于使用容器和应用程序编程接口(API)构建的微服务来构建和部署下一代应用程序，自动化开发运维流程只会变得更具挑战性。Grabner 补充说，作为降低交付这些应用程序成本的努力的一部分，Dynatrace 向云原生计算基金会(CNCF)捐赠了 Keptn，这是一个开源的企业级控制平面，用于自动持续交付云原生应用程序。他指出，总体目标是降低实施 DevOps 的总成本，作为开发更多应用的努力的一部分。

虽然大多数已经采用 DevOps 的组织在如何实现持续集成方面有很好的处理，但实现 DevOps 等式的持续交付方面被证明更具挑战性。在平台之间缺乏标准化的情况下，部署在单个 IT 系统上的每个应用程序都是“雪花”，因为它对于该平台是唯一的。这个问题在一定程度上是为什么如此多的 IT 团队仍然依赖手动流程来部署应用程序。

组织开始更积极地自动化应用程序的部署只是时间问题。如果他们在可以跨任何平台实现的 Kubernetes 应用程序编程接口(API)上实现标准化，这可能会变得更加容易。

然而，与此同时，很明显，随着数字业务转型步伐的加快，组织将面临的最大限制因素之一是应用程序的部署和更新速度。
