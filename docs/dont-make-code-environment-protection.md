# 不要让我编码:环境保护

> 原文：<https://devops.com/dont-make-code-environment-protection/>

感谢收听“不要让我编码”，这是一个讨论开发者体验设计和构建面向开发者的产品的独特挑战的播客。《不要让我写代码》由[史蒂夫·博克](https://twitter.com/sboak)和[联合创始人兼首席执行官](https://convox.com/)大卫·杜乐[主持，史蒂夫·博克是](https://heavybit-blog.squarespace.com/config#) [Opsee](https://opsee.com/) 的联合创始人兼首席执行官。

在这一集里，Docker 的首席产品设计师肖恩·李加入了史蒂夫和大卫的行列。Sean，Steve 和 David 讨论了 Docker 对 Kitematic 的收购，试图取代开发者现有工具的摩擦，以及设计在开发者工具中日益突出的作用。

这个播客是由 heavy bit(T1)带给你的，这是一个帮助以开发者为中心的公司将他们的产品推向市场的项目。

我们将这一集的“不要让我写代码”称为“环境保护”，我们请到了肖恩·李，他是 Docker 的首席产品设计师，也是 Kitematic 的创始人之一。

肖恩:谢谢。

史蒂夫:谢谢你和我们在一起。对我来说，这真是一次激动人心的谈话，因为我们可以和 Docker 这样的公司的产品设计师交谈。

肖恩:很高兴来到这里。

Steve: Kitematic 是一个非常有趣的项目，原因有很多。我是说，它引起了码头工人的兴趣。它也被证明是一种非常新颖的开发环境技术。所以我们很好奇这一切是如何开始的。

肖恩:这是在一所大学的宿舍里开始的。我是加拿大滑铁卢大学的学生，学校实际上提供了一个投球比赛。所以我们有了让开发环境变得更容易的想法，所以我们提出了这个想法并取得了成功。这就是我们如何得到足够的钱开始。

史蒂夫:哇，当时它和 Docker 有关系吗？

Sean: Docker 实际上是我们用于后端的东西。当时 Docker 是 0.3 版本。然后我们上了黑客新闻，我们就像，“哦，这很有趣。这可能是一项超级有用的技术，可以做它想做的事情。”我们的想法本质上就像你的代码的 Dropbox。

你在你的任务菜单上得到一个类似 Dropbox 的图标，然后你可以在里面放代码。例如，姜戈，它会为你运行它。你可以在浏览器上看到。

史蒂夫:哦，太好了。这样，您就有了生产环境，为您编译并准备就绪。

肖恩:是的。然后，当您添加代码时，它将同步到云环境，然后立即更新。

史蒂夫:很好。太棒了。这就像是超越 GitHub 的一步。不仅仅是提交代码；所有配置项和一切都内置在这一个工作流中。

Sean: 这就像是当时的未来计划，但起初，我们的 MVP 非常简单——只需将代码同步到远程服务器，然后看到变化。

Steve: 从一开始它就被实现为一个桌面应用程序，就像你说的 Dropbox 托盘应用程序。

肖恩:是的，因为 Dropbox 显然是灵感的来源。

史蒂夫:那么为什么是桌面应用呢？

**Sean:** 当时我们其实还没有意识到桌面应用的强大。就像初创公司通常做的那样，我们曾经在某一点上做了很多调整。然后我们与客户交谈，他们开始意识到我们在使用什么。

我们采访了加拿大最大的零售商加拿大轮胎公司。他们在寻找类似这样的东西，他们会说，“你在底层使用 Docker 吗？我们想利用这一点。”

当时，我们的产品在云中，就像一个本地的网络浏览器设置。然后我们开始思考，与其将 Docker 抽象化，为什么我们不拥抱它，然后让它对客户来说更容易？然后我们开始集思广益，结果发现拥有一个桌面应用程序实际上是非常有益的。

这就像是开发人员的最佳平台，因为他们可以超级容易地导入他们的代码。他们可以打开一个目录，而不必在浏览器中上传文件。您还可以访问他们的桌面，所以一旦您的工具变得更强大，您就可以做更多的事情。

它也非常酷，因为我们在电子层中使用了铬合金。我们在 React 中编写了 UI，这意味着我们只需要担心 UI 如何在一个浏览器中运行，而不必担心所有浏览器的兼容性。

史蒂夫:所以这不是 React 需要的东西，它只是在电子浏览器的网络视图中运行。

肖恩:我们尽可能让它有本土的感觉。

史蒂夫:好的，很好。所以从很多方面来说，它是一个真正的桌面应用程序，但具体到 UI，它是一个 web UI。

**Sean:** 很多灵感来自 GitHub 的 Atom 编辑器。他们使用同样的技术。

史蒂夫:所以这一切从一开始就是开源的？

Sean: 我们正在开发它，然后开源了它。

大卫:我很好奇是什么让你决定发布这个工具。开源服务是一个战略决策，还是你认为这样做可以获得更多用户？或者只是想要社区的贡献？你能谈谈你背后的一点点灵感吗？

Sean: 我认为最重要的一点是，如果它涉及到他们的代码，那么开发者通过查看代码来信任它是非常重要的。

我们对我们所做的事情非常开放，从社区收集反馈的部分也非常非常有帮助。我觉得，总的来说，开源项目从社区获得了更多的反馈。

Steve: 使用开发者工具似乎是鼓励参与的好方法。如果你有一个开源项目，你是在邀请人们为之做出贡献。显然，我们的 Opsee 产品是封闭的，是一个 SaaS 应用程序。所以我们没有从开发者那里得到我们在某些方面真正想要的社区支持。

Sean: 我们从 Docker 聘请了一名工程师，因为他是一名维护人员，他对这个项目非常有热情。这在这方面也很有帮助。

Steve: 接下来，你向一个开源社区发布了这个产品，你当时对你的目标用户有什么想法吗？

**Sean:** 我们在原有产品的基础上做了一些，Dropbox for-code 产品。我们有一份注册名单。这些用户中有很多更倾向于使用用户界面，喜欢更简单的东西。超级活跃的人实际上是为多个项目工作的顾问。

史蒂夫:这是一个有趣的部分。那里的人们维护着许多项目，他们正在寻找一种简单的方法来跟踪所有这些东西。且 Kitematic 适合模具。

**Sean:** 是的，Docker 被隔离也提供了那个值。

史蒂夫:那也是。现在，如果我维护多个项目，我可以为每个项目都提供一个 Docker 环境。这款应用可以让你轻松地在两者之间切换。特别是 React UI 和 GUI，作为开源项目的一部分是什么感觉？你也因此得到了贡献吗，或者那完全是内部的？

Sean:React UI 在某种程度上帮助了我们，因为根据谷歌的趋势，React 几乎和 Docker 走在同一条轨道上。所以有很多人愿意致力于 React 内置的东西。这在某种程度上是偶然的，因为当我们开始时，这是一项非常好的技术。

史蒂夫:这个项目有设计师参与吗？

肖恩:我们确实有有设计头脑的人，但没有专业的设计师。

史蒂夫:举例来说，你是 Docker 公司的首席产品设计师，我猜，你主要是一名工程师，或者你的编码背景。但是在开发人员工具空间中肯定存在这种混合世界。

肖恩:肯定的。一开始我们有黑客马拉松。许多参与者试图给 Kitematic 添加功能，然后花大力气找出合适的用户界面和合适的体验。默认情况下，我认为我想要的有点像设计一个操作系统。

一个 UI 是非常可扩展的，你想要找出你可以在其上扩展的正确点。您实际上关注的是每个组件，并且为您如何扩展当前 UI 树立了一个榜样。当人们做出贡献时，它可以遵循这种模式。你只需要稍微调整一下。

史蒂夫:这似乎也适用于开源项目。你需要建立一个广泛的基础，人们可以在此基础上进行定制。

肖恩:是的，但是最初的努力必须非常努力。因为否则你会被不同的观点所左右。由委员会设计并不总是一件好事。

**David:** 这很有趣，因为你经常会看到开源项目因为糟糕的设计或委员会设计而臭名昭著，尤其是在视觉上，最终变得有点怪异。我能想到一些我不会说出名字的 Linux 桌面管理器，但 Kitematic 实际上是一个漂亮的应用程序。

肖恩:谢谢。

大卫:我很好奇你是如何保持这一点的，即使你在做社区驱动的设计。

Sean: 我认为很大一部分是保持专注，并愿意削减开支。一开始，Kitematic 实际上有 build，你实际上可以构建一个图像，然后推它。但是，在我们弄清楚下一步之前，保持专注，让事情变得简单，实际上是让它只运行内容。

实际上，Kitematic 的重点是帮助用户在他们的电脑上安装 Docker。你知道，从视觉上来说，感觉很靠谱。我们一开始就用 VirtualBox 解决了很多潜在的问题。然后选择一个漂亮的目录，你可以点击一个按钮，运行一个容器。这就是我们追求和关注的体验。所以怎么做超级明显。

**Steve:** 你有没有因为必须在社区中坚持自己的立场而遭到社区的反对？

肖恩:不，我认为社区主要是支持的。我必须管理的是人们试图添加太多的东西，而这些东西目前并没有关注路线图。

Steve: 我想我记得你在软件工程每日播客上说过，过了一段时间后，你确实从产品中砍掉了一些东西。你能多谈谈吗？你想把它清理干净？

Sean: 有一件事被砍掉了，那就是用它来制作图像。尽管这是一件超级重要的事情，但我们决定推迟，因为我们想把它做好。我们不想过早发布一些东西，然后转向其他东西，而是希望保持专注，直到我们弄清楚这就是它。

然后把它作为一个整体释放。谈到削减更多，Kitematic 的下一个版本有点像 Mac 的 Docker。它真正解决了 Kitematic 最初试图解决的核心问题，即以一种几乎感觉无缝的方式在人们的桌面上可靠地安装 Docker。

你根本感觉不到它是在一个虚拟机中运行的，这是一开始的目标。你安装了 Mac 版 Docker。当你打开终端时，你可以输入“Docker run”，它就工作了。

Steve:Mac Docker 项目在收购 Kitematic 之前就已经开始了吗？

**肖恩:**那是收购之后。

史蒂夫:所以你从一开始就参与其中了。

肖恩:是的，我是 Mac Docker 的创始成员。

大卫:所以思考这种内部争论是很有趣的。我猜大多数设计问题都是可定制性和配置，而不是有限的选项，这创造了一个美好的体验。我觉得这种情况在开发工具中更常见。

看起来像 Kitematic 和 Docker for Mac，其中一些工具，肯定是倾向于后者，在那里你删除了 Docker 的一些可选性，以获得更好的用户体验。我觉得那真的很有趣。

Sean: 我认为这两个工具在某种程度上是互补的。人们通常通过让 Kitematic UI 与终端并排来使用 Kitematic。所以他们喜欢从 Kitematic 方面获得的透明度，以及这种感觉的可靠性。

“我看事情正在运转。我可以超级轻松地检查日志。”另一方面，终端是他们以前习惯的工作方式，这在一开始是意料之中的。然后你从那里开始。你永远也不想替换他们已经在使用的工具，他们非常熟悉的工具。

因此，我们接受了这一点，但让 Kitematic 更像是 Docker CLI 的补充。

史蒂夫:那真的很有趣。我见过类似的与 GitHub 和他们的一些工具的比较。GitHub 有他们的 Mac 应用程序和网络用户界面等等。我发现自己去它们那里是为了一些事情，比如差异和比较，合并状态，一些我想得到大量文本或者得到一些状态更新的事情。

但是对于动作，如果我想进行合并，我实际上做的所有事情都是通过命令行来完成的。但这是一个非常有趣的比较，因为听起来你也做了同样的区分。为了查看事物，也许你去 UI。而是采取实际行动？

Sean: GitHub 桌面也是所有这些决策的巨大灵感来源。

Steve: 我从来没有听说过这种明显的区别，有些事情开发者会继续想通过命令行和终端来做。但我们可以在某些方面让体验更好，比如弄清楚容器是否在运行。有一个可以一直看到这些的用户界面真的很好。
