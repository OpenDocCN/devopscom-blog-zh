# GitHub 更新了 CLI，使创建扩展更加容易

> 原文:[https://devo PS . com/github-updates-CLI-to-make-creating-extensions-easy/](https://devops.com/github-updates-cli-to-make-creating-extensions-easier/)

GitHub 本周对其命令行界面(CLI)进行了更新，使 DevOps 团队能够向[工作流](https://devops.com/?s=workflows)添加扩展。

GitHub 的工程总监 Billy Griffin 表示，该公司最初的重点是为 DevOps 团队提供一套可以通过 CLI 轻松调用的工作流。他补充说，现在 GitHub 正在鼓励开发者使用其 CLI 2.0 版本来扩展这些工作流，希望能够与更大的社区共享。

例如，已经创建了扩展，这些扩展按照分支创建的时间来排序，然后显示所有相关的拉请求。

另一方面，开发人员正在为他们自己的娱乐创造简单的屏保扩展。

开发运维团队对扩展的兴趣程度差异很大。他们中的许多人只是想采用代表他们自己创建的固执己见的工作流。其他 DevOps 团队希望能够进行适度的修改。其他人喜欢尽可能地定制工作流，以满足他们的特定目的。GitHub 提供的 CLI 2.0 版本使每个 DevOps 团队能够决定对他们的工作流进行多大程度的修改。然而，随着 DevOps 团队获得更多经验，他们越来越倾向于定制工作流。

随着越来越多的组织采用最佳 GitOps 实践，围绕 GitHub 存储库创建的工作流数量持续稳步增长。GitHub 是最广泛使用的管理软件工件的仓库，使用脚本或 GitHub 动作构建的工作流的复杂程度正在稳步提高。大部分努力都集中在简化流程上，否则会妨碍开发人员花更多时间编写代码。

当然，开发人员要更快地编写更多代码的压力从未如此之大。随着组织接受数字化业务转型，他们发现他们是多么依赖软件来创造客户体验，从而增加收入。这一挑战出现在许多开发人员在家工作以帮助对抗新冠肺炎疫情的传播的时候，许多现有的工作流程需要重新设计。与此同时，开发团队变得越来越分散，因为组织越来越习惯于在世界任何地方雇佣开发人员。

尚不清楚不断变化的应用程序开发环境会在多大程度上促使组织重新评估他们正在使用哪些平台来驱动 DevOps 流程。在过去的一年中，许多内部开发平台已经被基于云的替代平台所取代。然而，因为需要是发明之母，在疫情之前所依赖的工作流程现在不仅永远改变了，而且一旦疫情消退，随着开发团队在办公室和家里更加间歇地工作，它们也可能会再次改变。

不管这些工作流如何发展，单值 DevOps 属性将是灵活性。现在的挑战是，在不确定性持续存在的情况下，确定哪个平台最能实现这种灵活性。
