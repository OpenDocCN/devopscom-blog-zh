# 别看这个！这是个秘密！

> 原文：<https://devops.com/dont-look-at-this-its-a-secret/>

在阅读了 [GitGuardian](https://www.gitguardian.com/) 的这篇[精彩报告](https://blog.gitguardian.com/state-of-secrets-sprawl-2021/)之后，继续关于秘密的讨论——上次我对 IT 人士的个人储存库中曝光的秘密数量有点抓狂。而且是很多。我指的是*一个* *很多*的秘密。但你知道什么比“很多秘密在急于参加本周短跑的人们的个人储存库中泄露”更可怕吗？事实上，2020 年有 30，000 个秘密通过企业公共存储库泄露。那就是*三万*。

这不是某个可怜的 DevOps 人在赶时间，对个人存储库没有真正的安全支持的情况下，把一些秘密放到公共的个人存储库中。

这是一个由您的组织建立的系统，应该具有安全意识，应该定期进行扫描，并且应该*永远不要*成为黑客的信息来源。相反，it *是*信息的来源，甚至可能是黑客可以轻易窃取的关键信息。因为这个公共知识库里有[的秘密](https://devops.com/?s=secrets)。

避免这一问题的第一步是承认 IT 无法满足所有需求，并要求业务部门(BUs)提供他们背着你开发的任何应用程序，以便可以将(非侵入性)安全性应用于这些应用程序。

你要么得到一些应用程序，要么人们不信任你，你什么都得不到。在后一种情况下，如实解释隐藏在公共存储库中的秘密，并要求他们验证自己的秘密。如果人们不愿意分享他们所做的，并且仍然有人认为 It 参与是他们软件的一个巨大障碍，这是你能做的最好的事情。*耸耸肩::只要你能让他们检查他们的应用程序的漏洞，这应该不会太重要。但我更关心的是获取秘密，所以我不会在这篇博客中讨论集中式与分布式应用程序的哲学，只是说，对于秘密，如果你能让他们扫描/清理它们，谁在管理应用程序和过程的问题就不是很重要了。

我还在摇头——3 万！也就是说，每天*(当然是平均而言)有超过 100 个秘密被登记到公共的公司资料库中。请仔细检查你的组织，确保你没有参与这个总数。那是野生的；即使你考虑到这样一个事实，如果一个应用程序有一个秘密，它可能有多个；即使假设每个易受攻击的应用有四个，仍然有 7500 个应用以最可能预防的方式暴露秘密。*

*所以，如果你不确定你的公开回购，去确定一下。与“我们的 AWS 秘密被泄露”或“我们的 API 密钥被发现和滥用”相比，“几分钟的扫描或几个小时的搜索”的投资回报率相当高。*

*继续摇摆。大量的应用程序是共享秘密问题的一部分，随着自动化和 DevOps 的接管，秘密使用的增长也是如此。但你已经找到了解决办法，只要确保它们不会出现在公开回购中。然后回去继续做让企业生存下去的事情。*
