# ShiftLeft 报告揭示了应用程序的安全状态

> 原文：<https://devops.com/shiftleft-report-reveals-state-of-application-security/>

自动化应用程序安全测试平台 ShiftLeft 今天发布的一份[报告](https://www.businesswire.com/news/home/20220623005346/en/ShiftLeft-Releases-its-2022-AppSec-Progress-Report-2022)发现，只有三分之一的应用程序存在可攻击的漏洞。该报告还发现，根据实际威胁级别确定补救工作优先级的组织平均在两次持续 12 天的冲刺中修复了 76%的漏洞。

报告发现，根据去年使用 ShiftLeft 平台进行的数百万次扫描来确定漏洞，组织需要解决的漏洞数量同比减少了 97%。

此外，该报告指出，平均扫描时间中位数仅为 90 秒，这有助于推动每日扫描量同比增长 68%。报告还发现，总体平均修复时间(MTTR)下降了 37%。

ShiftLeft 首席执行官 Manish Gupta 表示，鉴于漏洞数量的增加，组织将补救工作的重点放在可在其 it 环境中利用的漏洞上比以往任何时候都更加重要。例如，该报告发现，在分析的 IT 环境中发现的所有 [Log4j](https://devops.com/?s=Log4j) 实例中，只有 4%易受攻击。Gupta 说，IT 团队与其花费数周时间寻找每个实例，不如将精力集中在实际可能被利用的实例上。

Gupta 补充说，面临的挑战是，没有足够的组织有办法根据漏洞给业务带来的风险对漏洞进行评分。

虽然在一系列高调的软件供应链违规事件之后，人们更加关注采用 DevSecOps 最佳实践，但 ShiftLeft 的报告表明，向开发人员提供一长串潜在漏洞会适得其反。Gupta 指出，被发现的大部分漏洞实际上是不可攻击的。

Gupta 补充说，这并不意味着应用程序安全性不需要改进，但是很明显，大量的时间和精力被浪费了。

至少在理论上，随着应用程序开发团队和网络安全团队之间的文化差异不断缩小，确定补救工作优先级的整体能力应该会提高。实现这一目标的第一步是使识别应用程序攻击面中真正易受攻击的百分比变得更加容易。否则，面对一长串理论上的漏洞，开发人员将继续优先开发新的特性和功能，而不是修复漏洞。

无论如何，漏洞补救问题很快就会出现。在拜登政府发布行政命令以提高联邦 IT 环境中的整体应用程序安全性之后，许多组织将采用类似的政策和流程。开发人员自然会不可避免地推后或绕过他们认为过于繁琐的过程。古普塔说，精明的 DevOps 团队不会让僵局发生，而是会投入更多时间来识别关键漏洞，这些漏洞需要在网络犯罪分子利用它们之前加以解决。

当然，问题是网络罪犯越来越善于发现和利用这些相同的漏洞。
