# 数据库可观察性:如何避开最薄弱的环节

> 原文:[https://devo PS . com/database-observability-how-going-your weak-link/](https://devops.com/database-observability-how-to-circumvent-your-weakest-link/)

如今，IT 专业人员需要监控越来越多的资产，以保护系统并了解数据库和应用程序中发生的情况。随着组织采用混合 IT 战略，IT 专业人员的任务是管理跨内部和公共云的技术堆栈，这变得更加复杂。

为了降低复杂性并创建对其 IT 系统的端到端可见性，组织必须制定[可观察性](https://devops.com/?s=observability)方法。

### **监控与可观察性**

当谈到可观察性时，许多人认为它与监控的意思相同。而监控是一个动词；可观测性是衡量系统的内部状态和条件可以从其外部输出中推断出来的程度。

建立监控系统时，仪表板是围绕 IT 专业人员可能预期会发生的某些假设构建的。在监控数据库时，团队正在构建多级组件 IP 系统。算法必须处理这些数据，用户界面必须与之交互。

然而，为了使系统高度可观察，必须包含跨各种不同端点的性能信息和其他类型的度量。特别是因为 IT 专业人员必须处理通过 Kubernetes 编排的微服务、网格、物联网或[容器。](https://containerjournal.com)

### 如何有效监控

过去，IT 专业人员通常会监控 SQL 服务器，甚至 PostgreSQL 或 MySQL 等其他数据库平台。另一方面，在这个可观察性的新时代，IT 专业人员需要对前端代码做一些不同的事情，以处理针对该数据集的不同类型的算法。通常，开发团队会编写代码，但是代码中并没有写任何东西来发出调试信号。

今天，这种情况发生了变化。以确保我们的系统是完全可观测的(因为我们不能预见所有可能的问题！)，我们必须记录不同的步骤来跟踪许多移动的部分。组织必须收集尽可能多的东西。数据越多，组织对其应用程序中发生的事情的了解就越多，从数据库到虚拟化或容器化层，再到虚拟化主机或流程编排，再到裸机。如果组织能够获得这些元素的可见性，我们就可以使我们的系统完全可见，并且更容易诊断发生的任何问题或异常。

### 可观察性方法

当使用可观察性方法时，高级仪表板很方便。他们可以帮助 IT 专业人员汇总正在发生的一切，并在出现问题时进行根本原因分析。然而，有时没有特定的问题；这只是一个一般性的性能问题。有了可观察性平台，团队可以快速发现、映射和理解依赖关系，从而更容易实现和维护服务级别目标(SLO)。

随着 IT 团队越来越多地区分仅仅监控 IT 环境和实现可观察性，他们可以使用机器学习算法和其他形式的数据科学主动发现异常。可用于管理 IT 的深度和洞察力也在急剧增加。可观察性正迅速成为任何现代 IT 组织的赌注。
