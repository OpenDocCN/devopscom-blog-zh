# 用于可观察性平台的 LogDNA 预览流模块

> 原文:[https://devo PS . com/log DNA-previews-streaming-module-for-observability-platform/](https://devops.com/logdna-previews-streaming-module-for-observability-platform/)

LogDNA 正在有限的基础上提供处理流数据到其 observability 平台的能力，作为给予 DevOps 团队对日志数据更多控制的努力的一部分。

LogDNA 首席执行官 Tucker Callaway 表示，LogDNA 已经提供的数据摄取管道可以摄取、解析和规范化大量的结构化和非结构化日志数据。 [LogDNA Streaming](https://www.businesswire.com/news/home/20210908005369/en/LogDNA-Unveils-Observability-Data-Streaming) 增加了向任何应用程序或分析工具自动发送日志数据的能力。

他补充说，这种功能还可以通过限制日志数据的副本数量来降低存储成本。

Callaway 指出，LogDNA 还希望 DevOps 团队也能够分析飞行中的数据，以此简化甚至在存储日志数据之前暴露问题的工作。

虽然有多种流数据平台可用，但 Callaway 表示，LogDNA 选择自己建立一个平台，以满足可观测性平台的要求。他说，最终目标是将可观察性从数据层面转移到实时提供更精细的见解。

Callaway 补充说，这种能力还将为广泛的自主过程奠定基础，这些过程将由通过 LogDNA 收集的可观测性数据驱动。

[可观察性](https://devops.com/?s=observability)一直是 DevOps 最佳实践的核心原则。大多数开发运维团队通常专注于某种形式的持续监控，以便更主动地管理应用环境。然而，发现问题的根本原因仍然需要几天甚至几周的时间。监控工具旨在使用预定义的指标来识别特定平台或应用程序何时在预期范围内运行。例如，被跟踪的度量通常集中在资源利用上。

可观察性将指标、日志和跟踪(一种特殊的日志记录形式)结合起来，以一种更简单的方式检测应用程序，从而解决问题，而不必仅仅依靠一组有限的预定义指标来监控特定的流程或功能。DevOps 团队可以更容易地使用查询来询问数据，从而更容易发现问题的根本原因。可观察性平台以某种方式关联事件，使得分析工具更容易识别指示 it 问题的异常行为。

平台提供商广泛采用机器学习算法来更快地分析机器数据，这只是时间问题。甚至可能有一天不再需要所谓的“作战室”会议，即今天需要召开的通过艰苦的排除过程来确定 IT 问题原因的会议。

与此同时，需要提高 IT 环境中应用程序的百分比来实现这一目标。幸运的是，由于开源代理软件的兴起，检测应用程序的成本开始下降。与此同时，DevOps 团队应该会看到传入数据量的显著增加，他们需要找到存储这些数据的地方，并确定这些数据应该保留的时间长度。
