# 新手错误:没有管理移动客户反馈

> 原文：<https://devops.com/rookie-mistake-not-managing-customer-feedback/>

有效地管理移动客户反馈可能是你的公司和客户最重要的成功因素。称之为上瘾或习惯，但每天我们都会遇到新的数据，证实移动应用程序的使用和与它们互动的总时间的增长。在“ [2015 美国移动应用报告](https://www.comscore.com/Insights/Presentations-and-Whitepapers/2015/The-2015-US-Mobile-App-Report)”中，CoreMetrics 团队分享了一些关于数字媒体消费增长和应用使用的有用数据指标。几个值得注意的主题或趋势是:

1.  **所有年龄段的美国人在移动应用上花费的时间都在增加:**到 2015 年年中，美国人平均每月在移动设备上花费大约 68 个小时。这一数字在千禧年一代中甚至上升到每月 90 多个小时。
2.  **高达 50%的时间花在一个应用上，大约 90%的时间花在排名前五的应用上:**不仅花在移动应用上的平均时间在增加，而且集中在几个有限的应用上。令人震惊的是，他们 50%的时间花在一个应用上，随着受欢迎程度、实用性和质量的下降，这个数字会迅速减少。

从这些数据中可以明显看出，你希望客户将你的应用放在他们的主屏幕上。这意味着不仅要有引人注目的实用程序，还要确保应用程序的最高质量。在本文中，我将说明如何通过在开发周期的每个阶段整合移动客户反馈循环，并在生产过程中让客户参与进来来实现这一点。在这个过程中，我们将研究各种开源和商业工具，这些工具增强了在持续交付管道的各个阶段收集反馈的能力。

大多数敏捷或精益软件组织都有一个自动化的持续集成和交付管道，帮助管理发布工件到生产的流程。虽然团队和企业之间可能存在细微的差异，但应用或服务的典型生产路径是通过所示的步骤来表示的:

## 持续交付和移动客户反馈

不久前，大多数负面用户反馈仅仅包括社交媒体平台上的糟糕评级、评论和咆哮。没有上下文、度量和数据，大多数开发团队只能猜测如何最好地重现和确定问题的程度。这通常会导致问题和特性以一种特别的方式被优先化。

今天，有许多开源和商业工具可以帮助解决反馈问题。其中大多数可分为以下几大类:

*   **仪器**
    尽管最近有报道称[脸书让其安卓应用](https://www.theverge.com/2016/1/4/10708590/facebook-google-android-app-crash-tests)崩溃以测试用户忠诚度，但大多数应用崩溃都不是故意的。有太多的崩溃测试和报告解决方案，最受欢迎的是[crash lytics](http://try.crashlytics.com/)(Twitter[Fabric](https://get.fabric.io/)的一部分)、 [Apteligent](https://www.apteligent.com/) (以前的 Crittercism)、TestFlight(苹果)和 [HockeyApp](http://hockeyapp.net/features/) (微软)。它们都提供了客户端 SDK 来捕获和报告崩溃报告，并提供了服务器端服务来管理和分析这些报告。
*   **分阶段发布——Alpha、beta、分阶段推出** Android 应用开发者可以利用作为 [Google Play](https://support.google.com/googleplay/android-developer/answer/3131213?hl=en) 一部分的 Alpha、Beta 和分阶段推出功能。对于α和β测试，不需要生产 APK。开发人员有许多选择来针对特定的组进行他们的 alpha 和 beta 测试。他们还可以利用应用程序中可用的反馈渠道来获得反馈。分阶段展示允许开发人员以分阶段的方式从 alpha 和 beta 测试进展到生产，不断收集反馈和指标，与基线(如果可用)进行比较，并根据满意的结果进展到 100%的生产。iOS 平台上的开发者可以利用 [TestFlight](https://developer.apple.com/testflight/) 提供的测试功能，并可以通过电子邮件邀请多达 2000 名测试人员来测试应用的预生产。
*   直接的应用内反馈
    电子邮件、信使或应用商店的评论提供表面反馈的日子已经一去不复返了。客户越来越希望通过应用程序进行交流，并在不离开应用程序的情况下提供反馈。与此同时，移动应用程序开发人员有更好的工具来扩展反馈并有效地管理它，以解决错误，优先考虑功能并获得上下文-截屏，系统信息等。—获取具体反馈。 [HelpStack](http://www.helpstack.io/) 是一个开源框架，使应用内的客户反馈变得相对容易，因此他们可以报告问题，附上屏幕截图，嵌入设备信息等等。客户反馈可以通过电子邮件报告，也可以提供帮助台集成。
    [appentive](http://www.apptentive.com/)、Crittercism、 [HelpShift](https://www.helpshift.com/) 、 [UserVoice](https://www.uservoice.com/) 和 [Parature](http://www.parature.com/about-parature/) 是一些著名的支持应用内反馈的商业解决方案。
*   **指标&性能** 当业务分析无法提供足够的答案时，深入研究技术数据可能会让你了解问题所在。值得跟踪的两个关键移动客户反馈指标是应用正常运行时间和性能。应用正常运行时间有助于识别面临崩溃的客户，而性能问题可能会影响参与率。本文中提到的一些工具确实提供了 APM 解决方案，可以监控这些关键指标。
*   **实验&分析** A/B 测试或多变量测试是一种数据驱动的实验方法，旨在促进应用功能的变化。虽然上述一些工具使您作为开发人员能够更好地倾听您的应用程序社区，但 A/B 分割测试允许您验证您的假设，并进一步投资于展示 lift 的特性和功能。
    Apptimize、Optimizely 和 [MixPanel](https://mixpanel.com/) 是少数几种提供多元测试功能的商业产品，这些功能可以在某些情况下无需修改代码或更新应用程序。它们还提供了将测试活动与基线进行比较的能力，并促进了展示积极反馈的功能。

## 在持续交付中纳入移动客户反馈

在整个持续交付渠道中利用数据驱动的工具增加了高质量和高效用的移动应用的机会。即使在不可能过早公开应用程序的情况下，也有可能“狗粮”你的应用程序——这个术语指的是积极利用自己的产品以测试它的团队或公司。例如，与开发人员提交相关的频繁构建使得 alpha 构建测试人员和其他团队成员能够下载并测试应用程序。同时，公司的其他成员也可以下载测试版。随着候选版本的出现，beta 社区可以扩展到包括应用程序社区的成员。

[![mobile customer feedback](img/a5994470b6949c250130326c37d7e599.png)](http://newstar-online.com/site/wp-content/uploads/2016/04/feedback.jpg)

因此，采取措施在开发周期的每一步整合移动客户反馈循环，并获得应用程序用户的大部分参与时间。我希望听到您和您的团队如何利用工具和技术来构建更好的应用程序，并不断听取客户反馈。
