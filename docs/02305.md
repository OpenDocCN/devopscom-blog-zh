# 价值流优化:将业务放在驾驶座上

> 原文:[https://devo PS . com/value-stream-optimization-puting-business-drivers-seat/](https://devops.com/value-stream-optimization-putting-business-drivers-seat/)

***价值流如何成为当今纯软件世界中 DevOps 成功的主要驱动力***

即使 DevOps 达到了接受的顶峰(截至 2018 年，只有 3%的公司没有采用 DevOps，也没有计划这样做)，许多组织仍在努力让高质量的软件走出国门。许多专家和顾问都将 DevOps 作为瀑布方法中常见的繁琐低效的解决方案。然而，正如许多运营活动中常见的那样，应用新的方法或技术不会自动解决根深蒂固的问题。

这一点在价值流管理中表现得最为明显(VSM)。这种关注于从过程中去除浪费以提高效率的方法，听起来像是解决软件交付挑战的灵丹妙药。实际上，它经常达不到目标。

这是为什么呢？价值流的概念没有错。将一个复杂的过程分解成几个组成部分，并优化每一个组成部分，从而优化整个流程，这是精益生产长期以来公认的支柱。在我看来，问题不在于 VSM，甚至也不在于德沃普斯。

问题是，在从瀑布迁移到 DevOps 和/或敏捷的过程中，许多组织形象地“把婴儿和洗澡水一起倒掉”他们不确定如何适应当团队从基本软件过程转移到现代过程时出现的极端加速，所以他们要么抛弃软件工程的重要元素，要么不完全适应它们。

## 为什么连续性很重要

举例来说，考虑测量和度量。由测量和度量支持的可追溯性和主动补救是软件开发的两个最重要的方面。然而，重要的不仅仅是开发和测试指标，还有业务指标——质量、用户接受度、安全性和易用性的指标。自从软件开发出现以来，这两组数据就有着千丝万缕的联系，但是一些现代软件团队似乎已经忘记了这个教训。

再多的过程改进度量也不能确保软件发布是安全的，或者用户会喜欢它。然而，如果重大软件升级出错，可能会对运营产生负面影响，导致销售势头停滞。(如果公司出售自己开发的软件，连锁反应就更强烈了。)更令人不安的是，在一次重大数据泄露后，一家公司可能会发现自己破产了。

不要误解我。我并不是说流量和其他 DevOps 指标不重要，因为它们非常重要。我的意思是，许多团队和他们的开发经理已经变得过于脱离业务度量，这可能会产生令人震惊的负面后果。

## 从那里到这里

在瀑布的日子里，开发工件、缺陷和其他数据被储存在一个非现场的位置，并且在每次发布之后(可能需要几天、几周甚至几个月)，团队将参与几天的数据提取。然后，一个单独的团队会尝试将开发工件映射到缺陷，并实现可追溯性。在 Excel 电子表格中进行数小时或数天的组织和交叉映射后，将为管理层和高管层生成报告。一旦涉众和管理层完成了他们的评估，团队就可以在下一个发布周期之前进行调整。

现在，我们的开发范例发生了如此巨大的变化，以至于许多团队没有给予[度量](https://devops.com/how-to-use-metrics-to-measure-the-success-of-your-agile-projects/)他们应得的东西。这是缺陷度量的一个问题，可能会影响用户体验或软件可靠性。当指标与安全性相关时，这是一个严重的、可能会终结游戏的问题。

## 完成价值流

为了[创建和管理](https://www.google.com/url?sa=t&rct=j&q=&esrc=s&source=web&cd=1&cad=rja&uact=8&ved=2ahUKEwj_6MO888XiAhWFrFkKHcutB0QQFjAAegQICRAB&url=https%3A%2F%2Fdevops.com%2Fthe-human-touch-the-key-to-optimizing-your-software-value-stream-network%2F&usg=AOvVaw1GAB9pY4oPGDDPM--a_O5G)完整的价值流，软件团队必须在传统的 DevOps 框架之外的三个领域得到支持，其中迭代开发活动是可追溯性度量的主要驱动力。有了这些元素，管理层可以将他们的软件团队的工作映射到*业务度量*，这是成功的最终度量:

*   组织必须有一个集成开发和测试工具的机制，以便缺陷度量可以实时地反馈给开发团队——与开发运维的速度相匹配。
*   仍然发生在开发和测试流程之外的关键软件活动，如变更单的生成和批准，也必须监控惰性，批准者要对拖延发布的书面工作负责。
*   安全缺陷必须与所有其他代码缺陷同等对待，并且优先于所有其他代码缺陷。所有影响用户端的缺陷，比如功能和启动速度，都是重要的业务度量。但是安全必须是最重要的。

如果您是一名开发团队成员，甚至是一名开发经理，您可能会认为业务度量和发生在您权限之外的相关活动对您不适用。我劝你再想想。软件已经成为许多公司运营成功的重要支柱，这种联系与日俱增。参与软件开发的每个人，从开发人员到运营副总裁，再到首席执行官，都必须从业务的角度来看待整个过程。
