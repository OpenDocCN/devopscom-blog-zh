# Radware 采用 API 来改进跨多个云的 AppSec

> 原文:[https://devo PS . com/radware-enchanges-APIs-to-improve-appsec-cross-multi-clouds/](https://devops.com/radware-embraces-apis-to-improve-appsec-across-multiple-clouds/)

Radware 本周推出了一种应用编程接口(API)方法来保护多个云，这种方法是从根本上设计的，是 DevSecOps 工作流的扩展。

Radware 安全部门的高级产品营销经理 Eyal Arazi 表示， [Radware SecurePath](https://www.globenewswire.com/news-release/2022/04/07/2418241/0/en/Radware-Introduces-Game-Changing-New-Way-to-Deploy-and-Consistently-Protect-Applications-Across-Any-Cloud.html) 使得通过 API 调用该公司的任何云安全产品成为可能。这消除了对需要管理单独控制台的内联安全平台的依赖。

他说，相反，Radware 实现的路径外方法使得跨多个云整合安全管理变得更简单。Arazi 指出, [API](https://devops.com/?s=API) 方法允许应用程序请求直接从客户端发送到应用服务器，降低了复杂性和延迟，消除了路由更改，提高了整体可用性。

![](../Images/5a79b3a0e948c060a70d870154ee877d.png)

Arazi 说，DevOps 团队现在可以访问整个 Radware 云安全产品组合，包括 Radware web 应用程序防火墙(WAF)、API 保护工具和一项通过一组一致的 API 阻止分布式拒绝服务(DDoS)攻击的服务。他补充说，Radware 将继续提供这些产品，以便在 IT 团队认为合适的时候进行内联或异地部署。

每当组织在另一个云平台上部署工作负载时，本已复杂的安全挑战就会变得更加复杂。如今，保护这些云的大部分责任由网络安全团队承担。然而，随着 DevOps 团队获得对工具和平台的访问，使得开发人员和 DevOps 团队以编程方式管理安全性变得更加可行，it 的更多责任正在进一步向左转移。

Arazi 指出，这种方法的部分好处是消除了在虚拟机上部署和管理防火墙的需要，或者通过内容交付网络(cdn)重新路由流量以检查流量的需要。他补充说，这也使 Radware 安全产品能够搭载在已经在云平台中使用的证书上。

从长远来看，目标是减少或彻底消除生产应用程序中常见的漏洞。组织经常面临的许多云安全问题可以追溯到开发人员犯下的错误配置。这并不是说开发人员故意开始构建和部署易受攻击的应用程序。相反，他们只是缺乏所需的专业知识，以确保，例如，港口没有开放。在缺乏专业知识的情况下，必须在开发运维工作流中嵌入更多工具，以更好地确保应用安全性。毕竟，大多数开发人员可能需要数年时间才能达到任何有意义的网络安全水平。

与此同时，DevOps 和网络安全专业人员之间需要更密切的合作。没有足够的网络安全专业人员参与应用程序开发项目的每个阶段。开发运维团队需要能够尽早识别和修复问题。否则，在最后一分钟发现的网络安全问题很可能会阻止在生产环境中部署工作负载。

在没有任何额外努力的情况下，网络犯罪分子变得更加擅长破坏云应用环境只是时间问题。
