# 调查显示越来越依赖 DORA 指标

> 原文:[https://devo PS . com/survey-shows-increased-reliance-on-Dora-metrics/](https://devops.com/survey-shows-increased-reliance-on-dora-metrics/)

一项新的[调查](https://www.prnewswire.com/news-releases/leanix-state-of-developer-experience-survey-reveals-devops-practice-falls-short-of-its-promise-301576552.html)发现，不到四分之一(23%)的受访者正在跟踪开发运维研究和评估团队(DORA)定义的所有四项开发运维指标，另有 17%的受访者正在跟踪三项指标。

开发人员体验状况调查调查了 129 名在软件开发中发挥作用的 IT 专业人员，由 LeanIX 进行，LeanIX 是一家管理企业应用程序构建的平台提供商。

LeanIX 产品管理和战略副总裁 Dominick Rose 表示，跟踪部署频率(77%)、故障率(73%)、变更前置时间(63%)和平均恢复时间(MTTR) (55%)等 DORA 指标(T1)的能力是 DevOps 成熟度的主要指标。

调查发现，由谷歌的一个部门定义和维护的 DORA metrics 被最频繁地用于推动敏捷规划(69%)和优化持续集成/持续交付(CI/CD)平台(58%)。

总体而言，调查显示不到一半的受访者(47%)表示他们的组织具有高水平的 DevOps 成熟度，定义为采用了三种或更多 DevOps 工作方法。这些工作方法是:灵活应对客户需求的变化；已实施 CI/CD 平台；所有工程师都构建、发布并拥有自己的代码；团队是围绕拓扑组织的，每个团队都可以自由选择自己的技术堆栈。

当然，每个单独的组织将为自己确定需要什么级别的开发运维深度。例如，并不是每个组织都认为需要围绕拓扑来组织团队，或者自由选择自己的技术堆栈。事实上，Rose 说，调查清楚地表明，较大的企业 it 组织往往具有较低的 DevOps 成熟度整体水平。Rose 指出，其中一个原因是许多大型组织仍然在使用遗留过程来构建和部署软件。

Rose 补充说，大多数开发人员在采用持续集成(CI)方面也比 IT 运营团队在采用持续交付(CD)方面走得更远。然而，他说，随着使用低代码和无代码工具开发的应用数量继续增加，应用交付需要变得更加自动化只是时间问题。罗斯补充说，随着更多的人工智能(AI)被纳入 IT 运营，实现这一目标将变得更加容易。

尽管如此，调查清楚地表明，在弥合开发运维团队与业务其余部分之间的鸿沟方面，仍有很长的路要走，只有 20%的受访者表示，他们使用价值流管理(VSM)平台来跟踪工程效率，以将开发活动与实际业务结果联系起来。

然而，总共有 71%的受访者将开放式支持票证作为衡量客户价值的标准，66%的受访者跟踪月度用户，45%的受访者跟踪功能采用情况。

DevOps 将如何在整个组织中进一步推广还有待观察，但随着越来越多的组织认识到他们对软件的依赖程度，随着应用程序的构建和部署速度不断加快，他们在多大程度上接受 DevOps 最佳实践就成了一个问题。
