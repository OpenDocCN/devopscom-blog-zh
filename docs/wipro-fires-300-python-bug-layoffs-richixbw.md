 *## 1.Wipro 对兼职者采取强硬措施

本周第一个问题是:你知道有谁在家偷偷做两份工作吗？DevOps-for-hire 公司 Wipro 正在反击这股潮流。

### 分析:很公平，如果为竞争对手工作的话

Wipro 正在打击那些兼职的人。如果一名员工也在为 HCL 工作，那么你可以**看到这一点**。

[Manish Singh](https://techcrunch.com/2022/09/21/it-services-group-wipro-fires-300-employees-for-moonlighting/ "read the full text") : **IT 服务集团 Wipro 解雇 300 名员工为竞争对手兼职**

Wipro 在 50 多个国家雇用了 25 万多名员工，该公司董事长里沙德·普雷姆吉在周三的一次会议上说，该公司发现为竞争对手兼职是“违反诚信的行为”。…没有人可以为 Wipro 和竞争对手 XYZ 工作。”随着公司采纳在家工作的规范，这种做法在全球范围内越来越流行。…从科技行业到银行业，越来越多的白领已经悄悄地从事第二职业…在家工作。

[u/accessible _ concept 36](https://www.reddit.com/r/overemployed/comments/xk4ew3/comment/ipc97o4/ "read the full text")几乎听起来让人羡慕:

被 Wipro 解雇可能是件好事。…这是一家维多利亚时代英国济贫院的印度版公司……不希望你为他们的竞争对手工作——比如 Infosys 或 HCL，它们同样糟糕——但他们绝对不会介意把你推给多个最终客户，只要他们能得到报酬，[即使]他们的最终客户是直接竞争对手，比如说如果他们同时让你与丰田和现代签订合同。我经常看到 HCL 和其他人的帖子，他们似乎做了所有让我生气的事情，我从来不想和他们一起工作。… 10 个人从该公司打电话/发电子邮件来询问完全相同的发布内容，但费率不同。

这不是什么新鲜事，T2 说:

这发生在 20 多年前:……我认识的一个兼职人员曾经接到一个电话，他的第二份工作的公司必须为他的第一份工作提供一些技术支持，尽管他不是领导，但他必须在电话中提供信息。他的老板们都在打电话，但他还是成功地做到了，以至于两个老板都没听懂。事实上，第一家公司的老板说他看起来很有研究。

## 2.Python Bug 挤 35 万项目

Python 中一个令人讨厌的 15 年之久的漏洞被重新发现，导致至少 35 万个项目陷入混乱。这个 bug 已经是旧闻了，但实际上从未被修复过— **只是被记录了下来**。

### 分析:软件供应链的薄弱环节

你分担组件安全的责任，你不能推卸。是时候意识到这种“供应链”类比是一派胡言了。叫它什么就是什么: ***代码重用*** 。

[约努特·伊拉斯库](https://www.bleepingcomputer.com/news/security/unpatched-15-year-old-python-bug-allows-code-execution-in-350k-projects/ "read the full text") : **15 岁的 Python bug 允许在 35 万个项目中执行代码**

**`Rediscovered the bug`**
2007 年披露并标记为 CVE-2007-4559 的安全问题从未得到修补，唯一提供的缓解措施是一个警告开发者风险的文档更新。…使用未清理的 tarfile.extract()函数或 tarfile.extractall()的内置默认值的代码存在路径遍历缺陷，使得攻击者能够覆盖任意文件。虽然没有关于这个漏洞被[利用]的报告，但它代表了软件供应链中的一个风险。…重新发现该漏洞的研究员卡西米尔·舒尔茨…发现易受 CVE-2007-4559 攻击的开源代码“跨越了大量行业”[他]估计有超过 350，000 个易受攻击的存储库，其中许多被机器学习工具(例如 GitHub Copilot)使用，帮助开发人员更快地完成项目。

他们在文档中“修复”了它？这个[匿名懦夫](https://forums.theregister.com/forum/all/2022/09/22/python_vulnerability_tarfile/#c_4535337 "read the full text")对 Python 团队非常愤怒:

“软件会对任何一个稍微不太勤奋的人做这种非常危险的事情，但这不是一个错误，因为它被记录在案”是一种几十年前就应该消失的态度。[现在]我们在 350，000 个单独的项目中有一个危险的安全风险，[即使]假设那些项目正在被积极地维护。
…
算一算。可悲的是，这种态度似乎没有下降的迹象，这就是为什么安全性需要保持不同于编码的纪律。

但这是 35 万消费者中每一个人的错，也是这 35 万消费者的错，lrvick 说:

我为几家高风险公司做软件供应链安全咨询，并且很大程度上同意…我们必须停止期望开发人员对他们生产的代码负责。责任在于消费它的人。
…
如果你的公司选择使用开源代码，而没有有能力的、有报酬的、全职的专业人员来审查它的安全性和质量，那么你的公司就要承担这个责任。如果你没有合理的尝试去检查你的供应链，结果伤害了用户，那么我认为你应该为疏忽负责。
……
这不应该有争议，但确实有。在医院洗手曾经也是有争议的。

## 3.忘记安静的辞职吧——下面是“精简裁员”

还记得上个月我问是否又回到了 1973 年吗？Meta、Google 和其他公司正在寻找新的裁员方法，而不使用 L-word。它包括强迫人们在内部申请新的工作，但是**给他们一个期限**。

### 分析:人员精简

当然，掩饰裁员的委婉说法并不新鲜。让它提醒你，没有人欠你一份工作。

杰夫·霍维茨、萨尔瓦多·罗德里格斯和迈尔斯·克鲁帕 : **梅塔和谷歌正在裁员。就是不提裁员**

**`Being pushed out on a regular basis`**
Meta Platforms Inc .…已经开始通过重组部门和给受影响的员工有限的申请其他职位的机会，悄悄地排挤掉大量员工。…这些举措是在 Meta 高管数周公开讨论“无情优先化”的必要性，同时避免使用裁员这个词之后出台的。在一些 Meta 员工中，在内部有限的时间内重新申请工作的过程被称为一种人力资源炼狱，他们称之为“30 天名单”…重新被聘用的努力正在进行中。“现实地说，公司里可能有一群人不应该在这里，”扎克伯格今年 6 月在公司的一次员工大会上说。(但)员工和经理们表示,(甚至)声誉良好、绩效考核优秀的员工也经常被排挤。

但是不要称之为“裁员”[tiger prish](https://tech.slashdot.org/comments.pl?sid=22095369&cid=62901739 "read the full text")提出低价竞标:

更新旧掘金的时间，例如:
o 调整规模
o 削减兵力(重新编制)
o 优化人员编制
o 人事调整
o 重组(重新组织)
o 重组

我跟不上。迈克尔·霍夫曼也不能:

时钟倒转得多快啊。各地的高管们一定在碰杯。仅在几个月内，我们就从改变了的工作现实和大辞职回到了“工作，农奴，或者被解雇，换成另一个农奴，他工作两倍的时间，拿一半的工资。”
