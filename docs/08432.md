# 为什么嵌入式分析成熟度框架至关重要

> 原文:[https://devo PS . com/why-an-embedded-analytics-maturity-framework-is-essential/](https://devops.com/why-an-embedded-analytics-maturity-framework-is-essential/)

[上下文分析](https://devops.com/turbocharge-enterprise-apps-with-contextual-analytics/) ，或者将分析直接嵌入到业务应用的交易环境和核心工作流中，是目前组织可用的[商业智能](https://devops.com/?s=business+intelligence) (BI)的最复杂阶段。它领先于传统的分析集成一个级别，传统的分析集成是一种独立的功能，以单独的仪表板或报告模块的形式呈现。

如今，组织越来越倾向于将交易分析嵌入到企业应用程序中。因为上下文分析是市场上最先进的选择，所以也越来越需要首先评估我们的分析准备情况。

上下文分析的根本区别在于它如何向用户提供分析，以在其交易环境中探索数据或获得洞察力，而不是作为外部工具访问。这种方法需要使用正式的框架对产品的分析成熟度进行更深入的评估。拥有这样一个框架可以作为您的开发人员在嵌入式 BI 实施的每个阶段的部署指南，包括上下文分析。

但是首先，是什么让上下文分析如此不同，以至于需要进行就绪性评估？

## 情境分析与嵌入式分析

有了上下文分析，仪表盘、报告和图表不再是分离的，而是主要用户体验的一部分。这种整体变化需要更仔细地观察您的数据和集成能力，以使它与您的软件的所有方面保持一致，包括现有的分析。

将这些元素直接嵌入核心软件工作流并不能取代任何现有的分析，而是朝着使数据和分析成为最终用户日常工作中更无缝的一部分迈出了又一大步。

应该强调的是，上下文分析适应不同层次的实现和业务需求。分析可以轻松嵌入，例如以用户在探索你的软件时可以看到的基本内嵌图表和图形的形式，或者更深入地嵌入，以提供人工智能辅助的见解和实时警报，从而在用户工作时触发他们的行动。它可以与经验丰富的 BI 供应商合作伙伴集成，如果您拥有内部技能，也可以与内部人员集成。

## 临时方法中可避免的陷阱

为了避免不必要的 BI 障碍，需要评估您的应用程序的分析成熟度以及将分析直接嵌入事务性工作流的准备情况。

对于许多组织来说，BI 和分析功能的优先级通常低于软件的其他方面。打算以后添加 BI 特性是善意的，也是可以理解的，但是让开发人员断断续续地添加特性，并让您的用户在需要时提出要求，会产生可见性问题。供应商通常看不到他们的应用程序目前能够分析什么，或者需要什么样的开发才能使新功能的添加不仅在技术上可以实现，而且值得时间和成本投资。

如果您花时间重新评估您的应用程序的分析能力，您的开发人员和领导可以更好地理解您的产品可能处于比以前认为的更早的准备阶段，并在寻求上下文或其他 BI 解决方案之前做好相应的准备。

例如，进行上下文分析可能意味着您发现了一个意想不到的旧仪表板模块，这是一个可以在正式成熟度评估中考虑的成本。

归结起来就是:实施基本的仪表板模块或数据导出功能，完全独立的产品功能，与用户核心工作流体验中的完全嵌入式分析有着根本的不同。寻求一个正式的评估来指导上下文分析的引入，甚至是一种早期的嵌入式分析形式，这意味着您的用户和产品可以获得您的企业今天已经准备好的最佳类型的分析能力。

加快情境分析的实施取决于您的技能组合，这将使您能够准确地识别和评估您的就绪状态，并确定 BI 解决方案的灵活性。如果您没有内部专业知识，在专业 BI 供应商的帮助下，实施上下文分析会更容易，这些供应商可以帮助您进行评估，并提供围绕开放性构建的上下文分析平台。我们建议您寻找具备以下条件的供应商和合作伙伴:

*   先进的 API 可降低总集成时间，并确保重新构建软件的需求降至零。

*   定制化的体验使你想要的上下文链接成为可能，比如当用户点击一个客户的表格时，向用户显示一个相关的嵌入式图表。
*   一个开放的平台，允许选择内部、云、多操作系统，保证您的分析的长期安全性、可扩展性和灵活性。

*   利用自动化流程和管理工具快速适应您的产品开发路线图，以降低总拥有成本

*   贴上白色标签，将分析解决方案的品牌重塑和融合与您的本地应用程序组件无缝结合。

最终，了解产品的准备情况并花时间检查应用程序的当前状态意味着您可以避免代价高昂的集成错误，节省时间并确保开发人员根据您的确切需求实施最佳解决方案。
