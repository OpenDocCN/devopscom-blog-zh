夏琳·奥汉隆:嘿，大家好。欢迎回到 *Techstrong TV* 。我是 Charlene O'Hanlon，现在和我在一起的是 Quali 的产品战略副总裁 David Williams。大卫，非常感谢你能加入我。很高兴见到你。

大卫·威廉姆斯:我也很高兴见到你，夏琳。很高兴来到这里。

奥汉隆:太好了，太好了。所以，这些天在 DevOps 的世界里发生了很多事情，但是我们听到的更多的是 DevOps 中的治理。特别是随着自动化的疯狂发展，加速了整个过程，很明显，事情很容易出问题。因此，我想听听你对 DevOps 中治理领域的看法，以及我们如何才能在治理方面摆脱羊群效应。

威廉姆斯:是的。这是 DevOps 上的一次旅行。当 DevOps 在很多很多年前作为一个概念被引入时，它立即与敏捷联系在一起，所以它变成了尽可能连续地、尽可能快地完成事情的唯一能力。所以当时的情况是，事情已经得到了很好的管理。是小团队在做大量的工作。随着企业开始涉足，开发运维成为向消费者交付应用的重要战略方式，参与其中的人数也在增加。

参与的团队数量不断增加。尽管 DevOps 的理念是小团队、快速移动、应用程序组件尽可能快地进入以太网，但它很久以前就不再是这样了，并开始变得更加主流。所以不幸的是，DevOps 成长了很多，其中一个问题是治理。从这个角度来看，治理实际上不仅仅是关于安全性——你知道，SOC 2 合规性，以及确保您的秘密在整个管道中得到有效保存和管理—

但也有人在身边，你怎么理解人们在做什么？他们之间的互动是什么？他们为什么会有这样的互动？基本上这是一个碎片化的挑战，比什么都重要。DevOps 管道是分散的，所以你如何在不抑制创新的情况下获得控制权？你如何获得可见性，有很多团队想在他们自己的领域创新，但不一定给你完整的端到端？这就是已经出现的挑战。

我认为在未来相当长的一段时间内，这种情况还会继续，因为在基础架构共享方面，应用程序变得越来越大，越来越基于争用。所以治理是关键。

奥汉隆:这很有意思，因为我认为这很像是一家公司在那里成长。当你有两三个人在一家公司工作时，如果不是每个人都全力以赴，很难取得很大的成就。但是随着公司的成长，你会有更多的人在做这项工作，但是也许他们不一定互相交流，而且不是每个人都有他们需要的所有信息。所以你必须建立一种技术，或者至少是一个过程，能够把信息传递出去，帮助每个人都在同一页上。

听起来，在 DevOps 领域，就更好地理解而言，同样的事情也在发生。不仅仅是对流程的理解，我想，从理解和沟通的角度来看，这是实现流程的一种更好的方式，同时确保所有正确的事情都在正确的时间完成。你们有什么提议？你认为什么能真正改变现状？

威廉姆斯:你用了两个词，当我说 DevOps 时，这两个词总能让我会心一笑。我曾经是一名分析师，与许多 DevOps 实践者交谈过，我曾经逃避的两个词是“过程；你知道，那是一回事。所以他们想要合作，但是他们想要能够合作而不受任何方式的控制。所以我认为这个过程词是一个，和“抽象”有关所以抽象不是一件好事。它让你远离核心。

因此，我要告诉你的是，任何能够让你更好地控制治理和理解的解决方案，都必须能够在不妨碍人们工作的情况下做到这一点。这应该是包含在他们正常呼吸中的东西。这不可能是另一项工作。这就推翻了精益实践的全部意义，任何与精益实践有关的事情都是关于去除东西的。我们是软件行业，所以我们让事情不那么复杂的方法就是添加一些东西，你知道；那是正常的倾向。

我认为 DevOps，当你真正使用它的时候，无论你添加什么，都必须包含在你的日常活动中。所以这不可能是另一件事。例如，安全性必须存在，但不能侵入。我之所以选择 Quali，是因为我曾在一些公司工作过很长时间，这些公司的基础设施、在云中调配资源的能力以及所有的事情都变得有点神秘。人们会认为基础设施。

因此，围绕您正在部署的内容以及您如何调配基础架构的监管不仅在您的管道中变得支离破碎，因为您计划了一些东西，然后您对其进行编码、构建、发布和测试，等等。每一个都是一个非常明确的领域。可能是一个人。可能会有很多团队。但是每一个都是你必须做的一件非常具体的事情。但是如果你能把测试包含到你的代码中，那不是很酷吗？如果你能在编码的时候消耗遵从性，那不是很酷吗？

因此，你正在编码，它会自动查看配置和修改，并适当地记录下来，而无需你添加另一层。我认为基础设施也是如此。我认为我以前工作过的公司真的不知道发生了什么，我工作过的 DevOps 公司试图解决，“好的，所以基础设施在那里；现在它不见了，”或者“它在那里，而且被修改了。”

你知道，程序员做的每一次提交都是一个时间点。"那么在那次提交和下一次提交之间发生了什么？"那么，基础设施的所有这些阶段，你如何构建呢？所以在 Quali，我们看到的是提供一种方法，让人们可以使用基础设施作为代码实践、配置管理工具。做你需要的，但它会自动归入一个模型。模型是你评价的标准。这就是应用访问等相关政策的原因。

但是它应该允许编码人员、发布管理人员、测试人员做他们需要做的事情，而且它还提供了一个类似于控制平面的东西，使你能够说话和被管理，但是没有老大哥类型的东西看着你。所以我认为 DevOps 的神奇之处在于，如果你添加一些东西，它不会影响速度。它不会提供另一个屏障。它必须做这些事情。这是一个很难解决的问题，但如果你以一种假设人们会做事情的方式来处理它，他们会打破东西——

人们那样做；他们就是这么做的。他们做自己的事。他们不喜欢别人告诉他们该做什么。所以，如果你能潜意识地提供对基础设施的控制，说“我知道你是怎么做的。我现在将提供我们在构建中使用的相同功能，以帮助测试人员了解要构建什么。”现在，您可以使用控制平面提供一些一致性。所以你在使用你的模型，你在移动它们。但是它包含了他们已经使用的东西，所以你可以添加进去。

您的基础架构堆栈和部署方式是在整合了策略、访问控制和机密的模型中实施的。这就是真正的理念，确保我们能够提供端到端的能力，加快交付生命周期，但不妨碍从业者的工作，而是为业务提供他们目前没有的可见性。所以我会继续说成本不是治理的事情，但是除非你治理它，并且你理解你为什么做某事，那么如果你不能把它和管道联系起来，你怎么把云的成本联系起来呢？

目前，成本与实例相关联，你必须将它们组合在一起。如果你能在做的时候为模型提供一个价值，你交付的应用程序，会怎么样？如果你能做到这一点，那么你就巧妙地将他们在从业者层面上所做的事情应用于一个应用程序，一个拥有者，一个成本。因此，这是治理实际上进入访问和控制的一种方式，而不是侵入性的。我们基本上可以在那里提供所有功能，并使企业能够更好地控制各种事情，包括第一次端到端的成本。

奥汉隆:是的，这很有道理。在我看来——如果我错了，请告诉我——但这也意味着开发人员真的不必考虑，我该怎么说呢？他们实际上不需要学习更多的东西——如果他们从一种常规的开发人员活动转变为更多的 DevOps 活动，这并不需要他们增加他们可能不需要的专业知识水平。这有点像为他们做的，有点像为他们填补空白，也有点像帮助其他组织或组织内的其他团队更好地理解流程，这很有意义。

我认为这也可能是治理对话的一部分，如果我们都更好地了解在任何特定时间发生的事情，我们就可以更好地管理它。

威廉姆斯:是的。我的意思是，秘密管理，人们听到了很多。当你和他们谈论秘密管理时，他们并不十分清楚这是什么。我的意思是，像密码或用户名这样的东西可能会在整个过程中使用，这是一个你不想暴露的共享知识。因此，您可以在整个环境中保持一致，但在整个管道中使用访问权限的所有人之间保持访问控制一致。我们学到的一件事是 DevOps 不是一个东西；是对一个事物的解读。

目标对人们来说是非常明确的，但是他们实现目标的方式可能会有所不同，这取决于组织、应用、设计等等。因此，我们还必须确保技能较低的人——我不是指基础设施方面的技能较低的人；基础架构可以进行云调配。但是，如果他们必须管理所有的网络组件和数据库实例，这些东西不是云资源调配环境所固有的，那该怎么办呢？

因此，我们必须确保不同技能水平的开发人员与基础设施互动时，我们可以为他们填补空白，这样他们就可以做他们需要做的事情，但是如果需要管理网络等，那么模型应该为他们提供护栏，使他们能够实施这些，以支持管道中该部分的应用需求。因此，这是我们正在努力做的事情，以确保我们能够将护栏安装到位。

奥汉隆:太好了。我想到了所有扩展了 DevOps 的事情，如果你愿意的话，超出了最初的参数，包括 DevSecOps 的实现和采用——

威廉姆斯:是的。

*奥汉隆:*——还有 DevNetOps，以及所有这些堆积在 DevOps 上面的东西。

威廉姆斯: *【咯咯地笑】*耶。

对于 DevSecOps 社区或 DevOps 社区，尤其是开发人员来说，管理所有这些知识可能会非常非常困难。因此，我想，有了这些防护栏，将会受到许多组织的欢迎，因为我们也在谈论这些天来开发人员社区人才的缺乏。

威廉姆斯:是的。

*O'Hanlon:* 因此，我认为，组织可以做的任何事情都有助于确保围绕他们的开发工具、实践和文化以及一切运行方式的治理水平，这是一个巨大的好处。David，非常感谢您向我详细介绍了 DevOps 和治理方面的情况。希望这个问题会随着我们的前进而变得更好。就你的观点而言，我们确实倾向于用技术来解决问题，并且希望这种技术能够坚持下去，如果你明白我的意思的话。

威廉姆斯:是的。

奥汉隆:再次感谢您的宝贵时间。我真的很感激。
