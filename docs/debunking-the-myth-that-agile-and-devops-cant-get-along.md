# 敏捷和 DevOps 能不能合得来？

> 原文：<https://devops.com/debunking-the-myth-that-agile-and-devops-cant-get-along/>

有一种普遍的误解，认为敏捷和 DevOps 不能一起工作，这对软件开发和交付领域的每个人都不利。事实上，这两种方法都可以显著增加透明性、检查和适应的 Scrum 支柱，从而更好地使产品团队开发、交付和维持复杂的产品。敏捷和 DevOps 都体现了伟大的思想和热情，可以增加团队工作的价值，确保他们的组织在一个业务既被软件驱动又被软件破坏的世界中取得成功。

为了让企业充分利用敏捷和 DevOps 带来的机会，两个社区——敏捷专家和 DevOps 专家——都需要纠正这两种方法之间的误解，这些误解扩大了它们之间的差异。他们必须理解为什么这些神话会渗透进来，并学习如何克服这些障碍，这样敏捷和开发人员才能最终在沙盒中玩得更好。

# **敏捷与开发:误解** 

敏捷和开发运维都追求以更有效的方式向客户交付价值的愿望，但是出发点不同——敏捷来自开发人员，而开发运维来自运营部门。他们的侧重点也不同，导致人们认为他们是不同的东西:敏捷强调团队互动、文化和价值观，而 DevOps 强调交付管道和流程。

这些明显的差异被误解进一步放大，误解只会让这两种方法相互对立:

*   一些敏捷论者认为所有的过程都是不好的，而 DevOps 沉浸在通过过程保持稳定的历史中。
*   DevOps 使用工具和自动化来提高速度，而一些敏捷人员认为工具在最好的情况下是不可避免的，在最坏的情况下是分散注意力的。
*   DevOps 使用整体系统来解决问题，而敏捷使用人来推动变化，并通过工作来发现变化。

这些误解的存在令人沮丧，因为敏捷和 DevOps 的共同点比大多数人意识到的要多。例如，两者都包括更快的交付周期、更小的增量或发布，并使用反馈来改进和消除浪费和障碍。这两种方法还可以影响组织中的文化转变，促进易于管理的小变化，以便随着时间的推移带来更大、更好的业务成果。

如果它们的相似之处多于不同之处，那么到底是什么阻碍了这两种方法的发展呢？

## **政治、组织筒仓扩大了差距**

组织变化，如新员工和管理人员变动等。，吸引个性鲜明的人，他们因推动变革而获得认可，并在变革产生成果时获得奖励。当他们的成功与其敏捷或 DevOps 项目的成功联系在一起时，相互竞争的政治目标被认为是一种威胁，并导致混乱:

*   **不同的工具策略:**你听过多少次“敏捷团队使用 JIRA，但是 DevOps 团队现在使用服务”？这些工具和销售这些工具的供应商关注的是推动购买的变革计划。这使得集成敏捷和开发运维变得更加困难，因为它们的数据和流程隐藏在不同的工具中。
*   **政治而非价值观来做决策:**俗话说，“重要的不是你知道什么，而是你认识谁”，这句话不仅在各国首都适用，在任何组织中都适用。政治是大多数组织的现实，这意味着相关的计划不仅带有工作的包袱，还带有谁在推动它们的政治。
*   具有讽刺意味的是，DevOps 和敏捷运动要求不同团队之间的统一和协作。然而，在实践中，由于他们不同的术语和想法，他们有时会导致传统团体内部更深的分歧。

## 不同的文化

敏捷论者认为过程是对形势需要的“紧急”反应。这意味着团队可以通过一个轻量级框架(比如 Scrum)来决定他们如何工作，然后添加他们认为需要的任何其他过程来实现更多的价值和更快的反馈。

DevOps 传统上采取了不同的策略，专注于用流程模型和价值流进行系统思考。

尽管存在文化差异，敏捷和开发人员需要一致的语言和概念来一起工作，否则沟通会中断，协作会受到影响。这导致利益相关者疏远，恐惧和防御行为增加，产品和服务质量差，缺乏透明度和可见性，更痛苦的是，从来没有完成任何事情。

## Scrum 打破了敏捷和开发的壁垒

通过软件向客户交付价值需要来自敏捷和开发运维的元素在一个全面的流程中结合起来，这个流程是流动的，并使团队能够以动态的方式工作。通过 Scrum，授权跨职能的、面向业务的团队持续地交付软件，工作在所有合适的涉众之间实时地无缝流动。

然而，仅仅在您的组织中拥有跨职能团队并不足以确保所有相关人员以整体的方式工作来交付客户价值。组织需要对谁应该加入或退出团队做出正确的决定，因为不同的技术需要专业技能和基础设施来支持它。

正如 [Scrum 指南](https://www.scrum.org/resources/scrum-guide?gclid=CjwKCAiAoqXQBRA8EiwAIIOWsjp3hJAWgu1mFfIA0x_NScM4jOMEn3P9L7TwbGup7uBVjwYezO2iHRoCKbcQAvD_BwE)中所描述的，一个跨职能团队必须具备完成工作所需的所有能力，而不依赖于团队之外的其他人来交付工作产品。

随着技术、市场和环境的复杂性及其相互作用的迅速增加，Scrum 在处理复杂性方面的效用每天都在被证明——Scrum 团队具有高度的灵活性和适应性，因此他们可以交付尽可能高价值的产品。当误解被放在一边，人们学会了相互交流，由敏捷人员和开发人员组成的跨职能团队拥有所有合适的技能，可以为您的组织更快地交付完成的软件。

— [戴夫·韦斯特](https://devops.com/author/famapr/)
