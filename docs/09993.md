# 在一个提交者动机混杂的世界中保护开源组件

> 原文:[https://devo PS . com/securing-open-source-components-in-a-the-world-of-mixed-committer-motivations/](https://devops.com/securing-open-source-components-in-a-world-of-mixed-committer-motivations/)

我们的世界运行在包含开源组件的软件上。这给作为这些组件的主要消费者和部署者的开发人员增加了负担，他们需要使用完全最新且安全的代码。大多数对项目存储库有写权限的开源提交者仍然专注于维护和积极改进代码，同时也帮助整个 [开源社区](https://cacm.acm.org/magazines/2022/5/260353-it-takes-a-community/abstract) 受益。然而，许多开源项目的复杂性为不同的、更阴险的和协同的攻击类型打开了大门，这些攻击可能会造成广泛和严重的伤害。

### 开发人员必须快速行动，重用给了他们这种速度

一个复杂的因素是，通过两种主要类型的依赖，第三方开源元素在大多数开源项目中的重用非常普遍。对于直接依赖，开源组件直接链接到第三方代码库，而传递依赖发生在组件调用库时，而库又与另一个依赖进行通信。

依赖关系使开发人员能够在不断增长的快速交付新软件的公司压力下，通过在同行现有工作的基础上进行构建而更快地前进。不利的一面是，这些依赖关系基础容易受到攻击，有些库可能在一个提交者的支持下，对他来说，代码收集可能是一个附带的项目，可能没有得到定期的关注。

sonatype[research](https://www.sonatype.com/state-of-the-software-supply-chain/open-source-supply-demand-security)和其他组织的研究告诉我们，使用 [Java 应用程序](https://devops.com/?s=Java+applications)的开发人员每年可能要处理惊人的 1500 个依赖关系变更，平均每个 Java 应用程序包含 148 个依赖关系，平均每个 Java 项目每年要更新 10 次。虽然开源项目的数量没有激增，但是这些项目的版本数量却在激增。当今影响开源项目的大多数漏洞都是由传递性依赖引起的，传递性依赖是一种更复杂的依赖类型。

在一个领域中，可能存在多个有效地解决相同问题的项目，开发人员需要知道哪些项目正在被积极地维护，哪些项目没有得到很好的培育或者不再运行。另一方面，成功的开源项目可能有数十或数百个提交者拥有写存储库访问权限，这些提交者解决一组完全不同的问题，包括就漏洞由什么组成以及如何修复或修补漏洞达成一致。

### 在一个好的、坏的演员和自毁事件出现的社区中

个人选择致力于开源软件的原因是多方面的；一种强烈的愿望来回馈社区，一种热情的，也许是非常专一的爱好，一个开发人员在工作中的角色和责任的一个不可或缺的和有报酬的部分。几年前，恶意开源提交者的概念还是一个抽象的理论。当时的想法是，在一个拥有数百万开发人员的社区中，可能存在少数“害群之马”。然而，自 2019 年以来，研究人员和其他组织的工具已经识别出超过 100，000 个恶意、可疑或概念验证包，表明网络犯罪分子的行为程度要高得多。

坏人故意利用开源软件的复杂性，故意引入可利用的漏洞。这些攻击变得越来越复杂，尤其是在将恶意代码注入开源库的源代码方面。这些攻击通常表现为无害的代码更改，并在传递恶意软件有效载荷之前保持隐藏状态。最近的一个例子是一次加密盗窃，盗窃了超过 300 万美元的加密货币，该加密货币利用了对私有 GitHub 存储库的入侵访问。

在过去的一年中，代码维护者选择自我破坏他们的开源项目，以引起人们对某个原因的注意，无论是地缘政治还是特定于个人的原因，这已经成为一个问题。在这种情况下，尽管声明的目标不是恶意的黑客攻击，但是对于使用该代码的开发人员来说，就重大破坏而言，可能会发生真正的伤害。

为了应对不良行为者和自我破坏对开源组件的影响，开发人员需要利用发布完整性特性来帮助保护他们自己和他们的组织免受伤害。他们必须能够完全了解和控制他们构建的软件以及作为开发工作一部分的开源组件。与此同时，包括美国在内的世界各地的政府都在考虑与软件供应链安全相关的问题，因此，增加监管和处罚也应该成为开发商及其组织未来行为的一个因素。

我们距离下一次潜在的 [Log4j](https://devops.com/?s=Log4j) 攻击只有一步之遥，每个开发人员和组织都需要提高警惕，准备好快速识别和修复漏洞。数据的收集和共享将有助于提高每个人的安全性。
