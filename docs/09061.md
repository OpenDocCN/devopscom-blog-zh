# 不可调和的分歧:DevOps 努力解决问题

> 原文:[https://devops.com/the-top-5-contentious-devops-issues/](https://devops.com/the-top-5-contentious-devops-issues/)

DevOps 问题可能是一个棘手的话题。开发人员(Dev)和运营人员(Ops)和谐地走到一起代表着某种包办婚姻，对一些人来说，这种婚姻永远不会有“从此幸福快乐”的结局，因为双方最终都要努力解决不可避免的分歧。

尽管有反对者，DevOps 作为一个组合体的存在(也许还有 DevSecOps 这样的姐妹品牌，以形成一个安全三明治)在许多方面都证明了在多年来经常相互不和的两个派别之间建立更紧密团结与和谐的纽带的必要性。

即使 DevOps 在过去十年里得到了普及、推广和民主化，也不是每个 T2 人都会相信它的价值和/或有效性——有些人甚至怀疑它的存在。有人说，如果你看看 IBM 在 25 年前倡导的软件开发实践，DevOps 已经存在，但没有正式的标签。

我们 可以确定的一件事是 [DevOps 几乎总是会抛出一些实质性的争论点](https://devops.com/mongodb-addresses-host-of-devops-issues/)。让我们更仔细地看看这些问题，希望我们可以利用一些焦虑，并将其转化为积极的能量，创造一个丰富多彩的未来。如果我们要 分析这个领域中最有争议、最敏感的 DevOps 问题，我们应该注意什么，以及这些大问题是如何分解的？

## 它是一只鸟吗？它是一架飞机吗？

可以说，DevOps 面临的最大问题是分类。有人称之为软件应用开发的方法论 。“经典”软件方法的支持者不同意这种观点，因为对于纯粹主义者来说，软件方法只能是敏捷的、瀑布式的、极端的、快速的、联合的或 Scrum 的。

这些语义意味着许多人更喜欢将员工志愿服务归类为一种文化、一种工作伦理、一种技术、一种文化和/或一种思维方式，或者用最模糊的术语来说，一种方法。要想在 DevOps 上取得真正的成功，你必须采取整体的方法，从头到尾地审视整个开发周期 。密切评估进入开发的过程，以找到生产力的瓶颈和障碍。通过找到当前流程中不足的领域，你将能够确定在实施强有力的发展战略时需要强调的领域。

## 云原生 DevOps

自从千禧年以来，任何没有住在山洞里的人都会知道，云计算已经成长起来，偶尔会步履蹒跚，最终会发展成为一个新的标准晴雨表 f 或所有组织走向所谓的数字化转型的旅程。

这是个好消息。不太好的消息是，从初始部署 到持续维护、管理和报告，云(以及当前的云原生化趋势)仍然很复杂。

将 DevOps 带入云原生的世界最终会有回报，但首先，随着团队习惯于 alto 的新形状、形式和功能，以及更抽象和虚拟化的软件工具集，将会面临挑战。

正确处理云原生开发的关键是采用谨慎的方法进行监控和观察。随着以云为中心的开发运维团队获取他们需要执行可靠的 和可靠的系统[可观测性](https://devops.com/?s=observability)的[日志、指标和跟踪](https://devops.com/?s=logs+metrics+traces)，他们将需要一套一致的工具来长期监控和观测系统。在本练习中，记住监控向您显示与系统性能相关的较低级别的粒度信息也很重要。可观察性包括从监控中收集的所有信息，还包括更高级的工具，其中一些将是 DevOps 从业者使用的数据可视化。当两者结合用于云原生 DevO ps 时，可以优化、保持和维护系统健康，同时在较低级别执行任何调试或根本原因分析。

## 应用循环中的机器

随着机器对机器事件现在在物联网(IoT)中更加普遍地发生，捕获、集成和分析在人类循环之外发生的事件流，并将这些日志指标带回更接近“人类”软件工程领域的需求不是即插即用的快速解决方案。

让机器 t 很好地发挥作用，成为扩展和进步的开发团队的一部分应该是理所当然的。毕竟他们只是机器。在现实中，将需要特定的技能，并且人类 DevOps 人员将需要习惯于他们以前没有考虑过的更新和其他警报 。

## AIOps 和自动化限制

关于机器事件的最后一点，[我们需要考虑 AIOps](https://www.gartner.com/smarterwithgartner/how-to-get-started-with-aiops) 。当我们构建 AI 驱动的 DevOps 和创建 AIOps 时，我们需要停下来，问一问我们在哪里对流程自动化设置了限制，自动化的更广泛元素在哪里碰到了 限制器。为了知道我们应该在多大程度上建立我们的 AIOps 实践，必须进行详尽的审计和分析；这里的整个技术主张不是简单的点击事件。即使我们确实建立了可行的 AIOps，也要知道会有的边界。

除此之外，当人工智能引擎不知道下一步该做什么，并且需要额外的数据和培训时，一些 AIOps 实施将需要我们所说的“人工移交”支持——只有到那时，我们才能开始意识到AI ops 本身并不是简单的即插即用事件。

## 提升和移动不缩放

今天，在太多现实世界的 IT 商店中，一个不幸的事实是，在使用云的地方，它的使用很差。太多的技术被毫不客气地 提升并转移到组织采购的云实例中。这使得以云为中心的开发运维更加难以实现。但是为什么呢？

因为在陆地(即非云)编程中发生的事情是 个人跨职能软件团队的开发。这种基于筒仓的方法的出现是由于公司的遗留结构、公司管理系统，有时也是出于个人、地域或文化的原因。当不同的团队都在重新发明 ng 轮子(运行诊断测试，建立 CI/CD 管道，交付升级等等)时，所有人都应该共享和访问，自然会出现低效率。

## DevOps 问题:结论

这是近期最有争议的五个 DevOps 问题 ，当然，找到更多并进行分类并不困难。在未来，我们甚至可能需要争论哪些开发人员应该被归类为核心开发人员，哪些运营部门的核心成员不应该包括在内。

对于软件开发人员、软件架构师——实际上，是所有从事系统管理、数据库管理、测试、安全和所有其他形式的操作的软件工程专业人员——来说，有相当多的风险和相当多的 要争夺。如果你问我，这就是 DevOps 的伟大之处。
