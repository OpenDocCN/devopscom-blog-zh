# DevOps 工程师存在吗？信念如何影响招聘

> 原文：<https://devops.com/does-devops-engineer-exist-how-beliefs-can-influence-hiring/>

DevOps 工程师真的存在吗？对于一个专门从事 DevOps 招聘的人来说，这是一个有风险的话题。对于外行人来说，简单的答案是，是的。然而，对于所有所谓的 DevOps 工程师来说，答案并不那么简单。

虽然围绕这一点的有效立场各不相同，但在雇用 DevOps 工程师时，有一些事情需要考虑，以及你的心态对决定你选择谁的影响。

DevOps 不仅仅是一套工具，而是一种将人们聚集在一起的哲学(提示“Kumbaya”的两个章节)，并构建一套促进个人所有权的思维。首先，让人们意识到个人行为的后果，从而产生创造成功产品的共同责任感。

因此，仅仅雇佣一个或多个 DevOps 工程师是远远不够的。当你认为这些雇佣是对你缺乏 DevOps 的治疗时，你无意中雇佣了与 DevOps 目标完全相反的人。DevOps 是一种思维模式，而不是一套线性工作流程。

这是一种必须在跨职能团队中流动的方法。它从您的 DevOps 工程师开始，这是肯定的，但它并没有在那里结束，因为目标是多个团队朝着同一个最终目标有效地协作。通过将您的 DevOps 工程师隔离在一个“DevOps 部门”，您巩固了 DevOps 最初创建时要消除的隔离效应。

## 我们能期待什么样的变化？

现在，你已经通过值得信赖的 DevOps 招聘人员雇佣了一名 DevOps 工程师，下一步呢？我们知道，仅仅雇佣一些具有 DevOps 技能的人是不行的，雇佣更多的人也不行。我们的目标不是雇佣有使用华丽工具经验的人，而是雇佣有技术专长和足够人际交往能力的人，以传播合作态度。你正在寻求重组团队协作方式的帮助，现在每个人都对最终产品负责。

当一群人分担这个责任时，一种健康的推诿和质疑的文化就形成了。当然，每个人仍然专注于他们的专业；然而，重组每个人对持续改进的优先顺序带来了一个新的共享目标，这对于几乎所有以前没有经历过的相关方来说通常是新的。

无论你是开发人员、管理员、QA 甚至是经理，这种健康的提问有助于团队更好地理解每个人所贡献的价值。人们开始通过更具批判性的视角来分析自己的工作，然后在分析同事的工作时也这样做(并不总是让每个人立即高兴)。曾经独立的部门现在可以真正了解每个小组的贡献。

人们开始质疑他们的同事做了多少工作，而更多的是他们自己的行为如何影响共同的最终目标，这在引入 DevOps 思维之前很少存在。

## 面试合适的人选

那么，当他们拥护你公司的新工作理念时，你真的能把他们简单地归类为 DevOps 工程师吗？

DevOps 雇佣带来的价值并不总是有形的，但有助于塑造公司，并使同事的优先事项朝着同一方向发展。它始于对流程的思考，并将每个人聚集到更大的利益上。

雇佣专业技术人员要容易得多。更大的挑战是让每个人都认同 DevOps 的整个理念。当然，也许 C 级管理层是被收买的，但是你如何让你的终身员工接受这样一个看似彻底的改变呢？在寻找合适的 DevOps 工程师的过程中，有效的面试流程可以发挥至关重要的作用。

考虑到 DevOps 工程师将这种变化带给技术和非技术团队的沟通能力，角色扮演法可能会有所帮助，在角色扮演法中，你的候选人向非技术团队成员解释 DevOps。当你能更有效地看到他们如何使 DevOps 的想法对你公司的每个人都切实可行时，仅仅拷问某人的技术敏锐度是不够的。

如果他们只能与同行的技术专家交流，他们可能不是合适的人选。然而，如果他们能用通俗易懂的语言向其他员工介绍 DevOps 的效果，那么他们肯定是适合这份工作的。无论如何，没有什么比现实世界中的情况更能真实地成为他们工作后的首要任务之一了。

## 那么，下一步是什么？

同样，这不仅仅是关于一个工具集和是否有人利用了 Kubernetes 或 AWS。这是关于雇佣能够重组团队和实施哲学的人。不存在购买 DevOps 或雇人处理您的“DevOps”这样的事情——从设定相互目标和创造集体责任的动态，DevOps 工程师的影响将远远超过利用 Jenkins，尤其是如果这是您的公司第一次来到 devo PS 街区。

所以，在角色扮演和相信你对一个人的直觉之后，你能真正称之为工程雇佣吗？有些人可能仍然认为是；然而，这个人所做的远不止是带来技术知识。他们已经看到并帮助公司进行数字化转型，也可以为您做同样的事情。最好的开始方式是知道除了简历上光鲜亮丽的技能矩阵之外，你在寻找什么。
