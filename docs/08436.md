# 如何为 DevSecOps 准备 DevOps 环境

> 原文:[https://devo PS . com/how-to-prepare-your-devo PS-environment-for-devsecops/](https://devops.com/how-to-prepare-your-devops-environment-for-devsecops/)

新冠肺炎疫情在工作环境中产生了深刻的、不可逆转的变化，因为大多数组织都在用完全[远程或混合工作](https://devops.com/?s=remote+work)的组合来重新构想这种“新常态”。DevOps 实践越来越正常化，并且越来越依赖于它来响应现代数字化公司的需求。从疫情出来后，组织现在更好地理解了高效部署软件应用的需求，这是在这个新的工作世界中取得成功所必需的。

这并不奇怪，因为组织已经意识到 DevOps 帮助产品团队快速交付高质量代码的价值。DevOps 使他们能够更敏捷、更快速地响应不断变化的市场和业务需求。为了保持软件交付的速度，产品团队正在将各种工具集成到他们的 CI/CD 管道中，并最大限度地减少管道外的工具引用和验证。

随着这些 DevOps 实践的变化率的增加，自动化企业安全控制的合规性变得比以往任何时候都更加重要，尤其是考虑到代码和配置中的安全漏洞会带来可怕的业务风险。要使开发运维真正取得成功，组织需要实现安全流程的现代化和自动化，使其成为开发运维构建和交付渠道中不可或缺的一部分，从而为开发运维准备好整体开发运维环境。

但是，成功的 DevSecOps 部署在很大程度上取决于组织当前 DevOps 环境和安全操作工作流的成熟度。这包括优先考虑自动化、测量和协作的核心原则，因为这些原则也推动积极的安全成果。参考这些原则，以下是组织在评估如何完善其 DevOps 生态系统和安全运营流程以便为 DevSecOps 做好准备时应考虑采用的四种最佳实践。

### 在开发人员工作流程和治理流程中嵌入安全工具

旅程的第一步是为开发团队优先考虑“安全第一”的心态和相关技能。安全不应该是事后的想法，开发团队也不应该依赖安全团队在生命周期的后期检查漏洞。相反，安全专家需要成为开发阶段不可或缺的一部分，以便代码从一开始就考虑到安全性。

这可以通过将安全技能的责任转移到开发团队中，或者通过让安全团队在生命周期的早期扮演顾问角色来实现。然后，开发团队可以自动执行必要的安全漏洞检查，如静态应用程序安全测试(SAST)、软件组合分析(SCA)和动态应用程序安全测试(DAST)。但更重要的是，在开发和安全团队之间共享见解和简化沟通对于确保策略有效并在整个组织中得到一致应用至关重要。

### 在管道之外继续自动化

Puppet 的 [2020 年 DevOps 状态报告](https://puppet.com/resources/report/2020-state-of-devops-report/)显示，通过改善治理流程(如变更管理)、准备企业基础设施和应用程序以应对组织变革，组织已经使其 DevOps 环境变得成熟。事实上，他们发现，拥有自动化变更管理和治理流程的组织在 DevOps 转换中的表现比竞争对手高出近三倍。

这是因为自动化改进了变更管理过程，这反过来消除了将治理应用于开发运维的障碍。这也得到了 Puppet 的调查的支持，该调查指出，自动化使组织更加相信他们的变更管理解决方案是有效的。一个全面的、自动化的变更管理过程通过在允许代码或配置变更进入生产之前将基于策略的规则应用于代码或配置变更来灌输治理。它通过利用从管道收集的数据来实时完成这项工作。正确应用时，DevSecOps 工具可防止漏洞，自动变更管理会进行“最后一英里”检查，以确保运行正确的扫描并遵守合规性策略。

### 提供跨开发运维的端到端可追溯性

产品团队——以及所有参与应用程序交付的团队——应该有一个可访问的审计跟踪，提供对安全检查、用户、代码和配置更改、代码质量、测试等的可见性。这清楚地描述了软件从构思到生产的过程。在一个理想的世界中，产品团队包括操作和安全人员，但是这些功能在大型组织中通常是孤立的。

公司可以打破这些孤岛，通过集中可见性使团队更容易协作。提供可见性和集中的安全策略使得团队更容易有效地操作软件的交付过程，因为他们被提供了单一的事实来源。此外，用于收集审计数据的自动化系统提高了审计信息的一致性和可靠性。从整个管道中自动收集数据，消除了开发人员输入信息的需要，降低了数据异常的可能性，从而节省了宝贵的时间和资源。

### 支持跨企业协作，解决开发运维管道中的安全问题

将人工智能和机器学习应用于 DevSecOps 流程有助于组织在安全问题出现在生产中之前快速识别和修复它们，从而增强跨产品团队的主动和可操作的协作和协调。

正如 Puppet 在其 [2019 年 DevOps 状态报告](https://www2.circleci.com/2019-state-of-devops-report.html?utm_source=gnb&utm_medium=SEM&utm_campaign=SEM-gnb-stateOfDevopsReport-Ld-uscan&utm_content=SEM-gnb-stateOfDevopsReport-Ld-uscan-StateOfDevops&utm_term=SoDReport&gclid=CjwKCAiAm-2BBhANEiwAe7eyFNSqa3hRVVaDAmjPtJBMyV9eGlryqayDpmgrO0K43Xf14fbamJEkTxoCY-QQAvD_BwE)中所引用的，共享和协作是团队可以优先考虑的两个最有影响力的事情，以改善 DevOps 计划，包括 DevSecOps，这在今天仍然适用。2020 年的调查结果表明，还有多少组织正在达到 DevOps 成熟度的这一顶点。

最终，我们已经看到了操作活动——如检查软件的安全漏洞——如何转移到 DevOps 管道中，并被作为“代码”处理但组织应该更进一步，在 DevOps 部署中牢记可靠性、可预测性、可测量性和可观察性等因素，以培养更安全的环境和管道。
