# 商业银行业务中的开发者挑战

> 原文:[https://devo PS . com/the-state-of-devo PS-in-banking-and-finance/](https://devops.com/the-state-of-devops-in-banking-and-finance/)

与许多其他业务部门一样，银行业也不得不因为新冠肺炎疫情而加快数字化转型的步伐。

今天的零售银行业与五年前相比已经大相径庭。消费者不再需要亲自去银行处理支付或开立新账户。移动和网络应用程序具有直观的体验和复杂的功能。总的来说，零售银行和他们的软件开发团队在创造工具方面做了大量的工作，这些工具是今天个人银行业务不可或缺的一部分。

然而，尽管零售银行业务在过去几年中取得了重大进展，但商业银行部门却落在了后面。对于零售银行体验领域的所有创新，商业银行领域仍像以往一样支离破碎且“缺乏技术含量”。对于开发人员来说，这意味着他们有自己的工作要做。

如果您是一名希望深入研究银行领域的开发人员，或者如果您刚刚开始从事商业银行产品的开发工作，请记住以下几点。

## **准备好使用过时的技术**

开发人员将面临的最大挑战之一是，商业银行往往依赖于几十年前的遗留基础设施。

随着银行的发展，它们的 IT 团队、供应商和顾问通常会在核心银行系统之上和之外构建一些层。问题是，在许多情况下，这些层是在身份认证委托、微服务和事件驱动架构过于新颖或不存在的情况下设计和实施的。

因此，开发人员需要满足银行在技术方面的要求。例如，他们可能会看到文件而不是 API，模糊的 XML 模式(更不用说 JSON 了)，意外的节流和批处理，以及总体上缺乏更高级的工具，如 webhooks、事件、流和报告。

## **耐心是关键**

虽然在 IT 行业的其他地方，速度是王道，但许多大型组织(如商业银行)的运营速度往往要慢得多。这不仅仅是使用一个“较慢”的方法或者有一个“较慢”的开发过程和较长的迭代。与初创公司不同，大型组织内的开发流程是更大业务流程的一部分，而银行的业务流程需要更多的时间和精力，并受到更严格的审查。开发人员应该预料到一个较慢的节奏，并且必须理解耐心的必要性。

## **拥抱未知并(负责任地)尝试**

银行、金融和会计是非常复杂的领域。此外，银行是大型复杂的组织，大多数开发人员通常不熟悉其内部工作方式。因此，开发人员可能害怕犯错误，选择安全性而不是创造性。这是绝对不能允许发生的。  当然，没有人喜欢犯错。然而，错误往往是[创新](https://devops.com/?s=innovation)的关键组成部分。与其让未知的特技创新，开发者应该拥抱这个不熟悉的领域，把它作为一个实验的机会。当然，需要有适当的质量控制步骤，以确保这些实验不会造成混乱或使关键功能偏离轨道。

这里有一些最佳实践，每个角色都可以遵循，以在实验和目标实现之间取得适当的平衡:

*   **工程经理:**
    *   建立一种文化，鼓励编写干净的代码，然后进行重构。
    *   确保开发人员了解业务并优先解决客户的痛点。
    *   确保组织分配足够的能力来承担创新的成本。
*   **开发者:**
    *   不要急着把东西放到你平台的核心。
    *   将副作用移到应用程序的边缘，并将它们从业务逻辑中分离出来。此外，当涉及到功能时，努力保持业务逻辑尽可能的纯粹；例如，确保相同的输入产生相同的输出，而不会产生副作用或依赖它们。
*   **大家:**
    *   运用 [理论中的](https://jobs-to-be-done.com/jobs-to-be-done-a-framework-for-customer-needs-c883cbf61c90) 来更好地了解客户的需求。理解银行业的运作方式比理解客户的工作和需求需要更长的时间。

有了这些路标，开发者将更容易在商业银行中激发创新，让创造力蓬勃发展。理解并接受这些挑战将使开发人员能够构建复杂的工具和体验，这些工具和体验在商业银行领域早就应该有了。
