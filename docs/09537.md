# Buildkite 增加了分析工具来识别不稳定的应用测试

> 原文:[https://devo PS . com/build kite-adds-analytics-tools-to-identify-flacky-app-tests/](https://devops.com/buildkite-adds-analytics-tools-to-identify-flaky-app-tests/)

Buildkite 为其持续集成/持续交付(CI/CD)平台添加了一个分析工具，用于识别不可靠的测试。

Buildkite 的联合首席执行官 Keith Pitt 表示，该公司的[测试分析](https://www.prnewswire.com/news-releases/buildkite-eradicates-nightmare-flaky-coding-tests-for-developers-with-revolutionary-new-tool-301560638.html)工具能够对测试套件进行持续的性能监控和实时洞察，如果编写不当，只会浪费开发团队的时间。Pitt 指出，大多数 DevOps 团队没有消除这些测试，而是选择再次运行它们，因为他们测试的代码最终会通过，因为测试的构造很差。

Buildkite 报告称，每个月有超过 8 亿秒——相当于 25 年——由开发人员在其平台上手动重新运行的零散测试。

除了与 Buildkite CI/CD 平台集成之外，测试分析工具还可以应用于 CircleCI、GitHub Actions 和 Jenkins 等 CI 平台中的测试。

测试分析包括一个仪表板，可识别最慢和最不可靠的测试，以帮助确定测试补救工作的优先级。还提供了历史视图和趋势图，以进一步了解重复出现的问题。

DevOps 团队还可以利用用户定义的阈值来定义测试是否缓慢或不可靠，这将在发现时生成定制的警报。最后，还有与常用测试框架的集成，以确定测试在哪里，例如，浪费时间对超时的应用程序编程接口(API)运行数据库查询。

Pitt 指出，今天的大多数测试是由开发人员和软件工程师创建的，而不是由专门的测试团队创建的。因此，返回不可靠结果的测试出现问题的可能性很高。

当然，组织正在试图确定应用程序测试的转变还有多远。开发人员可以编写他们自己的测试，但是这些测试的彻底性和客观性是有争议的。还有一些测试应该应用于组织可以嵌入到其 DevOps 工作流中的所有应用程序，以确保它们始终运行。

显然，测试越多，开发团队在应用程序部署后遇到的问题就越少。然而，当测试不可靠时，开发人员更容易证明不运行它们是合理的，即使该测试可能仍然暴露了他们应该解决的问题。除了增加对那些测试质量的信心之外，知道哪些测试是不可靠的，允许 DevOps 团队决定哪些测试应该首先运行。在越来越多的组织试图提高开发人员生产力的时候，这一点至关重要；在编写更多代码之前等待测试完成通常会对生产率产生负面影响。

最终，随着机器学习算法和其他形式的数据科学的应用，测试的整体质量应该会提高。然而，与此同时，改进应用程序测试的责任显然落在了创建它们的工程师身上。面临的挑战是，许多工程师可能甚至不知道测试有多不可靠，直到有人向他们展示一些无可争议的经验数据。
