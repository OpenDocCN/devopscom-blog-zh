# 从“水-Scrum-Fall”到 DevSecOps

> 原文:[https://devops.com/water-scrum-fall-devsecops/](https://devops.com/water-scrum-fall-devsecops/)

随着组织为了敏捷而放弃软件开发的瀑布方法，许多人陷入了哈桑·亚萨尔的术语“水-混乱-瀑布”。也就是说，组织还没有有效地接受敏捷和 DevOps 原则，仍然处于与业务目标没有联系的孤岛中。进入 DevOps，敏捷思维的延伸。敏捷拥抱不断的变化并将客户嵌入到过程中，而 DevOps 拥抱不断的测试和交付，并将操作嵌入到团队中，以将部署和维护方面的专业知识内化。

在[全天 DevOps 会议](https://www.alldaydevops.com/)上，Hasan 是这样开始他的演讲“DevOps 平台上的多重安全检查点”的。

![Picture5.png](../Images/4e3c0867b603dc8a144d825317363b77.png)

在他的演讲中，Hasan 展示了一个让组织开发 SecOps 的计划。实际上，DevOps 是一种风险缓解策略，建立在情景意识、自动化和重复的基础上。但是，安全性是许多 DevOps 实现失败的地方。每个组织的目标应该是:

*   保护私人用户数据
*   限制对数据/系统的访问
*   保护公司数据/知识产权
*   符合标准
*   保护处置/过渡

但是，组织如何实现这个目标呢？

第一，融合与沟通。产品开发生命周期的每一点都应该被集成和交流，包括在工具之间。一旦实现了这一点，您就可以自动化许多(如果不是大多数)任务。自动化步骤是软件开发过程中需要较少人工操作/输入的步骤。这使得每个人都可以专注于创新和更好的代码，而不是那些可以由自治系统自动化的任务。此外，可以自动化的任务不容易出错。

![Picture6.png](../Images/cb3a63f9e1cec6f5924c6e0243a67b99.png)

当然，最终设计、开发和交付软件的是团队。您的团队由开发、IT 运营、质量保证和安全部门组成。每个人都有自己的技能和重点，重叠部分是安全的开发工作。

![Picture7.png](../Images/f251e8cc5f1d53028712c3290be3a0ae.png)

团队已经到位，流程已经自动化，开发已经开始。这个时代的发展与几年前相比已经发生了巨大的变化。以前，软件受限于大小、功能和受众，供应链几乎不存在。你的团队构建了每个组件。现在，开发已经超出了一个组织在其核心能力之外进行开发的能力。供应链现在涉及许多代码来源。它更像是即插即用，这就产生了许多漏洞。

Hasan 指出了软件供应链的风险因素:

*   **供应商能力**–供应商是否遵循降低供应链风险的做法？
*   **产品安全性**–交付或更新的产品是否安全？
*   **产品分发**–将产品传送给购买者的方法是否能防止篡改？
*   **操作产品控制**–产品是否以安全的方式使用？

为了降低供应链风险，Hasan 建议:

*   确保供应商的安全承诺；
*   评估产品的抗威胁能力；
*   为所有开发人员创建一个经过审查的第三方组件的集中私有存储库；
*   建立良好的产品分销实践；和
*   尽量减少组件的变化，使事情变得更容易。

最后，当您过渡到 DevSecOps 时，请记住，必须在不破坏快速交付、持续反馈模型的情况下解决安全性问题。
