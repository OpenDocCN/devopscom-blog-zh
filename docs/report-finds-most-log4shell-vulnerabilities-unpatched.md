# 报告发现大多数 Log4Shell 漏洞未打补丁

> 原文：<https://devops.com/report-finds-most-log4shell-vulnerabilities-unpatched/>

跟踪和分析软件漏洞的平台提供商 Rezilion 今天发布的一份报告发现，尽管 Java 零日 [Log4Shell](https://devops.com/?s=Log4Shell) 漏洞吸引了所有的注意力，但似乎有近 60%的受影响软件包仍未打补丁。

Rezilion 使用谷歌的开源洞察工具扫描开源软件包，包括依赖关系，发现在总共 17840 个受影响的 Java 软件包中，只有 7140 个被打了补丁。

此外，Rezilion 报告引用了 Sonatype 维护的 Log4j 漏洞资源中心，表明主动从 Maven Central 下载的 Log4j Java 日志管理工具的漏洞版本中，有 36%仍然容易受到 Log4Shell 的攻击。

Rezilion 还利用 dive(一种用于探索 Docker 容器图像的开源工具)来验证容器中可能存在的 Log4j 版本，以及 Shodan.io(一种用于发现互联网连接设备的搜索引擎)。

Rezelion 的漏洞研究主管 Yotam Perkal 表示，这些结果表明，许多部署了 Java 应用程序的组织无法知道它是否包括广泛使用的开源 Log4j 工具，用于跟踪 Java 应用程序中的日志。该工具的许多实例都受到最近发现的零日漏洞的影响，该漏洞很容易被用来在 Java 应用程序中执行恶意代码。

挑战在于许多组织使用第三方提供给他们的 Java 软件组件，并且他们认为软件是安全的。然而，Perkal 指出，第三方通常不知道该软件可能是如何部署的。

他补充道，更令人不安的是，如今许多部署软件的组织并没有积极地管理任何可能被认为是软件供应链的东西。Perkal 说，软件是按需部署的，很少或根本不考虑出处。

即使组织可以跟踪软件组件，修补 Log4j 也是一个挑战。Java 文件可以嵌套在其他文件中。Java 应用程序的打包方式也使得寻找漏洞变得更加困难。

自然，发现易受攻击的 Java 应用程序实例的大部分责任落在了 DevOps 团队身上。随着组织采用围绕软件材料清单(SBOM)的 DevSecOps 最佳实践，跟踪易受攻击的软件包变得更加容易。问题是大多数组织要么没有 SBOM，要么在应用程序升级时没有更新它。在缺乏 SBOM 的情况下，在最初的漏洞被披露四个月后，许多 DevOps 团队仍然在寻找 Log4jShell 的实例。

Perkal 说，不幸的是，最近的漏洞危机似乎不是推动更多组织实施 DevSecOps 工作流的警钟。虽然一些组织正在主动修补应用程序，但许多其他组织仍在等待第三方的更新，他们希望第三方能够解决这个问题。然而，这些更新可能无法及时阻止知道 Log4jShell 漏洞的网络犯罪分子造成破坏。
