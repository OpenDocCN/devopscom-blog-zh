# 填补云 IT 技能缺口的 4 个技巧

> 原文：<https://devops.com/4-tips-to-plug-the-cloud-it-skills-gap/>

技术人才少之又少。编程世界面临着开发人员短缺，随之而来的是云技术的缺乏。到目前为止，大多数组织已经成为[多云](https://devops.com/majority-of-organizations-now-adopt-multi-cloud/)，然而 86%的 IT 专业人员认为缺乏这些技能将[减缓云项目](https://cloudcomputing-news.net/news/2021/jul/28/near-term-solutions-to-fill-the-cloud-skills-gap/)。

随着越来越多的团队从旧的架构过渡到云环境，IT 技能缺口可能会对软件开发和部署造成严重的瓶颈。云服务器需要持续维护，管理安全性和合规性具有挑战性，即使对于经验丰富的开发人员也是如此。此外,[很容易在云中超支](https://devops.com/how-to-respond-to-rising-cloud-costs/)—如果没有对供应商细微差别的敏锐洞察力，为手头的应用程序选择最具成本效益的软件包将变得非常困难。

我最近与 cloudtamer.io 的首席技术官兼联合创始人 Joe Spurrier 进行了交谈，以了解团队如何在缺乏 IT 云知识的情况下改进他们的云战略。根据 Spurrier 的说法，增加自动化、共享可重用模板、集中最佳实践和保持不可知论是放大云结果的策略。下面，我们将扩展这些领域，看看它们如何最大限度地提高云优势，同时保持精益工程足迹。

## 自动化，自动化，自动化

“自动化是进一步采用和使用云的关键，”Spurrier 说。对于许多 DevOps 和 SRE 的专业人士来说，自动化的趋势并不令人惊讶，他们的目标是将他们的工作自动化。当实现云自动化时，基础设施即代码(IaC)是一项基本需求。Spurrier 说:“如果你正在进入云计算领域，你可能需要一些脚本技能。

由于工程师必须重复类似的操作，如服务器供应、部署模式、监控等，因此在设置额外帐户时，自动化可以显著提高可扩展性。自动化对于左移测试和增加生产环境中的可观察性也是必不可少的。“通过自动化部署应用是一回事，但在自动化中进行测试是另一回事——这是下一个级别，”Spurrier 说。

既然如此多的工具都是 API 驱动的，编排就变得可编程了。还可以利用云服务提供商的 CLI 或 SDK 来挂钩这些集成，以构建这样的自动化。

另一方面，过多的高级权限自动化可能会适得其反。“有人可以闯入数据中心，偷走服务器，断开它，然后逃跑，”Spurrier 解释说。因此，自动化将需要更高的安全感，以确保每一个更改都得到适当的授权和记录。

## 可重用模板

除了自动化，记录工程团队之间共享的操作流程对于在精益人才环境中最大限度地提高效率至关重要。Spurrier 强调说，这里的关键是使过程自文档化和可重用。这很可能涉及到团队中的某个人为其他团队成员创建共享库和存储库。

一个这样的例子是亚马逊的 golden AMI pipeline。Golden AMIs 是虚拟机，团队可以“通过配置、一致的安全修补和强化来实现标准化”Golden AMIs 还可以实现一个可重复的流程，用于执行监控、日志记录，甚至分发到业务部门。

可重用模式对于构建 CI/CD 系统非常重要，这样当您提交代码时，它会自动部署到首选的云提供商。总的来说，Spurrier 建议构建易于克隆的共享存储库，并提供一步一步的使用说明。用不同语言演示组件的示例应用程序也可以帮助开发人员提高可用性并节省时间。

## 采用云卓越中心

另一种鼓励更好采用的方式是创建卓越云中心(CoCE)。对于企业来说，拥有一个以云为中心的项目管理团队非常有助于组织知识和提供支持。CoCE 可以集中运营，为工程师提供预配置的容器，并帮助分发对云帐户的访问。

Spurrier 说，CoCE 是一个很好的方式，可以开始将不同的团队团结在共享的最佳实践周围。他补充说，根据组织的需求和工作量，CoCE 将侧重于不同的技能。一开始，一家公司可能只有一个云知识中心。然而，随着采用的增加，围绕特定的云提供商也可能出现专门的中心。

## 保留不可知论

与普遍的看法相反，[多云平台并不总是提升和移动](https://devops.com/multi-cloud-is-not-about-lift-and-shift/)。业务部门通常采用特定的云来满足特定的工作负载需求。例如，研究人员可能会选择 GCP 的数据科学功能，或者微软商店可能更喜欢 Azure。

在一个多云世界中，管理层可以帮助保持少量的供应商中立性。例如，以[云托管](https://cloudcustodian.io/)为例，这是一个在 Capitol One 开发的工具，最近被贡献给了 CNCF。使用云托管，工程师可以跨 AWS、Azure 和 GCP 运行安全检查，并生成报告指标。这种控制和洞察力可以让团队自动处理一些多云管理负担。

Spurrier 说，当使用多重云时，考虑可以跨不同提供商工作的不可知工具也是很好的，如 Terraform 或 Kubernetes。

## 模板

大多数组织都在进行云转型。随着这种增长，更多的云原生流程和工具正在出现。然而，由于人才短缺，DevOps 并不总是准备充分。回顾一下，我们讨论了企业可以绕过这个问题以获得优势的四种方法:

*   增加云自动化是对劳动力减少的必要回应。尽管楼宇自动化需要前期的技术技能，但它能提高效率。
*   可重复使用的库和模板可以快速配置新项目，并在业务部门之间保持一致。
*   考虑维护一个卓越的云中心，以集中策略和提供支持。
*   选择不可知的工具。这有助于保持供应商中立，并跨云应用一致的控制。

在上述策略中，Spurrier 强调了创建共享模板的重要性。他说，如果使用得当，模板方法会非常有效。“如果您有一个模板模型，DevOps 团队应该很容易迁移到新功能。如果您正在利用设置模板的工具，则更容易计算出基础架构的成本。”
