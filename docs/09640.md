# 一切如代码革命和 OWASP 十大

> 原文:[https://devo PS . com/the-everything-as-code-revolution-and-the-owasp-top-10/](https://devops.com/the-everything-as-code-revolution-and-the-owasp-top-10/)

经过多年的停滞，开放 Web 应用程序安全项目(OWASP) [十大名单](https://owasp.org/www-project-top-ten/) 终于看到了一些变动。

最值得注意的是，不安全的设计是 web 应用程序的第四大安全风险。

OWASP 说:“如果我们真的想作为一个行业‘左移’，就需要更多地使用威胁建模、安全设计模式和原则以及参考架构。”

那一针见血。对于行业来说，组织正在适应并继续负责任地向左转是一个很好的信号，并指向我们一直想象的一切如代码的未来。

OWASP 的这一补充最终认识到了组织努力解决安全问题的一个关键原因。如今，许多安全措施都是被动的，组织在部署应用程序后很久才发现问题，在几乎每个行业都缺少安全专业人员的情况下，安全团队只能充当消防员。

为了有效地向前推进，我们需要转变思维。

研究表明，如果组织在设计中解决了这些相同的问题，他们将消除消防 的 [成本，并显著提高](https://www.researchgate.net/figure/IBM-System-Science-Institute-Relative-Cost-of-Fixing-Defects_fig1_255965523)[开发人员的生产力](https://devops.com/?s=developer+productivity)。它还将解放安全团队，让他们专注于安全的战略需求。

OWASP 的这一变化认识到，在制定关键软件设计决策时，安全性需要占有一席之地，这需要授权给安全团队，就像 DevOps 团队有权拥有和管理应用程序的安全设计和实施一样。

## 一个独特的转折点:DevSecOps

安全漏洞继续堆积，然而许多组织还没有重新检查他们的被动方法，这忽略了导致问题的架构问题。我们不能继续这样经营下去了。

在[云原生应用](https://containerjournal.com)中，我们需要一种更加动态和全面的安全方法。我们需要承认在开发生命周期中发现错误配置太晚的失败。这种方式是不可能扩展的，而且在这个过程中你最终会创建太多的规则。

我们需要更主动的参与，这在速度为王的应用程序开发世界中是很困难的。

开发人员和 DevOps 团队应该将安全性视为可信任的合作伙伴，并在开发生命周期的早期就让他们参与进来。他们应该明白，他们对安全负有共同的责任。安全团队应该接受适合开发人员工作流程的自动化，并通过民主化安全使他们的生活更轻松。

## **一切如码的未来**

如果我们能够继续负责任地向左转，我们将继续以一种将产生连锁反应的方式推动行业向前发展，包括在 OWASP 前十名中。

在一个完美的世界中，我们会看到框架和应用程序基础设施解决了输入验证问题，这在今天的十大问题中占了很大一部分。这些问题应该从列表中删除，10 大问题应该集中在导致当前列表的系统问题上，如不安全的设计。

随着我们走向一切如代码的世界，应用程序和基础设施之间的界限变得模糊，OWASP 可能会更广泛地关注一切如代码，这将包括一类新的问题，如云代码、基础设施如代码、管道如代码和安全如代码。虽然肯定还有更多工作要做，但这将是一个强烈的信号，表明我们正朝着正确的方向前进。
