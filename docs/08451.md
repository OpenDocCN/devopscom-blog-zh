# 开发运维与持续交付:不一样

> 原文:[https://devops.com/devops-and-continuous-delivery-not-same/](https://devops.com/devops-and-continuous-delivery-not-same/)

*DevOps 和连续交付是一回事吧？远非如此。我们透过迷雾来理解这两种相关但不同的方法之间的区别。*

我们生活在一个应用的时代。我们越来越多的个人、社交和工作生活都由应用程序控制——通过社交应用程序相互交流，通过移动设备预订电影票，或通过航空应用程序安排商务旅行。服务的数字化已经到来。了解这一巨变的组织拥有超越竞争对手、更快获得客户和增加收入的独特机会。

这项服务开发的战场是 [DevOps](http://automic.com/devops-ara) 。它不是一个产品，而是一种帮助组织建立团队和软件的方法。DevOps 是一种文化、运动或实践，强调软件开发人员和其他 IT 专业人员的协作和交流，同时自动化软件交付和基础设施变更的过程。它创造了一种文化和环境，在这种文化和环境中，构建、测试和发布软件可以快速、频繁和更可靠地进行。

组织在用脚行进。根据最近的一项研究，市场对开发运维技能的需求正在增长，开发运维工程师是当今薪酬最高的 IT 从业者之一。而且看起来会一直保持下去。[英国的研究](https://devops.com/2015/09/02/will-devops-10-years-time/)显示，在 2015 年 DevOps 峰会召开前接受调查的 300 家组织中，有 30%已经开始实施 DevOps 战略，或者已经合并了开发和运营部门，另有 13%计划沿着这条路走下去。

## 两个相关的孩子:DevOps 和连续分娩

所以现在我们到达[连续交货](http://automic.com/continuous-delivery)。如果 DevOps 是关于加速新的、高质量服务的交付，那么持续交付肯定是同样的事情，不是吗？换句话说，就是确保源源不断的新服务？您可能会将其视为制造生产线:DevOps 是构建服务的机器，而连续交付是将服务推出生产线的传送带——一个大的统一服务开发周期。

事实并非如此。虽然 DevOps 和连续交付之间的区别可能有点模糊并引起混淆，但实际上这两者之间是清晰的。持续交付已经成为团队进行迭代和增量软件交付的基本要素。这是一种方法，通过这种方法，团队可以确保系统的每个变更都可以发布，并且任何版本都可以通过一个按钮发布。请这样想:持续交付的目的是使发布变得枯燥和可靠，因此组织可以频繁地交付，风险更小，并更快地从最终用户那里获得反馈，直到部署成为业务流程和企业竞争力的一个不可或缺的部分。

Gartner 在一份报告中非常简洁地总结了 DevOps 和持续交付之间的区别，该报告称:“ [DevOps 不是一个市场，而是一种以工具为中心的理念，支持持续交付价值链。](http://apmblog.dynatrace.com/2016/01/05/the-digital-world-is-itil-relevant/)“分析师认为，双模和数字业务策略刺激了对提高软件交付速度和效率的需求。

尽管如此，持续交付和 DevOps 确实有一些共同的特征。这两种方法都是针对敏捷和精益思维的:每一种都交付小而快的变化；每一个都依赖于紧密的业务和 IT 协作；每个公司都有一个共同的目标，那就是加快新服务的上市时间。

然而，DevOps 的目标是将开发和运营角色以及他们管理的流程合并在一起，以实现业务目标。这都是关于一个共同的，共享的文化和加强合作。关于清晰定义的业务流程。

自动化在开发运维及持续交付中都扮演着重要的角色。自动化开发运维有助于您的组织管理和保护向业务和 IT 用户的持续交付。其结果是更快、低风险地迁移到复合应用程序的新世界，更快的业务创新和对快速变化的业务流程需求的更好支持。
