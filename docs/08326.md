# 第三方安全保证如何增强 DevSecOps

> 原文：<https://devops.com/how-third-party-security-assurance-enhances-devsecops/>

如今，企业不断尝试用更少的资源做更多的事情，并更快地完成工作，以获得竞争优势和增加收入。这在他们的内部软件开发过程中最为普遍。向第三方或外部源代码的转移是对“更快”发布周期的自然反应。然而，这种对速度的需求可能会给应用程序开发团队带来意想不到的下游安全后果。

随着网络犯罪与许多企业中数字流程的爆炸式增长同步发展，“设计安全性”需要从美好的概念变为现实。对于许多开发团队来说，这就是 DevSecOps 从流行词汇走向实践的原因。实施软件安全保证计划，将安全性作为编码和组装过程的一部分，而不是由安全人员执行的单独的事后质量测试功能，对于限制修复漏洞和防止数据泄露的成本至关重要。

## 软件攻击是一个日益严重的问题

最近最引人注目的网络犯罪事件之一是黑客利用网络安全管理软件产品软件平台作为入侵公司的跳板。随着威胁专门针对处于增长曲线上的软件，组织需要保护他们开发的软件和他们从供应商处获得的软件。这通常包括许多从供应商和开源库中获得的重用代码，这些代码为软件和应用程序提供了构建模块，以帮助开发人员更快、更便宜地工作——但存在潜在的安全成本。

维护软件安全变得越来越复杂，因为企业越来越依赖第三方组件来更快、更便宜地开发软件，并跟上数字革命的步伐。在过去的五年里，企业软件内部开发的代码份额从 43%下降到 38%,而商业软件和 T2 开源软件的使用却在增长。许多开发人员不再依赖定制代码，而是利用开源组件或 API 库来构建应用程序。一些软件供应商提供的应用程序库可以包含该供应商为其客户购买的第三方组件。

依赖第三方代码会让软件容易被滥用，因为开发人员对他们用作编程构建块的组件的内容了解不多。一些供应商库被供应商重用于不同的应用程序，一些在他们的代码中使用开源组件；一个开发人员可能正在使用不知什么来源的第三方或第四方代码。漏洞可能隐藏在代码深处的许多层，除非开发人员知道来源，否则它们很难被检测到。

这就是一个完善的软件安全保证计划可以降低网络安全事件的风险和安全缺陷的影响的地方。它通过建立网络安全实践来保护用户和企业，满足组织的风险管理和法规遵从性要求，并提供高级管理人员对流程的可见性。

## 软件安全保证清单

一个组织良好的安全保证计划确保已经为软件和软件开发过程建立了安全要求，并且部署的任何软件都没有漏洞。如果发现任何问题，它还会建立补救步骤来处理这些问题。

一些重要的实践:

● **信任但验证:**确保在整个软件开发生命周期(SDLC)中对软件进行了安全评估。主动验证软件代码的安全性和完整性，提供识别任何发现的漏洞的结果，并分配补救步骤，以便在发布到生产环境之前纠正这些漏洞。
● **知道软件里有什么:**由于许多开发者和软件供应商更频繁地依赖开源和第三方代码，所以知道那些组件是什么以及它们来自哪里比以往任何时候都更重要。软件安全
保证应包括任何开源代码和第三方组件的软件材料清单(SBOM)。
● **写下来:**产生文档化的证明，比如上面提到的 SBOM，生成证明软件完整性的漏洞报告。如果发现漏洞，根据通用漏洞评分系统(CVSS)评分(一种行业标准)确定网络安全风险和潜在影响，以指导补救。

一个好的软件安全保证计划确保每个软件审查或审计都包括对企业安全需求的评估。根据软件运行的功能，这些可能因行业而异，甚至在组织内部也是如此。为发电公司运行邮件列表的程序不像运行电网的软件那样任务关键。

随着所有行业都成为网络攻击者的目标，一个有效而勤奋的软件安全保证计划将使他们更难找到后门，并有助于保护您的组织的安全。
