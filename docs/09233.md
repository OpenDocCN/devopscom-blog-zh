# 修复 Spring4Shell 从软件供应链管理开始

> 原文：<https://devops.com/fixing-spring4shell-starts-with-software-supply-chain-management/>

Spring4Shell 是从根本上改善软件供应链完整性的最新行动号召。虽然 Spring4Shell 的调查仍在继续，但有一个结论是无可争议的:我们必须全面地重新思考我们持续盘点和管理相互关联的软件及其来源的复杂景观的方式。

无论 Spring4Shell 的影响范围是否超过 Log4j 的影响范围，在开源基础设施、框架和应用软件栈中，软件和 API 安全仍然存在巨大的潜在严重后果。

这是因为，正如我们在 Log4j 中看到的，这种攻击可能会影响全球大多数企业 Java 应用程序。似乎根本原因是广泛使用的免费社区开发的开源编程框架 Spring Core 中的一个漏洞。

根据一些数据，Spring 框架是大多数企业 Java 应用程序的基础——多达 74%。具体来说，Spring 是企业 Java 应用程序的基础，因此团队可以专注于应用程序级的业务逻辑，而不会被限制在特定的部署环境中。

由于 Spring Core 在企业软件、云服务、第三方软件和服务产品以及内部管理软件中的使用，它的无处不在带来了巨大的潜在广泛影响。这是我们改进软件供应链安全和供应链管理方法的又一个行动号召。谢天谢地，已经有了一个开始的地方。

每个基础设施和软件团队都可以通过利用来自 Linux 基金会的软件材料清单(SBOM)项目的资源来解决这个漏洞。2021 倡议包括 SBOM 准备情况调查、生成软件材料清单培训课程和基于 SPDX 标准(ISO/IEC 5962:2021)的 SBOM 生成器工具。

Linux 基金会在 SBOM 的贡献给我们所有人提供了一个解决软件供应链管理问题的良好开端。随着广泛采用，SBOM 为软件项目和用户提供了评估和解决 Spring4Shell 以及任何其他未知漏洞的设备，并让我们为毫无疑问的高影响基础设施软件漏洞季节做好准备。
