# 我们能没有 DevOps 吗？

> 原文：<https://devops.com/can-we-do-without-devops/>

最近，DevOps 受到了广泛的关注，每一家受人尊敬的公司都将拥有一个 DevOps 团队视为自己的责任。它被认为对任何公司都非常有益，以至于一个非技术人员在任何技术网站上浏览帖子都会认为它是治疗所有可能的公司疾病的万灵药。然而，鉴于 DevOps 的定义因公司而异，没有人确切知道一个真正的 DevOps 团队应该做什么，DevOps 得到的赞誉和实际产出之间是否存在一些误解？让我们深入挖掘，自己决定。

然而，尽管围绕这一概念的讨论沸沸扬扬，DevOps 却被认为是一件可行的事情。根据 Puppet 的[devo PS 状态报告](https://www.ciosummits.com/Online_Assets_Puppet_2016_State_of_DevOps_Report.pdf) ，它保证了公司采用者 200 倍的更频繁部署、24 倍的更快恢复和三倍的更低变更失败率。

这就是为什么，为了获得一个关于为什么应该“开发”企业流程的公正观点，在我们的文章中，我们给出了以下问题的答案:

*   devo PS 是什么？
*   它应该帮助解决什么问题？
*   DevOps 工程师是做什么的？
*   DevOps 工程师需要哪些技能？

所有这些问题得到解答后，将有助于我们了解为什么业界将开发运维作为常规实践，为什么贵公司需要尝试它，以及为什么我们不能像以前那样没有它，或者我们可以吗？

## **什么是 DevOps？**

根据 [AWS](https://aws.amazon.com/devops/what-is-devops/?nc1=h_ls) 的说法，DevOps 是文化哲学、实践和工具的结合，它提高了组织高速交付应用和服务的能力:以比使用传统软件开发和基础设施管理流程的组织更快的速度发展和改进产品。嗯，这个定义可能不像撒丁岛的水域那样清晰，所以我们会稍微为你澄清一下。

消息是，曾经(在 DevOps 之前)产品发布缓慢而麻烦，而今天，随着 DevOps 实践的到位，它几乎达到了宇宙速度和高端质量。通过工作方法和实践的成熟(开发和运营的合并)，以及自动化、云技术、虚拟化和大量顶级工具。DevOps 文化在一定程度上使开发人员能够执行 IT 团队的工作，反之亦然，这打破了团队之间的操作墙，同时侵蚀了“不是我的任务”的心态。

## DevOps 工程师是做什么的？

如果这就是今天正在发生的事情——程序员处理配置问题和 IT 团队代码——为什么每个人都在谈论 DevOps 工程师？如果你可以在开发人员/IT 人员的合同中写入更多的新职责，并保持名称不变，为什么要想出一个新的术语呢？事实是，过程和角色的转换是如此的重要，以至于他们最终合并到了一个全新的位置，这个位置既不是编码 IT 人员，也不是配置开发人员。

| **开发前的角色** |
| **开发者**

*   构建功能，试图将独立的代码片段合并在一起(通常会导致大量的错误，并最终导致漫长而麻烦的集成)。

 | **集成商或生产服务商**

*   众所周知，他参与了大型银行/保险项目。
*   这种现象在行业内的应用有限，这种中间工作通常由开发人员和运营人员共享，并取得不同程度的成功。

 | **IT/运营**

*   修复了开发人员的混乱，将可用的代码(通常需要大量的修改)压缩到可用的基础设施中。
*   支持办公室的技术生活(维修、技术支持、配置和安全等。).

 |
| **今天的角色，大概的职责分解** |
| **开发者**

*   基于规范的代码生产。
*   软件组件集成。
*   现有软件的故障排除、调试和升级。
*   用于参考和报告的技术文件的创建。
*   集成实现。

 | 开发工程师

*   基础设施规划、测试和开发。
*   集成实现。
*   更新和修复。
*   构建工具以最大限度地减少错误并改善客户体验。
*   开发软件与内部后端系统集成。
*   生产错误根本原因分析，
*   调查和解决技术问题，2 级技术支持。
*   自动化可视化的脚本开发。
*   系统故障排除和维护的程序设计。

 | **IT/运营**

*   安装和配置计算机硬件、软件、系统、网络、打印机和扫描仪。
*   监控和维护计算机系统和网络。
*   服务问题和请求。
*   整个公司的技术支持。
*   为新用户设置账户。
*   必要时修理和更换设备。

 |

总而言之，DevOps 工程师是一个所谓的“半人马”,在编码会议之间履行系统管理员的职责。他/她该项目的主要目标是找到一种解决方案，使产品部署快速高效，并使人们(开发人员和运营人员)的生活更加轻松。

## DevOps 工程师需要哪些技能？

根据 Qulix 的实践，DevOps 的理想技能组合是:

*   技术毕业生。
*   中级或高级英语。
*   一般编程技巧。
*   精通网络、Git、Linux、Docker、Nginx、Bash 的 PC 用户。
*   精益求精，求新若渴。
*   古玩，有内驱力要达到底线，热情。
*   很强的职业道德和时间管理技能。
*   开放且具有团队精神。

作为开始，这些就够了。同时，在同一家公司内，不同的部门可能要求他们的 DevOps 工程师掌握不同的技术。比如我们的另一个部门，期望从他们的候选人那里了解 TCPIP，DNS，Nginx，Tomcat，Git，Bash，Groovy，Ansible，Docker，Ubuntu，Debian，Macos， Gradle，Maven，CI/CD。

如你所见，在一个特定的部门里，开发运维工程师的职位很难找到理想的人选。所有这些都是因为 DevOps 工程师必须要做很多工作，包括技术和软技能。为什么？首先，在某种程度上，作为一名系统管理员，DevOps 工程师必须操作许多技术，这些技术有时远远超过普通开发人员日常使用的技术。在这方面，我曾经问过我一个在 DevOps 团队工作的朋友，“作为一个 DevOps 人员，你必须处理的所有困难事情中，哪一个是最困难的？”他的回答是“一切”

我朋友的“一切”都需要进一步解释，为此我去找了 Qulix Systems 的另一位 DevOps 工程师 Denis Golovnev。他告诉我，他选择 DevOps 职位是因为他想更深入地了解整个 SDLC，因为“这个职位需要对 SDLC 流程有 360 度的了解。”DevOps 的人是少数几个从开始到结束一直参与项目维护的专家之一。这就是为什么“一名 DevOps 工程师不仅要深入挖掘他/她自己的知识领域，还要瞄准相邻领域，看看产品是如何创造的，以及他/她的行为如何能为客户创造更多价值，”Golovnev 解释道。因此，这让我清楚了技术部分。

软技能呢？

DevOps 最困难的事情是以适合每个人的方式安排流程；这将在合理的程度上鼓励主动性和个人决策过程，”Golovnev 说。“要实现这一点，必须具备高度发达的软技能，才能找到问题的根源，让工程师能够向所有人清楚地阐述自己的观点。”

另一方面，DevOps 团队不可避免地会面临技术和组织两方面的问题，不应该混淆。

Golovnev 说:“我们不应该试图通过技术手段来解决与人类相关的问题。这也是为什么没有高度发展的软技能的候选人不太可能得到这个职位的另一个原因。软技能是 DevOps 候选人的绝对杀手锏。

## **总结一下**

我们已经逐渐从缓慢而麻烦的产品发布和独立工作块的交付过渡到快速、有序和一致的性能。最明确的是，这种转变不仅仅是 DevOps 实践实现的结果。但是没有它，没有快速协作交付、动态问题解决和最终自动化的文化，这是 DevOps 哲学的典型特征，这种转变是不可能的。

此外，革命从思想开始，而手段是后来才出现的。按照 Golovnev 的说法，如果一个典型的 DevOps 工程师“热爱他/她的工作，疯狂地将他/她力所能及的一切自动化，抓住阻碍快速产品交付或降低效率的任何约束的核心”，难怪我们作为消费者和整个行业都需要他们。加上杰出的工具和技术，DevOps 工程师是一名士兵，在 100%的情况下，开枪杀人(任何生产约束)。

然而，有一个问题仍不清楚。即使在同一个公司内，对于真正的 DevOps 的含义也有各种各样的说法，我们如何达成一个统一的 DevOps 定义呢？最肯定的是，对于那些试图获得该职位，但由于上述公司之间发展愿景的差异而最终被拒绝的候选人来说，一些一致性将是有用的。

另一方面，由于各种各样的技术以及不同公司使用的不同方法，缺乏明确性是合理的。到目前为止，技能差异仍将是 DevOps 招聘中的一个关键问题。尽管我们的主要问题是——我们能没有 DevOps 吗？-是不可能的。

——[维罗妮卡·奇日](https://devops.com/author/veronica-chizh/)
