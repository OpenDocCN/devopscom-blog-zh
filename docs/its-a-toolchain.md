# 这是工具链！

> 原文：<https://devops.com/its-a-toolchain/>

前几天我和一些朋友聊天，我们开始讨论“[工具链](https://devops.com/?s=toolchain)这个词是多么不准确。我们这些专家——甚至很多营销人员——经常使用工具链这个词，尽管它并不是对我们所做事情的可怕概括，但它充其量只是一种草率的概括。这有很多原因。

最大的原因是“您的工具链”在与单个 DevOps 团队交谈时可能有意义，但一个组织很少只有一个。越来越多的是，即使是单个团队也会根据他们当前关注的项目使用不同的工具。有很多语言、目标、环境、专门的工具、框架等等。独一无二又相互依存的。任何最终达到“一个工具链”的组织都将保持这一荣誉，直到下一次合并。或者直到下一个项目开始。因为最终，它是关于解决问题的，虽然像操作系统这样的一些东西得到了标准化，但其他的没有，因为它们专注于解决这些问题。

然后我们有了变化。即使“一个”工具链也不是同一个工具链；这是同一个 CI/CD 工具，用于管理其他工具。这对于“链条中最薄弱的环节”这一概念有着非常有趣的含义，无论我们是在谈论安全性、合规性还是测试，因为最薄弱的环节可能取决于这次运行产生了哪些工具。举一个简单的例子，这个例子不会与上面的最大原因重叠——将[容器](https://containerjournal.com/?s=containers)作为测试目标，将虚拟机(VM)作为部署目标。由于许可或空间问题，一些组织经常做这类事情。“一个”工具链中的两个不同部署步骤。像这样的例子比你想象的要多。“这个项目使用 make，那个项目使用 cmake”是我们正在谈论的场景类型的一个例子。这些微小的变化由从 CI 中调用的东西来处理。

最后，我接触的大多数现实生活中的组织都是基于项目的，并且在不断发展。这使得上述两种情况成为常态，而不是例外。虽然他们希望所有项目都有一个栈和一个工具链，但是没有人认为这种情况会很快发生。一些专门为一类受众开发一类应用程序的组织(例如，为特定市场开发的动态 web 应用程序)可以使用单一的 DevOps 工具链，但大多数组织不能。

像我这样的空谈者声称我们使用了错误的术语——并且在进行这些讨论时工具链应该总是复数——并不重要。我认为这很重要，因为对一个工具链有用的东西对下一个工具链并不有用。最古老、最成熟、也可以说是最彻底的代码分析工具是为 C/C++开发的。如果重点是赞美那些代码分析工具的优点，并且听众都是主要来自 Python 或 Go 商店的人，那么谈论我们如何有一个解决方案来帮助将测试添加到*您的工具链*就没有太大的用处。

现在，你要考虑的重要工具是那些使你的工作更容易的工具。整合应该是你优先考虑的事情；我们有太多的 DevOps 工具。合并应该是一个考虑因素。是的，Python 和 Ruby 有不同的需求，但是如果有一个工具可以基于管道中包含的信息为它们中的任何一个提供构建或测试，那么就抓住它，这样两个工具集就不会将您的构建过程分成两部分。

继续踢它。说真的。总的来说，我们进入了一个艰难的时期，但是你们一直在努力保持 org 的在线运行。现在不要停下来。如果你的用户没有被“关于”页面按钮上的拼写错误所困扰，他们会感谢你的。所以我会为他们做的。谢谢你。
