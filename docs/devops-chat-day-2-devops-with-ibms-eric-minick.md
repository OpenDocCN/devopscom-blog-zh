# DevOps 聊天:与 IBM 的 Eric Minick 的“第二天 DevOps”

> 原文：<https://devops.com/devops-chat-day-2-devops-with-ibms-eric-minick/>

现在，DevOps 的概念已经存在了几年，越来越多的组织接受了它，对于许多人来说，自然的问题是，“现在怎么办？”—尤其是考虑到 DevOps 的成功是以不同的标准来衡量的。IBM 的 Eric Minick 一直在努力解决“第二天开发运维”的概念——企业范围的部署、多工具链等。—也是让组织从 A 点到 b 点的最佳方式

在本次 DevOps 聊天中，我们将讨论第 2 天的 DevOps 和 IBM 在工具领域为帮助组织向前发展所做的工作，以及 Eric 和他的团队正在进行的其他项目。

**Shimel:** 很酷。Eric，我们只有 15 到 20 分钟的时间，我想直接进入我想谈论的几个话题。我想提出的第一件事是这个概念，就像你们所说的第二天 DevOps 或 DevOps 第二天。让我们来探索一下。首先让我们告诉我们的观众你这么说是什么意思？

**Minick:** 我认为，你知道，我们看看几年前的情况，有许多组织在运行他们的 DevOps 试点，试图找出，“这对我们有用吗？”诸如此类的东西。但是我想我们知道这是可行的，不是吗？

希梅尔: 嗯，我希望如此，但我们会继续发布，直到每个人都最终同意，但是的，你是对的。

**Minick:** 所以我觉得大家看着就像，“好吧，我这里有几个 DevOps 的小口袋，那边有几个 DevOps 的团队。我将如何在我的企业中推广这一点？我认为我们已经有了很好的东西，我们如何大规模地做呢？”所以存在一些可扩展性挑战，你知道，在组织中有一些非常好的基础工作——就像你有几个不同的团队，他们在基础上构建东西，然后他们看着对面，他们使用完全不同的工具链。

然后你会遇到一个被称为 DevOps 副总裁的可怜人，他会问:“嗯，DevOps 怎么样了？”因此，让不同工具链上的团队尝试从中提取数据，让新团队加入一个或两个或其他工具链，这是有意义的。因此，扩展和协调所有这些不同的工具似乎是当前的挑战。

希梅尔:是的，我同意。你知道吗，埃里克，你提到这整件事很有趣，就像“我们能同意这行得通吗？”你有什么，因为你和我一样，我们可能生活在一个小小的 DevOps 泡沫中，我们没有真正与那些尚未开始 DevOps 之旅或正在考虑开始 DevOps 之旅的公司交谈。但是在我们的泡泡之外还有人，对吗？

**微车:** 有。我想我看到一位分析师说，“是的，但现在是 ，——40%的人已经踏上了他们的旅程；40%的人现在正在认真地开始；然后是还有 20%——我和另一位分析师谈过，他说，“是的，我只是认为 20%会倒闭。”你前几天看到了我的文章:我同意。

所以，是的，如果你还没有准备好开始旅程——嗯，我想我应该说我真的很担心你。我本想说我没那么担心你，但现在是时候了。

**希梅尔:**是啊，没错。 对于那些即将进入开发计划第二天的人来说，我认为这是一个问题，很多人都在为第一天的 D 日袭击做准备，而没有真正制定第二天的计划。我们对他们有什么明智的建议？他们可以从哪里获得这方面的信息？他们能做什么？

**微车:** 所以我觉得不打乱正在进行的工作真的很重要。如果你的团队在一些奇怪的工具链上，这些工具链不是企业标准，但是真的真的在工作:让他们运行吧。我认为这种数据已经出现了——我们在 _ _ _ _ _ _ _ _的 Accelerate book 中看到，这一切都是 的，是的，你想授权给团队，你想让他们使用良好的工具链运行，这些工具链正在工作——嘿，这就是为什么我有一个非常活跃的工具业务——这是用真正工作良好的工具替换不工作的工具，或者帮助没有良好自动化能力的团队进行连续交付或测试。

但是我认为你真的应该考虑一下，“我怎样才能看到正在发生的事情？我如何使一些数据正常化？”如果我们知道 DevOps 的关键指标是从代码到产品的交付周期——好吧，“我要如何跨多个工具理解它并使之正常化？”

因此，这是我们在工具领域做了大量工作的一个领域——其他一些人也在这样做——如今这是很容易实现的。

**所以我希望这是第三天——所有这些人都在转变，并走向下一件大事。**
