# 自动化应用程序发布是否意味着与过去一样？

> 原文:[https://devo PS . com/automating-application-release-mean-used/](https://devops.com/automating-application-release-mean-used/)

在过去的十年中，我们已经看到了应用程序发布方式向自动化的转变。自动化首先受到敏捷对持续集成和交付的支持，现在更广泛地转向 DevOps。随着部署自动化离开了单个开发团队的领域，另一个趋势出现了。对于以自动化方式部署应用程序意味着什么的理解已经稳步扩展。对于持续交付早期的开发人员来说，生产问题通常是次要的。他们将采用单个构建，并自动将其部署到测试环境中。一些团队会成功地用成熟的交付管道将这种方法扩展到生产中，但是大多数会遇到困难。忽视底层环境中的生产问题导致他们的工作被认为在生产环境中不可行。然而，敏捷开发的成功意味着对生产变革的需求增加。与此同时，云服务的趋势扩大了给定部署将影响的目标的数量。生产支持团队必须自动化，并且在某种程度上也要适应开发和测试中的快速变化。构建管道模型很接近，但是不能满足增量变更(而不是完整构建)以及跨多个构建和层协调变更的运营需求。

UrbanCode 在 2012 年对 500 多名技术人员进行的调查发现，只有 12%的团队将产品发布局限于一个单独的版本。随着业务要求更快的变化，以及 DevOps 的增加，我们看到自动化在过去几年中在生产环境中迅速成熟。这包括应用程序部署/发布自动化在本世纪初作为一个工具类别的兴起。这些 ADA 工具是由业务应用范围内的自动化定义的——它们跨多个层和组件进行协调。因此，他们不仅部署了一个 web 服务的新版本，还部署了其他五个 web 服务(后端数据库等)。随着 DevOps 更好地连接开发和运营，以及在生产中自动化的意愿增加，我们看到了自动化范围的第一次大转变。从部署构建，到部署构建和相关构建，内容和数据库发生变化。web 应用仍然是最常见的场景。今天，我们看到下一个主要的扩张正在进行，它同时在几个方面发生。

1.  第一个方面是，其发布跨越多个业务应用程序的组织正在添加以发布为中心的编排，以加速耦合应用程序的发布。
2.  下一个前沿是超越 web 应用程序本身。与后端服务相关的移动开发正被纳入“业务应用程序”的定义，包括大型机在内的记录系统也是如此。
3.  第三条战线是向下移动。环境和基础设施越来越不被视为应用程序运行的地方，而是应用程序本身的一部分。随着这种转变的发生，我们看到对虚拟机中的 CPU 数量进行更改的推动会经历与 web 服务行为更新所经历的相同类型的自动化和治理。变化就是变化。
4.  最后，我们看到了像 Bluemix 这样的平台即服务产品的正常化。这导致对该技术的接受度增加，但也导致希望协调通过 PaaS 平台交付的服务和通过其他云或传统数据中心平台交付的服务的变化。

从广义上讲，IT 通过交付服务为企业提供价值。对这些服务的更改往往局限于复杂的应用程序。趋势是通过相同的渠道将软件定义的应用程序中的所有内容都视为相同。这使得系统级测试能够验证整个系统，并使整个系统一起向前推进。上面的讨论不仅告知了正在发生的变化，而且还提示了向变化迈出的一步。迁移到 DevOps 正在迅速带来变化。11 月 20 日，在纽约地区(泽西城)举行的 IBM DevOps 研讨会上，一些行业专家和领导者聚集在一起，讨论有效采用这种变化的效果和步骤。通过聆听杰出的行业领袖和专家，了解如何成功采用 DevOps。加入我们对网络广播的[重播。成为变革的一部分](https://w.on24.com/r.htm?e=879165&s=1&k=7557958DD1C9F547F12C3BA083632062)
