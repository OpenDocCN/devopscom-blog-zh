# 最低可行的应用程序并不意味着最低的安全性

> 原文：<https://devops.com/minimum-viable-app-doesnt-mean-minimum-security/>

据估计，今年移动应用程序的总市场将达到 1430 亿美元。该预算的很大一部分将用于企业移动应用开发。到明年，25%的企业 IT 预算将用于移动应用。然而，尽管做出了这些努力，大多数企业在移动应用程序开发方面仍然落后 12、15 甚至 20 周。

为了帮助清理积压并满足他们内部的移动应用程序需求，越来越多的企业专注于提供具有承诺产生最大回报的功能的移动应用程序；也就是说，人们可能会用最少的投资来使用和做最多的应用程序，以帮助改善他们的日常工作生活和生产力。或者说，这个最低可行产品(MVP)是一个应用程序证明其应用程序概念所需的最低要求。

但安全研发公司 Galois 的计算机安全研究主管 Isaac Potoczny-Jones 指出，当快速转移应用程序时，企业可能会在质量尤其是安全方面有所欠缺，这可能会给应该小心处理的数据带来风险。

“移动领域肯定会有更多的发展，”Potoczny-Jones 说。“我认为移动领域的(安全)最佳实践不如网络领域的最佳实践发展得好。我认为这一点正在变得更好，安全性(MVP)面临着一个非常有趣的挑战，因为安全性是一个非功能性的需求。”

当然，开发应用程序时必须牢记良好的移动安全性，包括应用程序密码管理、服务器端控制、加密和其他最佳实践。 [OWASP 十大移动安全风险](https://www.owasp.org/index.php/OWASP_Mobile_Security_Project#tab=Top_10_Mobile_Risks)提供了常见移动风险和相关缓解措施的出色概述。

移动安全测试面临的一个挑战是，最佳实践和相关流程没有被大多数人很好地理解，至少现在还没有，而且许多组织都没有部署高质量的移动安全测试工具。此外，那些使用一些通用移动安全测试工具的组织发现它们并不总是足够的，他们经常不得不构建自己的自动化工具集。

快速跟踪的 MVP 应用程序的风险在于，在交付生产之前，它们根本不会进行任何安全测试。但是，这些应用程序——就像任何其他接近企业数据和网络或可能访问敏感、机密或受监管数据的应用程序一样——必须经过审查、威胁建模和测试(静态和动态应用程序测试)，以及通常的质量测试。对于 MVP，同样重要的是，测试从一开始就是流程的一个组成部分。

当然，实现这一点的最佳方式是通过开发人员的意识和安全代码培训。

公司可能想考虑创建一个 MVP 开发指南，详细说明 MVP 的安全考虑。这种指南可以帮助开发人员确定要遵循的安全最佳实践，以及哪些应用程序需要发送给安全团队进行审查，以及何时发送。

Potoczny-Jones 说，除了可能的粗心之外，没有理由为什么 MVP 应用程序是在最低限度的安全考虑下构建的，因为大多数精益开发方法都将安全测试构建到了他们的过程中。“一旦你让这些团队将安全测试作为[他们所认为的]‘正在完成的事情'的一部分，我不认为很多人会这样做，那么他们就会做正确的事情。你要么开始在用户故事或验收测试中构建安全性，”他说。

这是肯定的，而且不应该留到开发过程的最后。“你不能把它留在过程的最后。如果你把安全测试留到最后，你的进度自然会下滑。你会发现还有很多工作要做。波托兹尼-琼斯说:“然后，你将处于一个不幸的境地，不得不决定是修复事情，让时间表溜走，还是选择让不安全的事情离开。”
