# DevSecOps 会取代安全运营中心吗？

> 原文：<https://devops.com/will-devsecops-replace-the-security-operations-center/>

传统上，安全运营中心(SOC)是一个组织执行信息安全活动的物理设施。SOC 团队分析并监控组织的安全系统。SOC 旨在通过识别、分析和响应网络安全威胁来保护企业免受安全漏洞的侵害。

[SOC 团队](https://www.cynet.com/incident-response/what-is-a-soc-10-core-functions-and-6-key-challenges/) 由管理员、安全分析师和安全工程师组成。他们与内部开发和 IT 运营团队合作。

SOC 是一种行之有效的方法，可以改进威胁检测和 [事件响应](https://www.bluevoyant.com/knowledge-center/what-is-incident-response-process-frameworks-and-tools) ，降低安全漏洞的可能性，并帮助组织在发生事件时做出适当的响应。SOC 团队隔离服务器、数据库、网络、终端和应用上的异常活动，识别和调查安全威胁，并在安全事件发生时做出响应。

SOC 曾经被认为只适合非常大的组织。但如今许多小型组织都在构建轻量级 SOC，如依赖兼职内部员工和外包专家的混合 SOC，或在没有物理设施的情况下运行的虚拟 SOC。

## SOC 挑战

SOC 在保护组织资产免受网络威胁以及确保其网络和系统的安全性和完整性方面发挥着至关重要的作用。然而，SOC 在检测和应对潜在威胁的工作中可能会面临各种挑战，包括:

### 警戒霸王

警报过载是 SOC 面临的常见挑战。当 SOC 团队被过多的安全警报淹没时，就会发生这种情况，使得很难对它们进行优先排序和有效响应。这可能是由大量警报、缺乏上下文或适当的优先级、有限的可见性或有限的资源造成的。为了应对警报过载的挑战，SOC 团队可能会实施一些策略，例如调整安全工具、实施分类流程、利用自动化以及增加团队规模。

### 能见度差距

可见性差距是指组织的网络和系统中没有受到安全工具和流程充分监控或保护的区域。这些差距造成了网络威胁可以利用的漏洞，使 SOC 团队难以检测和应对潜在威胁。

可见性差距可能是由复杂的环境、有限的资源、影子 IT 或错误配置的系统造成的。为了解决可见性差距，SOC 团队可以实施各种策略，例如实施安全信息和事件管理( [SIEM](https://www.exabeam.com/explainers/siem/what-is-siem/) )系统、进行定期评估、实施额外的安全控制、培训员工并确保系统配置正确。

### 复杂的调查

对于 SOC 团队来说，调查网络安全威胁可能是一项具有挑战性的任务。它要求他们快速准确地识别威胁的根本原因，并采取适当的措施来遏制和减轻威胁。由于有限的可见性、复杂的环境、缺乏背景或有限的资源，这可能很困难。

为了协助调查过程，SOC 团队可以使用取证工具来收集和分析与潜在安全事故相关的证据。常用工具包括网络取证工具和数字取证工具，前者允许团队分析网络流量并确定威胁来源，后者允许团队检查设备和系统以确定事件证据。

### 缓慢的检测和响应时间

检测和响应速度慢是安全运营中心(SOC)面临的一个普遍挑战。它们会影响组织的平均检测时间(MTTD)和平均响应时间(MTTR)，这两项指标衡量 SOC 团队检测和响应潜在安全事件所需的时间。

高 MTTD 和 MTTR 会给组织带来严重后果，因为它可能会让网络安全威胁在很长一段时间内保持活跃，从而可能造成重大损失。

## 什么是 DevSecOps？

DevSecOps 代表开发、安全和运营。这是一个新的工作流程，涉及将安全实践集成到所有 DevOps 流程中。DevSecOps 思想指导所有技术专家平等地获得安全专业知识。

DevSecOps 在开发过程的早期推进安全实践和实施。这创造了一种文化，安全是每个人的责任，而不仅仅是安全团队的责任。

将安全性构建到软件交付生命周期的每个阶段，可以实现持续集成和高开发速度，同时减少安全问题和降低合规成本。

## DevSecOps 如何提高安全性和质量

使用 DevSecOps 模型的组织明白安全性不应该是事后的想法，而应该是软件开发生命周期(SDLC)的核心部分。

DevSecOps 协作比 DevOps 更困难，因为它需要同时实现两个看似矛盾的目标——加快交付过程，同时花费更多时间来确保代码安全且没有错误。

为了在不损害产品质量的情况下实现 DevSecOps，组织需要创建一种将安全性作为代码来处理的文化，鼓励开发人员考虑他们项目的安全方面，并自动执行安全任务。组织需要在 IT 工程师、软件开发人员和安全团队之间建立持续的沟通和协作。

## DevSecOps 流程如何改变 SOC

传统上，SOC 流程与组织的其他部分完全隔离。开发人员构建系统，IT 运营人员运行系统，安全人员保护系统。如今，人们认识到，将这三项职能合并到一个共同承担安全责任的组织中，可以提高安全性并显著提高运营效率。

SecOps 是第二阶段，在此阶段，运营和安全团队成为一个组织。安全性被向左转移，从 IT 需求和系统设计的早期阶段开始，而不是在过程的末尾。

SecOps 对实施 DevOps 的组织产生了重大影响。它为第三阶段铺平了道路——安全、运营和软件开发团队之间更广泛的合作，被称为 [DevSecOps](https://devops.com/?s=DevSecOps) 。这使得安全性更加靠左，从开发的第一次迭代就将安全性构建到系统中。

DevSecOps 可以与 SOC 并行运行。以下是 SOC 实现流程现代化的一些方法，即使 SOC 本身是一个筒仓:

*   **与 DevOps 成员一起开发分散式 SOC:**熟悉 DevOps 的部门成员可以深入了解 IT 系统，了解安全人员面临的漏洞和威胁，从而有助于事件响应。
*   **与威胁猎人和开发运维团队协作:** 威胁猎人可以直接与开发或运营团队沟通，解决关键的安全漏洞，而不是隔离威胁并将其报告给管理层。
*   **创建卓越中心:**SOC 可以与开发和运营团队合作，采用安全最佳实践，并将这些积极成果传播到整个组织，以推广 DevSecOps 实践。
*   **允许 SOC 提供建议和指导:** 参与安全的每个人都应该能够方便地访问 SOC，并能够与组织的顶级安全专家合作。

## 结论

总之，虽然 DevSecOps 可以显著影响 SOC 的运行方式，但它不太可能完全取代 SOC。DevSecOps 强调从一开始就将安全实践和工具集成到开发过程中，这有助于从一开始就防止安全事故的发生。但是，组织仍然需要一个专门的团队来监控和响应潜在的安全事件。

SOC 将继续在保护组织资产免受网络安全威胁以及确保其网络和系统的安全性和完整性方面发挥关键作用。通过采用 DevSecOps 实践，SOC 可以提高组织的软件应用程序的安全性和质量，降低安全事故的风险，并提高其运营效率。然而，SOC 仍将是组织整体安全态势的重要组成部分，并将随着新挑战的出现而不断发展和适应。
