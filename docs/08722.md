# DevOps 文化变革的 10 大技巧

> 原文:[https://devops.com/10-top-tips-devops-cultural-change/](https://devops.com/10-top-tips-devops-cultural-change/)

DevOps 通常被认为主要是创造一种有效和无缝协作的文化，但当它涉及 [**改变**现有文化](http://www.ranger4.com/blog/bid/74892/Enterprise-DevOps-The-Three-H-s)时，这通常是最难实现的(与在创业型环境中从零开始创造文化并从第一天起就雇佣具有特定思维模式的个人并定义工作实践相反)。那么你如何改变文化呢？以下是我们的十大建议:

# 1.学会信任

我知道这听起来有点俗气，但 DevOps 试图解决的核心问题是多年来，甚至几十年来，开发和运营团队相互隔离，无法有效沟通，更不用说协作了。创新和推动快速变革的压力与稳定脆弱系统的需求之间的斗争加剧了这种情况，并创造了詹姆斯·特恩布尔所描述的“有毒”工作环境。

因此，首先需要达成的是一个统一的协议，并认识到我们都在同一个团队中。

**奉献技巧~= >** *花些时间回顾你的个人和团队目标，确保没有冲突。然后建立一个把所有人聚集在一起的单一目标。让每个人都同意它。解雇不能胜任的人。[有点开玩笑——但又不尽然。](http://web.experiencedevops.org/2014/11/18/3-reasons-why-building-a-devops-culture-wont-work/)*

文化的改变需要你改变日常工作的方式——习惯需要被打破。 ***奉献技巧~= >*** *[左移](http://www.ranger4.com/blog/bid/76363/DevOps-Shifting-Drifting-Left-or-Right)——围绕项目安排您的团队执行非线性任务，并让负责生产应用的运营人员参与**在流程的早期**。尽早参与将有助于他们计划代码何时准备好，并感觉他们也参与到游戏中来。*

# 2.理解动机

冲突经常发生在人们不理解对方的时候——我敢肯定，这些年来我们已经无数次听到过这样一句话:“他们有性格冲突”。人类和我们的大脑、个性、背景和性格有着各种各样的特点，这是一件好事，因为正如我母亲常说的那样:“如果我们都是一样的，那会很无聊。”然而，有时在工作中，在压力下，很难花时间去理解为什么有人对你有不同的反应或想法，而直接的反应是反抗或谴责它；不仅令人不快，而且完全不健康。

因此，如果我们接受我们都在同一个团队中，我们如何才能更好地了解彼此并促进合作呢？我知道这些年来我经历了无数的团队合作练习，我发现这些练习非常痛苦，所以我为新一波神经科学喝彩，它让我们能够更精确地了解我们的大脑中在发生什么，而不仅仅是试图决定我是绿色还是红色的性格类型。

我们目前特别喜欢的是《工作中的大脑》( SCARF 前提是，我们大脑中用于生存和防御的相同奖励回路也用于我们工作的动机。因此，如果我受到自主性(围巾中的“A”)的激励，而工作中有人攻击我在工作场所的自主感(例如，通过微观管理我)，我会有相同的神经反应，就好像他们直接攻击我的生活一样——包括释放相关的神经化学物质，在这种情况下，是压力荷尔蒙[皮质醇](https://en.wikipedia.org/wiki/Cortisol)。类似地，如果发生了一些事情来增强我的自主感(例如，我创造的一个事件有很好的结果和反馈)，我会感觉很好，并且会产生一股很好的[多巴胺](http://www.thebabbleout.com/health/dopamine/)。

***奉献技巧~= >*** *与你的团队开一个围巾研讨会，要求每个人先完成[这个自我评估](http://www.scarf360.com/individuals/scarf-self-assessment.shtml)，分享结果并进行讨论。我们并不总是同意这种评估工具的结果，但发现这只会给争论火上浇油。*

我在这里的关键观点是，我们应该首先试着了解自己，然后是我们周围的人(有点“松散”，但这是关于[正念](https://en.wikipedia.org/wiki/Mindfulness))。对我们自己和我们的同事有这种水平的洞察力意味着，当冲突发生时，我们有更好的机会喘口气，了解到底发生了什么，记住我们有共同的目标，并更紧密地合作，协作实现它。

我们最近也开始研究[无意识偏见](https://www.theguardian.com/women-in-leadership/2013/aug/15/guilty-of-unconscious-bias-job-roles)*–*如果你在这方面有任何经验，我们很乐意倾听你的意见。

**devop static Tip ~ =>** *如果你想知道自己是否有无意识的偏见，可以使用哈佛的[在线项目隐性测试工具](https://implicit.harvard.edu/implicit/)进行自我测试。*

# 4.拥抱聪明的失败

为了让企业繁荣、生存，甚至在今天竞争异常激烈的市场中，快速创新是必不可少的。当我们评估组织的 DevOps 文化成熟度时，我们衡量的特征之一是组织的能力[‘在失败的边缘跳舞’](https://www.appdynamics.com/blog/devops/quantified-devops/)。以前所未有的速度创新会带来一系列必须降低的风险。需要创建能够警告和预防故障的系统，并且需要实现一些功能，例如在出现故障时立即重新部署应用程序的最后一个已知良好状态或工作版本。我们绝不是在暗示应该庆祝[的失败。](http://www.plexxi.com/2014/11/dealing-vs-celebrating-failure/#sthash.bAgvICvR.dpbs)

除了创建一个安全的环境，在这个环境中，故障可以快速轻松地修复，您还可以通过花一些时间优先考虑功能和以可组合的方式设计应用程序来为故障做准备——例如，您的业务交易可能比您的社交订阅源更重要。在最近的一次研讨会上，我们被告知法国铁路在线票务应用程序如何因其分析问题而停机三天——如果组件没有紧密耦合，用户就不会受到影响，因为预订系统可以独立运行。

**devo static Tip ~ =>** *研究[网飞的混沌猴](http://techblog.netflix.com/2012/07/chaos-monkey-released-into-wild.html)方法，并确定一种在您的基础设施中安全测试故障的方法。*

# 5.关注瓶颈和流程

如果你读过[凤凰计划](https://www.amazon.co.uk/The-Phoenix-Project-Helping-Business-ebook/dp/B00AZRBLHO)，你就会知道关于[约束理论](https://en.wikipedia.org/wiki/Theory_of_constraints)的一切——我们对这个概念最喜欢的总结是“任何没有在约束条件下做出的改进都是幻觉”。作为一个团队努力工作，以确定真正的制约因素——并注意书中布伦特角色描述的常见陷阱，即那些看起来像英雄的人实际上是你的瓶颈。试着理解每项改进对你的业务的真正价值，并相应地对它们进行优先排序。

不要害怕重新思考你的过程，或者找到一双新的眼睛去看你因为太近而看不到的东西。观看此视频，了解赛车运动在过去五十多年中对其流程的改进——查看 2013 年的资源利用、完美流程和自动化:

# 6.消除计划外的工作

凤凰计划 的另一个主要话题，以及当我们开始与组织合作时，我们听到的普遍抱怨:“我如何找到时间来节省时间？”。人们知道他们的流程可以改进，自动化很有可能进一步发挥优势，但如果文化是继续做一些破碎的事情，就很难抛开消防之外的时间来实现开发运维。

几乎不可能告诉一个企业他们可以有创新或支持，但不能两者都有，但明智的做法是掌握项目的运行和可用的资源，以及通过管道进入的计划外工作量。一些优先级较低的项目可能需要暂停，而 DevOps 项目具有较高的优先级-这可能需要获得管理层的授权，并有坚实的理由支持，即一旦奠定了基础，所有后续项目都将加速。

**奉献提示~= >** *阅读[凤凰计划](https://www.amazon.co.uk/The-Phoenix-Project-Helping-Business-ebook/dp/B00AZRBLHO)——你到底在等什么？！* 

# 7.持续不断

有很多这样的例子——我们不断看到:

*   集成代码
*   试验
*   合并
*   部署
*   传递
*   基金

其理念是让软件始终处于可交付状态，并拥有一个能够按需部署的 DevOps 工具链——让您的软件开发生命周期像一条制造流水线。

不过，我们最喜欢的是“持续投资”——正如 Forrester 所说:

“以快速发布周期交付产品的组织为持续的创新提供了稳定的资金基础。他们利用客户反馈和业务优先级，不断对能够产生最大业务价值的功能进行优先排序。”
 **奉献技巧~= >** *试着设定一个预先批准的创新预算，缩短审批周期，而不是每年一次的大型项目预算。

# 8.组建专门的跨职能团队

很多人问我们关于[创建 DevOps 团队](https://continuousdelivery.com/2012/10/theres-no-such-thing-as-a-devops-team/)的事情，我们认为这很可能适得其反，因为它有创建另一个竖井的趋势。然而，这里存在挑战，因为不可能让每个人都成为所有事情的主人，特别是在更复杂的技术情况下。更简单的技术可能有让人们更接近商业的优势。

我们最近与剑桥 Satchel 公司的首席技术官 Jonny Wooldridge 举行了一次研讨会，讨论了他在马克斯和斯潘塞工作期间建立和管理的 DevOps 团队。Jonny 的团队表现得更像一个“老虎”团队，关键资源可以根据需要加入项目团队，以传播 DevOps 方式，并在项目生命周期中帮助交叉技能。

**奉献技巧~= >** *当你考虑引进一项新技术时，[考虑掌握它的时间](http://www.ranger4.com/blog/bid/76416/Forrester-on-Automation-and-Application-Performance-Management)——时间越短，你的团队就越有机会掌握它。*

关键是让每个人都以 DevOps 的方式思考，在时间和资源允许的情况下交叉技能并掌握技能——但要围绕项目安排资源，而不是他们的核心技能。

# 9.喜欢透明

俗话说知识就是力量——但事实并非总是如此。有时候，分享你的知识会更好——不仅你会获得尊重和信任，你还会发现你周围的团队能够更快地实现你的共同目标——并鼓励其他人也与你分享他们的知识。我们不可能什么都知道，但是如果我们把事情藏在心里，我们经常会妨碍我们团队的表现，最终也会妨碍我们自己。

**devo pstatic Tip ~ =>** 与您的团队分享。与世界分享。重复一遍。

知识占有者通常会成为瓶颈——例如，我们经常看到这样的情况，个人负责一个部署过程，厌倦了手动重复它，并编写脚本来加速它——越来越多的脚本，直到他们有了一个“脚本怪物”。现在，他们是唯一可能理解这部分流程的人，当出现任何问题时，他们是唯一能够解决问题的人——真是个英雄！如果一个人受到确定性的激励(回到围巾),他们肯定会享受到安全感和不可或缺感，但这对组织的其他人没有太大帮助，因为他们现在是一个严重的瓶颈。

透明度和可见性是 DevOps 文化的特征，回到技巧 2，理解动机，我们这些受公平激励的人会在开放和平等的文化中感到更快乐。秘密会适得其反，让人疏远。

# 10.建立自主、掌控和目标

改变你的文化可能意味着你需要重新审视你的目标和奖励方式，重新审视你设定的目标，确保它们没有冲突，并且每个人都有一个明确的、单一的目标。奖励心理学可能会让你大吃一惊——看看丹·平克的 Ted 演讲。

你需要能够创造一种工作环境，让人们觉得他们拥有自主权、掌控力和目标——这是最终的回报。请看[这篇博客](http://www.ranger4.com/blog/bid/76055/How-to-Celebrate-DevOps-Success)，了解 DevOps 目标设定和奖励的更多细节。
