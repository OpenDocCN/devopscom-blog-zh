# 过滤消防水管

> 原文：<https://devops.com/filter-the-firehose/>

我们累了。信息过载是现代世界的一个问题。我们会立即听到一些我们原本不知道的事情，或者是在事情发生几个月后才知道的事情。今天，在一个事件发生后，我们有数千名“专业人士”为我们分析它，数百万名业余爱好者告诉我们他们知道关于它的一切，还有一大批机器人告诉我们如何看待这个事件。不管是什么事件。通常“事件”甚至没有新闻价值，然而这个过程本身就在发展。

我们身处其中，情况更糟。我们有大量的安全和应用程序事件不断涌入，数据如此之多，我们称之为“数据湖”更像是一些组织的数据海洋。我们累了。他们给它起了各种各样的名字——警报疲劳、数据过载、事件泛滥、“消防软管”

不管叫什么，都是大问题。如果没有有效的管理，我们就会犯错误。特别是在安全方面，但也包括其他方面，我们不能让洪水分散我们的注意力或耗尽我们的精力。

调查 [AI](https://digitalcxo.com) 。数据、警报、信息、指标、日志、跟踪等的数量。我们正在积累的东西——以及我们积累的速度——增长的速度超过了人类可以合理跟上的速度——即使没有人员编制的限制。不要再犹豫了；不要以为你或供应商 NOC 只用眼球就能处理好这件事。人工智能解决方案已经足够成熟，可以可靠地[筛选至少噪音的顶层](https://devops.com/turning-off-devsecops-noise-for-functional-fidelity/)，在不增加风险的情况下减轻负担。事实上，随着业务量的增加，人为错误的可能性越来越大，风险可能会更小。

问问商贩吧；与您的安全供应商讨论他们的产品。大量的安全厂商现在都有 NOC，他们很乐意让你订阅。他们中的许多是高度自动化的，使用人工智能。数据供应商在这方面有点落后(原因很多)，但他们正在迎头赶上。不久以前，标准化不同数据集的绝大多数工作都是通过反复试验完成的。人工智能和大量经验编写的脚本至少减轻了这一负担。

所以去看看吧。想办法让事情恢复可控。我们都知道，随着举报数量的增加和举报地点的增多，事情越来越糟。不要等到你有了另一个[紧急](https://devops.com/?s=incident+response)；研究一下各种选择，让事情运转起来。

对于那些已经在使用 AI/ML 的人来说，除非你最近才实现它，否则你可能会想要验证你正在使用什么，AI/ML 解决方案在什么级别上有多大帮助，并进行调整或替换。这个空间发展得太快，以至于今天“设置好就忘了”。也许几年后，我们可以。

除了我每天都在使用并且认为您可能会从中受益的极少数例外，我坚持不在此处推荐产品的政策，因此我无法为您指出正确的方向，只能说您的供应商(数据、SIEM 和安全性)可以提供解决方案和/或建议。这一点，以及“你不应该等待”是我的建议。

继续摇摆。数据/日志/警报增长的一部分是您帮助投入的所有应用程序和基础架构，它们对组织至关重要。所以，用一切可用的工具保护它——并利用空闲时间创造更多奇迹。
