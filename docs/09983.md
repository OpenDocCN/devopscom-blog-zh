# Logz.io 推出托管开放式 360°可观测性平台

> 原文:[https://devo PS . com/logz-io-unveils-managed-open-360-observability-platform/](https://devops.com/logz-io-unveils-managed-open-360-observability-platform/)

在 [AWS re:Invent 2022](https://reinvent.awsevents.com/) 大会上，Logz.io 推出了一个 [Open 360](https://www.businesswire.com/news/home/20221129005647/en/Logz.io-Unveils-Open-360%E2%84%A2-Platform-for-Optimal-Observability?_gl=1*12mu7lc*_ga*MTE1NDgwNjUxNy4xNjUwNjQzNjM3*_ga_ZQWF70T3FK*MTY2OTczMDQ3My40Ny4xLjE2Njk3MzA1MTUuMC4wLjA.) 平台，该平台结合了多种开源技术，在现有的单片和新兴的云原生应用环境中提供[可观察性](https://devops.com/?s=observability)。

Open 360 平台中现有的 Logz.io 产品包括 Logz.io Kubernetes 360，这是一个基于开源工具的可观测性管理平台，如 OpenSearch 分析软件、Prometheus 监控工具和 Jaeger 分布式跟踪代理。

此外，Open 360 还包括 Logz.io 遥测收集器代理软件、Logz.io 数据优化中心仪表板、用于将日志数据转换为指标的 Logz.io LogMetrics 索引以及用于配置开源 OpenTelemetry 收集器代理软件的 Logz.io 跟踪采样向导工具。

![](../Images/f19d4c65f715b2e3804318a2a9f37d5e.png)

Logz.io 首席执行官 Tomer Levy 表示，Logz.io 已经策划了开源工具的实例，而不是要求组织将一组脱节的开放软件工具组合起来创建一个可观察性平台。他指出，目标是精简 DevOps 团队观察多类应用程序所需的工具和平台的数量。

就可观测性平台的采用而言，现在还为时尚早，但很明显，随着应用程序环境变得更加复杂，DevOps 团队今天所依赖的监控工具也需要发展。一般来说，可观察性平台承诺统一日志、指标和跟踪，使启动查询以确定问题的根本原因变得更简单，而不是简单地跟踪一组预定义的指标。

DevOps 团队接受可观测性的速度自然会有所不同，但许多 DevOps 团队正在尝试的选择是在多大程度上依赖专有的可观测性平台，而不是使用开源软件来构建他们自己的平台。Logz.io 正在通过依赖基于开源工具的托管服务来寻找这两个极端之间的中间地带。

可观察性一直是 DevOps 的核心原则，但是实现和维护它是具有挑战性的。如今，大多数 DevOps 团队都渴望保持某种程度的持续监控。然而，随着对应用程序进行检测变得越来越容易且成本越来越低，人们对可观测性平台的兴趣越来越大，这种平台可以简化对异常情况的调查，这些异常情况表示可能会破坏分布式应用程序环境的问题。

更具挑战性的是，DevOps 团队现在还管理着更广泛的应用类型，因为基于部署在 Kubernetes 集群上的微服务的云原生应用与运行在云和内部 IT 环境中的传统整体应用一起部署。Levy 指出，Logz.io 方法使 DevOps 团队能够以更低的总成本管理这些应用程序。

无论采用何种方法来实现可观察性，应用程序环境日益增加的复杂性已经成为对成本越来越敏感的组织所面临的主要挑战。他们中的许多人没有能力雇佣额外的 DevOps 工程师。相反，重点是使用新一代可观测性平台提高现有 DevOps 团队的生产力，与扩大现有 DevOps 工程师团队相比，获得和部署新一代可观测性平台的成本应该更低。这可能不一定会消除雇用额外的 DevOps 工程师的需要，但它应该确保组织充分利用现有的 DevOps 团队。
