# 如何实现卓越的云计算运营

> 原文：<https://devops.com/how-to-achieve-cloud-operational-excellence/>

云卓越运营意味着以最优的成本和质量提供基于云的服务的正确组合，以支持您组织的使命和战略。你可能会想，“为什么要这么麻烦呢？难道‘差不多够好’还不够吗？”事实证明这很重要——非常重要。每项相关交付指标中成本最高的 10%和最低的 10%之间的差异可以相差一个数量级或更多。

在 20 世纪 90 年代中期，Gartner 收购了一家名为 Real Decisions 的 IT 度量公司。他们提供基准测试服务，因此客户可以将他们的 IT 效率与类似的组织进行比较。拥有数百家“全球 2000 强”规模的客户，他们的数据库非常丰富。随着时间的推移，Real Decisions 团队改进了他们的指标目录，并通过重复研究丰富了他们的数据，以开发效率指数。

在他们的用户群中，前 10%用户的单位生产成本比后 10%用户高 11 倍。那不是说好 11%，是说好 1000%。

请注意，用户群是自选的。所有参与者都希望获得与其业务目标相关的客观绩效指标，并为涉及问卷调查、财务审计和技术基准的广泛研究付费。值得注意的是，所有参与者都制定了成本回收计划(不仅限于退款)。这一部分中最底层的 10%仍然在 IT 行业的前 10%之内，他们的分数比精英中的精英差 11 倍。

这就提出了一个问题:IT 效率的行业平均水平是多少？有没有可能数百个基准测试用户都做错了，对相关指标的搜索被误导了？我认为不是。

### 选择正确的指标

1911 年，弗雷德里克·w·泰勒(Fredrick W. Taylor)出版了《科学管理的原则》 ，其中讨论了优化两个重要变量的方法:产出质量和工人薪酬。泰勒认识到，成功的公司是相互协作的，管理层和员工共同设定目标，开发方法和工具，以实现盈利和相应的薪酬。在这篇文章中，或者在他的任何录音讲话或文件中，他都没有说过，“如果你不能衡量它，你就不能管理它。”他从来没有那样说过，原因有二:第一，他不相信；第二，不是真的。他*强调的是，如果你*去*衡量它，你就会管理它。这是一个警告:选择正确的度量标准，否则你将努力追求一个无意义的目标。*

以下是一些潜在的云性能[指标](https://devops.com/?s=metrics)，这些指标是从不同行业的几家云提供商和用户、非营利组织和政府机构收集的:

*   **服务指标**
    *   可靠性——平均无故障时间(MTBF)
    *   可用性——正常运行时间，表示为需求的有意义百分比
    *   适用性–平均维修时间(MTTR)
*   **IT 指标**
    *   容量
    *   潜伏
    *   带宽
    *   响应时间
*   **战略指标**
    *   业务敏捷性
    *   客户参与度
    *   客户到达
    *   财务影响
    *   解决方案性能

云计算卓越之旅始于制定与您的业务目标最相关的指标。用正确的尺度选择正确的指标很重要。根据经验，一般来说正确的会击败错误的。

### 持续卓越

在你同意了一套统计上可靠的、可重复的、客观的、与公司使命一致的衡量标准后，你如何实现并保持卓越？

20 世纪 70 年代，美国国防部购买了大量定制软件。有时效果很好，有时则不然。所以，国防部资助了关于代码质量的研究。事实证明，优秀和一般质量代码之间的关键区别在于组织如何管理问题。更具体地说，团队如何对意外事件做出反应？从困惑和沮丧到消防演习的混乱，再到冷静、理性的评估和补救。这种方法已经融入了代码本身。这项研究产生了由卡内基梅隆大学软件工程研究所创建的[能力成熟度模型](https://cmmiinstitute.com/) (CMMi)。

CMMi 框架确定了过程成熟度的五个层次。对一个组织的过程成熟度级别的简单评估归结为两个问题。首先，当前是否有全面的流程文档，包括如何处理缺陷？如果答案是肯定的，则该组织处于第三级或更高级别。如果答案是否定的，那么第二个问题就是，“有人知道是怎么回事吗？”如果答案是肯定的，那就是第二级；如果没有，一级。

一级组织没有处理问题的标准方法。当出现问题时，每个人都抓起工具，试图找出问题出在哪里，如何解决。像这样的组织不会在培训或分析上花费太多。他们的重点是继续生产他们想生产的东西。随着时间的推移，一个人可能会发展出诊断某个组件的专业技能，当出现问题时，就会发出“让 Fred 进来”的呼叫解决问题。拥有专业知识的组织正在进入第二级。大多数组织属于这两个级别中的一个。

组织转型非常困难，通常需要管理团队的重大变革，以及为不同的活动提供资金。培训和沟通技巧对于超越第二级至关重要。管理层会奖励那些能解决最棘手的 bug 的英雄。他们得到高额奖金、晋升、更好的办公室、靠近门口的停车位。这种行为强化了英雄文化。但是向前发展需要英雄们扮演新的角色。

一旦组织创建了文档，它就走上了第三级的道路。请注意，这些转换是痛苦的。很难告诉英雄们，他们对组织的最大价值是他们能写得多好或教得多好。但只要管理层给予适当的关注，这是可以做到的。而且往前走的好处很多:

*   危机显著减少。员工可以制定计划并坚持执行——没有紧急情况会打断家庭聚会、学校活动或朋友聚会。
*   **高质量代码。**维护任务变得更加简单，客户体验到了更高的可靠性。文档很有帮助，故障排除变得很平常，而不是让人不知所措。
*   **可靠的规划。**在一个成熟的组织中，计划是正确的，因为它们基于成熟的衡量标准、持续验证的流程和团队内一致的高能力。项目评估是准确的，因为数据来源于可靠的、可重复的证据。

卓越云不是一个幻影或无法实现的目标。它是清晰的思维和完善的文档的结果。随着时间的推移，实践会改进，技能会提高。引用麦克摩尔的话，“伟人不是因为出生时会画画才伟大，而是因为他们画了很多画才伟大。”  通过实践和专注，您的组织可以实现卓越的云计算。
