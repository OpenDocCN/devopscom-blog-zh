# 为什么你应该使用 GitOps 来试验人工智能

> 原文:[https://devo PS . com/why-you-should-use-gitops-to-experiment-with-ai/](https://devops.com/why-you-should-use-gitops-to-experiment-with-ai/)

随着人工智能( [AI](https://devops.com/?s=AI) )模型的构建和不可避免的更新速度开始加快，数据科学团队目前进行实验的速度也需要大幅加快，这一点变得越来越明显。

今天，如果许多数据科学团队能够在 12 个月的时间里成功地在生产环境中部署两个人工智能模型，那么他们就是幸运的。然而，部分由于数字业务转型的兴起，已经很明显，每个企业 It 环境中可能有数百个人工智能模型的用例。说实话，很难想象今天正在开发的应用程序会*而不是*包含一个或多个人工智能模型。挑战在于，对人工智能能力的需求现在远远超过了数据科学团队的交付能力。

解决这个问题的最好方法是认识到人工智能模型和用来构建它们的组件实际上只是另一种软件产品。事实上，使用工件加速应用程序开发的需求在很大程度上是一个已经解决的问题。与其重新发明轮子，不如把我们已经知道的东西应用到人工智能模型的开发中。

## 基于 GitOps 的流程的兴起

如今，现代软件开发团队经常使用基于 Git 的库来共享代码和协作。这些存储库塞满了经过审查的软件工件，开发团队通过基于一组 GitOps 最佳实践的 DevOps 工作流的固执己见的实例不断地重用这些软件工件。

作为 DevOps 的一个子集， [GitOps](https://devops.com/?s=gitops) 工作流使数据科学团队能够利用软件开发环境中已经存在的 Git 存储库来管理工件。Git 支持使用 Git 分支来创建文件的版本。他们还采用了标记(Git Tag)和远程(服务器所在位置)的概念。这三个不同的概念或引用存在于 Git 中。事实上，整个流程都是基于这些概念。当一个新的特性被实现时，一个新的分支被创建，然后被合并到生产分支中，并带有一个标签，类似于“这是一个需要被部署的新版本”

像任何其他软件一样，人工智能模型也需要更新。随着新的数据源变得可用，人工智能模型很可能会偏离最初打算的参数。商业环境的快速变化也可能需要人工智能模型的改变。DevOps 团队应用于更新任何软件工件的相同过程也可以应用于快速更新 AI 模型。使用 Git 存储库来管理 AI 工件将软件开发团队所享受的好处扩展到了数据科学团队。

通常，组织会有一个 MLOps 解决方案来简化 AI 模型开发，它完全独立于他们的 DevOps 工具。这些解决方案将具有专门的功能，如数据的版本跟踪和专门针对人工智能模型的实验。但许多 MLOps 解决方案存在于一个封闭的生态系统中，这意味着它们不依赖于 DevOps 端使用的 GitHub 等现有工具。

此外，数据科学团队必须单独购买、部署和管理这些封闭的 MLOps 解决方案。采用面向 GitOps 的解决方案简化了对添加的 MLOps 工具的管理，并使应用程序开发人员和数据科学团队更容易了解彼此的工作。最终结果不仅仅是以更低的成本进行更快的人工智能实验；更多的人工智能模型成功部署在生产环境中的可能性大大增加。

## 软件和人工智能团队一起工作

DevOps 和 MLOps 的融合已经不可避免。组织继续采用孤立的 MLOps 平台来构建和部署 AI 模型的时间越长，融合 MLOps 和 DevOps 流程的成本就越大。

数据科学团队天生保守。成功与失败的区别往往在于人工智能模型得到验证的彻底程度。然而，不必牺牲上市时间来确保人工智能模型如承诺的那样工作。当竞争对手竞相开发类似的人工智能能力时，不存在迟到的伟大人工智能模型——只有错过的机会。工件的可重用性使得使用 GitOps 工作流进行实验的速度比以往任何时候都要快。
